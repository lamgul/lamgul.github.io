folder — 아키텍처 패턴을 극한까지 밀어붙인 시범연구

2024년 3월• 아키텍처 · 패턴 · folder

한동안 저는 아키텍처 패턴을 책으로만 알고 있었습니다. 이름은 외웠는데 언제 써야 좋은지는 손끝으로 알지 못했습니다. 그래서 folder라는 이름의 작은 팀을 만들어, 패턴을 실제 코드에 한번 극한까지 밀어붙여 보기로 했습니다. 대단한 산출물을 내겠다는 계획은 없었고, 특별한 성적을 낸 것도 아닙니다. 다만 패턴이 어디까지 힘이 되고 어디부터 짐이 되는지를 직접 부딪혀 확인해 보려던 시범연구였고, 그때 알게 된 것을 잊지 않으려고 적어 둡니다.

먼저 짚고 가기
시범연구Pilot Study
본격적인 연구나 작업에 들어가기 전에, 작은 규모로 먼저 돌려 보며 방법이 실제로 통하는지·어디서 걸리는지를 확인하는 예비 단계입니다. 목적은 완성된 결과물이 아니라 다음에 무엇을 어떻게 할지에 대한 판단을 얻는 데 있습니다. folder에서 하려던 것도 그것이라, 잘 만든 시스템을 남기기보다 패턴을 실제로 적용해 보며 감을 얻는 쪽에 무게를 두었습니다.
Strategy 패턴Strategy
GoF가 정리한 행위 패턴으로, 서로 바꿔 쓸 수 있는 여러 알고리즘을 각각 하나의 객체로 캡슐화해 두고, 사용하는 쪽이 런타임에 그중 하나를 골라 끼우게 하는 방식입니다. 조건 분기(if·switch)로 흩어질 로직을 이름 있는 타입으로 나눠 교체와 확장을 쉽게 하는 것이 목적이라, 실제로 갈아 끼울 알고리즘이 여럿일 때 값을 합니다.
Template Method 패턴Template Method
알고리즘의 골격을 상위 클래스의 한 메서드에 고정해 두고, 그 안에서 달라지는 일부 단계만 하위 클래스가 재정의하도록 남겨 두는 패턴입니다. 전체 절차의 순서는 상위가 통제하고 세부만 아래로 위임하므로, 여러 구현이 큰 흐름은 같고 일부 단계만 다른 경우에 어울립니다.
Repository 패턴Repository
도메인 객체의 저장·조회를 마치 메모리 속 컬렉션을 다루듯 다룰 수 있게 감싸, 도메인 코드와 실제 데이터 매핑 사이를 중개하는 계층입니다. 도메인이 SQL이나 ORM의 어휘를 직접 알지 않아도 되게 만드는 것이 핵심이라, 저장 기술을 도메인에서 떼어 놓고 싶을 때 씁니다.
Pipe & FilterPipes and Filters
데이터를 독립된 처리 단계(필터)들에 차례로 흘려보내고, 단계 사이는 파이프로 잇는 아키텍처 스타일입니다. 각 필터가 입력을 받아 변환한 결과를 다음 필터에 넘기므로, 필터끼리 서로를 모르는 채로 순서를 바꾸거나 새 단계를 끼워 넣기 좋습니다. 처리가 단계로 자연스럽게 나뉘고 그 구성이 자주 바뀌는 데이터 흐름에 맞습니다.

folder라는 이름의 작은 실험실

folder는 거창한 조직이 아니라, 같은 것을 궁금해하던 사람 몇이 모여 만든 스터디에 가까웠습니다. 규칙은 하나였습니다. 배운 패턴은 슬라이드로만 넘기지 말고 돌아가는 코드에 끝까지 적용해 본다는 것이었습니다. 자료는 주로 책에서 가져왔습니다. Martin Fowler의 《엔터프라이즈 애플리케이션 아키텍처 패턴》, Eric Evans의 《도메인 주도 설계》, Robert C. Martin의 《클린 아키텍처》를 돌려 읽었고, 파이썬과 데이터 처리 쪽 책들도 함께 봤습니다.

초반에 정한 방향이 나중까지 성격을 갈랐습니다. 저희는 패턴을 "적당히" 쓰지 않고 일부러 과하게 써 보기로 했습니다. 적당히 쓰면 어디가 과한지 영영 모를 것 같았기 때문입니다. 작은 기능 하나에도 인터페이스를 세우고, 분기 하나에도 Strategy를 붙이고, 절차 하나에도 Template Method를 얹었습니다. 그렇게 밀어붙이고 나서야 비로소 "여기는 아니었다"가 눈에 보였습니다.

극한까지 밀어붙인다는 것

가장 먼저 손댄 것은 회원 등급별 할인 계산이었습니다. 규칙은 단순했습니다. 일반 회원은 할인이 없고 VIP는 10%를 뺀다, 그게 전부였습니다. 저희는 여기에 Strategy 패턴을 교과서대로 전부 적용했습니다. 알고리즘을 타입으로 나누고, 고르는 책임은 팩토리로 빼고, 공통 계약은 추상 기반 클래스로 세웠습니다.

# 등급별 할인율을 Strategy로 "제대로" 나눈 코드
from abc import ABC, abstractmethod

class DiscountStrategy(ABC):
    @abstractmethod
    def rate(self) -> float: ...

class NormalDiscount(DiscountStrategy):
    def rate(self) -> float:
        return 0.0

class VipDiscount(DiscountStrategy):
    def rate(self) -> float:
        return 0.1

class DiscountStrategyFactory:
    _table = {"normal": NormalDiscount, "vip": VipDiscount}

    def create(self, grade: str) -> DiscountStrategy:
        return self._table[grade]()

def final_price(price: int, grade: str) -> int:
    strategy = DiscountStrategyFactory().create(grade)
    return round(price * (1 - strategy.rate()))

동작은 완벽했고 리뷰도 통과했습니다. 그런데 며칠 뒤 "숫자 하나만 바꾸면 되는 일에 파일 네 개를 여닫고 있다"는 것을 알아차렸습니다. 할인율을 8%로 조정하는 데도 클래스를 찾아 들어가야 했고, 새 등급을 추가하면 클래스 하나와 팩토리 등록 한 줄이 또 늘었습니다. 이 코드가 지키려던 것은 "알고리즘이 여럿일 때의 교체 가능성"인데, 정작 여기엔 교체할 알고리즘이 없었습니다. 있는 것은 등급마다 다른 숫자 하나뿐이었습니다. 같은 규칙을 데이터로 두면 이렇게 줄었습니다.

# 알고리즘이 아니라 숫자였으므로, 표 하나로 충분했다
RATES = {"normal": 0.0, "vip": 0.1}

def final_price(price: int, grade: str) -> int:
    return round(price * (1 - RATES[grade]))

패턴을 걷어내고 나니 규칙이 오히려 한눈에 들어왔습니다. Strategy가 틀린 도구였다기보다, 변하는 것이 로직이 아니라 값인 자리에 쓴 것이 어긋남이었습니다. folder에서 이런 순간을 여러 번 만났습니다. Template Method로 감쌌던 절차가 알고 보니 단계가 하나뿐이었고, Repository로 격리했던 테이블이 규칙 없는 단순 조회였습니다. 과하게 써 봤기 때문에, 무엇이 과한지가 비로소 몸에 남았습니다.

패턴이 제값을 한 자리

반대편도 있었습니다. 수집 데이터를 정제하는 흐름에는 같은 패턴 계열이 오히려 잘 맞았습니다. 원본 레코드에서 품절을 걸러 내고, 가격 문자열을 숫자로 바꾸고, 수집원 표시를 붙이는 단계들이 있었는데, 이 단계 구성이 수집원마다 달랐고 자주 바뀌었습니다. 어떤 곳은 품절 정보를 안 주고, 어떤 곳은 가격 형식이 또 달랐습니다. 여기에 Pipe & Filter를 적용했더니, 각 단계는 서로를 모른 채 자기 일만 하고 조립하는 쪽에서 순서만 바꿔 끼우면 됐습니다.

# 정제 단계가 자주 바뀌어서 Pipe & Filter가 맞았다
from typing import Callable, Iterable

Record = dict
Filter = Callable[[Record], Record | None]  # None을 반환하면 그 레코드는 버린다

def drop_out_of_stock(r: Record) -> Record | None:
    return None if r["stock"] == 0 else r

def normalize_price(r: Record) -> Record:
    r["price"] = int(r["price"].replace(",", "").rstrip("원"))
    return r

def tag_source(source: str) -> Filter:
    def _apply(r: Record) -> Record:
        r["source"] = source
        return r
    return _apply

def run_pipeline(rows: Iterable[Record], filters: list[Filter]) -> list[Record]:
    out = []
    for r in rows:
        for f in filters:
            r = f(r)
            if r is None:
                break
        if r is not None:
            out.append(r)
    return out

# 수집원마다 단계 구성만 바꿔 끼운다
pipeline = [drop_out_of_stock, normalize_price, tag_source("shop-a")]

여기서는 패턴이 값을 했습니다. 새 수집원이 붙어도 파이프라인 골격은 그대로였고, 단계를 함수 하나로 추가해 목록에 끼우면 끝이었습니다. 앞의 할인 코드와 무엇이 달랐나를 나중에 정리해 보니 답은 한 문장이었습니다. 여기엔 실제로 갈아 끼우고 다시 배열할 무언가가 있었다는 점입니다. 변하는 것이 값이 아니라 구성 그 자체였고, 그때 패턴은 짐이 아니라 힘이었습니다.

힘과 짐을 가르는 선

두 경우를 나란히 놓고 보니, 패턴이 힘이 되는지 짐이 되는지를 가르는 것은 패턴의 우열이 아니었습니다. 변하는 것이 무엇인가였습니다. 변하는 것이 값 하나면 표에 담는 편이 나았고, 변하는 것이 알고리즘이나 단계 구성이면 그것을 이름 있는 타입으로 세우는 값이 있었습니다. 극한까지 밀어붙여 본 덕분에, 저는 이 구분을 규칙이 아니라 감각으로 갖게 됐습니다.

*

folder에서 스스로에게 물어보던 것을 적어 둡니다. "이 추상을 세우면 실제로 두 번째 구현이 생기는가." 두 번째가 그려지면 패턴을 세웠고, 하나뿐인데 미래를 위해 세우고 있다면 대개 과한 자리였습니다. 지금 없는 유연성을 위해 지금 있는 단순함을 내주고 있던 셈이니까요.

과하게 적용한 코드가 남긴 비용도 분명했습니다. 파일이 늘어 흐름을 따라가기 어려웠고, 같은 필드를 여러 곳에 반복해서 적는 의식(儀式)이 생겼고, 무엇보다 정작 중요한 규칙이 구조에 묻혀 잘 보이지 않았습니다. 반대로 알맞게 쓴 자리에서는 구조가 오히려 규칙을 드러내 주었습니다. 같은 패턴이 자리에 따라 이렇게 다르게 작동한다는 것을, 저는 두 코드를 직접 짜 보고 나서야 믿게 됐습니다.

남은 것 — 균형 감각

folder로 대단한 성과를 낸 것은 아닙니다. 내세울 결과물도, 특별한 성적도 없습니다. 다만 시작할 때 얻고 싶던 것은 얻었다고 생각합니다. 패턴의 정의가 아니라 적용의 균형 감각입니다. 어디까지가 힘이고 어디부터가 과함인지, 이제는 코드를 보면 어렴풋이 가늠이 됩니다. 이 감각은 책을 한 번 더 읽어서가 아니라, 일부러 과하게 써 보고 그 비용을 직접 치러 보고 나서 생겼습니다.

패턴을 극한까지 밀어붙여 본 경험은 이후 설계에 계속 남았습니다. 새 추상을 세우려 할 때마다 folder에서 물었던 질문이 먼저 떠올랐고, 덕분에 필요 없는 구조를 세우기 전에 한 번 멈추게 됐습니다. 이 패턴들이 어떤 이름으로 정리돼 있는지는 Fowler의 카탈로그를 읽고 나서 더 또렷해졌는데, 그 이야기는 서평에 따로 적어 두었습니다.
