논문 분석: Raft — 이해할 수 있는 합의 알고리즘

2020년 9월• Raft · 분산시스템 · 논문

분산 시스템에서 합의(consensus)는 피해 갈 수 없는 주제인데, 오랫동안 표준처럼 쓰이던 Paxos에는 배우기 어렵다는 말이 늘 따라다녔습니다. Diego Ongaro와 John Ousterhout의 2014년 논문 In Search of an Understandable Consensus Algorithm은 바로 그 지점을 문제로 삼습니다. 성능이나 새로운 보장이 아니라 이해 가능성(understandability)을 1급 설계 목표로 놓은 논문입니다. 읽기만 해서는 손에 잡히지 않아, 핵심인 리더 선출과 로그 복제를 실행 가능한 미니 코드로 옮겨 가며 읽었습니다.

출처

In Search of an Understandable Consensus Algorithm

Diego Ongaro, John Ousterhout · USENIX ATC 2014

공식 PDF ↗ · USENIX ATC ↗

먼저 짚고 가기
합의consensus
여러 서버가 하나의 값 — Raft에서는 명령 로그의 각 자리에 어떤 명령이 오는가 — 에 동의하고, 한번 확정된 결정은 이후에도 뒤집히지 않게 보장하는 문제입니다. Raft와 Paxos가 가정하는 장애 모델은 노드가 멈추거나 메시지가 지연·유실·중복되는 경우(crash fault)까지이고, 거짓 응답을 꾸며 보내는 비잔틴 장애는 다루지 않습니다. 서버 2f+1대 중 f대가 죽어도 남은 과반이 서로 통신되면 새 결정을 이어 갈 수 있으며, 과반이 깨지면 잘못된 값을 고르는 대신 진행을 멈춥니다.
임기term
Raft가 시간을 구획하는 데 쓰는 단조 증가 정수, 즉 논리적 시계입니다. 각 노드는 자기가 본 가장 큰 값을 currentTerm으로 디스크에 남기고 모든 RPC에 실어 보내며, 자기보다 큰 term을 받으면 그 값으로 갱신한 뒤 즉시 follower로 물러납니다. 투표는 term당 한 표뿐이고 당선에는 과반이 필요하므로, 한 term에 리더는 최대 한 명입니다(Election Safety). 벽시계와는 무관해서 term 값에서 경과 시간을 읽을 수 없고, 선거가 실패하면 리더 없이 지나가는 term도 생깁니다.
로그 복제log replication
리더가 클라이언트 명령을 { term, command } 항목으로 자기 로그 끝에 붙인 뒤, AppendEntries RPC로 팔로워의 로그를 자기 로그와 같아지도록 맞추는 절차입니다. 요청에는 직전 항목의 위치와 임기(prevLogIndex, prevLogTerm)가 함께 실리고, 팔로워는 자기 로그의 그 자리가 일치할 때만 뒤를 이어 붙입니다. 항목은 리더에서 팔로워로 한 방향으로만 흐르므로 서로 맞춰 가는 협상은 없고, 어긋난 팔로워의 로그는 일치 지점까지 잘려 나간 뒤 리더의 것으로 덮입니다. 로그 중간에 빈 자리(구멍)는 허용되지 않습니다.
상태 머신state machine
합의 알고리즘이 전제로 삼는 실행 모형으로, 같은 초기 상태에서 같은 명령을 같은 순서로 적용하면 반드시 같은 상태에 도달하는 결정적(deterministic) 계산을 뜻합니다. 그래서 Raft는 상태 자체를 주고받지 않고 명령의 순서 — 로그 — 만 복제하며, 각 서버가 커밋된 항목을 index 순서대로 적용하면 상태 일치는 결정성에서 따라옵니다. 뒤집어 말하면 명령이 현재 시각이나 난수처럼 노드마다 다른 값을 읽는 순간 로그가 같아도 상태는 갈라집니다. 결정성은 Raft가 보장해 주는 것이 아니라 응용이 지켜야 할 조건입니다.
커밋commit
로그 항목이 과반 서버의 로그에 저장되어 앞으로 어떤 리더가 와도 지워지지 않는다고 확정된 상태이며, 노드는 자기 commitIndex 이하의 항목만 상태 머신에 적용하고 클라이언트에 응답합니다. 판정은 리더가 내리는데, 리더는 자기 임기의 항목이 과반에 복제될 때만 복제 수를 세어 commitIndex를 올립니다. 이전 임기의 항목은 이미 과반에 있어도 그 사실만으로는 커밋으로 치지 않고, 현재 임기의 항목이 커밋될 때 로그 매칭 프로퍼티에 의해 간접적으로 함께 커밋됩니다(§5.4.2). 팔로워는 스스로 판단하지 않고 AppendEntries에 실려 오는 leaderCommit을 따라 자기 commitIndex를 올립니다.

1. 이해 가능성이라는 설계 목표

논문이 출발하는 자리는 Paxos의 서술 방식입니다. Paxos는 정확성이 증명된 알고리즘이지만 단일 값 합의(single-decree)를 중심으로 서술되어 있어서, 실제 시스템이 필요로 하는 복제 로그까지 가는 길은 논문에 나와 있지 않습니다. 그래서 실무 구현들은 저마다 "Paxos 비슷한 무언가"를 만들게 됐고, 증명된 것과 배포된 것 사이가 벌어졌다고 저자들은 적습니다.

Raft는 이 간극을 두 가지 방법으로 줄입니다. 하나는 문제 분해로, 합의를 리더 선출·로그 복제·안전성·멤버십 변경이라는 독립적인 조각으로 나눠 하나씩 설명합니다. 다른 하나는 상태 공간 축소인데, 로그에 구멍을 허용하지 않고 로그가 항상 리더에서 팔로워로 한 방향으로만 흐르게 해서(strong leader) 고려해야 할 경우의 수를 줄입니다. 저자들은 이 목표를 실증까지 해 두었습니다. 두 대학 학생 43명에게 Raft와 Paxos를 각각 강의하고 퀴즈를 치르게 했더니 Raft 쪽 평균 점수가 더 높았고, 설문에서도 대다수가 구현하기 쉬운 쪽으로 Raft를 꼽았습니다. 이해하기 쉽다는 말을 주장이 아니라 측정으로 보인 셈인데, 시스템 논문에서 이런 절은 처음 봤습니다.

2. 세 가지 역할과 임기 — 상태 머신으로 읽기

Raft 노드는 언제나 follower, candidate, leader 세 상태 중 하나에 있습니다. 시간은 임기(term)라는 단조 증가 정수로 구획되고, 한 임기에 리더는 최대 한 명입니다(Election Safety). 기본 합의에 쓰이는 RPC는 두 개뿐이라, 표를 요청하는 RequestVote와 로그를 밀어 넣으면서 하트비트를 겸하는 AppendEntries가 전부입니다(스냅샷 전송용 InstallSnapshot이 뒤쪽 장에서 하나 더 붙습니다). 이 골격을 코드로 옮겨 보면 이렇습니다.

// 노드는 세 상태 중 하나 — 모든 규칙은 두 개의 RPC 위에서 움직인다
const Role = { FOLLOWER: "follower", CANDIDATE: "candidate", LEADER: "leader" };

class RaftNode {
  constructor(id, peers) {
    this.id = id; this.peers = peers;
    this.role = Role.FOLLOWER;
    this.term = 0;          // currentTerm: 단조 증가하는 논리적 시계
    this.votedFor = null;   // 이 임기에 투표한 후보 — 임기당 1표
    this.log = [];          // { term, command } 의 배열, 구멍 없음
    this.commitIndex = 0;
  }
  // 모든 RPC에 공통인 단 하나의 강등 규칙:
  // 더 높은 term을 보면 즉시 follower로 내려간다
  observeTerm(term) {
    if (term > this.term) {
      this.term = term; this.votedFor = null;
      this.role = Role.FOLLOWER;
    }
  }
}

임기가 곧 논리적 시계 역할을 하기 때문에, "낡은 리더가 돌아와서 명령을 내리면 어떻게 되나" 같은 경우들이 observeTerm 한 규칙으로 정리됩니다. 낡은 term의 요청은 거부되고, 낡은 노드는 더 큰 term을 보는 순간 스스로 물러납니다. 규칙이 이만큼 줄어든 덕분에 상태 전이가 그림 한 장(Figure 4)에 다 들어갑니다.

3. 리더 선출 — 무작위 타임아웃이 하는 일

follower는 리더의 하트비트를 받지 못한 채 선출 타임아웃이 지나면 candidate가 되어 term을 올리고 표를 모읍니다. 이때 여러 노드가 동시에 후보로 나서면 표가 갈려 아무도 과반을 얻지 못하는 분할 투표(split vote)가 생깁니다. Paxos 계열이라면 노드 순위 같은 메커니즘을 얹었을 텐데, Raft는 타임아웃을 노드마다 무작위로 주는 것으로 이 문제를 넘어갑니다. 논문이 예시로 쓰는 구간은 150~300ms입니다. 먼저 깨어난 노드가 먼저 표를 가져가고, 충돌하더라도 다음 라운드의 무작위성이 다시 흩어 줍니다.

// 리더 선출 미니 시뮬레이션 뼈대 — requestVote 전송부(RPC)는 생략
const rand = () => 150 + Math.random() * 150;  // 150~300ms 무작위 타임아웃

function resetTimer(node) {
  clearTimeout(node.timer);
  node.timer = setTimeout(() => startElection(node), rand());
}

function startElection(node) {
  node.role = Role.CANDIDATE;
  node.term += 1; node.votedFor = node.id;     // 자기 자신에게 1표
  let votes = 1;
  for (const peer of node.peers) {
    const granted = requestVote(peer, {
      term: node.term, candidateId: node.id,
      lastLogIndex: node.log.length,
      lastLogTerm: node.log.at(-1)?.term ?? 0,
    });
    if (granted && ++votes > (node.peers.length + 1) / 2) {
      node.role = Role.LEADER;
      console.log(`term ${node.term}: ${node.id} 당선 (${votes}표)`);
      return;  // 이후엔 주기적 AppendEntries(하트비트)로 재선출을 막는다
    }
  }
  resetTimer(node);  // 과반 실패(분할 투표) → 다음 무작위 타임아웃에 재도전
}

저자들은 처음에 순위 기반 선출을 검토했다가 미묘한 코너 케이스가 늘어나자, 무작위화가 더 단순하고 이해하기 쉽다는 이유로 그 방식을 버렸다고 적어 두었습니다. 결정적인 분기를 무작위성으로 덮어 상태 공간을 줄인 셈인데, 이해 가능성이라는 목표가 알고리즘의 모양까지 바꾼 대목이라 오래 기억에 남았습니다.

글로만 읽으니 무작위 타임아웃이 실제로 어떻게 엉킴을 푸는지 잘 그려지지 않아서, 위 코드의 규칙을 그대로 옮긴 시뮬레이터를 만들어 두었습니다. 노드 다섯이 각자 무작위 타임아웃 게이지를 채우다가, 먼저 만료된 노드가 후보로 나서 투표를 요청하고 과반(3표)을 얻으면 리더가 됩니다. 리더가 된 뒤에는 주기적인 하트비트(회색 점)가 다른 노드의 게이지를 계속 0으로 되돌려 재선거를 막습니다. [리더 죽이기]를 누르면 하트비트가 끊긴 노드들의 게이지가 다시 차오르고, term이 올라가며 재선거가 일어납니다. 운이 나쁘면 두 노드가 동시에 후보로 나서 표가 갈리는 분할 투표도 나오는데, 그다음 라운드의 무작위 타임아웃이 교착을 푸는 과정이 눈에 보입니다.

노드 둘레의 게이지가 선거 타임아웃입니다 — 먼저 만료된 노드가 후보가 되고, 과반(3표)을 얻으면 리더가 됩니다. 오가는 점은 투표 요청(주황) · 득표(청록) · 하트비트(회색)입니다.

*

논문의 타이밍 조건은 broadcastTime ≪ electionTimeout ≪ MTBF입니다. etcd 같은 Raft 구현을 운영할 때 디스크가 느려지거나 네트워크 RTT가 길어져 하트비트 왕복(broadcast time)이 타임아웃에 가까워지면, 멀쩡한 리더를 두고도 재선출이 반복됩니다. 리더가 자꾸 바뀔 때 코드보다 디스크 fsync 지연과 네트워크를 먼저 보게 되는 이유이기도 합니다. 논문은 선출 타임아웃을 하트비트 왕복 시간보다 최소 한 자릿수(약 10배) 크게 잡기를 권합니다.

4. 로그 복제와 로그 매칭 프로퍼티

리더가 뽑히면 클라이언트 명령은 리더의 로그에 붙고, AppendEntries로 팔로워에 복제되며, 과반에 복제되는 순간 커밋되어 상태 머신에 적용됩니다. 이 부분에서 중심이 되는 정리가 로그 매칭 프로퍼티(Log Matching Property)입니다.

"서로 다른 두 로그에서 같은 index에 같은 term을 가진 항목이 있다면, 두 항목은 같은 명령을 담고 있고, 그 이전의 로그 전체도 동일하다."

이 성질은 저절로 얻어지는 것이 아니라 AppendEntries가 매번 수행하는 일관성 검사를 통해 귀납적으로 유지됩니다. 리더는 새 항목과 함께 "직전 항목의 index와 term"(prevLogIndex, prevLogTerm)을 보내고, 팔로워는 자기 로그의 그 자리와 일치할 때만 받아들입니다. 파이썬으로 옮겨 보니 검사 전체가 함수 하나에 들어갔습니다.

# 로그 매칭 프로퍼티를 유지하는 AppendEntries 일관성 검사 (논문 §5.3)
def append_entries(node, term, prev_index, prev_term, entries, leader_commit):
    if term < node.term:              # 낡은 리더의 요청은 거부
        return False
    # prev 자리의 (index, term)이 일치해야만 이어붙인다 — 귀납의 한 걸음
    if prev_index > 0:
        if len(node.log) < prev_index or node.log[prev_index - 1].term != prev_term:
            return False              # 리더는 nextIndex를 낮춰 재시도한다
    for i, entry in enumerate(entries):
        pos = prev_index + i
        if pos < len(node.log) and node.log[pos].term != entry.term:
            del node.log[pos:]        # 충돌: 그 지점 이후를 전부 잘라낸다
        if pos >= len(node.log):
            node.log.append(entry)
    if leader_commit > node.commit_index:
        node.commit_index = min(leader_commit, len(node.log))
    return True

검사가 실패하면 리더는 그 팔로워의 nextIndex를 하나 낮춰 다시 보냅니다. 두 로그가 일치하는 지점까지 내려간 다음, 거기서부터는 리더의 로그로 덮어씁니다. 팔로워의 로그가 리더와 달라도 서로 맞춰 가는 협상 과정은 없고, 리더의 로그가 그대로 기준이 됩니다. 이 단방향성 덕분에 저는 Paxos를 볼 때보다 흐름을 훨씬 수월하게 따라갈 수 있었습니다.

5. 안전성 — 표를 주지 않을 권리

여기까지 읽고 남은 의문은 리더가 교체되는 순간이었습니다. 리더가 죽고 새 리더가 뽑힐 때 커밋된 항목을 갖고 있지 않은 노드가 리더가 되면, 커밋된 로그가 덮여 사라질 수 있습니다. Raft는 이 문제를 커밋 시점이 아니라 선출 시점에 막습니다(선거 제한, §5.4.1). 투표자는 후보의 마지막 로그가 자기 로그보다 낡았으면 표를 주지 않습니다. 커밋된 항목은 과반에 존재하므로, 과반의 표를 얻은 리더는 반드시 그 항목을 가지고 있게 됩니다(Leader Completeness).

# §5.4.1 선거 제한 — "내 로그보다 낡은 후보에게는 표를 주지 않는다"
def is_up_to_date(my_log, last_log_index, last_log_term):
    my_term = my_log[-1].term if my_log else 0
    if last_log_term != my_term:
        return last_log_term > my_term    # 마지막 항목의 term이 크면 더 최신
    return last_log_index >= len(my_log)  # term이 같으면 더 긴 로그가 최신

def request_vote(node, term, candidate_id, last_log_index, last_log_term):
    if term < node.term:
        return False
    if node.voted_for in (None, candidate_id) \
       and is_up_to_date(node.log, last_log_index, last_log_term):
        node.voted_for = candidate_id         # 임기당 한 표, 선착순
        return True
    return False

여기에 규칙이 하나 더 붙는데, 문장만으로는 이유가 잘 잡히지 않았습니다. 리더는 자기 임기의 항목만 복제 수를 세어 커밋합니다. 이전 임기의 항목이 과반에 복제되어 있어도 그것만으로는 커밋으로 치지 않고, 현재 임기의 항목이 커밋될 때 로그 매칭에 의해 간접적으로 함께 커밋됩니다. 논문의 Figure 8이 이 규칙이 없을 때 커밋된 항목이 덮여 사라지는 반례를 단계별로 보여 주는데, 이 논문에서 제가 가장 오래 붙들고 본 그림입니다.

6. 논문 밖에서 — 무엇을 가져갈 것인가

Raft가 자리 잡은 과정은 논문 바깥에서 확인됩니다. 쿠버네티스의 저장소인 etcd, Consul, TiKV 같은 시스템이 Raft를 채택했고, 여러 언어로 구현이 나와 있습니다. 이해할 수 있으니 구현이 나왔고, 구현이 쌓이니 표준처럼 쓰이게 된 셈입니다.

저는 합의 알고리즘을 직접 구현할 일이 없는 백엔드 개발자지만, 이 논문에서 가져온 것이 두 가지 있습니다. 하나는 문제를 보는 눈입니다. 여러 워커가 나눠 도는 크롤링 파이프라인에서 "지금 이 작업의 주인이 누구인가"를 정하는 문제나, 정산 리팩토링에서 겪은 "어디까지가 확정된 상태인가"라는 질문은 결국 리더 선출과 커밋의 축소판이었습니다. 문제에 이름을 붙여 분해하고 나면 락 하나, 상태 컬럼 하나를 어디에 둘지를 정하기가 한결 쉬워집니다.

다른 하나는 태도입니다. 이 논문은 읽는 사람을 설계에 포함시켰습니다. 더 배우기 쉬운 쪽을 고르기 위해 알고리즘의 모양을 바꿨고, 그 선택이 정확성을 해치지 않음을 증명했으며, 이해 가능성 자체를 실험으로 측정했습니다. 코드 리뷰에서 "동작은 같은데 왜 이렇게 고치자는 거예요?"라는 질문에 어떻게 답해야 할지를, 저는 이 논문을 읽으며 정리할 수 있었습니다.
