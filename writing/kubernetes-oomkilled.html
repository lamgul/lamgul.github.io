쿠버네티스 requests/limits — OOMKilled의 진짜 이유

2025년 7월• 약 10분• Kubernetes · 메모리 · 운영

kubectl describe에 찍힌 OOMKilled는 쿠버네티스가 내린 판결처럼 보입니다. 하지만 실제로 프로세스를 죽인 것은 리눅스 커널입니다. limits가 어떻게 cgroup 값이 되는지 작은 실험 매니페스트로 눈으로 확인하고, 그 위에서 JVM 컨테이너가 유독 잘 죽는 전형적 패턴을 정리합니다.

먼저 짚고 가기
requests / limitsrequests / limits
컨테이너에 배정하는 메모리·CPU의 두 값입니다. requests는 "이만큼은 예약해 줘"라는 최소 요청이고, limits는 "여기까지만 써"라는 최대 한도입니다. 방을 예약할 때 잡아 두는 최소 인원(requests)과 정원(limits)에 비유할 수 있습니다. 이 글은 이 둘이 서로 다른 층에서 작동한다는 점에서 출발합니다.
cgroupcontrol group
리눅스 커널이 프로세스 묶음에 쓸 수 있는 자원(메모리·CPU 등)을 제한·측정하는 기능입니다. 쿠버네티스의 limits는 결국 이 cgroup의 설정 파일 값으로 바뀌어 커널이 직접 집행합니다. 즉 한도를 지키게 만드는 진짜 주체는 쿠버네티스가 아니라 커널입니다.
OOMKilled / OOM killerOut Of Memory
메모리가 한도를 넘으면 리눅스 커널의 OOM killer가 프로세스를 강제로 종료시키는데, 그 결과를 쿠버네티스가 파드 상태에 OOMKilled라고 적어 둡니다. 경고 없이 프로세스가 죽기 때문에, 로그가 갑자기 끊기는 장애의 흔한 원인입니다.
힙Heap
JVM(자바 가상 머신)이 객체를 담아 두는 메모리 영역입니다. 다만 자바 프로세스가 쓰는 전체 메모리는 힙보다 큽니다 — 힙 밖에도 스레드 스택·메타스페이스 같은 "세입자"들이 있어서, 힙만 계산하면 컨테이너가 한도를 넘겨 죽는 함정에 빠집니다.

1. requests와 limits는 다른 기계를 움직인다

둘은 한 쌍처럼 쓰이지만 작동하는 층이 다릅니다. requests는 스케줄러의 언어입니다. "이 파드를 어느 노드에 놓을까"를 계산할 때 쓰는 예약값이고, 런타임에 사용량을 막지 않습니다. requests가 100Mi인 파드도 노드에 여유가 있으면 그 이상을 씁니다. 반면 limits는 커널의 언어입니다. kubelet이 컨테이너 런타임을 거쳐 cgroup에 기록하는 강제 한도이고, 스케줄러는 이 값을 보지 않습니다.

그리고 자원의 성격에 따라 초과의 결과가 다릅니다. CPU limits를 넘으면 스로틀링으로 느려질 뿐이지만, 메모리는 압축할 수 없는 자원이라 limits를 넘는 순간 커널 OOM killer가 프로세스를 죽입니다. 이 비대칭이 이 글 전체의 핵심입니다. 실험에 쓸 매니페스트는 이렇습니다.

# memory-demo.yaml — limits 안쪽(150M)만 쓰는 얌전한 파드
apiVersion: v1
kind: Pod
metadata:
  name: memory-demo
spec:
  containers:
  - name: stress
    image: polinux/stress
    resources:
      requests:
        memory: "100Mi"
      limits:
        memory: "200Mi"
    command: ["stress"]
    args: ["--vm", "1", "--vm-bytes", "150M", "--vm-hang", "1"]

kind든 minikube든 로컬 클러스터면 충분합니다. stress가 150M를 할당한 채 버티므로, limits 200Mi 안에서 멀쩡히 돌아갑니다.

2. limits는 cgroup 파일이 된다 — 눈으로 확인

이 파드에 들어가 cgroup 파일을 읽어보면, YAML에 적은 문자열이 커널 설정값이 되어 있는 것을 그대로 볼 수 있습니다.

$ kubectl apply -f memory-demo.yaml
$ kubectl exec memory-demo -- cat /sys/fs/cgroup/memory.max
209715200        # = 200Mi (200 × 1024 × 1024), cgroup v2

# cgroup v1 노드라면 경로가 다릅니다
$ kubectl exec memory-demo -- cat /sys/fs/cgroup/memory/memory.limit_in_bytes
209715200

전달 경로는 단순합니다. YAML의 limits → kubelet → 컨테이너 런타임 → cgroup의 memory.max. 여기까지 오고 나면 쿠버네티스의 역할은 끝났습니다. 이후 이 한도를 집행하는 것은 순수하게 리눅스 커널입니다. cgroup의 메모리 사용량이 memory.max에 닿으면 커널이 그 cgroup 안에서 희생자를 골라 SIGKILL을 보내고, kubelet은 사후에 그 결과를 읽어 파드 상태에 OOMKilled라고 적어줄 뿐입니다. "쿠버네티스가 죽였다"는 말은 절반만 맞는 셈입니다.

3. 일부러 죽여본다 — 137이라는 서명

이제 같은 매니페스트에서 --vm-bytes만 250M으로 올려 limits(200Mi)를 넘겨봅니다. 컨테이너는 곧바로 죽고, describe에 전형적인 흔적이 남습니다.

$ kubectl describe pod memory-demo
    ...
    Last State:     Terminated
      Reason:       OOMKilled
      Exit Code:    137
    Restart Count:  3
    ...
Events:
  Warning  BackOff  ...  Back-off restarting failed container

137 = 128 + 9. 시그널 9번, 즉 SIGKILL로 죽었다는 뜻입니다. 이 숫자가 중요한 이유는 SIGKILL은 가로챌 수 없는 시그널이기 때문입니다. 애플리케이션은 종료 훅도, 마지막 로그 한 줄도, graceful shutdown도 수행하지 못합니다. 로그가 아무 예고 없이 뚝 끊기고 재시작 카운트만 올라가 있다면, 가장 먼저 의심할 것이 이 경로입니다.

4. JVM 컨테이너가 죽는 전형적 패턴

이 무대 위에서 JVM은 두 가지 방향으로 자주 미끄러집니다. 요즘 JVM(JDK 8u191+, JDK 10+)은 UseContainerSupport가 기본이라 cgroup 한도를 "가용 메모리"로 올바르게 인식하지만, 문제는 그다음의 기본 비율입니다. 최대 힙의 기본값은 가용 메모리의 25%(MaxRAMPercentage=25)입니다.

$ kubectl exec jvm-app -- java -XX:+PrintFlagsFinal -version | grep -i maxheapsize
   size_t MaxHeapSize = 268435456    {ergonomic}
# limits.memory가 1Gi인데 힙은 256Mi — 기본 25%의 결과

실패 1 — 기본값 방치. 1Gi를 예약해 놓고 힙은 256Mi만 쓰는 상태입니다. 나머지는 놀고, 힙이 좁아 GC가 일찍부터 바빠집니다. 그래서 -XX:MaxRAMPercentage=75.0처럼 비율을 올리는 것이 컨테이너의 정석입니다.

실패 2 — 힙만 계산에 넣는 것. 반대로 90% 이상으로 올리면 이번엔 커널에게 죽습니다. cgroup이 세는 것은 힙이 아니라 프로세스 전체이기 때문입니다. 힙 밖에도 메타스페이스, 스레드 스택(64비트 리눅스 기본 1MB × 스레드 수), 코드 캐시, 다이렉트 버퍼, GC 오버헤드가 삽니다. 힙이 한도의 90%를 차지하면 이 나머지가 10%를 넘는 순간 — 힙은 멀쩡한데도 — 컨테이너가 137로 사라집니다. 75%는 이 "힙 밖 세입자들"의 몫을 남겨두는 경험칙입니다.

# Deployment 컨테이너 스펙 — 힙 비율과 힙덤프 준비를 함께
containers:
- name: api
  image: registry.example.com/api:1.4.2
  env:
  - name: JAVA_TOOL_OPTIONS
    value: >-
      -XX:MaxRAMPercentage=75.0
      -XX:+HeapDumpOnOutOfMemoryError
      -XX:HeapDumpPath=/var/log/app/heap.hprof
  resources:
    requests:
      memory: "1Gi"
    limits:
      memory: "1Gi"
*

죽음의 종류를 먼저 구분하세요. 로그에 java.lang.OutOfMemoryError가 있으면 힙 문제이고 HeapDumpOnOutOfMemoryError가 힙덤프를 남겨줍니다. 로그 없이 exit code 137이면 컨테이너 예산 문제이고, SIGKILL이라 힙덤프도 남지 않습니다. 137 앞에서 힙덤프 파일을 찾아 헤매는 것이 가장 흔한 시간 낭비입니다.

5. 운영에서의 결론 — 예산을 정하고, 실측한다

메모리에 한해서는 requests = limits로 맞추는 편을 권합니다. 메모리는 CPU처럼 잠깐 양보할 수 있는 자원이 아니어서, 오버커밋된 노드가 압박을 받으면 파드 축출로 이어집니다. requests와 limits를 같게 두면 스케줄러가 예약한 만큼만 쓰는 정직한 파드가 되고, CPU까지 같게 맞추면 QoS도 Guaranteed가 됩니다. 그리고 75%든 얼마든 비율은 출발점일 뿐이라, 실제 부하에서 container_memory_working_set_bytes를 보며 한도와의 여유를 실측해야 합니다.

이 감각은 온프레미스에서 vLLM을 서빙하며 gpu-memory-utilization과 동시 슬롯 수의 균형을 잡던 일과 정확히 같은 종류였습니다. 예산을 정하고, 예산 안의 세입자를 전부 세고, 실제 트래픽으로 스윗스팟을 확인한다 — 자원이 VRAM이든 cgroup 메모리든 순서는 같습니다. 그리고 한도가 아니라 애플리케이션 안쪽이 문제로 판명되면, 그때부터는 힙덤프를 읽는 일이 시작됩니다. 7일마다 죽던 서버를 추적했던 순서를 아래 글에 정리해 두었습니다.
