쿠버네티스 requests/limits — OOMKilled의 진짜 이유

2025년 7월• Kubernetes · 메모리 · 운영

kubectl describe에 OOMKilled가 찍혀 있는 것을 보고, 한동안은 쿠버네티스가 파드를 죽인 것이라고 생각했습니다. 실제로 프로세스를 종료시킨 쪽은 리눅스 커널이었습니다. limits가 어떻게 cgroup 값이 되는지 작은 실험 매니페스트로 확인해 보고, 그 위에서 JVM 컨테이너가 자주 죽던 전형적 패턴을 함께 적어 둡니다.

먼저 짚고 가기
requests / limitsrequests / limits
파드 스펙의 spec.containers[].resources 아래에 선언하는 두 필드로, 같은 자원에 대해 서로 다른 주체가 읽습니다. requests는 kube-scheduler가 "이 노드에 남은 할당량으로 파드를 받을 수 있는가"를 계산할 때 쓰는 예약값이라, 런타임 사용량을 막지 않습니다. limits는 kubelet이 컨테이너 런타임을 거쳐 cgroup 한도로 기록하는 강제 상한이고, 집행은 커널이 합니다. 두 값의 조합이 파드의 QoS 클래스(Guaranteed / Burstable / BestEffort)도 결정합니다.
cgroupcontrol group
프로세스를 계층 트리로 묶어 메모리·CPU 같은 자원의 사용을 제한하고 집계하는 리눅스 커널 기능입니다. /sys/fs/cgroup 아래 파일 인터페이스로 노출되며, 메모리 상한 필드는 v2가 memory.max, v1이 memory.limit_in_bytes입니다. 쿠버네티스가 하는 일은 limits 값을 이 파일에 써 넣는 데까지이고, 초과를 판정하고 집행하는 주체는 커널입니다. 세는 단위도 개별 프로세스가 아니라 그 cgroup에 속한 프로세스 전체의 사용량 합입니다.
OOMKilled / OOM killerOut Of Memory
OOM killer는 메모리를 회수할 수 없을 때 커널이 희생자 프로세스를 골라 SIGKILL을 보내는 커널 루틴입니다. cgroup 한도를 넘어 발생한 경우에는 시스템 전체가 아니라 그 cgroup 안에서만 대상을 고릅니다. OOMKilled는 원인이 아니라 그 종료를 kubelet이 사후에 읽어 컨테이너 상태의 Reason 필드에 적어 둔 문자열이고, 종료 코드는 128+9인 137로 남습니다. SIGKILL은 핸들러를 등록할 수 없는 시그널이라 종료 훅도 힙덤프도 남지 않습니다.
힙Heap
JVM이 생성한 객체와 배열을 담고 GC가 회수를 담당하는 메모리 영역으로, 상한은 -Xmx 또는 -XX:MaxRAMPercentage로 정합니다. 자바 프로세스가 실제로 점유하는 메모리는 힙에 메타스페이스·스레드 스택·코드 캐시·GC 내부 구조체·다이렉트 버퍼가 더해진 값이라 언제나 힙보다 큽니다. cgroup이 재는 값은 이 합계이므로, -Xmx를 limits에 가깝게 맞추면 힙에는 여유가 있는데도 컨테이너가 종료됩니다. 즉 힙 상한은 프로세스 전체 메모리의 상한이 아닙니다.

1. requests와 limits는 다른 기계를 움직인다

둘은 한 쌍처럼 쓰이지만 작동하는 층이 다릅니다. requests는 스케줄러가 보는 값입니다. "이 파드를 어느 노드에 놓을까"를 계산할 때 쓰는 예약값이고, 런타임에 사용량을 막지는 않습니다. 그래서 requests가 100Mi인 파드도 노드에 여유가 있으면 그 이상을 씁니다. 반면 limits는 커널이 보는 값이라, kubelet이 컨테이너 런타임을 거쳐 cgroup에 강제 한도로 기록하고 스케줄러는 이 값을 보지 않습니다.

자원의 성격에 따라 한도를 넘겼을 때의 결과도 다릅니다. CPU limits를 넘으면 스로틀링으로 느려질 뿐이지만, 메모리는 압축할 수 없는 자원이라 limits를 넘는 순간 커널 OOM killer가 프로세스를 종료시킵니다. 이 비대칭이 아래 내용의 바탕입니다. 실험에 쓴 매니페스트는 이렇습니다.

# memory-demo.yaml — limits 안쪽(150M)만 쓰는 얌전한 파드
apiVersion: v1
kind: Pod
metadata:
  name: memory-demo
spec:
  containers:
  - name: stress
    image: polinux/stress
    resources:
      requests:
        memory: "100Mi"
      limits:
        memory: "200Mi"
    command: ["stress"]
    args: ["--vm", "1", "--vm-bytes", "150M", "--vm-hang", "1"]

kind든 minikube든 로컬 클러스터면 충분했습니다. stress가 150M를 할당한 채 버티므로, limits 200Mi 안에서 문제없이 돌아갑니다.

2. limits는 cgroup 파일이 된다 — 눈으로 확인

이 파드에 들어가 cgroup 파일을 읽어보면, YAML에 적은 문자열이 커널 설정값이 되어 있는 것을 그대로 확인할 수 있었습니다.

$ kubectl apply -f memory-demo.yaml
$ kubectl exec memory-demo -- cat /sys/fs/cgroup/memory.max
209715200        # = 200Mi (200 × 1024 × 1024), cgroup v2

# cgroup v1 노드라면 경로가 다릅니다
$ kubectl exec memory-demo -- cat /sys/fs/cgroup/memory/memory.limit_in_bytes
209715200

전달 경로는 단순합니다. YAML의 limits가 kubelet과 컨테이너 런타임을 거쳐 cgroup의 memory.max에 적히고, 거기까지가 쿠버네티스의 몫입니다. 이후 이 한도를 집행하는 것은 리눅스 커널이라, cgroup의 메모리 사용량이 memory.max에 닿으면 커널이 그 cgroup 안에서 희생자를 골라 SIGKILL을 보냅니다. kubelet은 사후에 그 결과를 읽어 파드 상태에 OOMKilled라고 적어 둘 뿐입니다. 그래서 "쿠버네티스가 죽였다"는 말은 절반만 맞는 표현이었습니다.

3. 일부러 죽여본다 — 137이라는 서명

같은 매니페스트에서 --vm-bytes만 250M으로 올려 limits(200Mi)를 넘겨 봤습니다. 컨테이너는 곧바로 종료됐고, describe에 전형적인 흔적이 남았습니다.

$ kubectl describe pod memory-demo
    ...
    Last State:     Terminated
      Reason:       OOMKilled
      Exit Code:    137
    Restart Count:  3
    ...
Events:
  Warning  BackOff  ...  Back-off restarting failed container

137 = 128 + 9, 시그널 9번인 SIGKILL로 종료됐다는 뜻입니다. SIGKILL은 애플리케이션이 가로챌 수 없는 시그널이라 종료 훅도, 마지막 로그 한 줄도, graceful shutdown도 수행되지 않습니다. 로그가 예고 없이 끊기고 재시작 카운트만 올라가 있을 때 이 경로를 먼저 확인하게 된 것도 그래서입니다.

4. JVM 컨테이너가 죽는 전형적 패턴

이 구조 위에서 JVM은 두 방향으로 자주 미끄러집니다. 요즘 JVM(JDK 8u191+, JDK 10+)은 UseContainerSupport가 기본이라 cgroup 한도를 "가용 메모리"로 올바르게 인식하는데, 걸리는 것은 그다음의 기본 비율이었습니다. 최대 힙의 기본값은 가용 메모리의 25%(MaxRAMPercentage=25)입니다.

$ kubectl exec jvm-app -- java -XX:+PrintFlagsFinal -version | grep -i maxheapsize
   size_t MaxHeapSize = 268435456    {ergonomic}
# limits.memory가 1Gi인데 힙은 256Mi — 기본 25%의 결과

실패 1 — 기본값 방치. 1Gi를 예약해 놓고 힙은 256Mi만 쓰는 상태입니다. 남은 공간은 놀고, 힙이 좁으니 GC는 일찍부터 바빠집니다. 그래서 -XX:MaxRAMPercentage=75.0처럼 비율을 직접 올려 두는 편이 낫습니다.

실패 2 — 힙만 계산에 넣는 것. 반대로 90% 이상으로 올리면 이번엔 커널이 프로세스를 종료시킵니다. cgroup이 세는 것은 힙이 아니라 프로세스 전체이기 때문입니다. 힙 밖에도 메타스페이스, 스레드 스택(64비트 리눅스 기본 1MB × 스레드 수), 코드 캐시, 다이렉트 버퍼, GC 오버헤드가 함께 삽니다. 힙이 한도의 90%를 차지하면 이 나머지가 10%를 넘는 순간, 힙은 멀쩡한데도 컨테이너가 137로 종료됩니다. 75%는 이 "힙 밖 세입자들"의 몫을 남겨두는 경험칙입니다.

# Deployment 컨테이너 스펙 — 힙 비율과 힙덤프 준비를 함께
containers:
- name: api
  image: registry.example.com/api:1.4.2
  env:
  - name: JAVA_TOOL_OPTIONS
    value: >-
      -XX:MaxRAMPercentage=75.0
      -XX:+HeapDumpOnOutOfMemoryError
      -XX:HeapDumpPath=/var/log/app/heap.hprof
  resources:
    requests:
      memory: "1Gi"
    limits:
      memory: "1Gi"
*

종료의 종류를 먼저 구분해 두면 헤매는 시간이 줄어듭니다. 로그에 java.lang.OutOfMemoryError가 있으면 힙 문제이고 HeapDumpOnOutOfMemoryError가 힙덤프를 남겨줍니다. 로그 없이 exit code 137이면 컨테이너 예산 문제이고, SIGKILL이라 힙덤프도 남지 않습니다. 137 앞에서 힙덤프 파일을 찾아 헤매다 시간을 쓰기 쉬운 지점입니다.

5. 운영에서의 결론 — 예산을 정하고, 실측한다

메모리에 한해서는 requests = limits로 맞추는 편이 마음이 편했습니다. 메모리는 CPU처럼 잠깐 양보할 수 있는 자원이 아니어서, 오버커밋된 노드가 압박을 받으면 파드 축출로 이어집니다. 두 값을 같게 두면 스케줄러가 예약한 만큼만 쓰는 파드가 되고, CPU까지 같게 맞추면 QoS도 Guaranteed가 됩니다. 다만 75%든 얼마든 비율은 출발점일 뿐이라, 실제 부하에서 container_memory_working_set_bytes를 보며 한도와의 여유를 실측해 두는 편이 좋습니다.

돌아보면 온프레미스에서 vLLM을 서빙하며 gpu-memory-utilization과 동시 슬롯 수의 균형을 잡던 일과 비슷한 종류였습니다. 예산을 정하고, 예산 안의 세입자를 전부 세고, 실제 트래픽으로 스윗스팟을 확인하는 순서는 자원이 VRAM이든 cgroup 메모리든 크게 다르지 않았습니다. 한도가 아니라 애플리케이션 안쪽이 문제인 경우도 있었는데, 그때는 힙덤프를 읽어야 했습니다. 7일마다 죽던 서버를 추적했던 순서는 아래 글에 정리해 두었습니다.
