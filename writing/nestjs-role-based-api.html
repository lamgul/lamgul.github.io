관리자와 근로자는 같은 데이터를 다르게 쓴다 — NestJS 역할 분리

2026년 3월• 약 7분• NestJS · Guard · 권한

SafeIn에는 두 종류의 사용자가 있습니다. 전체를 조망하고 처리하는 관리자, 자기 것만 보고 신고하는 근로자. 둘은 같은 Report 엔티티를 만지지만, 볼 수 있는 것과 할 수 있는 것이 완전히 다릅니다. 이걸 한 컨트롤러에 if (user.isAdmin)로 욱여넣으면 지옥이 열립니다.

먼저 짚고 가기
엔티티Entity
데이터베이스의 한 종류를 코드로 표현한 설계도입니다. 예를 들어 "신고서" 한 장 한 장의 모양(누가·언제·무슨 내용)을 정의한 Report가 엔티티입니다. 이 글은 관리자와 근로자가 같은 엔티티를 서로 다르게 다루는 이야기이므로 출발점이 됩니다.
Guard가드
요청이 컨트롤러에 도착하기 직전에 통과 여부를 검사하는 문지기입니다. 놀이공원 입구에서 표를 확인하는 직원처럼, "이 사람이 이 요청을 해도 되는가"를 먼저 판단해 걸러 냅니다. 권한 검사를 이 문지기에게 맡겨 컨트롤러 본문에서 덜어 내는 것이 핵심입니다.
DTOData Transfer Object
클라이언트가 서버로 보낼 수 있는 데이터의 모양과 규칙을 정해 둔 양식입니다. 신청서에 적을 수 있는 칸을 미리 정해 두는 것과 같아서, 정해지지 않은 칸(예: status)은 적어 보내도 무시됩니다. 근로자와 관리자에게 서로 다른 양식을 주어 입구를 좁힙니다.
화이트리스트whitelist
"허용할 것만 명단에 올리고, 명단에 없으면 전부 버린다"는 방식입니다. 반대로 "나쁜 것만 골라 막는" 블랙리스트보다 안전한데, 예상 못 한 값이 새어 들어올 틈을 원천 봉쇄하기 때문입니다. DTO에 정의 안 한 필드가 자동으로 걸러지는 근거입니다.
상태 전이State Transition
어떤 대상이 정해진 단계 사이를 옮겨 가는 것입니다. 신호등이 초록→노랑→빨강 순서로만 바뀌듯, 신고서도 "신고됨→확인→처리완료"처럼 허용된 순서로만 넘어가야 데이터가 어긋나지 않습니다.

먼저 모듈로 가른다

관리자 API와 앱 API를 별도 모듈로 분리했습니다. 공통 도메인 로직은 서비스 계층에 두고, 컨트롤러는 역할별로 나눠 노출 표면 자체를 다르게 했습니다. 근로자 App에는 관리자용 엔드포인트가 아예 존재하지 않습니다.

권한은 Guard로, 선언적으로

// 역할을 메타데이터로 선언 → Guard가 강제
@Roles('admin')
@UseGuards(JwtAuthGuard, RolesGuard)
@Patch('reports/:id/resolve')
resolve(@Param('id') id: string, @CurrentUser() admin: User) {
  return this.reports.resolve(id, admin.id);  // 처리 이력이 남는다
}

RolesGuard는 핸들러의 @Roles 메타데이터를 읽어 요청자의 역할과 대조합니다. 권한 검사가 비즈니스 로직에서 빠져나와 선언으로 바뀌니, 컨트롤러가 하는 일이 한눈에 보입니다.

DTO로 입구를 좁힌다

근로자가 보내는 신고와 관리자가 보내는 처리는 다른 DTO입니다. class-validator로 필드를 검증하고, 근로자 DTO에는 status 같은 필드를 아예 정의하지 않아 넘겨도 무시되게 했습니다. 화이트리스트 방식(whitelist: true)이 기본입니다.

상태와 이력은 한곳에서

근로자의 신고가 관리자의 처리로 이어지는 흐름에서, 상태 전이는 서비스의 단일 메서드를 거치게 했습니다. "신고됨 → 확인 → 처리완료"가 아무 데서나 바뀌면 정합성이 깨집니다. 전이 규칙을 한곳에 모으고, 각 전이에서 이력 레코드를 남깁니다. 안전 도메인에서 "누가 언제 무엇을 했나"는 기능이 아니라 요구사항입니다.
