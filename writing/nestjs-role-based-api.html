관리자와 근로자는 같은 데이터를 다르게 쓴다 — NestJS 역할 분리

2026년 3월• NestJS · Guard · 권한

SafeIn에는 사용자가 두 종류 있습니다. 전체를 조망하고 처리하는 관리자와, 자기 것만 보고 신고하는 근로자입니다. 둘은 같은 Report 엔티티를 쓰지만 볼 수 있는 범위와 할 수 있는 일이 서로 달랐습니다. 한 컨트롤러 안에서 if (user.isAdmin)로 갈라 볼까 하다가, 분기가 늘어날수록 어느 줄이 누구를 위한 코드인지 알기 어려워질 것 같아 모듈과 Guard, DTO 세 군데에서 나눠 두었습니다. 그때 정한 기준을 적어 둡니다.

먼저 짚고 가기
엔티티Entity
ORM이 데이터베이스 테이블 하나에 대응시켜 관리하는 클래스입니다. 각 프로퍼티가 컬럼에 매핑되고, 개별 레코드의 동일성은 필드 값이 같은지가 아니라 기본 키로 판별합니다. 엔티티가 나타내는 것은 저장 구조일 뿐이라, 클라이언트가 보내거나 받는 데이터의 모양과는 다른 층입니다. 이 글에서 관리자와 근로자가 공유하는 것은 Report 엔티티 한 벌이고, 갈라지는 것은 그 위에 얹은 접근 경로입니다.
Guard가드
NestJS에서 CanActivate 인터페이스를 구현한 클래스로, canActivate()가 돌려준 boolean 하나가 이 요청을 핸들러까지 보낼지를 결정합니다. 요청 처리 순서상 미들웨어 다음·파이프보다 앞에서 실행되며, false를 반환하면 Nest가 ForbiddenException(403)을 던집니다. 인자로 받는 ExecutionContext에 대상 핸들러와 클래스가 들어 있어서, Reflector로 거기 붙은 메타데이터를 읽어 판단 근거로 쓸 수 있습니다. 다만 파이프보다 먼저 도는 만큼 검증·변환을 마친 DTO는 볼 수 없어, 본문 값에 따라 갈리는 판단은 Guard의 몫이 아닙니다.
DTOData Transfer Object
계층 경계를 넘는 데이터의 형태를 정의해 둔 클래스입니다. NestJS에서는 class-validator 데코레이터로 필드마다 규칙을 적어 두고, ValidationPipe가 class-transformer로 요청 본문을 그 클래스의 인스턴스로 바꾼 뒤 규칙을 검사합니다. 타입만 적은 인터페이스로는 같은 일을 할 수 없는데, 컴파일 후에도 런타임까지 남아 데코레이터를 붙일 수 있는 쪽은 클래스이기 때문입니다. 그래서 DTO는 그 자체로 무엇을 막지 않고, ValidationPipe가 걸려 있을 때에만 규칙으로 작동합니다.
화이트리스트whitelist
허용할 항목만 열거하고 목록에 없는 것은 전부 거부하는 기본 차단(default-deny) 방식의 설계 원칙입니다. 금지 항목을 나열하는 블랙리스트와의 차이는, 명단을 갱신하지 않아도 새로 등장한 미지의 입력이 자동으로 막힌다는 데 있습니다. NestJS에서는 ValidationPipe의 whitelist: true가 이 역할을 맡아, 검증 데코레이터가 하나도 붙지 않은 프로퍼티를 요청 본문에서 조용히 잘라 냅니다. 잘라 내는 대신 400으로 되돌려주려면 forbidNonWhitelisted: true를 따로 켜야 합니다.
상태 전이State Transition
유한 상태 기계에서 (현재 상태, 이벤트) 짝을 다음 상태로 대응시키는 규칙입니다. 허용된 짝만 정의해 두고 나머지 요청은 거부하면, "처리완료된 신고서가 다시 확인 중으로 돌아가는" 식의 있어서는 안 될 조합을 흩어진 조건문이 아니라 규칙표 하나로 막을 수 있습니다. 다만 규칙을 한곳에 모으는 것만으로는 두 요청이 같은 행을 동시에 읽고 각자 전이하는 경우까지 막지 못해서, 마지막 보장은 트랜잭션과 잠금 쪽에서 나옵니다.

먼저 모듈로 가른다

관리자 API와 앱 API를 별도 모듈로 분리했습니다. 공통 도메인 로직은 서비스 계층에 그대로 두고, 컨트롤러만 역할별로 나눠 노출 표면을 다르게 가져갔습니다. 이렇게 해 두면 근로자 App 쪽에는 관리자용 엔드포인트가 아예 존재하지 않아서, 권한을 잘못 열어 둘 여지가 그만큼 줄어듭니다.

권한은 Guard로, 선언적으로

// 역할을 메타데이터로 선언 → Guard가 강제
@Roles('admin')
@UseGuards(JwtAuthGuard, RolesGuard)
@Patch('reports/:id/resolve')
resolve(@Param('id') id: string, @CurrentUser() admin: User) {
  return this.reports.resolve(id, admin.id);  // 처리 이력이 남는다
}

RolesGuard는 핸들러에 붙은 @Roles 메타데이터를 읽어 요청자의 역할과 대조합니다. 권한 검사를 비즈니스 로직 안에서 하지 않고 선언으로 옮겨 두니, 나중에 컨트롤러만 훑어도 이 엔드포인트를 누가 쓸 수 있는지 알 수 있었습니다.

DTO로 입구를 좁힌다

근로자가 보내는 신고와 관리자가 보내는 처리는 서로 다른 DTO로 두었습니다. class-validator로 필드를 검증하고, 근로자 DTO에는 status 같은 필드를 정의하지 않았습니다. 그래서 클라이언트가 그 값을 실어 보내도 서버에서는 무시됩니다. 화이트리스트 방식(whitelist: true)을 기본으로 켜 둔 덕입니다.

상태와 이력은 한곳에서

근로자의 신고가 관리자의 처리로 이어지는 흐름에서, 상태 전이는 서비스의 단일 메서드를 거치게 했습니다. "신고됨 → 확인 → 처리완료"를 여러 곳에서 바꿀 수 있게 두면 정합성이 어긋나기 쉬워서, 전이 규칙을 한곳에 모으고 각 전이마다 이력 레코드를 남기도록 했습니다. 안전 도메인에서는 누가 언제 무엇을 했는지가 부가 기능이라기보다 요구사항에 가깝다고 봤습니다.
