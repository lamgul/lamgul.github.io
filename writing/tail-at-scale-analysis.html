논문 분석: The Tail at Scale — p99가 서비스를 결정한다

2025년 3월• 지연시간 · p99 · 논문

서버 한 대가 100번에 1번만 느리다면 꽤 훌륭한 서버입니다. 그런데 한 요청이 그런 서버 100대를 부채꼴로 병렬 호출한다면, 사용자 요청의 63%가 그 "100번에 1번"을 밟습니다. 훌륭한 부품 100개가 모여 형편없는 전체가 되는 이 산수가, Jeffrey Dean과 Luiz André Barroso가 2013년 CACM에 쓴 The Tail at Scale의 출발점입니다. 구글이 꼬리 지연을 어떻게 계산하고, 어떻게 감내하는지를 따라가며 읽었습니다.

출처

The Tail at Scale

Jeffrey Dean, Luiz André Barroso · Communications of the ACM 56(2) · 2013

공식 페이지 ↗ · PDF ↗

먼저 짚고 가기
꼬리 지연tail latency
가장 느린 소수의 요청이 겪는 지연입니다. 응답시간을 빠른 순으로 줄 세웠을 때 분포의 오른쪽 끝, 그러니까 "꼬리"에 해당하는 부분이라 이렇게 부릅니다. 평균 그래프에는 거의 드러나지 않습니다.
p9999th percentile
요청 100개를 빠른 순으로 세웠을 때 99번째 값입니다. "p99가 1초"라면 요청의 99%는 1초 안에 끝나지만 1%는 그보다 느리다는 뜻입니다. 평균과 달리 꼬리가 그대로 보이는 지표입니다.
팬아웃fan-out
한 요청이 여러 하위 서버로 부채꼴처럼 갈라져 병렬 호출되는 구조입니다. 검색처럼 데이터가 수백 대에 흩어져 있으면 전부에게 물어보고 답을 모아야 하는데, 이때 전체 속도는 가장 느린 한 대가 정합니다.
헤지 요청hedged request
첫 요청의 응답이 일정 시간 안에 안 오면 같은 요청을 다른 복제본에 한 발 더 보내고, 먼저 도착한 답을 쓰는 기법입니다. 보험을 들어두는(hedge) 것에 비유한 이름입니다.
마이크로 파티션micro-partition
데이터를 기계 수보다 훨씬 많은 조각으로 잘게 나눠 두는 분할 방식입니다. 어떤 기계가 느려지면 그 기계가 맡던 조각 일부만 다른 기계로 옮기면 되므로, 부하 이동과 장애 복구가 조각 단위로 빨라집니다.

1. 산수 — 개별 서버의 p99가 전체에선 p37이 된다

논문의 핵심 주장은 사실 곱셈 한 번입니다. 어떤 서버가 요청의 1%에서만 느리다고 합시다. 즉 그 서버의 p99 지연이 우리가 "느리다"고 부르는 기준선입니다. 한 사용자 요청이 이런 서버 n대를 병렬로 호출하고 전부의 응답을 모아야 답할 수 있다면, 요청이 빠르려면 n대가 전부 빨라야 합니다. 각 서버가 빠를 확률이 0.99이고 서버들이 서로 독립적으로 느려진다고 보면, 전체가 빠를 확률은 0.99n이고 n=100이면 0.99100 ≈ 0.366입니다. 뒤집으면 요청의 63.4%가 최소 한 대의 느린 서버를 밟는다는 뜻이죠.

같은 사실을 퍼센타일의 언어로 다시 말하면 이렇게 됩니다. 개별 서버에서는 100번에 1번만 나타나던 p99 수준의 지연이, 팬아웃 100대의 전체 요청에서는 3번에 2번 나타납니다. 개별의 p99 지연값이 전체 분포에서는 대략 p37 지점에 내려와 앉는 겁니다. 꼬리가 몸통이 되는 순간입니다. 논문은 'Component-Level Variability Amplified by Scale' 절에서 평균 10ms·p99 1초인 서버 100대의 예로 이 63%를 제시하고, 실제 구글 서비스 측정(Table 1)에서는 단말 서버 하나의 p99가 10ms인데 모든 단말을 기다리는 전체 요청의 p99는 140ms였다고 보고합니다. 특히 가장 느린 5%의 단말을 기다리는 시간이 전체 p99 지연의 절반을 차지했다는 대목이 뼈아픕니다.

# 개별 서버가 1% 확률로 "느린 쪽"일 때, n대 팬아웃이 꼬리를 밟을 확률
def p_tail(n, p=0.01):
    return 1 - (1 - p) ** n      # "전부 빠르다"의 여집합

for n in (1, 10, 50, 100):
    print(f"{n:>3}대 → {p_tail(n):6.1%}")

#   1대 →   1.0%
#  10대 →   9.6%
#  50대 →  39.5%
# 100대 →  63.4%  ← 개별 p99의 지연값이 전체에선 p36.6, 즉 p37 근처

이 곱셈이 무서운 건 지수적으로 악화되기 때문입니다. 서버를 열 대만 묶어도 이미 9.6% — 개별 p99가 전체의 p90으로 내려옵니다. 아래 위젯에서 서버 수를 직접 움직여 보면, "우리 서버는 다 빠른데 왜 전체가 느리지?"라는 질문의 답이 곡선 하나로 보입니다.

2. 서버는 왜 가끔 느려지는가

"그럼 서버가 안 느리게 만들면 되잖아요"가 자연스러운 반응인데, 논문의 'Why Variability Exists?' 절은 그게 왜 불가능에 가까운지를 목록으로 보여줍니다. 지연 변동의 출처는 코드 바깥 어디에나 있습니다.

  • 자원 공유 — 같은 기계 위의 다른 프로세스와 CPU 코어, 캐시, 메모리 대역폭, 네트워크를 나눠 씁니다. 이웃이 바쁘면 내가 느려집니다.
  • 백그라운드 데몬과 유지보수 작업 — 주기적으로 깨어나는 데몬, 로그 컴팩션, 가비지 컬렉션 같은 내부 청소가 순간적인 지연 스파이크를 만듭니다.
  • 전역 자원 — 네트워크 스위치나 공유 파일시스템처럼 기계 바깥에서 여러 기계가 함께 쓰는 자원도 경합 지점입니다.
  • 큐잉 — 운영체제, 네트워크, 애플리케이션 등 여러 계층의 큐에서 대기가 겹치며 변동이 증폭됩니다.
  • 하드웨어 스스로 — 전력 한도에 걸린 CPU의 스로틀링, SSD 내부의 가비지 컬렉션처럼 하드웨어도 성능이 일정하지 않습니다.

하나하나는 줄일 수 있어도 전부 없앨 수는 없습니다. 그리고 규모가 크면 드문 일이 일상이 됩니다. 기계 한 대에서 만 번에 한 번 나는 지연 스파이크도, 수천 대가 초당 수만 요청을 처리하는 시스템에서는 매초 일어나는 일이니까요. 그래서 논문은 관점을 바꿉니다. 장애 감내(fault-tolerance)가 신뢰할 수 없는 부품으로 신뢰할 수 있는 전체를 만드는 기술이듯, 예측할 수 없는 부품으로 예측 가능하게 반응하는 전체를 만들자 — 저자들은 이걸 tail-tolerant, 꼬리 감내라고 부릅니다. 이 관점 전환이 이 논문의 진짜 기여라고 생각합니다. 꼬리는 박멸의 대상이 아니라 설계의 전제입니다.

3. 짧은 호흡의 기법 — 헤지 요청과 tied request

요청 하나가 살아 있는 몇십 ms 안에서 즉석으로 대응하는 기법들부터 봅니다. 가장 단순한 아이디어는 "같은 요청을 복제본 두 곳에 쏘고 빠른 답을 쓰기"인데, 이러면 부하가 그대로 2배가 됩니다. 논문의 헤지 요청은 여기에 절제를 더한 버전입니다. 일단 한 발만 쏘고, 이 호출의 p95쯤 되는 시간까지 기다려도 답이 없으면 그때 두 번째 발을 다른 복제본에 쏩니다. 요청의 95%는 첫 발로 끝나므로 추가 부하는 최대 5% 언저리로 묶이면서, 꼬리의 대부분이 잘려나갑니다.

# 헤지 요청의 뼈대 — p95만큼 기다렸다가, 늦으면 복제본에 한 발 더
import asyncio

async def hedged_get(replicas, key, hedge_after=0.010):  # 10ms ≈ 이 호출의 p95
    first = asyncio.ensure_future(replicas[0].get(key))
    done, _ = await asyncio.wait({first}, timeout=hedge_after)
    if done:
        return first.result()        # 요청의 95%는 여기서 끝 — 추가 비용 0

    second = asyncio.ensure_future(replicas[1].get(key))  # 늦다 → 헤지 발사
    done, pending = await asyncio.wait(
        {first, second}, return_when=asyncio.FIRST_COMPLETED)
    for t in pending:
        t.cancel()                       # 진 쪽은 취소 — tied request의 아이디어로 이어진다
    return done.pop().result()

효과는 논문이 직접 측정해 보여줍니다. 100대 서버에 흩어진 1,000개 키를 BigTable에서 읽는 벤치마크에서, 10ms를 기다린 뒤 헤지를 보내는 것만으로 p99.9 지연이 1,800ms에서 74ms로 떨어졌습니다. 추가로 보낸 요청은 전체의 2%에 불과했고요('Hedged requests' 절). 두 자릿수 배의 개선을 한 자릿수 %의 비용으로 사는, 논문에서 가장 통쾌한 숫자입니다.

한 걸음 더 나간 것이 tied request입니다. 헤지는 두 번째 발을 쏜 뒤 둘 다 실행될 수 있다는 낭비가 남는데, 꼬리 지연의 큰 원인이 실행이 아니라 큐에서의 대기라는 점을 이용합니다. 요청을 처음부터 두 서버의 큐에 같이 넣되, 각 요청에 "상대가 누구인지"를 태그로 붙여둡니다. 한쪽 서버가 실행을 시작하는 순간 상대 서버에 취소를 보내, 다른 쪽 복사본은 큐에서 조용히 사라집니다('Tied requests' 절). 중복 실행 없이 "둘 중 덜 밀린 큐"를 고르는 효과를 얻는 셈입니다.

*

헤지 요청은 멱등한 읽기에만 안전합니다. 같은 요청이 두 번 실행될 수 있다는 전제이므로, 쓰기나 부수효과가 있는 호출에 걸면 사고가 됩니다. 그리고 임계값을 평균이 아니라 p95에 거는 것이 핵심입니다 — 임계값이 낮을수록 꼬리는 더 잘리지만 추가 부하가 커지는 트레이드오프이고, p95는 "추가 부하 5% 상한"이라는 예산을 명시적으로 정하는 선택입니다.

4. 긴 호흡의 기법 — 마이크로 파티션, 선택적 복제, 보호관찰

요청 단위의 응급처치와 별개로, 수십 초에서 분 단위로 시스템의 불균형 자체를 고쳐가는 기법들이 'Cross-Request Long-Term Adaptations' 절에 나옵니다.

먼저 마이크로 파티션. 데이터를 기계 수와 1:1로 나누면 한 기계가 느려졌을 때 옮길 수 있는 단위가 "그 기계 전부"라서 재배치가 느리고 거칩니다. 대신 기계 수보다 훨씬 많은 파티션을 만들어 한 기계가 여러 조각을 맡게 하면(BigTable의 태블릿이 이 방식입니다), 느려진 기계에서 조각 몇 개만 덜어내는 세밀한 부하 이동이 가능해집니다. 장애 복구도 빨라집니다. 죽은 기계의 조각들을 수십 대가 나눠 받으면 되니까요.

선택적 복제는 여기에 비대칭을 더합니다. 모든 데이터를 똑같이 복제하는 대신, 인기 있거나 지연에 민감한 항목의 복제본을 더 만들어 읽기 부하를 흩뿌립니다. 그리고 지연 유발 보호관찰(latency-induced probation) — 유난히 느려진 기계를 감지하면 일단 라우팅에서 빼되, 완전히 버리지는 않고 그림자 요청을 계속 보내며 상태를 지켜보다가 회복되면 복귀시킵니다. 흥미롭게도 논문은 이렇게 용량을 덜어내는데도 전체 지연이 오히려 좋아지는 일이 생긴다고 말합니다. 느린 한 대가 만드는 꼬리 비용이 그 한 대에서 얻는 처리 용량보다 비싸다는 뜻입니다.

대규모 검색 시스템을 위한 장치도 두 가지 눈에 띕니다('Large Information Retrieval Systems' 절). 충분히 좋은(good-enough) 응답 — 마감 시간이 되면 단말 전부를 기다리는 대신 도착한 결과만으로 응답합니다. 99%의 단말에서 온 결과로 만든 검색 결과가, 100%를 기다려 느려진 결과보다 사용자에게 낫다는 판단입니다. 그리고 카나리아 요청 — 수천 대 팬아웃 전에 한두 대에 먼저 요청을 보내 정상 응답을 확인합니다. 처음 보는 이상한 질의가 크래시를 일으킬 때, 수천 대가 동시에 같이 죽는 사태를 막는 안전핀입니다.

5. 규모가 작아도 산수는 같다 — SafeIn에서 느린 요청을 쫓던 경험

저는 수천 대 팬아웃과는 거리가 먼, SafeIn 백엔드를 혼자 설계하고 운영하는 입장에서 이 논문을 읽었습니다. 그런데도 1장의 산수가 남 얘기가 아니었습니다. 요청 하나가 처리되는 동안 DB를 몇 번 두드리고, Redis를 거치고, 외부 API를 호출한다면 — 곱셈의 지수가 "서버 100대"가 아니라 "한 요청이 거치는 하위 호출의 수"로 바뀔 뿐, 하나만 느려도 전체가 느려지는 구조는 동일하기 때문입니다. 직렬 호출이면 오히려 지연이 더해지기까지 하죠.

실제로 평균 응답시간 그래프는 늘 평온한데 "가끔 이상하게 느리다"는 제보가 이어지던 시기가 있었습니다. 평균과 p50은 꼬리를 숨깁니다. 요청 단위로 구간별 소요 시간을 로그에 남기고 느린 요청만 따로 모아 보기 시작하자, 범인은 대부분 특정 외부 호출과 무거운 조회 몇 개로 좁혀졌습니다. 외부 호출에는 타임아웃을 걸고 — 크롤링 파이프라인에서 백오프와 함께 몸에 익힌 습관입니다 — 반복되는 무거운 조회는 Redis 캐시로 앞단에서 끊었습니다. 논문의 언어로 옮기면, 헤지처럼 꼬리를 우회한 게 아니라 곱셈의 밑 p 자체를 줄이는 쪽이었죠. 규모가 작을 때는 그쪽이 먼저라고 지금도 생각합니다.

*

어느 쪽이든 관측이 먼저입니다. "어느 하위 호출이 꼬리를 만드는가"를 보지 못하면 헤지도 캐시도 과녁 없이 쏘는 것과 같습니다. 대시보드에 평균만 있다면 오늘 p99 패널부터 추가하세요 — 트래픽이 하루 수만 건만 되어도 p99 바깥의 1%는 매일 수백 번의 실제 사용자 경험입니다.

6. 정리 — 꼬리를 없애는 게 아니라 감내한다

  • 팬아웃 n대의 꼬리 확률은 1-(1-p)n으로 지수적으로 악화됩니다. 개별 서버의 p99 지연값이 100대 팬아웃에서는 전체의 p37 지점에 내려옵니다.
  • 지연 변동의 출처는 자원 공유, 데몬, GC, 큐잉, 하드웨어까지 어디에나 있어 제거가 아니라 감내가 설계 목표가 되어야 합니다 — fault-tolerant처럼, tail-tolerant.
  • 짧은 호흡: 헤지 요청(p95까지 기다렸다 한 발 더, 추가 부하 ~5% 예산)과 tied request(양쪽 큐에 넣고 시작하는 쪽이 상대를 취소).
  • 긴 호흡: 마이크로 파티션으로 세밀한 재배치를, 선택적 복제로 인기 데이터의 여유분을, 보호관찰로 느린 기계의 격리와 복귀를.
  • 규모가 작아도 한 요청이 거치는 홉 수가 같은 곱셈을 만듭니다. 퍼센타일 계측으로 꼬리를 먼저 보이게 만들고, 작은 시스템에서는 곱셈의 밑 p를 줄이는 것부터.

이 논문이 십 년 넘게 계속 인용되는 이유는 특정 기법 때문이 아니라 관점 때문이라고 생각합니다. 부품의 성능을 약속할 수 없다면 전체의 반응성을 설계로 약속하라는 것. 그리고 그 출발점은 언제나 같습니다 — 평균이 아니라 꼬리를 보는 눈을 갖추는 일입니다.
