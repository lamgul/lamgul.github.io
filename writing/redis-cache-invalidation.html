Redis로 목록을 캐싱할 때 진짜 어려운 건 무효화

2024년 9월• 약 7분• Redis · 캐시 · 무효화

"캐시 쓰면 되잖아요"는 절반만 맞습니다. 읽기를 빠르게 하는 cache-aside는 몇 줄이면 됩니다. 진짜 문제는 "언제 이 캐시를 버리나"입니다. 컴퓨터 과학의 오래된 농담이 괜히 있는 게 아니죠.

먼저 짚고 가기
캐시Cache
자주 쓰는 데이터의 사본을 빠른 저장소(여기선 Redis)에 잠깐 보관해 두는 것입니다. 원본 데이터베이스까지 매번 다녀오지 않아도 되니 읽기가 빨라집니다. 대신 원본이 바뀌면 사본은 낡아 버리는 게 이 글의 핵심 고민입니다.
cache-asideCache-Aside
캐시를 애플리케이션이 직접 챙기는 방식입니다. 먼저 캐시를 들여다보고, 없으면(miss) 원본에서 읽어 캐시에 채워 넣습니다. 구현이 단순해 가장 널리 쓰이지만, 언제 캐시를 지울지는 여전히 직접 정해야 합니다.
TTLTime To Live
캐시에 붙이는 "유통기한"입니다. 이 시간이 지나면 캐시가 자동으로 사라져 다음 요청 때 원본에서 다시 읽어 옵니다. 가장 손쉬운 무효화 방법이라, "얼마나 낡아도 괜찮은가"만 정하면 됩니다.
무효화Invalidation
낡은 캐시를 버려서 다음 읽기가 최신 원본을 가져오게 만드는 일입니다. 캐시가 어려운 이유의 대부분이 여기 있습니다 — 만드는 건 쉬워도, 제때 버리는 건 어렵습니다.
캐시 스탬피드Cache Stampede
인기 있는 캐시가 한꺼번에 만료되는 순간, 수많은 요청이 동시에 원본 DB로 몰려가 부하를 주는 현상입니다. 소 떼(stampede)가 한 방향으로 몰리는 모습에서 온 이름으로, 이 글의 방어 전략들이 겨냥하는 문제입니다.

기본: cache-aside

async get(key) {
  let v = await redis.get(key);
  if (v) return JSON.parse(v);      // hit
  v = await db.query(key);            // miss → 원본
  await redis.set(key, JSON.stringify(v), "EX", 300);  // TTL 5분
  return v;
}

TTL은 가장 값싼 무효화입니다. "최대 5분까지 낡아도 괜찮은가?"를 도메인에 물어 정하세요. 대시보드 통계는 5분이 괜찮고, 잔액은 아닙니다.

스탬피드 — 캐시가 동시에 만료될 때

인기 키의 TTL이 만료되는 순간, 수백 요청이 동시에 DB로 몰립니다(cache stampede). 방어는 두 가지 — 재계산을 단일 요청만 하도록 짧은 락을 걸거나, 만료 직전에 미리 갱신(probabilistic early recomputation)하는 것. 저는 핫 키에 락 방식을, 나머지에 TTL+지터를 씁니다.

이벤트 기반 무효화

TTL로 못 버티는 정합성이 필요하면, 데이터가 바뀌는 바로 그 순간 캐시를 지웁니다. 쓰기 경로에서 관련 키를 DEL하거나, 도메인 이벤트를 구독해 무효화 워커가 처리하게 합니다. 핵심은 무효화 지점을 한곳에 모으는 것 — 쓰기 로직 여기저기 redis.del이 흩어지면, 그게 바로 다음 버그의 씨앗입니다.

*

캐시는 성능을 위해 정합성을 조금 파는 거래입니다. 얼마나 낡아도 되는지를 도메인에 먼저 물어야, 무효화 전략이 정해집니다. 기술이 아니라 요구사항의 문제입니다.
