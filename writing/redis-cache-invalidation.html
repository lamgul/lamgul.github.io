Redis로 목록을 캐싱할 때 진짜 어려운 건 무효화

2024년 9월• Redis · 캐시 · 무효화

목록 조회를 Redis로 캐싱하면서, 읽기를 빠르게 만드는 cache-aside 자체는 몇 줄로 끝났습니다. 오래 붙잡고 있던 쪽은 언제 이 캐시를 버릴지 정하는 일이었습니다. TTL로 시작해 스탬피드 방어와 이벤트 기반 무효화까지, 그때 정리해 둔 순서를 적어 둡니다.

먼저 짚고 가기
캐시Cache
원본 데이터의 사본을 원본보다 접근이 빠른 저장소에 키로 찾을 수 있게 얹어 둔 계층입니다. 여기서는 인메모리 키-값 저장소인 Redis가 그 역할을 하고, 조회는 키 하나로 값을 찾아 있으면 hit, 없으면 miss로 갈리는 판정으로 끝납니다. 캐시에 있는 값은 사본일 뿐 진실의 원천(source of truth)이 아니라서, 원본이 바뀐 순간부터 낡기 시작하고 메모리 정책에 따라 만료 전에도 축출(eviction)될 수 있습니다.
cache-asideCache-Aside
애플리케이션 코드가 캐시와 원본 사이를 직접 중개하는 캐싱 패턴입니다. 읽기는 캐시를 먼저 조회하고, miss일 때만 원본에서 읽어 TTL과 함께 되채웁니다 — 값이 필요해진 시점에야 적재되므로 lazy loading이라고도 부릅니다. 캐시 계층이 스스로 원본을 읽어 오는 read-through와 달리 적재와 삭제 시점을 전부 애플리케이션이 정해야 하고, 같은 키에 동시에 miss가 나면 그 요청들이 서로를 모른 채 전부 원본으로 향한다는 약점이 여기서 나옵니다.
TTLTime To Live
키마다 따로 걸어 두는 만료 시각 메타데이터입니다. Redis는 값과 함께 만료 시각을 보관하고, 해당 키에 접근하는 순간(lazy expiration)이나 주기적인 백그라운드 샘플링(active expiration)으로 만료된 키를 지웁니다 — EXPIRE 명령이나 SET의 EX 옵션으로 초 단위를 지정합니다. 원본의 변경을 감지하는 게 아니라 시간만으로 끊는 방식이라, 값은 최대 TTL 길이만큼 낡은 채로 제공될 수 있습니다. 그래서 TTL을 정하는 일은 "이 데이터가 얼마나 낡아도 되는가"라는 도메인 허용치를 시간 값으로 옮겨 적는 일에 가깝습니다.
무효화Invalidation
원본이 바뀌었을 때 캐시의 해당 항목을 지우거나 만료로 표시해, 다음 읽기가 miss로 떨어져 최신 원본을 다시 적재하도록 만드는 조작입니다. 캐시를 새 값으로 곧장 덮어쓰는 갱신(write-through)과 달리, 버리기만 하고 다시 채우는 일은 다음 요청에 맡깁니다. 어려움은 "무엇을 지우는가"보다 "언제 지우는가"에 있습니다 — 원본 쓰기와 캐시 삭제, 그 사이에 끼어드는 재적재의 순서가 어긋나면 방금 지운 자리에 낡은 값이 다시 눌러앉을 수 있기 때문입니다.
캐시 스탬피드Cache Stampede
인기 있는 하나의 키가 만료·축출되는 순간, 그 키를 동시에 조회하던 요청들이 전부 miss로 떨어져 서로를 모른 채 각자 원본을 재계산하면서 부하가 순간적으로 몇 배로 튀는 동시성 문제입니다. thundering herd 또는 dogpile이라고도 부릅니다. 요청 수가 많다는 것 자체보다 재계산이 한 키에 겹친다는 데 원인이 있어서, 방어책도 재계산을 한 요청으로 묶는 락(single-flight)이나 만료 전 조기 갱신, TTL에 지터를 섞어 만료 시점을 흩뜨리는 쪽으로 향합니다.

기본: cache-aside

async get(key) {
  let v = await redis.get(key);
  if (v) return JSON.parse(v);      // hit
  v = await db.query(key);            // miss → 원본
  await redis.set(key, JSON.stringify(v), "EX", 300);  // TTL 5분
  return v;
}

TTL은 가장 값싸게 붙일 수 있는 무효화라, 저는 여기서부터 시작했습니다. 정하는 기준은 "이 데이터가 최대 5분까지 낡아도 괜찮은가"를 도메인에 물어보는 것이었습니다. 대시보드 통계는 5분이 지나도 괜찮았지만, 잔액처럼 항상 맞아야 하는 값에는 이 방식을 쓸 수 없었습니다.

스탬피드 — 캐시가 동시에 만료될 때

인기 있는 키의 TTL이 만료되는 순간에는 수백 개의 요청이 한꺼번에 DB로 몰립니다. 이걸 cache stampede라고 부릅니다. 방어하는 방법은 크게 두 가지인데, 재계산을 한 요청만 하도록 짧은 락을 걸거나, 만료 직전에 미리 갱신해 두는 것(probabilistic early recomputation)입니다. 저는 핫 키에는 락 방식을 쓰고, 나머지에는 TTL에 지터를 섞어 두고 있습니다.

이벤트 기반 무효화

TTL만으로는 감당이 안 되는 정합성이 필요한 데이터도 있었습니다. 그럴 때는 데이터가 바뀌는 시점에 캐시를 지웁니다. 쓰기 경로에서 관련 키를 DEL하거나, 도메인 이벤트를 구독한 무효화 워커가 처리하도록 넘깁니다. 여기서 신경 쓴 부분은 무효화 지점을 한곳에 모으는 일이었습니다. 쓰기 로직 여기저기에 redis.del이 흩어지면, 나중에 어디를 빠뜨렸는지 찾기가 어려워집니다.

*

캐시는 성능을 얻는 대신 정합성을 조금 내주는 거래입니다. 그래서 무효화 전략은 Redis 설정이 아니라, 데이터가 얼마나 낡아도 되는지를 도메인에 물어본 답에서 정해졌습니다. 돌아보면 기술을 고르기 전에 요구사항부터 확인했어야 하는 일이었습니다.
