이 사이트를 배포하며 부딪힌 다섯 개의 벽

2025년 7월• 약 8분• 디버깅 · 배포

지금 보고 계신 이 포트폴리오 자체가 하나의 프로젝트였습니다. 프레임워크 없이 짜고 GitHub Pages까지 올리는 동안, 예상 못 한 벽에 다섯 번 부딪혔습니다. 전부 커밋 이력에 흉터로 남아 있어서, 기억이 아니라 기록으로 씁니다. 증상 → 원인 → 해결 → 교훈 순으로.

먼저 짚고 가기
가변폰트Variable Font
굵기·기울기 같은 성질을 슬라이더처럼 무단계로 조절할 수 있는 한 개의 폰트 파일입니다. 예전엔 Bold·Italic을 각각 다른 파일로 받았지만, 가변폰트는 이 값들을 "축(axis)"으로 담아 하나로 묶습니다. 첫 번째 벽이 바로 이 축들을 다루다 생긴 일입니다.
가변폰트의 축axis
가변폰트가 조절 가능한 성질 하나하나를 축이라고 부릅니다. 무게(wght), 기울기(slnt)처럼 네 글자 이름을 갖습니다. font-variation-settings는 이 축들을 CSS에서 지정하는 속성인데, 여기 적지 않은 축은 기본값으로 되돌아갑니다.
베지어 곡선Bézier curve
몇 개의 점(끝점과 방향을 잡아 주는 조절점)만으로 매끈한 곡선을 정의하는 방식입니다. 폰트 외곽선부터 그래픽 편집기의 펜 도구까지 널리 쓰입니다. 두 번째 벽에서 다루기 까다로운 SVG arc 대신 이 곡선을 골랐습니다.
타입 검사기tsc, TypeScript
코드를 실행하기 전에 "이 변수에 엉뚱한 값이 들어갈 수 있는지"를 미리 훑어 주는 도구입니다. tsc --noEmit은 결과물을 만들지 않고 검사만 합니다. 세 번째 벽에서 실제로 터질 뻔한 버그를 배포 전에 잡아 줬습니다.
IntersectionObserver
어떤 요소가 화면(뷰포트)에 들어왔는지를 브라우저가 대신 감시해 알려 주는 기능입니다. 스크롤하면 나타나는 애니메이션을 만들 때 씁니다. 다섯 번째 벽은 이 감시 대상을 "언제 수집하는가"라는 타이밍에서 생겼습니다.

① 슬라이더를 움직였더니 다른 축이 리셋됐다

증상. 활자 실험실에서 무게(weight)를 바꿨더니, 방금 조절해 둔 기울기(slant)가 슬그머니 0으로 돌아갔습니다.

원인. font-variation-settings는 명시하지 않은 축을 전부 기본값으로 되돌립니다. "wght" 700만 넣으면 나머지 축은 "리셋해라"라는 뜻이 됩니다.

해결. 상태는 JS가 들고, 매 렌더마다 다섯 축을 통째로 직렬화해서 넘겼습니다.

el.style.fontVariationSettings =
  `"wght" ${w}, "slnt" ${s}, "CASL" ${c}, "MONO" ${m}, "CRSV" ${r}`;

교훈. 부분 갱신처럼 보이는 API가 사실은 전체 치환일 수 있다. 문서를 안 읽었으면 "브라우저 버그"로 오해했을 겁니다.

② SVG arc의 sweep 플래그와 두 번 싸웠다

증상. 정규식 철길에서 분기 레일을 SVG arc(a 명령)로 그렸더니, 곡선이 엉뚱한 방향으로 감겼습니다. 방향을 바꾸면 이번엔 반대쪽이 뒤집혔습니다.

원인. arc의 large-arc-flag와 sweep-flag는 네 방향 회전마다 조합이 달라지는데, 화면 좌표계(y가 아래로 증가)까지 겹쳐 머릿속 계산이 계속 어긋났습니다.

해결. arc를 버리고 끝점 두 개로 큐빅 베지어를 뽑았습니다. 수평 접선을 준 S자 곡선이라 방향 계산이 아예 필요 없어졌습니다.

const mx = (x1 + x2) / 2;
return `M${x1} ${y1} C${mx} ${y1} ${mx} ${y2} ${x2} ${y2}`;
도구가 자꾸 나를 이기면, 더 단순한 도구를 안 찾아본 것일 수도 있다.

③ 타입 검사가 막아준 런타임 버그

증상. Livepoll 클라이언트를 tsc로 검사하니 딱 하나가 걸렸습니다. 'a' is possibly 'undefined'.

원인. 선두 항목을 찾는 reduce의 초깃값으로 options[0]을 넣었는데, noUncheckedIndexedAccess를 켠 탓에 배열 인덱스 접근이 Option | undefined가 됩니다. 빈 배열이면 실제로 터질 수 있는 코드였습니다.

// before — 빈 배열이면 런타임 에러
opts.reduce((a, b) => b.votes > a.votes ? b : a, opts[0]);
// after — undefined에서 시작해 안전하게
opts.reduce<Option | undefined>((a, b) => !a || b.votes > a.votes ? b : a, undefined);

교훈. 엄격한 컴파일러 옵션은 잔소리가 아니라 배포 전에 켜둔 지뢰 탐지기였습니다. esbuild 빌드는 타입을 무시하고 통과시켰지만, tsc --noEmit은 이걸 잡아냈습니다. 그래서 빌드와 타입 검사를 분리해 둡니다.

④ 빌드한 결과물이 GitHub에 안 올라갔다

증상. 철길 프로젝트를 컴파일한 dist/를 커밋하려는데, git이 조용히 무시했습니다. 로컬에선 잘 도는데 배포하면 스크립트가 404.

원인. 표준 .gitignore에 습관적으로 넣어둔 dist/가 범인이었습니다. 보통은 빌드 산출물을 커밋하지 않는 게 맞지만, Pages는 그 산출물을 직접 서빙해야 합니다.

해결. 전역 규칙은 두되, 이 폴더만 부정 패턴으로 되살렸습니다.

dist/
# ...단, Rail은 컴파일된 TS를 Pages가 서빙하므로 예외
!projects/rail/dist/
!projects/rail/dist/**

교훈. "베스트 프랙티스"는 맥락 위에서만 참입니다. 정적 호스팅에선 빌드 산출물이 곧 배포물입니다.

⑤ 통계 타일이 화면에 안 나타났다

증상. 새벽의 커밋의 요약 숫자 타일들이 투명하게 사라진 채였습니다. 다른 카드는 멀쩡한데 이것들만.

원인. 스크롤 등장 애니메이션은 IntersectionObserver가 .reveal 요소를 초기화 시점에 한 번 수집합니다. 그런데 이 타일들은 fetch로 데이터를 받은 뒤에 JS로 생성돼서, 옵저버가 이미 훑고 지나간 뒤였습니다. 관찰되지 않으니 영영 opacity: 0.

해결. 나중에 만들어지는 요소에는 .reveal을 붙이지 않고 바로 보이게 했습니다. 근본적으론 동적 요소를 옵저버에 다시 등록하는 방법도 있지만, 이 경우엔 애니메이션을 포기하는 게 더 정직했습니다.

교훈. "선언적으로 한 번 수집"하는 패턴은 DOM이 시간에 따라 자라날 때 조용히 어긋납니다. 정적 마크업엔 완벽했던 코드가, 비동기 렌더 앞에서 반쪽이 됐습니다.

공통점

다섯 개 다 "환경이 내 예상과 다르게 동작한" 경우였습니다. 폰트 API의 리셋 규칙, 좌표계, 컴파일러의 엄격함, git의 무시 규칙, 옵저버의 수집 타이밍 — 전부 제가 가정했지만 확인하지 않은 것들이었죠. 버그는 대개 코드가 아니라 가정에 있었습니다. 그래서 요즘은 막히면 코드를 노려보기 전에 "내가 뭘 사실이라고 믿고 있지?"부터 적어봅니다.
