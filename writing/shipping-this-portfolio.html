Devlog · 트러블슈팅

이 사이트를 배포하며 부딪힌 다섯 개의 벽

2025년 7월• 디버깅 · 배포

지금 보고 계신 이 포트폴리오도 제가 만든 프로젝트 중 하나입니다. 프레임워크 없이 짜서 GitHub Pages에 올리는 동안 다섯 번 막혔는데, 그때마다 무엇 때문에 막혔는지를 얼마 지나면 잊어버리곤 해서 이번엔 적어 두기로 했습니다. 다행히 저장소의 커밋 이력이 남아 있어 당시 상황을 되짚을 수 있었습니다. 다섯 가지 모두 증상 → 원인 → 해결 → 교훈 순으로 정리했습니다.

먼저 짚고 가기
가변폰트Variable Font
OpenType 1.8(2016)에서 추가된 폰트 규격으로, 여러 디자인을 하나의 파일 안에 담습니다. fvar 테이블이 정의한 축들이 이루는 좌표 공간에서, 정해 둔 기준 디자인(마스터) 사이를 보간해 임의의 중간 굵기·기울기를 그려 냅니다. 다만 조절할 수 있는 범위는 디자이너가 마스터를 마련해 둔 축과 그 구간 안으로 한정되며, 파일 자체는 여전히 하나의 바이너리입니다.
가변폰트의 축axis
디자인 공간의 한 차원으로, 네 글자짜리 축 태그로 식별됩니다. wght·wdth·slnt·ital·opsz 다섯은 OpenType에 등록된 소문자 축이고, 그 밖의 커스텀 축은 대문자 태그(이 글의 CASL·MONO·CRSV)를 씁니다. 각 축은 fvar에 최소·기본·최대값이 정의되며, CSS font-variation-settings에 나열하지 않은 축은 직전 값이 아니라 이 기본값으로 되돌아갑니다 — ①번 문제의 원인이 바로 이 규칙이었습니다.
베지어 곡선Bézier curve
제어점들의 가중 합(번스타인 기저 다항식)으로 정의하는 매개변수 곡선입니다. SVG C 명령이 쓰는 3차 베지어는 제어점 네 개로 그리는데, 양 끝점은 곡선이 지나가지만 가운데 두 점은 지나가지 않은 채 곡선을 끌어당겨 끝점의 접선 방향을 정하고, 곡선은 항상 이 제어점들의 볼록 껍질 안에 머뭅니다. ②번에서는 방향 계산이 까다로운 arc 대신 이 곡선을 골라, 가운데 두 점의 x를 끝점 중앙에 두어 양 끝을 수평 접선으로 만들었습니다.
타입 검사기tsc, TypeScript
TypeScript 컴파일러 tsc가 수행하는 정적 타입 검사로, 코드를 실행하지 않고 컴파일 시점에 타입이 서로 일관되는지 확인하는 과정입니다. --noEmit은 JS를 내보내지 않고 검사만 하며, ③번에서는 noUncheckedIndexedAccess 옵션이 배열 인덱스 접근의 타입을 T | undefined로 넓혀 빈 배열 버그를 걸러 냈습니다. 다만 타입은 컴파일되면서 전부 지워져 런타임 보장을 주지 못하고, 타입을 보지 않는 esbuild 빌드는 이런 오류를 그대로 통과시킵니다.
IntersectionObserver
요소가 조상 요소나 뷰포트(root)와 겹치는 정도가 지정한 임계치(threshold)를 넘는 순간을 비동기로 알려 주는 브라우저 API입니다. observe(el)로 등록한 대상에 한해, 겹침이 바뀌면 isIntersecting 값을 담은 IntersectionObserverEntry로 콜백이 불리며, 스크롤 이벤트에 매이지 않고 레이아웃과 별개로 돕니다. 감시 대상은 observe()에 넘긴 요소로 한정되므로, ⑤번처럼 fetch 뒤에 생성된 요소는 다시 등록하기 전까지 관찰되지 않습니다.

① 슬라이더를 움직였더니 다른 축이 리셋됐다

증상. 활자 실험실에서 무게(weight)를 바꿀 때마다, 방금 조절해 둔 기울기(slant)가 0으로 돌아가 있었습니다.

원인. font-variation-settings는 명시하지 않은 축을 전부 기본값으로 되돌린다는 것을 나중에야 알았습니다. "wght" 700만 적으면 나머지 축은 기본값으로 되돌리라는 뜻이 됩니다.

해결. 축 값은 JS 쪽에서 들고 있게 하고, 렌더할 때마다 다섯 축을 통째로 직렬화해서 넘기도록 고쳤습니다.

el.style.fontVariationSettings =
  `"wght" ${w}, "slnt" ${s}, "CASL" ${c}, "MONO" ${m}, "CRSV" ${r}`;

교훈. 부분만 갱신하는 것처럼 보이는 API가 실제로는 전체를 갈아 끼우기도 한다는 걸 알게 됐습니다. 문서를 읽어보기 전까지는 브라우저 버그라고 생각하고 있었습니다.

② SVG arc의 sweep 플래그와 두 번 싸웠다

증상. 정규식 철길에서 분기 레일을 SVG arc(a 명령)로 그렸는데, 곡선이 의도한 것과 반대 방향으로 감겼습니다. 플래그를 바꾸면 이번에는 다른 쪽 분기가 뒤집혔습니다.

원인. arc의 large-arc-flag와 sweep-flag는 네 방향 회전마다 조합이 달라지는데, 화면 좌표계(y가 아래로 증가)까지 겹치다 보니 머릿속 계산이 계속 어긋났습니다.

해결. 두 번을 그렇게 헤매고 나서 arc를 그만 쓰기로 하고, 끝점 두 개로 큐빅 베지어를 뽑았습니다. 수평 접선을 준 S자 곡선이어서 방향을 따로 계산할 일이 없어졌습니다.

const mx = (x1 + x2) / 2;
return `M${x1} ${y1} C${mx} ${y1} ${mx} ${y2} ${x2} ${y2}`;
도구와 계속 씨름하고 있다면 더 단순한 방법을 아직 안 찾아본 것일 수도 있겠다고, 이때 생각했습니다.

③ 타입 검사가 막아준 런타임 버그

증상. Livepoll 클라이언트를 tsc로 검사했더니 하나가 걸렸습니다. 'a' is possibly 'undefined'라는 메시지였습니다.

원인. 선두 항목을 찾는 reduce의 초깃값으로 options[0]을 넣어 뒀는데, noUncheckedIndexedAccess를 켜 둔 탓에 배열 인덱스 접근이 Option | undefined가 됩니다. 빈 배열이 들어오면 실제로 터질 수 있는 코드였습니다.

// before — 빈 배열이면 런타임 에러
opts.reduce((a, b) => b.votes > a.votes ? b : a, opts[0]);
// after — undefined에서 시작해 안전하게
opts.reduce<Option | undefined>((a, b) => !a || b.votes > a.votes ? b : a, undefined);

교훈. 엄격한 컴파일러 옵션은 성가시다고만 생각했는데, 이번에는 배포 전에 문제를 걸러 줬습니다. esbuild 빌드는 타입을 보지 않고 그대로 통과시켰고, tsc --noEmit이 이걸 잡았습니다. 그래서 빌드와 타입 검사는 따로 돌리도록 해 뒀습니다.

④ 빌드한 결과물이 GitHub에 안 올라갔다

증상. 철길 프로젝트를 컴파일한 dist/를 커밋하려는데 git이 아무 말 없이 무시했습니다. 로컬에서는 잘 돌아가는데 배포하면 스크립트가 404였습니다.

원인. 표준 .gitignore에 습관적으로 넣어 둔 dist/ 때문이었습니다. 보통은 빌드 산출물을 커밋하지 않는 게 맞지만, Pages는 그 산출물을 직접 서빙해야 합니다.

해결. 전역 규칙은 그대로 두고, 이 폴더만 부정 패턴으로 되살렸습니다.

dist/
# ...단, Rail은 컴파일된 TS를 Pages가 서빙하므로 예외
!projects/rail/dist/
!projects/rail/dist/**

교훈. 관행처럼 따르던 설정도 환경이 달라지면 맞지 않을 수 있다는 걸 이때 알았습니다. 정적 호스팅에서는 빌드 산출물이 곧 배포물이었습니다.

⑤ 통계 타일이 화면에 안 나타났다

증상. 새벽의 커밋의 요약 숫자 타일들이 투명한 채로 나타나지 않았습니다. 같은 페이지의 다른 카드들은 잘 보이는 상태였습니다.

원인. 스크롤 등장 애니메이션은 IntersectionObserver가 .reveal 요소를 초기화 시점에 한 번 수집합니다. 그런데 이 타일들은 fetch로 데이터를 받은 뒤에 JS로 생성돼서, 옵저버가 이미 훑고 지나간 뒤에 만들어졌습니다. 관찰 대상이 아니니 opacity: 0인 상태로 남아 있었습니다.

해결. 나중에 만들어지는 요소에는 .reveal을 붙이지 않고 바로 보이게 했습니다. 동적 요소를 옵저버에 다시 등록하는 방법도 있지만, 이 경우엔 애니메이션을 빼는 쪽이 간단해서 그렇게 뒀습니다.

교훈. 초기화 때 한 번만 대상을 수집하는 방식은 DOM이 나중에 늘어나면 어긋납니다. 정적 마크업만 있을 때는 문제가 없던 코드였는데, 비동기로 그리는 부분에서 걸렸습니다.

공통점

다섯 개 모두 환경이 제 예상과 다르게 동작한 경우였습니다. 폰트 API의 리셋 규칙, 좌표계, 컴파일러의 엄격함, git의 무시 규칙, 옵저버의 수집 타이밍 — 전부 제가 가정은 했지만 확인은 하지 않은 것들이었습니다. 돌아보면 문제는 코드보다 가정 쪽에 있었던 셈입니다. 그래서 요즘은 막히면 코드를 들여다보기 전에 내가 무엇을 사실이라고 믿고 있는지부터 적어 봅니다.
