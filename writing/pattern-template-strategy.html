아키텍처 패턴 · Template Method와 Strategy — 코드 40%를 지운 조합

2021년 10월• Template Method · Strategy · Java

사례 글에는 결과만 적어 두었습니다. 정산 로직 5곳이 1곳이 되고 코드가 40% 줄었다는 이야기였는데, 그 사이에 코드가 어떻게 바뀌었는지는 남겨 두지 않았습니다. 당시 저는 Template Method와 Strategy를 이름만 알고 있었고, 둘 중 하나만 써서는 정리가 되지 않아 한참 헤맸습니다. 왜 하나로는 부족했는지부터 순서대로 적어 둡니다.

먼저 짚고 가기
리팩토링Refactoring
겉으로 관찰되는 동작을 바꾸지 않은 채 코드의 내부 구조만 고치는 변경을 가리킵니다(마틴 파울러의 정의). 같은 입력에 같은 결과가 나오는지를 테스트로 확인하며 작은 단계로 옮기는 것이 전제라, 기능 추가나 버그 수정처럼 결과가 달라지는 작업은 리팩토링과 분리해서 다룹니다. 이 글에서 다섯 사본을 골격 하나로 합친 부분은 리팩토링이지만, 사본마다 뒤집혀 있던 할인·절사 순서를 하나로 맞춘 부분은 금액 자체가 달라지므로 엄밀히는 동작 변경입니다.
Template Method템플릿 메서드 패턴
GoF의 행위(behavioral) 패턴으로, 상위 클래스의 메서드 하나가 알고리즘의 단계와 호출 순서를 확정하고 각 단계의 내용만 하위 클래스에 위임하는 구조입니다. 위임 창구는 두 종류인데, 반드시 달라야 하는 단계는 abstract 메서드로 두어 구현하지 않으면 컴파일이 막히게 하고, 대개 같지만 예외가 있는 단계는 기본 구현을 가진 훅(hook) 메서드로 엽니다. 제어 흐름은 끝까지 상위 클래스가 쥐고 있고 하위 클래스는 불려 가기만 합니다. 다만 골격 메서드에 final이 없으면 하위 클래스가 그것까지 재정의해 순서를 다시 정할 수 있어, 이 글의 calculate()는 final로 막아 두었습니다.
Strategy전략 패턴
역시 GoF의 행위 패턴으로, 교체 가능한 알고리즘을 공통 인터페이스(이 글의 DiscountPolicy.apply)로 추상화하고 그 구현 객체를 쓰는 쪽이 필드로 들고 있다가 위임하는 구조입니다. 어떤 구현을 넣을지는 객체를 만들거나 주입하는 시점에 정해지므로, 타입으로 고정되는 상속과 달리 실행 중에야 갈리는 변이도 담을 수 있습니다. 구현체는 자기가 언제 호출되는지 알지 못하고 알 필요도 없어서, 호출 순서는 전적으로 쓰는 쪽이 소유합니다. 조건 분기를 없애 주는 패턴은 아니고, 흩어져 있던 분기를 '어떤 객체를 고를 것인가' 한 곳으로 모을 뿐입니다.
상속과 조합Inheritance & Composition
상속(extends)은 하위 타입이 상위 타입의 인터페이스와 구현을 함께 물려받는 is-a 관계이고, Java에서 클래스 상속은 하나만 허용됩니다. 조합은 다른 객체를 필드로 보유해 필요한 기능만 위임하는 has-a 관계라, 여러 개를 동시에 품을 수도 실행 중에 갈아 끼울 수도 있습니다. 실무에서 갈리는 지점은 클래스 수의 증가 방식인데, 변이의 축이 둘일 때 두 축을 모두 상속으로 표현하면 수단 5개 × 정책 4개 = 20개가 필요하고 두 번째 축을 필드로 내리면 5 + 4 = 9개로 끝납니다. GoF가 "클래스 상속보다 객체 조합을 우선하라"고 적어 둔 이유이자, 이 글이 할인을 상속이 아니라 주입으로 넣은 이유입니다.
컴파일 타임과 런타임Compile time & Runtime
컴파일 타임은 소스가 바이트코드로 번역되는 시점(javac → .class)이라 이때 확정되는 것은 타입과 소스에 적힌 구조뿐이고, 런타임은 JVM이 그 바이트코드를 실행하며 실제 값과 객체가 정해지는 시점입니다. 어떤 결정이 어느 쪽인지는 "그 결정에 요청 파라미터나 DB 조회 같은 외부 입력이 필요한가"로 갈립니다. 키오스크가 올림이라는 규칙은 소스에 이미 적혀 있고, 이 회원이 정기권인지는 조회해 봐야 압니다. 주의할 점은 오버라이드된 메서드를 고르는 동적 디스패치 자체는 어느 경우든 런타임에 일어난다는 것이라, 여기서 나누는 기준은 실행 메커니즘이 아니라 변이의 내용이 소스에 미리 적혀 있는가이고, 이 글은 그 기준으로 확정된 변이는 타입(abstract 구현)으로, 요청마다 갈리는 변이는 값(주입되는 전략 객체)으로 표현합니다.

1. 복붙이 만든 다섯 개의 진실

주차 관제 시스템의 정산 로직을 맡았을 때 일입니다. 결제 경로가 웹, 키오스크, 카카오페이, 월정액, 관리자 수동 정산까지 다섯 개였는데, 정산 코드도 다섯 벌이 있었습니다. 처음부터 다섯 벌이었을 리는 없고, 새 경로가 생길 때마다 기존 코드를 복사해 붙인 다음 각자 자리에서 조금씩 고쳐졌을 겁니다. 제가 열어봤을 때는 이런 모양이었습니다.

// 채널별 서비스에 흩어져 있던 다섯 사본 — 비교를 위해 한 메서드로 모아 본 모습
public int calculate(Parking p, String channel, Member m) {
  long min = p.durationMinutes();
  int fare = 0;
  if ("WEB".equals(channel)) {
    fare = (int)(min / 30) * 1500;             // 30분 단위 — 내림
    if (m.isSubscriber()) fare = (int)(fare * 0.9); // ① 할인 먼저
    fare = fare / 100 * 100;                    // ② 100원 절사
  } else if ("KIOSK".equals(channel)) {
    fare = (int) Math.ceil(min / 30.0) * 1500;  // 여기만 올림!
    fare = fare / 100 * 100;                    // ① 절사 먼저
    if (m.isSubscriber()) fare = (int)(fare * 0.9); // ② 할인 나중 — 순서가 뒤집혔다
  } else if ("KAKAO".equals(channel)) {
    // … 또 미묘하게 다른 30여 줄 …
  }
  return Math.max(fare, 0);
}

같은 100분 주차인데 웹은 내림으로 계산한 4,500원에 할인과 절사를 거쳐 4,000원이 나왔고, 키오스크는 올림으로 계산한 데다 절사와 할인 순서까지 뒤집혀 5,400원이 나왔습니다. 그런데 사본을 하나씩 열어보면 그 안에서는 나름의 논리가 서 있어서, 어느 쪽이 틀렸다고 말하기가 어려웠습니다. 버그가 사본 하나의 안이 아니라 사본들 사이에 있다 보니 diff로도 리뷰로도 잘 잡히지 않았습니다.

처음엔 이 코드의 문제를 그냥 중복이라고 불렀는데, 정리하다 보니 그 말로는 절반밖에 설명되지 않았습니다. 복붙은 코드를 복사하는 게 아니라 결정권을 복사합니다. 다섯 사본은 각자 "할인과 절사 중 무엇이 먼저인가"라는 비즈니스 결정을 매번 다시 내리고 있었고, 다른 시기에 다른 사람이 고친 결정들이 서로 어긋나 있었습니다. 그래서 목표도 코드를 합치는 쪽이 아니라 결정을 한 곳으로 되돌리는 쪽으로 잡았습니다.

2. 변하는 것, 변하지 않는 것, 그리고 변하는 '시점'

패턴 이름을 먼저 고르지는 않았습니다. 다섯 사본을 나란히 놓고 계산을 단계로 분해하는 일부터 했는데, 돌아보면 설계의 절반은 여기서 끝나 있었습니다.

// 다섯 사본을 나란히 놓고 얻은 단계 분해
[1] 기본요금   주차 시간 → 금액       수단마다 다르다 (과금 단위 · 회차 유예 · 올림 규칙)
[2] 할인       정책에 따라 차감·감률  수단과 무관하다 (회원 · 경차 · 월정액 …)
[3] 절사       100원 미만 버림        전 수단 공통 — 반드시 할인 뒤
[4] 하한 보정  음수 금액 방지         전 수단 공통

이렇게 적어 놓고 보니 성질이 세 가지로 갈라졌고, 각각에 필요한 도구도 달랐습니다.

첫째, 단계의 순서는 바뀌지 않습니다. "절사는 할인 뒤"는 코딩 스타일이 아니라 요금 규정이었습니다. 순서가 불변이라면 순서를 결정하는 코드도 한 곳에만 있어야 하고, 나머지 코드에는 순서를 결정할 능력 자체가 없는 편이 안전합니다. Template Method가 하는 일을 저는 이렇게 이해했습니다. 제어 흐름의 소유권을 상위 클래스 한 곳으로 옮겨 두는 것입니다.

둘째, 기본요금은 결제 수단이 정해지면 같이 고정됩니다. 키오스크의 올림 규칙은 배포 시점에 이미 정해져 있습니다. 이렇게 컴파일 타임에 정해지는 변이는 상속(abstract 메서드)으로 표현해도 안전했습니다.

셋째, 할인은 요청이 와야 정해집니다. 이 차가 경차인지, 이 회원이 정기권자인지는 런타임에야 아는 정보입니다. 런타임에 갈리는 변이를 상속으로 표현하려면 조합마다 클래스를 만들어야 해서, 객체를 주입하는 Strategy 쪽이 맞았습니다. 당시엔 변이의 방향만 보고 패턴을 고르려 했는데, 변이가 확정되는 시점까지 봐야 도구가 정해진다는 걸 이때 알았습니다.

3. 골격 — 순서를 컴파일러에게 맡긴다 (Template Method)

공통 골격은 추상 클래스 하나로 옮겼습니다. 지금 다시 봐도 이 코드에서 제일 중요한 부분은 calculate()에 붙인 final입니다.

public abstract class Settlement {
  private final DiscountPolicy discount;
  protected Settlement(DiscountPolicy discount) { this.discount = discount; }

  /** 정산의 골격 — 계산 순서는 이 메서드만 결정한다 */
  public final Money calculate(Parking p) {
    Money base = baseFare(p.duration());        // [1] 수단별 차이 — abstract
    Money discounted = discount.apply(base, p); // [2] 정책별 차이 — 전략에 위임
    Money truncated = truncate(discounted);     // [3] 공통 — 훅(기본 구현)
    return Money.max(truncated, Money.ZERO);    // [4] 공통 — 고정
  }

  protected abstract Money baseFare(Duration d);  // 하위 클래스가 반드시 구현

  protected Money truncate(Money m) {             // 대부분 같지만 예외를 허용
    return m.floorTo(100);
  }
}

final을 붙인 이유를 적어 둡니다. 골격 메서드가 final이 아니면 하위 클래스가 calculate()를 오버라이드해 순서를 다시 정할 수 있습니다. 그러면 1절의 다섯 사본이 상속 계층 안에서 그대로 되살아나고, 복붙이 상속으로 이름만 바꾼 셈이 됩니다. final을 붙여 두면 "순서는 한 곳"이라는 규칙을 코드 리뷰가 아니라 컴파일러가 지켜줍니다. 사람의 주의력에 기대던 불변식을 타입 시스템의 제약으로 옮겨 둔 셈인데, 이 리팩토링에서 가장 든든했던 부분이었습니다.

하위 클래스에 열어주는 문은 두 종류로 나눴습니다. 수단마다 반드시 달라야 하는 것은 abstract로 뒀습니다. 구현하지 않으면 컴파일이 되지 않으니 빠뜨릴 수가 없습니다. 대부분 같지만 예외가 있는 것은 기본 구현이 있는 훅(hook)으로 열었습니다. 관리자 수동 정산만 절사 규칙이 달라서, 그 한 곳에서만 truncate()를 재정의했습니다.

*

골격은 final, 훅은 최소로. 훅을 하나 열 때마다 하위 클래스는 골격의 일부를 다시 결정할 권한을 얻습니다. 언젠가 필요할지 모른다는 이유로 미리 열어둔 훅이 쌓이면 Template Method도 사본 다섯 벌과 비슷한 상태로 돌아갑니다. 저는 지금 실제로 다른 것만 훅으로 열어 두기로 했습니다. 필요해지면 그때 여는 편이 쌉니다.

4. 교체 지점 — 할인은 상속이 아니라 주입 (Strategy)

처음 스케치에서는 할인도 abstract 메서드로 만들려고 했습니다. 그런데 클래스 수를 세어 보다가 접었습니다. 수단이 5개, 할인 정책이 회원·경차·월정액·무할인으로 4개면 상속만으로는 WebSubscriberSettlement, KioskLightCarSettlement처럼 5 × 4 = 20개의 클래스가 필요했습니다. 상속 계층은 변이의 축을 하나밖에 표현하지 못하는데, 이 문제는 축이 둘(수단, 정책)이고 서로 직교했기 때문입니다.

두 번째 축을 클래스 계층에서 꺼내 필드로 옮기면 곱셈이 덧셈이 됩니다. 수단 5개에 정책 4개를 더해 9개 클래스로 끝납니다. Strategy를 쓴 이유는 이것뿐이었습니다. 패턴이 좋아서가 아니라, 직교하는 두 번째 축은 조합(composition)으로 담아야 클래스 수가 선형으로 늘기 때문입니다.

public interface DiscountPolicy {
  Money apply(Money base, Parking p);

  DiscountPolicy NONE = (base, p) -> base;   // null 대신 무할인 전략
}

public class SubscriberDiscount implements DiscountPolicy {
  public Money apply(Money base, Parking p) {
    return base.rate(0.9);                   // 정기권 10% 감률
  }
}

public class LightCarDiscount implements DiscountPolicy {
  public Money apply(Money base, Parking p) {
    return base.rate(0.5);                   // 경차 50% — 조례 기준
  }
}

결정 시점을 따져 봐도 같은 결론이었습니다. 어떤 Settlement 클래스를 쓸지는 요청의 채널만 보면 정해지지만, 어떤 할인이 적용될지는 회원 조회가 끝나야 알 수 있었습니다. 컴파일 타임 결정은 타입으로, 런타임 결정은 값(주입되는 객체)으로 두는 식으로 결정 시점과 표현 수단을 맞추자 코드가 제자리를 찾았습니다. 테스트도 같이 편해졌습니다. DiscountPolicy는 값 객체(Money, Parking)만 주고받는 순수한 인터페이스라, DB도 채널도 없이 정책 하나만 단위 테스트할 수 있었습니다.

5. 결합 지점 — 골격이 '언제'를, 전략이 '무엇을'

두 패턴이 만나는 지점은 골격의 [2]번 단계인 discount.apply(base, p) 한 줄입니다. Template Method가 "할인은 기본요금 뒤, 절사 앞에 적용된다"는 언제를 소유하고, Strategy가 "10%인지 50%인지"라는 무엇을 소유합니다. 1절의 순서 역전 버그가 다시 나오지 않는 이유도 이 분업에 있습니다. 전략 구현체는 자기가 언제 불리는지 알 수 없고, 알 필요도 없습니다.

수단별 구현체는 이만큼 얇아졌습니다. 80줄짜리 사본 다섯 벌이 골격 하나와 10줄 남짓한 구현체 다섯 개가 되었습니다.

public class KioskSettlement extends Settlement {
  public KioskSettlement(DiscountPolicy discount) { super(discount); }

  @Override protected Money baseFare(Duration d) {
    long units = ceilDiv(d.toMinutes(), 30);   // 키오스크: 30분 단위 올림
    return UNIT_FARE.times(units);             // 이 클래스가 아는 전부
  }
}

// 조립 지점 — if-else가 "이사 간" 곳
public Settlement settlementFor(Channel ch, Member m) {
  DiscountPolicy policy = discountFor(m);      // 런타임 결정은 여기서 한 번
  return switch (ch) {
    case WEB   -> new WebSettlement(policy);
    case KIOSK -> new KioskSettlement(policy);
    case KAKAO -> new KakaoSettlement(policy);
    case MONTHLY, MANUAL -> /* … */
  };
}
Money fee = settlementFor(channel, member).calculate(parking);

한 가지는 짚어 두는 게 좋겠습니다. if-else는 사라지지 않고 자리를 옮겼을 뿐입니다. 분기는 조립 지점(settlementFor) 한 곳에 그대로 남아 있습니다. 달라진 점은 그 분기 안에서 금액을 계산하지 않는다는 것입니다. 고르는 대상이 숫자가 아니라 객체라서, 이 분기가 틀리면 요금이 미묘하게 다른 버그가 아니라 채널이 뒤바뀐 버그가 나고, 그건 테스트 한 줄로 잡힙니다. 애초에 목표도 분기를 없애는 것이 아니라 분기와 계산을 다른 방에 두는 것이었습니다.

새 결제 수단을 추가하는 작업도 성격이 달라졌습니다. 전에는 기존 사본 하나를 통째로 복사해 여섯 번째 사본을 만들고 조금씩 고치는 일이었고, 1절의 다섯 사본도 그렇게 생겨난 것들이었습니다. 이제는 baseFare() 하나를 가진 새 클래스를 만들고 조립 지점에 한 줄 등록하면 되고, 기존 코드는 열어보지 않아도 됩니다. 사례 글에 적은 "신규 수단 추가 시간 절반"과 "코드 40% 감소"는 이 구조 변화에서 나온 숫자입니다.

6. 한계 — 전략 폭발, 그리고 패턴이 과한 순간

여기까지는 잘 풀린 이야기고, 이 조합이 버거워지는 자리에도 두 번 부딪혔습니다. 그쪽도 같이 적어 둡니다.

첫 번째는 할인이 겹치기 시작했을 때입니다. 경차이면서 정기권 회원인 경우를 어떻게 할지 정해야 했는데, 손 가는 대로 만들면 SubscriberAndLightCarDiscount 같은 클래스가 생깁니다. 조합마다 클래스를 만들면 4개였던 정책이 2ⁿ으로 불어나는 전략 폭발이 일어납니다. Composite로 묶어 클래스 수는 잡았는데, 그다음에 더 미묘한 문제가 남았습니다.

public class CompositeDiscount implements DiscountPolicy {
  private final List<DiscountPolicy> policies;  // 적용 순서 = 리스트 순서

  public Money apply(Money base, Parking p) {
    Money result = base;
    for (DiscountPolicy policy : policies)
      result = policy.apply(result, p);
    return result;
  }
}
// 정액 2,000원 차감과 정률 50% 할인 — 순서에 따라 금액이 달라진다.
// 조합 순서를 호출부마다 정하게 두면, 1절의 순서 버그가 전략 층에서 부활한다.

정액 차감과 정률 할인은 교환법칙이 성립하지 않습니다. 조합 순서가 다시 비즈니스 규칙이 된 것입니다. 호출부마다 리스트를 임의로 조립하게 두면, 애써 없앤 "사본마다 다른 순서"가 층만 바꿔 재발합니다. 그래서 조합을 만드는 팩토리를 한 곳에 두고 정렬 규칙(정률 먼저, 정액 나중)을 그 안에 고정해 막았습니다. 적고 보니 이것도 전략 조합에 대한 또 하나의 작은 Template Method였습니다. 같은 문제가 층을 바꿔 다시 나오고, 같은 방법이 다시 들었습니다.

!

전략 폭발의 조기 신호는 이름에서 먼저 보였습니다. 전략 구현체 이름에 And가 들어가기 시작하면(SubscriberAndLightCar…) 조합이 클래스 수를 곱셈으로 늘리고 있다는 뜻입니다. 그 시점에 Composite로 전환하면서 조합 순서를 결정하는 코드는 한 곳에 모아 두는 편이 좋습니다. 순서가 흩어지면 처음의 불일치 버그가 전략 레벨에서 되살아납니다.

두 번째는 이 조합 자체가 과한 상황이 있다는 것입니다. 결제 수단이 2개뿐이고 할인도 하나뿐이었다면, 추상 클래스 하나와 인터페이스 하나, 구현체 서너 개로 늘어난 파일들이 if 두 줄보다 비쌌을 겁니다. 패턴은 간접 참조를 만들고, 간접 참조는 코드를 처음 읽는 사람에게 점프 비용을 물립니다. 분기가 있으니 패턴을 쓴다는 규칙은 아니고, 변이의 축이 늘어나거나(수단 × 정책) 같은 결정이 여러 곳에서 반복될 때 패턴이 비용을 회수한다고 정리해 두었습니다. 저도 다섯 사본과 직교하는 두 축이라는 사정이 없었다면 이 구조를 만들지 않았을 겁니다.

Template Method 쪽에 원래 있는 한계도 적어 둡니다. 상속 기반이라 하위 클래스가 단일 상속을 소모하고, 골격과 구현체가 protected로 은근히 결합하기 쉽습니다. 지금 다시 짠다면 baseFare까지 Function<Duration, Money>로 주입해 골격을 상속 없는 일반 클래스로 두는 선택지도 괜찮다고 생각합니다. 그 경우 Template Method는 상속 패턴이 아니라 순서를 소유한 하나의 final 함수로 남습니다. 겉모습은 달라져도 1절에서 세운 원칙, 그러니까 순서의 결정권은 한 곳이라는 점은 그대로입니다. 이후 NestJS 레거시 UseCase를 정리할 때는 이 함수형 변형을 썼습니다.

7. 정리 — 두 패턴을 조합하는 규칙

이 리팩토링에서 남은 규칙을 잊지 않으려고 정리해 둡니다.

  • 복붙의 비용은 결정권이 흩어지는 것이었습니다. 사본마다 계산 순서를 다시 정하게 되고, 버그는 사본 안이 아니라 사본 사이에 생깁니다.
  • 패턴보다 분해가 먼저였습니다. 단계를 나눠 무엇이 변하고 무엇이 불변인지, 그리고 변이가 컴파일 타임에 확정되는지 런타임에 확정되는지까지 봅니다.
  • 순서가 불변이면 Template Method를 씁니다. 골격 메서드는 final로 둬서 순서 불변식을 리뷰가 아니라 컴파일러가 지키게 하고, 훅은 실제로 지금 다른 것만 엽니다.
  • 변이의 축이 둘이고 직교하면 두 번째 축은 Strategy로 뺍니다. 상속은 축 하나만 담고, 필드로 꺼내면 M × N이 M + N이 됩니다.
  • 결합 지점은 한 줄로 둡니다. 골격이 '언제'를, 전략이 '무엇'을 소유하고, 전략은 자기가 언제 불리는지 몰라도 됩니다.
  • if-else는 없애는 게 아니라 옮기는 것이었습니다. 분기는 조립 지점 한 곳에 두고, 분기와 계산을 같은 방에 두지 않습니다.
  • 전략이 조합되면 조합 순서도 다시 한 곳에 고정해 둡니다. 같은 문제가 층을 바꿔 재발합니다.
  • 축이 하나뿐이면 그냥 둡니다. 분기 두어 개짜리 코드에 이 조합을 들이미는 건 문제보다 비싼 해법입니다.

골격과 교체 지점을 나누는 이 감각은 이후 크롤링 파이프라인에 Adapter를 붙일 때도, 레거시 UseCase를 정리할 때도 그대로 다시 썼습니다. 패턴 이름은 매번 달랐지만 스스로에게 묻는 것은 늘 같았습니다. 무엇이 변하고, 언제 변하는가 하는 것이었습니다.
