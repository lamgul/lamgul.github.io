정규식을 손으로 파싱하기

2025년 4월• TypeScript · 재귀 하강

정규식은 "쓰기 전용 언어"라는 농담이 있습니다. 짤 땐 알겠는데, 두 달 뒤엔 내가 짠 것도 못 읽죠. 못 읽는다는 건 결국 머릿속에서 파싱이 안 된다는 뜻입니다. 그럼 그 파싱을 코드로 옮기면, 기계가 대신 읽어주지 않을까 — Rail은 그 생각에서 시작했습니다.

먼저 짚고 가기
파싱parsing
글자의 나열을 "의미가 있는 구조"로 해석하는 일입니다. 사람이 문장을 읽고 주어·동사로 나누듯, a(b|c)* 같은 문자열을 기계가 이해할 수 있는 형태로 분해합니다. 이 글은 그 분해 과정을 직접 손으로 짜는 이야기입니다.
파서parser
파싱을 실제로 수행하는 프로그램입니다. 입력 문자열을 받아 뒤에 나오는 트리 구조로 바꿔 주는 기계라고 보면 됩니다.
재귀 하강 파서recursive descent parser
문법 규칙 하나를 함수 하나로 만들고, 더 안쪽 규칙은 다른 함수를 부르며 파고 내려가는 방식의 파서입니다. 큰 상자를 열면 그 안에 더 작은 상자가 있고, 다시 그 안에… 하는 식으로 위에서 아래로(하강) 자기 자신 같은 규칙을 반복해서(재귀) 처리한다고 해서 붙은 이름입니다.
우선순위precedence
어떤 연산을 먼저 묶어서 계산하는지의 순서입니다. 수학의 곱셈이 덧셈보다 먼저이듯, 정규식에서도 |(선택)보다 수량자 *가 더 단단히 붙습니다. 이 글의 핵심은 이 우선순위 순서가 그대로 함수 호출 순서가 된다는 점입니다.
토크나이저tokenizer
문자열을 의미 단위(토큰)로 미리 잘라 두는 앞 단계입니다. 문장을 단어별로 띄어 놓는 작업에 가깝습니다. 이 글에서는 정규식이 대부분 한 글자가 곧 한 토큰이라 이 단계를 통째로 생략합니다.
ASTabstract syntax tree
파싱의 결과물인 나무 모양 구조로, 우리말로는 추상 구문 트리라고 합니다. 문법의 중첩 관계를 가지가 뻗은 나무처럼 담아 두어, 이후 코드가 다루기 쉽게 만든 형태입니다.

파서 생성기를 쓰지 않은 이유

PEG.js나 ANTLR 같은 도구를 쓰면 문법 파일 하나로 파서가 뚝 떨어집니다. 하지만 저는 파서가 어떻게 동작하는지 손끝으로 알고 싶었습니다. 그리고 정규식 정도의 문법은, 도구를 배우는 시간보다 직접 짜는 게 빠릅니다. 의존성은 결국 TypeScript 하나뿐입니다.

우선순위가 곧 함수다

재귀 하강 파싱의 핵심은 단순합니다. 문법의 우선순위를 함수 호출의 깊이로 옮기는 것. 정규식에서 가장 느슨하게 묶이는 건 선택(|)이고, 그 안에 연속이, 그 안에 수량자가, 가장 안쪽에 원자(문자·그룹)가 있습니다. 이 순서를 그대로 함수로 적습니다.

parseAlternation() {          // 가장 바깥 = 가장 느슨함
  const opts = [this.parseSequence()];
  while (this.peek() === "|") {
    this.next();
    opts.push(this.parseSequence());   // 한 단계 더 조인 것에게 위임
  }
  return opts.length === 1 ? opts[0] : { t: "alt", opts };
}

각 함수는 자기 층만 처리하고, 더 단단히 묶이는 건 한 단계 아래 함수에게 위임합니다. parseSequence는 parseQuantified를, 그건 다시 parseAtom을 부릅니다. 문법 규칙 하나에 함수 하나 — 트리를 세우는 코드가 문법 정의서처럼 읽힙니다. 이 대응 관계가 재귀 하강의 가장 아름다운 지점입니다.

토크나이저는 없다

보통 파싱은 "토큰으로 자르기 → 트리로 세우기" 두 단계지만, 정규식은 대부분 한 글자가 한 토큰입니다. 그래서 문자열 위를 커서 하나로 훑는 걸로 충분했습니다. peek()로 엿보고 next()로 삼킵니다. 단계를 하나 없애니 코드가 눈에 띄게 짧아졌습니다.

에러는 위치를 안다

잘못된 입력을 만나면 그냥 던지지 않고, 어디서 틀렸는지를 함께 던집니다.

if (this.peek() !== ")")
  throw new RegexError("')' 를 기대했습니다", this.pos);

이 pos 하나 덕분에 UI는 입력창 바로 아래에 ▲를 정확한 칸에 찍어줍니다. 사용자에게 "어딘가 틀렸어요"가 아니라 "여기가 틀렸어요"라고 말할 수 있는 건, 파서가 위치를 버리지 않고 끝까지 들고 온 덕분입니다.

덤: 트리를 그리다 배운 것

AST를 얻은 뒤엔 철도 다이어그램으로 눕혀야 했는데, 거기서 예상 못 한 교훈이 하나 나왔습니다. 분기 레일을 SVG의 arc로 그리다 sweep 플래그를 방향마다 맞추느라 두 번 틀렸습니다. 결국 끝점 두 개로 큐빅 베지어를 뽑는 방식으로 바꿨더니 코드도 짧아지고 곡선도 부드러워졌습니다.

도구가 복잡하게 느껴진다면, 더 단순한 도구를 안 찾아본 것일 수도 있다.

수량자를 그리는 것도 비슷했습니다. *(0회 이상)를 위한 특별한 도형을 따로 만들 뻔했지만, 그건 그냥 optional(oneOrMore(x)) — "생략 가능"과 "1회 이상"의 합성이었습니다. 조각이 자기를 몇 개의 숫자로만 설명하도록 잘 설계하면, 복잡한 건 합성으로 공짜로 나옵니다.
