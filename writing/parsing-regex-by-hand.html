정규식을 손으로 파싱하기

2025년 4월• TypeScript · 재귀 하강

두 달 전에 제가 짠 정규식을 다시 읽지 못한 적이 있습니다. 정규식을 두고 "쓰기 전용 언어"라는 농담을 하는 이유를 그때 알았습니다. 읽지 못한다는 건 머릿속에서 파싱이 되지 않는다는 뜻이니, 그 파싱을 코드로 옮겨 두면 기계가 대신 읽어 줄 수 있겠다고 생각했습니다. Rail은 거기서 시작했고, 이 글은 라이브러리 없이 재귀 하강 파서를 짜면서 정리해 둔 기록입니다.

먼저 짚고 가기
파싱parsing
형식 문법이 정한 규칙에 따라 기호의 나열을 읽어, 그 나열이 문법으로 유도되는지 판정하고 유도 과정을 트리 구조로 복원하는 작업입니다. a(b|c)*는 글자 일곱 개지만, 파싱을 거치면 "a 다음에 (b 또는 c)의 0회 이상 반복"이라는 중첩 관계가 드러납니다. 입력이 올바른지만 확인하는 인식(recognition)과 달리 파싱은 구조까지 만들어 내며, 그 구조로 무엇을 할지는 파싱 다음 단계의 몫입니다.
파서parser
문법을 코드로 옮겨 파싱을 수행하는 프로그램입니다. 입력 문자열과 현재 읽는 위치를 상태로 들고, 성공하면 AST를, 문법에 맞지 않으면 실패한 위치와 함께 오류를 돌려줍니다. 이 글의 파서는 pos 하나만 들고 왼쪽에서 오른쪽으로 나아가며 peek()로 다음 글자를 보고 next()로 위치를 옮깁니다. 파서는 구조를 세울 뿐 그 정규식을 실행하지는 않습니다. 문자열 매칭은 별개의 단계입니다.
재귀 하강 파서recursive descent parser
하향식(top-down) 파싱 기법의 하나로, 문법의 비단말 기호 하나에 함수 하나를 대응시키고 규칙 우변에 등장하는 비단말을 그대로 함수 호출로 옮겨 적습니다. 문법이 재귀적이면 호출도 재귀가 되고, 실행 중의 호출 스택이 그 순간 세워지고 있는 트리의 경로와 일치합니다. 다음 글자 하나만 미리 보고 어느 규칙으로 갈지 정할 수 있는 문법이면 되돌아가기 없이 동작하지만, A → A x 같은 좌재귀 규칙은 그대로 옮기면 자기 자신을 무한히 부르므로 반복문으로 펴 주어야 합니다. 아래 parseAlternation의 while이 그 변환의 결과입니다.
우선순위precedence
한 식에 여러 연산자가 섞였을 때 어느 연산자가 피연산자를 먼저 가져가는지를 정해 두는 문법상의 규칙입니다. 정규식은 선택(|)이 가장 느슨하고 연속, 수량자(* + ?) 순으로 결합이 강해져서 ab*|c는 (a(b*))|c로 묶입니다. 재귀 하강에서는 이 순서가 함수의 중첩 깊이로 그대로 나타나, 느슨한 규칙이 바깥 함수가 되고 단단한 규칙이 안쪽 함수가 됩니다. 같은 우선순위끼리 왼쪽부터 묶을지 오른쪽부터 묶을지를 정하는 결합 방향은 우선순위와 별개로 정해야 합니다.
토크나이저tokenizer
입력 문자열을 어휘 규칙에 따라 토큰, 즉 "종류 + 값 + 위치" 단위로 잘라 파서에게 넘기는 앞 단계입니다. 식별자나 숫자 리터럴, >=처럼 여러 글자가 모여 하나의 의미 단위가 되는 언어에서 파서가 글자 단위 판단을 떠맡지 않도록 덜어 주는 역할을 합니다. 정규식은 대부분 한 글자가 곧 한 토큰이라 이 글에서는 이 단계를 두지 않고 파서가 문자열을 직접 읽습니다. 대신 \d나 {2,5}처럼 여러 글자가 한 단위인 경우는 파서 안에서 직접 처리해야 합니다.
ASTabstract syntax tree
파싱 결과를 담는 트리 자료구조로, 노드 하나가 구문 구조 하나(연산자나 리터럴)를 가리키고 자식이 그 피연산자가 됩니다. 문법을 그대로 옮긴 파스 트리와 달리 괄호나 구분자처럼 구조를 표시하려고 있었을 뿐인 기호는 남기지 않고 중첩 관계 자체로 흡수하는데, "추상"이라는 이름이 붙은 이유가 그것입니다. 아래 코드가 돌려주는 { t: "alt", opts }가 그 형태이고, 선택지가 하나뿐이면 alt 노드조차 만들지 않고 자식을 그대로 올려보냅니다.

파서 생성기를 쓰지 않은 이유

PEG.js나 ANTLR 같은 도구를 쓰면 문법 파일 하나로 파서가 만들어집니다. 처음엔 저도 그쪽을 보다가, 파서가 어떻게 동작하는지 모른 채 결과만 받아 쓰게 될 것 같아 직접 짜기로 했습니다. 정규식 정도의 문법이라면 도구 사용법을 익히는 시간이나 직접 짜는 시간이나 비슷하겠다는 계산도 있었습니다. 그렇게 해서 의존성은 TypeScript 하나로 끝났습니다.

우선순위가 곧 함수다

재귀 하강 파싱은 문법의 우선순위를 함수 호출의 깊이로 옮기는 방식입니다. 정규식에서 가장 느슨하게 묶이는 건 선택(|)이고, 그 안에 연속이, 그 안에 수량자가, 가장 안쪽에 원자(문자·그룹)가 있습니다. 저는 이 순서를 그대로 함수 이름으로 적어 내려갔습니다.

parseAlternation() {          // 가장 바깥 = 가장 느슨함
  const opts = [this.parseSequence()];
  while (this.peek() === "|") {
    this.next();
    opts.push(this.parseSequence());   // 한 단계 더 조인 것에게 위임
  }
  return opts.length === 1 ? opts[0] : { t: "alt", opts };
}

각 함수는 자기 층만 처리하고, 더 단단히 묶이는 건 한 단계 아래 함수에게 위임합니다. parseSequence는 parseQuantified를, 그건 다시 parseAtom을 부릅니다. 문법 규칙 하나에 함수 하나가 대응하는 구조라, 다 짜고 나서 읽어 보니 트리를 세우는 코드가 문법 정의서와 같은 순서로 늘어서 있었습니다. 이 대응 관계가 마음에 들어서, 이후에 파서를 짤 일이 있으면 이 방식을 먼저 떠올립니다.

토크나이저는 없다

보통 파싱은 "토큰으로 자르기 → 트리로 세우기" 두 단계로 나누는데, 정규식은 대부분 한 글자가 한 토큰이라 앞 단계를 따로 두지 않았습니다. 문자열 위를 커서 하나로 훑으면서 peek()로 다음 글자를 확인하고 next()로 한 칸 넘기는 정도로 충분했습니다. 단계 하나를 덜어 낸 만큼 코드도 짧아졌습니다.

에러는 위치를 안다

잘못된 입력을 만났을 때 예외만 던지면 사용자가 어디를 고쳐야 할지 알 수 없어서, 틀린 위치를 함께 담아 던지도록 했습니다.

if (this.peek() !== ")")
  throw new RegexError("')' 를 기대했습니다", this.pos);

이 pos 값을 UI로 넘기면 입력창 바로 아래 해당하는 칸에 ▲를 찍을 수 있습니다. "어딘가 틀렸어요" 대신 "여기가 틀렸어요"라고 말해 주려면 파서가 위치를 버리지 않고 끝까지 들고 와야 한다는 걸, 이 부분을 고치면서 알았습니다.

덤: 트리를 그리다 배운 것

AST를 얻은 뒤에는 이걸 철도 다이어그램으로 눕히는 작업이 남아 있었습니다. 분기 레일을 SVG의 arc로 그렸는데 sweep 플래그를 방향마다 맞추는 걸 두 번 틀렸습니다. 그러다 끝점 두 개로 큐빅 베지어를 뽑는 방식으로 바꿨더니 코드도 짧아지고 곡선도 부드러워졌습니다.

도구가 복잡하게 느껴진다면, 더 단순한 도구를 안 찾아본 것일 수도 있다.

수량자를 그릴 때도 비슷한 일이 있었습니다. *(0회 이상)를 위한 도형을 따로 만들려다가, 그게 optional(oneOrMore(x)), 그러니까 "생략 가능"과 "1회 이상"의 합성이라는 걸 뒤늦게 알아차렸습니다. 조각이 자기를 몇 개의 숫자로만 설명하도록 설계해 두면 복잡한 모양은 합성으로 따라온다는 걸, 도형을 하나 덜 만들고 나서 이해했습니다.
