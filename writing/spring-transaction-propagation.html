Spring 트랜잭션 전파 — REQUIRES_NEW는 언제 롤백되는가

2022년 2월• 약 10분• Spring · Transaction · JPA

다섯 곳에 흩어진 정산 로직을 한 곳으로 모으고 나니 다음 질문이 따라왔습니다. 정산 저장과 이력 저장은 같이 롤백되어야 하는가? 답은 @Transactional의 전파(propagation) 옵션이 정하는데, 문서만 읽어서는 커밋·롤백 경계가 눈에 안 보입니다. REQUIRED · REQUIRES_NEW · NESTED 세 옵션을 실험 코드와 로그로 확인합니다.

먼저 짚고 가기
트랜잭션Transaction
여러 DB 작업을 하나로 묶어 전부 성공하거나 전부 취소하는 단위입니다. 계좌 이체에서 "출금과 입금은 함께 이뤄지거나 둘 다 없던 일이 돼야 한다"는 규칙이 트랜잭션이에요. 이 글은 그 "묶음의 경계"를 어디에 그을지를 다룹니다.
커밋 · 롤백Commit · Rollback
커밋은 지금까지의 변경을 확정 저장하는 것, 롤백은 없던 일로 되돌리는 것입니다. 트랜잭션은 반드시 이 둘 중 하나로 끝납니다. 이 글의 핵심 질문 "정산과 이력이 같이 롤백되는가"가 바로 이 경계 이야기입니다.
전파Propagation
트랜잭션이 걸린 메서드가 또 다른 트랜잭션 메서드를 호출할 때, 두 번째가 기존 트랜잭션에 참여할지 · 새로 열지 · 중첩할지를 정하는 규칙입니다. REQUIRED / REQUIRES_NEW / NESTED가 바로 이 선택지고, 롤백 경계가 여기서 갈립니다.
프록시 기반 AOPProxy-based AOP
스프링은 원본 객체를 얇게 감싼 대리인(프록시)을 만들어, 트랜잭션을 열고 닫는 코드를 그 대리인이 대신 실행합니다. 그래서 같은 클래스 안에서 this.메서드()로 직접 부르면 대리인을 건너뛰어 트랜잭션 설정이 무시됩니다(6장의 함정).
세이브포인트Savepoint
하나의 트랜잭션 안에 찍어 두는 중간 저장 지점입니다. 문제가 생기면 처음까지 다 되돌리지 않고 이 지점까지만 부분 롤백할 수 있어요. NESTED 옵션이 이 기능을 쓰는데, JPA에서는 지원되지 않는다는 게 5장의 벽입니다.

1. 전파는 "경계를 어떻게 이어붙일 것인가"의 문제

@Transactional 메서드가 또 다른 @Transactional 메서드를 호출할 때, 두 번째 메서드가 첫 번째의 트랜잭션에 참여할지, 새로 만들지, 중첩할지를 정하는 규칙이 전파 옵션입니다. 이때 두 개념을 구분해야 사고가 정리됩니다.

논리 트랜잭션은 @Transactional 스코프 하나하나이고, 물리 트랜잭션은 실제 DB 커넥션 위에서 도는 BEGIN … COMMIT입니다. REQUIRED는 여러 논리 트랜잭션을 하나의 물리 트랜잭션에 태우고, REQUIRES_NEW는 물리 트랜잭션을 하나 더 만듭니다. 롤백 관련 사고는 대부분 이 둘을 혼동할 때 생깁니다.

2. 실험 준비 — 경계를 로그로 보이게 만든다

트랜잭션 매니저의 DEBUG 로그를 켜면 트랜잭션이 언제 만들어지고, 참여하고, 정지(suspend)되는지 전부 찍힙니다. 이 글의 모든 로그는 이 설정으로 뽑았습니다.

# application.yml — 트랜잭션 경계를 로그로 확인
logging:
  level:
    org.springframework.orm.jpa.JpaTransactionManager: DEBUG
    org.springframework.transaction.interceptor: TRACE

실험 시나리오는 정산 시스템에서 그대로 가져왔습니다. SettlementService.settle()이 정산을 저장하고, HistoryService.record()가 정산 이력을 남깁니다. 이력 저장이 실패했을 때 정산까지 함께 롤백되는지가 관전 포인트입니다.

@Service
public class SettlementService {
  private final SettlementRepository repository;
  private final HistoryService historyService;

  @Transactional  // propagation 기본값 = REQUIRED
  public void settle(Order order) {
    repository.save(Settlement.of(order));
    try {
      historyService.record(order);  // 이력 저장 — 실패할 수 있다
    } catch (RuntimeException e) {
      log.warn("이력 저장 실패 — 정산은 살리고 싶다", e);
    }
  }
}

3. REQUIRED — 함께 살고, 함께 죽는다

record()가 REQUIRED(기본값)이고 내부에서 RuntimeException이 터지면, 바깥에서 try-catch로 잡았으니 정산은 커밋될 것 같습니다. 실제로는 정산까지 전부 롤백되고, 커밋을 시도한 바깥이 UnexpectedRollbackException을 받습니다.

JpaTransactionManager : Creating new transaction with name
  [..SettlementService.settle]: PROPAGATION_REQUIRED,ISOLATION_DEFAULT
JpaTransactionManager : Participating in existing transaction
JpaTransactionManager : Participating transaction failed
  - marking existing transaction as rollback-only
JpaTransactionManager : Global transaction is marked as rollback-only
  but transactional code requested commit

-- 바깥으로 던져지는 예외 --
UnexpectedRollbackException: Transaction rolled back because
  it has been marked as rollback-only

이유는 로그 세 번째 줄에 있습니다. REQUIRED로 참여한 논리 트랜잭션이 예외로 끝나는 순간, 스프링은 공유 중인 물리 트랜잭션에 rollback-only 마크를 찍습니다. 바깥이 예외를 잡아도 마크는 지워지지 않으니, 커밋 시점에 "커밋해 달라더니 롤백 예약이 돼 있다"며 예외가 나는 겁니다. try-catch는 예외 전파만 막을 뿐 트랜잭션의 운명은 못 바꿉니다.

참고로 이 롤백 규칙은 기본적으로 RuntimeException과 Error에만 적용됩니다. 체크 예외는 기본값으로는 롤백을 일으키지 않으므로, 필요하면 @Transactional(rollbackFor = …)로 명시해야 합니다.

4. REQUIRES_NEW — 따로 커밋되지만, 예외는 남는다

이력은 정산과 운명을 같이할 필요가 없다고 판단하면 record()를 REQUIRES_NEW로 바꿉니다.

@Service
public class HistoryService {
  private final HistoryRepository historyRepository;

  @Transactional(propagation = Propagation.REQUIRES_NEW)
  public void record(Order order) {
    historyRepository.save(History.of(order));
  }
}
JpaTransactionManager : Suspending current transaction, creating
  new transaction with name [..HistoryService.record]
JpaTransactionManager : Committing JPA transaction on EntityManager [..]
JpaTransactionManager : Resuming suspended transaction after
  completion of inner transaction

바깥 트랜잭션은 정지되고, 새 커넥션 위에 새 물리 트랜잭션이 열립니다. 이제 제목의 질문에 답할 수 있습니다. REQUIRES_NEW는 자기 스코프 안에서 예외가 던져질 때, 그때만 롤백됩니다. 방향을 나눠 보면:

안쪽이 실패하면 — 안쪽 물리 트랜잭션만 롤백됩니다. 다만 예외 객체 자체는 그대로 바깥으로 던져지므로, 바깥이 안 잡으면 바깥도 (자기 예외로) 롤백됩니다. 잡으면 바깥은 무사히 커밋됩니다. REQUIRED와 달리 rollback-only 마크가 바깥에 찍히지 않기 때문에, 이번에는 try-catch가 진짜로 통합니다.

바깥이 실패하면 — 안쪽이 이미 커밋된 뒤라면 되돌릴 방법이 없습니다. 정산은 롤백됐는데 "정산 완료" 이력만 남는, 데이터가 서로를 부정하는 상태가 됩니다. 독립 커밋은 공짜가 아니라, "바깥이 죽어도 이 기록은 남아야 하는가"라는 업무 질문에 예라고 답했을 때만 쓰는 도구입니다. 감사 로그나 실패 기록처럼 실패했다는 사실 자체를 남기고 싶을 때 적합합니다.

*

REQUIRES_NEW는 바깥 커넥션을 반납하지 않은 채 커넥션을 하나 더 잡습니다. 요청 하나가 커넥션 두 개를 점유하므로, 풀 크기가 빠듯하면 서로가 서로의 반납을 기다리는 교착으로 풀이 말라붙을 수 있습니다. REQUIRES_NEW를 늘릴 때는 HikariCP maximum-pool-size를 같이 봐야 합니다.

5. NESTED — 세이브포인트, 그리고 JPA의 벽

NESTED는 새 물리 트랜잭션을 만들지 않습니다. 대신 바깥 물리 트랜잭션 안에 JDBC 세이브포인트를 찍고, 안쪽이 실패하면 그 지점까지만 부분 롤백합니다. 안쪽이 성공해도 독립 커밋은 없고, 최종 커밋은 바깥과 함께 일어납니다. "안쪽 실패가 바깥을 죽이면 안 되지만, 바깥이 죽으면 안쪽도 같이 사라져야 한다"는 요구에 정확히 맞는 옵션입니다.

다만 JPA에서는 벽에 부딪힙니다. JpaTransactionManager는 세이브포인트 기반 중첩 트랜잭션을 지원하지 않아서, NESTED를 걸면 시작조차 못 하고 예외가 납니다.

@Transactional(propagation = Propagation.NESTED)
public void record(Order order) { .. }

-- JpaTransactionManager 환경에서의 결과 --
NestedTransactionNotSupportedException:
  JpaDialect does not support savepoints
  - check your JPA provider's capabilities

JdbcTemplate이나 MyBatis처럼 DataSourceTransactionManager를 쓰는 환경에서는 NESTED가 세이브포인트로 동작합니다. 정리하면, JPA 위에서는 사실상 REQUIRED와 REQUIRES_NEW의 양자택일이고, 부분 롤백이 꼭 필요하면 해당 작업만 JDBC 경로로 내리는 식의 우회가 필요합니다.

6. 정산 코드에서 부딪힌 함정 — 프록시를 거치지 않는 호출

정산 로직을 한 곳으로 모으는 리팩토링 중에 부딪힌 함정입니다. 이력 저장을 같은 클래스의 메서드로 옮기고 REQUIRES_NEW를 걸었는데, 로그에 Suspending current transaction이 한 번도 안 찍혔습니다.

@Service
public class SettlementService {
  @Transactional
  public void settle(Order order) {
    repository.save(Settlement.of(order));
    record(order);  // this.record() — 프록시를 거치지 않는다!
  }

  @Transactional(propagation = Propagation.REQUIRES_NEW)
  public void record(Order order) {
    historyRepository.save(History.of(order));  // 그냥 REQUIRED처럼 동작
  }
}

@Transactional은 프록시 기반 AOP입니다. 트랜잭션을 열고 닫는 코드는 빈을 감싼 프록시에 있는데, 같은 클래스 안에서의 this.record() 호출은 프록시를 우회해 원본 객체로 직행합니다. 그래서 record()의 전파 설정은 조용히 무시되고, 호출자의 트랜잭션에 그대로 묻어갑니다. 예외도 로그도 없이 의도와 다르게 동작하니 알아채기 어렵습니다. 해법은 위 실험처럼 별도 빈으로 분리하거나, 경계를 코드로 직접 긋는 TransactionTemplate을 쓰는 것입니다.

*

전파 옵션은 기술이 아니라 업무 요구가 정합니다. "이력은 정산이 실패해도 남아야 하는가?"에 답할 수 있어야 옵션을 고를 수 있습니다 — 같이 죽어야 하면 REQUIRED, 바깥과 무관하게 남아야 하면 REQUIRES_NEW, 부분 롤백이 필요하고 JDBC라면 NESTED.
