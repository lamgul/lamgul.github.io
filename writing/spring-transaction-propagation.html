Spring 트랜잭션 전파 — REQUIRES_NEW는 언제 롤백되는가

2022년 2월• Spring · Transaction · JPA

다섯 곳에 흩어져 있던 정산 로직을 한 곳으로 모으고 나니, 정산 저장과 이력 저장이 같이 롤백되어야 하는지가 남았습니다. 이 답은 @Transactional의 전파(propagation) 옵션이 정하는데, 문서를 읽어도 커밋·롤백 경계가 잘 그려지지 않았습니다. 그래서 REQUIRED · REQUIRES_NEW · NESTED 세 옵션을 실험 코드와 로그로 하나씩 확인해 봤고, 그때 정리한 내용을 적어 둡니다.

먼저 짚고 가기
트랜잭션Transaction
여러 DB 작업을 하나로 묶어 전부 성공하거나 전부 취소하는 단위입니다. 계좌 이체에서 "출금과 입금은 함께 이뤄지거나 둘 다 없던 일이 돼야 한다"는 규칙이 트랜잭션이에요. 이 글은 그 "묶음의 경계"를 어디에 그을지를 다룹니다.
커밋 · 롤백Commit · Rollback
커밋은 지금까지의 변경을 확정 저장하는 것, 롤백은 없던 일로 되돌리는 것입니다. 트랜잭션은 반드시 이 둘 중 하나로 끝납니다. 이 글의 핵심 질문 "정산과 이력이 같이 롤백되는가"가 바로 이 경계 이야기입니다.
전파Propagation
트랜잭션이 걸린 메서드가 또 다른 트랜잭션 메서드를 호출할 때, 두 번째가 기존 트랜잭션에 참여할지 · 새로 열지 · 중첩할지를 정하는 규칙입니다. REQUIRED / REQUIRES_NEW / NESTED가 바로 이 선택지고, 롤백 경계가 여기서 갈립니다.
프록시 기반 AOPProxy-based AOP
스프링은 원본 객체를 얇게 감싼 대리인(프록시)을 만들어, 트랜잭션을 열고 닫는 코드를 그 대리인이 대신 실행합니다. 그래서 같은 클래스 안에서 this.메서드()로 직접 부르면 대리인을 건너뛰어 트랜잭션 설정이 무시됩니다(6장의 함정).
세이브포인트Savepoint
하나의 트랜잭션 안에 찍어 두는 중간 저장 지점입니다. 문제가 생기면 처음까지 다 되돌리지 않고 이 지점까지만 부분 롤백할 수 있어요. NESTED 옵션이 이 기능을 쓰는데, JPA에서는 지원되지 않습니다(5장).

1. 전파는 "경계를 어떻게 이어붙일 것인가"의 문제

@Transactional 메서드가 또 다른 @Transactional 메서드를 호출할 때, 두 번째 메서드가 첫 번째의 트랜잭션에 참여할지, 새로 만들지, 중첩할지를 정하는 규칙이 전파 옵션입니다. 저는 여기서 두 개념을 나눠 놓고 나서야 로그가 읽히기 시작했습니다.

논리 트랜잭션은 @Transactional 스코프 하나하나이고, 물리 트랜잭션은 실제 DB 커넥션 위에서 도는 BEGIN … COMMIT입니다. REQUIRED는 여러 논리 트랜잭션을 하나의 물리 트랜잭션에 태우고, REQUIRES_NEW는 물리 트랜잭션을 하나 더 만듭니다. 롤백이 예상과 다르게 흘러갈 때는 대개 제 머릿속에서 이 둘이 섞여 있었습니다.

2. 실험 준비 — 경계를 로그로 보이게 만든다

실험을 하려면 경계가 보여야 해서 트랜잭션 매니저의 DEBUG 로그부터 켰습니다. 이렇게 해 두면 트랜잭션이 언제 만들어지고, 참여하고, 정지(suspend)되는지가 로그에 남습니다. 아래 로그는 모두 이 설정으로 뽑은 것입니다.

# application.yml — 트랜잭션 경계를 로그로 확인
logging:
  level:
    org.springframework.orm.jpa.JpaTransactionManager: DEBUG
    org.springframework.transaction.interceptor: TRACE

실험 시나리오는 정산 시스템에서 그대로 가져왔습니다. SettlementService.settle()이 정산을 저장하고, HistoryService.record()가 정산 이력을 남깁니다. 확인하고 싶었던 것은 이력 저장이 실패했을 때 정산까지 함께 롤백되는지 하나였습니다.

@Service
public class SettlementService {
  private final SettlementRepository repository;
  private final HistoryService historyService;

  @Transactional  // propagation 기본값 = REQUIRED
  public void settle(Order order) {
    repository.save(Settlement.of(order));
    try {
      historyService.record(order);  // 이력 저장 — 실패할 수 있다
    } catch (RuntimeException e) {
      log.warn("이력 저장 실패 — 정산은 살리고 싶다", e);
    }
  }
}

3. REQUIRED — 함께 살고, 함께 죽는다

record()가 REQUIRED(기본값)이고 내부에서 RuntimeException이 터져도, 바깥에서 try-catch로 잡았으니 정산은 커밋되리라고 생각했습니다. 실행해 보니 정산까지 함께 롤백됐고, 커밋을 시도한 바깥은 UnexpectedRollbackException을 받았습니다.

JpaTransactionManager : Creating new transaction with name
  [..SettlementService.settle]: PROPAGATION_REQUIRED,ISOLATION_DEFAULT
JpaTransactionManager : Participating in existing transaction
JpaTransactionManager : Participating transaction failed
  - marking existing transaction as rollback-only
JpaTransactionManager : Global transaction is marked as rollback-only
  but transactional code requested commit

-- 바깥으로 던져지는 예외 --
UnexpectedRollbackException: Transaction rolled back because
  it has been marked as rollback-only

이유는 로그 세 번째 줄에 적혀 있었습니다. REQUIRED로 참여한 논리 트랜잭션이 예외로 끝나면 스프링은 공유 중인 물리 트랜잭션에 rollback-only 마크를 찍고, 바깥에서 예외를 잡아도 이 마크는 지워지지 않습니다. 그래서 커밋 시점에 "커밋해 달라고 했는데 롤백 예약이 돼 있다"며 예외가 납니다. try-catch가 막아 주는 범위는 예외 전파까지였고, 트랜잭션이 어떻게 끝날지는 거기서 바뀌지 않았습니다.

참고로 이 롤백 규칙은 기본적으로 RuntimeException과 Error에만 적용됩니다. 체크 예외는 기본값으로는 롤백을 일으키지 않아서, 필요하면 @Transactional(rollbackFor = …)로 명시해 줘야 합니다.

4. REQUIRES_NEW — 따로 커밋되지만, 예외는 남는다

이력은 정산과 함께 롤백될 필요가 없다고 판단해서, record()를 REQUIRES_NEW로 바꿔 봤습니다.

@Service
public class HistoryService {
  private final HistoryRepository historyRepository;

  @Transactional(propagation = Propagation.REQUIRES_NEW)
  public void record(Order order) {
    historyRepository.save(History.of(order));
  }
}
JpaTransactionManager : Suspending current transaction, creating
  new transaction with name [..HistoryService.record]
JpaTransactionManager : Committing JPA transaction on EntityManager [..]
JpaTransactionManager : Resuming suspended transaction after
  completion of inner transaction

바깥 트랜잭션은 정지되고, 새 커넥션 위에 새 물리 트랜잭션이 열립니다. 제목에 적어 둔 질문의 답은 여기에 있었습니다. REQUIRES_NEW는 자기 스코프 안에서 예외가 던져질 때만 롤백됩니다. 실패가 어느 쪽에서 나느냐로 나눠 보면 이렇습니다.

안쪽이 실패하면 — 안쪽 물리 트랜잭션만 롤백됩니다. 다만 예외 객체 자체는 그대로 바깥으로 던져지므로, 바깥이 잡지 않으면 바깥도 자기 예외로 롤백됩니다. 잡으면 바깥은 그대로 커밋됩니다. REQUIRED와 달리 rollback-only 마크가 바깥에 찍히지 않아서, 이번에는 try-catch가 의도한 대로 동작했습니다.

바깥이 실패하면 — 안쪽이 이미 커밋된 뒤라면 되돌릴 방법이 없습니다. 정산은 롤백됐는데 "정산 완료" 이력만 남아서, 두 데이터가 서로 어긋난 상태가 됩니다. 그래서 독립 커밋은 "바깥이 실패해도 이 기록은 남아야 하는가"라는 업무 질문에 예라고 답할 수 있을 때만 쓰기로 했습니다. 감사 로그나 실패 기록처럼 실패했다는 사실 자체를 남겨야 하는 경우가 여기에 해당합니다.

*

REQUIRES_NEW는 바깥 커넥션을 반납하지 않은 채 커넥션을 하나 더 잡습니다. 요청 하나가 커넥션 두 개를 점유하므로, 풀 크기가 빠듯하면 서로가 서로의 반납을 기다리는 교착으로 풀이 고갈될 수 있습니다. REQUIRES_NEW를 늘릴 때는 HikariCP maximum-pool-size도 같이 봐 두는 편이 좋았습니다.

5. NESTED — 세이브포인트, 그리고 JPA의 벽

NESTED는 새 물리 트랜잭션을 만들지 않습니다. 대신 바깥 물리 트랜잭션 안에 JDBC 세이브포인트를 찍고, 안쪽이 실패하면 그 지점까지만 부분 롤백합니다. 안쪽이 성공해도 독립 커밋은 없고, 최종 커밋은 바깥과 함께 일어납니다. "안쪽 실패가 바깥을 무너뜨리면 안 되지만, 바깥이 롤백되면 안쪽도 같이 사라져야 한다"는 요구에는 이 옵션이 맞습니다.

다만 제가 쓰던 JPA 환경에서는 쓸 수 없었습니다. JpaTransactionManager는 세이브포인트 기반 중첩 트랜잭션을 지원하지 않아서, NESTED를 걸면 트랜잭션이 시작되지 않고 예외가 납니다.

@Transactional(propagation = Propagation.NESTED)
public void record(Order order) { .. }

-- JpaTransactionManager 환경에서의 결과 --
NestedTransactionNotSupportedException:
  JpaDialect does not support savepoints
  - check your JPA provider's capabilities

JdbcTemplate이나 MyBatis처럼 DataSourceTransactionManager를 쓰는 환경에서는 NESTED가 세이브포인트로 동작합니다. 정리하면 JPA 위에서는 REQUIRED와 REQUIRES_NEW 중에서 고르는 셈이고, 부분 롤백이 꼭 필요하면 해당 작업만 JDBC 경로로 내리는 식으로 우회해야 합니다.

6. 정산 코드에서 부딪힌 함정 — 프록시를 거치지 않는 호출

정산 로직을 한 곳으로 모으는 리팩토링 중에 한동안 헤맸던 부분입니다. 이력 저장을 같은 클래스의 메서드로 옮기고 REQUIRES_NEW를 걸었는데, 로그에 Suspending current transaction이 한 번도 찍히지 않았습니다.

@Service
public class SettlementService {
  @Transactional
  public void settle(Order order) {
    repository.save(Settlement.of(order));
    record(order);  // this.record() — 프록시를 거치지 않는다!
  }

  @Transactional(propagation = Propagation.REQUIRES_NEW)
  public void record(Order order) {
    historyRepository.save(History.of(order));  // 그냥 REQUIRED처럼 동작
  }
}

@Transactional은 프록시 기반 AOP입니다. 트랜잭션을 열고 닫는 코드는 빈을 감싼 프록시에 있는데, 같은 클래스 안에서의 this.record() 호출은 프록시를 거치지 않고 원본 객체로 바로 갑니다. 그래서 record()의 전파 설정은 무시되고, 호출자의 트랜잭션에 그대로 묻어갑니다. 예외도 로그도 남지 않은 채 의도와 다르게 동작해서, 저는 로그에서 Suspending이 빠진 걸 보고서야 알았습니다. 정리한 해결책은 위 실험처럼 별도 빈으로 분리하거나, 경계를 코드로 직접 긋는 TransactionTemplate을 쓰는 것이었습니다.

*

이 실험을 하고 나서 전파 옵션은 기술보다 업무 요구가 정한다고 정리해 두었습니다. "이력은 정산이 실패해도 남아야 하는가?"에 답이 서야 옵션이 골라집니다 — 같이 롤백돼야 하면 REQUIRED, 바깥과 무관하게 남아야 하면 REQUIRES_NEW, 부분 롤백이 필요하고 JDBC라면 NESTED.
