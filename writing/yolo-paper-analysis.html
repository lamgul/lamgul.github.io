논문 분석: YOLO — 검출을 회귀 문제로 바꾼 한 발상

2023년 1월• YOLO · Object Detection · 논문

2023년에 사전학습된 YOLOv5로 주차장 차량감지 MVP를 2주 만에 만들었습니다. 모델은 잘 돌아갔는데, 정작 저는 왜 "한 번만 보는" 것으로 검출이 되는지를 설명하지 못했습니다. 그래서 Redmon 등의 원 논문 "You Only Look Once"(2015)를 다시 읽고, 격자 분할과 바운딩박스 회귀, NMS를 numpy로 직접 만들어 봤습니다. 그때 정리해 둔 내용을 잊지 않으려고 적어 둡니다.

Joseph Redmon 외 · CVPR 2016 · arXiv:1506.02640

먼저 짚고 가기
객체 검출Object Detection
한 이미지에서 객체마다 클래스(무엇)와 바운딩박스 좌표(어디)를 함께 출력하는 컴퓨터 비전 과제입니다. 이미지 한 장에 라벨 하나를 매기는 이미지 분류와 달리 객체를 인스턴스 단위로 찾아 각각 상자와 클래스를 내놓아야 하고, 픽셀마다 라벨을 붙이는 의미론적 분할과 달리 위치를 사각형 하나로 근사합니다. YOLO를 포함한 검출기의 최종 출력은 '클래스 · 상자 좌표 · 점수'의 목록입니다.
바운딩박스Bounding Box
객체를 감싸는 축 정렬 직사각형 영역이며, 중심 좌표와 크기 (cx, cy, w, h) 또는 두 대각 꼭짓점 (x1, y1, x2, y2)으로 표현합니다. YOLO는 상자마다 x, y를 책임 셀 내부의 오프셋(0~1)으로, w, h를 이미지 전체 대비 비율로 회귀하고, 추론 때 이를 (x1, y1, x2, y2) 꼭짓점으로 되돌립니다. 축에 정렬된 사각형이라 기울어진 객체의 회전각은 표현하지 못하며, 윤곽선을 따라가는 분할 마스크와도 다릅니다.
IOUIntersection over Union
두 상자의 교집합 넓이를 합집합 넓이로 나눈 0~1 사이의 겹침 지표입니다(집합의 자카드 지수를 넓이에 적용한 형태). YOLO에서는 두 곳에 쓰입니다 — confidence의 학습 목표값 Pr(Object)·IOU와, NMS에서 중복 여부를 판정하는 임계값입니다. 겹치지 않는 두 상자는 거리와 무관하게 언제나 0이 되므로, 얼마나 멀리 떨어졌는지까지는 구분하지 못합니다.
NMSNon-Maximum Suppression, 비최대 억제
네트워크 출력 뒤에 붙는, 학습되지 않는 그리디 후처리 알고리즘입니다. 후보 상자를 점수 내림차순으로 정렬한 뒤, 가장 점수가 높은 상자를 확정하고 그것과 IOU가 임계값(iou_thr) 이상인 나머지를 제거하는 과정을 목록이 빌 때까지 반복합니다. 규칙이 고정돼 있어 겹침 임계값 하나로만 판정하므로, 실제로 가까이 겹쳐 있는 서로 다른 두 객체를 하나로 뭉개 버릴 수 있습니다.
mAPmean Average Precision
검출기의 정확도를 하나의 스칼라로 요약하는 평가 지표입니다. 클래스마다 정밀도-재현율 곡선 아래 넓이인 AP(Average Precision)를 구한 뒤 전 클래스에 대해 평균낸 값으로, 예측 상자와 정답의 IOU가 임계값(VOC는 0.5)을 넘어야 정답(TP)으로 셉니다. 특정 IOU 기준에서의 종합값이라, 정의가 다른 벤치마크(VOC와 COCO) 사이에서는 같은 mAP라도 곧바로 비교할 수 없습니다.

1. 2015년의 검출은 두 단계였다

당시 객체 검출의 주류는 분류기를 재활용하는 구조였습니다. R-CNN 계열은 먼저 Selective Search로 약 2,000개의 후보 영역을 뽑고, 각 영역을 CNN 분류기에 통과시킨 뒤, 박스를 정제하고 중복을 지우는 후처리를 거칩니다. 후보 생성과 분류, 후처리가 각각 따로 학습·튜닝되는 파이프라인이라 손볼 곳이 많았고 속도도 느렸습니다. 논문에 실린 수치로 Fast R-CNN은 70.0 mAP를 내지만 초당 0.5장이니, 이미지 한 장에 2초가 걸리는 셈입니다.

YOLO는 이 파이프라인을 하나로 접는 쪽을 택했습니다. 논문은 검출을 "이미지 픽셀에서 바운딩박스 좌표와 클래스 확률로 가는 단일 회귀 문제"로 다시 정의한다고 적어 두었습니다. 네트워크를 이미지에 한 번 통과시키면 검출이 끝나는 구조입니다. 그 결과가 Titan X 기준 base 모델 45fps에 VOC 2007 63.4 mAP, 축소판 Fast YOLO는 155fps에 52.7 mAP였습니다. 같은 표에서 Faster R-CNN(VGG-16)은 73.2 mAP로 더 정확했지만 7fps였으니, 정확도를 조금 내주고 실시간 동작을 가져간 절충입니다.

2. 한 번만 본다 — 7×7 격자와 텐서 하나

설계 자체는 단순한 편입니다. 이미지를 S×S 격자(논문은 S=7)로 나누고, 객체의 중심이 떨어진 셀이 그 객체의 검출을 책임집니다. 각 셀은 바운딩박스 B개(논문은 B=2)를 예측하는데, 상자 하나당 x, y, w, h, confidence 다섯 값입니다. x, y는 셀 기준 오프셋(0~1), w, h는 이미지 전체 기준 비율이고, confidence는 Pr(Object)·IOU, 그러니까 "여기 뭔가 있고, 내 상자가 그걸 얼마나 잘 감쌌나"에 해당하는 값입니다. 클래스 확률 C개(VOC는 20)는 상자가 아니라 셀마다 한 벌만 예측합니다. 그래서 출력은 7×7×(2·5+20) = 7×7×30 텐서 하나가 됩니다.

네트워크는 GoogLeNet에서 영감을 받은 합성곱 24층 + 완전연결 2층이고, 검출 시 입력은 448×448입니다. 논문의 구조 그림은 층이 많아 복잡해 보이지만, 읽고 나니 이미지가 들어가면 7×7×30이 나온다는 이야기였습니다. 학습이 되려면 정답도 같은 모양이어야 하니, 정답 인코딩부터 만들어 봤습니다.

import numpy as np

S, C = 7, 20                                  # 격자 7×7, 클래스 20개 (VOC)

def encode_target(boxes, labels, S=S, C=C):
    # boxes: (N, 4) — [cx, cy, w, h], 이미지 기준 0~1 정규화
    target = np.zeros((S, S, 5 + C))
    for (cx, cy, w, h), cls in zip(boxes, labels):
        col, row = int(cx * S), int(cy * S)       # 중심이 속한 셀
        x, y = cx * S - col, cy * S - row         # 셀 내부 오프셋 0~1
        target[row, col, :5] = [x, y, w, h, 1.0]  # 다섯 번째 = 객체 존재
        target[row, col, 5 + cls] = 1.0           # 클래스 one-hot
    return target

car = np.array([[0.52, 0.48, 0.30, 0.42]])      # 화면 중앙의 차 한 대
t = encode_target(car, [6])                       # VOC에서 6 = car
print(t.shape)                                    # (7, 7, 25) — 정답은 셀당 상자 1개
print(np.argwhere(t[..., 4] == 1))               # [[3 3]] — (3,3) 셀이 책임진다

이 인코딩을 손으로 짜 보고 나서야 "회귀 문제로 바꿨다"는 말이 구체적으로 잡혔습니다. 검출이 고정 크기 텐서를 맞히는 문제로 바뀐 것이고, 출력 크기가 고정이면 마지막 완전연결층에서 한 번에 뽑을 수 있습니다. 후보 영역을 따로 만드는 단계가 필요 없어지는 셈입니다.

3. 손실 함수의 디테일 — √w가 있는 이유

손실은 전 항목 제곱합 오차입니다. 최적화가 쉽다는 이유인데, 논문은 그대로 쓰면 생기는 문제를 셋 짚고 각각에 장치를 답니다. 첫째, 49개 셀 대부분에는 객체가 없어서 confidence를 0으로 미는 기울기가 압도합니다. 그래서 좌표 오차에 λ_coord=5를 곱해 키우고, 객체 없는 셀의 confidence 오차에는 λ_noobj=0.5를 곱해 줄입니다. 둘째, 셀의 상자 2개 중 정답과 IOU가 가장 높은 쪽 하나만 "responsible"로 지정해 좌표 손실을 받게 합니다. 학습이 진행되면 두 예측기가 서로 다른 크기·비율에 특화된다고 합니다.

셋째는 상자 크기에 관한 것입니다. 큰 상자의 5px 오차와 작은 상자의 5px 오차는 제곱합 오차에서 같은 값이 되지만, 실제로 같은 정도의 잘못은 아닙니다. 논문의 처방은 w, h 대신 √w, √h를 회귀하는 것이었습니다. 차이가 얼마나 나는지 숫자로 확인해 봤습니다.

big, small = 0.60, 0.06            # 너비 60%짜리 상자, 6%짜리 상자
err = 0.03                          # 둘 다 3%p씩 예측이 빗나갔다고 하자

# w를 그대로 회귀하면 — 손실이 똑같다
print(err ** 2, err ** 2)                            # 0.0009  0.0009

# √w를 회귀하면 — 작은 상자의 실수가 8배쯤 비싸진다
print((np.sqrt(big + err) - np.sqrt(big)) ** 2)      # ≈ 0.00037
print((np.sqrt(small + err) - np.sqrt(small)) ** 2)  # ≈ 0.00303

제곱근을 씌우는 것만으로 작은 상자의 같은 오차가 8배 넘게 비싸집니다. 물론 완전한 해결은 아니어서, 뒤에서 보듯 작은 객체의 위치 오차는 끝까지 YOLO의 주된 약점으로 남습니다. 그래도 방향은 맞춘 절충입니다. Attention 논문의 √d_k를 읽었을 때도 그랬듯, 수식에 붙은 자잘한 상수와 변형에는 대개 이유가 있다는 걸 여기서도 확인했습니다.

4. 마지막 관문 — 텐서를 상자로, 그리고 NMS

추론할 때는 출력 텐서를 다시 상자 목록으로 되돌립니다. 셀마다 상자 2개씩 7×7×2 = 98개 후보가 나오고, 각 상자의 점수는 클래스 확률과 confidence의 곱, 즉 Pr(Class_i)·IOU에 해당하는 값입니다.

def decode(pred, S=7, B=2, C=20, score_thr=0.2):
    boxes, scores, classes = [], [], []
    for row in range(S):
        for col in range(S):
            cell = pred[row, col]                    # 길이 30짜리 벡터
            cls_prob = cell[B * 5:]                  # Pr(Class|Object) — 셀당 한 벌
            for b in range(B):
                x, y, w, h, conf = cell[b * 5:(b + 1) * 5]
                score = conf * cls_prob.max()        # Pr(Object)·IOU × Pr(Class|Object)
                if score < score_thr:
                    continue
                cx, cy = (col + x) / S, (row + y) / S  # 셀 좌표 → 이미지 좌표
                boxes.append([cx - w/2, cy - h/2, cx + w/2, cy + h/2])
                scores.append(score)
                classes.append(cls_prob.argmax())
    return np.array(boxes), np.array(scores), np.array(classes)

큰 객체는 여러 셀에 걸치기 마련이라 후보들이 서로 겹칩니다. 이 중복을 정리하는 후처리가 비최대 억제(NMS)입니다. 점수가 가장 높은 상자를 살리고, 그것과 많이 겹치는(IOU가 임계값 이상인) 나머지를 지우는 일을 반복합니다. 논문에도 NMS로 mAP가 2~3%p 오른다고 적혀 있습니다.

def iou(a, b):
    # a, b: [x1, y1, x2, y2]
    ix1, iy1 = max(a[0], b[0]), max(a[1], b[1])
    ix2, iy2 = min(a[2], b[2]), min(a[3], b[3])
    inter = max(0.0, ix2 - ix1) * max(0.0, iy2 - iy1)
    area = lambda r: (r[2] - r[0]) * (r[3] - r[1])
    return inter / (area(a) + area(b) - inter + 1e-9)

def nms(boxes, scores, iou_thr=0.5):
    order = list(np.argsort(scores)[::-1])        # 점수 내림차순
    keep = []
    while order:
        best = order.pop(0)                       # 가장 확신하는 상자는 살린다
        keep.append(best)
        order = [i for i in order
                 if iou(boxes[best], boxes[i]) < iou_thr]  # 많이 겹치면 제거
    return keep

pred = np.random.rand(7, 7, 30)                  # 네트워크 출력이라 치자
b, s, c = decode(pred)
print(len(s), len(nms(b, s)))                     # 문턱 넘은 후보 수 → NMS 뒤 남은 수
*

차량감지 MVP에서 오검출과 미검출을 조정할 때 제가 처음 손댄 것도 모델이 아니라 위 코드의 score_thr와 iou_thr였습니다. 임계값을 낮추면 놓침이 줄고 오검출이 늘어나는데, 그 트레이드오프가 어느 줄에서 생기는지 알고 나니 값을 만질 때 한결 덜 헤맸습니다. 검출 라이브러리의 conf·iou 파라미터가 하는 일이 이것입니다.

NMS 시뮬레이터 — 임계값 두 개가 상자를 거르는 법직접 조작해 보세요

후보 8개 → 신뢰도 통과 8 · NMS 억제 3 · 생존 5

생존 IoU로 억제 신뢰도 미달

위 nms() 코드와 똑같이 매번 실제 IoU를 계산합니다 — IoU를 0.20까지 내리면 차 한 대에 상자 하나로 정리되지만, 겹치는 상대가 없는 배경 오검출(0.28)은 신뢰도를 0.30까지 올려야 사라집니다

5. 공짜는 없다 — 격자가 지불한 대가

논문은 한계 절에서 그 대가를 스스로 밝혀 둡니다. 셀 하나는 상자 2개와 클래스 한 벌만 예측하므로, 한 셀 안에 중심이 여러 개 떨어지면 물리적으로 다 잡을 수 없습니다. 논문이 드는 예가 새 떼처럼 무리 지은 작은 객체입니다. 여러 번의 다운샘플링을 지나며 거칠어진 특징으로 상자를 그리는 데다, √w로 완화했어도 작은 상자의 위치 오차가 주된 오류원으로 남습니다. 대신 얻은 것도 있습니다. 후보 영역만 보는 방식과 달리 이미지 전체를 보고 추론하므로 문맥이 반영되고, 배경을 객체로 오인하는 오류가 Fast R-CNN의 절반 이하였습니다.

이 한계 목록은 읽고 나서 제 체크리스트가 됐습니다. 주차장 CCTV 화면 속 차량은 크고, 주차 구역 덕에 간격이 일정해서 한 셀에 두 대가 몰릴 일이 거의 없었습니다. 돌아보면 YOLO의 약점 목록에 걸리는 항목이 하나도 없는 문제였고, 사전학습 모델을 그대로 쓰는 결정이 편했던 이유가 논문에 이미 적혀 있었던 셈입니다. 제가 쓴 YOLOv5는 여러 해상도의 격자와 앵커로 v1의 한계를 상당 부분 완화했지만, "격자가 담을 수 있는 밀도까지만 잡는다"는 트레이드오프의 뼈대는 지금도 같습니다.

*

검출 모델을 고르기 전에 데이터에서 객체 중심을 격자에 뿌려 한 셀에 2개 이상 몰리는 비율을 먼저 재 보면 도움이 됩니다. 2절의 encode_target에서 덮어쓰기가 발생하는 비율을 세면 되니 다섯 줄이면 충분합니다. 그 비율이 높다면 하이퍼파라미터를 만지기보다 입력 해상도나 타일링, 더 촘촘한 격자의 모델을 먼저 보는 편이 낫습니다.

6. 논문에서 실무로 — 2주짜리 MVP의 뒷배

대흥정보에서 만든 차량감지 MVP는 이 논문 계보 위에 얹힌 작업이었습니다. 사전학습된 '차량' 클래스를 필터링하고 바운딩박스 좌표로 주차 구역 점유를 판단하는 구조는, 단일 네트워크가 상자와 점수, 클래스를 한 번에 내주기에 가능한 설계였습니다. 실시간 30fps라는 요구도 같은 성질 덕에 맞출 수 있었습니다. 그보다 앞서 ETRI에서 Vision AI 분야 연구에 참여했을 때도 검출은 늘 파이프라인의 출발점이었는데, 그때는 도구로만 쓰던 그 출발점을 이제는 이 논문의 격자 그림으로 설명할 수 있게 됐습니다.

  • 문제를 다시 정의하는 것도 기여가 됩니다. YOLO는 더 좋은 분류기를 만든 것이 아니라 검출을 텐서 하나 맞히는 회귀로 바꿨고, 문제를 세우는 방식이 달라지자 여러 단계가 한 단계로 줄었습니다.
  • 한계 절은 실무의 체크리스트로 쓸 만합니다. 작은 객체와 군집, 격자 밀도처럼 논문이 적어 둔 항목에 내 데이터가 걸리는지부터 확인하고 나면, 모델 선택을 유행이 아니라 근거로 설명할 수 있게 됩니다.
  • 미니 구현 수십 줄이 남긴 것도 있습니다. 인코딩과 디코딩, NMS를 직접 짜 보면 라이브러리의 conf·iou 파라미터가 어느 수식에서 왔는지 읽힙니다. 그다음부터는 튜닝할 때 무엇을 움직이고 있는지 알고 만지게 됩니다.
