작은 팀의 AWS — EC2 한 대에서 시작하는 아키텍처

2024년 11월• AWS · 아키텍처 · 비용

두세 명으로 서비스를 만들면서 AWS 아키텍처 문서를 찾아 읽으면, 대개 멀티 AZ와 오토스케일링과 관리형 데이터베이스가 처음부터 그려진 완성형이 나옵니다. 정작 제 손에 들린 첫 계산서와는 거리가 있었습니다. 그래서 EC2 한 대 → ALB+ASG → 관리형 서비스로 단계를 넓혀오면서, 각 단계로 넘어갈 때 무엇을 보고 판단했는지와 그때의 월 비용을 정리해 두었습니다. 금액은 us-east-1 온디맨드 기준의 대략치라 리전과 시점에 따라 달라지니, 절대값보다는 단계 사이의 비율로 읽어 주시면 좋겠습니다.

먼저 짚고 가기
EC2Elastic Compute Cloud
AWS가 가상 머신을 빌려주는 IaaS 컴퓨트 서비스입니다(리눅스 온디맨드는 초 단위, 최소 60초 과금). AMI라는 디스크 이미지와 vCPU·메모리·네트워크 성능의 조합인 인스턴스 타입을 골라 기동하고, 루트 디스크는 보통 인스턴스와 분리된 네트워크 블록 스토리지인 EBS를 붙여 씁니다. 인스턴스를 정지했다 다시 켜면 다른 물리 호스트에서 뜨기 때문에 퍼블릭 IPv4 주소가 바뀌는데, 1절 코드에서 EIP를 따로 붙여 둔 이유가 그것입니다.
ALBApplication Load Balancer
HTTP·HTTPS·gRPC를 다루는 L7 로드 밸런서로, Elastic Load Balancing 제품군의 한 종류입니다. 리스너가 받은 요청을 호스트 헤더·경로·HTTP 헤더 같은 조건의 규칙으로 매칭해 타깃 그룹으로 넘기고, 타깃 그룹은 등록된 인스턴스에 주기적으로 헬스 체크를 보내 정상 판정을 받은 대상에게만 트래픽을 흘립니다. 요금은 시간당 고정요금에 LCU(새 연결·활성 연결·처리 바이트·규칙 평가 네 지표 중 가장 큰 값)가 더해지는 구조입니다. 임의의 TCP/UDP를 그대로 넘겨야 하거나 고정 IP가 필요한 경우는 ALB가 아니라 NLB의 영역입니다.
ASGAuto Scaling Group
EC2 인스턴스 묶음을 목표 대수(desired capacity)에 맞춰 유지하는 관리 단위입니다. AMI·인스턴스 타입·user_data 같은 기동 사양을 담은 launch template과 여러 AZ의 서브넷 목록을 받아, 헬스 체크에 실패한 인스턴스를 종료하고 같은 사양으로 새로 띄우면서 min_size와 max_size 사이에서 대수를 맞춥니다. health_check_type을 ELB로 두면 EC2 자체 상태 검사 대신 타깃 그룹의 헬스 체크 결과로 인스턴스의 생사를 판정합니다. 하는 일이 복구가 아니라 교체라서, 죽은 인스턴스 안에만 있던 데이터는 돌아오지 않습니다.
관리형 서비스Managed Service
특정 제품이 아니라 운영 책임의 경계가 어디에 그어지는지를 가리키는 분류입니다. 프로비저닝, 마이너 버전 패치, 자동 백업, 기반 하드웨어 장애 시의 교체 같은 작업을 AWS가 맡고, 사용자는 그 위에서 하는 일(스키마·쿼리·데이터)만 책임집니다. 대신 통제권도 같이 넘어가서, RDS라면 OS에 SSH로 들어갈 수 없고 슈퍼유저 권한도 주어지지 않으며 유지 관리 창에 재시작이 일어날 수 있습니다. 자동 백업이 켜져 있다는 사실과 그 백업에서 실제로 복구된다는 사실은 여전히 별개이고, 확인은 사용자 몫입니다.
무상태Stateless
요청과 요청 사이에 남아 있어야 하는 데이터를 서버 프로세스의 메모리나 로컬 디스크에 두지 않는 설계 원칙입니다. 세션은 Redis나 DB에, 업로드 파일은 S3에 두고, 각 요청은 필요한 상태를 그때그때 바깥에서 읽어 옵니다. 그래야 로드 밸런서가 sticky session 없이 아무 인스턴스에나 요청을 보내도 되고, ASG가 어느 인스턴스를 언제 종료해도 잃을 것이 없습니다. 상태가 사라지는 게 아니라 옮겨 가는 것이라 그 저장소가 새로운 단일 장애점이 되며, 다시 만들 수 있는 로컬 캐시 정도는 남겨 두어도 전제가 깨지지 않습니다.

1. 한 대로 시작해도 되는가

제 경우에는 됐습니다. 검증 단계의 트래픽은 t4g.small 한 대가 감당하고도 남았습니다. 제가 만들었던 YOLO 차량 감지 MVP처럼 "이게 되는가"를 확인하는 단계라면 더 그렇습니다. 처음엔 멀티 AZ부터 그려 두는 게 맞나 싶었는데, 아직 아무에게도 약속하지 않은 가용성에 매달 돈을 내는 셈이라 접어 두었습니다.

대신 감수하기로 한 리스크는 적어 두었습니다. 한 대짜리 아키텍처는 인스턴스 장애나 AZ 장애가 나면 그대로 내려갑니다. SLA를 약속한 고객이 없고 몇십 분의 다운타임을 스스로 감당할 수 있는 동안에는 그럭저럭 맞는 거래였습니다. 그 경계를 넘어가면 2단계로 가야 하는데, 어떤 신호를 보고 판단했는지는 3절에 적었습니다.

# main.tf — 시작은 이 정도면 충분합니다
resource "aws_instance" "app" {
  ami                    = data.aws_ami.al2023.id
  instance_type          = "t4g.small"   # Arm, 온디맨드 약 $12/월
  vpc_security_group_ids = [aws_security_group.app.id]
  user_data              = file("cloud-init.sh")

  root_block_device {
    volume_type = "gp3"
    volume_size = 30                     # $0.08/GB·월 → 약 $2.4
  }

  tags = { Name = "app-single" }
}

resource "aws_eip" "app" {
  instance = aws_instance.app.id         # 인스턴스를 갈아끼워도 IP 고정
}

월 비용은 인스턴스 약 $12, EBS 약 $2.4, 여기에 2024년부터 과금되는 퍼블릭 IPv4 주소 약 $3.6을 더해 $20 미만이었습니다. 프리 티어나 세이빙 플랜을 쓰면 더 내려갑니다. 이 금액에서 시작해도 부담이 없었던 건, 위 코드가 그대로 다음 단계의 재료가 되기 때문입니다.

2. 죽는 것은 괜찮다, 못 돌아오는 것이 문제다

한 대로 굴리면서 더 신경이 쓰였던 쪽은 다운타임보다 복구 불능이었습니다. SSH로 들어가 손으로 패키지를 깔고 설정을 고친 서버는 장애가 났을 때 같은 상태로 다시 만들기가 어렵습니다. 터미널에서 손으로 한 일은 어디에도 남지 않아서, 정작 필요한 순간에 그대로 재현할 방법이 없었습니다.

그래서 한 대일 때일수록 규칙을 정해 두었습니다. 부팅 시 상태는 전부 user_data에, 앱은 컨테이너에, 데이터는 EBS 스냅샷에 두는 식입니다. 이렇게 해 두면 인스턴스가 소모품이 되어서, 상태가 이상하면 오래 붙잡지 않고 지운 뒤 다시 띄울 수 있었습니다.

#!/bin/bash
# cloud-init.sh — 부팅마다 같은 상태를 만든다
dnf install -y docker
systemctl enable --now docker
usermod -aG docker ec2-user

# 앱은 컨테이너로, 재시작 정책은 데몬에게 맡긴다
docker run -d --name app --restart unless-stopped \
  -p 80:8000 --env-file /etc/app/.env \
  ghcr.io/example/app:1.4.2   # latest 대신 버전 고정

스냅샷은 손으로 뜨지 않고 Data Lifecycle Manager에 맡겨 두었습니다. "매일 새벽 1시, 7개 보관" 정책 하나면 충분합니다. 다만 정책을 걸어 두는 것과 실제로 복구되는 것은 다른 이야기라, 한 번은 스냅샷에서 인스턴스를 새로 띄워 보는 편이 좋습니다. 백업이 있다는 말과 복구를 해봤다는 말 사이가 생각보다 멀었습니다. 정산 코드를 리팩토링하면서 얻은 감각이기도 한데, 돈이 걸린 데이터 앞에서는 그 차이가 특히 크게 느껴집니다.

이렇게 정리해 둔 것이 다음 단계에서 도움이 됐습니다. user_data에 모아 둔 부팅 스크립트는 거의 그대로 ASG의 launch template이 됐습니다.

3. 두 번째 인스턴스가 필요해지는 신호

스케일 아웃을 고민하게 된 계기는 CPU 그래프가 아니었습니다. 배포할 때마다 수십 초씩 끊기는 것이 사용자에게 보이기 시작했고, 회의에서 "이 한 대가 죽으면"이라는 말이 나왔습니다. 트래픽이 문제라면 인스턴스 크기를 키우는 스케일 업이 더 싸고 단순한 경우가 많습니다. 두 대로 넘어간 이유도 성능보다는 무중단 배포와 장애 허용 쪽이었습니다.

# ALB가 받고, ASG가 두 AZ에 최소 2대를 유지한다
resource "aws_lb" "app" {
  load_balancer_type = "application"    # 기본 약 $16/월 + LCU
  subnets            = module.vpc.public_subnets
}

resource "aws_autoscaling_group" "app" {
  min_size            = 2
  max_size            = 4
  vpc_zone_identifier = module.vpc.private_subnets
  target_group_arns   = [aws_lb_target_group.app.arn]
  health_check_type   = "ELB"            # 포트가 아니라 /health로 판정

  launch_template {
    id      = aws_launch_template.app.id  # 2절의 cloud-init이 여기로
    version = "$Latest"
  }
}

비용은 그만큼 올라갑니다. ALB 기본요금 약 $16에 LCU 사용량이 붙고 인스턴스가 두 대가 되면서, 1단계의 세 배쯤인 월 $60 안팎이 됐습니다. 정작 더 걸렸던 건 돈이 아니라, 두 대가 되는 순간 앱이 무상태여야 한다는 전제였습니다. 메모리에 두던 세션은 Redis나 DB로, 로컬 디스크에 받던 업로드 파일은 S3로 옮겨야 합니다. 제 경우 이 코드 작업이 테라폼 작업보다 오래 걸렸습니다. 반대로 처음부터 무상태로 짜 두면 2단계 전환은 인프라 코드 몇십 줄로 끝납니다.

4. 상태는 관리형으로 밀어낸다

DB를 EC2에 직접 올리면 백업, 마이너 버전 패치, 장애 시 복구가 전부 제 일이 됩니다. RDS db.t4g.micro는 약 $12/월(+스토리지)로 자동 백업과 시점 복구(PITR), 패치 창을 대신 맡아 줍니다. 처음엔 이 요금이 비싸 보였는데, 비교 대상을 인스턴스 요금이 아니라 제 시간으로 놓고 나서 생각이 달라졌습니다. 온프레미스에서 vLLM을 서빙하던 때는 패치와 모니터링과 새벽 장애 대응이 전부 사람 몫이었습니다.

resource "aws_db_instance" "main" {
  engine            = "postgres"
  instance_class    = "db.t4g.micro"  # 약 $12/월 + 스토리지
  allocated_storage = 20

  multi_az                = false      # 처음엔 스냅샷 복구로 충분, 켜면 2배
  backup_retention_period = 7          # 7일 PITR
  deletion_protection     = true       # terraform destroy 사고 방지
  db_subnet_group_name    = aws_db_subnet_group.main.name
}

같은 기준을 다른 상태에도 적용했습니다. 파일은 S3(스탠더드 $0.023/GB·월)로, 큐는 SQS로 옮겼습니다. 크롤링 파이프라인처럼 작업 큐가 필요한 구조에서는 SQS가 월 백만 요청까지 무료 구간이 있어서, 브로커를 직접 띄워 돌보는 쪽을 택할 이유를 찾기 어려웠습니다. 정리하면 차별화가 아닌 부분은 밀어내고 팀의 시간은 앱에 쓴다 정도의 기준입니다. 다만 vLLM 서빙처럼 관리형이 아직 비싸거나 요건에 맞지 않는 워크로드는 직접 운영하는 편이 나을 수 있어서, 기준으로 두었을 뿐 규칙으로 삼지는 않았습니다.

5. 월 비용 감각 — 계산서에서 놀라는 항목들

단계별로 대략 이렇게 잡혔습니다. 1단계 한 대는 $20 미만, 2단계 ALB+ASG 두 대는 $60 안팎, 3단계에서 RDS와 S3를 더하면 $80~100 정도입니다. 여기까지는 예상하던 범위였고, 계산서에서 따로 눈에 걸린 항목이 몇 개 있었습니다.

먼저 NAT 게이트웨이입니다. 시간당 $0.045라 켜 두기만 해도 약 $32/월이고, 처리하는 트래픽에 GB당 $0.045가 또 붙습니다. 프라이빗 서브넷을 만들면서 무심코 하나 띄워 두면 인스턴스보다 비싼 항목이 됩니다. 다음은 데이터 전송 아웃으로, 인터넷으로 나가는 트래픽이 GB당 약 $0.09라서 이미지나 영상을 EC2에서 직접 서빙하면 금액이 조용히 쌓입니다. 마지막은 앞서 적은 퍼블릭 IPv4인데, 주소 하나에 약 $3.6/월이라 쓰지 않고 잡아 둔 EIP부터 정리했습니다.

*

NAT 게이트웨이는 띄우기 전에 한 번 따져 볼 만합니다. 프라이빗 서브넷의 인스턴스가 밖으로 나갈 일이 S3와 ECR뿐이라면, S3는 무료인 게이트웨이 엔드포인트로, ECR은 시간당 과금이 있지만 대개 NAT보다 싼 인터페이스 엔드포인트로 대체할 수 있습니다. 초기에는 퍼블릭 서브넷에 보안 그룹 인바운드를 막아 두는 것으로 충분한 경우도 많았습니다. "프라이빗 서브넷이니까 NAT"라고 반사적으로 정하기보다, 한 번 계산해 보고 결정하는 편이 낫습니다.

워크로드가 안정된 뒤에는 세이빙 플랜이나 예약 인스턴스로 컴퓨트 비용을 상당 폭 줄일 수 있습니다. 다만 아키텍처가 자리를 잡은 다음의 이야기라, 어떤 인스턴스 타입으로 갈지 아직 흔들리는 단계에서 1년 약정부터 걸면 오히려 손해가 될 수 있습니다.

6. 판단 기준 요약

돌아보면 단계를 넘길 때 제가 본 것은 트래픽 수치보다 약속이었습니다. 누구에게 무엇을 약속했는지를 먼저 적고, 그 약속을 지킬 수 있는 최소 구조를 골랐습니다.

정리하면 이렇습니다. 아무에게도 SLA를 약속하지 않은 동안에는 EC2 한 대로 두되, user_data와 스냅샷으로 복구할 수 있게 해 둡니다. 배포 중단이 사용자에게 보이고 한 대 장애가 사업 문제가 되면 ALB+ASG로 넘어가는데, 그 전에 앱을 무상태로 바꿔 두어야 합니다. 데이터를 잃으면 안 되는 단계가 되면 상태를 관리형으로 밀어내고, 복구 리허설까지 마쳐야 도입이 끝납니다. 각 단계가 앞 단계에서 정해 둔 규칙 위에 서 있어서, 순서를 건너뛰면 그만큼 다음 단계에서 다시 손이 갔습니다.

그리고 어느 단계에 있든, 두 대로 넘어가기 전에 갖춰 두어야 했던 것이 있습니다. 지금 서버 안에서 무슨 일이 벌어지는지 보이게 만드는 일, 그러니까 로그와 메트릭과 알림입니다. 위에 적은 판단들도 결국 그 데이터를 보고 내린 것이었습니다.
