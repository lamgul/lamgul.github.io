분석 · 아키텍처

아키텍처 패턴 · 레이어드에서 헥사고날로

2021년 5월• 약 12분• 레이어드 · 포트와 어댑터 · DIP

레이어드 아키텍처는 죄가 없습니다. 컨트롤러→서비스→리포지토리라는 세 칸 그림은 처음 몇 달 동안 잘 작동합니다. 문제는 그다음입니다. 어느 날 돌아보면 도메인 규칙이 ORM 호출 틈새에 흩어져 있고, 테스트 하나 돌리는 데 DB가 필요해져 있습니다. SafeIn 백엔드를 NestJS로 단독 설계하며, 크롤링 파이프라인을 DDD로 다시 세우며 겪은 그 붕괴의 순서와 복구의 순서를, 하나의 도메인으로 끝까지 따라가 봅니다.

먼저 짚고 가기
레이어드 아키텍처Layered Architecture
코드를 컨트롤러→서비스→리포지토리처럼 위아래 층으로 쌓고, 위층은 바로 아래층만 부르도록 정리하는 방식입니다. 건물의 1·2·3층처럼 역할을 나눈다고 보면 됩니다. 이 글은 이 익숙한 구조가 왜 시간이 지나면 무너지는지에서 출발합니다.
ORMObject-Relational Mapping
데이터베이스의 표(테이블)를 프로그래밍 언어의 객체처럼 다루게 해 주는 도구입니다(글의 TypeORM이 그 예). 번역기처럼 편하지만, 코드가 ORM의 말투에 물들면 업무 규칙이 기술 문장으로 변질된다는 것이 글의 핵심 문제의식입니다.
의존성 역전 원칙DIP
중요한 쪽(도메인 규칙)이 덜 중요한 쪽(저장 기술)에 끌려다니지 않도록, 화살표의 방향을 뒤집어 둘 다 추상(인터페이스)을 바라보게 만드는 원칙입니다. 이 글이 레이어드를 고치는 열쇠로 삼는 바로 그 규칙입니다.
헥사고날 아키텍처 (포트와 어댑터)Hexagonal / Ports & Adapters
안쪽에 도메인과 그 도메인이 요구하는 계약(포트)을 두고, 바깥의 어댑터가 그 계약을 실제로 이행하게 만드는 구조입니다. 포트는 콘센트 규격, 어댑터는 거기에 맞춰 꽂히는 플러그라고 비유할 수 있습니다. 의존성 역전을 시스템 전체의 규칙으로 끌어올린 것입니다.
유스케이스Use Case
"수집한 상품을 저장한다"처럼 시스템이 수행하는 하나의 업무 단위를 담은 코드입니다. 이 글에서는 유스케이스가 구체적인 DB가 아니라 포트에만 기대도록 만드는 것이 목표입니다.

무너지는 순서는 항상 같다

예제 도메인은 크롤링 파이프라인의 한 조각입니다. 수집기가 상품 페이지를 긁어 오면, 서버는 두 가지 규칙으로 저장합니다. 규칙 1 — 같은 수집원의 같은 상품이면 새로 만들지 않고 갱신한다. 규칙 2 — 가격이 달라졌을 때만 가격 변동 이력을 남긴다. 레이어드로 쓰면 보통 이런 서비스가 나옵니다.

// product.service.ts — 레이어드의 "모범 답안"처럼 보이는 코드
@Injectable()
export class ProductService {
  constructor(
    @InjectRepository(ProductEntity)
    private readonly repo: Repository<ProductEntity>,
  ) {}

  async saveCrawled(dto: CrawledProductDto) {
    // 규칙 1: 같은 수집원·같은 키면 갱신
    const found = await this.repo.findOne({
      where: { source: dto.source, sourceKey: dto.sourceKey },
    });
    if (!found) {
      return this.repo.save(this.repo.create({ ...dto, priceHistory: [] }));
    }
    // 규칙 2: 가격이 달라졌을 때만 이력을 남긴다
    if (found.price !== dto.price) {
      found.priceHistory.push({ price: found.price, at: new Date() });
      found.price = dto.price;
    }
    return this.repo.save(found);
  }
}

이 코드는 리뷰를 통과합니다. 계층도 지켰고, 중복도 없고, 동작도 맞습니다. 그런데 여기서부터 붕괴가 시작되는 순서는 어느 코드베이스에서나 거의 같았습니다.

1단계 — 서비스가 ORM의 언어로 말하기 시작합니다. findOne({ where }), create, save는 전부 TypeORM의 어휘입니다. 규칙 1은 "같은 상품이면 갱신"이라는 업무 문장이 아니라 "findOne이 null이면 create" 라는 기술 문장으로 기록됐습니다. 2단계 — 도메인 규칙이 쿼리 속으로 숨습니다. 목록 화면이 생기면 "판매 중인 상품"의 정의가 QueryBuilder의 where 절 안으로 들어가고, 같은 정의가 화면마다 조금씩 다르게 복제됩니다. 3단계 — 엔티티가 모든 계층을 관통합니다. @Column이 붙은 ProductEntity가 컨트롤러 응답으로도, 서비스 인자로도 쓰이면서 DB 스키마 변경이 곧 API 변경이 됩니다. 4단계 — 테스트가 DB를 요구합니다. "가격이 같으면 이력을 안 남긴다"라는 순수한 규칙 하나를 검증하는 데 테스트 컨테이너가 뜹니다. 여기까지 오면 서비스 계층은 존재하지만, 도메인은 어디에도 없습니다.

문제는 계층이 아니라 화살표다

흔한 오해는 "계층을 덜 지켜서" 무너졌다는 진단입니다. 위 코드는 레이어드의 규칙 — 위는 아래만 호출한다 — 을 완벽히 지키고 있습니다. 무너뜨린 건 규칙이 아니라 규칙이 다루지 않는 것, 즉 화살표의 종점이 구체 타입이라는 사실입니다. 서비스의 import 목록에 typeorm이 있는 한, 도메인 규칙은 TypeORM의 어휘로 표현될 수밖에 없습니다. 의존한다는 건 그 대상의 언어로 말하게 된다는 뜻이니까요.

여기서 왜 "의존 방향을 뒤집어야 한다"는 규칙이 성립하는지 한 번은 논증해 볼 가치가 있습니다. 핵심 전제는 두 가지입니다. 첫째, 의존은 변경이 전파되는 통로입니다. A가 B에 의존하면 B의 변경은 A를 흔들 수 있지만 역은 성립하지 않습니다. 둘째, 도메인 규칙과 저장 기술은 변경의 이유가 다릅니다. "가격 이력을 언제 남길 것인가"는 업무가 바꾸고, "PostgreSQL이냐 어떤 ORM이냐"는 기술 사정이 바꿉니다. 이유가 다르면 시점도 다르고, 시점이 다른 두 코드가 의존으로 묶여 있으면 한쪽의 변경 때마다 다른 쪽이 재검증 대상이 됩니다. 그렇다면 화살표는 더 안정적이어야 하는 쪽, 즉 도메인을 향해야 합니다. 이것이 의존성 역전 원칙(DIP)이 이 상황에서 성립하는 이유입니다 — 도그마여서가 아니라, 변경 전파 비용의 부등식이 그렇게 말하기 때문입니다.

한 가지를 분리해서 봐야 합니다. 런타임 호출은 뒤집을 수 없습니다. 실행 시점에는 어차피 유스케이스가 DB를 부릅니다. 뒤집을 수 있는 것은 소스코드 의존성뿐이고, 그 도구가 인터페이스입니다. 헥사고날 아키텍처(포트와 어댑터)는 이 역전을 시스템 전체의 규칙으로 승격한 것입니다. 안쪽에는 도메인과 도메인이 소유한 계약(포트)이 있고, 바깥의 어댑터들이 그 계약을 이행합니다. 육각형 그림은 장식이고, 본질은 "계약의 소유권이 안쪽에 있다" 한 문장입니다.

도메인과 포트 — 계약의 소유권을 뒤집는다

복구의 첫걸음은 프레임워크가 닿지 않는 곳으로 규칙을 꺼내는 일입니다. @Column도 @Injectable도 없는, 순수 TypeScript 클래스로요.

// domain/product.ts — 이 파일의 import 목록은 거의 비어 있다
export type PriceChange = { price: number; at: Date };

export class Product {
  constructor(
    readonly source: string,
    readonly sourceKey: string,
    private _price: number,
    private readonly _history: PriceChange[] = [],
  ) {}

  get price() { return this._price; }
  get history() { return [...this._history]; }

  // 규칙 2가 if문이 아니라 "이름 있는 메서드"가 됐다
  applyCrawledPrice(next: number, now = new Date()): boolean {
    if (next === this._price) return false;
    this._history.push({ price: this._price, at: now });
    this._price = next;
    return true;
  }
}

그다음이 포트입니다. 저장이 필요하다는 사실은 도메인의 사정이므로, 도메인이 원하는 어휘로 인터페이스를 선언합니다. 시그니처를 보면 "찾는다, 저장한다"만 있고 where도 QueryBuilder도 없습니다. 유스케이스는 이 포트에만 의존합니다.

// domain/product.repository.port.ts — 계약의 소유자는 도메인
export interface ProductRepositoryPort {
  findBySourceKey(source: string, key: string): Promise<Product | null>;
  save(product: Product): Promise<void>;
}
export const PRODUCT_REPOSITORY = Symbol('PRODUCT_REPOSITORY');

// application/save-crawled-product.usecase.ts
export type CrawledProduct = { source: string; sourceKey: string; price: number };

@Injectable()
export class SaveCrawledProduct {
  constructor(
    @Inject(PRODUCT_REPOSITORY)
    private readonly products: ProductRepositoryPort,
  ) {}

  async execute(cmd: CrawledProduct): Promise<void> {
    const found = await this.products.findBySourceKey(cmd.source, cmd.sourceKey);
    const product = found ?? new Product(cmd.source, cmd.sourceKey, cmd.price);
    if (found) product.applyCrawledPrice(cmd.price);
    await this.products.save(product);
  }
}

같은 두 규칙인데 코드의 성격이 달라졌습니다. 규칙 1은 유스케이스의 흐름으로, 규칙 2는 applyCrawledPrice라는 이름으로 자리를 잡았고, 어느 쪽에도 ORM의 흔적이 없습니다. 레거시 코드를 UseCase 단위로 개선할 때도 순서는 같았습니다 — 규칙을 먼저 꺼내고, 규칙이 필요로 하는 바깥세상을 포트로 선언하는 것.

*

실무 팁: TypeScript의 interface는 컴파일하면 사라지므로 NestJS가 주입 시점에 붙잡을 토큰이 없습니다. 포트 파일에서 Symbol 토큰을 함께 export하고 @Inject(PRODUCT_REPOSITORY)로 받는 것이 정석입니다. 그리고 포트 파일은 반드시 domain(또는 application) 폴더에 두세요. "계약의 소유권이 안쪽에 있다"는 원칙을 폴더 구조가 물리적으로 강제하게 됩니다.

어댑터 — 세부는 바깥에서 계약을 이행한다

이제 TypeORM은 사라지는 게 아니라 제자리로 갑니다. 어댑터는 포트를 구현하면서, ORM의 행(row)과 도메인 모델 사이의 번역을 전담합니다. 지저분함이 없어지는 게 아니라, 지저분함이 한 파일에 격리되는 것입니다.

// infrastructure/typeorm-product.repository.ts — 계약의 "이행자"
@Injectable()
export class TypeOrmProductRepository implements ProductRepositoryPort {
  constructor(
    @InjectRepository(ProductEntity)
    private readonly repo: Repository<ProductEntity>,
  ) {}

  async findBySourceKey(source: string, key: string) {
    const row = await this.repo.findOne({ where: { source, sourceKey: key } });
    return row ? ProductMapper.toDomain(row) : null;
  }

  async save(product: Product) {
    await this.repo.save(ProductMapper.toRow(product));
  }
}

// product.module.ts — 화살표가 뒤집히는 유일한 지점
providers: [
  SaveCrawledProduct,
  { provide: PRODUCT_REPOSITORY, useClass: TypeOrmProductRepository },
],

모듈의 provide 한 줄이 이 아키텍처의 급소입니다. 소스코드 상에서 유스케이스는 포트(추상)를 바라보고, 어댑터도 포트를 바라봅니다. 두 화살표가 모두 안쪽을 향하고, 둘을 실제로 연결하는 결정은 조립 지점인 모듈에만 존재합니다. NestJS의 DI 컨테이너가 바로 이 역전을 위한 도구인데, 레이어드로 쓸 때는 그 능력을 "구체 클래스를 편하게 주입하는 데"만 쓰고 있었던 셈입니다.

교체 시연 — 첫 번째 수혜자는 테스트다

"어댑터를 갈아 끼운다"라고 하면 다들 DB 교체를 상상하지만, 실제로 가장 먼저, 가장 자주 갈아 끼우게 되는 어댑터는 테스트용입니다. 포트가 제대로 설계됐다면 두 번째 구현체는 20줄이면 나옵니다.

// test/in-memory-product.repository.ts — 두 번째 어댑터
class InMemoryProductRepository implements ProductRepositoryPort {
  private rows = new Map<string, Product>();
  async findBySourceKey(s: string, k: string) { return this.rows.get(`${s}:${k}`) ?? null; }
  async save(p: Product) { this.rows.set(`${p.source}:${p.sourceKey}`, p); }
}

// DB도, NestJS 컨테이너도 없이 도메인 규칙을 검증한다
it('가격이 같으면 이력을 남기지 않는다', async () => {
  const repo = new InMemoryProductRepository();
  const usecase = new SaveCrawledProduct(repo);

  await usecase.execute({ source: 'shop-a', sourceKey: '42', price: 12900 });
  await usecase.execute({ source: 'shop-a', sourceKey: '42', price: 12900 });

  const saved = await repo.findBySourceKey('shop-a', '42');
  expect(saved!.history).toHaveLength(0);
});

처음의 레이어드 서비스에서는 이 테스트가 테스트 DB 없이는 불가능했습니다. 지금은 new 두 번으로 끝납니다. 그리고 이 구조가 실전에서 진짜 효자였던 곳이 크롤링 파이프라인입니다. 저장소만이 아니라 수집원 자체를 포트로 두면 — "페이지를 가져와 상품으로 해석한다"라는 계약 — 수집원마다 HTML 구조가 제각각이어도 수집원별 어댑터가 각자 계약을 이행할 뿐, 파이프라인의 도메인 코드는 한 줄도 바뀌지 않습니다. 새 수집원 추가가 "어댑터 클래스 하나 + 모듈 등록 한 줄"로 끝나는 경험을 하고 나면 이 구조를 되돌릴 수 없게 됩니다.

!

함정: 포트 시그니처에 인프라 타입이 하나라도 스며들면 전부 무너집니다. findWith(qb: SelectQueryBuilder<...>), save(p: Product, em: EntityManager) 같은 시그니처가 보이면 그 포트는 이름만 포트인 레이어드입니다. 어댑터를 교체하는 순간 EntityManager를 흉내 낼 방법이 없으니까요. 판별법은 간단합니다 — InMemory 어댑터가 20줄로 안 나오면 포트가 새고 있는 것입니다.

경계 — 이 원칙이 손해가 되는 곳

여기까지 읽으면 모든 모듈을 헥사고날로 바꾸고 싶어지는데, 이 원칙이 성립하지 않는 경계가 분명히 있습니다. 앞의 논증을 다시 보면, 역전의 이득은 "격리해서 지킬 규칙이 있다"는 전제에서 나옵니다. 전제가 무너지는 곳에서는 비용만 남습니다.

경계 사례 1 — 규칙 없는 CRUD. 관리자 화면의 공지사항처럼 도메인 규칙이 사실상 없는 모듈에 포트·매퍼·도메인 모델을 도입하면, 같은 필드 목록을 엔티티·도메인·DTO에 세 번 쓰는 의식(儀式)이 됩니다. 지킬 규칙이 없으니 격리의 이득은 0이고, 파일 세 개와 매핑 코드라는 비용만 확정됩니다. 이런 모듈은 컨트롤러—리포지토리 직결의 얇은 레이어드로 두는 것이 정직합니다. 아키텍처는 모듈 단위로 선택하는 것이지, 코드베이스 전체에 일괄 적용하는 종교가 아닙니다.

경계 사례 2 — 복잡한 조회. 목록 화면의 페이지네이션·정렬·검색 조건·조인이 늘어나기 시작하면, 이를 포트로 감싸려는 시도는 필연적으로 ORM의 쿼리 언어를 인터페이스로 재발명하는 방향으로 흘러갑니다(findByFilter(filter: ProductFilter)의 필드가 화면 요구마다 늘어나는 걸 보게 됩니다). 이때는 조회 유스케이스를 어댑터 가까이에 두고 쿼리 결과를 바로 화면 DTO로 반환하는 편이 낫습니다. 명령(쓰기) 경로는 헥사고날로 지키고, 조회 경로는 얇게 — 트랜잭션 경계도 마찬가지여서, 여러 포트를 한 트랜잭션으로 묶겠다고 Unit of Work 포트를 도입하는 순간 추상화가 다시 인프라를 흉내 내기 시작합니다. 그건 원칙의 실패가 아니라 적용 범위를 잘못 고른 것입니다.

정리 — 남겨둘 규칙

레이어드에서 헥사고날로 넘어가는 과정을 한 도메인으로 따라왔습니다. 남겨둘 것은 육각형 그림이 아니라 다음 규칙들입니다.

  • 도메인 파일의 import 목록이 곧 아키텍처 검사기입니다. 프레임워크·ORM import가 보이면 이미 오염이 시작된 것입니다.
  • 포트는 도메인이 소유하고, 도메인의 어휘로만 씁니다. 시그니처에 인프라 타입이 한 개라도 들어가면 이름만 포트입니다.
  • 런타임 호출이 아니라 소스코드 의존성을 뒤집는 것입니다. NestJS에서는 모듈의 provide 한 줄이 그 역전을 고정합니다.
  • 어댑터 교체의 첫 수요는 DB 교체가 아니라 테스트입니다. InMemory 어댑터가 20줄로 나오는지가 포트 품질의 리트머스지입니다.
  • 규칙 없는 CRUD와 복잡한 조회에는 적용하지 않습니다. 역전의 이득은 "격리해서 지킬 규칙"에서 나오므로, 규칙이 없는 곳에서는 비용만 남습니다.

이렇게 도메인을 안쪽으로 꺼내고 나면 다음 질문이 생깁니다 — 안쪽에서 반복되는 절차들은 무엇으로 접는가. 다음 글에서 정산 도메인을 예제로 Template Method와 Strategy를 따라갑니다.
