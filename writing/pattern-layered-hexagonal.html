아키텍처 패턴 · 레이어드에서 헥사고날로

2021년 5월• 레이어드 · 포트와 어댑터 · DIP

컨트롤러→서비스→리포지토리로 코드를 나누는 레이어드 아키텍처는 처음 몇 달 동안은 잘 굴러갔습니다. 그런데 어느 시점부터 도메인 규칙이 ORM 호출 틈새에 흩어져 있었고, 규칙 하나를 확인하는 테스트를 돌리는 데도 DB가 필요해져 있었습니다. SafeIn 백엔드를 NestJS로 혼자 설계하고 크롤링 파이프라인을 DDD로 다시 세우면서 비슷한 일을 두 번 겪었는데, 그때 무너지던 순서와 되돌리던 순서를 크롤링 도메인 하나로 이어서 적어 둡니다.

먼저 짚고 가기
레이어드 아키텍처Layered Architecture
코드를 책임별 계층으로 쌓고 계층 간 호출을 한 방향으로만 허용하는 아키텍처 스타일입니다. 이 글의 예에서는 컨트롤러(표현)→서비스(응용)→리포지토리(데이터 접근) 순이고, 각 계층은 자기 바로 아래 계층만 호출합니다. 다만 이 규칙이 정하는 것은 호출 방향뿐이라, 각 계층이 무엇에 의존하는지 — 화살표의 종점이 구체 클래스인지 인터페이스인지 — 는 규정하지 않습니다. 이 글은 규칙을 다 지키고도 구조가 무너지는 이유를 그 빈틈에서 찾습니다.
ORMObject-Relational Mapping
관계형 데이터베이스의 테이블·행을 객체와 대응시켜 두고, 객체에 대한 조작을 SQL로 번역해 실행해 주는 라이브러리 계층입니다. 글에 나오는 TypeORM은 클래스에 @Entity·@Column을 붙여 매핑을 선언하고, findOne({ where })·save() 같은 자체 API로 질의합니다. 매핑 자체가 문제가 되는 일은 없고, 그 API의 어휘가 호출하는 쪽 코드에 그대로 남는다는 점이 문제입니다. 어휘가 서비스 계층까지 올라오면 업무 문장이어야 할 규칙이 기술 문장으로 기록됩니다.
의존성 역전 원칙DIP
Robert C. Martin이 정리한 SOLID의 한 항목으로, 상위 수준 모듈이 하위 수준 모듈에 의존하지 말고 양쪽 다 추상에 의존해야 하며 추상이 세부에 맞추는 것이 아니라 세부가 추상에 맞춰야 한다는 원칙입니다. 구현 수단은 인터페이스이고, 그 인터페이스의 소유권을 호출하는 쪽에 두는 것이 핵심입니다. 뒤집히는 대상은 소스코드 의존성(import 방향)뿐이고 런타임 호출 순서는 그대로라, 어떤 구현을 쓸지 고르는 결정은 조립 지점(NestJS라면 모듈의 provide 한 줄)에만 남습니다.
헥사고날 아키텍처 (포트와 어댑터)Hexagonal / Ports & Adapters
Alistair Cockburn이 2005년에 정리한 아키텍처 스타일이고, 포트와 어댑터라는 이름으로도 불립니다. 애플리케이션이 바깥세상과 주고받는 상호작용을 애플리케이션 쪽이 소유한 인터페이스, 곧 포트로 선언하고, 포트마다 기술별 어댑터를 붙여 UI든 DB든 테스트용 구현이든 대칭적인 바깥 자리에 놓습니다. 포트는 바깥이 애플리케이션을 부르는 구동 포트(primary)와 애플리케이션이 바깥을 부르는 피구동 포트(secondary)로 나뉘며, 이 글의 ProductRepositoryPort는 후자입니다. 육각형이라는 도형에는 의미가 없어 변의 수와 포트 개수는 무관하고, 남는 것은 계약의 소유권이 안쪽에 있다는 한 가지입니다.
유스케이스Use Case
Ivar Jacobson이 요구사항 기술 기법으로 도입한 용어로, 시스템이 외부 요청 하나를 받아 끝까지 완결하는 업무 절차 단위를 뜻합니다. 코드에서는 응용 계층의 클래스 하나로 옮겨져, 도메인 객체가 가진 규칙을 어떤 순서로 실행할지 지휘하고 조회·저장 같은 바깥일은 포트를 통해서만 호출합니다. 그래서 이 글의 SaveCrawledProduct는 생성자로 받는 것이 포트 하나뿐이고, DB도 DI 컨테이너도 없이 new 두 번으로 테스트에 세울 수 있습니다.

무너지는 순서는 항상 같다

예제로 삼을 도메인은 크롤링 파이프라인의 한 조각입니다. 수집기가 상품 페이지를 긁어 오면 서버는 두 가지 규칙으로 저장합니다. 규칙 1 — 같은 수집원의 같은 상품이면 새로 만들지 않고 갱신한다. 규칙 2 — 가격이 달라졌을 때만 가격 변동 이력을 남긴다. 레이어드로 쓰던 시절의 서비스는 대략 이런 모양이었습니다.

// product.service.ts — 레이어드의 "모범 답안"처럼 보이는 코드
@Injectable()
export class ProductService {
  constructor(
    @InjectRepository(ProductEntity)
    private readonly repo: Repository<ProductEntity>,
  ) {}

  async saveCrawled(dto: CrawledProductDto) {
    // 규칙 1: 같은 수집원·같은 키면 갱신
    const found = await this.repo.findOne({
      where: { source: dto.source, sourceKey: dto.sourceKey },
    });
    if (!found) {
      return this.repo.save(this.repo.create({ ...dto, priceHistory: [] }));
    }
    // 규칙 2: 가격이 달라졌을 때만 이력을 남긴다
    if (found.price !== dto.price) {
      found.priceHistory.push({ price: found.price, at: new Date() });
      found.price = dto.price;
    }
    return this.repo.save(found);
  }
}

이 코드는 리뷰를 무리 없이 통과했습니다. 계층도 지켰고 중복도 없고 동작도 맞았으니까요. 다만 제가 겪은 코드베이스들에서 여기서부터 흐트러지는 순서는 대체로 비슷했습니다.

1단계로 서비스가 ORM의 언어로 말하기 시작했습니다. findOne({ where }), create, save는 전부 TypeORM의 어휘입니다. 규칙 1은 "같은 상품이면 갱신"이라는 업무 문장이 아니라 "findOne이 null이면 create"라는 기술 문장으로 기록돼 있었습니다. 2단계로 도메인 규칙이 쿼리 속으로 숨었습니다. 목록 화면이 생기면 "판매 중인 상품"의 정의가 QueryBuilder의 where 절 안으로 들어가고, 같은 정의가 화면마다 조금씩 다르게 복제됩니다. 3단계로 엔티티가 모든 계층을 관통했습니다. @Column이 붙은 ProductEntity가 컨트롤러 응답으로도 서비스 인자로도 쓰이면서 DB 스키마 변경이 곧 API 변경이 됐습니다. 4단계에서는 테스트가 DB를 요구했습니다. "가격이 같으면 이력을 안 남긴다"라는 규칙 하나를 확인하는 데 테스트 컨테이너를 띄우고 있었습니다. 여기까지 오고 나서야 서비스 계층은 있는데 도메인은 어디에도 없다는 걸 알아차렸습니다.

문제는 계층이 아니라 화살표다

처음에는 계층을 덜 지켜서 이렇게 됐다고 생각했습니다. 그런데 위 코드는 레이어드의 규칙 — 위는 아래만 호출한다 — 을 어기지 않고 있었습니다. 문제는 그 규칙이 다루지 않는 쪽, 그러니까 화살표의 종점이 구체 타입이라는 데 있었습니다. 서비스의 import 목록에 typeorm이 있는 한 도메인 규칙은 TypeORM의 어휘로 표현될 수밖에 없습니다. 무언가에 의존한다는 건 그 대상의 언어로 말하게 된다는 뜻이기도 하니까요.

의존 방향을 뒤집어야 한다는 말은 자주 들었지만 왜 그런지는 한동안 제 말로 설명하지 못했습니다. 정리해 보니 전제가 두 가지였습니다. 하나는 의존이 변경을 전파하는 통로라는 점입니다. A가 B에 의존하면 B의 변경은 A를 흔들 수 있지만 그 반대는 성립하지 않습니다. 다른 하나는 도메인 규칙과 저장 기술이 변경되는 이유가 서로 다르다는 점입니다. "가격 이력을 언제 남길 것인가"는 업무가 바꾸고, "PostgreSQL이냐 어떤 ORM이냐"는 기술 사정이 바꿉니다. 이유가 다르면 바뀌는 시점도 다르고, 시점이 다른 두 코드가 의존으로 묶여 있으면 한쪽이 바뀔 때마다 다른 쪽까지 재검증 대상이 됩니다. 그래서 화살표는 더 안정적이어야 하는 쪽, 즉 도메인을 향하는 편이 낫습니다. 의존성 역전 원칙(DIP)이 이 상황에서 성립하는 이유를 저는 이렇게 이해했습니다. 원칙이라서가 아니라 변경 전파 비용의 부등식이 그쪽을 가리키기 때문입니다.

한동안 헷갈렸던 것이 하나 있어서 따로 적어 둡니다. 런타임 호출은 뒤집히지 않습니다. 실행 시점에는 어차피 유스케이스가 DB를 부릅니다. 뒤집을 수 있는 것은 소스코드 의존성뿐이고, 그 도구가 인터페이스였습니다. 헥사고날 아키텍처(포트와 어댑터)는 이 역전을 시스템 전체의 규칙으로 올려놓은 구조입니다. 안쪽에는 도메인과 도메인이 소유한 계약(포트)이 있고, 바깥의 어댑터들이 그 계약을 이행합니다. 육각형 그림 자체는 크게 중요하지 않았고, 계약의 소유권이 안쪽에 있다는 한 문장으로 요약됐습니다.

도메인과 포트 — 계약의 소유권을 뒤집는다

되돌리는 작업은 규칙을 프레임워크가 닿지 않는 곳으로 꺼내는 일부터 시작했습니다. @Column도 @Injectable도 없는 순수 TypeScript 클래스로 옮겼습니다.

// domain/product.ts — 이 파일의 import 목록은 거의 비어 있다
export type PriceChange = { price: number; at: Date };

export class Product {
  constructor(
    readonly source: string,
    readonly sourceKey: string,
    private _price: number,
    private readonly _history: PriceChange[] = [],
  ) {}

  get price() { return this._price; }
  get history() { return [...this._history]; }

  // 규칙 2가 if문이 아니라 "이름 있는 메서드"가 됐다
  applyCrawledPrice(next: number, now = new Date()): boolean {
    if (next === this._price) return false;
    this._history.push({ price: this._price, at: now });
    this._price = next;
    return true;
  }
}

그다음은 포트였습니다. 저장이 필요하다는 사실 자체는 도메인의 사정이라, 인터페이스도 도메인이 쓰는 어휘로 선언했습니다. 시그니처에는 찾는다와 저장한다만 남고 where나 QueryBuilder는 들어가지 않습니다. 유스케이스는 이 포트에만 의존하게 됩니다.

// domain/product.repository.port.ts — 계약의 소유자는 도메인
export interface ProductRepositoryPort {
  findBySourceKey(source: string, key: string): Promise<Product | null>;
  save(product: Product): Promise<void>;
}
export const PRODUCT_REPOSITORY = Symbol('PRODUCT_REPOSITORY');

// application/save-crawled-product.usecase.ts
export type CrawledProduct = { source: string; sourceKey: string; price: number };

@Injectable()
export class SaveCrawledProduct {
  constructor(
    @Inject(PRODUCT_REPOSITORY)
    private readonly products: ProductRepositoryPort,
  ) {}

  async execute(cmd: CrawledProduct): Promise<void> {
    const found = await this.products.findBySourceKey(cmd.source, cmd.sourceKey);
    const product = found ?? new Product(cmd.source, cmd.sourceKey, cmd.price);
    if (found) product.applyCrawledPrice(cmd.price);
    await this.products.save(product);
  }
}

같은 두 규칙인데 코드의 성격이 달라졌습니다. 규칙 1은 유스케이스의 흐름으로, 규칙 2는 applyCrawledPrice라는 이름으로 자리를 잡았고, 어느 쪽에도 ORM의 흔적이 남지 않았습니다. 레거시 코드를 UseCase 단위로 개선할 때도 순서는 같았습니다. 규칙을 먼저 꺼내고, 그 규칙이 필요로 하는 바깥세상을 포트로 선언했습니다.

*

실무 팁: TypeScript의 interface는 컴파일하면 사라지므로 NestJS가 주입 시점에 붙잡을 토큰이 없습니다. 그래서 포트 파일에서 Symbol 토큰을 함께 export하고 @Inject(PRODUCT_REPOSITORY)로 받았습니다. 포트 파일 자체는 domain(또는 application) 폴더에 두었습니다. 그래야 계약의 소유권이 안쪽에 있다는 원칙을 폴더 구조가 대신 지켜 줍니다.

어댑터 — 세부는 바깥에서 계약을 이행한다

이 단계에서 TypeORM은 없어지지 않고 제자리로 갑니다. 어댑터가 포트를 구현하면서 ORM의 행(row)과 도메인 모델 사이의 번역을 전담합니다. 지저분한 코드가 사라지는 것은 아니고, 한 파일 안에 모이는 셈입니다.

// infrastructure/typeorm-product.repository.ts — 계약의 "이행자"
@Injectable()
export class TypeOrmProductRepository implements ProductRepositoryPort {
  constructor(
    @InjectRepository(ProductEntity)
    private readonly repo: Repository<ProductEntity>,
  ) {}

  async findBySourceKey(source: string, key: string) {
    const row = await this.repo.findOne({ where: { source, sourceKey: key } });
    return row ? ProductMapper.toDomain(row) : null;
  }

  async save(product: Product) {
    await this.repo.save(ProductMapper.toRow(product));
  }
}

// product.module.ts — 화살표가 뒤집히는 유일한 지점
providers: [
  SaveCrawledProduct,
  { provide: PRODUCT_REPOSITORY, useClass: TypeOrmProductRepository },
],

화살표가 실제로 뒤집히는 지점은 모듈의 provide 한 줄입니다. 소스코드 상에서 유스케이스는 포트(추상)를 바라보고 어댑터도 포트를 바라보니, 두 화살표가 모두 안쪽을 향하게 됩니다. 둘을 실제로 연결하는 결정은 조립 지점인 모듈에만 남습니다. NestJS의 DI 컨테이너가 원래 이 역전을 위한 도구인데, 레이어드로 쓰던 동안 저는 그 능력을 구체 클래스를 편하게 주입하는 데만 쓰고 있었습니다.

교체 시연 — 첫 번째 수혜자는 테스트다

어댑터를 갈아 끼운다고 하면 저는 DB 교체부터 떠올렸는데, 실제로 가장 먼저 그리고 가장 자주 갈아 끼운 것은 테스트용 어댑터였습니다. 포트가 제대로 잡혀 있으면 두 번째 구현체는 20줄 정도로 끝납니다.

// test/in-memory-product.repository.ts — 두 번째 어댑터
class InMemoryProductRepository implements ProductRepositoryPort {
  private rows = new Map<string, Product>();
  async findBySourceKey(s: string, k: string) { return this.rows.get(`${s}:${k}`) ?? null; }
  async save(p: Product) { this.rows.set(`${p.source}:${p.sourceKey}`, p); }
}

// DB도, NestJS 컨테이너도 없이 도메인 규칙을 검증한다
it('가격이 같으면 이력을 남기지 않는다', async () => {
  const repo = new InMemoryProductRepository();
  const usecase = new SaveCrawledProduct(repo);

  await usecase.execute({ source: 'shop-a', sourceKey: '42', price: 12900 });
  await usecase.execute({ source: 'shop-a', sourceKey: '42', price: 12900 });

  const saved = await repo.findBySourceKey('shop-a', '42');
  expect(saved!.history).toHaveLength(0);
});

앞의 레이어드 서비스에서는 같은 테스트를 테스트 DB 없이 돌릴 수 없었는데, 지금은 new 두 번이면 됩니다. 이 구조가 실제로 도움이 됐던 곳은 크롤링 파이프라인이었습니다. 저장소만이 아니라 수집원 자체를 포트로 두면 — "페이지를 가져와 상품으로 해석한다"라는 계약입니다 — 수집원마다 HTML 구조가 제각각이어도 수집원별 어댑터가 각자 계약을 이행할 뿐, 파이프라인의 도메인 코드는 한 줄도 바뀌지 않습니다. 새 수집원을 붙이는 일이 어댑터 클래스 하나와 모듈 등록 한 줄로 끝나는 걸 보고 나서는, 그 뒤로 이 구조를 계속 쓰게 됐습니다.

!

함정: 포트 시그니처에 인프라 타입이 스며들면 격리가 의미를 잃습니다. findWith(qb: SelectQueryBuilder<...>), save(p: Product, em: EntityManager) 같은 시그니처가 보이면 이름만 포트이고 실제로는 레이어드입니다. 어댑터를 교체하는 순간 EntityManager를 흉내 낼 방법이 없으니까요. 저는 InMemory 어댑터가 20줄로 나오는지로 확인해 왔습니다. 그보다 길어지면 포트가 새고 있다고 봤습니다.

경계 — 이 원칙이 손해가 되는 곳

이쯤 되면 모든 모듈을 헥사고날로 바꾸고 싶어지는데, 저도 한동안 그랬습니다. 다만 앞의 정리를 다시 보면 역전의 이득은 격리해서 지킬 규칙이 있다는 전제에서 나옵니다. 그 전제가 성립하지 않는 곳에서는 비용만 남았습니다.

경계 사례 1 — 규칙 없는 CRUD. 관리자 화면의 공지사항처럼 도메인 규칙이 사실상 없는 모듈에 포트와 매퍼, 도메인 모델을 도입하면 같은 필드 목록을 엔티티·도메인·DTO에 세 번 쓰는 의식(儀式)이 됩니다. 지킬 규칙이 없으니 격리의 이득은 0이고, 파일 세 개와 매핑 코드라는 비용만 남습니다. 이런 모듈은 컨트롤러—리포지토리 직결의 얇은 레이어드로 두는 편이 나았습니다. 아키텍처는 코드베이스 전체에 일괄 적용하기보다 모듈 단위로 고르는 것이라고 정리해 두었습니다.

경계 사례 2 — 복잡한 조회. 목록 화면의 페이지네이션·정렬·검색 조건·조인이 늘어나기 시작하면, 이를 포트로 감싸려는 시도는 필연적으로 ORM의 쿼리 언어를 인터페이스로 재발명하는 방향으로 흘러갑니다(findByFilter(filter: ProductFilter)의 필드가 화면 요구마다 늘어나는 걸 보게 됩니다). 이때는 조회 유스케이스를 어댑터 가까이에 두고 쿼리 결과를 바로 화면 DTO로 반환하는 편이 나았습니다. 명령(쓰기) 경로는 헥사고날로 지키고 조회 경로는 얇게 두는 식입니다. 트랜잭션 경계도 비슷해서, 여러 포트를 한 트랜잭션으로 묶겠다고 Unit of Work 포트를 도입하면 추상화가 다시 인프라를 흉내 내기 시작합니다. 이런 경우는 원칙이 틀렸다기보다 적용 범위를 잘못 고른 쪽에 가깝다고 봤습니다.

정리 — 남겨둘 규칙

레이어드에서 헥사고날로 넘어가는 과정을 한 도메인으로 따라와 봤습니다. 나중에 다시 볼 때 필요한 건 육각형 그림보다 아래 항목들이었습니다.

  • 도메인 파일의 import 목록이 가장 빠른 검사기였습니다. 프레임워크나 ORM import가 보이면 이미 섞이기 시작한 것입니다.
  • 포트는 도메인이 소유하고 도메인의 어휘로만 씁니다. 시그니처에 인프라 타입이 하나라도 들어가면 이름만 포트입니다.
  • 뒤집는 대상은 런타임 호출이 아니라 소스코드 의존성입니다. NestJS에서는 모듈의 provide 한 줄이 그 역전을 고정합니다.
  • 어댑터 교체가 처음 필요해지는 곳은 DB가 아니라 테스트였습니다. InMemory 어댑터가 20줄로 나오는지가 포트 품질을 가늠하는 기준이 됩니다.
  • 규칙 없는 CRUD와 복잡한 조회에는 적용하지 않았습니다. 역전의 이득은 격리해서 지킬 규칙에서 나오므로, 규칙이 없는 곳에서는 비용만 남습니다.

도메인을 안쪽으로 꺼내고 나니 다음 질문이 남았습니다. 안쪽에서 반복되는 절차들은 무엇으로 접을 것인가 하는 문제였습니다. 다음 글에서는 정산 도메인을 예제로 Template Method와 Strategy를 따라가 보려 합니다.
