Livepoll · 백엔드

데이터베이스 없이 상태를 버티기

2025년 6월• Node · Socket.IO · 상태 설계

실시간 투표를 만들면서 데이터베이스를 아예 켜지 않아 봤습니다. 저는 오랫동안 일단 스키마부터 그리는 사람이었습니다. 커머스 백오피스와 ERP를 만들다 보면 모든 데이터가 언젠가 조회되고 정산되고 감사받는다는 전제가 몸에 붙어서, 이번엔 그 전제를 한 번 빼 보고 싶었습니다. 영속성을 덜어내면 무엇이 단순해지고 무엇이 무너지는지, 그때 확인한 것들을 적어 둡니다.

먼저 짚고 가기
영속성Persistence
프로그램의 생애주기와 무관하게 데이터가 살아남게 하는 성질입니다. 값을 디스크 같은 비휘발성 매체에 기록해, 프로세스가 종료되거나 재시작해도 그대로 다시 읽을 수 있게 합니다. 속도나 정합성이 아니라 '프로세스가 죽어도 남는가'라는 한 축의 성질이며, 이 글은 그 축을 일부러 포기해 스키마·마이그레이션·트랜잭션 같은 부대 비용을 덜어냅니다.
인메모리In-memory
데이터를 디스크가 아니라 실행 중인 프로세스의 주소 공간(RAM)에만 두는 저장 전략입니다. Livepoll은 전체 데이터셋을 Map 하나에 담아 읽기·쓰기를 O(1)로 처리하고, 트랜잭션이나 커넥션 풀을 두지 않습니다. 대신 상태가 그 프로세스에 묶여 있어 재시작하면 사라지고, 프로세스를 둘 이상으로 늘리는 순간 상태를 프로세스가 소유한다는 전제가 깨집니다.
스냅샷과 델타Snapshot / Delta
상태 변화를 클라이언트에 전달하는 두 가지 방식입니다. 스냅샷은 매번 현재 상태 전체를 보내 클라이언트가 받은 값을 그대로 렌더하게 하고, 델타는 직전 상태와의 차이(diff)만 보내 클라이언트가 자기 사본 위에 그 변경을 다시 적용하게 합니다. 델타는 대역폭을 아끼는 대신 계산·재조립 비용과 서버·화면이 어긋날 여지를 함께 들이는데, 이 글은 payload가 수백 바이트(선택지 최대 8개)라 스냅샷을 택합니다.
룸Room
Socket.IO에서 여러 소켓 연결을 하나의 이름 아래 묶는 서버 측 그룹입니다. 소켓이 socket.join(id)로 들어가면 io.to(id).emit(...)이 그 묶음에게만 방송되고, 어댑터가 소속을 추적하므로 rooms.get(id).size가 곧 현재 인원 수가 됩니다. 클라이언트에는 없는 서버 측 개념이고, 기본 어댑터에서는 그 소속이 단일 프로세스 메모리에만 있어 프로세스를 넘기려면 Redis 어댑터가 필요합니다. Livepoll은 이 room 크기를 시청자 수로 그대로 씁니다.

어떤 데이터는 오래 살 필요가 없다

라이브 투표는 수명이 짧습니다. 발표장에서 손을 들게 하고 몇 분 뒤 결과를 보여주면 끝이고, 그 뒤로 "3주 전 그 투표의 12번째 표"를 다시 꺼낼 일은 거의 없습니다. 그런데 저는 이 프로젝트에서도 습관대로 테이블부터 그리고 있었습니다. 마이그레이션을 쓰고 인덱스를 고민하다가, 수명이 5분인 데이터에 영속성의 비용을 전부 치르는 중이라는 걸 알아챘습니다.

그래서 Livepoll의 전체 데이터셋은 Map 하나로 두었습니다. 투표를 만들면 키가 하나 생기고, 표가 들어오면 숫자가 하나 오르고, 시간이 지나면 통째로 사라집니다.

class PollStore {
  private polls = new Map<string, Poll>();
  // 6시간마다 오래된 투표를 쓸어낸다 — 그게 유일한 '삭제 정책'
  constructor() { setInterval(() => this.sweep(), 600_000).unref?.(); }
}

모든 변경이 O(1)이고, 트랜잭션도 커넥션 풀도 없습니다. 대신 재시작하면 전부 날아갑니다. 처음엔 이게 약점 같아서 마음에 걸렸는데, 만들다 보니 날아간다는 사실 자체가 이 설계의 계약 조건에 가깝다고 정리됐습니다. 영속성을 포기한 대가로 복잡성을 덜어낸 셈입니다.

"누가 보고 있나"를 직접 세지 않기

실시간 기능에서 은근히 귀찮은 게 접속자 수입니다. 연결과 해제를 추적하고, 중복을 거르고, 죽은 소켓을 청소해야 합니다. 저도 그 자료구조를 절반쯤 만들다가, Socket.IO가 이미 같은 일을 하고 있다는 걸 뒤늦게 알았습니다. 소켓이 특정 투표에 join하면 그 room의 크기가 곧 시청자 수였습니다.

const viewersOf = (pollId) =>
  io.sockets.adapter.rooms.get(pollId)?.size ?? 0;

그래서 presence를 위한 별도 자료구조는 남기지 않았습니다. 프레임워크가 이미 아는 값을 옆에서 다시 세면, 둘이 어긋나는 순간 어느 쪽이 맞는지 확인할 방법이 없어집니다.

델타가 아니라 스냅샷

상태가 바뀔 때마다 클라이언트에 무엇을 보낼지도 정해야 했습니다. 바뀐 부분만 보내는 델타(delta)가 정석처럼 느껴져서 한동안 그쪽을 들여다봤는데, 마지막에는 매번 전체 스냅샷을 보내는 쪽으로 갔습니다.

const broadcast = (pollId) => {
  const state = store.snapshot(pollId, viewersOf(pollId));
  if (state) io.to(pollId).emit("poll:state", state);
};

이유는 도메인의 크기였습니다. 선택지는 최대 8개, payload는 수백 바이트입니다. 이 규모에서는 델타를 계산하고 클라이언트에서 재조립하는 로직이 절약하는 대역폭보다 비싸 보였습니다. 지금은 클라이언트가 받은 스냅샷을 그대로 렌더하기만 해서, 서버와 화면의 상태가 어긋날 자리가 없습니다.

*

선택지가 수천 개거나 초당 수만 표가 들어오는 상황이었다면 이 결정은 틀렸을 겁니다. 설계는 도메인의 크기에 붙어 있어서, 이건 정답인 아키텍처라기보다 지금 이 크기에 맞춰 둔 선택에 가깝습니다.

정직한 정합성

표 무결성은 소켓 단위로 잡았습니다. 한 연결은 한 표고, 다른 항목을 누르면 표가 옮겨 가고, 연결이 끊기면 표를 회수합니다. 그래서 새 탭을 열면 새 표가 되는데, 이건 모르고 둔 게 아니라 알고도 남겨 둔 구멍입니다. 진지한 선거라면 세션과 핑거프린트가 필요하겠지만, 발표장에서 재미로 손을 드는 도구에는 이 정도가 정직한 선이라고 봤습니다. 무엇을 안 지킬지 정하는 것도 설계라는 걸 이때 알았습니다.

인메모리의 유효기간

다만 이 단순함에는 유효기간이 있습니다. 프로세스를 둘 이상으로 늘리면 상태를 프로세스가 소유한다는 전제가 바로 깨집니다. 그때는 Socket.IO의 Redis 어댑터로 방송을 공유하고, 표도 Redis로 옮겨야 합니다. 그래서 지금의 인메모리는 이 규모에서는 이걸로 충분하다는 판단이지, 계속 가져갈 답은 아니라고 생각하고 있습니다.

그래도 이 실험은 해 볼 만했습니다. 무조건 DB부터 놓던 습관을 한 번 의심해 봤고, 덜어낸 자리에서 코드가 얼마나 읽기 쉬워지는지도 봤습니다. 다음에 일하는 제품에서 영속성이 필요할 땐 왜 필요한지 조금 더 또렷하게 말할 수 있을 것 같습니다.
