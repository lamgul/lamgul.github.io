Livepoll · 백엔드

데이터베이스 없이 상태를 버티기

2025년 6월• 약 7분• Node · Socket.IO · 상태 설계

저는 오랫동안 "일단 스키마부터" 그리는 사람이었습니다. 커머스 백오피스와 ERP를 만들다 보면 그렇게 됩니다. 모든 것은 언젠가 조회되고, 정산되고, 감사받으니까요. 그래서 실시간 투표를 만들 땐 일부러 반대로 걸어봤습니다. 데이터베이스를 아예 켜지 않으면, 실시간 기능은 얼마나 단순해질까?

먼저 짚고 가기
영속성Persistence
데이터를 디스크에 저장해, 프로그램을 껐다 켜도 남아 있게 하는 성질입니다. 서랍에 서류를 넣어 두는 것과 같죠. 이 글은 바로 이 영속성을 일부러 빼면 무엇이 단순해지는지를 따라갑니다.
인메모리In-memory
데이터를 디스크가 아니라 실행 중인 프로그램의 메모리(RAM)에만 두는 방식입니다. 아주 빠르지만, 프로그램이 꺼지면 함께 사라집니다. Livepoll의 모든 상태가 이 메모리 위에만 있습니다.
스냅샷과 델타Snapshot / Delta
변화를 전달하는 두 방식입니다. 스냅샷은 "현재 전체 모습"을 통째로 보내고, 델타는 "바뀐 부분만" 보냅니다. 사진 한 장을 다시 찍어 보내느냐, 달라진 곳만 짚어 주느냐의 차이죠. 이 글은 일부러 스냅샷을 택합니다.
룸Room
Socket.IO에서 같은 채널로 묶인 접속자들의 묶음입니다. 단체 채팅방을 떠올리면 됩니다. 방에 몇 명이 들어와 있는지를 프레임워크가 이미 세고 있어서, 시청자 수를 따로 관리하지 않아도 됩니다.

어떤 데이터는 오래 살 필요가 없다

라이브 투표는 수명이 짧습니다. 발표장에서 손을 들게 하고, 몇 분 뒤 결과를 보여주면 끝입니다. 그 뒤로 "3주 전 그 투표의 12번째 표"를 다시 꺼낼 일은 거의 없습니다. 그런데도 우리는 습관적으로 이런 데이터를 디스크에 앉힙니다. 테이블을 만들고, 마이그레이션을 쓰고, 인덱스를 고민합니다. 수명이 5분인 데이터에 영속성의 비용을 전부 치르는 겁니다.

그래서 Livepoll의 전체 데이터셋은 Map 하나입니다. 투표를 만들면 키가 하나 생기고, 표가 들어오면 숫자가 하나 오르고, 시간이 지나면 통째로 사라집니다.

class PollStore {
  private polls = new Map<string, Poll>();
  // 6시간마다 오래된 투표를 쓸어낸다 — 그게 유일한 '삭제 정책'
  constructor() { setInterval(() => this.sweep(), 600_000).unref?.(); }
}

모든 변경이 O(1)이고, 트랜잭션도 커넥션 풀도 없습니다. 재시작하면 전부 날아갑니다. 이게 약점처럼 들리지만 — 그 날아감이 이 설계의 계약 조건입니다. 나는 영속성을 포기하는 대가로 복잡성을 통째로 덜어냈습니다.

"누가 보고 있나"를 직접 세지 않기

실시간 기능에서 은근히 귀찮은 게 접속자 수입니다. 연결·해제를 추적하고, 중복을 거르고, 죽은 소켓을 청소해야 합니다. 그런데 Socket.IO는 이미 그걸 하고 있었습니다. 소켓이 특정 투표에 join하면, 그 room의 크기가 곧 시청자 수입니다.

const viewersOf = (pollId) =>
  io.sockets.adapter.rooms.get(pollId)?.size ?? 0;

presence를 위한 별도 자료구조를 하나도 만들지 않았습니다. 프레임워크가 이미 아는 걸 다시 추적하는 건, 버그를 두 벌로 늘리는 일입니다.

델타가 아니라 스냅샷

상태가 바뀔 때마다 클라이언트에 무엇을 보낼지도 결정해야 했습니다. 바뀐 부분만 보내는 델타(delta)가 정석처럼 느껴지지만, 저는 매번 전체 스냅샷을 던집니다.

const broadcast = (pollId) => {
  const state = store.snapshot(pollId, viewersOf(pollId));
  if (state) io.to(pollId).emit("poll:state", state);
};

이유는 도메인의 크기입니다. 선택지는 최대 8개, payload는 수백 바이트입니다. 이 규모에서 델타를 계산하고 클라이언트에서 재조립하는 로직은 절약하는 대역폭보다 비싼 복잡성입니다. 클라이언트는 그냥 받은 스냅샷을 그대로 렌더합니다. 상태 동기화 버그가 생길 여지가 없습니다.

*

만약 선택지가 수천 개거나, 초당 수만 표가 들어왔다면 이 결정은 틀렸을 겁니다. 설계는 늘 도메인의 크기에 붙어 있습니다. "정답인 아키텍처"가 아니라 "이 크기에 맞는 아키텍처"가 있을 뿐입니다.

정직한 정합성

표 무결성은 소켓 단위로 잡았습니다. 한 연결은 한 표, 다른 항목을 누르면 표가 옮겨가고, 연결이 끊기면 표를 회수합니다. 새 탭을 열면 새 표가 되죠. 진지한 선거라면 세션과 핑거프린트가 필요하겠지만, 재미로 손을 드는 도구에는 이 정도가 정직한 선입니다. 무엇을 안 지킬지 정하는 것도 설계입니다.

인메모리의 유효기간

이 모든 단순함에는 만료일이 있습니다. 프로세스를 하나 이상으로 늘리는 순간, 상태를 "프로세스가 소유한다"는 전제가 깨집니다. 그때는 Socket.IO의 Redis 어댑터로 방송을 공유하고, 표도 Redis로 옮겨야 합니다. 즉 인메모리는 "지금은 이걸로 충분하다"는 선언이지, 영원한 답이 아닙니다.

그래도 저는 이 실험이 좋았습니다. "무조건 DB"라는 반사신경을 한 번 의심해봤고, 덜어낸 자리에서 코드가 얼마나 읽기 쉬워지는지를 봤으니까요. 다음에 진짜 제품에서 영속성이 필요할 땐, 적어도 왜 그게 필요한지 더 또렷하게 말할 수 있을 것 같습니다.
