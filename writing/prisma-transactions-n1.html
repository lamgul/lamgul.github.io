Prisma 트랜잭션과 N+1 — ORM이 숨기는 쿼리

2026년 2월• Prisma · NestJS · ORM

SafeIn 백엔드는 NestJS와 Prisma, PostgreSQL로 돌아갑니다. 코드에는 findMany 한 줄뿐인데 목록 API가 느린 적이 있었습니다. 추측만으로는 어디를 손대야 할지 알 수 없었고, ORM이 SQL을 대신 써 주는 만큼 실제로 나가는 쿼리도 눈에 보이지 않았습니다. 그래서 로그를 켜는 일부터 다시 했습니다. Prisma가 실제로 무엇을 실행하는지 확인하는 법과 N+1이 생기는 지점, 그걸 잡는 세 가지 방법을 비교해 적어 둡니다.

먼저 짚고 가기
ORMObject-Relational Mapping
객체 모델과 관계형 테이블의 대응 관계를 선언해 두면, 그 선언을 근거로 SQL 생성과 결과 매핑을 대신 수행해 주는 라이브러리 계층입니다. Prisma는 schema.prisma의 모델 정의에서 타입이 붙은 클라이언트를 생성하고, findMany 같은 호출을 쿼리 엔진이 SQL로 번역해 실행한 뒤 돌아온 행을 다시 객체로 되돌립니다. 대신해 주는 것은 SQL 작성이지 실행 계획이 아니라서, 호출 한 줄이 쿼리 몇 개로 풀리는지는 API 형태만 봐서는 드러나지 않습니다.
N+1 문제N+1 problem
연관 데이터를 가져오는 방식에서 생기는 접근 패턴으로, 부모 목록을 1번 조회한 뒤 결과 행 N개마다 자식 조회를 1번씩 더 보내 총 1+N번의 DB 왕복이 발생하는 상태를 가리킵니다. 개별 쿼리는 외래 키 인덱스를 타서 각각 빠를 수 있지만, 왕복마다 붙는 네트워크 지연과 커넥션 점유가 N에 비례해 누적됩니다. 비용의 원인이 쿼리 하나의 성능이 아니라 왕복 횟수인 만큼 인덱스를 더 얹어도 사라지지 않고, N번을 IN 한 번이나 조인으로 묶어야만 없어집니다. 병렬로 던지는 것 역시 해결이 아닌데, 총 왕복 횟수는 그대로이기 때문입니다.
트랜잭션transaction
여러 문(statement)을 하나로 묶어 원자성·일관성·격리성·지속성(ACID)을 보장받는 DBMS의 실행 단위입니다. BEGIN으로 열고 COMMIT으로 닫을 때까지의 변경은 다른 세션에 보이지 않다가 커밋 순간 한꺼번에 반영되고, 도중에 오류가 나거나 ROLLBACK하면 전부 없던 일이 됩니다. Prisma에서는 $transaction에 넘긴 배열이나 콜백이 이 범위에 해당하며, 콜백 폼은 그동안 커넥션 하나를 계속 붙잡고 있습니다. 되돌릴 수 있는 것은 DB 안의 변경뿐이라, 트랜잭션 안에서 보낸 푸시 알림이나 외부 API 호출은 롤백해도 취소되지 않습니다.
격리수준isolation level
동시에 실행되는 트랜잭션들이 서로의 중간 상태를 어디까지 볼 수 있는지 규정한 SQL 표준의 설정값으로, READ UNCOMMITTED · READ COMMITTED · REPEATABLE READ · SERIALIZABLE 네 단계가 정의돼 있습니다. 단계를 올릴수록 dirty read, non-repeatable read, phantom read 순으로 허용되는 이상 현상이 줄어듭니다. PostgreSQL의 기본값은 READ COMMITTED이고, 읽기를 MVCC 스냅샷으로 처리하는 구조라 READ UNCOMMITTED를 요청해도 READ COMMITTED와 동일하게 동작합니다. 올린 단계가 공짜는 아니어서, SERIALIZABLE은 읽기를 막아 세우는 대신 직렬화 순서가 깨진 것을 감지한 시점에 트랜잭션 한쪽을 실패시키므로 재시도는 애플리케이션 몫으로 남습니다.
커넥션 풀connection pool
DB 연결을 미리 열어 두고 요청들이 빌려 쓰고 반납하도록 관리하는 자원 풀입니다. PostgreSQL은 연결 하나마다 서버 쪽에 별도 프로세스가 붙어 매번 새로 여는 비용이 크기 때문에, 열어 둔 것을 재사용하는 이 구조를 씁니다. Prisma Client는 기본적으로 물리 코어 수 × 2 + 1개를 유지하고, 풀이 비면 요청은 대기 상태로 들어갔다가 한도(pool_timeout, 트랜잭션이라면 maxWait)를 넘기는 순간 에러가 됩니다. 풀 크기가 곧 동시에 실행될 수 있는 쿼리 수의 상한이라, Promise.all로 N개를 한꺼번에 던져도 실제로는 풀 크기만큼씩 나뉘어 처리됩니다.

1. 로그부터 켠다 — 추측 금지

손댈 곳을 찾으려면 쿼리 로그부터 봐야 했습니다. Prisma는 log 옵션으로 실행되는 SQL을 이벤트로 뽑아 줍니다. NestJS라면 PrismaService에서 한 번만 연결해 두면 이후로는 따로 신경 쓸 일이 없습니다.

// prisma.service.ts — 쿼리를 이벤트로 받아 NestJS 로거로 흘린다
@Injectable()
export class PrismaService
  extends PrismaClient<Prisma.PrismaClientOptions, 'query'>
  implements OnModuleInit {

  private readonly logger = new Logger('SQL');

  constructor() {
    super({ log: [{ emit: 'event', level: 'query' }] });
  }

  async onModuleInit() {
    this.$on('query', (e) => {
      this.logger.debug(`${e.duration}ms ${e.query} -- ${e.params}`);
    });
    await this.$connect();
  }
}

이벤트 객체에는 실행된 SQL(e.query), 바인딩된 파라미터(e.params), 소요 시간(e.duration, ms)이 들어 있습니다. 개발 환경에서 이걸 켜 두니 "요청 하나에 쿼리 몇 개가 나가는가"가 매 요청마다 눈에 보였습니다. 느린 API를 열어 보기 전에 쿼리 개수부터 세는 습관이 생긴 것도 이때부터입니다.

2. include는 N+1이 아닙니다

저도 처음에는 include를 쓰면 관계 개수만큼 쿼리가 나가는 줄 알았는데, Prisma의 기본 전략은 그렇지 않았습니다. 부모를 한 번 조회하고, 자식은 부모 id들을 모아 IN 쿼리 한 번으로 가져온 뒤 애플리케이션 메모리에서 조립합니다.

// TBM 목록 + 참가자 — 쿼리가 몇 개 나갈까?
const tbms = await prisma.tbm.findMany({
  where: { siteId },
  include: { participants: true },
  take: 20,
});

-- 로그에 찍힌 것: 단 2개 (요약)
SELECT "id", "title", "state" FROM "Tbm"
  WHERE "siteId" = $1 LIMIT $2
SELECT "tbmId", "employeeNameAtAssign", "state" FROM "TbmParticipant"
  WHERE "tbmId" IN ($1,$2, ... ,$20)

부모 20건에 자식이 몇백 건이어도 쿼리는 2개였습니다. include 자체가 N+1을 만들지는 않는 셈입니다. 다만 공짜는 아니어서, 자식 테이블의 컬럼을 전부 끌고 오는 탓에 관계가 깊어지고 include가 중첩될수록 전송량과 메모리에서의 조립 비용이 커집니다. 목록 화면이라면 select로 필요한 필드만 좁히는 편이 먼저였습니다.

3. 진짜 N+1은 반복문에서 태어난다

로그에서 실제로 만난 N+1은 ORM이 만든 것이 아니라 제가 쓴 반복문이었습니다. "목록에 집계 하나만 붙이자"고 시작한 코드가 대체로 그랬습니다.

// 안티패턴 — TBM마다 확인 인원을 세러 간다
const tbms = await prisma.tbm.findMany({ where: { siteId } });
const rows = await Promise.all(
  tbms.map(async (t) => ({
    ...t,
    confirmed: await prisma.tbmParticipant.count({
      where: { tbmId: t.id, state: 'CONFIRMED' },
    }),
  })),
);  // 쿼리 1 + N개. 목록이 100건이면 101개.

// 처방 — groupBy 한 번으로 같은 답을 얻는다
const counts = await prisma.tbmParticipant.groupBy({
  by: ['tbmId'],
  where: { tbmId: { in: tbms.map((t) => t.id) }, state: 'CONFIRMED' },
  _count: true,
});  // 쿼리 2개로 끝. 이후 Map으로 붙이면 된다.

Promise.all로 감싸면 병렬이라 빨라 보이지만, 커넥션 풀을 N개가 나눠 쓰며 DB에 왕복 N번을 하는 것은 그대로였습니다. 부하가 없는 개발 DB에서는 티가 나지 않다가, 운영에서 목록이 길어지면 응답 시간이 목록 길이에 비례해 늘어납니다. 로그에 같은 모양의 쿼리가 파라미터만 바뀌며 줄줄이 찍히는 것이 제가 N+1을 알아보는 표시가 됐습니다.

참고로 GraphQL 리졸버처럼 항목 단위 조회가 구조적으로 강제되는 곳에서는, Prisma의 fluent API(prisma.tbm.findUnique({ where: { id } }).participants())가 같은 틱에 발생한 findUnique들을 내부 dataloader로 묶어 IN 쿼리 하나로 배치해 줍니다. 반복문에서 생긴 N+1과 달리 이 경로는 처음부터 그렇게 설계된 것이라 그대로 써도 됩니다.

4. 처방 비교 — IN 배치, DB 조인, 수동 SQL

N+1을 없애는 방법은 제가 써 본 범위에서 세 갈래였습니다. 각각 코드부터 보겠습니다.

// (a) 기본 전략 — 쿼리 분리 + IN 배치 (2절의 include가 이것)

// (b) DB 조인 — preview 기능 relationJoins를 켜면 선택 가능
const tbms = await prisma.tbm.findMany({
  relationLoadStrategy: 'join',   // PostgreSQL은 LATERAL JOIN + JSON 집계
  where: { siteId },
  include: { participants: true },
});

// (c) 수동 SQL — 조인과 집계를 한 방에. 태그드 템플릿이라 파라미터는 안전하다
const rows = await prisma.$queryRaw`
  SELECT t."id", t."title",
         COUNT(*) FILTER (WHERE p."state" = 'CONFIRMED') AS "confirmed",
         COUNT(p."tbmId")                                AS "total"
  FROM "Tbm" t
  LEFT JOIN "TbmParticipant" p ON p."tbmId" = t."id"
  WHERE t."siteId" = ${siteId}
  GROUP BY t."id"`;

고를 때 보는 기준은 이렇게 정리해 두었습니다.

(a) IN 배치는 기본값이고 대부분의 화면에는 이걸로 충분했습니다. 쿼리가 단순해 인덱스를 태우기 쉽고, 관계별로 나뉘어 있어 로그를 읽기도 쉽습니다. 약점이라면 왕복이 관계 수만큼 늘고 조립을 애플리케이션이 떠안는다는 점입니다.

(b) relationLoadStrategy: 'join'은 왕복을 한 번으로 줄입니다. 다만 이 글 시점 기준 preview 기능이라 schema.prisma의 previewFeatures = ["relationJoins"]를 켜야 하고, 생성되는 SQL이 JSON 집계를 낀 조인이라 플래너가 어떤 계획을 세우는지는 직접 확인해야 했습니다. 조인이 항상 빠르지도 않아서, 부모가 적고 자식이 아주 많은 분포에서는 (a)가 나은 경우도 있었습니다.

(c) $queryRaw는 집계·윈도우 함수·조건부 카운트처럼 ORM 문법 바깥의 일이 필요할 때 꺼냈습니다. 위처럼 "목록 + 확인 인원 + 전체 인원"을 쿼리 하나로 끝낼 수 있습니다. 대신 타입 안전이 약해지고(결과 타입을 직접 선언해야 합니다) 스키마를 바꿔도 컴파일러가 잡아 주지 못합니다. 그래서 raw SQL은 리포지토리 한 곳에 모아 격리해 두었습니다.

*

어느 쪽을 고르든 같은 데이터 분포로 로그를 떠서 비교한 다음에 정했습니다. 1절의 e.duration과 PostgreSQL의 EXPLAIN ANALYZE면 충분했습니다. "조인이 빠르다더라"는 이야기는 남의 데이터 분포에서 나온 결론일 수 있습니다.

5. $transaction — 배열, 인터랙티브, 격리수준

쿼리 개수를 줄이는 것과 별개로, 쿼리를 한 단위로 묶는 문제가 남습니다. Prisma의 $transaction은 두 가지 형태입니다. 배열 폼은 준비된 쿼리들을 순서대로 한 트랜잭션에서 실행하는데, 앞 결과를 읽어 다음 쿼리를 결정할 필요가 없다면 이쪽이 가볍습니다. 읽고-판단하고-쓰는 흐름이 필요하면 인터랙티브 폼을 씁니다.

// TBM 발행 — 상태 전이 + 참가자 스냅샷 + 번역 작업 적재를 한 단위로
const tbm = await prisma.$transaction(
  async (tx) => {
    const tbm = await tx.tbm.update({
      where: { id, state: 'DRAFT' },      // 상태 조건을 where에 — 이중 발행 방지
      data: { state: 'PUBLISHED', publishedAt: new Date() },
    });
    await tx.tbmParticipant.createMany({ data: snapshots });
    await tx.translationJob.createMany({ data: pendingJobs });
    return tbm;
  },
  {
    isolationLevel: Prisma.TransactionIsolationLevel.Serializable,
    maxWait: 2000,   // 커넥션 확보 대기 한도 (기본 2초)
    timeout: 5000,   // 트랜잭션 전체 시간 한도 (기본 5초)
  },
);

제가 오래 눈여겨보지 않았던 것이 세 번째 인자였습니다. 격리수준(isolationLevel)을 지정하지 않으면 DB의 기본값을 따르는데, PostgreSQL이라면 READ COMMITTED입니다. 대부분의 CRUD에는 그걸로 충분해서, 위처럼 "읽은 상태를 전제로 쓰는" 경합 구간에만 올려 잡고 있습니다. Serializable은 동시 발행 같은 경합을 걸러 주는 대신 충돌 시 트랜잭션이 P2034(write conflict/deadlock) 에러로 실패할 수 있어서, 격리수준을 올렸다면 재시도 로직까지 붙여야 마무리가 됩니다.

이전 직장에서 정산 코드를 리팩토링할 때도 비슷한 생각을 했습니다. 돈이든 안전 기록이든 부분 성공은 실패보다 다루기 까다롭습니다. "발행은 됐는데 참가자 스냅샷이 없다"는 상태는 조회 화면의 버그처럼 보이지만, 실제로는 데이터 자체가 어긋난 것이기 때문입니다.

*

인터랙티브 트랜잭션 안에서는 외부 I/O를 하지 않는 편이 좋았습니다. FCM 푸시나 LLM 번역 호출 같은 것이 안에 들어오면 커넥션을 쥔 채 대기하다 timeout으로 통째로 롤백됩니다. 위 코드가 번역을 직접 호출하지 않고 translationJob 레코드만 적재하는 것도 그래서입니다. 외부 호출은 커밋 이후에 워커가 맡습니다.

6. 숨겨진 것을 꺼내 보는 습관

정리하면 순서는 이렇습니다. 쿼리 로그를 켜고, 요청당 쿼리 개수를 세고, 같은 모양의 쿼리가 반복되면 반복문을 의심합니다. 처방은 IN 배치 → DB 조인 → 수동 SQL 순으로 단순한 것부터 씁니다. 묶여야 하는 쓰기는 $transaction으로 묶되, 격리수준을 올렸다면 재시도를 붙이고 외부 I/O는 밖으로 뺍니다. ORM을 쓰지 말자는 이야기는 아닙니다. Prisma 덕분에 스키마와 타입이 하나로 움직이는 생산성은 SafeIn처럼 모델이 많은 제품에서 포기하기 어렵습니다. 다만 ORM이 숨겨 준 것을 언제든 꺼내 볼 수 있는 상태로 두는 편이 오래 쓰기에 낫다고, 지금은 생각하고 있습니다.
