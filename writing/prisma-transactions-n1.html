Prisma 트랜잭션과 N+1 — ORM이 숨기는 쿼리

2026년 2월• Prisma · NestJS · ORM

SafeIn 백엔드는 NestJS와 Prisma, PostgreSQL로 돌아갑니다. 코드에는 findMany 한 줄인데 목록 API가 느릴 때, 추측으로는 아무것도 못 고칩니다. ORM은 SQL을 숨기는 게 일이고, 우리는 그걸 다시 꺼내 보는 게 일입니다. 이 글은 Prisma가 실제로 무엇을 실행하는지를 로그로 확인하는 법, N+1이 진짜로 생기는 지점, 그리고 그걸 잡는 세 가지 처방을 비교합니다.

먼저 짚고 가기
ORMObject-Relational Mapping
프로그램 속 객체와 데이터베이스 테이블을 자동으로 이어 주는 번역기입니다. 덕분에 SQL을 직접 쓰지 않아도 되지만, 그만큼 어떤 SQL이 실제로 나가는지는 가려집니다. 이 글은 바로 그 가려진 SQL을 다시 꺼내 보는 이야기입니다.
N+1 문제N+1 problem
목록을 한 번 불러온 뒤(1), 목록 속 항목마다 추가 쿼리를 한 번씩(N) 더 날려 총 N+1번의 왕복이 생기는 성능 함정입니다. 목록이 길어질수록 느려지는 이 글의 핵심 문제입니다.
트랜잭션transaction
여러 DB 작업을 "전부 되거나 전부 안 되거나" 하나로 묶는 단위입니다. 중간까지만 성공해 데이터가 어긋나는 상황을 막아 줍니다.
격리수준isolation level
동시에 도는 트랜잭션들이 서로의 작업을 얼마나 볼 수 있는지 정하는 단계입니다. 높일수록 경합은 안전해지지만 충돌로 실패할 확률이 늘어 재시도가 필요해집니다.
커넥션 풀connection pool
DB 연결을 미리 몇 개 만들어 두고 돌려 쓰는 창고입니다. 개수가 한정돼 있어서, N+1처럼 왕복이 많아지면 이 한정된 연결을 나눠 쓰며 병목이 됩니다.

1. 로그부터 켠다 — 추측 금지

모든 이야기의 출발점은 쿼리 로그입니다. Prisma는 log 옵션으로 실행되는 SQL을 이벤트로 뽑아줄 수 있습니다. NestJS라면 PrismaService에서 한 번만 연결해 두면 됩니다.

// prisma.service.ts — 쿼리를 이벤트로 받아 NestJS 로거로 흘린다
@Injectable()
export class PrismaService
  extends PrismaClient<Prisma.PrismaClientOptions, 'query'>
  implements OnModuleInit {

  private readonly logger = new Logger('SQL');

  constructor() {
    super({ log: [{ emit: 'event', level: 'query' }] });
  }

  async onModuleInit() {
    this.$on('query', (e) => {
      this.logger.debug(`${e.duration}ms ${e.query} -- ${e.params}`);
    });
    await this.$connect();
  }
}

이벤트 객체에는 실행된 SQL(e.query), 바인딩된 파라미터(e.params), 소요 시간(e.duration, ms)이 들어 있습니다. 개발 환경에서 이걸 켜 두면 "요청 하나에 쿼리 몇 개가 나가는가"가 매 요청마다 눈에 보입니다. 느린 API를 열어보기 전에, 먼저 쿼리 개수부터 세는 습관이 이 글의 전부라고 해도 됩니다.

2. include는 N+1이 아닙니다

흔한 오해부터 정리합니다. include를 쓰면 관계 개수만큼 쿼리가 나간다고 생각하기 쉽지만, Prisma의 기본 전략은 그렇지 않습니다. 부모를 한 번 조회하고, 자식은 부모 id들을 모아 IN 쿼리 한 번으로 가져온 뒤 애플리케이션 메모리에서 조립합니다.

// TBM 목록 + 참가자 — 쿼리가 몇 개 나갈까?
const tbms = await prisma.tbm.findMany({
  where: { siteId },
  include: { participants: true },
  take: 20,
});

-- 로그에 찍힌 것: 단 2개 (요약)
SELECT "id", "title", "state" FROM "Tbm"
  WHERE "siteId" = $1 LIMIT $2
SELECT "tbmId", "employeeNameAtAssign", "state" FROM "TbmParticipant"
  WHERE "tbmId" IN ($1,$2, ... ,$20)

부모 20건에 자식이 몇백 건이어도 쿼리는 2개입니다. 즉 include 자체는 무죄입니다. 다만 공짜는 아닙니다 — 자식 테이블의 컬럼을 전부 끌고 오므로, 관계가 깊어지고 include가 중첩될수록 전송량과 메모리에서의 조립 비용이 커집니다. 목록 화면이라면 select로 필요한 필드만 좁히는 것이 먼저입니다.

3. 진짜 N+1은 반복문에서 태어난다

제가 로그에서 실제로 만나는 N+1은 ORM이 만든 게 아니라 제 손으로 쓴 반복문이었습니다. "목록에 집계 하나만 붙이자"가 전형적인 시작입니다.

// 안티패턴 — TBM마다 확인 인원을 세러 간다
const tbms = await prisma.tbm.findMany({ where: { siteId } });
const rows = await Promise.all(
  tbms.map(async (t) => ({
    ...t,
    confirmed: await prisma.tbmParticipant.count({
      where: { tbmId: t.id, state: 'CONFIRMED' },
    }),
  })),
);  // 쿼리 1 + N개. 목록이 100건이면 101개.

// 처방 — groupBy 한 번으로 같은 답을 얻는다
const counts = await prisma.tbmParticipant.groupBy({
  by: ['tbmId'],
  where: { tbmId: { in: tbms.map((t) => t.id) }, state: 'CONFIRMED' },
  _count: true,
});  // 쿼리 2개로 끝. 이후 Map으로 붙이면 된다.

Promise.all로 감싸면 병렬이라 빨라 보이지만, 커넥션 풀을 N개가 나눠 쓰며 DB에 왕복 N번을 강요하는 건 그대로입니다. 부하가 없는 개발 DB에서는 티가 안 나다가, 운영에서 목록이 길어지는 순간 응답 시간이 목록 길이에 비례해 늘어납니다. 로그에 같은 모양의 쿼리가 파라미터만 바뀌며 줄줄이 찍힌다면 그게 N+1의 지문입니다.

참고로 GraphQL 리졸버처럼 항목 단위 조회가 구조적으로 강제되는 곳에서는, Prisma의 fluent API(prisma.tbm.findUnique({ where: { id } }).participants())가 같은 틱에 발생한 findUnique들을 내부 dataloader로 묶어 IN 쿼리 하나로 배치해 줍니다. 반복문 N+1과 달리 이 경로는 설계된 탈출구입니다.

4. 처방 비교 — IN 배치, DB 조인, 수동 SQL

N+1을 없애는 방법은 결국 세 갈래입니다. 각각 코드부터 보겠습니다.

// (a) 기본 전략 — 쿼리 분리 + IN 배치 (2절의 include가 이것)

// (b) DB 조인 — preview 기능 relationJoins를 켜면 선택 가능
const tbms = await prisma.tbm.findMany({
  relationLoadStrategy: 'join',   // PostgreSQL은 LATERAL JOIN + JSON 집계
  where: { siteId },
  include: { participants: true },
});

// (c) 수동 SQL — 조인과 집계를 한 방에. 태그드 템플릿이라 파라미터는 안전하다
const rows = await prisma.$queryRaw`
  SELECT t."id", t."title",
         COUNT(*) FILTER (WHERE p."state" = 'CONFIRMED') AS "confirmed",
         COUNT(p."tbmId")                                AS "total"
  FROM "Tbm" t
  LEFT JOIN "TbmParticipant" p ON p."tbmId" = t."id"
  WHERE t."siteId" = ${siteId}
  GROUP BY t."id"`;

선택 기준은 이렇게 정리하고 있습니다.

(a) IN 배치는 기본값이자 대부분의 화면에 충분합니다. 쿼리가 단순해 인덱스를 태우기 쉽고, 관계별로 나뉘어 있어 로그를 읽기도 쉽습니다. 약점은 왕복이 관계 수만큼 늘고, 조립을 애플리케이션이 떠안는다는 것.

(b) relationLoadStrategy: 'join'은 왕복을 한 번으로 줄입니다. 다만 이 글 시점 기준 preview 기능이라 schema.prisma의 previewFeatures = ["relationJoins"]를 켜야 하고, 생성되는 SQL이 JSON 집계를 낀 조인이라 플래너가 어떤 계획을 세우는지는 직접 확인해야 합니다. 조인이 항상 빠르다는 보장은 없습니다 — 부모가 적고 자식이 아주 많은 분포에서는 (a)가 나은 경우도 있습니다.

(c) $queryRaw는 집계·윈도우 함수·조건부 카운트처럼 ORM 문법 바깥의 일이 필요할 때 꺼냅니다. 위처럼 "목록 + 확인 인원 + 전체 인원"을 쿼리 하나로 끝낼 수 있습니다. 대가는 타입 안전이 약해지고(결과 타입을 직접 선언해야 합니다) 스키마 변경 시 컴파일러가 못 잡아준다는 것. 그래서 저는 raw SQL을 리포지토리 한 곳에 모아 격리합니다.

*

어느 처방이든 채택 전에 같은 데이터 분포로 로그를 떠서 비교하세요. 1절의 e.duration과 PostgreSQL의 EXPLAIN ANALYZE면 충분합니다. "조인이 빠르다더라"는 남의 분포에서 나온 결론일 수 있습니다.

5. $transaction — 배열, 인터랙티브, 격리수준

쿼리 개수를 줄이는 것과 별개로, 쿼리를 한 단위로 묶는 문제가 있습니다. Prisma의 $transaction은 두 가지 형태입니다. 배열 폼은 준비된 쿼리들을 순서대로 한 트랜잭션에서 실행합니다 — 앞 결과를 읽어 다음 쿼리를 결정할 필요가 없다면 이쪽이 가볍습니다. 읽고-판단하고-쓰는 흐름이 필요하면 인터랙티브 폼을 씁니다.

// TBM 발행 — 상태 전이 + 참가자 스냅샷 + 번역 작업 적재를 한 단위로
const tbm = await prisma.$transaction(
  async (tx) => {
    const tbm = await tx.tbm.update({
      where: { id, state: 'DRAFT' },      // 상태 조건을 where에 — 이중 발행 방지
      data: { state: 'PUBLISHED', publishedAt: new Date() },
    });
    await tx.tbmParticipant.createMany({ data: snapshots });
    await tx.translationJob.createMany({ data: pendingJobs });
    return tbm;
  },
  {
    isolationLevel: Prisma.TransactionIsolationLevel.Serializable,
    maxWait: 2000,   // 커넥션 확보 대기 한도 (기본 2초)
    timeout: 5000,   // 트랜잭션 전체 시간 한도 (기본 5초)
  },
);

세 번째 인자가 핵심입니다. 격리수준(isolationLevel)을 지정하지 않으면 DB의 기본값을 따릅니다 — PostgreSQL이라면 READ COMMITTED. 대부분의 CRUD에는 그걸로 충분하고, 위처럼 "읽은 상태를 전제로 쓰는" 경합 구간에만 올려 잡습니다. Serializable은 동시 발행 같은 경합을 확실히 걸러 주는 대신, 충돌 시 트랜잭션이 P2034(write conflict/deadlock) 에러로 실패할 수 있으므로 재시도 로직이 세트입니다. 격리수준을 올리는 코드에는 재시도가 따라와야 완성입니다.

이전 직장에서 정산 코드를 리팩토링하며 새긴 원칙도 같은 것이었습니다. 돈이든 안전 기록이든, 부분 성공은 실패보다 나쁩니다. "발행은 됐는데 참가자 스냅샷이 없다"는 상태는 조회 화면의 버그가 아니라 데이터의 거짓말이 되기 때문입니다.

*

인터랙티브 트랜잭션 안에서 외부 I/O를 하지 마세요. FCM 푸시, LLM 번역 호출 같은 것이 안에 들어오면 커넥션을 쥔 채 대기하다 timeout으로 통째로 롤백됩니다. 위 코드가 번역을 직접 호출하는 대신 translationJob 레코드만 적재하는 이유입니다 — 외부 호출은 커밋 이후, 워커의 일입니다.

6. 숨겨진 것을 꺼내 보는 습관

정리하면 순서는 이렇습니다. 쿼리 로그를 켜고, 요청당 쿼리 개수를 세고, 같은 모양의 쿼리가 반복되면 반복문을 의심합니다. 처방은 IN 배치 → DB 조인 → 수동 SQL 순으로, 단순한 것부터 씁니다. 그리고 묶여야 하는 쓰기는 $transaction으로 묶되, 격리수준을 올렸다면 재시도를 붙이고 외부 I/O는 밖으로 뺍니다. ORM을 쓰지 말자는 이야기가 아닙니다 — Prisma 덕에 스키마와 타입이 하나로 움직이는 생산성은 SafeIn처럼 모델이 많은 제품에서 포기할 수 없습니다. 다만 ORM이 숨겨 준 것을 언제든 꺼내 볼 수 있는 상태를 유지하는 것, 그게 ORM을 오래 쓰는 방법이라고 생각합니다.
