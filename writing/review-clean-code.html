서평: Clean Code — 규칙보다 태도

2021년 8월• 서평 · 클린코드 · 리팩토링

《Clean Code》를 처음 읽은 건 개발을 배우기 시작한 지 얼마 안 됐을 때였습니다. 그때는 이 책을 규칙집으로 읽었습니다. 함수는 짧게, 인자는 적게, 주석은 되도록 쓰지 말 것. 규칙은 외웠는데 제 코드는 좀처럼 깨끗해지지 않았고, 왜 그런지는 당시엔 몰랐습니다. 정산 로직을 리팩토링하고 크롤링 파이프라인을 다시 설계해 본 뒤에 다시 펼쳐 보니, 이 책이 줄곧 말하고 있던 건 규칙이라기보다 태도 쪽이었습니다. "고치며 떠난다"는 태도 말입니다. 그렇게 다시 읽으며 정리한 것을 적어 둡니다.

먼저 짚고 가기
리팩토링Refactoring
Martin Fowler가 《Refactoring》(1999)에서 정의한, 겉으로 관찰되는 동작(observable behavior)은 보존한 채 코드의 내부 구조만 바꾸는 변형입니다. 이름 바꾸기·함수 추출·이동처럼 작고 되돌릴 수 있는 단계를 하나씩 적용하고 매번 테스트로 동작이 그대로인지 확인하며, Fowler는 기능 추가와 리팩토링을 두 개의 모자(Two Hats)로 나눠 한 번에 하나만 쓰라고 했습니다. 즉 기능 추가와 버그 수정은 동작을 바꾸는 일이라 리팩토링이 아니고, 한 커밋에 섞이면 무엇이 동작 변경이고 무엇이 정리인지 리뷰어가 구분할 수 없게 됩니다. 이 글에서 말하는 "고치며 떠난다"는 태도는 이 작은 변형을 손댈 때마다 한두 개씩 반복하는 쪽에 가깝습니다.
추상화Abstraction
대상에서 지금 필요한 속성만 남기고 나머지 세부를 이름 뒤로 감추는 설계 행위입니다. 호출하는 쪽이 이름과 시그니처만 보고 "무엇을 하는지"는 알되 "어떻게 하는지"는 몰라도 되게 만드는 것이 목적이며, Dijkstra의 말대로 그 목적은 모호해지는 것이 아니라 "절대적으로 정확할 수 있는 새로운 의미 층"을 세우는 데 있습니다. 여기서 추상화 수준이란 그 이름이 서 있는 층을 가리키는데, isSettlementTarget은 업무 정책 층이고 getType() == 2는 저장 구조 층이라 한 함수 안에 섞이면 읽는 사람이 층을 오르내리게 됩니다. 이 글 4장의 "한 수준의 추상화"는 함수마다 그 층을 하나로 맞추라는 뜻이지, 무조건 잘게 쪼개라는 뜻이 아닙니다.
레거시Legacy
Michael Feathers는 《Working Effectively with Legacy Code》(2004)에서 레거시 코드를 테스트가 없는 코드로 정의했습니다. 테스트가 없으면 구조를 바꿨을 때 동작이 보존됐는지 확인할 수단이 없어서, 손댈 때마다 회귀 위험을 그대로 떠안게 되고 결국 아무도 건드리지 않게 됩니다. 이 정의에서 코드가 오래됐다거나 남이 짰다는 사실 자체는 조건이 아니어서, 어제 내가 짠 코드도 테스트가 없으면 여기 속합니다. 이 글 2장에서 말하는 레거시는 여기에 "설계 결정을 내가 쥐고 있지 않다"는 조건이 겹친 경우이고, 그래서 전면 재작성 대신 보이스카우트 규칙식 개선을 택했습니다.
예외Exception
실패를 반환값이 아니라 별도의 제어 경로로 전달하는 언어 장치입니다. throw가 실행되면 호출 스택을 되감아 그 타입과 일치하는 가장 가까운 catch로 제어가 넘어가므로, 사이에 낀 호출부들은 실패 코드를 검사해 되돌려 주는 코드를 쓰지 않아도 됩니다. Java에서는 RuntimeException이나 Error 계열이 아닌 예외(checked)가 호출부에 catch 또는 throws 선언을 강제하는데, 《Clean Code》 7장은 이것이 하위 모듈의 변경을 상위 시그니처까지 줄줄이 번지게 해 개방-폐쇄 원칙(OCP)을 깬다며 unchecked 쪽을 권합니다. 다만 "결과가 없음"처럼 정상적으로 일어나는 경우까지 예외로 던지면 안 되고, 그건 이 글 5장에서 쓴 특수 케이스 객체나 Optional의 몫입니다.
DRY 원칙Don't Repeat Yourself
Andrew Hunt와 David Thomas가 《The Pragmatic Programmer》(1999)에서 제시한 설계 원칙으로, 원문은 "모든 지식은 시스템 안에서 단 하나의, 모호하지 않은, 권위 있는 표현을 가져야 한다"입니다. 중복의 단위가 코드 문자열이 아니라 지식이라는 점이 핵심이어서, 판단 기준은 "생김새가 같은가"가 아니라 "요구사항이 바뀔 때 함께 바뀌는가"입니다. 지금 우연히 모양만 같은 두 코드를 성급히 합치면 한쪽만 바뀌어야 하는 날부터 플래그와 조건 분기가 그 자리로 파고듭니다. 이 글 6장에서 이 원칙을 맹신하지 말자고 적어 둔 이유가 그것입니다.

1. 규칙집으로 읽으면 실패한다

이 책이 규칙집처럼 읽히는 데는 이유가 있습니다. 마지막 장이 아예 "냄새와 휴리스틱"이라는 제목의 목록이고, 각 장도 "이렇게 하라"는 항목의 나열처럼 보이기 때문입니다. 그래서 많은 입문자가 이 책에서 규칙만 뽑아 체크리스트를 만드는데, 저도 처음엔 그렇게 읽었습니다. 그런데 근거를 뺀 규칙은 화물 숭배에 가까워서, 형태만 흉내 낼 뿐 왜 그렇게 하는지를 모르니 상황이 조금 달라지면 오히려 코드를 망치게 됩니다.

다시 읽어 보니 책의 척추는 그 목록이 아니라 1장에 있었습니다. Robert C. Martin은 유명한 개발자들에게 "깨끗한 코드란 무엇인가"를 물었는데, 돌아온 답은 대체로 규칙이 아니라 읽는 경험에 관한 것이었습니다. Grady Booch는 "잘 쓴 산문처럼 읽힌다"고 했고, Ward Cunningham은 "각 루틴이 당신이 예상한 그대로일 때"라고 했습니다. 같은 장에서 저자는 코드를 읽는 시간과 쓰는 시간의 비율이 10 대 1을 훌쩍 넘는다고 지적하는데, 코드는 결국 다음 사람이 읽으라고 쓰는 글이라는 뜻으로 받아들였습니다.

그렇게 보면 이 책이 던지는 질문은 "이 코드가 규칙을 지켰는가"보다 "다음 사람이 이 코드를 예상대로 읽을 수 있는가"에 가깝습니다. 질문을 이렇게 바꿔 놓고 나서야 뒤따르는 규칙들이 목적이 아니라 수단으로 읽혔습니다. 이 글은 그 관점으로 다시 읽으며 남긴 기록입니다.

2. 척추는 보이스카우트 규칙

책 전체를 한 문장으로 줄이라면 저는 1장의 보이스카우트 규칙을 고르겠습니다. "캠핑장은 처음 왔을 때보다 깨끗하게 하고 떠나라." 코드로 옮기면 어떤 파일을 손댔다면 떠날 때 조금이라도 더 깨끗한 상태로 남기라는 뜻인데, 여기서 말하는 건 전면 재작성이 아니라 지나가는 김에 이름 하나나 가드 하나를 정리하는 정도의 작은 개선입니다.

// 버그 하나 고치러 들어왔다가, 떠날 땐 조금 더 깨끗하게
// Before — 손대러 온 김에 눈에 밟히던 코드
public void process(Order o) {
  if (o.getS() == 1) {            // 1 = 결제 완료
    // ...결제 완료 처리
  }
}

// After — 버그 수정과 함께 이름 하나, 가드 하나만
public void process(Order order) {
  if (isPaid(order)) {
    // ...결제 완료 처리
  }
}
private boolean isPaid(Order order) {
  return order.getStatus() == OrderStatus.PAID;
}

거대한 리팩토링보다 이 작은 태도 쪽이 오래가는 이유는 같은 장에 나오는 르블랑의 법칙에 있습니다. "나중은 결코 오지 않는다(Later equals never)"는 말인데, 제 경험으로도 "언젠가 날 잡아 싹 정리하자"고 미뤄 둔 그 날은 오지 않았고 코드는 그동안 계속 낡았습니다. 반대로 손댈 때마다 조금씩 고치면 그 개선이 복리처럼 쌓입니다. 저자가 1장에서 "빠르게 가는 유일한 길은 코드를 계속 깨끗하게 유지하는 것"이라고 쓴 근거도 여기에 있습니다.

제 경우엔 유지보수 현장에서 이 태도가 그나마 지속 가능한 자세였습니다. 대흥정보에서 SM 업무를 할 때는 제가 짜지 않았고 함부로 갈아엎을 수도 없는 레거시를 물려받았고, 그때 할 수 있는 것도 보이스카우트 규칙 정도였습니다. SafeIn의 정산 리팩토링 역시 처음부터 큰 프로젝트로 시작한 게 아니라 "온 김에 이것만"이라는 작은 정리를 반복하다가 중복의 지형이 드러나면서 판이 커진 경우였습니다.

*

실무 팁 — 보이스카우트 규칙을 "온 김에 다 뜯어고치기"로 오해하면 곤란해집니다. 버그 수정 커밋에 500줄짜리 리팩토링을 섞으면 리뷰어는 무엇이 동작 변경이고 무엇이 단순 정리인지 구분할 수 없고, 나중에 git blame도 엉킵니다. 그래서 정리는 작게 하고 가능하면 커밋을 분리해 두는 편입니다. "조금 더 깨끗하게"의 방점은 '조금'에 있습니다.

3. 이름 — 규칙이 아니라 의도

2장 "의미 있는 이름"은 규칙이 가장 많은 장입니다. 의도를 드러내라, 검색 가능하게 하라, 발음 가능하게 하라, 인코딩(헝가리안 표기)을 피하라 등등. 처음엔 이걸 다 외우려 했는데, 묶어 놓고 보니 결국 의도를 드러내는 이름이라는 하나의 목적으로 모이고 나머지는 거기에 봉사하는 수단이었습니다.

// Before — 규칙(검색·발음 가능)은 지켰지만 의도가 없다
List<Merchant> ms = repo.findAll();
List<Merchant> ms2 = new ArrayList<>();
for (Merchant m : ms)
  if (m.getType() == 2) ms2.add(m);   // 2 = 정산 대상

// After — 이름이 곧 이 루프가 하는 일의 설명이 된다
List<Merchant> allMerchants = merchantRepository.findAll();
List<Merchant> settlementTargets = allMerchants.stream()
    .filter(Merchant::isSettlementTarget)
    .toList();

두 코드는 같은 일을 합니다. 차이는 isSettlementTarget이라는 서술어 하나가 "타입이 2인 것"이라는 정체불명의 조건에 이름을 붙였다는 것뿐인데, Booch가 말한 "산문처럼 읽힌다"가 이런 것이겠구나 싶었습니다. 같은 장의 "개념 하나에 단어 하나(Pick One Word per Concept)" 규칙은 한 걸음 더 나아가 DDD의 유비쿼터스 언어와 맞닿습니다 — 팀이 회의에서 쓰는 단어를 코드에서도 똑같이 쓰자는 것이죠. 정산 리팩토링에서 가장 큰 문제도 사실 이름의 충돌, 즉 같은 개념이 calcCharge·computeFee·getSettleAmt로 흩어져 grep에도 잡히지 않던 것이었습니다.

그래서 이 장에서 제가 가져온 건 규칙 목록이 아니라 질문 하나였습니다. "이 이름이 왜 존재하고, 무엇을 하고, 어떻게 쓰이는지를 스스로 답할 수 있는가?" 여기에 답하지 못하면 나머지 규칙을 지켜도 이름은 잘 읽히지 않았습니다. 이름 규칙을 나쁜 예→좋은 예로 하나하나 뜯은 이야기는 클린코드 · 이름 글에 따로 정리해 두었습니다.

4. 함수 — "작게"의 진짜 뜻과 함정

3장에서 가장 자주 인용되는 문장은 "함수의 첫 번째 규칙은 작아야 한다는 것이다. 두 번째 규칙은 그보다 더 작아야 한다는 것이다"일 텐데, 이어지는 "함수는 한 가지를 해야 한다. 그것을 잘 해야 한다. 그것만 해야 한다"까지 읽고 나서야 판단 기준이 줄 수가 아님을 알았습니다. 기준은 세 가지였습니다 — 한 가지 일을 하는가, 한 수준의 추상화만 담는가, 위에서 아래로 이야기처럼 읽히는가(스텝다운 규칙).

이걸 줄 수에 대한 명령으로 오해하면 곤란해집니다. "쪼갤 수 있으면 끝까지 쪼개라"를 곧이곧대로 적용하면 한 줄짜리 함수들이 덩굴처럼 얽혀서, 흐름 하나를 따라가려고 파일을 위아래로 헤매게 됩니다.

// "작게"를 맹신한 결과 — 한 줄짜리 함수의 덩굴
public Money settle(Order order) {
  return applyFee(applyDiscount(baseAmount(order)), order);
}
private Money baseAmount(Order o)        { return o.getAmount(); }
private Money applyDiscount(Money m)    { return m.minus(discountOf(m)); }
private Money discountOf(Money m)       { return m.times(RATE); }
private Money applyFee(Money m, Order o) { return m.minus(feeOf(o)); }
// ...흐름을 읽으려면 다섯 곳을 오르내려야 한다

// 한 단계 덜 쪼갠 편 — 한 화면에서 흐름이 읽힌다
public Money settle(Order order) {
  Money base     = order.getAmount();
  Money discount = base.times(DISCOUNT_RATE);
  Money fee      = feePolicy.of(order);
  return base.minus(discount).minus(fee);
}

저는 아래쪽이 더 읽기 좋다고 생각합니다. 흐름이 한 화면에 담기고 base → discount → fee가 순서대로 읽히기 때문입니다. 위쪽은 "작게"라는 규칙을 더 잘 지켰지만 읽는 비용은 늘었습니다. 함수를 나누는 기준이 줄 수가 아니라 의미라는 건, 이렇게 둘을 나란히 놓고 나서야 알았습니다.

!

함정 — "함수는 몇 줄 이하"라는 숫자는 이 책 어디에도 없습니다. 저자가 든 기준은 "한 가지 일 / 한 수준의 추상화"이지 줄 수가 아닙니다. 줄 수를 목표로 삼아 무리하게 추출하면 정작 버그를 쫓을 때 호출 사슬을 다섯 겹씩 헤집게 됩니다. 저는 (1) 개념에 이름을 붙일 때, (2) 중복을 없앨 때, (3) 추상화 수준을 한 단계 낮출 때 정도로 추출 시점을 정해 두고 있습니다.

이 장의 나머지 조언 — 플래그 인자를 쓰지 말 것, 명령과 조회를 분리할 것(Command-Query Separation), 그리고 에러 코드보다 예외를 선호할 것 — 도 모두 "함수가 한 가지만, 예상대로 읽히게" 하려는 같은 목적의 변주입니다. 함수 규칙을 실전 리팩토링으로 더 파고든 이야기는 클린코드 · 함수 글에 있습니다.

5. 주석과 에러 처리 — 표현의 실패

4장은 "주석은 잘해야 필요악이다"라는 문장으로 시작합니다. 이어서 저자는 "주석의 올바른 용도는 코드로 의도를 표현하지 못한 우리의 실패를 메우는 것이다"라고 씁니다. 그래서 주석을 달고 싶어질 때 먼저 확인하게 된 건 이름과 함수로 표현할 수는 없었는가였습니다. 이름과 함수 장이 이 장보다 앞에 오는 이유도 여기에 있는 듯합니다.

다만 이 장은 "주석은 나쁘니 다 지워라"로 오독되기 쉽습니다. 저자는 같은 장에서 좋은 주석도 나열해 두었습니다 — 의도 설명, 결과 경고, TODO, 법적 고지, 강조. 사라져야 할 건 중복되거나 거짓말하는 주석이지 모든 주석은 아닙니다. 특히 "왜"를 설명하는 주석은 이름이 대신하기 어렵습니다. 코드는 "무엇을" 하는지까지만 말해 주고, "왜 하필 이 값인지"는 여전히 주석과 커밋 메시지의 몫이기 때문입니다.

// Before — 에러 코드와 null이 호출부를 오염시킨다
Settlement s = settlementService.find(id);
if (s == null) return ERROR_NOT_FOUND;   // 모든 호출부가 null을 기억해야
if (s.getStatus() == -1) { /* -1 = 실패, 매직 코드 */ }

// After — 예외는 흐름에서 분리하고, 정상 경로만 남긴다
Settlement settlement = settlementService.findById(id)
    .orElseThrow(() -> new SettlementNotFoundException(id));

// "결과 없음"이 정상이라면 null 대신 특수 케이스 객체(Null Object)
Payout payout = payoutRepository.findLatest(merchantId)
    .orElse(Payout.NONE);            // 호출부에 if가 필요 없다

7장 "에러 처리"의 요지가 이 코드에 담겨 있습니다. 에러 코드 대신 예외를, null 반환 대신 특수 케이스 객체를 쓰라는 것인데, 에러 코드와 null은 모든 호출부에 방어 코드를 요구해서 정작 중요한 정상 흐름을 잡음 속에 파묻기 때문입니다. 예외로 실패를 흐름에서 떼어내면 본문에는 "성공했을 때 할 일"만 남습니다. SafeIn 백엔드를 혼자 설계할 때는 이 에러 계약을 초반에 잡아 둔 것이 두고두고 도움이 됐습니다. 설계를 쥐고 있는 사람이 저 하나뿐이라 나중에 맞추기가 더 어려웠을 테니까요. 이 주제는 클린코드 · 에러 처리 글에서 더 다뤘습니다.

6. 규칙을 맹신하면 — 이 책의 한계

다시 읽으면서 이 책의 한계도 같이 눈에 들어왔습니다. 우선 이 책은 2008년에 나온, 예제가 온통 Java인 책입니다. 람다도 Optional도 없던 시절의 장황한 관용구가 예제에 그대로 남아 있어서, 어떤 조언은 그 시절 언어의 제약을 반영한 것으로 보입니다. 2026년의 Kotlin이나 TypeScript로 옮기면 성립하지 않는 세부도 있습니다.

논쟁적인 대목도 있습니다. 저자는 함수를 "깨끗하게" 유지하려고 인자를 인스턴스 필드로 옮기는 기법을 종종 쓰는데, 이건 인자 전달을 숨은 가변 상태로 바꾸는 거래라 동시성 환경에서는 위험할 수 있습니다. 저도 공유되던 가변 상태에 동기화가 빠져 객체가 회수되지 못하던 누수를 힙덤프로 쫓은 적이 있어서, 이 지점은 지금도 조심스럽습니다. DRY도 마찬가지입니다. Sandi Metz가 남긴 "잘못된 추상화보다 중복이 훨씬 싸다"는 말이 이 원칙에 대한 좋은 교정이라고 생각합니다. 닮아 보이는 두 코드가 사실은 다른 개념일 때, 성급한 통합은 나중에 더 비싼 값을 치르게 됩니다.

그래서 지금은 이렇게 읽고 있습니다. 규칙은 맥락에 의존하는 휴리스틱이고, 그보다 덜 흔들리는 건 12장의 단순 설계 4원칙(모든 테스트를 통과할 것, 중복이 없을 것, 의도를 드러낼 것, 요소 수를 최소화할 것)과 그 밑에 흐르는 태도입니다. 이 책은 판단을 대신해 주는 매뉴얼이라기보다 판단을 시작하게 하는 출발점에 가깝습니다. 규칙을 맹신하면 오히려 원래 목적에서 멀어지는데, 클린코드는 읽는 사람을 위한 것인 반면 도그마는 그 사람의 맥락을 보지 않기 때문입니다.

7. 정리 — 규칙은 잊혀도 태도는 남는다

다시 읽고 나서 손에 남은 것은 규칙 목록이 아니라 몇 가지 태도였습니다. 나중에 다시 찾아보려고 정리해 둡니다.

  • 보이스카우트 규칙 — 손댄 코드는 조금이라도 더 깨끗하게 남기고 떠난다. '나중'은 오지 않는다.
  • 코드는 읽는 사람에게 쓰는 글 — 판단 기준은 "규칙 준수"가 아니라 "다음 사람이 예상대로 읽는가".
  • 규칙은 수단, 의도가 목적 — 이름·함수 규칙은 전부 "의도를 드러내라"에 봉사한다.
  • 표현의 실패를 이름·함수로 메운다 — 그래도 "왜"는 주석·커밋에 남긴다.
  • 예외로 정상 흐름을 지킨다 — 에러 코드와 null을 호출부에 흘리지 않는다.
  • 규칙은 맥락에 따라 조정한다 — 단순 설계 4원칙과 "읽는 사람" 원칙을 기준으로 두고 나머지는 상황에 맞춘다.

규칙은 언젠가 낡습니다. 2008년의 Java 관용구는 오늘의 언어에서 이미 어색하고 앞으로도 계속 바뀌겠지만, 태도 쪽은 그만큼 빨리 낡지 않는 것 같습니다. Michael Feathers가 1장에 남긴 "깨끗한 코드는 언제나 누군가 신경 써서 짠 것처럼 보인다"는 문장이 그 태도를 짧게 요약해 줍니다. 이 책에서 한 문장만 기억한다면 저는 이걸로 두려고 합니다.

대신 규칙의 실전 적용은 네 편으로 따로 정리했습니다 — 이름, 함수, 에러 처리, 테스트. 이 서평이 척추라면 그 네 편은 거기 붙는 갈비뼈쯤 됩니다. 규칙이 필요할 때는 그쪽을 펼치고, 규칙을 지키는 이유가 흐려질 때는 다시 이 태도로 돌아오려고 합니다.
