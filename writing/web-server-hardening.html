웹 서버 하드닝 체크리스트 — ServerTokens부터 헤더까지

2024년 12월• 약 7분• Apache · Nginx · 헤더

보안 감사의 첫 지적은 "서버 버전이 응답 헤더에 노출된다"였습니다. 사소해 보였지만, 공격자에게 정확한 버전을 알려주는 건 CVE 목록을 손에 쥐여주는 것과 같습니다. 그 일을 계기로 정리한 체크리스트입니다.

1. 나를 소개하지 않기

# Apache — Server 헤더에서 버전 제거
ServerTokens Prod
ServerSignature Off

# Nginx
server_tokens off;

변경 자체는 한 줄이지만, 이 헤더를 참조하는 모니터링·LB가 있는지 먼저 확인하고 개발 환경에서 테스트한 뒤 반영했습니다. "간단한 변경"이라는 방심이 장애를 만듭니다.

2. 보안 헤더 세트

# 최소 세트 — 대부분의 서비스에 안전
Strict-Transport-Security: max-age=63072000; includeSubDomains
X-Content-Type-Options: nosniff
X-Frame-Options: DENY            # 클릭재킹 방지
Referrer-Policy: strict-origin-when-cross-origin
Content-Security-Policy: default-src 'self'   # 점진 도입

CSP는 강력하지만 잘못 넣으면 화면이 깨집니다. Content-Security-Policy-Report-Only로 먼저 위반을 수집만 하다가, 리포트가 깨끗해지면 강제로 전환하는 게 안전합니다.

3. 표면 줄이기

  • 쓰지 않는 HTTP 메서드(TRACE, PUT 등) 차단
  • 디렉터리 리스팅 끄기(autoindex off, Options -Indexes)
  • TLS는 1.2 이상만, 약한 cipher 제거
  • 관리자 경로는 IP 화이트리스트/별도 인증
*

보안은 하나의 큰 방패가 아니라 작은 노출을 하나씩 닫는 일의 합입니다. 화려하지 않지만, 정찰 단계에서 공격자를 지치게 만드는 게 방어의 절반입니다.
