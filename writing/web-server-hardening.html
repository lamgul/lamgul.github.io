웹 서버 하드닝 체크리스트 — ServerTokens부터 헤더까지

2024년 12월• Apache · Nginx · 헤더

보안 감사에서 처음 받은 지적은 서버 버전이 응답 헤더에 그대로 노출된다는 것이었습니다. 처음엔 사소한 지적이라고 생각했는데, 버전이 드러나면 공격자가 그 버전에 해당하는 CVE 목록을 그대로 참고할 수 있다는 설명을 듣고 생각을 바꿨습니다. 그 뒤로 서버 설정을 하나씩 점검하면서 정리해 둔 체크리스트입니다.

먼저 짚고 가기
응답 헤더Response Header
HTTP 응답에서 상태 줄(status line) 다음, 본문 앞에 놓이는 이름: 값 형식의 메타데이터 필드들입니다. 각 필드는 CRLF로 구분되고 빈 줄 하나가 헤더와 본문의 경계가 됩니다(RFC 9110·9112). Server, Content-Type처럼 응답을 처리하는 데 필요한 정보를 담지만 화면에 렌더링되는 본문 자체는 아니며, 여기에 서버 종류·버전이 실려 나가는 게 이 글의 출발점입니다.
CVECommon Vulnerabilities and Exposures
공개적으로 알려진 보안 취약점 하나하나에 CVE-연도-일련번호 형식의 고유 식별자를 부여하는 체계입니다. MITRE가 관장하고 번호 발급 기관(CNA)이 ID를 매겨, 서로 다른 보안 도구·문서가 같은 취약점을 같은 번호로 가리키게 합니다. CVE는 식별자일 뿐 심각도 점수(그것은 CVSS)는 아니며, 서버 버전이 노출되면 공격자가 그 버전에 걸린 CVE 목록을 그대로 조회할 수 있어 버전 숨기기가 첫 방어가 됩니다.
HSTSHTTP Strict Transport Security
서버가 Strict-Transport-Security 응답 헤더로 브라우저에 "이 호스트는 max-age 기간 동안 HTTPS로만 접속하라"고 지시하는 정책입니다(RFC 6797). 헤더를 받은 브라우저는 그 기간 내 모든 요청을 HTTPS로 자동 승격시키고, 인증서 오류가 나도 사용자가 경고를 무시하고 진입하는 것을 막습니다. 다만 헤더를 처음 받기 전의 최초 접속은 보호되지 않아, 이 공백은 브라우저 내장 preload 목록으로 메웁니다.
CSPContent Security Policy
Content-Security-Policy 응답 헤더에 default-src, script-src 같은 지시어로 자원 종류별 허용 출처를 선언하는 정책입니다. 브라우저는 이 목록에 어긋나는 스크립트·이미지 로드와 인라인 실행을 차단해, 주입된 악성 스크립트(XSS) 실행을 어렵게 만드는 심층 방어 계층으로 작동합니다. 완전한 XSS 차단은 아니며 허용 출처를 좁게 잡으면 정상 자원까지 막혀 화면이 깨지므로, Content-Security-Policy-Report-Only로 위반만 수집하며 지켜본 뒤 강제 적용합니다.
TLSTransport Layer Security
TCP 같은 전송 계층 위에서 통신을 암호화하고 무결성을 보장하는 프로토콜로, SSL의 후속입니다. 핸드셰이크에서 인증서로 서버를 인증하고 키를 교환한 뒤 대칭키 암호로 이후 트래픽을 암호화하며, HTTPS는 HTTP를 이 TLS 위에 실어 나른 것입니다. 1.0·1.1은 알려진 약점으로 폐기되어(RFC 8996) 1.2(RFC 5246)·1.3(RFC 8446)만 허용하는 것이 권장됩니다.

1. 나를 소개하지 않기

# Apache — Server 헤더에서 버전 제거
ServerTokens Prod
ServerSignature Off

# Nginx
server_tokens off;

설정 자체는 한 줄이라 금방 끝나지만, 이 헤더를 참조하는 모니터링·LB가 있는지 먼저 확인하고 개발 환경에서 테스트한 뒤에 반영했습니다. 간단해 보이는 변경일수록 확인 없이 넘어가기 쉬워서, 이 순서만은 지키기로 해 두었습니다.

2. 보안 헤더 세트

# 최소 세트 — 대부분의 서비스에 안전
Strict-Transport-Security: max-age=63072000; includeSubDomains
X-Content-Type-Options: nosniff
X-Frame-Options: DENY            # 클릭재킹 방지
Referrer-Policy: strict-origin-when-cross-origin
Content-Security-Policy: default-src 'self'   # 점진 도입

CSP는 효과가 큰 만큼 잘못 넣으면 정상 자원까지 막혀 화면이 깨집니다. 그래서 Content-Security-Policy-Report-Only로 위반 내역을 수집만 하며 한동안 지켜보다가, 리포트가 조용해진 뒤에 강제 적용으로 바꿨습니다.

3. 표면 줄이기

  • 쓰지 않는 HTTP 메서드(TRACE, PUT 등) 차단
  • 디렉터리 리스팅 끄기(autoindex off, Options -Indexes)
  • TLS는 1.2 이상만, 약한 cipher 제거
  • 관리자 경로는 IP 화이트리스트/별도 인증
*

돌아보면 이 작업은 큰 조치 하나를 넣는 일이라기보다 작은 노출을 하나씩 닫는 일이 쌓이는 쪽에 가까웠습니다. 눈에 띄는 성과는 아니지만, 정찰 단계에서 공격자가 얻어 갈 정보를 줄여 두는 것만으로도 해 둘 만한 일이라고 생각합니다.
