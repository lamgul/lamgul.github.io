DDD 애그리거트 경계 — 크롤링 파이프라인에 그은 선

2023년 8월• DDD · 설계 · 애그리거트

크롤링 파이프라인을 DDD로 재설계하면서 가장 오래 매달린 질문은 "어디까지가 하나의 애그리거트인가"였습니다. 클래스 이름을 정하거나 레이어를 나누는 일보다 이 선을 긋는 쪽이 훨씬 어려웠습니다. 처음에 잘못 그은 탓에 워커를 늘릴수록 트랜잭션이 잡 하나로 몰리는 것을 겪고 나서야 기준이 잡혔습니다. 그때 어떻게 긋고 어떻게 고쳐 그었는지 잊지 않으려고 적어 둡니다.

먼저 짚고 가기
애그리거트Aggregate
DDD의 전술 패턴으로, 하나의 루트 엔티티를 통해서만 외부에서 접근할 수 있게 묶은 엔티티와 값 객체의 집합입니다. 리포지토리는 이 묶음을 통째로 읽고 통째로 저장하므로, 애그리거트는 곧 잠금과 커밋이 걸리는 단위가 됩니다. 관련 있는 것끼리 모아 둔 패키지·폴더 구분이 아니라 무엇을 함께 잠글지를 정하는 선이라는 점이 이 글의 출발점입니다.
트랜잭션 경계transaction boundary
한 트랜잭션이 원자성(전부 커밋되거나 전부 롤백)을 보장하는 연산의 범위입니다. 격리 수준과 잠금도 이 범위 안에서만 적용되므로, 두 트랜잭션에 나뉘어 실행되는 작업은 중간 상태가 다른 트랜잭션에 보일 수 있습니다. "한 트랜잭션은 한 애그리거트만 수정한다"는 규칙은 이 경계와 애그리거트 경계를 일치시키라는 요구이고, 이 글은 그 선을 어디에 그었는지를 다룹니다.
불변식invariant
상태 변경이 끝난 시점에 반드시 참이어야 하는 도메인 규칙, 즉 상태에 대한 참·거짓 명제입니다. 연산 도중에는 일시적으로 깨질 수 있고 커밋 시점에 다시 참이어야 한다는 쪽이 정확한 표현이며, 이 검증은 애그리거트 루트가 자기 경계 안에서 책임집니다. 그래서 커밋 순간에 함께 참이어야 하는 불변식의 범위가 곧 애그리거트의 크기를 정합니다. 모든 도메인 규칙이 불변식은 아니어서, 잠시 어긋나도 되는 집계는 여기에 넣지 않습니다.
결과적 일관성eventual consistency
분산 시스템의 일관성 모델 중 하나로, 새로운 쓰기가 멈추면 언젠가 모든 복제본과 파생 데이터가 같은 값으로 수렴한다는 보장입니다. 수렴까지의 시간 상한은 규정하지 않으므로 그 전까지는 낡은 값을 읽을 수 있고, 대신 트랜잭션 경계를 좁게 유지할 수 있습니다. 수렴은 저절로 일어나지 않아서 이벤트 발행·재전달·멱등 처리를 직접 설계해 만들어 내야 합니다.
멱등성idempotency
같은 연산을 두 번 이상 적용한 결과가 한 번 적용한 것과 같은 성질, 즉 f(f(x)) = f(x)입니다. 여기서 같아야 하는 대상은 응답값이 아니라 시스템에 남는 상태입니다. 카운터 증가처럼 그 자체로는 멱등하지 않은 연산은, 처리한 이벤트 ID를 상태 변경과 같은 트랜잭션에 기록해 두 번째 도착을 걸러내는 방식으로 멱등하게 만듭니다. 재시도를 안전하게 만들 뿐 이벤트의 도착 순서까지 보장하지는 않습니다.

1. 애그리거트는 폴더가 아니라 트랜잭션 경계다

저는 한동안 애그리거트를 "관련 있는 객체 묶음" 정도로 이해하고 있었습니다. 그렇게 보면 경계를 그을 기준이 생기지 않습니다. 크롤링 도메인에서는 잡·페이지·결과·로그가 전부 서로 관련 있기 때문입니다. 에릭 에반스의 정의에서 실무에 쓸모가 있었던 쪽은 묶음이 아니라 일관성 경계(consistency boundary)라는 표현이었습니다. 애그리거트란 하나의 트랜잭션이 커밋되는 순간 반드시 함께 참이어야 하는 불변식(invariant)의 범위입니다.

이 정의를 받아들이면 규칙 세 가지가 따라 나옵니다. 반 버논(Vaughn Vernon)이 정리해 둔 것과 같은 내용입니다.

첫째, 한 트랜잭션은 한 애그리거트만 수정한다. 둘째, 그래서 애그리거트는 작게 만든다. 셋째, 다른 애그리거트는 객체가 아니라 ID로 참조하고, 애그리거트 사이의 일관성은 결과적 일관성(eventual consistency)으로 맞춘다. 처음 읽었을 때는 왜 이렇게까지 해야 하는지 잘 와닿지 않았습니다. 저는 이 규칙들을 모르는 채로 한 번 어겨 보고 나서야 이유를 알았습니다.

2. 잘못 그은 선 — 모든 것을 품은 CrawlJob

처음 만든 모델은 단순했습니다. 잡이 페이지를 만들고 결과를 받으니 잡이 전부 가지고 있으면 된다고 생각했습니다.

// crawl-job.ts — 잘못 그은 경계: 잡이 모든 것을 품는다
export class CrawlJob {
  private status: JobStatus = 'QUEUED';
  private readonly pages: PageTask[] = [];  // 잡 하나에 수천 건이 매달린다

  addPage(url: string) {
    if (this.status === 'COMPLETED') throw new DomainError('완료된 잡');
    this.pages.push(PageTask.queued(url));
  }

  recordResult(url: string, result: FetchResult) {
    const page = this.pages.find((p) => p.url === url);
    if (!page) throw new DomainError('등록되지 않은 페이지');
    page.complete(result);
    if (this.pages.every((p) => p.isTerminal())) this.status = 'COMPLETED';
  }
}

// 워커 쪽 — 결과 하나를 쓰려고 잡 전체를 든다
const job = await jobRepo.findById(jobId);  // 페이지 수천 건 로드
job.recordResult(url, result);
await jobRepo.save(job);  // 낙관적 잠금 버전 충돌 — 다른 워커도 같은 잡을 고치는 중

이 모델은 단위 테스트에서는 문제없이 통과했고, 워커를 여러 개 띄우고 나서야 어긋나기 시작했습니다. 애그리거트는 저장소에서 통째로 읽고 통째로 저장하는 단위이므로, 워커 열 개가 페이지 열 개를 각각 완료해도 전부 같은 CrawlJob 하나를 놓고 경합하게 됩니다. 낙관적 잠금을 쓰면 버전 충돌과 재시도가 계속 쌓였고, 비관적 잠금으로 바꾸면 워커들이 한 줄로 서서 순서를 기다렸습니다. 병렬로 긁으라고 만든 파이프라인이 잡 단위로 직렬화되고 있었습니다.

동시성 말고도 걸리는 곳이 있었습니다. 페이지 하나를 갱신하려고 수천 건을 로드하니 메모리와 쿼리가 낭비됐고, "잡을 완료 처리하는 규칙"과 "페이지 하나를 재시도하는 규칙"이 한 클래스에 섞여 코드도 같이 비대해졌습니다. 경계를 크게 잡으면 이런 것들이 함께 무거워진다는 걸 그때 알았습니다.

3. 진짜 불변식 찾기 — "지금 당장 참이어야 하는가"

경계를 다시 그을 기준으로 삼은 것은 "이 규칙은 커밋 순간에 반드시 참이어야 하는가, 아니면 잠시 어긋나도 괜찮은가"라는 질문이었습니다. 크롤링 도메인의 규칙들을 하나씩 이 질문에 넣어 분류해 봤습니다.

커밋 순간에 참이어야 하는 쪽에는 페이지 태스크의 상태 전이가 들어갔습니다. FETCHING이 아닌 태스크에 결과가 붙으면 안 되고, 재시도 횟수는 상한을 넘으면 안 됩니다. 잡의 상태 전이(QUEUED → RUNNING → COMPLETED/CANCELED)도 마찬가지로 순간마다 유효해야 합니다. 이런 규칙이 트랜잭션 일관성의 대상입니다.

잠시 어긋나도 되는 쪽에는 "잡의 완료 카운트 = 완료된 페이지 수" 같은 집계가 들어갔습니다. 페이지가 완료된 뒤 잡의 진행률에 반영되기까지 몇백 밀리초가 걸려도 문제가 되지 않습니다. 진행률은 대시보드가 보는 숫자이지 도메인이 그 순간에 지켜야 할 규칙은 아니었습니다. 이런 규칙이 결과적 일관성의 대상입니다.

이렇게 나누고 나니 처음 모델에서 무엇을 잘못했는지 보였습니다. 결과적 일관성으로 충분한 집계 규칙(모든 페이지가 끝나면 잡을 완료로)을 트랜잭션 불변식으로 올려 두었고, 그 때문에 애그리거트가 도메인 대부분을 끌어안게 된 것이었습니다.

*

경계를 점검할 때 보는 신호를 두 가지로 정리해 두었습니다. 서로 다른 워커나 사용자가 동시에 수정하는 두 객체가 한 애그리거트 안에 있다면 경계가 너무 큽니다. 반대로 두 애그리거트를 한 트랜잭션에서 함께 고쳐야 하는 요구가 반복된다면 경계가 너무 잘게 잘렸거나 잘못 잘린 것입니다. 정산 리팩토링을 할 때도 같은 질문을 했는데, 도메인이 달라도 신호는 비슷했습니다.

4. 다시 그은 선 — 작게 나누고 ID로 참조한다

불변식을 분류한 대로 모델을 옮기니 애그리거트는 둘로 나뉘었습니다. 잡은 자신의 상태 전이만, 태스크는 자신의 상태 전이만 책임지도록 했습니다.

// page-task.ts — 워커가 잠그는 것은 이제 태스크 하나뿐이다
export class PageTask {
  constructor(
    readonly id: PageTaskId,
    readonly jobId: CrawlJobId,  // 객체가 아니라 ID로 참조
    private status: TaskStatus,
    private attempts: number,
  ) {}

  complete(result: FetchResult) {
    if (this.status !== 'FETCHING')
      throw new DomainError(`${this.status}에서 완료 불가`);
    this.status = 'DONE';
    this.record(new PageTaskCompleted(this.id, this.jobId));  // 도메인 이벤트
  }

  fail(maxAttempts: number) {
    this.attempts += 1;  // 불변식: attempts는 상한을 넘지 않는다
    this.status = this.attempts >= maxAttempts ? 'FAILED' : 'QUEUED';
  }
}

// crawl-job.ts — 페이지 목록이 사라졌다
export class CrawlJob {
  private status: JobStatus = 'QUEUED';

  start() {
    if (this.status !== 'QUEUED') throw new DomainError('이미 시작된 잡');
    this.status = 'RUNNING';
  }

  cancel() {
    if (this.status === 'COMPLETED') throw new DomainError('완료된 잡은 취소 불가');
    this.status = 'CANCELED';
  }
}

바뀐 점을 정리하면, 워커가 결과 하나를 기록할 때 로드하고 저장하는 것은 PageTask 한 건뿐입니다. 워커 수십 개가 서로 다른 태스크를 물고 있으면 트랜잭션이 서로 겹치지 않습니다. 잡 전체를 직렬화하던 경합이 태스크 단위의 독립적인 커밋으로 바뀌었습니다. 애그리거트를 작게 만들라는 규칙이 코드를 정돈하라는 이야기가 아니라 동시성 설계에 대한 이야기였다는 것을 이때 이해했습니다.

jobId를 객체 참조가 아니라 ID로 들고 있게 한 것도 같은 이유였습니다. 객체를 직접 참조할 수 있으면 편하다는 이유로 상대 애그리거트를 그 자리에서 수정하게 되고, 트랜잭션이 슬그머니 두 애그리거트를 넘나들게 됩니다. ID로만 참조하면 그렇게 하고 싶어도 컴파일 단계에서 막힙니다.

5. 애그리거트 사이 — 도메인 이벤트와 결과적 일관성

남은 문제는 잘라낸 쪽이었습니다. "모든 태스크가 끝나면 잡은 완료된다"는 규칙은 여전히 지켜야 하는데, 이제 두 애그리거트에 걸쳐 있게 됐습니다. 이 틈은 도메인 이벤트로 이었습니다. PageTask.complete()가 기록한 PageTaskCompleted를 태스크 커밋 뒤에 발행하고, 핸들러가 별도의 트랜잭션으로 잡의 진행률을 갱신합니다.

# progress_handler.py — 애그리거트 사이는 이벤트로 잇는다
class PageTaskCompletedHandler:
    def handle(self, event: PageTaskCompleted) -> None:
        with self.uow:  # 여기서도 트랜잭션은 잡 하나만 만진다
            if self.uow.processed.contains(event.event_id):
                return  # 이벤트 재전송 대비 — 멱등 처리
            self.uow.jobs.apply_task_done(event.job_id)
            self.uow.processed.add(event.event_id)
            self.uow.commit()

여기서 두 가지를 챙겨야 결과적 일관성이 제대로 동작했습니다. 첫째는 멱등성입니다. 이벤트는 브로커 재전송이든 워커 재시작이든 언젠가 두 번 오게 되어 있어서, 처리한 이벤트 ID를 같은 트랜잭션에 기록해 두 번째 도착을 무시해야 카운트가 두 번 오르지 않습니다. 둘째는 갱신 자체의 원자성입니다. 진행률을 읽고–더하고–쓰는 세 단계로 짰더니 이번에는 핸들러끼리 경합했습니다. 그래서 카운터 증가와 완료 판정을 UPDATE 한 문장 안에 넣었습니다.

# job_repository.py 내부 — 읽고-더하고-쓰기 대신 UPDATE 한 문장
SQL = """
UPDATE crawl_job
   SET done_count = done_count + 1,
       status = CASE
                  WHEN done_count + 1 + failed_count = total_count
                  THEN 'COMPLETED' ELSE status
                END
 WHERE id = %(job_id)s
   AND status = 'RUNNING'
"""

def apply_task_done(self, job_id: JobId) -> None:
    self.session.execute(SQL, {"job_id": job_id})

마지막 태스크의 이벤트가 처리되는 순간 같은 문장 안에서 잡이 COMPLETED로 넘어가므로, "완료 판정을 누가 하나"라는 레이스도 데이터베이스의 행 잠금 안에서 정리됩니다. 이벤트 발행 자체의 유실이 걱정되는 지점에는 아웃박스 패턴 — 이벤트를 도메인 변경과 같은 트랜잭션으로 테이블에 적재하고 별도 프로세스가 발행 — 을 씁니다. 결과적 일관성은 값이 대충 맞아도 된다는 뜻이 아니라 멱등성과 원자적 갱신으로 언젠가 맞춰지도록 설계하는 일이라는 것을 이 과정에서 정리했습니다.

6. 경계를 검증하는 세 가지 질문

이 경계로 재설계한 뒤로는 어댑터를 늘려 사이트를 확장해도 트랜잭션 구조가 흔들리지 않았습니다. 워커를 늘리는 일이 더는 잠금 경합을 늘리는 일이 아니게 되었기 때문입니다. 그 뒤로 새 도메인을 모델링할 때는 세 가지 질문으로 경계를 점검하고 있습니다.

이 규칙은 커밋 순간에 참이어야 하는가? 아니라면 애그리거트 밖으로 — 이벤트와 결과적 일관성으로 보냅니다. 이 두 객체를 서로 다른 주체가 동시에 수정하는가? 그렇다면 같은 애그리거트에 두지 않습니다. 트랜잭션이 두 애그리거트를 함께 수정하고 있는가? 그렇다면 경계를 다시 봅니다 — 둘이 하나이거나, 둘 사이에 이벤트가 빠진 것입니다.

돌아보면 애그리거트 경계는 클래스 다이어그램 위가 아니라 트랜잭션과 잠금 위에 긋는 선에 가까웠습니다. 선을 어디에 긋는지가 코드의 모양보다 시스템의 동시성 성격을 먼저 정했습니다. 이 결정이 실제 파이프라인에서 어떻게 굴러갔는지는 아래 사례에 정리해 두었습니다.
