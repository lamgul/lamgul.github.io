DDD 애그리거트 경계 — 크롤링 파이프라인에 그은 선

2023년 8월• 약 10분• DDD · 설계 · 애그리거트

크롤링 파이프라인을 DDD로 재설계하면서 가장 오래 붙잡고 있던 질문은 클래스 이름도, 레이어 구조도 아니었습니다. "어디까지가 하나의 애그리거트인가"였습니다. 이 선을 잘못 그으면 코드가 아니라 트랜잭션이 무너집니다. 잘못 그었던 선과, 다시 그은 선을 기록합니다.

먼저 짚고 가기
애그리거트Aggregate
함께 저장하고 함께 지켜야 하는 객체들을 하나로 묶은 덩어리입니다. 마치 주문서 한 장과 그 안의 주문 항목들이 한 묶음으로 처리되는 것처럼요. 이 글은 "그 묶음을 어디까지로 잡을 것인가"를 다룹니다.
트랜잭션 경계transaction boundary
"이 작업들은 전부 성공하거나 전부 취소된다"가 보장되는 한 번의 저장 단위입니다. 은행 이체에서 출금과 입금이 한 세트로 처리되는 그 범위와 같습니다. 애그리거트는 곧 이 경계와 겹치도록 그어야 한다는 것이 이 글의 핵심입니다.
불변식invariant
어느 순간에도 반드시 참이어야 하는 규칙입니다. "잔액은 0원 밑으로 내려갈 수 없다" 같은 것이죠. 커밋되는 순간 함께 참이어야 하는 불변식이 무엇인지가 애그리거트 경계를 정합니다.
결과적 일관성eventual consistency
지금 당장은 값이 살짝 어긋나 있어도 잠시 뒤에는 반드시 맞춰진다는 방식입니다. SNS 좋아요 수가 몇 초 뒤 정확해지는 것을 떠올리면 됩니다. 즉시 맞을 필요가 없는 규칙은 이 방식으로 애그리거트 바깥에서 처리합니다.
멱등성idempotency
같은 작업을 여러 번 해도 결과가 한 번 한 것과 똑같다는 성질입니다. 엘리베이터 버튼을 열 번 눌러도 한 번 누른 것과 같은 것처럼요. 같은 이벤트가 두 번 도착해도 카운트가 두 번 오르지 않게 하는 안전장치입니다.

1. 애그리거트는 폴더가 아니라 트랜잭션 경계다

애그리거트를 "관련 있는 객체 묶음"으로 이해하면 경계를 그을 기준이 없어집니다. 크롤링 도메인에서는 잡·페이지·결과·로그가 전부 서로 관련 있으니까요. 에릭 에반스의 정의에서 실무적으로 중요한 부분은 묶음이 아니라 일관성 경계(consistency boundary)라는 쪽입니다. 애그리거트란 하나의 트랜잭션이 커밋되는 순간, 반드시 함께 참이어야 하는 불변식(invariant)의 범위입니다.

여기서 실전 규칙 세 가지가 따라 나옵니다. 반 버논(Vaughn Vernon)이 정리한 것과 같습니다.

첫째, 한 트랜잭션은 한 애그리거트만 수정한다. 둘째, 그래서 애그리거트는 작게 만든다. 셋째, 다른 애그리거트는 객체가 아니라 ID로 참조하고, 애그리거트 사이의 일관성은 결과적 일관성(eventual consistency)으로 맞춘다. 이 규칙이 왜 필요한지는 규칙을 어겨보면 가장 잘 보입니다. 제가 그랬습니다.

2. 잘못 그은 선 — 모든 것을 품은 CrawlJob

처음 모델은 직관적이었습니다. 잡이 페이지를 만들고 결과를 받으니, 잡이 전부 가지면 된다고 생각했습니다.

// crawl-job.ts — 잘못 그은 경계: 잡이 모든 것을 품는다
export class CrawlJob {
  private status: JobStatus = 'QUEUED';
  private readonly pages: PageTask[] = [];  // 잡 하나에 수천 건이 매달린다

  addPage(url: string) {
    if (this.status === 'COMPLETED') throw new DomainError('완료된 잡');
    this.pages.push(PageTask.queued(url));
  }

  recordResult(url: string, result: FetchResult) {
    const page = this.pages.find((p) => p.url === url);
    if (!page) throw new DomainError('등록되지 않은 페이지');
    page.complete(result);
    if (this.pages.every((p) => p.isTerminal())) this.status = 'COMPLETED';
  }
}

// 워커 쪽 — 결과 하나를 쓰려고 잡 전체를 든다
const job = await jobRepo.findById(jobId);  // 페이지 수천 건 로드
job.recordResult(url, result);
await jobRepo.save(job);  // 낙관적 잠금 버전 충돌 — 다른 워커도 같은 잡을 고치는 중

단위 테스트에서는 아름답게 통과합니다. 문제는 워커를 여러 개 띄우는 순간입니다. 애그리거트는 저장소에서 통째로 읽고 통째로 저장하는 단위이므로, 워커 열 개가 페이지 열 개를 각각 완료해도 전부 같은 CrawlJob 하나를 놓고 경합합니다. 낙관적 잠금이면 버전 충돌과 재시도가 쏟아지고, 비관적 잠금이면 워커들이 한 줄로 서서 순서를 기다립니다. 병렬로 긁으라고 만든 파이프라인이 잡 단위로 직렬화되는 것입니다.

증상은 동시성만이 아닙니다. 페이지 하나를 갱신하려고 수천 건을 로드하니 메모리와 쿼리가 낭비되고, "잡을 완료 처리하는 규칙"과 "페이지 하나를 재시도하는 규칙"이 한 클래스에 섞여 코드도 함께 비대해집니다. 경계가 크면 전부 같이 무거워집니다.

3. 진짜 불변식 찾기 — "지금 당장 참이어야 하는가"

경계를 다시 긋는 기준은 하나였습니다. 이 규칙은 커밋 순간에 반드시 참이어야 하는가, 아니면 잠시 어긋나도 괜찮은가. 크롤링 도메인의 규칙들을 이 질문으로 분류해 봤습니다.

커밋 순간에 참이어야 하는 것 — 페이지 태스크의 상태 전이입니다. FETCHING이 아닌 태스크에 결과가 붙으면 안 되고, 재시도 횟수는 상한을 넘으면 안 됩니다. 마찬가지로 잡의 상태 전이(QUEUED → RUNNING → COMPLETED/CANCELED)도 순간마다 유효해야 합니다. 이런 것이 트랜잭션 일관성의 대상입니다.

잠시 어긋나도 되는 것 — "잡의 완료 카운트 = 완료된 페이지 수" 같은 집계입니다. 페이지가 완료된 뒤 잡의 진행률에 반영되기까지 몇백 밀리초가 걸려도 아무도 다치지 않습니다. 진행률은 대시보드가 보는 숫자이지, 도메인이 그 순간에 걸고 넘어져야 할 규칙이 아닙니다. 이것이 결과적 일관성의 대상입니다.

이렇게 나누고 보니 처음 모델의 문제가 선명해졌습니다. 저는 결과적 일관성으로 충분한 집계 규칙(모든 페이지가 끝나면 잡을 완료로)을 트랜잭션 불변식으로 승격시켰고, 그 대가로 애그리거트가 도메인 전체를 삼킨 것입니다.

*

경계를 검증하는 실무 신호 두 가지. 서로 다른 워커·사용자가 동시에 수정하는 두 객체가 한 애그리거트 안에 있다면 경계가 너무 큽니다. 반대로 두 애그리거트를 한 트랜잭션에서 함께 고쳐야 하는 요구가 반복된다면 경계가 너무 잘게 잘렸거나 잘못 잘린 것입니다. 저는 정산 리팩토링에서도 같은 질문을 반복했는데, 도메인이 달라도 신호는 같았습니다.

4. 다시 그은 선 — 작게 나누고 ID로 참조한다

불변식 분류를 그대로 모델에 옮기면 애그리거트는 둘로 나뉩니다. 잡은 자신의 상태 전이만, 태스크는 자신의 상태 전이만 책임집니다.

// page-task.ts — 워커가 잠그는 것은 이제 태스크 하나뿐이다
export class PageTask {
  constructor(
    readonly id: PageTaskId,
    readonly jobId: CrawlJobId,  // 객체가 아니라 ID로 참조
    private status: TaskStatus,
    private attempts: number,
  ) {}

  complete(result: FetchResult) {
    if (this.status !== 'FETCHING')
      throw new DomainError(`${this.status}에서 완료 불가`);
    this.status = 'DONE';
    this.record(new PageTaskCompleted(this.id, this.jobId));  // 도메인 이벤트
  }

  fail(maxAttempts: number) {
    this.attempts += 1;  // 불변식: attempts는 상한을 넘지 않는다
    this.status = this.attempts >= maxAttempts ? 'FAILED' : 'QUEUED';
  }
}

// crawl-job.ts — 페이지 목록이 사라졌다
export class CrawlJob {
  private status: JobStatus = 'QUEUED';

  start() {
    if (this.status !== 'QUEUED') throw new DomainError('이미 시작된 잡');
    this.status = 'RUNNING';
  }

  cancel() {
    if (this.status === 'COMPLETED') throw new DomainError('완료된 잡은 취소 불가');
    this.status = 'CANCELED';
  }
}

바뀐 것을 정리하면 이렇습니다. 워커가 결과 하나를 기록할 때 로드하고 저장하는 것은 PageTask 한 건입니다. 워커 수십 개가 서로 다른 태스크를 물고 있으면 트랜잭션은 전혀 겹치지 않습니다. 잡 전체를 직렬화하던 경합이, 태스크 단위의 독립적인 커밋으로 바뀝니다. 애그리거트를 작게 만들라는 규칙은 미학이 아니라 동시성 설계였던 것입니다.

jobId를 객체 참조가 아니라 ID로 든 것도 같은 이유입니다. 객체를 직접 참조하면 "편하니까" 상대 애그리거트를 그 자리에서 수정하게 되고, 트랜잭션이 슬그머니 두 애그리거트를 넘나들기 시작합니다. ID 참조는 그 유혹을 컴파일 타임에 끊는 장치입니다.

5. 애그리거트 사이 — 도메인 이벤트와 결과적 일관성

남은 숙제는 잘라낸 쪽입니다. "모든 태스크가 끝나면 잡은 완료된다"는 여전히 지켜야 할 규칙이고, 이제 두 애그리거트에 걸쳐 있습니다. 이 틈을 잇는 것이 도메인 이벤트입니다. PageTask.complete()가 기록한 PageTaskCompleted를 태스크 커밋 뒤에 발행하고, 핸들러가 별도의 트랜잭션으로 잡의 진행률을 갱신합니다.

# progress_handler.py — 애그리거트 사이는 이벤트로 잇는다
class PageTaskCompletedHandler:
    def handle(self, event: PageTaskCompleted) -> None:
        with self.uow:  # 여기서도 트랜잭션은 잡 하나만 만진다
            if self.uow.processed.contains(event.event_id):
                return  # 이벤트 재전송 대비 — 멱등 처리
            self.uow.jobs.apply_task_done(event.job_id)
            self.uow.processed.add(event.event_id)
            self.uow.commit()

여기서 두 가지가 결과적 일관성의 품질을 결정합니다. 첫째는 멱등성입니다. 이벤트는 브로커 재전송이든 워커 재시작이든 언젠가 두 번 옵니다. 처리한 이벤트 ID를 같은 트랜잭션에서 기록해 두 번째 도착을 무시해야 카운트가 두 번 오르지 않습니다. 둘째는 갱신 자체의 원자성입니다. 진행률을 읽고–더하고–쓰는 세 단계로 짜면 핸들러끼리 또 경합합니다. 저는 카운터 증가와 완료 판정을 UPDATE 한 문장으로 밀어 넣었습니다.

# job_repository.py 내부 — 읽고-더하고-쓰기 대신 UPDATE 한 문장
SQL = """
UPDATE crawl_job
   SET done_count = done_count + 1,
       status = CASE
                  WHEN done_count + 1 + failed_count = total_count
                  THEN 'COMPLETED' ELSE status
                END
 WHERE id = %(job_id)s
   AND status = 'RUNNING'
"""

def apply_task_done(self, job_id: JobId) -> None:
    self.session.execute(SQL, {"job_id": job_id})

마지막 태스크의 이벤트가 처리되는 순간 같은 문장 안에서 잡이 COMPLETED로 넘어가므로, "완료 판정을 누가 하나"라는 레이스도 데이터베이스의 행 잠금 안에서 정리됩니다. 이벤트 발행 자체의 유실이 걱정되는 지점에는 아웃박스 패턴 — 이벤트를 도메인 변경과 같은 트랜잭션으로 테이블에 적재하고 별도 프로세스가 발행 — 을 씁니다. 요컨대 결과적 일관성은 "대충 맞는다"가 아니라, 멱등성과 원자적 갱신으로 언젠가 반드시 맞는다를 설계하는 일입니다.

6. 경계를 검증하는 세 가지 질문

이 경계로 재설계한 파이프라인은 어댑터를 늘려 사이트를 확장해도 트랜잭션 구조가 흔들리지 않았습니다. 워커를 늘리는 일이 잠금 경합을 늘리는 일이 아니게 되었기 때문입니다. 새 도메인을 모델링할 때마다 저는 이제 세 질문으로 경계를 검증합니다.

이 규칙은 커밋 순간에 참이어야 하는가? 아니라면 애그리거트 밖으로 — 이벤트와 결과적 일관성으로 보냅니다. 이 두 객체를 서로 다른 주체가 동시에 수정하는가? 그렇다면 같은 애그리거트에 두지 않습니다. 트랜잭션이 두 애그리거트를 함께 수정하고 있는가? 그렇다면 경계를 다시 봅니다 — 둘이 하나이거나, 둘 사이에 이벤트가 빠진 것입니다.

애그리거트 경계는 클래스 다이어그램에 긋는 선이 아니라 트랜잭션과 잠금에 긋는 선입니다. 선을 어디에 긋느냐가 코드의 모양보다 시스템의 동시성 성격을 먼저 결정합니다. 이 결정이 실제 파이프라인에서 어떻게 굴러갔는지는 아래 사례에 정리해 두었습니다.
