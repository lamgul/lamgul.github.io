PostgreSQL 느린 목록 화면 — 인덱스와 keyset 페이지네이션

2025년 11월• 약 8분• PostgreSQL · 인덱스 · 페이지네이션

"목록 뒤로 갈수록 느려져요." 흔한 민원이고, 대개 범인은 정해져 있습니다. 인덱스 없는 정렬과 OFFSET 페이지네이션. 추측 대신 EXPLAIN ANALYZE부터 켰습니다.

먼저 짚고 가기
실행 계획EXPLAIN ANALYZE
DB에게 "이 쿼리를 어떤 순서로 처리할 셈이었고, 실제로 얼마나 걸렸는지" 계획서를 받아 보는 명령입니다. 자동차 내비게이션이 경로와 소요 시간을 보여 주는 것과 비슷합니다. 느린 화면을 고칠 때 추측 대신 여기서부터 출발합니다.
인덱스index
책 뒤의 색인처럼, 특정 값을 가진 행을 페이지를 전부 넘기지 않고 바로 찾게 해 주는 보조 자료입니다. 인덱스가 없으면 DB는 조건에 맞는 행을 찾으려고 테이블을 처음부터 끝까지 훑습니다.
복합 인덱스composite index
여러 컬럼을 묶어 하나로 만든 인덱스입니다. 전화번호부가 "성 → 이름" 순서로 정렬돼 있듯, 컬럼을 나열한 순서가 성능을 좌우합니다. 이 글의 핵심 장면 중 하나입니다.
OFFSET 페이지네이션OFFSET pagination
"앞의 N개는 건너뛰고 그다음부터 달라"는 방식의 페이지 나누기입니다. 편하지만, 건너뛸 N개를 매번 실제로 세고 지나가야 해서 뒤 페이지로 갈수록 느려집니다.
keyset 페이지네이션keyset / seek
"마지막으로 본 값 다음부터 달라"는 방식입니다. 읽던 책에 끼워 둔 책갈피에서 이어 읽는 것과 같아서, 몇 페이지째든 속도가 일정합니다. OFFSET의 대안으로 뒤에서 자세히 다룹니다.

먼저 실행 계획을 읽는다

EXPLAIN ANALYZE
SELECT * FROM reports
WHERE site_id = 42
ORDER BY created_at DESC
LIMIT 20 OFFSET 4000;
-- Seq Scan + Sort (actual time=210..214)  ← 전량 스캔 후 정렬

Seq Scan에 Sort가 붙으면 신호입니다. 조건에 맞는 행을 전부 읽어 정렬한 뒤 20개만 버리고 가져오는 겁니다. 4000번째부터 보면 앞의 4020개를 매번 다시 만듭니다. 뒤 페이지일수록 버리는 양이 늘어나니, 페이지가 깊어질수록 느려지는 겁니다.

복합 인덱스는 순서가 전부

-- WHERE로 좁히고(site_id) 그 안에서 정렬(created_at)
CREATE INDEX idx_reports_site_created
  ON reports (site_id, created_at DESC);

(site_id, created_at DESC) 순서가 핵심입니다. 등치 조건 컬럼이 앞, 정렬 컬럼이 뒤. 이러면 인덱스만 타고 정렬 없이 상위 N개를 바로 집습니다. 순서를 뒤집으면 인덱스가 놀아요.

OFFSET을 버리고 keyset으로

인덱스를 깔아도 OFFSET 4000은 여전히 앞 4000개를 훑어 버립니다. 깊은 페이지가 근본적으로 느린 이유죠. 그래서 keyset(seek) 페이지네이션으로 바꿉니다 — "마지막으로 본 값 다음부터".

-- 직전 페이지 마지막 (created_at, id)를 커서로
SELECT * FROM reports
WHERE site_id = 42
  AND (created_at, id) < ('2025-11-01 09:00', 88123)
ORDER BY created_at DESC, id DESC
LIMIT 20;   -- 몇 페이지든 일정하게 빠름
*

keyset은 "3페이지로 점프" 같은 임의 접근을 포기하는 대신, 어느 깊이에서든 일정한 속도를 얻습니다. 무한 스크롤·타임라인에는 완벽하고, 페이지 번호 UI에는 안 맞습니다. 트레이드오프를 알고 고르는 게 중요합니다.

덤: N+1도 같이 잡는다

목록이 느린 또 다른 흔한 이유는 각 행마다 연관을 다시 조회하는 N+1입니다. ORM의 지연 로딩을 믿지 말고 JOIN이나 배치 로딩으로 한 방에 가져오세요. 쿼리 로그를 켜서 "화면 하나에 쿼리 몇 개 나가나"를 세는 습관이 반쯤은 답입니다.
