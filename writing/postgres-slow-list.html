PostgreSQL 느린 목록 화면 — 인덱스와 keyset 페이지네이션

2025년 11월• PostgreSQL · 인덱스 · 페이지네이션

"목록 뒤로 갈수록 느려져요"라는 문의를 받은 적이 있습니다. 짐작 가는 데는 있었지만 확인해 보기 전에는 알 수 없어서 EXPLAIN ANALYZE부터 돌려 봤고, 원인은 인덱스 없는 정렬과 OFFSET 페이지네이션이었습니다. 그때 따라간 순서를 적어 둡니다.

먼저 짚고 가기
실행 계획EXPLAIN ANALYZE
쿼리를 실제로 실행한 뒤, 플래너가 세운 실행 계획 트리와 노드별 실측치를 함께 출력하는 명령입니다. EXPLAIN만 쓰면 추정 비용(cost는 startup..total 형태)과 예상 행 수만 나오고, ANALYZE 옵션을 붙여야 노드마다 실제 소요 시간(actual time)·행 수(rows)·반복 횟수(loops)가 찍힙니다. 이때 actual time과 rows는 루프 한 번분의 값이라, 안쪽 노드가 여러 번 도는 경우 loops를 곱해야 총량이 됩니다. 추정 행 수와 실제 행 수가 크게 벌어진 노드가 대개 문제의 출발점이며, 쓰기 쿼리에 붙일 때는 실제로 실행되므로 트랜잭션으로 감싸 롤백해야 합니다.
인덱스index
테이블과 별도로 유지되는 보조 자료구조로, PostgreSQL의 기본형은 B-tree입니다. 키 값을 정렬된 상태로 보관하고 각 키마다 실제 행의 물리 위치(ctid)를 함께 저장해, 테이블 전체를 훑는 대신 트리를 로그 단계만큼만 내려가 대상 행에 도달합니다. 대신 행을 삽입·갱신할 때마다 인덱스도 같이 갱신되어 쓰기 비용과 저장 공간이 늘고, 조건이 테이블의 큰 비율을 고르는 경우에는 랜덤 접근이 오히려 비싸져 플래너가 일부러 순차 스캔을 택하기도 합니다.
복합 인덱스composite index
두 개 이상의 컬럼을 하나의 키로 묶어, 컬럼을 나열한 순서대로 사전식 정렬을 유지하는 인덱스입니다. 선행 컬럼이 등치 조건으로 고정되면 후행 컬럼은 그 구간 안에서 이미 정렬된 상태가 되어, (site_id, created_at)는 site_id로 좁히고 created_at으로 정렬하는 쿼리에서 Sort 노드를 통째로 없애 줍니다. 다만 왼쪽 컬럼부터 연속으로 조건이 걸려야 탐색 시작점을 잡을 수 있어(leftmost prefix), 선행 컬럼을 건너뛰고 후행 컬럼만 거는 조건은 인덱스가 있어도 이점이 거의 없습니다.
OFFSET 페이지네이션OFFSET pagination
LIMIT n OFFSET m으로 조건에 맞는 행을 정렬한 뒤 앞의 m개를 건너뛰고 그다음 n개만 돌려받는 방식입니다. 서버는 건너뛸 m개도 실제로 만들어 낸 다음 버리므로 읽는 비용이 m+n에 비례하고, m이 커지는 깊은 페이지일수록 선형으로 느려집니다. 페이지 번호로 임의 접근이 되는 게 장점이지만, 페이지를 넘기는 사이 앞쪽 행이 추가·삭제되면 경계가 밀려 같은 행을 두 번 보거나 건너뛰게 됩니다.
keyset 페이지네이션keyset / seek
직전 페이지 마지막 행의 정렬 키를 커서로 삼아, WHERE (created_at, id) < ('…', …) 같은 튜플 비교로 읽기 시작 지점을 직접 지정하는 방식입니다. 건너뛸 행을 세는 대신 인덱스에서 시작 위치로 곧장 이동해 n개만 읽으므로, 한 페이지를 가져오는 비용이 인덱스 탐색 한 번에 n행 읽기(log N + n)로 고정되어 페이지 깊이와 무관합니다. 다만 정렬 키가 행마다 유일해야 하고(동점을 깨는 컬럼을 함께 묶어야 하고), 비교 순서가 인덱스 정렬 순서와 맞아야 하며, 임의의 페이지 번호로 건너뛰는 건 포기해야 합니다. 읽던 자리에 끼운 책갈피에서 이어 읽는 쪽에 가깝습니다.

먼저 실행 계획을 읽는다

EXPLAIN ANALYZE
SELECT * FROM reports
WHERE site_id = 42
ORDER BY created_at DESC
LIMIT 20 OFFSET 4000;
-- Seq Scan + Sort (actual time=210..214)  ← 전량 스캔 후 정렬

Seq Scan 옆에 Sort가 붙어 있는 걸 보고 원인을 짐작했습니다. 조건에 맞는 행을 전부 읽어 정렬한 뒤, 그중 20개만 남기고 나머지는 버린다는 뜻이었습니다. 4000번째부터 보려면 앞의 4020개를 매번 다시 만들어야 하고, 뒤 페이지로 갈수록 버리는 양이 늘어나니 페이지가 깊어질수록 느려집니다.

복합 인덱스는 순서가 전부

-- WHERE로 좁히고(site_id) 그 안에서 정렬(created_at)
CREATE INDEX idx_reports_site_created
  ON reports (site_id, created_at DESC);

인덱스는 (site_id, created_at DESC) 순서로 만들었습니다. 등치 조건으로 쓰는 컬럼을 앞에, 정렬에 쓰는 컬럼을 뒤에 두는 순서입니다. 이렇게 두면 인덱스를 따라가면서 별도 정렬 없이 상위 N개를 바로 집어 올 수 있습니다. 순서를 뒤집으면 같은 인덱스라도 정렬에 쓰이지 못해 제 몫을 못 합니다.

OFFSET을 버리고 keyset으로

인덱스를 만든 뒤에도 OFFSET 4000은 앞 4000개를 그대로 훑고 지나갔습니다. 깊은 페이지가 느린 이유가 여기 있어서, 페이지를 넘기는 방식 자체를 keyset(seek) 페이지네이션으로 바꿨습니다. 마지막으로 본 값 다음부터 읽어 오는 방식입니다.

-- 직전 페이지 마지막 (created_at, id)를 커서로
SELECT * FROM reports
WHERE site_id = 42
  AND (created_at, id) < ('2025-11-01 09:00', 88123)
ORDER BY created_at DESC, id DESC
LIMIT 20;   -- 몇 페이지든 일정하게 빠름
*

keyset은 "3페이지로 점프" 같은 임의 접근을 포기하는 대신 어느 깊이에서든 일정한 속도를 얻는 방식입니다. 무한 스크롤이나 타임라인에는 잘 맞고, 페이지 번호를 그대로 노출하는 UI에는 맞지 않습니다. 어느 쪽을 고를지는 화면이 페이지를 어떻게 보여 주느냐에 달려 있습니다.

덤: N+1도 같이 잡는다

목록이 느린 또 다른 이유로 각 행마다 연관 데이터를 다시 조회하는 N+1이 있습니다. ORM의 지연 로딩에 맡겨 두면 눈에 띄지 않게 늘어나서, JOIN이나 배치 로딩으로 한 번에 가져오도록 바꿨습니다. 그 뒤로는 쿼리 로그를 켜 두고 화면 하나에 쿼리가 몇 개 나가는지 세어 보는 편입니다.
