회고: 2020년부터의 학습 지도 — 무엇을 왜 공부했나

2026년 6월• 회고 · 학습 · 커리어

2020년에는 남이 준 데이터를 읽고 다듬는 일을 했고, 지금은 온프레미스 GPU에 LLM을 올려 두고 운영합니다. 그 사이에 무엇을 왜 공부했는지 한 번 정리해 두고 싶어서 적습니다. 돌아보면 순서를 정한 건 계획이 아니라 그때그때 앞에 놓인 문제였습니다. 각 지점에서 따로 쓴 글은 본문에 링크로 달아 두었습니다.

먼저 짚고 가기
피처 엔지니어링Feature Engineering
원시 데이터를 모델이 학습하기 좋은 새 변수로 다듬는 일입니다. 예컨대 기온·습도라는 날것 대신 '불쾌지수'를 만들어 넣는 식이죠. 본문에서 "데이터를 얼마나 이해했는가"를 이야기하는 대목이 여기에 해당합니다.
역전파backpropagation
신경망이 답을 틀린 정도를 거꾸로 되짚어, 각 부품(가중치)을 얼마나 고쳐야 할지 계산하는 과정입니다. 시험 오답을 뒤에서부터 되짚어 어디서 틀렸는지 찾는 것과 비슷합니다. 라이브러리 뒤에 있는 이 계산을 손으로 따라가 본 이야기가 본문 '바닥부터 보고 싶었다' 절에 나옵니다.
애그리거트Aggregate
함께 바뀌어야 하는 데이터들을 한 덩어리로 묶어, 그 안에서만 규칙(불변식)을 지키게 하는 단위입니다. 주문과 주문 항목을 따로 놀게 두지 않고 '주문' 하나로 다루는 식입니다. 경계를 어디에 그을지가 이 글의 크롤링 파이프라인 이야기의 중심입니다.
부패 방지 계층ACL, Anti-Corruption Layer
바깥 시스템의 낯선 데이터 모양이 내 도메인 안으로 새어 들지 못하게, 문 앞에서 내 언어로 번역해 주는 중간층입니다. 외국어 손님을 통역사가 걸러 응대하는 것과 같습니다. 사이트마다 제각각인 크롤링 응답을 하나로 모을 때 이 계층이 오염을 막아 줍니다.
OOMKilledOut Of Memory Killed
컨테이너가 허용된 메모리 한계를 넘으면 쿠버네티스가 그 프로세스를 강제로 종료시키는 것을 말합니다. 정해진 방보다 짐이 많으면 쫓겨나는 셈이죠. 온프레미스에 모델을 올릴 때 메모리 한계(limits)를 잘못 잡으면 파드가 조용히 죽는 이유가 이것입니다.

2020: 데이터 가공에서 출발

처음 맡은 일은 데이터를 다루는 쪽이었습니다. 남이 준 원시 데이터를 읽고, 씻고, 쓸 수 있는 모양으로 바꾸는 일부터 배웠습니다. 매번 손으로 만들던 보고서를 pandas·openpyxl로 자동화해 본 것도 이때인데, 코드로 시간을 되살 수 있다는 걸 그때 처음 실감했습니다.

데이터를 오래 들여다보며 얻은 생각은, 결과를 가르는 게 모델보다 데이터를 얼마나 이해했는가에 가깝다는 것이었습니다. 나중에 전력 예측 문제에서 원시 기상값 대신 불쾌지수 같은 파생 변수를 만들었던 Feature Engineering 실전도 같은 생각의 연장이었습니다. 도구는 여러 번 바뀌었지만 데이터를 한 번 의심해 보고 가공한다는 습관은 그대로 남아 있습니다.

바닥부터 보고 싶었다 — Vision AI

ETRI에서 연구연수로 CNN 기반 객체탐지를 다루던 때입니다. 딥러닝을 쓰는 일과 이해하는 일이 같지 않다는 걸 이 무렵에 알았습니다. 코드는 돌아갔지만 라이브러리가 대신 해주는 계산을 제 입으로 설명하지 못하는 게 계속 걸려서, 사이토 고키의 「밑바닥부터 시작하는 딥러닝」을 집어 들었습니다.

이 책은 넘파이만으로 신경망을 처음부터 쌓아 올립니다. 가장 오래 남은 건 계산 그래프(computational graph)로 역전파를 손으로 따라가는 장이었습니다. 곱셈·덧셈 노드의 국소적 미분이 연쇄법칙으로 이어지는 걸 코드로 확인하고 나서야, '기울기'가 비유가 아니라 계산이라는 게 이해됐습니다.

# 「밑바닥부터 시작하는 딥러닝」의 곱셈 계층 — 역전파가 곧 국소 미분
class MulLayer:
    def forward(self, x, y):
        self.x, self.y = x, y
        return x * y
    def backward(self, dout):
        # x와 y를 서로 바꿔 곱한다 — 그게 곱셈 노드의 국소 미분
        return dout * self.y, dout * self.x

책 뒤쪽에는 합성곱을 im2col로 펼쳐 행렬곱 하나로 바꾸는 장이 나옵니다. 객체탐지에서 별생각 없이 쓰던 conv 레이어의 속을 그제야 납득했습니다. 이 시기에 읽은 논문은 따로 정리해 두었습니다. 검출을 회귀 문제로 바꾼 YOLO와, 이후로도 계속 다시 보게 된 Attention입니다.

*

라이브러리를 못 믿어서 바닥을 판 건 아니었습니다. 추상화를 믿고 쓰려면 한 번쯤은 그 아래를 봐 두는 편이 낫다고 생각했을 뿐입니다. 실제로 한 번 보고 나니 그 뒤로는 오히려 마음 놓고 프레임워크를 쓰게 됐습니다.

제품을 짜려면 설계가 필요했다

대흥정보에서 SM과 웹개발을 맡으면서, 연구실에서 쓰던 코드와 남이 유지보수할 제품 코드의 기준이 다르다는 걸 알게 됐습니다. 돌아가기만 하는 코드와 읽히는 코드 사이의 거리를 매일 확인하는 자리였습니다. 로버트 마틴의 「클린 코드」를 정독한 것도 이 시기입니다.

책이 하는 이야기는 단순합니다. 함수는 한 가지 일만 하고, 이름은 의도를 드러내며, 주석보다 코드가 말하게 하라는 것입니다. 처음 읽을 때는 당연한 소리 같았는데, 남이 짜 둔 레거시를 고치다 보니 이 당연한 것들이 지켜지지 않으면 하루가 통째로 사라진다는 걸 알게 됐습니다. 장별로 곱씹은 기록은 네 편으로 남겨 두었습니다. 이름, 함수, 에러, 테스트입니다.

같은 시기에 프레임워크가 감춰 둔 규칙도 하나씩 들여다봤습니다. 스프링의 트랜잭션 전파가 그런 쪽이었는데, 얼핏 알아서 되는 것처럼 보이지만 규칙을 모르면 데이터가 조용히 어긋나는 지점이었습니다. 구조를 보는 눈도 이때 조금 자랐습니다. 레이어드에서 헥사고날로 넘어가면서 의존성의 방향이 곧 설계라는 걸 배웠습니다.

고치기 쉽게 먼저 만든다

제품을 오래 만지다 보면 새로 만드는 일보다 고치는 일이 많아집니다. 여기저기 흩어진 정산 로직을 손대야 했을 때 마틴 파울러의 「리팩터링」을 참고했습니다. 이 책이 말하는 건 기교보다 순서였습니다. 테스트로 안전망을 친 뒤에, 겉보기 동작을 바꾸지 않는 작은 변형을 쌓아 간다는 것입니다.

파울러는 '긴 함수'나 '중복'처럼 손봐야 할 신호를 코드 냄새(code smell)로 부르고, 그때마다 쓸 표준 동작(함수 추출, 이름 바꾸기 등)을 카탈로그로 정리해 둡니다. 정산 코드에서는 종류별로 20줄씩 늘어붙어 있던 분기를 의도 단위의 함수로 뽑아내고, 종류는 전략으로 갈랐습니다.

// Before — 한 메서드에 정산 종류별 분기·계산이 뒤엉킨다
if (type == MONTHLY) { /* 20여 줄 */ }
else if (type == USAGE) { /* 또 20여 줄 */ }

// After — 의도 단위로 함수를 뽑고(Extract Function), 종류를 전략으로
Money fee = policyOf(type).calculate(usage);

결과는 정산 시스템 모듈화 사례에 남겼고, 패턴 쪽 정리는 Template Method와 Strategy의 조합에 따로 적어 두었습니다. 켄트 벡이 말한 변경을 쉽게 만든 다음 그 쉬운 변경을 하라는 문장이 오래 남았는데, 리팩토링과 기능 추가를 한 번에 섞지 말라는 뜻으로 읽었습니다.

경계를 긋는 법 — 크롤링과 DDD

여러 사이트를 긁어 하나의 파이프라인으로 모으는 일은 코드보다 경계를 잡는 쪽이 어려웠습니다. 사이트마다 다른 데이터 모양이 도메인 안으로 그대로 들어오면 안쪽 코드까지 같이 지저분해집니다. 에릭 에반스의 「도메인 주도 설계」에서 두 개념을 가져다 썼습니다.

하나는 애그리거트로, 함께 바뀌어야 하는 것들을 한 덩어리로 묶고 불변식을 그 경계 안에서만 지키는 단위입니다. 다른 하나는 부패 방지 계층(ACL)으로, 외부의 모델을 내 도메인 언어로 번역하는 어댑터를 문 앞에 세우는 방식입니다. 크롤러에 둔 어댑터가 이 계층에 해당했습니다.

# 부패 방지 계층 — 사이트별 원시 응답을 도메인 모델로 '번역'한다
class SourceAdapter(Protocol):
    def fetch(self, q: Query) -> list[RawItem]: ...
    def to_domain(self, raw: RawItem) -> Listing: ...  # 외부 모양을 아는 건 여기까지

이 어댑터를 런타임에 갈아끼우는 방법은 FastAPI 의존성 주입에, 차단당하지 않고 오래 도는 방법은 Adapter와 지수 백오프에 정리해 두었습니다. 어디에 선을 그을지에 대한 판단은 애그리거트 경계 글에, 전체 그림은 DDD 크롤링 파이프라인 사례에 적었습니다.

*

에반스가 말한 보편 언어(ubiquitous language)는 문서에 붙이는 용어집이 아니었습니다. 코드의 클래스명과 회의에서 쓰는 말이 같아지면 그 사이를 번역하는 비용이 줄어듭니다. 경계를 긋는 이유도 그 정도로 실용적인 것이었습니다.

혼자 다 지는 자리 — SafeIn과 온프레미스

SafeIn에서는 백엔드를 혼자 설계하고 혼자 운영했습니다. 설계하는 사람과 운영하는 사람이 같다 보니 그동안 배운 것들을 한꺼번에 꺼내 쓰게 됐습니다. 관제와 앱이 같은 데이터를 다르게 쓰기 때문에 권한을 역할별로 갈랐고(Guard·DTO), 새벽에 깨는 일을 줄이려고 로그·메트릭·트레이스를 먼저 깔아 두었습니다.

산업안전 현장의 데이터는 밖으로 내보낼 수 없어서, 모델도 온프레미스 GPU에 직접 올려야 했습니다. 인프라를 아는 일이 부수적인 관심사가 아니라 요구사항이 되는 자리였습니다. 컨테이너에는 쓸 자원과 넘으면 안 되는 한계를 함께 지정해 줘야 했습니다.

# 컨테이너에 '살 방'과 '못 넘을 선'을 함께 준다
resources:
  requests: { memory: "4Gi", cpu: "1" }   # 스케줄러가 보장하는 하한
  limits:   { memory: "6Gi" }             # 이 선을 넘으면 OOMKilled

이 값을 잘못 잡으면 파드가 별다른 메시지 없이 죽습니다. 왜 그렇게 되는지는 requests/limits와 OOMKilled에 정리했고, 컨테이너가 아니라 힙 쪽이 문제일 때 본 것들은 힙덤프 읽는 법에 적어 두었습니다. 모델 서빙은 vLLM으로 했습니다. 왜 빠른지는 PagedAttention 논문 분석에, 실제로 운영한 기록은 온프레미스 LLM 서빙 글과 SafeIn 사례에 있습니다. 이 시기에 옆에 두고 읽은 책은 마틴 클레프만의 「데이터 중심 애플리케이션 설계」였는데, 복제와 파티셔닝, 일관성을 한 번 훑고 나니 인프라 선택이 취향이 아니라 트레이드오프로 보이기 시작했습니다.

정리 — 무엇을 왜 공부했나

돌아보면 순서는 대체로 비슷했습니다. 늘 문제가 먼저 오고 공부가 뒤따랐습니다. 데이터가 맞지 않아서 EDA를 봤고, 딥러닝이 이해되지 않아서 바닥을 봤고, 코드가 잘 고쳐지지 않아서 리팩토링을, 경계가 흐려져서 DDD를, 서버가 죽어서 인프라를 들여다봤습니다. 책은 그때마다 누군가 먼저 정리해 둔 길을 빌려 읽은 것에 가까웠습니다.

  • 데이터를 이해하는 일 → Feature Engineering, 보고서 자동화
  • 바닥부터 보는 일 → YOLO · Attention · PagedAttention 논문
  • 읽히는 코드 → 클린코드 함수, UseCase·Repository
  • 경계를 긋는 일 → DDD 애그리거트, 어댑터·백오프
  • 운영까지 지는 일 → 온프레미스 서빙, OOMKilled, 힙덤프

기술 이름을 나열하는 방식으로는 제가 어떤 사람인지 잘 설명되지 않는 것 같습니다. 대신 모르는 걸 만나면 한 겹 아래를 보는 편이라고 적어 두고 싶습니다. 그 습관 덕에 2020년의 데이터 한 장에서 온프레미스 GPU까지 오게 됐고, 다음 문제도 비슷하게 넘어가지 않을까 싶습니다.
