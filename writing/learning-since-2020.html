회고: 2020년부터의 학습 지도 — 무엇을 왜 공부했나

2026년 6월• 약 12분• 회고 · 학습 · 커리어

어떤 기술을 아느냐보다 왜 그걸 공부했느냐가 저를 더 정확히 설명합니다. 2020년 데이터를 만지던 사람이 온프레미스 GPU에 LLM을 올리기까지, 매번 앞에 놓인 문제가 다음 공부를 정했습니다. 이 글은 그 순서를 과장 없이 그린 지도이자, 각 지점에서 따로 쓴 글로 가는 이정표입니다.

먼저 짚고 가기
피처 엔지니어링Feature Engineering
원시 데이터를 모델이 학습하기 좋은 새 변수로 다듬는 일입니다. 예컨대 기온·습도라는 날것 대신 '불쾌지수'를 만들어 넣는 식이죠. 이 글에서 "데이터를 얼마나 이해했는가"가 결과를 가른다는 첫 확신이 바로 여기서 나옵니다.
역전파backpropagation
신경망이 답을 틀린 정도를 거꾸로 되짚어, 각 부품(가중치)을 얼마나 고쳐야 할지 계산하는 과정입니다. 시험 오답을 뒤에서부터 되짚어 어디서 틀렸는지 찾는 것과 비슷합니다. 저자가 라이브러리 뒤에 감춰진 이 계산을 손으로 따라간 대목이 이 글의 '바닥부터 보기'의 핵심입니다.
애그리거트Aggregate
함께 바뀌어야 하는 데이터들을 한 덩어리로 묶어, 그 안에서만 규칙(불변식)을 지키게 하는 단위입니다. 주문과 주문 항목을 따로 놀게 두지 않고 '주문' 하나로 다루는 식입니다. 경계를 어디에 그을지가 이 글의 크롤링 파이프라인 이야기의 중심입니다.
부패 방지 계층ACL, Anti-Corruption Layer
바깥 시스템의 낯선 데이터 모양이 내 도메인 안으로 새어 들지 못하게, 문 앞에서 내 언어로 번역해 주는 중간층입니다. 외국어 손님을 통역사가 걸러 응대하는 것과 같습니다. 사이트마다 제각각인 크롤링 응답을 하나로 모을 때 이 계층이 오염을 막아 줍니다.
OOMKilledOut Of Memory Killed
컨테이너가 허용된 메모리 한계를 넘으면 쿠버네티스가 그 프로세스를 강제로 종료시키는 것을 말합니다. 정해진 방보다 짐이 많으면 쫓겨나는 셈이죠. 온프레미스에 모델을 올릴 때 메모리 한계(limits)를 잘못 잡으면 파드가 조용히 죽는 이유가 이것입니다.

2020: 데이터 가공에서 출발

처음 붙잡은 건 거창한 프레임워크가 아니라 데이터였습니다. 남이 준 원시 데이터를 읽고, 씻고, 쓸 수 있는 모양으로 바꾸는 일부터 배웠습니다. 반복되는 수작업 보고서를 pandas·openpyxl로 자동화하며, 코드가 시간을 되사는 감각을 처음 익혔습니다.

데이터를 오래 들여다보며 얻은 첫 확신은 분명했습니다 — 결과를 가르는 건 화려한 모델이 아니라 데이터를 얼마나 이해했는가였습니다. 뒷날 전력 예측 문제에서 원시 기상값 대신 불쾌지수 같은 파생 변수를 만든 Feature Engineering 실전은 이 확신의 연장선이었습니다. 도구는 바뀌어도, 데이터를 의심하고 가공한다는 태도는 그대로 남았습니다.

바닥부터 보고 싶었다 — Vision AI

딥러닝을 쓰는 것과 이해하는 것은 다릅니다. ETRI에서 CNN 기반 객체탐지를 연구연수로 다루며, 라이브러리가 대신 해주던 계산을 스스로 설명하지 못한다는 불편함이 커졌습니다. 그래서 집어 든 책이 사이토 고키의 「밑바닥부터 시작하는 딥러닝」이었습니다.

이 책은 넘파이만으로 신경망을 처음부터 쌓습니다. 제게 가장 크게 남은 건 계산 그래프(computational graph)로 역전파를 손으로 따라가는 장이었습니다. 곱셈·덧셈 노드의 국소적 미분이 연쇄법칙으로 이어지는 걸 코드로 보고 나서야, '기울기'가 비유가 아니라 계산이라는 게 손에 잡혔습니다.

# 「밑바닥부터 시작하는 딥러닝」의 곱셈 계층 — 역전파가 곧 국소 미분
class MulLayer:
    def forward(self, x, y):
        self.x, self.y = x, y
        return x * y
    def backward(self, dout):
        # x와 y를 서로 바꿔 곱한다 — 그게 곱셈 노드의 국소 미분
        return dout * self.y, dout * self.x

책의 뒤로 가면 합성곱을 im2col로 펼쳐 거대한 행렬곱 하나로 바꾸는 장이 나옵니다. 객체탐지에서 당연하게 쓰던 conv 레이어의 속을 그제야 납득했습니다. 이 시기에 파고든 논문들은 따로 정리해 두었습니다 — 검출을 회귀 문제로 바꾼 YOLO, 그리고 두고두고 다시 만나게 되는 Attention까지.

*

라이브러리를 못 믿어서 바닥을 판 게 아닙니다. 추상화를 신뢰하려면 한 번은 그 아래를 봐야 한다고 믿습니다. 바닥을 한 번 보고 나면, 그 뒤로는 오히려 마음 놓고 프레임워크를 쓰게 됩니다.

제품을 짜려면 설계가 필요했다

연구실의 코드와 남이 유지보수할 제품 코드는 기준이 달랐습니다. 대흥정보에서 SM과 웹개발을 맡으며, '돌아가는 코드'와 '읽히는 코드' 사이의 거리를 매일 실감했습니다. 로버트 마틴의 「클린 코드」를 이 시기에 정독했습니다.

이 책의 주장은 단순합니다 — 함수는 한 가지 일만 하고, 이름은 의도를 드러내며, 주석보다 코드가 말하게 하라. 처음엔 당연한 소리로 읽혔지만, 남의 레거시를 고쳐 보면 이 당연한 것들이 지켜지지 않아 하루가 통째로 녹는다는 걸 알게 됩니다. 장별로 곱씹은 기록은 네 편으로 남겼습니다 — 이름, 함수, 에러, 테스트.

동시에 프레임워크가 감춰 둔 규칙을 하나씩 헤집었습니다. 스프링의 트랜잭션 전파처럼, 얼핏 마법 같지만 규칙을 모르면 데이터가 조용히 깨지는 지점들이었습니다. 구조를 보는 눈도 이때 자랐습니다 — 레이어드에서 헥사고날로 넘어가며, 의존성의 방향이 곧 설계라는 걸 배웠습니다.

고치기 쉽게 먼저 만든다

제품을 오래 만지면 결국 고치는 일이 본업이 됩니다. 여기저기 흩어진 정산 로직을 손대야 했을 때, 마틴 파울러의 「리팩터링」이 방법을 줬습니다. 이 책의 핵심은 기교가 아니라 순서입니다 — 테스트로 안전망을 친 뒤, 겉보기 동작을 바꾸지 않는 작은 변형을 쌓아 간다.

파울러는 '긴 함수'나 '중복'처럼 손봐야 할 신호를 코드 냄새(code smell)로 부르고, 그때마다 쓸 표준 동작(함수 추출, 이름 바꾸기 등)을 카탈로그로 정리합니다. 정산에서 종류별로 20줄씩 늘어붙던 분기를, 의도 단위의 함수로 뽑고 종류를 전략으로 갈랐습니다.

// Before — 한 메서드에 정산 종류별 분기·계산이 뒤엉킨다
if (type == MONTHLY) { /* 20여 줄 */ }
else if (type == USAGE) { /* 또 20여 줄 */ }

// After — 의도 단위로 함수를 뽑고(Extract Function), 종류를 전략으로
Money fee = policyOf(type).calculate(usage);

그 결과는 정산 시스템 모듈화 사례에 남겼고, 패턴의 시선으로는 Template Method와 Strategy의 조합으로 따로 풀어 썼습니다. 켄트 벡의 말을 빌리면, 변경을 쉽게 만든 다음 그 쉬운 변경을 하라. 리팩토링과 기능 추가를 같은 손놀림으로 섞지 말라는 규율이 특히 오래 남았습니다.

경계를 긋는 법 — 크롤링과 DDD

여러 사이트를 긁어 하나의 파이프라인으로 모으는 일은, 코드보다 경계가 어려웠습니다. 사이트마다 다른 데이터 모양이 도메인 안으로 새어 들어오면 결국 전부가 오염됩니다. 에릭 에반스의 「도메인 주도 설계」에서 두 개념을 실무로 가져왔습니다.

하나는 애그리거트 — 함께 바뀌어야 하는 것들을 한 덩어리로 묶고, 불변식을 그 경계 안에서만 지킨다. 다른 하나는 부패 방지 계층(ACL) — 외부의 모델을 내 도메인 언어로 번역하는 어댑터를 문 앞에 세운다. 크롤러의 어댑터는 정확히 이 계층이었습니다.

# 부패 방지 계층 — 사이트별 원시 응답을 도메인 모델로 '번역'한다
class SourceAdapter(Protocol):
    def fetch(self, q: Query) -> list[RawItem]: ...
    def to_domain(self, raw: RawItem) -> Listing: ...  # 외부 모양을 아는 건 여기까지

이 어댑터를 런타임에 갈아끼우는 방법은 FastAPI 의존성 주입으로, 차단당하지 않고 오래 도는 방법은 Adapter와 지수 백오프로 정리했습니다. 어디에 선을 그을지에 대한 판단은 애그리거트 경계 글에, 전체 그림은 DDD 크롤링 파이프라인 사례에 남겼습니다.

*

에반스가 강조한 보편 언어(ubiquitous language)는 문서용 용어집이 아니었습니다. 코드의 클래스명과 회의에서 쓰는 말이 같아질 때 번역 비용이 사라진다 — 경계를 긋는 진짜 이유는 여기에 있었습니다.

혼자 다 지는 자리 — SafeIn과 온프레미스

SafeIn에서는 백엔드를 혼자 설계하고 혼자 운영했습니다. 아키텍트도 운영자도 저 하나였으므로, 그동안 배운 것들이 한꺼번에 필요했습니다. 관제와 앱이 같은 데이터를 다르게 쓰기에, 권한을 역할별로 갈랐고(Guard·DTO), 새벽에 안 깨려고 로그·메트릭·트레이스를 먼저 깔았습니다.

산업안전 현장의 데이터는 밖으로 못 나가서, 모델도 온프레미스 GPU에 직접 올려야 했습니다. 여기서 인프라가 취미가 아니라 요구사항이 됩니다. 컨테이너에는 살 방과 못 넘을 선을 함께 줘야 했습니다.

# 컨테이너에 '살 방'과 '못 넘을 선'을 함께 준다
resources:
  requests: { memory: "4Gi", cpu: "1" }   # 스케줄러가 보장하는 하한
  limits:   { memory: "6Gi" }             # 이 선을 넘으면 OOMKilled

이 선을 잘못 그으면 파드가 조용히 죽습니다 — 그 진짜 이유를 requests/limits와 OOMKilled에서 파헤쳤고, 컨테이너가 아니라 힙이 문제일 때는 힙덤프 읽는 법으로 범인을 좁혔습니다. 모델 서빙은 vLLM으로 갔습니다 — 왜 빠른지는 PagedAttention 논문 분석에, 실제로 굴린 기록은 온프레미스 LLM 서빙 글과 SafeIn 사례에 있습니다. 이 시기의 배경 독서는 마틴 클레프만의 「데이터 중심 애플리케이션 설계」였습니다. 복제·파티셔닝·일관성을 한 권으로 훑고 나면, 인프라 선택이 취향이 아니라 트레이드오프로 보입니다.

정리 — 무엇을 왜 공부했나

돌아보면 순서에 규칙이 하나 있었습니다. 먼저 문제가 오고, 공부가 뒤따랐습니다. 데이터가 안 맞아서 EDA를, 딥러닝이 안 보여서 바닥을, 코드가 안 고쳐져서 리팩토링을, 경계가 무너져서 DDD를, 서버가 죽어서 인프라를 팠습니다. 책은 그때마다 이미 누군가 정리해 둔 지름길이었을 뿐입니다.

  • 데이터를 이해하는 일 → Feature Engineering, 보고서 자동화
  • 바닥부터 보는 일 → YOLO · Attention · PagedAttention 논문
  • 읽히는 코드 → 클린코드 함수, UseCase·Repository
  • 경계를 긋는 일 → DDD 애그리거트, 어댑터·백오프
  • 운영까지 지는 일 → 온프레미스 서빙, OOMKilled, 힙덤프

기술 목록으로 저를 소개하면 정작 남는 게 없습니다. 대신 이렇게 말하고 싶습니다 — 저는 모르는 걸 만나면 한 겹 아래를 보는 사람입니다. 그 습관이 2020년의 데이터 한 장에서 온프레미스 GPU까지 저를 데려왔고, 다음 문제도 아마 같은 방식으로 넘을 겁니다.
