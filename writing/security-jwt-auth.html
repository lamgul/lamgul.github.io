보안: JWT 인증 미들웨어 — 토큰은 어디서 검증되는가

2022년 5월• JWT · 서명 검증 · NestJS

SafeIn 백엔드에 인증 미들웨어를 붙이면서 가장 오래 붙잡고 있던 질문은 토큰이 어디서 검증되느냐였습니다. DB를 조회하지도 않는데 서버가 이 요청을 어떻게 믿는지 처음엔 감이 잡히지 않았는데, 판단은 토큰의 세 번째 조각인 서명에서 끝나고 있었습니다. 그때 검증 지점을 한곳으로 모으며 알게 된 것들을 적어 둡니다.

먼저 짚고 가기
JWTJSON Web Token
header.payload.signature 세 조각을 점(.)으로 이어 붙이고 각 조각을 base64url로 인코딩한 토큰 형식입니다(RFC 7519). 서버는 로그인 상태를 따로 저장하지 않고 마지막 조각의 서명만 검증해 앞의 두 조각이 발급 이후 변조되지 않았음을 확인합니다. payload는 암호화가 아니라 인코딩일 뿐이라 서버 비밀이 없어도 누구나 디코드해 읽을 수 있어, 비밀번호 같은 민감정보는 담지 않습니다.
서명Signature
JWT의 세 번째 조각으로, base64url로 인코딩된 header와 payload를 이어 붙인 문자열에 키와 alg가 지정한 알고리즘(HS256이면 HMAC-SHA256, RS256이면 RSA)을 적용해 계산한 값입니다. 검증 측은 같은 입력으로 서명을 다시 만들어 토큰의 세 번째 조각과 대조하며, 앞 두 조각이 한 글자라도 바뀌면 값이 어긋나 거부합니다. 서명은 무결성과 발급자 증명을 담당할 뿐 payload를 감추지는 못하고, 서명이 유효하다고 해서 만료나 취소 여부까지 보장하는 것은 아닙니다.
클레임Claim
JWT payload(JSON 객체)를 이루는 이름·값 쌍 하나로, "이 사용자는 worker-92이고 역할은 worker"처럼 주체에 관한 정보 한 조각을 담습니다. RFC 7519가 이름을 예약해 둔 등록 클레임(sub·exp·iat·jti 등)과 응용이 자유롭게 정의하는 커스텀 클레임(role 등)으로 나뉩니다. 클레임은 그 자체로는 검증되지 않은 주장일 뿐이라, 서명 검증을 통과하기 전에는 참이라고 믿을 수 없습니다.
대칭키 · 비대칭키Symmetric · Asymmetric key
JWT 서명에 쓰는 키 방식 두 가지입니다. 대칭키(HS256, HMAC-SHA256)는 하나의 비밀키로 서명과 검증을 모두 수행하고, 비대칭키(RS256, RSA-SHA256)는 개인키로 서명하고 공개키로는 검증만 합니다. 대칭키에서는 검증할 수 있는 쪽이면 같은 키로 서명도 할 수 있어 토큰을 위조할 수 있지만, 비대칭키에서는 공개키만 가진 검증 측이 서명을 만들어 낼 수 없다는 점에서 둘이 갈립니다.
무상태Stateless
서버가 세션이나 로그인 상태를 어디에도 저장하지 않고, 요청이 실어 온 토큰의 서명 검증만으로 인증을 끝내는 설계 원칙입니다. 매 요청마다 세션 저장소를 조회하지 않으므로 서버를 여러 대로 늘려도 상태를 공유할 필요가 없어 수평 확장에 유리합니다. 대신 서버에 아무 기록이 없다는 특성 때문에 이미 발급한 토큰을 만료 전에 회수할 수 없어, 이 글에서는 jti와 TTL만 저장하는 블랙리스트로 무상태를 일부 반납해 그 빈틈을 메웁니다.

세 조각으로 쪼개진 토큰

JWT는 header.payload.signature 형태로, 점 두 개를 기준으로 세 조각이 이어 붙은 문자열입니다. 각 조각은 Base64URL로 인코딩되어 있을 뿐이라 암호문은 아닌데, 저는 한동안 이 부분을 반대로 알고 있었습니다.

# 점 두 개가 토큰을 세 조각으로 나눈다
header . payload . signature

# 실제로는 각 조각이 base64url 문자열이다
eyJhbGci… . eyJzdWIiOiJ3b3JrZXIt… . g0T5rQ8p…

# payload를 그냥 디코드하면 — 암호화가 아니라 인코딩이다
{ "sub": "worker-92", "role": "worker", "iat": 1651655520, "exp": 1651656420 }
#  ↑ 서버 비밀 없이 누구나 읽는다. 그래서 여기엔 비밀번호를 담지 않는다.

payload는 암호화가 아니라 인코딩이라, 서버 비밀이 없어도 누구나 디코드해 읽을 수 있습니다. 토큰이 통째로 암호화되어 있다고 생각했던 게 첫 오해였고, 그래서 비밀번호나 민감정보는 payload에 넣지 않습니다. payload는 "이 사람은 worker-92이고 역할은 worker"라고 주장하는 자리일 뿐이고, 그 주장을 믿어도 되는지는 세 번째 조각이 정합니다.

서명이 신뢰의 전부다

서명은 header와 payload가 발급 이후 바뀌지 않았음을 보장합니다. 한 글자라도 고치면 서명이 어긋나 서버가 거부하니, 인증에서 믿을 근거는 이 서명 검증 한 번에 모여 있는 셈입니다. 그다음에 정해야 했던 것은 무엇으로 서명하느냐였습니다.

// HS256 — 하나의 비밀키가 서명도 검증도 한다 (대칭)
jwt.verify(token, process.env.JWT_SECRET, { algorithms: ['HS256'] });
//   검증하는 모든 서비스가 '서명할 수도 있는' 키를 나눠 갖는다

// RS256 — 개인키로 서명, 공개키로 검증만 한다 (비대칭)
jwt.verify(token, PUBLIC_KEY, { algorithms: ['RS256'] });
//   서명 서버만 개인키를 쥔다 → 검증 측은 위조할 수 없다

HS256은 대칭키라 서명과 검증이 같은 비밀키를 씁니다. 단순하고 빠른 대신 검증하려는 모든 서비스가 "서명도 할 수 있는" 키를 나눠 가져야 하는데, 서비스가 하나뿐인 단일 서버였을 때는 이걸로 충분했습니다. RS256은 비대칭키라 서명 서버만 개인키를 쥐고 나머지는 공개키로 검증만 하므로, 검증 주체가 여럿으로 흩어지거나 검증하는 쪽이 토큰을 위조하면 안 되는 구조에서는 RS256을 씁니다.

*

algorithms를 지정하지 않으면, 공격자가 토큰 헤더의 alg를 none이나 HS256으로 바꿔 넣을 수 있습니다. RS256을 기대하는 서버에서 공개키를 HMAC 비밀키로 재활용해 서명을 위조하는 알고리즘 혼동 공격입니다. 그래서 alg는 토큰이 주장하는 값을 그대로 따르지 않고, 검증 측에서 고정해 둡니다.

서명을 exp보다 먼저 검사하는 이유 — exp는 payload 안에 있고, payload는 서명이 맞다고 확인되기 전까지 그저 주장일 뿐이기 때문입니다





만료·리프레시·블랙리스트


exp 클레임 덕분에 토큰은 스스로 만료되고, 검증기는 매번 exp > now를 확인합니다. 다만 토큰만 보고 판단이 끝난다는 점이 그대로 약점이 됩니다. 서명만 맞으면 만료 전까지는 유효하니 서버가 중간에 돌려받을 방법이 없고, 사용자가 로그아웃을 해도 토큰이 유출돼도 만료 시각까지는 살아 있습니다.


그래서 토큰을 둘로 나눠 씁니다. 수명이 짧은 access 토큰과 상대적으로 긴 refresh 토큰을 함께 발급해, access가 15분 뒤 만료되면 refresh로 새로 받는 식입니다. 그 15분의 틈까지 막아야 할 때 쓴 것이 블랙리스트였습니다.


// access는 검증 때마다 블랙리스트를 한 번 확인한다
async assertNotRevoked(payload: JwtPayload) {
  const revoked = await this.redis.get(`bl:${payload.jti}`);
  if (revoked) throw new UnauthorizedException('revoked');
}

// 로그아웃 = 남은 수명만큼만 블랙리스트에 올린다 (TTL이 알아서 지운다)
async revoke(jti: string, exp: number) {
  const ttl = exp - Math.floor(Date.now() / 1000);
  if (ttl > 0) await this.redis.set(`bl:${jti}`, '1', 'EX', ttl);
}


블랙리스트를 두는 순간 JWT의 무상태(stateless)를 일부 반납하게 됩니다. 다만 Redis에 올리는 것은 jti와 남은 수명(TTL)뿐이라, 상태는 만료 시점까지만 남습니다. 무상태의 이점과 취소 가능성 사이의 타협점을 여기에 잡아 두었고, 로그아웃이나 토큰 탈취, 권한 강등처럼 취소가 요구사항인 경우에만 이 비용을 치르도록 했습니다.



검증을 한곳에 모으기 — Guard와 Strategy


처음의 질문으로 돌아오면, NestJS에서 토큰이 실제로 검증되는 자리는 passport-jwt Strategy였습니다.


@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor() {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,        // 만료 검사는 절대 끄지 않는다
      secretOrKey: process.env.JWT_PUBLIC_KEY,
      algorithms: ['RS256'],          // alg는 우리가 고정한다
    });
  }
  // 서명과 만료를 통과한 요청만 여기까지 온다
  async validate(payload: JwtPayload) {
    return { id: payload.sub, role: payload.role };  // → req.user
  }
}


secretOrKey와 algorithms로 서명을, ignoreExpiration: false로 만료를 검사합니다. 이 둘을 통과한 요청만 validate()에 도착하고, 그 반환값이 req.user가 됩니다. 그러니까 validate() 안은 이미 검증이 끝난 뒤의 코드입니다.


이 Strategy는 Guard로 감싸 경계에 세워 두었습니다. 검증을 컨트롤러마다 흩어 놓으면 어디선가 빠뜨리게 되는데, 하나의 Guard 체인을 관문으로 두면 그 선을 넘은 요청은 이미 자신을 증명한 상태가 됩니다. 토큰이 어디서 검증되느냐는 질문의 답도 이 한 줄이었습니다.



같은 토큰, 다른 경계


SafeIn에는 웹으로 전체를 처리하는 관리자와 앱으로 자기 것만 다루는 근로자, 두 종류의 사용자가 있습니다. 둘은 같은 형식의 JWT를 받고 payload의 role 클레임만 다릅니다. role은 서명 안에 묶여 있어서 사용자가 임의로 바꿀 수 없습니다.


// 순서가 곧 경계다: 진짜인가? → 그다음, 허용되는가?
@UseGuards(JwtAuthGuard, RolesGuard)
@Roles('admin')
@Patch('reports/:id/resolve')
resolve(@Param('id') id: string) { /* 관리자만 도달한다 */ }


경계는 순서로 만들어집니다. JwtAuthGuard가 "진짜인가"를 먼저 묻고 통과하면 RolesGuard가 "허용되는가"를 묻는 식으로, 인증과 인가가 두 겹의 관문으로 나뉩니다. 다만 role은 토큰이 발급된 순간의 스냅샷이라, 근로자가 관리자로 바뀌어도 예전 토큰에는 예전 역할이 남아 있습니다. 앞서 만료를 짧게 잡고 블랙리스트를 둔 이유가 여기서 다시 걸립니다. 역할별 경계를 코드로 어떻게 갈랐는지는 아래 링크에 따로 적어 두었습니다.



정리 — 어디서 검증되는가


정리해 두면, JWT의 payload는 주장을 나르고 signature는 신뢰를 나릅니다. 그래서 지켜야 할 비밀은 payload가 아니라 서명 키 쪽에 있습니다. 무상태를 택한 대가로 이미 발급한 토큰을 취소하기 어려워지는데, 짧은 만료와 refresh, 블랙리스트가 그 빈틈을 메웁니다. 검증도 컨트롤러마다 흩어 놓지 않고 하나의 경계에 모아 두었습니다.


작업을 마치고 나서는, 인증 미들웨어가 하는 일이 로그인을 시키는 게 아니라 이 선을 넘는 요청이 이미 자신을 증명했음을 보장하는 일이라고 정리하게 됐습니다. 지금도 토큰이 어디서 검증되느냐는 질문을 받으면 그 경계 한 곳을 가리킵니다.
