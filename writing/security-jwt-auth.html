보안: JWT 인증 미들웨어 — 토큰은 어디서 검증되는가

2022년 5월• JWT · 서명 검증 · NestJS

인증 미들웨어를 처음 붙일 때 가장 헷갈렸던 질문은 "토큰이 대체 어디서 검증되는가"였습니다. DB를 찌르지도 않는데, 서버는 이 요청이 진짜라고 어떻게 확신할까요. 답은 토큰 안에, 더 정확히는 서명 안에 있었습니다. SafeIn 백엔드를 혼자 설계하며 이 경계를 한곳으로 모았던 과정을 정리합니다.

먼저 짚고 가기
JWTJSON Web Token
로그인한 사용자가 "나는 이런 사람"이라고 들고 다니는 디지털 출입증입니다. 놀이공원의 손목 밴드처럼, 한 번 받으면 그다음부터는 밴드만 보여 주면 됩니다. 서버가 매번 회원 명부(DB)를 뒤지지 않아도 되는 게 이 글의 출발점입니다.
서명Signature
토큰이 위조되지 않았음을 증명하는, 서버만 만들 수 있는 봉인입니다. 편지 봉투의 밀랍 도장과 같아서, 내용을 한 글자라도 고치면 도장이 깨집니다. 인증의 신뢰가 전부 이 도장 하나에 걸려 있어서 이 글의 핵심입니다.
클레임Claim
토큰 안에 담긴 "나는 worker-92이고 역할은 근로자"라는 식의 정보 한 조각입니다. 말 그대로 주장이라, 그 자체로는 참이 아니고 서명이 뒷받침해 줘야 믿을 수 있습니다. 만료 시각(exp), 역할(role) 같은 것들이 모두 클레임입니다.
대칭키 · 비대칭키Symmetric · Asymmetric key
서명에 쓰는 열쇠 방식입니다. 대칭키(HS256)는 하나의 열쇠로 잠그고 여는 반면, 비대칭키(RS256)는 잠그는 열쇠(개인키)와 여는 열쇠(공개키)가 따로입니다. 이 선택이 "누가 토큰을 위조할 수 있는가"를 가릅니다.
무상태Stateless
서버가 로그인 상태를 따로 기억해 두지 않는다는 뜻입니다. 손님을 기억하지 못해도 손목 밴드만 확인하는 놀이공원 같아서 가볍고 확장하기 쉽지만, 대신 "이미 나눠 준 밴드를 되돌려 받기가 어렵다"는 약점이 뒤에서 문제가 됩니다.

세 조각으로 쪼개진 토큰

JWT는 이름 그대로 세 조각입니다. header.payload.signature, 점 두 개로 나뉩니다. 각 조각은 Base64URL로 인코딩되어 있을 뿐, 암호문이 아닙니다.

# 점 두 개가 토큰을 세 조각으로 나눈다
header . payload . signature

# 실제로는 각 조각이 base64url 문자열이다
eyJhbGci… . eyJzdWIiOiJ3b3JrZXIt… . g0T5rQ8p…

# payload를 그냥 디코드하면 — 암호화가 아니라 인코딩이다
{ "sub": "worker-92", "role": "worker", "iat": 1753238400, "exp": 1753239300 }
#  ↑ 서버 비밀 없이 누구나 읽는다. 그래서 여기엔 비밀번호를 담지 않는다.

여기서 첫 오해가 풀립니다. payload는 암호화가 아니라 인코딩이라, 서버 비밀이 없어도 누구나 디코드해 읽습니다. 그러니 비밀번호나 민감정보를 payload에 넣으면 안 됩니다. payload는 "이 사람은 worker-92이고 역할은 worker"라고 주장하는 자리일 뿐입니다. 그 주장을 믿어도 되는지는 세 번째 조각이 정합니다.

서명이 신뢰의 전부다

서명은 "header와 payload가 위조되지 않았다"를 보장합니다. 한 글자라도 바꾸면 서명이 깨지고, 서버는 즉시 거부합니다. 그래서 인증의 신뢰는 전부 이 서명 검증 한 번에 걸려 있습니다. 남는 문제는 무엇으로 서명하느냐입니다.

// HS256 — 하나의 비밀키가 서명도 검증도 한다 (대칭)
jwt.verify(token, process.env.JWT_SECRET, { algorithms: ['HS256'] });
//   검증하는 모든 서비스가 '서명할 수도 있는' 키를 나눠 갖는다

// RS256 — 개인키로 서명, 공개키로 검증만 한다 (비대칭)
jwt.verify(token, PUBLIC_KEY, { algorithms: ['RS256'] });
//   서명 서버만 개인키를 쥔다 → 검증 측은 위조할 수 없다

HS256은 대칭키라 서명과 검증이 같은 비밀키를 씁니다. 단순하고 빠르지만, 검증하려는 모든 서비스가 "서명도 할 수 있는" 키를 나눠 가져야 합니다. 서비스 하나짜리 단일 서버라면 이걸로 충분합니다. 반대로 RS256은 비대칭키라, 서명 서버만 개인키를 쥐고 나머지는 공개키로 검증만 합니다. 검증 주체가 여럿으로 흩어지거나 검증하는 쪽이 토큰을 위조하면 안 되는 구조라면, 답은 RS256입니다.

*

algorithms를 지정하지 않으면, 공격자가 토큰 헤더의 alg를 none이나 HS256으로 바꿔 넣을 수 있습니다. RS256을 기대하는 서버에서 공개키를 HMAC 비밀키로 재활용해 서명을 위조하는 알고리즘 혼동 공격입니다. 방어는 간단합니다 — alg는 토큰이 아니라 검증 측이 못 박습니다.

만료·리프레시·블랙리스트

exp 클레임 덕분에 토큰은 스스로 만료됩니다. 검증기는 매번 exp > now를 확인하죠. 그런데 이 자족성이 곧 약점입니다. 서명만 맞으면 만료 전까지 유효하니, 서버가 중간에 돌려받을 방법이 없습니다. 로그아웃했는데도, 토큰이 유출됐는데도 만료까지는 살아 있습니다.

그래서 실무는 토큰을 둘로 나눕니다. 수명이 짧은 access 토큰과 긴 refresh 토큰. access가 15분 뒤 죽어도 refresh로 조용히 새로 받습니다. 그 15분의 틈마저 막아야 할 때 블랙리스트가 등장합니다.

// access는 검증 때마다 블랙리스트를 한 번 확인한다
async assertNotRevoked(payload: JwtPayload) {
  const revoked = await this.redis.get(`bl:${payload.jti}`);
  if (revoked) throw new UnauthorizedException('revoked');
}

// 로그아웃 = 남은 수명만큼만 블랙리스트에 올린다 (TTL이 알아서 지운다)
async revoke(jti: string, exp: number) {
  const ttl = exp - Math.floor(Date.now() / 1000);
  if (ttl > 0) await this.redis.set(`bl:${jti}`, '1', 'EX', ttl);
}

블랙리스트는 JWT가 팔던 무상태(stateless)를 일부 반납하는 일입니다. 하지만 jti와 남은 수명(TTL)만 Redis에 올리므로, 상태는 만료 시점까지만 존재합니다. 무상태의 이점과 취소 가능성 사이의 타협점을 여기서 잡는 겁니다. 로그아웃, 토큰 탈취, 권한 강등처럼 취소가 곧 요구사항인 순간에만 이 비용을 치릅니다.

검증을 한곳에 모으기 — Guard와 Strategy

이제 처음 질문으로 돌아옵니다. 토큰은 어디서 검증될까요. NestJS에서는 passport-jwt Strategy가 바로 그 자리입니다.

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor() {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,        // 만료 검사는 절대 끄지 않는다
      secretOrKey: process.env.JWT_PUBLIC_KEY,
      algorithms: ['RS256'],          // alg는 우리가 고정한다
    });
  }
  // 서명과 만료를 통과한 요청만 여기까지 온다
  async validate(payload: JwtPayload) {
    return { id: payload.sub, role: payload.role };  // → req.user
  }
}

secretOrKey와 algorithms로 서명을, ignoreExpiration: false로 만료를 검사합니다. 이 둘을 통과한 요청만 validate()에 도착하고, 그 반환값이 req.user가 됩니다. 즉 validate()는 이미 "검증이 끝난 뒤"의 세계입니다.

그리고 이 Strategy를 Guard로 감싸 경계에 세웁니다. 검증을 컨트롤러마다 흩어 놓으면 반드시 빠뜨리는 곳이 생깁니다. 대신 하나의 Guard 체인을 관문으로 두면, 그 선을 넘은 모든 요청은 이미 자신을 증명한 것입니다. "토큰은 어디서 검증되는가"의 답은 결국 이 한 줄의 경계입니다.

같은 토큰, 다른 경계

SafeIn에는 두 종류의 사용자가 있습니다. 웹으로 전체를 처리하는 관리자와, 앱으로 자기 것만 다루는 근로자. 둘은 같은 형식의 JWT를 받지만 payload의 role 클레임이 다릅니다. 그리고 role은 서명 안에 묶여 있으니 사용자가 함부로 바꿀 수 없습니다.

// 순서가 곧 경계다: 진짜인가? → 그다음, 허용되는가?
@UseGuards(JwtAuthGuard, RolesGuard)
@Roles('admin')
@Patch('reports/:id/resolve')
resolve(@Param('id') id: string) { /* 관리자만 도달한다 */ }

경계는 순서로 만들어집니다. JwtAuthGuard가 "진짜인가"를 먼저 묻고, 통과하면 RolesGuard가 "허용되는가"를 묻습니다. 인증과 인가가 두 겹의 관문으로 분리되죠. 다만 role은 토큰이 발급된 순간의 스냅샷이라, 근로자가 관리자로 바뀌어도 옛 토큰엔 옛 역할이 남습니다. 앞서 만료를 짧게 잡고 블랙리스트를 둔 이유가 여기서 다시 맞물립니다. 역할별 경계를 코드로 어떻게 갈랐는지는 아래 링크에서 더 깊이 다뤘습니다.

정리 — 어디서 검증되는가

정리하면 이렇습니다. JWT의 payload는 주장을 나르고 signature는 신뢰를 나릅니다 — 그래서 비밀은 payload가 아니라 서명 키에 있습니다. 무상태를 택한 대가로 취소가 어려워지고, 짧은 만료·refresh·블랙리스트가 그 빈틈을 메웁니다. 마지막으로 검증은 컨트롤러에 흩어질 게 아니라 하나의 경계에 모여야 합니다.

인증 미들웨어가 하는 일은 "로그인시키기"가 아닙니다. 이 선을 넘는 모든 요청이 이미 자신을 증명했음을 보장하는 것입니다. 토큰이 어디서 검증되느냐고 물으면, 저는 그 경계 한 곳을 가리킵니다.
