보안: TLS와 리버스 프록시 — NGINX 앞단에서 끝내기

2024년 9월• 약 11분• TLS · NGINX · 리버스 프록시

요청이 애플리케이션 코드에 닿기 전에, 이미 참이어야 하는 것들이 있습니다. 연결은 암호화되어 있어야 하고, 클라이언트는 평문 대신 HTTPS로 유도되어 있어야 하며, 뒤에 선 백엔드는 바깥의 거친 트래픽에서 가려져 있어야 합니다. 리버스 프록시는 그 모든 것을 한자리에서, 한 번에 참으로 만드는 곳입니다. NGINX 앞단에서 보안을 끝낸다는 건 그런 의미입니다.

먼저 짚고 가기
리버스 프록시reverse proxy
사용자와 실제 서버 사이에 앉아 요청을 대신 받아 뒤쪽 서버로 넘겨 주는 중간 관문입니다. 건물 정문의 안내 데스크처럼, 방문객은 데스크만 상대하고 안쪽 사무실은 노출되지 않습니다. 이 글은 그 관문 한 곳에 보안을 몰아넣는 이야기입니다.
TLS 핸드셰이크TLS handshake
암호화 통신을 시작하기 전, 클라이언트와 서버가 "어떤 방식으로 암호를 걸지"와 "둘만 아는 비밀 열쇠"를 짧게 주고받으며 합의하는 첫 인사입니다. 이 인사가 끝나야 그 뒤 오가는 내용이 안전하게 잠깁니다.
TLS 종단TLS termination
암호화된 연결을 프록시에서 풀어(복호화해) 주고, 그 안쪽 내부망은 평문으로 백엔드에 넘기는 구조입니다. 인증서 관리를 한 곳으로 모을 수 있어, 뒤에 선 서버들은 TLS를 몰라도 됩니다.
전방 안전성forward secrecy
세션마다 쓰고 버리는 임시 열쇠를 써서, 훗날 서버의 개인키가 유출돼도 과거에 오간 트래픽은 풀리지 않게 하는 성질입니다. 오늘 설정의 핵심 목표 중 하나입니다.
HSTSHTTP Strict Transport Security
한 번 HTTPS로 응답을 받은 브라우저에게 "앞으로 이 도메인엔 평문으로 접속조차 하지 마"라고 못 박는 규칙입니다. 리다이렉트만으로 남는 첫 요청의 틈을 메웁니다.

1. 핸드셰이크의 큰 그림부터

TLS를 설정하기 전에 핸드셰이크가 무엇을 합의하는지 큰 그림만 잡고 갑니다. 클라이언트가 ClientHello로 지원 가능한 프로토콜과 암호 스위트 목록을 던지면, 서버는 ServerHello로 그중 하나를 고르고 자신의 인증서를 내밉니다. 이어 키 교환으로 양쪽만 아는 대칭키를 세우고, 이후의 모든 바이트는 그 대칭키로 암호화됩니다. 공개키 암호는 이 첫 악수에서 키를 안전하게 세우는 데만 쓰이고, 실제 데이터는 훨씬 빠른 대칭 암호가 나릅니다.

여기서 오늘의 설정을 좌우하는 개념이 전방 안전성(forward secrecy)입니다. 키 교환에 ECDHE처럼 세션마다 버려지는 임시 키를 쓰면, 훗날 서버의 개인키가 유출되어도 과거에 오간 트래픽은 복호화되지 않습니다. TLS 1.3은 아예 전방 안전성이 없는 방식을 스펙에서 제거했고, 왕복도 한 번으로 줄여 handshake가 눈에 띄게 가벼워졌습니다. 그래서 오늘의 목표는 단순합니다 — 1.2 이상만 열고, 임시 키 교환만 쓰는 것.

핸드셰이크는 비싸기 때문에, 잘 짠 프록시는 이 악수를 재사용합니다. 세션 캐시나 세션 티켓으로 재접속의 왕복을 줄이고, OCSP Stapling으로 인증서 폐기 확인의 지연을 서버가 미리 삼켜 둡니다. 큰 그림은 여기까지입니다. 나머지는 이 그림을 NGINX 지시어로 옮기는 일입니다.

2. 프록시에서 TLS를 끝낸다

TLS 종단(termination)은 암호화된 연결을 프록시에서 풀고, 그 뒤 내부망은 평문 HTTP로 백엔드에 넘기는 구조입니다. 인증서와 프로토콜 정책을 한 곳에서 관리하니 백엔드는 TLS를 몰라도 되고, 배포도 단순해집니다. 아래가 종단 서버 블록의 뼈대입니다.

# /etc/nginx/conf.d/app.conf
server {
    listen 443 ssl;
    http2 on;                       # NGINX 1.25.1+ 권장 방식
    server_name app.example.com;

    ssl_certificate     /etc/ssl/app/fullchain.pem;
    ssl_certificate_key /etc/ssl/app/privkey.pem;

    # 1.2 이상만 연다. 1.0/1.1은 완전히 닫는다
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-CHACHA20-POLY1305;

    # 악수를 재사용해 재접속의 왕복을 줄인다
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;

    # OCSP Stapling — 폐기 확인을 서버가 대신 물어와 붙인다
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 1.1.1.1 8.8.8.8 valid=300s;
}

한 가지 오해하기 쉬운 지점이 있습니다. ssl_ciphers 목록은 TLS 1.2까지만 적용되고, TLS 1.3은 자신이 정한 강한 기본 스위트를 씁니다. 그러니 1.3 스위트를 손대려 애쓸 필요 없이, 위 목록은 1.2 클라이언트를 위한 안전망으로 두면 됩니다. 모든 스위트가 ECDHE로 시작하는 것도 의도된 선택입니다 — 앞 절에서 말한 전방 안전성을 프로토콜 수준에서 강제하는 겁니다.

평문으로 들어온 80 포트는 그냥 닫지 말고, 301로 HTTPS에 넘겨 사용자가 옛 링크로도 안전하게 도착하게 합니다. 이 리다이렉트가 다음 절의 HSTS와 짝을 이룹니다.

server {
    listen 80;
    server_name app.example.com;
    return 301 https://$host$request_uri;
}

3. HSTS와 보안 헤더로 문을 잠근다

리다이렉트만으로는 첫 요청이 평문으로 새어 나갈 틈이 남습니다. HSTS는 그 틈을 메웁니다 — 한 번 HTTPS로 응답을 받은 브라우저는 정해진 기간 동안 그 도메인에 평문 연결을 시도조차 하지 않습니다. 여기에 몇 개의 방어 헤더를 더 얹어, 콘텐츠 스니핑과 클릭재킹처럼 값싼 공격을 응답 한 줄로 막습니다.

# 한 번 HTTPS로 온 브라우저는 앞으로 평문을 시도하지 않는다
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options        "DENY" always;
add_header Referrer-Policy        "strict-origin-when-cross-origin" always;
add_header Content-Security-Policy "default-src 'self'; frame-ancestors 'none'" always;

끝에 붙은 always가 사소해 보이지만 중요합니다. 이게 없으면 4xx·5xx 같은 오류 응답에는 헤더가 빠지는데, 정작 공격 트래픽이 오류를 많이 유발한다는 걸 생각하면 항상 붙어야 할 헤더입니다. HSTS의 max-age는 2년으로 넉넉히 잡되, 서브도메인까지 걸 때는 그 아래 모든 호스트가 정말 HTTPS로만 서비스되는지 먼저 확인해야 합니다.

*

NGINX의 add_header는 상속되지 않고 덮어씁니다. 하위 location이 자기만의 add_header를 하나라도 선언하면, 상위에서 물려주던 보안 헤더가 그 블록에서 통째로 사라집니다. 헤더를 한 곳에 모으거나, 위험한 경로마다 다시 선언했는지 curl -I로 반드시 확인하세요.

HSTS에는 preload 옵션이 더 있습니다. 브라우저 내장 목록에 도메인을 올려, 첫 방문부터 평문을 원천 차단하는 강력한 수단이죠. 다만 이건 들어가긴 쉽고 빠져나오긴 어려운 결정입니다. 목록 반영과 회수는 브라우저 릴리스 주기를 타기 때문에, 준비가 덜 된 서브도메인이 하나라도 있으면 접속 불능이 오래 갈 수 있습니다. 그래서 저는 preload를 "확신이 설 때 마지막에 켜는 스위치"로 둡니다.

4. 업스트림을 지키는 버퍼링과 타임아웃

프록시의 또 다른 역할은 백엔드를 바깥의 변덕에서 떼어놓는 것입니다. 느린 클라이언트, 죽은 노드, 오래 걸리는 요청 — 이 셋이 그대로 업스트림에 전달되면 애플리케이션 워커가 금세 말라붙습니다. 버퍼링·타임아웃·헬스체크는 그 사이에 놓는 완충 장치입니다.

upstream app_backend {
    server 10.0.1.11:8080 max_fails=3 fail_timeout=15s;
    server 10.0.1.12:8080 max_fails=3 fail_timeout=15s;
    keepalive 32;                    # 업스트림 연결을 재사용
}

server {
    # ... TLS 설정은 위와 동일 ...
    location / {
        proxy_pass http://app_backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";          # keepalive를 살린다

        # 원래 요청의 맥락을 업스트림에 넘긴다
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 느린 업스트림이 프록시를 물고 늘어지지 않게
        proxy_connect_timeout 3s;
        proxy_send_timeout    30s;
        proxy_read_timeout    30s;

        # 버퍼링: 느린 클라이언트에서 업스트림을 떼어놓는다
        proxy_buffering   on;
        proxy_buffer_size 16k;
        proxy_buffers     8 16k;

        # 죽은 노드는 건너뛴다 (아래 주의 참고)
        proxy_next_upstream error timeout http_502 http_503;
    }
}

proxy_buffering on은 NGINX가 업스트림의 응답을 버퍼(필요하면 디스크까지)에 받아 두고 워커를 빨리 놓아준다는 뜻입니다. 그러면 응답을 찔끔찔끔 받아가는 느린 클라이언트가 백엔드 스레드를 붙잡는 사태를 막을 수 있습니다. 반대로 SSE나 스트리밍 응답에서는 버퍼링이 지연을 만드니, 그 경로만 proxy_buffering off로 예외를 둡니다. 켜고 끄는 판단의 기준은 언제나 "이 응답이 실시간으로 흘러야 하는가"입니다.

헬스체크는 두 종류를 구분해야 합니다. 오픈소스 NGINX가 제공하는 건 max_fails·fail_timeout 기반의 수동 헬스체크로, 실패가 쌓인 노드를 일정 시간 빼는 방식입니다. 능동적으로 /health를 주기적으로 찔러 보는 액티브 헬스체크(health_check 지시어)는 NGINX Plus 기능이니, 오픈소스라면 이 역할을 외부 모니터나 로드밸런서에 맡기는 편이 정직합니다.

*

proxy_next_upstream에 http_502 같은 상태를 넣으면 재시도가 생깁니다. 편리하지만, POST처럼 멱등하지 않은 요청이 재전송되면 결제·주문이 중복 처리될 수 있습니다. 안전하지 않은 메서드에는 재시도 범위를 좁히거나 non_idempotent를 의식적으로 다루세요 — 자동 재시도는 언제나 공짜가 아닙니다.

마지막으로 X-Forwarded-For는 클라이언트가 위조할 수 있는 헤더입니다. 그러니 가장 바깥 프록시에서는 들어온 값을 신뢰하지 말고 새로 세팅해, 그 뒤 백엔드가 보는 클라이언트 IP가 믿을 수 있는 값이 되도록 합니다.

5. 버전을 숨기는 작은 하드닝

기본 설정의 NGINX는 응답 헤더와 오류 페이지에 자기 버전을 적어 보냅니다. Server: nginx/1.25.3 한 줄은, 자동화된 스캐너에게 "이 버전에 알려진 취약점을 바로 시도해 보라"는 안내판과 같습니다. 이걸 지우는 게 하드닝의 가장 값싼 한 걸음입니다.

# 응답과 오류 페이지에서 버전 문자열을 지운다
server_tokens off;

# 업스트림이 흘리는 정보 헤더도 프록시에서 걷어낸다
proxy_hide_header X-Powered-By;
proxy_hide_header Server;

# 버전 서명이 남는 기본 오류 페이지를 자체 페이지로 교체
error_page 500 502 503 504 /50x.html;

솔직히 말하면 이건 결정적인 방어가 아닙니다. 작정한 공격자는 응답의 미묘한 차이만으로도 서버 종류를 추정하니까요. 그럼에도 버전을 숨기는 이유는, 세상 공격의 대부분이 사람이 아니라 버전 배너를 훑고 지나가는 자동 스캐너이기 때문입니다. 정보 노출을 줄이면 그 자동화된 소음의 상당 부분이 표적을 놓칩니다. 참고로 Apache라면 같은 일을 ServerTokens Prod와 ServerSignature Off가 맡습니다.

이 감각은 SafeIn에서 백엔드를 혼자 설계하며 온프레미스로 vLLM을 서빙할 때 더 또렷해졌습니다. 클라우드의 관리형 방어막이 없는 온프레미스에서는, 바깥으로 나가는 응답 한 줄 한 줄이 곧 공격 표면이 됩니다. 그래서 불필요한 정보는 흘리지 않는다는 원칙을 프록시 단에서 기본값으로 박아 두었습니다 — 백엔드 코드가 무엇을 실수로 노출하든, 경계에서 한 번 더 걸러지도록.

6. 책 한 권: Bulletproof SSL and TLS

TLS 설정을 "복사·붙여넣기"에서 "이해"로 넘어가게 해 준 책이 이반 리스티치(Ivan Ristić)의 《Bulletproof SSL and TLS》입니다. 저자는 서버 설정 등급을 매기는 SSL Labs 테스트와, 초기 ModSecurity를 만든 사람입니다. 그러니 이 책은 이론서가 아니라 "실제로 무엇이 A+를 만드는가"를 아는 사람의 실무서입니다.

책의 구성은 크게 네 덩어리로 읽힙니다. 앞부분은 프로토콜과 PKI — 핸드셰이크가 실제로 어떻게 키를 세우고, 인증서 체인이 어떻게 검증되는지. 가운데는 공격의 카탈로그로, BEAST·CRIME·Lucky 13·POODLE·Heartbleed 같은 이름들이 각각 어떤 가정을 깨뜨렸는지 짚습니다. 뒤로는 성능(세션 재사용·OCSP Stapling)과 배포·설정이 이어지고, OpenSSL로 직접 확인하는 법까지 다룹니다. 앞 절들에서 제가 켠 옵션 하나하나가 이 지도 위 어딘가에 놓여 있습니다.

이 책이 준 가장 큰 관점은 "이국적인 암호를 고르는 것보다 올바른 설정이 훨씬 중요하다"는 태도입니다. 화려한 스위트를 찾기 전에, 낡은 프로토콜을 닫고 전방 안전성을 강제하고 단순하게 유지하라는 것. 그 철학을 코드로 옮기면 결국 암호 스위트 이름 한 줄을 읽을 줄 아는 일이 됩니다.

# 스위트 이름은 네 부분으로 읽힌다
ECDHE-ECDSA-AES128-GCM-SHA256
#  │     │      │       │
#  │     │      │       └ 해시(무결성): SHA-256
#  │     │      └ 대칭 암호(기밀성): AES-128-GCM
#  │     └ 인증(서버 신원): ECDSA 서명
#  └ 키 교환: ECDHE = Ephemeral → 전방 안전성

이렇게 읽고 나면 ssl_ciphers 목록이 더는 주문(呪文)이 아니라 정책의 문장으로 보입니다. 그리고 그 정책이 맞게 적용됐는지는 리스티치가 만든 SSL Labs 테스트가 등급으로 되돌려 줍니다. 저는 이 책을 읽는 순서와 SSL Labs로 점수를 올려 보는 피드백 루프를, TLS를 몸으로 익히는 가장 빠른 길로 추천합니다.

7. 정리 — 경계에서 끝내기

리버스 프록시가 좋은 이유는 보안을 한 곳에서 표준화할 수 있기 때문입니다. 종단을 깔끔히 하고, HSTS와 보안 헤더로 HTTPS를 강제하고, 버퍼링·타임아웃·헬스체크로 업스트림을 지키고, 새어 나가는 정보를 줄인다 — 이 네 가지를 프록시에 박아 두면, 뒤에 선 백엔드는 본연의 로직에만 집중할 수 있습니다. 경계에서 끝낸 만큼 안쪽이 단순해집니다.

큰 그림은 핸드셰이크에서 시작해, 《Bulletproof SSL and TLS》라는 지도를 거쳐, SSL Labs 등급이라는 피드백 루프로 닫힙니다. 도구는 정책을 강제해 줄 뿐, 어떤 프로토콜을 열고 무엇을 숨길지 정하는 마지막 판단은 결국 사람의 몫입니다. 그 판단을 자신 있게 내리는 데 이 글이 작은 지도가 되었으면 합니다.
