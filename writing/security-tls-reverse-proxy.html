보안: TLS와 리버스 프록시 — NGINX 앞단에서 끝내기

2024년 9월• TLS · NGINX · 리버스 프록시

요청이 애플리케이션 코드에 닿기 전에 이미 참이어야 하는 조건들이 있습니다. 연결이 암호화되어 있을 것, 평문으로 들어온 요청이 HTTPS로 넘어가 있을 것, 뒤에 선 백엔드가 바깥 트래픽에 직접 노출되지 않을 것. 이 조건들을 애플리케이션 쪽에서 각자 챙기다 보면 어디가 빠졌는지 알기 어려워서, 리버스 프록시 한 곳으로 모아 두고 그 설정만 확인하는 방식으로 정리했습니다. NGINX 앞단에서 보안을 끝낸다는 말은 그런 뜻이었습니다. 그때 짚어 가며 익힌 내용을 적어 둡니다.

먼저 짚고 가기
리버스 프록시reverse proxy
클라이언트의 연결을 대신 받아 뒤쪽 원본 서버(업스트림)로 중계하는, 서버 측 경계에 놓인 프록시입니다. 클라이언트를 대신해 바깥으로 나가는 포워드 프록시와 방향이 반대라, 클라이언트는 프록시의 주소만 알 뿐 실제 백엔드의 주소나 대수는 드러나지 않습니다. TLS 종단·부하 분산·캐싱·헤더 필터링을 이 한 지점에 모을 수 있어, 이 글은 보안 정책을 여기서 표준화한 기록입니다.
TLS 핸드셰이크TLS handshake
TLS 연결을 여는 초반 협상 단계입니다. 클라이언트가 ClientHello로 지원하는 프로토콜 버전과 암호 스위트 목록을 보내면 서버가 ServerHello로 하나를 고르고 인증서를 제시하며, 이어 키 교환으로 양쪽만 아는 대칭키를 세웁니다. 이때 공개키 암호는 대칭키를 안전하게 합의하는 데만 쓰이고 이후 실제 데이터는 그 대칭키로 암호화되며, TLS 1.3은 이 협상 왕복을 1-RTT로 줄였습니다.
TLS 종단TLS termination
프록시가 인증서와 개인키를 쥐고 들어온 TLS 연결을 복호화한 뒤, 그 안쪽 내부망에는 평문 HTTP로 백엔드에 넘기는 배치 구조입니다. 인증서와 프로토콜 정책을 이 한 곳에서 관리하므로 뒤에 선 서버들은 TLS를 직접 구현하지 않아도 됩니다. 다만 프록시-백엔드 구간은 암호화되지 않으므로 내부망이 신뢰 가능하다는 전제가 필요하며, 그 구간까지 암호를 유지하는 종단 간(end-to-end) 방식이나 암호문을 그대로 흘려보내는 통과(passthrough)와는 다릅니다.
전방 안전성forward secrecy
키 교환 방식이 갖는 보안 성질로, 세션마다 새로 만들고 버리는 임시 키(ECDHE·DHE의 E는 Ephemeral)를 써서 세션 키를 서버의 장기 개인키와 분리합니다. 그 덕에 훗날 개인키가 유출되어도, 공격자가 미리 기록해 둔 과거 세션의 트래픽은 복호화되지 않습니다. 어디까지나 '과거 세션을 미래의 키 유출로부터' 지키는 성질이라, 세션이 진행 중일 때의 도청이나 임시 키 자체의 유출까지 막아 주지는 않습니다. TLS 1.3은 전방 안전성이 없는 키 교환을 스펙에서 아예 제거했습니다.
HSTSHTTP Strict Transport Security
HTTPS 강제를 브라우저에 각인시키는 응답 헤더 정책입니다(RFC 6797). 서버가 Strict-Transport-Security 헤더를 보내면 브라우저는 max-age로 지정한 기간 동안 그 호스트로의 요청을 자동으로 HTTPS로 승격하고 평문 연결을 거부하며, includeSubDomains를 붙이면 하위 도메인까지 같은 규칙이 걸립니다. 다만 헤더를 처음 받기 전의 최초 요청은 보호되지 않는 신뢰 시점(TOFU) 공백이 남고, 이 틈은 브라우저 내장 목록에 도메인을 미리 올리는 preload로만 메울 수 있습니다.

1. 핸드셰이크의 큰 그림부터

설정에 손대기 전에 핸드셰이크가 무엇을 합의하는지부터 확인했습니다. 클라이언트가 ClientHello로 지원 가능한 프로토콜과 암호 스위트 목록을 보내면, 서버는 ServerHello로 그중 하나를 고르고 자신의 인증서를 내밉니다. 이어 키 교환으로 양쪽만 아는 대칭키를 세우고, 이후의 모든 바이트는 그 대칭키로 암호화됩니다. 공개키 암호는 이 첫 악수에서 키를 안전하게 세우는 데만 쓰이고, 실제 데이터는 훨씬 빠른 대칭 암호가 나른다는 점은 그림을 그려 보고 나서야 정리됐습니다.

이후 설정을 좌우한 개념은 전방 안전성(forward secrecy)이었습니다. 키 교환에 ECDHE처럼 세션마다 버려지는 임시 키를 쓰면, 훗날 서버의 개인키가 유출되어도 과거에 오간 트래픽은 복호화되지 않습니다. TLS 1.3은 전방 안전성이 없는 방식을 스펙에서 제거했고, 왕복도 한 번으로 줄여 handshake가 그만큼 가벼워졌습니다. 그래서 제가 정한 기준은 1.2 이상만 열고, 임시 키 교환만 쓰는 것이었습니다.

핸드셰이크는 비싼 작업이라, 프록시 쪽에서 이 악수를 재사용하도록 맞춰 두었습니다. 세션 캐시나 세션 티켓으로 재접속의 왕복을 줄이고, OCSP Stapling으로 인증서 폐기 확인의 지연을 서버가 미리 삼켜 두는 식입니다. 큰 그림은 여기까지고, 남은 일은 이 그림을 NGINX 지시어로 옮기는 것이었습니다.

2. 프록시에서 TLS를 끝낸다

TLS 종단(termination)은 암호화된 연결을 프록시에서 풀고, 그 뒤 내부망은 평문 HTTP로 백엔드에 넘기는 구조입니다. 인증서와 프로토콜 정책을 한 곳에서 관리하게 되니 백엔드는 TLS를 몰라도 되고, 배포도 그만큼 단순해졌습니다. 아래가 제가 쓴 종단 서버 블록의 뼈대입니다.

# /etc/nginx/conf.d/app.conf
server {
    listen 443 ssl;
    http2 on;                       # NGINX 1.25.1+ 권장 방식
    server_name app.example.com;

    ssl_certificate     /etc/ssl/app/fullchain.pem;
    ssl_certificate_key /etc/ssl/app/privkey.pem;

    # 1.2 이상만 연다. 1.0/1.1은 완전히 닫는다
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-CHACHA20-POLY1305;

    # 악수를 재사용해 재접속의 왕복을 줄인다
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;

    # OCSP Stapling — 폐기 확인을 서버가 대신 물어와 붙인다
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 1.1.1.1 8.8.8.8 valid=300s;
}

여기서 한 번 헷갈렸던 게 있습니다. ssl_ciphers 목록은 TLS 1.2까지만 적용되고, TLS 1.3은 자신이 정한 기본 스위트를 씁니다. 1.3 쪽 스위트를 목록으로 조정하려다 한참 헤맸는데, 손댈 필요가 없는 부분이었고 위 목록은 1.2 클라이언트를 위한 안전망으로 남겨 두면 되는 것이었습니다. 모든 스위트를 ECDHE로 시작하게 맞춘 것도 같은 맥락으로, 앞 절에서 적은 전방 안전성을 프로토콜 수준에서 강제하기 위해서였습니다.

평문으로 들어온 80 포트는 닫아 두는 대신 301로 HTTPS에 넘겨, 예전 링크로 들어온 사용자도 그대로 도착하게 했습니다. 이 리다이렉트가 다음 절의 HSTS와 짝을 이룹니다.

server {
    listen 80;
    server_name app.example.com;
    return 301 https://$host$request_uri;
}

3. HSTS와 보안 헤더로 문을 잠근다

리다이렉트를 걸어 두어도 첫 요청이 평문으로 나가는 틈은 남습니다. 그 틈을 메우는 것이 HSTS로, 한 번 HTTPS로 응답을 받은 브라우저는 정해진 기간 동안 그 도메인에 평문 연결을 시도하지 않습니다. 여기에 방어 헤더 몇 개를 더 얹으면 콘텐츠 스니핑이나 클릭재킹 같은 공격은 응답 헤더 선에서 막을 수 있어서, 아래 네 줄을 함께 넣어 두었습니다.

# 한 번 HTTPS로 온 브라우저는 앞으로 평문을 시도하지 않는다
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options        "DENY" always;
add_header Referrer-Policy        "strict-origin-when-cross-origin" always;
add_header Content-Security-Policy "default-src 'self'; frame-ancestors 'none'" always;

끝에 붙은 always는 사소해 보여서 처음엔 그냥 지나쳤습니다. 이게 없으면 4xx·5xx 같은 오류 응답에는 헤더가 빠지는데, 공격 트래픽이 오히려 오류를 많이 유발한다는 걸 생각하면 오류 응답에야말로 붙어 있어야 할 헤더였습니다. HSTS의 max-age는 2년으로 넉넉히 잡았고, 서브도메인까지 걸 때는 그 아래 모든 호스트가 HTTPS로만 서비스되는지 먼저 확인했습니다.

*

NGINX의 add_header는 상속되지 않고 덮어씁니다. 하위 location이 자기만의 add_header를 하나라도 선언하면, 상위에서 물려주던 보안 헤더가 그 블록에서 통째로 사라집니다. 헤더를 한 곳에 모아 두거나, 경로마다 다시 선언했는지 curl -I로 확인해 두는 편이 안전합니다.

HSTS에는 preload 옵션이 더 있습니다. 브라우저 내장 목록에 도메인을 올려 첫 방문부터 평문 접속을 막는 방식인데, 들어가기는 쉽고 빠져나오기는 어렵다는 점이 걸렸습니다. 목록 반영과 회수가 브라우저 릴리스 주기를 타기 때문에, 준비가 덜 된 서브도메인이 하나라도 있으면 접속 불능이 오래 갈 수 있습니다. 그래서 저는 preload를 확신이 설 때 마지막에 켜는 스위치로 남겨 두고 있습니다.

4. 업스트림을 지키는 버퍼링과 타임아웃

프록시의 또 다른 역할은 백엔드를 바깥의 변덕에서 떼어놓는 일입니다. 느린 클라이언트나 죽은 노드, 오래 걸리는 요청이 그대로 업스트림까지 전달되면 애플리케이션 워커가 금세 소진됩니다. 버퍼링과 타임아웃, 헬스체크를 그 사이에 완충 장치로 두었습니다.

upstream app_backend {
    server 10.0.1.11:8080 max_fails=3 fail_timeout=15s;
    server 10.0.1.12:8080 max_fails=3 fail_timeout=15s;
    keepalive 32;                    # 업스트림 연결을 재사용
}

server {
    # ... TLS 설정은 위와 동일 ...
    location / {
        proxy_pass http://app_backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";          # keepalive를 살린다

        # 원래 요청의 맥락을 업스트림에 넘긴다
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 느린 업스트림이 프록시를 물고 늘어지지 않게
        proxy_connect_timeout 3s;
        proxy_send_timeout    30s;
        proxy_read_timeout    30s;

        # 버퍼링: 느린 클라이언트에서 업스트림을 떼어놓는다
        proxy_buffering   on;
        proxy_buffer_size 16k;
        proxy_buffers     8 16k;

        # 죽은 노드는 건너뛴다 (아래 주의 참고)
        proxy_next_upstream error timeout http_502 http_503;
    }
}

proxy_buffering on은 NGINX가 업스트림의 응답을 버퍼(필요하면 디스크까지)에 받아 두고 워커를 먼저 놓아준다는 뜻입니다. 그래야 응답을 조금씩 받아가는 느린 클라이언트가 백엔드 스레드를 오래 붙잡는 일을 막을 수 있습니다. 반대로 SSE나 스트리밍 응답에서는 버퍼링이 지연을 만들기 때문에, 그 경로에만 proxy_buffering off로 예외를 두었습니다. 켜고 끌지 정하는 기준은 이 응답이 실시간으로 흘러야 하는지 하나였습니다.

헬스체크는 두 종류를 구분해 두는 게 좋습니다. 오픈소스 NGINX가 제공하는 것은 max_fails·fail_timeout 기반의 수동 헬스체크로, 실패가 쌓인 노드를 일정 시간 빼는 방식입니다. 능동적으로 /health를 주기적으로 찔러 보는 액티브 헬스체크(health_check 지시어)는 NGINX Plus 기능이라, 오픈소스를 쓰는 동안에는 이 역할을 외부 모니터나 로드밸런서에 맡겼습니다.

*

proxy_next_upstream에 http_502 같은 상태를 넣으면 재시도가 생깁니다. 편리하지만, POST처럼 멱등하지 않은 요청이 재전송되면 결제나 주문이 중복 처리될 수 있습니다. 안전하지 않은 메서드에는 재시도 범위를 좁히거나 non_idempotent를 의식적으로 다뤄야 했습니다. 자동 재시도가 공짜는 아니었습니다.

X-Forwarded-For가 클라이언트 쪽에서 위조할 수 있는 헤더라는 것도 뒤늦게 알았습니다. 그래서 가장 바깥 프록시에서는 들어온 값을 신뢰하지 않고 새로 세팅해, 그 뒤 백엔드가 보는 클라이언트 IP는 믿을 수 있는 값이 되도록 맞춰 두었습니다.

5. 버전을 숨기는 작은 하드닝

기본 설정의 NGINX는 응답 헤더와 오류 페이지에 자기 버전을 적어 보냅니다. Server: nginx/1.25.3 한 줄은 스캐너 입장에서 어떤 취약점을 먼저 시도해 볼지 알려 주는 단서가 됩니다. 지우는 데 드는 비용이 거의 없어서 하드닝 항목 중에는 이걸 먼저 손봤습니다.

# 응답과 오류 페이지에서 버전 문자열을 지운다
server_tokens off;

# 업스트림이 흘리는 정보 헤더도 프록시에서 걷어낸다
proxy_hide_header X-Powered-By;
proxy_hide_header Server;

# 버전 서명이 남는 기본 오류 페이지를 자체 페이지로 교체
error_page 500 502 503 504 /50x.html;

물론 이것만으로 방어가 되지는 않습니다. 작정한 공격자는 응답의 미묘한 차이만으로도 서버 종류를 추정합니다. 그럼에도 버전을 숨겨 둔 것은, 실제로 들어오는 요청의 대부분이 사람이 아니라 버전 배너를 훑고 지나가는 자동 스캐너였기 때문입니다. 정보 노출을 줄이면 그 자동화된 소음의 상당 부분은 표적을 놓칩니다. Apache라면 같은 일을 ServerTokens Prod와 ServerSignature Off가 맡습니다.

이 감각은 SafeIn에서 백엔드를 혼자 설계하며 온프레미스로 vLLM을 서빙할 때 좀 더 분명해졌습니다. 클라우드의 관리형 방어막이 없는 환경에서는 바깥으로 나가는 응답 한 줄 한 줄이 그대로 공격 표면이 됩니다. 그래서 불필요한 정보는 흘리지 않는다는 원칙을 프록시 단의 기본값으로 두었습니다. 백엔드 코드가 무언가를 실수로 노출하더라도 경계에서 한 번 더 걸러지도록 해 둔 셈입니다.

6. 책 한 권: Bulletproof SSL and TLS

그때까지 저는 TLS 설정을 남의 것을 복사해 붙이는 수준으로 쓰고 있었는데, 이반 리스티치(Ivan Ristić)의 《Bulletproof SSL and TLS》를 읽고 나서야 각 줄의 의미를 짚을 수 있게 됐습니다. 저자는 서버 설정 등급을 매기는 SSL Labs 테스트와 초기 ModSecurity를 만든 사람이라, 이 책도 이론서보다는 실제로 무엇이 A+를 만드는지 아는 사람의 실무서에 가깝습니다.

책의 구성은 크게 네 덩어리로 읽힙니다. 앞부분은 프로토콜과 PKI를 다루면서 핸드셰이크가 실제로 어떻게 키를 세우고 인증서 체인이 어떻게 검증되는지 설명합니다. 가운데는 공격의 카탈로그로, BEAST·CRIME·Lucky 13·POODLE·Heartbleed 같은 이름들이 각각 어떤 가정을 깨뜨렸는지 짚습니다. 뒤로는 성능(세션 재사용·OCSP Stapling)과 배포·설정이 이어지고, OpenSSL로 직접 확인하는 법까지 다룹니다. 앞 절들에서 제가 켠 옵션 하나하나가 이 지도 위 어딘가에 놓여 있었습니다.

이 책에서 가장 오래 남은 것은 이국적인 암호를 고르는 것보다 올바른 설정이 더 중요하다는 태도였습니다. 화려한 스위트를 찾기 전에 낡은 프로토콜을 닫고, 전방 안전성을 강제하고, 설정을 단순하게 유지하라는 이야기입니다. 그 태도를 설정으로 옮기는 일은 암호 스위트 이름 한 줄을 읽을 줄 아는 데서 시작했습니다.

# 스위트 이름은 네 부분으로 읽힌다
ECDHE-ECDSA-AES128-GCM-SHA256
#  │     │      │       │
#  │     │      │       └ 해시(무결성): SHA-256
#  │     │      └ 대칭 암호(기밀성): AES-128-GCM
#  │     └ 인증(서버 신원): ECDSA 서명
#  └ 키 교환: ECDHE = Ephemeral → 전방 안전성

이렇게 읽고 나서는 ssl_ciphers 목록이 외워 쓰는 주문(呪文)이 아니라 정책의 문장으로 보였습니다. 그 정책이 맞게 적용됐는지는 리스티치가 만든 SSL Labs 테스트가 등급으로 되돌려 줍니다. 저는 책을 읽으면서 SSL Labs 점수를 올려 보는 순서로 익혔는데, 설정 하나를 바꿀 때마다 등급이 반응하니 무엇이 왜 필요한지 붙잡기 수월했습니다.

7. 정리 — 경계에서 끝내기

리버스 프록시를 두고 좋았던 점은 보안 설정을 한 곳에서 표준화할 수 있다는 것이었습니다. TLS 종단을 정리하고, HSTS와 보안 헤더로 HTTPS를 강제하고, 버퍼링·타임아웃·헬스체크로 업스트림을 지키고, 새어 나가는 정보를 줄이는 네 가지를 프록시에 넣어 두니 뒤에 선 백엔드는 본연의 로직만 보면 됐습니다. 경계에서 정리한 만큼 안쪽이 단순해졌습니다.

정리하고 보니 핸드셰이크를 이해하는 데서 시작해 《Bulletproof SSL and TLS》를 지도로 삼고, SSL Labs 등급으로 확인하는 순서였습니다. 설정은 정해진 정책을 강제해 줄 뿐이고, 어떤 프로토콜을 열고 무엇을 숨길지는 제가 판단해야 하는 부분이었습니다. 지금도 preload처럼 아직 확신이 서지 않는 항목이 남아 있어서, 다음에 같은 설정을 만질 때는 이 글부터 다시 열어 볼 생각입니다.
