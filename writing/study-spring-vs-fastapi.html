학습 노트: Spring과 FastAPI를 오가며 배운 것

2025년 9월• 약 11분• Spring · FastAPI · 비교

한동안 오전에는 Spring으로 정산 모듈을 만지고, 오후에는 FastAPI로 크롤링 파이프라인과 vLLM 게이트웨이를 짰습니다. 두 세계를 오갈 때 진짜 비용은 문법이 아니라 머릿속 모델을 갈아 끼우는 일이었습니다. 둘 다 의존성 주입·검증·트랜잭션이라는 같은 문제를 푸는데, 방향이 정반대입니다 — 한쪽은 컨테이너로, 한쪽은 함수 시그니처로. 이 노트는 같은 REST 엔드포인트를 양쪽으로 구현해 그 철학 차이를 비교하고, 제가 DI의 뿌리를 이해하게 해준 『토비의 스프링』을 그 렌즈로 다시 읽습니다. 어느 쪽이 낫다가 아니라, 언제 무엇을 고르느냐의 이야기입니다.

먼저 짚고 가기
의존성 주입Dependency Injection, DI
어떤 객체가 필요로 하는 다른 객체를 스스로 만들지 않고 밖에서 받아 쓰는 방식입니다. 요리사가 재료를 직접 밭에서 기르지 않고 배달받는 것과 비슷합니다. 이 글은 Spring과 FastAPI가 이 '재료 배달'을 각각 어떻게 하는지를 축으로 삼습니다.
DTOData Transfer Object
계층 사이에서 데이터를 실어 나르기 위한 전용 상자 객체입니다. 요청용(RegisterWorker)과 응답용(WorkerResponse)을 나눠 두면, 받는 데이터와 돌려주는 데이터의 모양을 따로 관리할 수 있어 안전합니다.
트랜잭션Transaction
여러 DB 작업을 하나로 묶어 전부 성공하거나 전부 취소되게 하는 단위입니다. 계좌 이체에서 출금과 입금이 함께 성사되어야 하는 것과 같습니다. 두 프레임워크가 이 '묶는 경계'를 어디에 긋느냐가 5장의 주제입니다.
이벤트 루프Event Loop
한 명의 일꾼이 여러 일을 번갈아 처리하는 방식입니다. 어떤 일이 응답을 기다리는 동안 놀지 않고 다른 일을 처리하다가, 답이 오면 다시 이어서 합니다. 여러 서버에 동시에 요청을 뿌리고 기다리는 작업에서 힘을 발휘합니다.
프록시Proxy
진짜 객체를 감싸서 대신 받는 대리인 객체입니다. 요청이 진짜 객체에 닿기 전에 프록시가 먼저 가로채 트랜잭션 열기 같은 부가 작업을 몰래 끼워 넣습니다. Spring의 @Transactional이 이 원리로 동작합니다.

1. 같은 엔드포인트, 두 개의 세계

비교를 추상적으로 하면 끝이 없으니 하나의 엔드포인트에 못 박겠습니다. 산업안전 시스템에서 흔한 근로자 등록 — 요청을 검증하고, 서비스가 저장하고, 결과를 돌려주는 POST입니다. 먼저 Spring입니다.

@RestController
@RequestMapping("/api/workers")
public class WorkerController {
  private final WorkerService service;   // 생성자 주입 — 컨테이너가 넣어준다
  public WorkerController(WorkerService service) { this.service = service; }

  @PostMapping
  public ResponseEntity<WorkerResponse> register(@Valid @RequestBody RegisterWorker req) {
    return ResponseEntity.status(HttpStatus.CREATED).body(service.register(req));
  }
}

// 요청 DTO — Bean Validation 애너테이션이 곧 규칙이다
public record RegisterWorker(
    @NotBlank String name,
    @Email    String email,
    @Min(1)   long siteId) {}

@Service
public class WorkerService {
  private final WorkerRepository repository;
  public WorkerService(WorkerRepository repository) { this.repository = repository; }

  @Transactional
  public WorkerResponse register(RegisterWorker req) {
    Worker saved = repository.save(Worker.from(req));
    return WorkerResponse.from(saved);
  }
}

같은 엔드포인트를 FastAPI로 옮기면 이렇게 됩니다.

from typing import Annotated
from fastapi import Depends, FastAPI, status
from pydantic import BaseModel, EmailStr, Field
from sqlalchemy.ext.asyncio import AsyncSession

app = FastAPI()

class RegisterWorker(BaseModel):
    name: str = Field(min_length=1)
    email: EmailStr
    site_id: int = Field(ge=1)          # 타입과 제약이 곧 검증이다

class WorkerResponse(BaseModel):
    id: int
    name: str

@app.post("/api/workers", status_code=status.HTTP_201_CREATED)
async def register(req: RegisterWorker,
                   session: Annotated[AsyncSession, Depends(get_session)]) -> WorkerResponse:
    worker = Worker.from_request(req)
    session.add(worker)
    await session.flush()               # 트랜잭션 경계는 get_session이 그린다
    return WorkerResponse(id=worker.id, name=worker.name)

두 코드는 겉으로 놀랍도록 닮았습니다. 검증 → 서비스 → 저장 → 응답이라는 뼈대가 같고, 요청 DTO와 응답 DTO를 나눈 것도 같습니다. 차이는 눈에 보이는 코드가 아니라 조각들이 언제, 어떻게 조립되는가에 있습니다. Spring은 시작할 때 컨테이너가 그래프를 통째로 엮어 싱글턴으로 세워 두고, FastAPI는 요청이 올 때마다 함수 시그니처를 읽어 그 요청만의 조각을 맞춥니다. 이 한 문장의 차이가 DI·검증·비동기·트랜잭션 전부로 번집니다. 하나씩 보겠습니다.

2. 컨테이너 대 시그니처

Spring의 세계는 컨테이너가 중심입니다. @Service·@Repository가 붙은 클래스를 애플리케이션이 뜰 때 스캔해 ApplicationContext가 객체 그래프를 통째로 엮고, 대부분 싱글턴으로 한 번만 만들어 재사용합니다. 객체의 생성과 조립·수명을 전부 컨테이너가 소유하고, 내 코드는 생성자로 받기만 합니다. FastAPI에는 그런 전역 컨테이너가 없습니다. Depends가 요청이 올 때마다 핸들러의 시그니처를 읽어 그 요청만의 의존성 트리를 그때그때 조립합니다. 같은 DI인데 미리 통째로와 매번 그때그때로 갈립니다.

이 차이를 원리로 이해하게 된 건 『토비의 스프링』 덕이었습니다. 1권은 스프링 문법으로 시작하지 않습니다. UserDao 하나가 직접 DB 커넥션을 만드는 평범한 코드에서 출발해, '이게 왜 문제인가'를 집요하게 파고듭니다.

// 출발점: UserDao가 커넥션 만드는 법을 "직접" 안다 — 강하게 결합돼 있다
public class UserDao {
  public void add(User user) throws SQLException {
    Connection c = DriverManager.getConnection(...);  // DB가 바뀌면 여기가 바뀐다
    ...
  }
}

// 관심사를 인터페이스 뒤로 밀어낸다 — "어떻게 만드는지"를 밖으로
public interface ConnectionMaker { Connection makeConnection() throws SQLException; }

public class UserDao {
  private final ConnectionMaker connectionMaker;      // 사용하지만 만들지 않는다
  public UserDao(ConnectionMaker connectionMaker) {   // 그럼 누가 넣어주나? → 컨테이너
    this.connectionMaker = connectionMaker;
  }
}

책은 답을 먼저 주는 대신 질문을 반복합니다 — 누가 만들고, 누가 조립하는가. 커넥션 만드는 책임을 ConnectionMaker 인터페이스 뒤로 밀어내자 UserDao는 '사용하지만 만들지 않는' 코드가 되고, 그러면 '그럼 누가 넣어주지?'라는 질문이 남습니다. 그 질문의 끝에 DaoFactory가, 다시 그 끝에 스프링 컨테이너가 자연스럽게 놓입니다. DI를 기능으로 소개하지 않고 객체지향 설계 원칙(OCP·DIP)에서 연역해 낸다는 게 이 책의 태도이고, 저는 여기서 DI가 스프링의 발명이 아니라 스프링이 산업화한 원리라는 걸 배웠습니다.

그 렌즈를 끼고 FastAPI를 보면 Depends가 더 이상 낯선 마법이 아닙니다. 컨테이너가 생성자를 읽어 조립하던 일을, FastAPI는 inspect로 함수 시그니처를 읽어 요청 단위로 할 뿐입니다. '사용하는 쪽은 만들지 않는다'는 원리는 똑같고, 조립도를 어디에 적는가만 다릅니다 — 컨테이너라는 한 곳에 모으느냐, 시그니처마다 흩뿌리느냐. 이 이야기의 실전은 따로 정리해 두었으니 여기서는 원리만 짚고 넘어갑니다.

3. 검증 — 애너테이션 대 타입

Spring에서 검증은 횡단 관심사입니다. DTO에 @NotBlank·@Email 같은 Bean Validation 애너테이션으로 규칙을 선언하면, 이미 만들어진 객체를 프레임워크가 @Valid 시점에 훑어 규칙을 적용합니다. 실패하면 MethodArgumentNotValidException이 튀고, 보통 @RestControllerAdvice에서 공통으로 400으로 변환합니다. 검증이 객체 생성과 별개의 단계이고, 실패 처리도 별도의 자리에서 이뤄지는 구조입니다.

Pydantic의 태도는 정반대입니다. 타입 자체가 검증기입니다. 파싱과 검증이 같은 행위라서, 모델이 만들어졌다는 것은 이미 규칙을 통과했다는 뜻입니다. 잘못된 입력은 모델 생성 단계에서 걸려 내 함수 본문에 도달조차 못 합니다 — FastAPI는 그 실패를 자동으로 422로 돌려주고, 저는 예외 핸들러도 @Valid 같은 트리거도 쓰지 않았습니다. 도메인 규칙까지 타입 안에 넣을 수 있습니다.

from pydantic import field_validator

class RegisterWorker(BaseModel):
    site_id: int = Field(ge=1)

    @field_validator("site_id")
    @classmethod
    def must_be_active_site(cls, v: int) -> int:
        if v not in ACTIVE_SITE_IDS:
            raise ValueError("inactive site")   # 객체가 만들어지는 바로 그 순간 검증된다
        return v

정리하면 Spring은 객체를 만든 뒤 검사하고, Pydantic은 생성과 검증을 한 몸으로 만듭니다. 전자는 검증을 한곳에서 일괄 관리하기 좋고, 후자는 '유효한 값'과 '아무 값'을 타입 수준에서 갈라놓아 함수 안쪽이 방어 코드로부터 자유로워집니다. 파싱을 통과한 데이터는 이미 신뢰할 수 있다 — 이 감각이 FastAPI로 코드를 짤 때 가장 편한 지점이었습니다.

4. 비동기 — 스레드 풀 대 이벤트 루프

Spring MVC의 기본 모델은 요청 하나에 스레드 하나입니다. 블로킹 코드를 써도 넉넉한 스레드 풀이 흡수하고, @Transactional은 스레드에 묶인(thread-local) 트랜잭션을 전제로 동작합니다. 비동기·논블로킹이 필요하면 WebFlux(Reactor)라는 별도의 세계로 넘어가야 하는데, 사실상 프로그래밍 모델 전체가 바뀝니다. 반면 FastAPI는 처음부터 단일 이벤트 루프 위에서 돕니다. async def 핸들러는 await 지점에서 제어를 양보하고, 그 사이 루프는 다른 요청을 처리합니다. 스레드에 묶인 트랜잭션 같은 개념은 없습니다.

이 차이가 실무에서 갈린 순간이 뚜렷합니다. 온프레미스 vLLM 게이트웨이는 여러 백엔드로 동시에 요청을 뿌리고 기다리는 I/O 바운드 작업이라, 이벤트 루프가 정확히 물 만난 곳이었습니다.

import asyncio

@app.post("/v1/ensemble")
async def ensemble(req: Prompt,
                   backends: Annotated[list[Backend], Depends(get_backends)]):
    # 여러 vLLM 백엔드에 동시에 던지고, 기다리는 동안 루프는 놀지 않는다
    results = await asyncio.gather(*(b.generate(req.text) for b in backends))
    return {"best": pick_best(results)}

반대로 Spring으로 만든 정산 모듈은 CPU와 DB에 붙은 동기적 작업이 대부분이었고, 스레드-퍼-리퀘스트 모델과 스레드에 묶인 @Transactional이 오히려 사고를 단순하게 해줬습니다. 비동기가 공짜가 아니라는 걸 여기서 배웠습니다. 팬아웃·스트리밍처럼 기다림이 많은 일에는 이벤트 루프가 압도적이지만, 경계가 또렷한 트랜잭션 처리에는 블로킹 모델의 단순함이 자산입니다.

*

FastAPI에서 async def 핸들러는 이벤트 루프에서, 평범한 def 핸들러는 스레드풀에서 돕니다. 그래서 동기 블로킹 드라이버를 async def 안에서 부르면 루프 전체가 그 시간만큼 멈춰, 무관한 다른 요청까지 함께 느려집니다. 비동기로 갈 거면 드라이버도 끝까지 비동기여야 하고, 어중간하면 차라리 def로 두어 스레드풀에 맡기는 편이 안전합니다.

5. 트랜잭션 — 선언 대 명시

트랜잭션 경계를 긋는 방식에서 두 프레임워크의 성격이 가장 선명하게 드러납니다. Spring의 @Transactional은 프록시 기반 AOP입니다. 트랜잭션을 열고 닫는 코드는 빈을 감싼 프록시에 숨어 있고, 메서드 본문에는 경계가 보이지 않습니다. 보일러플레이트가 사라지는 대신, 어디서 트랜잭션이 시작되고 끝나는지가 애너테이션 뒤로 감춰집니다.

// Spring: 경계가 애너테이션 뒤에 숨는다 — 프록시가 열고 닫는다
@Transactional
public WorkerResponse register(RegisterWorker req) {
    Worker saved = repository.save(Worker.from(req));
    events.publish(new WorkerRegistered(saved.id()));  // 같은 트랜잭션 안
    return WorkerResponse.from(saved);
}

FastAPI에는 이런 프록시가 없습니다. 트랜잭션 경계는 SQLAlchemy 세션의 컨텍스트 매니저로 코드에 직접 긋고, 흔히 그 경계를 의존성 함수 하나에 넣어 요청 수명에 맞춥니다.

# FastAPI/SQLAlchemy: 경계를 코드로 직접 긋는다 — 숨은 프록시가 없다
async def get_session() -> AsyncIterator[AsyncSession]:
    async with Session() as session:
        async with session.begin():     # BEGIN … COMMIT/ROLLBACK이 이 블록이다
            yield session          # 핸들러가 끝나면 여기로 돌아와 커밋/롤백

이건 취향이 아니라 교환입니다. @Transactional은 반복을 없애주지만, 경계가 코드에 없으니 통제도 프레임워크의 규칙을 따라야 합니다. 명시적 세션은 매번 async with를 써야 하는 대신, 경계가 내가 쓴 그 자리에 정확히 있습니다. 그리고 이 명시성은 함정 하나를 통째로 없앱니다.

*

Spring의 선언적 경계는 프록시 위에 있어서, 같은 클래스 안에서 this.method()로 부르면 경계가 조용히 무시됩니다(자기 호출 함정 — 전파 글에서 자세히 다뤘습니다). FastAPI 쪽엔 이 함정이 존재할 수 없습니다. 경계가 async with session.begin()이라는 눈에 보이는 코드라서, 건너뛰려면 내가 직접 안 써야 하기 때문입니다. 숨겨서 편한 것과, 드러나서 안전한 것의 교환입니다.

덧붙이면 『토비의 스프링』이 트랜잭션을 다루며 강조한 건 애너테이션이 아니라 서비스 추상화(PSA)였습니다. JDBC든 JPA든 JTA든, 코드는 PlatformTransactionManager라는 하나의 추상에만 기대게 만든다는 것. 정작 중요한 가치는 '선언적'이라는 편의가 아니라 기술이 바뀌어도 흔들리지 않는 추상에 있었고, FastAPI는 그 추상을 프레임워크가 주는 대신 세션이라는 형태로 손수 잡는 셈입니다.

6. 타입 시스템이 만드는 설계 차이

이 모든 차이의 뿌리를 거슬러 가면 결국 타입 시스템이 나옵니다. Java는 명목적(nominal) 타이핑에 컴파일 타임 검사입니다. 그래서 Spring은 인터페이스와 프록시에 크게 기댑니다 — 구현체는 인터페이스를 implements로 명시적으로 선언하고, AOP는 JDK 동적 프록시나 CGLIB로 런타임에 대상 객체를 감쌉니다. 조립은 시작 시점의 리플렉션이 담당하고, 대부분의 오류는 컴파일러가 먼저 잡습니다.

Python은 점진적(gradual) 타이핑에 구조적(structural) 타이핑, 그리고 런타임 검사입니다. FastAPI에서 타입 힌트는 '힌트'가 아니라 실행에 관여하는 실물입니다. inspect가 시그니처를 읽어 의존성과 파라미터를 해석하고, Pydantic은 힌트를 검증기로 컴파일합니다. 인터페이스도 implements 없이 모양만 맞으면 성립합니다.

# Python: 구조적 타이핑 — 이 모양이면 이 타입이다 (implements 선언이 없다)
class WorkerRepo(Protocol):
    async def save(self, w: Worker) -> Worker: ...

같은 '인터페이스'라는 단어가 두 세계에서 다른 것을 뜻합니다. Java에서는 계약을 먼저 선언하고 구현이 거기 서명하는 반면, Python의 Protocol은 필요한 모양을 뒤늦게 기술해도 기존 클래스가 소급해 만족합니다. 그래서 Spring 코드는 계약을 앞세우는 무게가 있고, FastAPI 코드는 가볍게 시작해 나중에 타입으로 조이는 유연함이 있습니다. 오류가 드러나는 시점도 다릅니다 — 한쪽은 컴파일러가, 한쪽은 요청이 들어와 데이터를 파싱하는 순간이. 어느 쪽이 낫다기보다, 안전을 앞에서 사느냐 뒤에서 사느냐의 차이입니다.

7. 정리 — 언제 무엇을 선택하나

두 프레임워크를 오간 끝에 남은 판단 기준을 목록으로 정리합니다. 전부 '어느 게 좋다'가 아니라 '언제 무엇을'의 규칙입니다.

  • 강한 트랜잭션 정합성이 핵심이고 팀이 JVM에 익숙하면 → Spring. 선언적 트랜잭션과 성숙한 생태계가 동기적 업무 로직의 무게를 잘 받쳐줍니다. 정산 모듈이 그랬습니다.
  • I/O 바운드 동시성·ML/데이터 생태계·빠른 반복이 핵심이면 → FastAPI. 이벤트 루프의 팬아웃과 가벼운 기동이 크롤링·vLLM 게이트웨이에 맞았습니다.
  • 검증을 도메인 타입에 붙이고 싶으면 → Pydantic, 횡단 관심사로 일괄 처리하고 싶으면 → Bean Validation. 응집이냐 일관된 처리냐의 선택입니다.
  • 경계를 숨겨 보일러플레이트를 줄이려면 → @Transactional, 경계를 눈에 보이게 통제하려면 → 명시적 세션. 편의와 통제의 교환입니다.

2020년에 저는 ETRI에서 CNN 객체탐지를 다루며 Python으로 개발을 시작했고, Java와 Spring은 이후 실무를 거치며 익혔습니다. 두 언어를 양쪽에서 밀고 들어온 셈인데, 그 덕에 얻은 가장 큰 소득은 문법 두 벌이 아니었습니다. Depends든 컨테이너든 결국 사용하는 쪽은 만들지 않는다는 한 원리의 두 구현이고, 선언적 트랜잭션이든 명시적 세션이든 관심사를 추상 뒤로 밀어낸다는 같은 목표의 두 표현이라는 것 — 『토비의 스프링』이 스프링 문법이 아니라 그 원리를 가르치려 했던 이유를, FastAPI를 만지고 나서야 완전히 이해했습니다. 두 프레임워크를 오간 진짜 배당금은 도구를 원리로 착각하지 않게 된 것이었습니다. 그러고 나면 다음 도구가 무엇이든, 새로 배우는 건 문법 몇 개뿐입니다.
