논문 분석: Kafka — 로그라는 단순한 진실

2023년 4월• Kafka · 이벤트 · 논문

메시지 큐라고 하면 떠오르는 장치들 — 브로커가 소비 여부를 추적하고, 전달을 확인하고, 못 받은 메시지를 재전송하는 복잡한 기계 — 을 이 2011년 논문은 대부분 버립니다. 남긴 것은 디스크에 이어 쓰기만 하는 추가 전용 로그와, 어디까지 읽었는지를 가리키는 정수 하나(오프셋)뿐입니다. LinkedIn의 Jay Kreps 팀이 쓴 Kafka 논문은 기능을 더해서가 아니라 덜어내서 처리량을 얻은 설계의 교과서라서, 십 년이 지난 지금 읽어도 배울 게 많습니다.

출처

Kafka: a Distributed Messaging System for Log Processing

Jay Kreps, Neha Narkhede, Jun Rao · NetDB 2011

1. 왜 기존 메시징으로는 안 됐나

논문 1~2장의 문제의식은 구체적입니다. LinkedIn이 다루려던 건 주문이나 결제 같은 "귀한" 데이터가 아니라 로그인, 페이지 뷰, 클릭 같은 활동 이벤트였습니다. 건당 가치는 낮은데 양은 실데이터보다 몇 배 크고, 오프라인 분석만이 아니라 실시간 소비도 필요했습니다.

당시의 선택지는 두 진영이었습니다. JMS 계열의 전통 메시징 시스템은 메시지 하나하나에 수신 확인을 붙이는 식의 강한 전달 보장에 초점이 맞춰져 있었습니다. 로그 데이터에는 과한 보장이고, 그 보장을 유지하는 비용 때문에 처리량이 1급 설계 목표가 되지 못했습니다. 반대편의 Scribe·Flume 같은 로그 수집기는 대체로 데이터를 HDFS에 밀어 넣는 오프라인 소비를 전제했고, 브로커가 소비자에게 밀어 주는 push 모델이었습니다. Kafka는 이 사이의 빈자리 — 대용량이면서 실시간이고, 소비자가 주도권을 갖는 메시징 — 를 겨냥합니다.

2. 추가 전용 로그와 오프셋 — 브로커에서 상태를 지우다

논문 3.1의 저장 구조는 허무할 만큼 단순합니다. 파티션 하나는 논리적으로 로그 하나이고, 물리적으로는 비슷한 크기(논문 기준 약 1GB)의 세그먼트 파일들입니다. 메시지가 오면 마지막 세그먼트 끝에 이어 쓰고, 끝. 더 놀라운 건 메시지 ID가 별도로 없다는 점입니다. 메시지는 로그 안에서의 논리적 오프셋으로 주소가 정해지고, 다음 메시지의 오프셋은 현재 오프셋에 메시지 길이를 더해 계산합니다. 메시지 ID를 실제 위치로 바꿔 줄 B-tree 같은 색인 구조가 통째로 사라집니다.

# 브로커의 본질 (논문 3.1 Simple storage) — 상태는 로그와 위치뿐
class Partition:
    def __init__(self):
        self.log = []              # 추가 전용 — 수정도, 중간 삭제도 없다
        self.base = 0              # 보존 기간이 지나 잘려 나간 앞부분

    def append(self, message):   # 생산자를 위한 API의 전부
        self.log.append(message)
        return self.base + len(self.log) - 1    # 이 메시지의 오프셋

    def read(self, offset, max_n):             # 소비자를 위한 API의 전부
        start = offset - self.base
        return self.log[start : start + max_n]

# 여기에 "누가 어디까지 읽었나"는 없다 — 그 상태는 컨슈머의 소유물이다

이 단순함의 정점이 무상태 브로커(stateless broker)입니다. 전통적인 브로커는 소비자마다 "받았는지"를 기록하고, 받았으면 지우고, 못 받았으면 재전송합니다. Kafka 브로커는 이 장부를 아예 쓰지 않습니다. 어디까지 읽었는지는 컨슈머가 스스로 기억하고, 브로커는 소비 여부와 무관하게 시간 기준으로만 데이터를 지웁니다 — 논문 3.1 기준으로 보통 7일입니다. 소비 추적, 수신 확인, 재전송 큐, 이 세 가지 복잡성이 "컨슈머가 오프셋을 소유한다"는 결정 하나로 브로커에서 통째로 증발하는 것입니다.

3. 컨슈머가 오프셋을 소유한다 — pull 모델의 진짜 의미

브로커가 밀어주는(push) 대신 컨슈머가 당겨 가는(pull) 방식은 단순한 취향 문제가 아닙니다. push 모델에서는 소비자가 느려지면 브로커가 선택을 강요받습니다 — 버퍼에 쌓아 두다 터지거나, 버리거나. pull 모델에서는 소비자가 자기 속도로 가져가면 그만이고, 밀린 만큼은 로그에 그대로 남아 있습니다. 처리 속도의 주도권이 데이터를 받는 쪽에 있는 것이죠.

그리고 오프셋을 컨슈머가 소유하기 때문에 생기는 능력이 하나 더 있습니다. 논문 3.1이 "중요한 부수 효과"로 짚는 되감기(rewind)입니다. 소비란 오프셋을 앞으로 밀어가는 일일 뿐이므로, 오프셋을 뒤로 돌리면 같은 데이터를 다시 읽을 수 있습니다. 소비가 곧 삭제인 전통 큐에서는 불가능한 일입니다. 처리 로직에 버그가 있었음을 뒤늦게 알았을 때, 데이터는 로그에 그대로 있으니 오프셋만 되돌려 재처리하면 됩니다.

*

공짜는 아닙니다. 논문 3.3(Delivery Guarantees)이 밝히듯 Kafka의 보장은 at-least-once — 컨슈머가 오프셋을 저장하기 전에 죽으면 재시작 후 일부 메시지를 중복으로 받습니다. 그래서 소비 쪽 처리는 멱등(idempotent)하게 — 같은 메시지를 두 번 처리해도 결과가 같게 — 만들어야 합니다. 이 원칙은 Kafka를 안 쓰는 파이프라인에서도 유효합니다.

4. 파티션 — 순서 보장의 정확한 범위

로그가 하나뿐이면 쓰기가 한 지점에 몰려 확장이 안 됩니다. Kafka의 답은 토픽을 여러 파티션으로 쪼개는 것인데, 여기서 논문 3.3의 문장을 정확히 읽어야 합니다. Kafka가 보장하는 순서는 "한 파티션에서 나온 메시지가 컨슈머에게 전달되는 순서"까지입니다. 파티션끼리는 순서가 없습니다. 대신 같은 키(예: 같은 사용자 ID)를 가진 메시지를 해시로 같은 파티션에 보내면, 그 키에 대해서는 전체 순서가 생깁니다. 순서가 필요한 범위를 키로 선언하는 셈이고, 그래서 키 설계가 곧 순서 설계입니다.

파티션은 병렬성의 최소 단위이기도 합니다(논문 3.2). 한 파티션은 컨슈머 그룹 안에서 정확히 한 컨슈머에게 배정되므로, 파티션 안에서 락이나 순서 조정이 필요 없습니다. 그럼 "누가 어느 파티션을 맡나"는 누가 정할까요? 중앙 마스터가 없습니다. 각 컨슈머가 ZooKeeper에서 같은 멤버 목록을 읽고, 같은 결정적 알고리즘을 돌려 각자 자기 몫을 계산합니다. 모두가 같은 입력에 같은 계산을 하니 조율 없이도 결론이 같습니다.

# 논문 3.2의 재할당(rebalance) — 마스터 없이 모두가 같은 결론에 도달한다
def my_partitions(partitions, consumers, me):
    parts   = sorted(partitions)      # 모두가 같은 순서로 정렬하고
    members = sorted(consumers)       # 자기 순번의 몫만 가져가면
    chunk   = len(parts) // len(members)
    i       = members.index(me)       # 조율자 없이도 배정이 겹치지 않는다
    return parts[i * chunk : (i + 1) * chunk]

# 컨슈머가 죽거나 새로 오면? 멤버 목록이 바뀐 걸 모두가 감지하고
# 같은 계산을 다시 돌린다 — 이것이 리밸런스의 전부다

말로는 여기까지입니다. 아래 위젯에서 직접 굴려 보세요. 메시지를 생산하면 키의 해시가 파티션을 정하고, 두 컨슈머 그룹 A(컨슈머 2명)와 B(컨슈머 1명)가 같은 로그를 각자의 오프셋으로 읽어 나갑니다. 리밸런스 버튼은 그룹 A의 컨슈머 한 명이 이탈했다 복귀하는 상황을 재연합니다.

파티션 로그와 컨슈머 그룹 오프셋직접 조작해 보세요
그룹 A의 다음 읽기 위치 그룹 B의 다음 읽기 위치 흐린 칸 = 두 그룹 모두 읽음
그룹 A: A1, A2 (파티션 분담) · 그룹 B: B1 (전 파티션)
P0A: A1 · B: B1
P1A: A2 · B: B1

키 해시가 파티션을 정하고, 그룹별 오프셋이 각자 속도로 전진합니다. 파티션이 가득 차면 가장 오래된 칸이 지워집니다(보존 기간).

몇 번 눌러 보면 논문의 문장들이 손에 잡힙니다. 소비 A만 연타해도 B의 마커는 제자리입니다 — 두 그룹의 오프셋이 독립이라는 뜻입니다. 리밸런스로 A2를 이탈시켜도 오프셋은 그대로입니다 — 오프셋은 컨슈머가 아니라 그룹의 소유라서, 파티션을 넘겨받은 A1이 A2가 읽던 자리에서 그대로 이어 읽습니다. 그리고 생산을 계속 누르다 보면 가장 오래된 칸이 지워지는데, 이것이 소비 여부와 무관한 시간 기준 보존입니다.

5. 순차 IO와 zero-copy — OS를 이기려 하지 않는다

논문 3.1의 나머지 절반은 효율 이야기인데, 관통하는 태도는 "운영체제가 이미 잘하는 일을 다시 하지 않는다"입니다. Kafka는 메시지를 애플리케이션 메모리에 캐시하지 않습니다. 쓰면 파일 시스템에 쓰고, 읽으면 파일 시스템에서 읽습니다. 그러면 느릴 것 같지만, 접근이 전부 순차라서 OS의 페이지 캐시와 미리 읽기(read-ahead)·모아 쓰기(write-behind)가 최고 효율로 동작합니다. 덤으로 캐시를 프로세스 밖(커널)에 두니 이중 버퍼링이 없고, JVM 힙이 가벼워 GC 부담이 줄고, 브로커를 재시작해도 캐시가 따뜻한 채로 남습니다.

소비 경로에는 한 가지 최적화가 더 붙습니다. 보통 파일을 소켓으로 보내려면 데이터가 네 번 복사되고 시스템 콜이 두 번 필요합니다. 브로커는 메시지 내용을 볼 필요가 없는데도 말이죠. 그래서 Kafka는 리눅스의 sendfile을 씁니다.

# 일반적인 파일 → 소켓 전송 (논문 3.1: 복사 4번 + 시스템 콜 2번)
read(file, app_buf)     # ① 디스크 → 페이지 캐시  ② 페이지 캐시 → 앱 버퍼
write(socket, app_buf)  # ③ 앱 버퍼 → 커널 버퍼    ④ 커널 버퍼 → NIC

# zero-copy 전송 — 앱은 데이터를 만지지 않는다
sendfile(socket, file)  # 페이지 캐시에서 소켓으로 바로 — 복사 2번 절약

이 효율 설계들이 합쳐진 결과가 논문 5장의 실험입니다. 생산자 처리량은 배치 크기 1에서 초당 약 5만 건, 배치 50에서 약 40만 건으로, 비교 대상인 ActiveMQ보다 한 자릿수 이상, RabbitMQ보다 2배 이상 높았습니다. 특별한 자료구조나 새 알고리즘 없이, 순차 IO + 페이지 캐시 + zero-copy + 무상태 브로커라는 덜어내기의 합만으로 나온 숫자입니다.

6. 크롤링 파이프라인에서 다시 만난 로그

제가 이 논문을 실감한 곳은 크롤링 파이프라인이었습니다. 수집기가 페이지를 가져오고, 파서가 해석하고, 저장 단계가 적재하는 흐름에서 단계 사이를 잇는 것이 결국 이벤트의 열(列)인데, 이 글의 개념들이 그대로 대응됩니다. 파싱이 수집보다 느려지는 시간대에도 pull 모델이라면 수집기는 제 속도로 쌓기만 하면 되고, 밀린 양은 lag(로그 끝과 오프셋의 거리)라는 숫자 하나로 보입니다. 저는 파이프라인 상태를 물을 때 이 질문이 제일 정직하다고 생각합니다 — "지금 어느 단계의 lag이 자라고 있는가."

순서 문제도 같습니다. 같은 사이트에 대한 수집 이벤트는 순서가 섞이면 곤란하지만, 사이트끼리는 순서가 필요 없습니다. 도메인을 키로 잡으면 정확히 필요한 범위만 순서가 보장되고 나머지는 병렬로 풀립니다. 그리고 파서를 고친 뒤 "지난 데이터를 다시 돌리고 싶다"는 요구는, 소비가 곧 삭제인 구조에서는 재수집이지만 로그 구조에서는 오프셋 되감기입니다. 파이프라인을 로그 중심으로 볼수록 재처리가 무서운 일에서 사소한 일로 바뀝니다.

7. 정리 — 단순함은 기능이다

이 논문에서 가져갈 것을 세 줄로 줄이면 이렇습니다. 첫째, 상태의 소유자를 옮기면 복잡성이 사라진다 — 소비 상태를 컨슈머에게 넘기자 브로커의 추적·확인·재전송이 통째로 없어졌습니다. 둘째, 보장의 범위를 정직하게 좁히면 성능이 나온다 — 순서는 파티션 안까지, 전달은 at-least-once까지만 약속하고, 그 이상은 키 설계와 멱등성으로 사용자에게 돌려줬습니다. 셋째, 플랫폼이 잘하는 일은 플랫폼에 맡긴다 — 캐시는 OS의 페이지 캐시가, 전송은 sendfile이 합니다.

"로그라는 단순한 진실"이라고 제목을 붙인 이유가 여기 있습니다. 시스템의 상태를 "추가 전용 로그의 어느 위치"로 환원하고 나면, 소비는 전진이고 재처리는 되감기이며 지연은 거리입니다. 설계를 검토할 때마다 저는 이 논문의 질문을 다시 씁니다 — 이 상태는 누가 소유해야 하는가, 그리고 이 보장은 정말 여기서 해야 하는가.
