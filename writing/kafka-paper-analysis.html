논문 분석: Kafka — 로그라는 단순한 진실

2023년 4월• Kafka · 이벤트 · 논문

2011년에 나온 Kafka 논문을 읽고 정리해 둡니다. 제가 알던 메시지 큐는 브로커가 소비 여부를 추적하고, 전달을 확인하고, 못 받은 메시지를 재전송하는 물건이었는데, 이 논문의 설계에는 그 장치들이 대부분 없었습니다. 남은 것은 디스크에 이어 쓰기만 하는 추가 전용 로그와, 어디까지 읽었는지를 가리키는 정수 하나, 곧 오프셋이었습니다. LinkedIn의 Jay Kreps 팀은 기능을 더하는 대신 덜어내는 쪽으로 처리량을 얻었고, 십 년이 지나 읽는 저에게도 그 방식이 남는 게 많았습니다.

출처

Kafka: a Distributed Messaging System for Log Processing

Jay Kreps, Neha Narkhede, Jun Rao · NetDB 2011

새 레코드를 파일 끝에만 이어 붙이고, 이미 기록된 부분은 수정도 중간 삽입도 하지 않는 저장 자료구조입니다. Kafka에서는 파티션 하나가 논리적으로 이 로그 하나이며, 물리적으로는 비슷한 크기(논문 기준 약 1GB)의 세그먼트 파일이 이어진 형태입니다. 쓰기 지점이 항상 파일 끝이라 디스크 접근이 순차가 되고, 삭제는 개별 레코드 단위가 아니라 보존 기간이 지난 앞쪽 세그먼트를 통째로 잘라내는 방식으로만 일어납니다.
파티션partition
하나의 토픽(메시지 분류 이름)을 나눈 독립된 로그이자, Kafka에서 순서와 병렬성이 정의되는 유일한 범위입니다. 생산자가 키의 해시 같은 분할 함수로 목적지 파티션을 고르고, 논문 3.3이 약속하는 순서는 "한 파티션에서 나온 메시지가 컨슈머에게 전달되는 순서"까지입니다. 서로 다른 파티션 사이에는 순서 관계가 정의되지 않으므로, 토픽 전체의 전역 순서는 파티션을 하나로 두지 않는 한 성립하지 않습니다.
오프셋offset
로그 안에서 메시지가 시작하는 위치를 가리키는 정수 주소입니다. 논문의 Kafka는 메시지 ID나 위치 색인을 따로 두지 않고, 다음 메시지의 오프셋을 현재 오프셋에 메시지 길이를 더해 얻습니다. 소비 진행 상태는 (토픽, 파티션, 오프셋) 조합만으로 전부 표현되며, 이 값을 브로커가 아니라 소비하는 쪽이 보관합니다(논문에서는 ZooKeeper에 기록합니다). 다만 오프셋은 해당 파티션 로그 안에서만 유효한 좌표라, 보존 기간이 지나 앞부분이 잘려 나가면 그보다 작은 오프셋은 더 이상 읽을 수 없습니다.
컨슈머 그룹consumer group
같은 토픽을 함께 나눠 읽는 컨슈머들의 집합이자, 오프셋이 저장되는 단위입니다. 그룹 안에서 파티션 하나는 최대 한 컨슈머에게만 배정되므로 파티션 내부에는 락도 순서 조정도 필요 없고, 그룹이 다르면 서로의 진행 상태를 모른 채 같은 로그를 각자의 오프셋으로 읽습니다. 배정은 중앙 조율자 없이, 각 컨슈머가 ZooKeeper에서 같은 멤버 목록을 읽고 같은 결정적 알고리즘을 돌려 계산합니다. 이 규칙의 결과로 그룹의 컨슈머 수를 파티션 수보다 늘리면 남는 컨슈머는 아무 파티션도 받지 못합니다.
zero-copy
파일 데이터를 사용자 공간 버퍼로 복사하지 않고, 커널 안에서 페이지 캐시로부터 소켓으로 곧장 넘기는 전송 기법입니다. 리눅스의 sendfile 시스템 콜(자바에서는 FileChannel.transferTo)이 그 일을 하며, 논문 3.1은 이 방식으로 일반적인 read+write 경로의 복사 4번 중 2번과 시스템 콜 1번이 사라진다고 적습니다. 애플리케이션이 바이트를 전혀 만지지 않는다는 전제 위에서만 성립하므로, 브로커가 메시지 내용을 들여다보거나 변환해야 하는 순간 이 경로는 쓸 수 없습니다.

1. 왜 기존 메시징으로는 안 됐나

논문 1~2장은 문제 상황부터 설명합니다. LinkedIn이 다루려던 데이터는 주문이나 결제 같은 것이 아니라 로그인, 페이지 뷰, 클릭 같은 활동 이벤트였습니다. 건당 가치는 낮은데 양은 실데이터보다 몇 배 크고, 오프라인 분석뿐 아니라 실시간 소비도 필요했다고 합니다.

당시 선택지는 크게 둘이었습니다. JMS 계열의 전통 메시징 시스템은 메시지 하나하나에 수신 확인을 붙이는 식의 강한 전달 보장에 초점이 맞춰져 있었습니다. 로그 데이터에는 과한 보장이었고, 그 보장을 유지하는 비용 때문에 처리량이 1급 설계 목표가 되기 어려웠습니다. 다른 쪽의 Scribe나 Flume 같은 로그 수집기는 대체로 데이터를 HDFS에 밀어 넣는 오프라인 소비를 전제했고, 브로커가 소비자에게 밀어 주는 push 모델이었습니다. Kafka가 채우려 한 자리는 그 사이, 대용량이면서 실시간이고 소비자가 주도권을 갖는 메시징이었습니다.

2. 추가 전용 로그와 오프셋 — 브로커에서 상태를 지우다

논문 3.1의 저장 구조는 제가 예상한 것보다 단순했습니다. 파티션 하나는 논리적으로 로그 하나이고, 물리적으로는 비슷한 크기(논문 기준 약 1GB)의 세그먼트 파일들입니다. 메시지가 오면 마지막 세그먼트 끝에 이어 쓰는 것이 쓰기의 전부입니다. 처음 읽을 때 눈에 걸린 부분은 메시지 ID가 별도로 없다는 점이었습니다. 메시지는 로그 안에서의 논리적 오프셋으로 주소가 정해지고, 다음 메시지의 오프셋은 현재 오프셋에 메시지 길이를 더해 계산합니다. 메시지 ID를 실제 위치로 바꿔 줄 B-tree 같은 색인 구조가 통째로 필요 없어집니다.

# 브로커의 본질 (논문 3.1 Simple storage) — 상태는 로그와 위치뿐
class Partition:
    def __init__(self):
        self.log = []              # 추가 전용 — 수정도, 중간 삭제도 없다
        self.base = 0              # 보존 기간이 지나 잘려 나간 앞부분

    def append(self, message):   # 생산자를 위한 API의 전부
        self.log.append(message)
        return self.base + len(self.log) - 1    # 이 메시지의 오프셋

    def read(self, offset, max_n):             # 소비자를 위한 API의 전부
        start = offset - self.base
        return self.log[start : start + max_n]

# 여기에 "누가 어디까지 읽었나"는 없다 — 그 상태는 컨슈머의 소유물이다

이 구조의 중심에 무상태 브로커(stateless broker)가 있습니다. 전통적인 브로커는 소비자마다 "받았는지"를 기록하고, 받았으면 지우고, 못 받았으면 재전송합니다. Kafka 브로커는 그 장부를 두지 않습니다. 어디까지 읽었는지는 컨슈머가 스스로 기억하고, 브로커는 소비 여부와 무관하게 시간 기준으로만 데이터를 지웁니다. 논문 3.1 기준으로는 보통 7일입니다. 소비 추적과 수신 확인, 재전송 큐라는 세 가지 복잡성이 컨슈머가 오프셋을 소유한다는 결정 하나에서 함께 사라집니다.

3. 컨슈머가 오프셋을 소유한다 — pull 모델의 진짜 의미

브로커가 밀어주는(push) 대신 컨슈머가 당겨 가는(pull) 방식은 취향 차이로 읽히기 쉬운데, 논문을 보면 이유가 분명합니다. push 모델에서는 소비자가 느려지면 브로커가 버퍼에 계속 쌓아 두거나 메시지를 버리거나 둘 중 하나를 골라야 합니다. pull 모델에서는 소비자가 자기 속도로 가져가고, 밀린 만큼은 로그에 그대로 남습니다. 처리 속도의 주도권이 데이터를 받는 쪽에 있습니다.

오프셋을 컨슈머가 소유하면 따라오는 일이 하나 더 있습니다. 논문 3.1이 "중요한 부수 효과"로 짚는 되감기(rewind)입니다. 소비란 오프셋을 앞으로 밀어가는 일이므로, 오프셋을 뒤로 돌리면 같은 데이터를 다시 읽을 수 있습니다. 소비가 곧 삭제인 전통 큐에서는 할 수 없는 일입니다. 처리 로직에 버그가 있었다는 걸 뒤늦게 알아도 데이터는 로그에 그대로 있으니, 오프셋만 되돌려 재처리하면 됩니다.

*

대신 공짜는 아닙니다. 논문 3.3(Delivery Guarantees)이 밝히듯 Kafka의 보장은 at-least-once라서, 컨슈머가 오프셋을 저장하기 전에 죽으면 재시작 후 일부 메시지를 중복으로 받습니다. 그래서 소비 쪽 처리는 같은 메시지를 두 번 처리해도 결과가 같도록 멱등(idempotent)하게 만들어야 합니다. Kafka를 쓰지 않는 파이프라인에도 그대로 적용되는 이야기입니다.

4. 파티션 — 순서 보장의 정확한 범위

로그가 하나뿐이면 쓰기가 한 지점에 몰려 확장이 어렵습니다. Kafka는 토픽을 여러 파티션으로 쪼개는데, 이 지점에서는 논문 3.3의 문장을 그대로 읽는 편이 좋습니다. Kafka가 보장하는 순서는 "한 파티션에서 나온 메시지가 컨슈머에게 전달되는 순서"까지입니다. 파티션끼리는 순서가 없습니다. 같은 키(예: 같은 사용자 ID)를 가진 메시지를 해시로 같은 파티션에 보내면 그 키에 대해서는 전체 순서가 생깁니다. 순서가 필요한 범위를 키로 선언하는 셈이어서, 저는 이 글을 쓴 뒤로 키 설계를 순서 설계로 생각하게 됐습니다.

파티션은 병렬성의 최소 단위이기도 합니다(논문 3.2). 한 파티션은 컨슈머 그룹 안에서 정확히 한 컨슈머에게 배정되므로, 파티션 안에서 락이나 순서 조정이 필요 없습니다. 그럼 누가 어느 파티션을 맡을지는 누가 정하는지가 궁금했는데, 중앙 마스터가 따로 없었습니다. 각 컨슈머가 ZooKeeper에서 같은 멤버 목록을 읽고, 같은 결정적 알고리즘을 돌려 각자 자기 몫을 계산합니다. 모두가 같은 입력에 같은 계산을 하니 조율 없이도 결론이 같아집니다.

# 논문 3.2의 재할당(rebalance) — 마스터 없이 모두가 같은 결론에 도달한다
def my_partitions(partitions, consumers, me):
    parts   = sorted(partitions)      # 모두가 같은 순서로 정렬하고
    members = sorted(consumers)       # 자기 순번의 몫만 가져가면
    chunk   = len(parts) // len(members)
    i       = members.index(me)       # 조율자 없이도 배정이 겹치지 않는다
    return parts[i * chunk : (i + 1) * chunk]

# 컨슈머가 죽거나 새로 오면? 멤버 목록이 바뀐 걸 모두가 감지하고
# 같은 계산을 다시 돌린다 — 이것이 리밸런스의 전부다

글로 설명하기는 여기까지가 한계여서 아래에 위젯을 하나 붙여 뒀습니다. 메시지를 생산하면 키의 해시가 파티션을 정하고, 두 컨슈머 그룹 A(컨슈머 2명)와 B(컨슈머 1명)가 같은 로그를 각자의 오프셋으로 읽어 나갑니다. 리밸런스 버튼은 그룹 A의 컨슈머 한 명이 이탈했다 복귀하는 상황을 재연합니다.

파티션 로그와 컨슈머 그룹 오프셋직접 조작해 보세요
그룹 A의 다음 읽기 위치 그룹 B의 다음 읽기 위치 흐린 칸 = 두 그룹 모두 읽음
그룹 A: A1, A2 (파티션 분담) · 그룹 B: B1 (전 파티션)
P0A: A1 · B: B1
P1A: A2 · B: B1

키 해시가 파티션을 정하고, 그룹별 오프셋이 각자 속도로 전진합니다. 파티션이 가득 차면 가장 오래된 칸이 지워집니다(보존 기간).

몇 번 눌러 보면서 논문의 문장들이 눈에 들어왔습니다. 소비 A만 계속 눌러도 B의 마커는 제자리인데, 두 그룹의 오프셋이 독립이기 때문입니다. 리밸런스로 A2를 이탈시켜도 오프셋은 그대로입니다. 오프셋은 컨슈머가 아니라 그룹의 소유라서, 파티션을 넘겨받은 A1이 A2가 읽던 자리에서 그대로 이어 읽습니다. 생산을 계속 누르다 보면 가장 오래된 칸이 지워지는데, 이것이 소비 여부와 무관한 시간 기준 보존입니다.

5. 순차 IO와 zero-copy — OS를 이기려 하지 않는다

논문 3.1의 나머지 절반은 효율 이야기인데, 바탕에 깔린 태도는 "운영체제가 이미 잘하는 일을 다시 하지 않는다"입니다. Kafka는 메시지를 애플리케이션 메모리에 캐시하지 않습니다. 쓰면 파일 시스템에 쓰고, 읽으면 파일 시스템에서 읽습니다. 느릴 것 같지만 접근이 전부 순차라서 OS의 페이지 캐시와 미리 읽기(read-ahead)·모아 쓰기(write-behind)가 제 성능을 냅니다. 캐시를 프로세스 밖(커널)에 두니 이중 버퍼링이 없고, JVM 힙이 가벼워 GC 부담이 줄고, 브로커를 재시작해도 캐시가 따뜻한 채로 남는다는 이점도 함께 따라옵니다.

소비 경로에는 최적화가 하나 더 붙습니다. 보통 파일을 소켓으로 보내려면 데이터가 네 번 복사되고 시스템 콜이 두 번 필요합니다. 브로커는 메시지 내용을 볼 일이 없는데도 그렇습니다. 그래서 Kafka는 리눅스의 sendfile을 씁니다.

# 일반적인 파일 → 소켓 전송 (논문 3.1: 복사 4번 + 시스템 콜 2번)
read(file, app_buf)     # ① 디스크 → 페이지 캐시  ② 페이지 캐시 → 앱 버퍼
write(socket, app_buf)  # ③ 앱 버퍼 → 커널 버퍼    ④ 커널 버퍼 → NIC

# zero-copy 전송 — 앱은 데이터를 만지지 않는다
sendfile(socket, file)  # 페이지 캐시에서 소켓으로 바로 — 복사 2번 절약

이 설계들이 합쳐진 결과가 논문 5장의 실험입니다. 생산자 처리량은 배치 크기 1에서 초당 약 5만 건, 배치 50에서 약 40만 건으로, 비교 대상인 ActiveMQ보다 한 자릿수 이상, RabbitMQ보다 2배 이상 높았습니다. 새로운 자료구조나 알고리즘이 있어서 나온 숫자는 아니고, 순차 IO와 페이지 캐시, zero-copy, 무상태 브로커를 더한 결과입니다.

6. 크롤링 파이프라인에서 다시 만난 로그

제가 이 논문을 실감한 곳은 크롤링 파이프라인이었습니다. 수집기가 페이지를 가져오고, 파서가 해석하고, 저장 단계가 적재하는 흐름에서 단계 사이를 잇는 것이 결국 이벤트의 열(列)인데, 이 글의 개념들이 그대로 대응됐습니다. 파싱이 수집보다 느려지는 시간대에도 pull 모델이라면 수집기는 제 속도로 쌓기만 하면 되고, 밀린 양은 lag(로그 끝과 오프셋의 거리)라는 숫자 하나로 보입니다. 그래서 파이프라인 상태를 물을 때 저는 "지금 어느 단계의 lag이 자라고 있는가"부터 확인하게 됐습니다.

순서도 마찬가지였습니다. 같은 사이트에 대한 수집 이벤트는 순서가 섞이면 곤란하지만, 사이트끼리는 순서가 필요 없습니다. 도메인을 키로 잡으면 필요한 범위에만 순서가 보장되고 나머지는 병렬로 풀립니다. 파서를 고친 뒤 지난 데이터를 다시 돌리고 싶다는 요구도, 소비가 곧 삭제인 구조에서는 재수집이지만 로그 구조에서는 오프셋 되감기입니다. 파이프라인을 로그 중심으로 보고 나서는 재처리를 앞두고 덜 긴장하게 됐습니다.

7. 정리 — 단순함은 기능이다

읽고 나서 남은 것을 세 가지로 적어 둡니다. 첫째, 상태의 소유자를 옮기면 복잡성이 줄어듭니다. 소비 상태를 컨슈머에게 넘기자 브로커의 추적·확인·재전송이 함께 없어졌습니다. 둘째, 보장의 범위를 정직하게 좁히면 성능이 따라옵니다. 순서는 파티션 안까지, 전달은 at-least-once까지만 약속하고, 그 이상은 키 설계와 멱등성으로 사용자에게 돌려줬습니다. 셋째, 플랫폼이 잘하는 일은 플랫폼에 맡깁니다. 캐시는 OS의 페이지 캐시가, 전송은 sendfile이 합니다.

제목을 "로그라는 단순한 진실"로 붙인 이유가 여기 있습니다. 시스템의 상태를 "추가 전용 로그의 어느 위치"로 환원하고 나면, 소비는 전진이고 재처리는 되감기이며 지연은 거리입니다. 설계를 검토할 때 저는 이 논문에서 가져온 두 질문을 씁니다. 이 상태는 누가 소유해야 하는가, 그리고 이 보장은 정말 여기서 해야 하는가.
