Feature Engineering 실전 — 불쾌지수가 강수량을 이긴 이유

2024년 6월• EDA · 다중공선성 · 파생변수

같은 데이터, 같은 모델이라도 순위를 가르는 건 데이터를 어떻게 이해했는가였습니다. "다 넣으면 되겠지"로 시작했다가, EDA를 하며 생각이 완전히 바뀐 이야기입니다.

먼저 짚고 가기
EDAExploratory Data Analysis
본격적으로 모델을 만들기 전에 데이터를 그래프로 그려 보고 이리저리 뜯어보며 성격을 파악하는 단계입니다. 처음 만난 사람과 대화해 보며 성향을 알아가는 것과 비슷합니다. 이 글에서 "다 넣으면 되겠지"라는 생각이 뒤집힌 것도 바로 이 EDA 덕분입니다.
상관계수correlation, r
두 값이 얼마나 나란히 움직이는지를 −1에서 1 사이 숫자로 나타낸 것입니다. 1에 가까울수록 한쪽이 오를 때 다른 쪽도 같이 오릅니다. 다만 함께 움직인다고 해서 한쪽이 다른 쪽의 원인이라는 뜻은 아닙니다. 이 글에서 기온과 전력사용량의 r≈0.85가 그 예입니다.
다중공선성multicollinearity
모델에 넣는 입력 변수들끼리 서로 너무 닮아서 겹치는 상태입니다. 같은 이야기를 하는 사람을 회의에 여럿 부른 셈이라, 누구 말이 진짜 영향을 준 건지 구분이 어려워집니다. 그래서 각 변수의 계수(영향력)가 불안정해지고 해석이 흔들립니다.
VIFVariance Inflation Factor
다중공선성이 얼마나 심한지를 변수별로 재는 점수입니다. 보통 10을 넘으면 "겹침이 심하다"는 경고 신호로 봅니다. 눈대중이 아니라 숫자로 문제를 짚어 주기 때문에 어떤 변수를 손봐야 할지 판단하는 근거가 됩니다.
파생변수derived feature
원본 데이터를 그대로 쓰지 않고 조합·가공해 새로 만든 변수입니다. 기온과 습도를 합쳐 만든 불쾌지수처럼요. 이 글의 핵심은 바로 이 파생변수를 도메인 지식으로 잘 만드는 일이 모델 선택보다 순위를 크게 좌우했다는 것입니다.

상관은 신호이자 함정

기온은 전력사용량과 강한 상관(r≈0.85), 강수량은 거의 무관(r≈0.12)이었습니다. 여기까지는 쉽습니다. 함정은 기온·습도·풍속이 서로도 강하게 상관한다는 것 — 다중공선성입니다. 이걸 그대로 넣으면 계수가 불안정해지고 해석이 무너집니다.

# VIF로 다중공선성을 수치로 확인
from statsmodels.stats.outliers_influence import variance_inflation_factor
vif = [variance_inflation_factor(X.values, i) for i in range(X.shape[1])]
# temp, humidity, wind 이 10을 훌쩍 넘김 → 손봐야 한다

도메인 지식으로 압축한다

"사람이 냉방을 켜는 건 온도가 높아서가 아니라 불쾌해서가 아닐까?" 이 한 문장이 방향을 바꿨습니다. 얽힌 세 변수를 의미가 분명한 둘로 압축했습니다.

# 불쾌지수(Thom) — 기온+습도를 하나로
df["discomfort"] = 0.81*T + 0.01*RH*(0.99*T - 14.3) + 46.3
# 체감온도 — 기온+풍속을 하나로
df["apparent"]  = T + 0.33*e - 0.70*wind - 4.0
df = df.drop(columns=["rain", "nonelec_cool", "solar"])  # 무의미·중복 제거

시간과 건물의 결

전력에는 하루의 리듬이 있습니다(22–05 저점, 10–16 고점). 데이터를 평평하게 다루면 이 구조를 놓칩니다. 시간대와 건물을 축으로 분류해, 각자의 고유 패턴을 모델이 볼 수 있게 했습니다.

*

교훈은 분명했습니다 — 원시 데이터를 그대로 쓰는 게 아니라, 도메인 지식으로 가공한다. 결과는 AI 부문 리더보드 10위였지만, 남은 건 순위보다 이 확신이었습니다.
