가변 폰트로 UI에 숨을 불어넣기

2025년 2월• CSS · Variable Fonts

가변 폰트(variable font)는 얇은 것과 굵은 것 사이의 중간값까지 한 파일에 담고 있고, 그 값을 CSS에서 연속적으로 움직일 수 있습니다. 그걸 알고 나서 축을 직접 만져볼 도구가 있으면 좋겠다 싶어 만든 것이 활자 실험실입니다. 만들면서 걸렸던 것들을 적어 둡니다.

먼저 짚고 가기
가변 폰트Variable Font
얇음-굵음처럼 여러 스타일을 따로 저장하는 대신, 그 사이를 다이얼처럼 무단계로 조절할 수 있게 만든 하나의 폰트 파일입니다. 이 글의 모든 실험은 이 "무단계 조절"이 가능하다는 데서 출발합니다.
축Axis
가변 폰트에서 조절할 수 있는 하나의 다이얼입니다. 굵기(wght), 기울기(slant) 같은 것이 각각 하나의 축이며, 폰트는 보통 여러 축을 동시에 가집니다. 이 글에서 다루는 "다섯 축"이 이 다이얼 다섯 개입니다.
font-variation-settingsCSS 속성
CSS에서 이 축들의 값을 직접 지정하는 속성입니다. 편리하지만, 뒤에서 이야기할 함정도 여기서 나옵니다.
이산 축Discrete Axis
중간값이 무의미하고 몇 개의 정해진 값(예: 0 또는 1)만 뜻을 갖는 축입니다. 연속처럼 보여도 실제로는 켜짐/꺼짐 스위치에 가까워, 다루는 방식이 달라집니다.
직렬화Serialize
여러 값을 하나의 문자열처럼 정해진 형식으로 이어 붙이는 일입니다. 이 글에서는 다섯 축의 값을 매번 통째로 하나의 문자열로 만들어 CSS에 넘기는 전략을 가리킵니다.

한 줄이 전부를 리셋한다

가변 폰트의 축은 font-variation-settings 한 줄로 제어합니다. 다만 이 속성은 명시하지 않은 축을 전부 기본값으로 되돌립니다. 당시엔 그걸 모르고, 무게만 바꾸려고 이렇게 썼습니다.

el.style.fontVariationSettings = '"wght" 700';
// 방금 -6도로 기울여 둔 slant가... 0으로 돌아간다

그러면 직전에 조절해 둔 기울기가 아무 표시 없이 사라집니다. CSS 커스텀 속성으로 축을 쪼개 관리하는 방법도 있었지만, 저는 매 렌더마다 다섯 축을 통째로 직렬화하는 쪽으로 갔습니다. 상태는 JS가 들고 있고 CSS에는 완성된 문자열만 넘기는 식이라, 어느 축이 초기화됐는지 다시 찾아다닐 일은 없었습니다.

모든 축이 연속인 건 아니다

Recursive 폰트에는 필기체로 바꾸는 CRSV 축이 있습니다. 슬라이더를 연결해 두고 한참을 "왜 중간에서 안 변하지?" 하며 헤맸는데, 스펙을 읽어 보니 이 축은 0과 1만 의미가 있는 이산(discrete) 축이었습니다. 그래서 슬라이더는 연속으로 두고, 값을 적용할 때만 0.5를 기준으로 스냅했습니다.

디자인 축에도 자료형이 있어서, 연속인 줄 알았던 값이 사실은 불리언일 때가 있다.

슬라이더 열 개보다 손끝 하나

축마다 슬라이더를 하나씩 다는 건 정직한 방법이지만, 쓰다 보니 손이 많이 갔습니다. 그래서 제일 자주 함께 만지는 두 축인 무게와 기울기는 2D 패드로 묶었습니다. 가로로 끌면 굵어지고 세로로 끌면 기울어지는데, 구현은 좌표를 각 축의 범위로 선형 매핑하는 게 전부입니다. 그런데도 손가락 하나로 두 값을 동시에 잡는 느낌은 슬라이더 두 개를 번갈아 만질 때와 꽤 달랐고, 원하는 지점을 찾는 것도 이쪽이 빨랐습니다.

프리셋은 점프가 아니라 이동

프리셋 버튼은 값이 툭 바뀌지 않게 만들어 두었습니다. 누르면 현재 상태에서 목표 상태로 다섯 축이 동시에 미끄러지듯 움직입니다. 개인적으로 가장 마음에 드는 부분입니다.

const ease = (t) => 1 - Math.pow(1 - t, 3);   // ease-out cubic
keys.forEach((k) => state[k] = from[k] + (target[k] - from[k]) * ease(p));

560ms 동안 '그로테스크'에서 '손글씨'로 글자가 흘러가는데, 정적인 폰트만 쓰던 때에는 만들 수 없던 화면입니다. 애니메이션의 대상이 위치나 색이 아니라 글자의 형태 자체라는 점이 지금도 신기합니다.

성능과 남은 빚

입력할 때마다 폰트를 리레이아웃하지만, 대상이 단일 요소라 60fps는 유지됐습니다. 문단이 길어지면 content-visibility가 필요해질 것 같습니다. 미리보기를 contenteditable로 만든 탓에 붙여넣기 서식이 딸려 오는 문제도 그대로 남아 있는데, plaintext-only의 크로스브라우징이 아직 애매해서 알고 있는 채로 남겨 두었습니다.

돌아보면 이 도구를 만든 이유는 폰트를 읽기 위해서였던 것 같습니다. 축을 직접 만져 보고 나서야 "이 폰트는 왜 이 축을 넣었을까"가 조금 이해됐습니다. 다음에 다른 가변 폰트를 볼 때도 같은 방식으로 열어 볼 생각입니다.
