PostgreSQL MVCC — UPDATE가 사실은 INSERT인 이유

2022년 9월• PostgreSQL · MVCC · VACUUM

지난 글에서 인덱스와 keyset 페이지네이션으로 느린 목록을 손봤는데, 인덱스가 멀쩡한데도 시간이 지나면 도로 느려지는 테이블이 남았습니다. 한동안 쿼리만 들여다보다가 원인을 못 찾았고, 저장 구조를 보고 나서야 이유를 알았습니다. PostgreSQL에서 UPDATE는 제자리 수정이 아니라 새 버전의 INSERT라는 것, 그 한 문장에서 시작해 튜플 버전과 VACUUM까지 따라가며 알게 된 것들을 적어 둡니다.

먼저 짚고 가기
MVCCMulti-Version Concurrency Control
한 행을 고칠 때 기존 레코드를 제자리에서 덮어쓰는 대신 새 버전을 따로 만들고, 각 버전에 그것을 만든 트랜잭션 ID(xmin)와 그것을 지운 트랜잭션 ID(xmax)를 적어 두는 동시성 제어 기법입니다. 읽는 쪽은 자기 스냅샷과 이 ID들을 대조해 자신에게 보여야 할 버전 하나를 고르므로, 읽기가 쓰기를 기다리지 않고 쓰기도 읽기를 기다리지 않습니다. 잠금이 사라지는 것은 아니어서 같은 행을 동시에 고치려는 쓰기끼리는 여전히 행 잠금으로 줄을 섭니다. 같은 MVCC라도 옛 버전을 undo 영역에 따로 보관하는 구현이 있고 PostgreSQL처럼 테이블 파일 안에 그대로 두는 구현이 있는데, 이 글의 팽창 이야기는 후자이기 때문에 생깁니다.
튜플tuple
힙 페이지 안에 실제로 저장된 행의 한 버전, 즉 물리적 레코드 하나를 가리킵니다. 헤더에 t_xmin·t_xmax·t_ctid 같은 시스템 필드를 갖고, 페이지 안에서는 라인 포인터(lp) 배열을 통해 참조되며 위치는 ctid, 즉 (페이지 번호, 슬롯 번호) 쌍으로 표현되는데, 이 ctid는 UPDATE로 새 버전이 생기거나 VACUUM FULL로 테이블이 재작성되면 바뀌므로 행의 영구 식별자로는 쓸 수 없습니다. 논리적인 행 하나와 튜플 하나가 1:1로 대응하지는 않습니다 — UPDATE될 때마다 새 튜플이 생기므로 같은 행이 여러 튜플로 페이지에 남아 있을 수 있습니다. 관계 모델에서 tuple은 곧 행을 뜻하지만, PostgreSQL 문서에서 이 말은 대개 행 버전(row version)을 가리킵니다.
dead tuple죽은 튜플
xmax에 커밋을 마친 트랜잭션 ID가 기록되어(또는 xmin 쪽 트랜잭션이 중단되어) 지금 살아 있는 어떤 스냅샷에서도 더는 보이지 않게 된 튜플 버전입니다. 논리적으로만 사라졌을 뿐 VACUUM이 회수하기 전까지 페이지 공간을 계속 점유하고, 스캔은 이 튜플까지 읽은 뒤 가시성 판정에서 버립니다. xmax에 값이 적혀 있다고 전부 dead인 것은 아닙니다 — 그 트랜잭션이 아직 커밋 전이거나 롤백됐거나, 옛 버전을 여전히 볼 수 있는 오래된 스냅샷이 열려 있으면 회수 대상이 아닙니다(3절 lp 2가 그런 경우입니다).
스냅샷snapshot
어떤 튜플 버전이 나에게 보이는지를 판정하는 기준을 담은 자료구조입니다. 스냅샷을 뜨는 시점에 아직 실행 중이던(커밋도 롤백도 되지 않은) 트랜잭션 ID 목록과 그 시점의 ID 경계값을 들고 있어서, 튜플의 xmin·xmax를 이 기준과 대조해 이미 커밋된 변경인지를 가립니다. READ COMMITTED에서는 문장마다 새로 뜨고, REPEATABLE READ 이상에서는 트랜잭션의 첫 쿼리 때 한 번 떠서 끝까지 그대로 씁니다. 데이터를 복사해 두는 것이 아니라 판정 기준일 뿐이라, 스냅샷을 오래 붙잡고 있으면 그 시점에 보이던 옛 버전을 VACUUM이 지우지 못합니다 — 6절 쿼리의 backend_xmin이 각 세션이 붙잡고 있는 그 하한값입니다.
VACUUM배큠
테이블을 훑어 어떤 스냅샷에도 보이지 않게 된 튜플을 모으고, 그것을 가리키던 인덱스 항목을 먼저 지운 뒤 힙의 라인 포인터를 회수해 그 공간을 재사용 가능으로 등록하는 유지보수 명령입니다. 임계치를 넘긴 테이블에 대해 autovacuum 데몬이 같은 일을 자동으로 돌립니다. 일반 VACUUM은 SHARE UPDATE EXCLUSIVE 잠금만 잡아 읽기·쓰기와 나란히 돌 수 있고, 테이블을 통째로 재작성하는 VACUUM FULL과는 다른 명령입니다. 회수한 자리는 그 테이블의 다음 INSERT·UPDATE가 쓸 뿐 파일 크기 자체는 그대로여서(파일 끝의 빈 페이지를 잘라내는 경우만 예외), VACUUM은 이미 부푼 테이블을 되돌리는 도구가 아니라 더 부풀지 않게 막는 도구입니다.

1. UPDATE는 제자리에서 고치지 않는다

PostgreSQL은 MVCC(다중 버전 동시성 제어)로 읽기와 쓰기가 서로를 막지 않게 합니다. 행을 제자리에서 고치는 대신 버전을 쌓아 두는 방식입니다. 모든 튜플(행 버전)에는 숨은 컬럼이 있어서, xmin에는 이 버전을 만든 트랜잭션 ID가, xmax에는 이 버전을 지운(또는 지우는 중인) 트랜잭션 ID가 들어갑니다.

그래서 UPDATE 한 번은 새 튜플을 추가하고 옛 튜플의 xmax에 자기 트랜잭션 ID를 적는, 두 가지 일을 하게 됩니다. DELETE는 xmax만 적습니다. 어떤 트랜잭션이 어떤 버전을 보느냐는 각자의 스냅샷이 결정하기 때문에 긴 SELECT가 도는 동안에도 UPDATE는 기다리지 않는데, 대신 옛 버전이 파일 안에 그대로 남습니다.

2. xmin·xmax를 눈으로 확인한다

숨은 컬럼은 이름을 직접 적어 주면 조회할 수 있습니다. 같이 보는 ctid는 튜플의 물리 위치, 즉 (페이지 번호, 페이지 안 순번)입니다.

CREATE TABLE account (id int PRIMARY KEY, balance int);
INSERT INTO account VALUES (1, 100);

SELECT ctid, xmin, xmax, * FROM account;
--  ctid  | xmin | xmax | id | balance
-- (0,1)  |  801 |    0 |  1 |     100

UPDATE account SET balance = 90 WHERE id = 1;

SELECT ctid, xmin, xmax, * FROM account;
--  ctid  | xmin | xmax | id | balance
-- (0,2)  |  802 |    0 |  1 |      90   ← 위치가 (0,1)→(0,2). 새 튜플이다

UPDATE 후 ctid가 바뀐 것을 보고, 같은 페이지의 다음 슬롯에 새 버전이 추가됐다는 것을 확인했습니다. xmax가 0이 아닌 순간도 보고 싶어서 세션을 두 개 열어 봤습니다.

-- 세션 A
BEGIN;
DELETE FROM account WHERE id = 1;   -- 아직 COMMIT하지 않는다

-- 세션 B: 행이 여전히 보이고, xmax에 A의 트랜잭션 ID가 적혀 있다
SELECT ctid, xmin, xmax FROM account;
--  ctid  | xmin | xmax
-- (0,2)  |  802 |  803

xmax는 "이 버전을 지우려는 트랜잭션"의 표시일 뿐이라, 실제로 행을 지우지는 않습니다. A가 ROLLBACK하면 그 표시는 무효가 되고 행은 그대로 남습니다. PostgreSQL의 롤백이 빨리 끝나는 것도 같은 구조 때문인데, 되돌려 놓을 것이 애초에 없기 때문입니다.

3. pageinspect로 페이지 속을 본다

일반 SELECT는 스냅샷에 보이는 버전만 돌려주기 때문에, 페이지에 실제로 무엇이 들어 있는지는 pageinspect 확장으로 확인했습니다. (위 실험의 세션 A가 ROLLBACK을 마친 상태입니다.)

CREATE EXTENSION IF NOT EXISTS pageinspect;

SELECT lp, t_xmin, t_xmax, t_ctid
FROM heap_page_items(get_raw_page('account', 0));
-- lp | t_xmin | t_xmax | t_ctid
--  1 |    801 |    802 | (0,2)   ← 옛 버전. 다음 버전 (0,2)를 가리킨다
--  2 |    802 |    803 | (0,2)   ← 현재 버전. ROLLBACK한 DELETE의 803이 남아 있다

SELECT에는 안 보이던 옛 버전(lp 1)이 페이지에 그대로 남아 있습니다. t_xmax에 UPDATE 트랜잭션의 ID가 적혀 있고, t_ctid는 버전 체인의 다음 칸을 가리킵니다. 현재 버전(lp 2)의 t_xmax에도 ROLLBACK으로 무효가 된 803이 지워지지 않고 남아 있습니다. PostgreSQL은 803이 중단된 트랜잭션이라는 사실만 따로 기록해 두고 이 표시를 무시할 뿐, 페이지를 되돌리러 오지는 않습니다. 롤백이 빨리 끝나는 이유를 페이지 위에서 눈으로 확인한 셈입니다. 반대로 커밋이 끝나 어떤 스냅샷에도 더는 보이지 않게 된 lp 1 같은 버전이 dead tuple이고, 이런 튜플은 자리를 차지한 채로 남아 스캔이 그 위를 지나가게 됩니다.

*

pageinspect는 슈퍼유저 권한이 필요한 저수준 도구라 운영 DB에서 자주 쓸 물건은 아닙니다. 저는 로컬에서 UPDATE·DELETE·VACUUM을 한 줄씩 실행하며 페이지가 어떻게 변하는지 따라가 보는 데 썼는데, 구조를 눈에 익히는 데는 이 방법이 가장 도움이 됐습니다.

4. dead tuple이 테이블을 느리게 만드는 방식

UPDATE가 잦은 테이블은 살아있는 행 수가 그대로여도 파일은 계속 자랍니다. 스캔 비용은 행 수가 아니라 페이지 수를 따라가기 때문에, 이런 테이블은 시간이 지날수록 느려집니다. 제가 만들던 크롤링 파이프라인의 큐 테이블이 그랬습니다. 대기 중인 몇백 건을 집어 오는 LIMIT 쿼리가, 완료 처리로 UPDATE되며 쌓인 수많은 죽은 버전을 헤치고 나서야 살아있는 행에 닿았습니다. 살아있는 행이 적을수록 오히려 더 느려지는 구조라는 것을 그때 알았습니다.

인덱스도 같은 영향을 받습니다. 인덱스 항목은 ctid를 가리키므로 새 버전이 생길 때마다 항목이 추가되고, 옛 버전을 가리키는 항목도 남습니다(예외는 뒤에 나올 HOT). 지난 글에서 인덱스를 깔아 살려 둔 목록도 그대로 두면 같은 경로로 다시 느려집니다. 그래서 확인은 통계 뷰부터 했습니다.

SELECT relname, n_live_tup, n_dead_tup,
       round(n_dead_tup * 100.0 / nullif(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,
       last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 10;

n_dead_tup은 통계 수집기의 추정치라, 정확한 수치가 필요하면 pgstattuple 확장의 dead_tuple_percent로 확인합니다. last_autovacuum이 한참 전이라면 그것만으로도 짚어 볼 만한 단서가 됩니다.

5. VACUUM이 하는 일, 하지 않는 일

VACUUM은 어떤 스냅샷에도 더 이상 보이지 않는 버전을 찾아 그 자리를 재사용 가능으로 표시하고, 인덱스에서 해당 항목을 걷어냅니다. 다만 파일을 줄여 OS에 돌려주지는 않습니다(파일 끝의 빈 페이지를 잘라내는 정도가 전부입니다). 이미 부푼 테이블에 VACUUM을 돌려도 크기가 그대로인 것은 그 때문입니다. 크기를 되돌리려면 VACUUM FULL을 써야 하는데, 테이블을 통째로 재작성하는 동안 ACCESS EXCLUSIVE 잠금을 잡으므로 운영 중에는 쓰기 어렵습니다. 무중단이 필요하면 pg_repack 같은 도구를 검토합니다.

튜플 버전 체인 시뮬레이터직접 조작해 보세요
페이지 사용률
13%
live 1 dead 0 재사용 가능 0 다음 xid 802

UPDATE는 옛 버전을 dead로 남기고 새 버전을 추가합니다 — VACUUM은 그 자리를 재사용 가능으로 되돌릴 뿐, 파일을 줄이지는 않습니다

UPDATE 비용 자체를 줄이는 장치로는 HOT(Heap-Only Tuple)이 있습니다. 인덱스에 걸린 컬럼이 바뀌지 않았고 같은 페이지에 빈자리가 있으면, 인덱스 항목을 새로 만들지 않고 페이지 안에서 버전 체인만 잇습니다. 그래서 fillfactor를 낮춰 페이지마다 여유를 남겨 두면 HOT으로 처리될 확률이 올라갑니다.

-- 페이지를 80%만 채워 HOT 업데이트 여지를 남긴다 (기본 100)
ALTER TABLE jobs SET (fillfactor = 80);

-- 전체 UPDATE 중 HOT 비율 확인
SELECT n_tup_upd, n_tup_hot_upd
FROM pg_stat_user_tables WHERE relname = 'jobs';

6. autovacuum 튜닝 포인트

autovacuum은 n_dead_tup이 대략 threshold(기본 50) + scale_factor(기본 0.2) × 행 수를 넘으면 그 테이블을 청소합니다. 계산해 보면 천만 행 테이블에서는 dead tuple이 2백만 개를 넘어야 청소가 시작된다는 뜻이라, 기본값만 두고 쓰면 한참 늦게 돕니다. 큰 테이블이나 UPDATE가 잦은 테이블은 테이블 단위로 값을 조여 두었습니다.

-- 이 테이블만 더 일찍, 더 자주 청소하게
ALTER TABLE jobs SET (
  autovacuum_vacuum_scale_factor = 0.02,   -- 기본 0.2 → dead 2%면 시작
  autovacuum_vacuum_threshold    = 1000
);

autovacuum이 돌긴 하는데 쌓이는 속도를 따라가지 못한다면 청소 속도 제한이 원인일 수 있습니다. 이때는 autovacuum_vacuum_cost_limit을 올리거나 autovacuum_vacuum_cost_delay(기본 2ms)를 낮춰 autovacuum에 더 많은 I/O 예산을 줍니다.

VACUUM이 돌아도 지우지 못하는 경우도 있습니다. 오래된 트랜잭션이 옛 스냅샷을 붙잡고 있으면 그 시점 이후의 dead tuple은 회수 대상이 되지 않습니다. 흔한 원인은 idle in transaction 상태로 방치된 세션과 잊고 지나간 replication slot이었습니다.

-- 스냅샷을 오래 붙잡고 있는 트랜잭션 찾기
SELECT pid, state, now() - xact_start AS xact_age, left(query, 40) AS query
FROM pg_stat_activity
WHERE backend_xmin IS NOT NULL
ORDER BY xact_age DESC;
*

autovacuum이 부담스럽다고 꺼 두면 상황은 더 나빠집니다. 문제가 된 쪽은 대개 "너무 자주"가 아니라 "너무 늦게" 도는 경우였고, 더 작게 더 자주 돌도록 맞추는 편이 결과가 좋았습니다. 정산 배치처럼 한 번에 대량 UPDATE를 치는 작업 뒤에는 VACUUM ANALYZE를 수동으로 한 번 돌려 두면 다음 트래픽이 한결 수월했습니다.

지난 글이 쿼리를 어떻게 쓰는지에 대한 기록이었다면, 이번 글은 테이블이 왜 부푸는지를 따라간 기록입니다. UPDATE가 사실은 INSERT라는 것을 알고 나서는 느려지는 테이블 앞에서 보는 곳이 달라졌습니다. 요즘은 실행 계획을 열기 전에 n_dead_tup과 last_autovacuum을 먼저 봅니다.
