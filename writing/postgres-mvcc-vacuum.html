PostgreSQL MVCC — UPDATE가 사실은 INSERT인 이유

2022년 9월• 약 10분• PostgreSQL · MVCC · VACUUM

지난 글에서는 인덱스와 keyset 페이지네이션으로 느린 목록을 살렸습니다. 그런데 인덱스가 멀쩡한데도 시간이 지나면 도로 느려지는 테이블이 있습니다. 이번에는 쿼리가 아니라 저장 구조를 봅니다. 출발점은 한 문장입니다 — PostgreSQL에서 UPDATE는 제자리 수정이 아니라 새 버전의 INSERT입니다.

먼저 짚고 가기
MVCCMulti-Version Concurrency Control
데이터를 고칠 때 원본을 덮어쓰지 않고 새 버전을 하나 더 만들어 쌓는 방식입니다. 문서를 수정할 때마다 새 사본을 저장해 두는 것과 비슷합니다. 덕분에 누가 읽는 중이어도 다른 사람이 동시에 고칠 수 있습니다. 이 글의 모든 이야기가 여기서 출발합니다.
튜플tuple
테이블에 저장된 행 하나하나의 버전을 가리키는 말입니다. 같은 행이라도 수정될 때마다 튜플이 새로 생기므로, 한 행에 여러 튜플이 존재할 수 있습니다.
dead tuple죽은 튜플
더 이상 아무도 볼 필요가 없어진 옛날 버전 행입니다. 지워졌지만 파일에는 그대로 남아 자리만 차지합니다. 이것이 쌓이면 테이블이 부풀고 느려집니다.
스냅샷snapshot
한 트랜잭션이 "지금 이 시점의 데이터"로 보기로 정한 기준 시점입니다. 카메라로 순간을 찍어 두는 것과 같아서, 다른 사람이 나중에 데이터를 바꿔도 내 스냅샷에는 영향이 없습니다.
VACUUM배큠
쌓인 dead tuple을 찾아 그 자리를 재사용할 수 있게 청소하는 관리 작업입니다. PostgreSQL이 알아서 돌리는(autovacuum) 버전도 있습니다. 이 청소가 제때 안 되면 테이블이 계속 부풉니다.

1. UPDATE는 제자리에서 고치지 않는다

PostgreSQL은 MVCC(다중 버전 동시성 제어)로 읽기와 쓰기가 서로를 막지 않게 합니다. 방법은 단순합니다. 행을 고치는 대신 버전을 쌓습니다. 모든 튜플(행 버전)에는 숨은 컬럼이 있습니다 — xmin은 이 버전을 만든 트랜잭션 ID, xmax는 이 버전을 지운(또는 지우는 중인) 트랜잭션 ID입니다.

그래서 UPDATE 한 번은 실제로 두 가지 일을 합니다. 새 튜플을 추가하고, 옛 튜플의 xmax에 자기 트랜잭션 ID를 적습니다. DELETE는 xmax만 적고 끝입니다. 어떤 트랜잭션이 어떤 버전을 보느냐는 각자의 스냅샷이 결정하므로, 긴 SELECT가 도는 동안에도 UPDATE는 기다리지 않습니다. 대가도 분명합니다 — 옛 버전이 파일 안에 그대로 남습니다.

2. xmin·xmax를 눈으로 확인한다

숨은 컬럼은 이름을 직접 부르면 조회됩니다. ctid는 튜플의 물리 위치, 즉 (페이지 번호, 페이지 안 순번)입니다.

CREATE TABLE account (id int PRIMARY KEY, balance int);
INSERT INTO account VALUES (1, 100);

SELECT ctid, xmin, xmax, * FROM account;
--  ctid  | xmin | xmax | id | balance
-- (0,1)  |  801 |    0 |  1 |     100

UPDATE account SET balance = 90 WHERE id = 1;

SELECT ctid, xmin, xmax, * FROM account;
--  ctid  | xmin | xmax | id | balance
-- (0,2)  |  802 |    0 |  1 |      90   ← 위치가 (0,1)→(0,2). 새 튜플이다

UPDATE 후 ctid가 바뀌었습니다. 같은 페이지의 다음 슬롯에 새 버전이 추가된 겁니다. xmax가 0이 아닌 순간도 잡을 수 있습니다. 세션을 두 개 열면 됩니다.

-- 세션 A
BEGIN;
DELETE FROM account WHERE id = 1;   -- 아직 COMMIT하지 않는다

-- 세션 B: 행이 여전히 보이고, xmax에 A의 트랜잭션 ID가 적혀 있다
SELECT ctid, xmin, xmax FROM account;
--  ctid  | xmin | xmax
-- (0,2)  |  802 |  803

xmax는 "이 버전을 지우려는 트랜잭션"의 표시일 뿐, 실제로 지우는 게 아닙니다. A가 ROLLBACK하면 그 표시는 무효가 되고 행은 그대로 삽니다. PostgreSQL의 롤백이 순식간에 끝나는 것도 같은 구조의 결과입니다 — 되돌릴 게 없으니까요.

3. pageinspect로 페이지 속을 본다

일반 SELECT는 스냅샷에 보이는 버전만 돌려줍니다. 페이지에 실제로 뭐가 있는지는 pageinspect 확장으로 봅니다. (위 실험의 세션 A가 ROLLBACK을 마친 상태입니다.)

CREATE EXTENSION IF NOT EXISTS pageinspect;

SELECT lp, t_xmin, t_xmax, t_ctid
FROM heap_page_items(get_raw_page('account', 0));
-- lp | t_xmin | t_xmax | t_ctid
--  1 |    801 |    802 | (0,2)   ← 옛 버전. 다음 버전 (0,2)를 가리킨다
--  2 |    802 |    803 | (0,2)   ← 현재 버전. ROLLBACK한 DELETE의 803이 남아 있다

SELECT에는 안 보이던 옛 버전(lp 1)이 페이지에 그대로 있습니다. t_xmax에 UPDATE 트랜잭션의 ID가 적혀 있고, t_ctid는 버전 체인의 다음 칸을 가리킵니다. 눈여겨볼 곳이 하나 더 있습니다 — 현재 버전(lp 2)의 t_xmax에는 ROLLBACK으로 무효가 된 803이 지워지지 않고 남아 있습니다. PostgreSQL은 803이 중단된 트랜잭션이라는 사실만 따로 기록해 두고 이 표시를 무시할 뿐, 페이지를 되돌리러 오지 않습니다. 롤백이 순식간에 끝나는 이유가 페이지 위에 그대로 보이는 셈입니다. 반면 커밋이 끝나 어떤 스냅샷에도 더는 보이지 않게 된 lp 1 같은 버전이 dead tuple입니다. 자리를 차지하고, 스캔은 그 위를 지나갑니다.

*

pageinspect는 슈퍼유저 권한이 필요한 저수준 도구라 운영 DB에서 남발할 물건은 아닙니다. 로컬에서 UPDATE·DELETE·VACUUM을 한 줄씩 실행하며 페이지가 어떻게 변하는지 따라가 보는 용도로는 최고의 교재입니다.

4. dead tuple이 테이블을 느리게 만드는 방식

UPDATE가 잦은 테이블은 살아있는 행 수는 그대로인데 파일은 계속 자랍니다. 그리고 스캔 비용은 행 수가 아니라 페이지 수를 따라갑니다. 전형적인 피해자가 작업 큐입니다. 제가 만들던 크롤링 파이프라인의 큐 테이블이 그랬습니다 — 대기 중인 몇백 건을 집는 LIMIT 쿼리가, 완료 처리로 UPDATE된 수많은 죽은 버전을 헤치고 나서야 살아있는 행에 닿았습니다. 살아있는 행이 적을수록 역설적으로 더 느려지는 구조입니다.

인덱스도 무사하지 않습니다. 인덱스 항목은 ctid를 가리키므로, 새 버전이 생길 때마다 항목이 추가되고 옛 버전을 가리키는 항목이 남습니다(예외는 뒤에 나올 HOT). 지난 글에서 인덱스를 깔아 살린 목록도, 방치하면 이 경로로 다시 무너집니다. 관찰은 통계 뷰부터 시작합니다.

SELECT relname, n_live_tup, n_dead_tup,
       round(n_dead_tup * 100.0 / nullif(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,
       last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 10;

n_dead_tup은 통계 수집기의 추정치입니다. 정확한 수치가 필요하면 pgstattuple 확장의 dead_tuple_percent로 확인합니다. last_autovacuum이 오래전이라면 그 자체가 단서입니다.

5. VACUUM이 하는 일, 하지 않는 일

VACUUM은 어떤 스냅샷에도 더 이상 보이지 않는 버전을 찾아 그 자리를 재사용 가능으로 표시하고, 인덱스에서 해당 항목을 걷어냅니다. 중요한 건 하지 않는 일입니다 — 파일을 줄여 OS에 돌려주지 않습니다(파일 끝의 빈 페이지를 잘라내는 정도가 전부입니다). 이미 부풀 대로 부푼 테이블에 VACUUM을 돌려도 크기가 그대로인 이유입니다. 크기를 되돌리려면 VACUUM FULL인데, 테이블을 통째로 재작성하는 동안 ACCESS EXCLUSIVE 잠금을 잡으므로 운영 중에는 사실상 못 씁니다. 무중단이 필요하면 pg_repack 같은 도구를 검토합니다.

UPDATE 비용 자체를 줄이는 장치로는 HOT(Heap-Only Tuple)이 있습니다. 인덱스에 걸린 컬럼이 바뀌지 않았고 같은 페이지에 빈자리가 있으면, 인덱스 항목을 새로 만들지 않고 페이지 안에서 버전 체인만 잇습니다. fillfactor를 낮춰 페이지마다 여유를 남겨두면 HOT 확률이 올라갑니다.

-- 페이지를 80%만 채워 HOT 업데이트 여지를 남긴다 (기본 100)
ALTER TABLE jobs SET (fillfactor = 80);

-- 전체 UPDATE 중 HOT 비율 확인
SELECT n_tup_upd, n_tup_hot_upd
FROM pg_stat_user_tables WHERE relname = 'jobs';

6. autovacuum 튜닝 포인트

autovacuum은 n_dead_tup이 대략 threshold(기본 50) + scale_factor(기본 0.2) × 행 수를 넘으면 그 테이블을 청소합니다. 기본값의 함정이 여기 있습니다 — 천만 행 테이블이면 dead tuple이 2백만 개를 넘어야 시작한다는 뜻입니다. 큰 테이블, UPDATE가 잦은 테이블은 테이블 단위로 조입니다.

-- 이 테이블만 더 일찍, 더 자주 청소하게
ALTER TABLE jobs SET (
  autovacuum_vacuum_scale_factor = 0.02,   -- 기본 0.2 → dead 2%면 시작
  autovacuum_vacuum_threshold    = 1000
);

돌긴 도는데 쌓이는 속도를 못 따라간다면 청소 속도 제한이 원인일 수 있습니다. autovacuum_vacuum_cost_limit을 올리거나 autovacuum_vacuum_cost_delay(기본 2ms)를 낮춰 autovacuum에 더 많은 I/O 예산을 줍니다.

마지막으로, VACUUM이 아무리 돌아도 못 지우는 경우가 있습니다. 오래된 트랜잭션이 옛 스냅샷을 붙잡고 있으면 그 시점 이후의 dead tuple은 회수 대상이 아닙니다. 단골 범인은 idle in transaction으로 방치된 세션과 잊힌 replication slot입니다.

-- 스냅샷을 오래 붙잡고 있는 트랜잭션 찾기
SELECT pid, state, now() - xact_start AS xact_age, left(query, 40) AS query
FROM pg_stat_activity
WHERE backend_xmin IS NOT NULL
ORDER BY xact_age DESC;
*

autovacuum이 부담스럽다고 끄는 건 최악의 수입니다. 문제는 거의 항상 "너무 자주"가 아니라 "너무 늦게"입니다 — 더 작게, 더 자주 돌게 만드세요. 그리고 정산 배치처럼 한 번에 대량 UPDATE를 치는 작업 뒤에는 VACUUM ANALYZE를 수동으로 한 번 돌려두는 편이 다음 트래픽을 편하게 합니다.

지난 글이 "쿼리를 어떻게 쓰나"였다면 이번 글은 "테이블이 왜 부푸나"였습니다. UPDATE가 INSERT라는 사실 하나를 받아들이면, 느려지는 테이블 앞에서 볼 곳이 달라집니다 — 실행 계획보다 먼저 n_dead_tup과 last_autovacuum을 여는 습관, 그게 이 글의 결론입니다.
