서평 · 아키텍처

서평: 엔터프라이즈 애플리케이션 아키텍처 패턴 — 이름이 생기면 보인다

2023년 3월• PoEAA · Fowler · 패턴

제 책장에서 가장 손때가 많이 묻은 책은 Martin Fowler의 《Patterns of Enterprise Application Architecture》(2002)입니다. 2002년에 나온 두꺼운 카탈로그를 아직도 꺼내 보는 이유는, 이 책이 새로운 기술이 아니라 이름을 다루기 때문입니다. 제가 이미 짜고 있던 코드에 이름이 붙고 나서야 그 코드를 두고 이야기할 수 있게 되었는데, 그 전까지는 무엇이 불편한지조차 설명하지 못했습니다. 이 책이 제 설계 언어를 어떻게 바꿨는지 정리해 둡니다.

먼저 짚고 가기
트랜잭션 스크립트Transaction Script
도메인 로직을 조직하는 세 방식 중 하나로, 프레젠테이션에서 들어온 요청 하나를 프로시저 하나가 조회·계산·저장까지 위에서 아래로 절차적으로 처리하는 패턴입니다. 데이터와 행동이 분리되어 있어 로직이 단순할 때는 가장 직관적이지만, 규칙이 복잡해지면 같은 계산이 여러 프로시저에 중복되고 조건 분기가 쌓입니다. 이 글은 제가 맡았던 정산 코드가 이 방식에서 시작해 왜 감당하기 어려워졌는지를 출발점으로 삼습니다.
도메인 모델Domain Model
도메인 로직을 조직하는 또 다른 방식으로, 도메인의 개념 하나하나를 데이터와 행동(계산 규칙)을 함께 가진 객체로 만들고 그 객체들이 서로 참조하는 그물망으로 업무를 표현하는 패턴입니다. 한 객체는 주문 하나, 상품 하나처럼 의미 있는 개체 하나에 대응하고, 규칙은 그 개체를 아는 객체 안에 놓입니다. 객체-관계 매핑 같은 초기 비용이 크기 때문에 도메인 복잡도가 낮으면 트랜잭션 스크립트보다 오히려 비싸고, 복잡도가 교차점을 넘어설 때 유리해집니다.
Service LayerService Layer
응용의 경계를 서비스 한 층으로 정의하는 패턴으로, 시스템이 제공하는 오퍼레이션의 목록을 정하고 각 오퍼레이션에서 트랜잭션 경계·외부 호출·순서 조율을 담당합니다. Fowler는 도메인 모델 위에 얇게 얹는 도메인 파사드형과 응용 로직까지 두툼하게 품는 오퍼레이션 스크립트형을 구분합니다. 업무 규칙 자체를 담는 자리가 아니라 흐름을 조율하는 자리라, 규칙까지 이 층으로 끌어오면 도메인 객체는 데이터만 남은 껍데기가 됩니다.
RepositoryRepository
도메인 계층과 데이터 매핑 계층 사이를 중재하며, 컬렉션 같은 인터페이스로 도메인 객체에 접근하게 하는 패턴입니다. 인터페이스에는 SQL이나 테이블 같은 저장 기술의 흔적을 두지 않아, 도메인 코드는 객체가 어디에 어떻게 저장되는지 모른 채 add·find·remove만 다룹니다. 메모리 속 컬렉션을 다루는 듯한 환상을 주지만, 그 환상을 유지하려면 행과 객체를 번역하는 Data Mapper가 뒤에 필요합니다.
Unit of WorkUnit of Work
하나의 업무 트랜잭션이 건드린 객체들(신규·변경·삭제)을 목록으로 추적해 두었다가, 커밋 시점에 그 변경을 한꺼번에 DB에 써 내리고 동시성 문제를 정리하는 패턴입니다. 개별 저장을 즉시 날리는 대신 무엇을 쓸지 모아 두므로, 전부 함께 반영되거나 전부 취소되는 하나의 작업 단위가 됩니다. JPA·Hibernate의 영속성 컨텍스트가 이 패턴과 Identity Map을 한 몸에 넣은 구현이고, 이 글 후반의 "은행 이체를 커밋 뒤로 미루는" 판단이 바로 이 경계에서 나옵니다.

1. 이름이 없으면 논쟁도 없다

이 책은 두 부분으로 나뉩니다. 앞쪽 절반은 계층화·도메인 로직·객체와 관계형 DB의 매핑을 설명하는 서술(narratives)이고, 뒤쪽 절반은 50개가 넘는 패턴을 하나씩 정의한 참조용 카탈로그입니다. 처음 읽을 땐 뒤쪽이 사전처럼 지루해 보였는데, 몇 년 뒤 다시 펴 보니 제가 계속 들춰 보게 되는 쪽은 그 사전이었습니다. 이 책이 제게 남긴 것은 코드가 아니라 어휘였습니다.

제 첫 실무는 오래된 웹 시스템을 유지보수하고 화면을 찍어내는 일이었습니다. 돌아보면 그때도 이름이 있는 구조를 짜고 있었는데, 정작 저는 그 이름을 몰랐습니다. 그래서 코드 리뷰에서 "이건 이대로 두는 게 낫다"거나 "여긴 갈아엎어야 한다"를 느낌으로만 말했습니다. 가리킬 단어가 없으니 논증이 되지 못하고 취향 이야기로 끝나곤 했습니다. 패턴의 값은 정답을 주는 데 있다기보다, 설계 회의에서 손가락으로 가리킬 대상을 만들어 주는 데 있었습니다.

Fowler는 이 패턴들을 GoF와 크리스토퍼 알렉산더의 전통 위에 올려놓습니다. 패턴은 문제·맥락·해법을 한 이름 아래 묶은 공용어라는 관점입니다. 아래에서는 다섯 개 패턴의 이름 — 트랜잭션 스크립트, 도메인 모델, Service Layer, Repository, Unit of Work — 이 제 설계 언어를 어떻게 바꿨는지를 순서대로 적었습니다.

2. 트랜잭션 스크립트와 도메인 모델의 분기

제가 가장 자주 다시 읽는 장은 도메인 로직을 조직하는 방법을 다루는 2장입니다. Fowler는 선택지를 셋으로 정리합니다. 트랜잭션 스크립트(요청 하나를 절차 하나로 처리), 도메인 모델(데이터와 행동을 가진 객체망), 그리고 그 사이의 테이블 모듈(테이블 단위로 로직을 모으는 .NET DataSet 계열)입니다. 제가 정산 시스템에서 처음 받아 든 코드는 전형적인 트랜잭션 스크립트였습니다.

// settlement.service.ts — 트랜잭션 스크립트: 한 메서드가 전부 한다
@Injectable()
export class SettlementService {
  constructor(private prisma: PrismaService, private bank: BankClient) {}

  async confirm(merchantId: string, month: string) {
    // 1) 원장 조회
    const rows = await this.prisma.ledger.findMany({
      where: { merchantId, month, status: 'PENDING' },
    });
    // 2) 금액 계산 — 수수료·세금·보정이 if로 쌓인다
    let payout = 0;
    for (const r of rows) {
      let fee = r.amount * 0.033;                // 실수(float)로 다루는 돈 — 이미 버그다
      if (r.channel === 'CARD') fee = r.amount * 0.025;
      if (r.promo) fee = fee - r.amount * 0.005;
      payout += r.amount - Math.round(fee);
    }
    // 3) 저장과 4) 외부 통보가 한 메서드에 뒤섞인다
    await this.prisma.settlement.create({
      data: { merchantId, month, payout, status: 'CONFIRMED' },
    });
    await this.bank.transfer(merchantId, payout); // 커밋과 무관하게 날아가는 이체
    return payout;
  }
}

Fowler는 트랜잭션 스크립트 자체가 잘못은 아니라고 적어 둡니다. 로직이 단순할 때는 이 방식이 가장 정직합니다. 절차가 위에서 아래로 읽히고, 처음 보는 사람도 따라 읽을 수 있습니다. 이 구조가 버거워지는 건 도메인 로직의 복잡도가 자랄 때입니다. 위 코드의 요금 계산이 지점·채널·감면·정산주기로 곱해지기 시작하자, if 중첩이 제가 머릿속으로 따라갈 수 있는 깊이를 넘어섰습니다.

이 대목에서 Fowler는 그래프 하나를 그립니다. 가로축은 도메인 로직의 복잡도, 세로축은 변경에 드는 노력입니다. 트랜잭션 스크립트는 낮게 시작해 가파르게 오르고, 도메인 모델은 높게 시작해 완만하게 오릅니다. 두 선은 어디선가 교차하고, 그 교차점을 넘어서면 도메인 모델이 싸집니다. 제가 맡은 정산 코드는 이미 그 교차점을 지나 있었습니다.

*

Fowler 본인도 이 그래프가 측정에서 나온 게 아니라 경험에 기댄 인상이라고 밝혀 둡니다. 교차점의 위치는 팀의 객체지향 설계 숙련도에 따라 좌우로 움직이고요. 그래서 이 그림은 "언제 갈아타라"는 답을 주기보다, "지금 우리가 어느 축 위에 있는가"를 한 번 묻게 만드는 쪽에 가깝습니다.

3. Service Layer — 응용 로직이 사는 층

도메인 모델로 넘어가기로 하고 나니 다음 질문이 따라왔습니다. 계산 규칙은 객체 안으로 넣는다 치고, 트랜잭션과 통보와 순서 같은 것은 어디에 두어야 하는가였습니다. 위 예제의 3·4번 — 저장하고 은행에 이체를 날리는 부분 — 은 정산의 업무 규칙이 아니라 한 번의 요청을 처리하는 흐름입니다. 이걸 도메인 객체에 넣으면 객체가 인프라를 알게 되고, 아무 데도 안 넣으면 컨트롤러가 비대해집니다.

Fowler가 이 자리에 놓는 패턴이 Service Layer입니다. 응용의 경계를 서비스 한 층으로 정의해, 시스템이 제공하는 오퍼레이션의 목록과 각 오퍼레이션의 응답 조율(트랜잭션 경계, 외부 호출, 이벤트 발행)을 담당하게 합니다. 책은 두 가지 결을 구분합니다. 도메인 모델 위에 얇게 얹는 도메인 파사드형과, 응용 로직까지 두툼하게 품는 오퍼레이션 스크립트형입니다.

SafeIn 백엔드를 NestJS로 설계할 때 저는 이 구분을 그대로 가져다 썼습니다. Provider 하나가 유스케이스 하나가 되고, 업무 규칙은 도메인 객체에, 흐름은 서비스 계층에 두었습니다. 이렇게 나눠 둔 이유는 규칙과 흐름이 서로 다른 이유로 바뀌기 때문입니다. 수수료율이 바뀌는 것과 "이체는 커밋 뒤에 한다"가 바뀌는 것은 서로 다른 사건이고, 둘을 다른 층에 두면 한쪽을 고치다 다른 쪽을 깨뜨리는 일이 줄어듭니다. Service Layer라는 이름을 알기 전에는 이 둘을 한 메서드에 욱여넣고도 무엇이 잘못됐는지 설명하지 못했습니다.

4. Repository와 Data Mapper — Prisma를 다시 보다

도메인 모델을 세우고 나면 다음 질문은 저장입니다. 객체는 메모리에 살고 데이터는 테이블에 사는데, 이 둘을 잇는 방식을 책은 Active Record와 Data Mapper로 나눕니다. Active Record는 행을 감싼 객체가 DB 접근까지 스스로 하는 방식이고, Data Mapper는 객체와 DB, 그리고 매퍼가 서로를 모르게 갈라놓는 방식입니다. 도메인 모델과 테이블 구조가 어긋나기 시작하면 후자가 유리해집니다.

이 대목에서 쓰고 있던 Prisma를 다시 보게 됐습니다. Prisma가 돌려주는 것은 메서드 없는 평범한 데이터 객체이니, Active Record는 아닙니다. TypeORM이 문서에서 "Active Record냐 Data Mapper냐"를 선택지로 제시하는 것과는 결이 다릅니다. 2002년 책의 어휘가 20년 뒤 ORM 문서의 목차가 되어 있는 셈입니다. Prisma는 데이터 매핑의 절반(행 읽기)만 해 주고, 행을 도메인 객체로 바꾸는 나머지 절반은 제 몫으로 남깁니다. 그래서 도메인과 Money를 먼저 세웠습니다.

// money.ts — Fowler의 Money 패턴: 금액은 '정수 최소단위 + 통화'다
export class Money {
  private constructor(readonly minor: bigint, readonly currency: Currency) {}

  static krw(won: number) { return new Money(BigInt(won), 'KRW'); }
  plus(o: Money) { return new Money(this.minor + o.minor, this.currency); }
  rate(bp: number) { return new Money(this.minor * BigInt(bp) / 10000n, this.currency); }
}

// settlement.ts — 도메인 모델: 계산 규칙이 객체 안에서 산다
export class Settlement {
  private constructor(
    readonly id: SettlementId,
    private status: SettlementStatus,
    private readonly lines: LedgerLine[],
  ) {}

  payout(policy: FeePolicy): Money {
    return this.lines.reduce(
      (sum, line) => sum.plus(line.net(policy)),
      Money.krw(0),
    );
  }

  confirm() {
    if (this.status !== 'PENDING') throw new DomainError('이미 확정된 정산');
    this.status = 'CONFIRMED';
  }
}

2장 예제에 있던 r.amount * 0.033 같은 실수 연산은 여기서 사라졌습니다. 책의 Base Patterns 장에 있는 Money 패턴이 돈을 부동소수점이 아니라 최소 단위 정수 + 통화로 다루라고 말하기 때문인데, 정산 도메인에서는 이게 취향이 아니라 정확성의 문제였습니다. 그리고 payout()이라는 업무 규칙이 서비스가 아니라 Settlement 안으로 들어왔습니다. 이 도메인 객체를 저장소가 컬렉션처럼 담아 줍니다.

// settlement.repository.ts — 인메모리 컬렉션인 '척'하는 경계
export interface SettlementRepository {
  findPending(merchantId: string, month: Month): Promise<Settlement | null>;
  save(settlement: Settlement): Promise<void>;
}

// prisma-settlement.repository.ts — Data Mapper: 행 ↔ 도메인 변환은 여기서만
@Injectable()
export class PrismaSettlementRepository implements SettlementRepository {
  constructor(private prisma: PrismaClient) {}

  async findPending(merchantId: string, month: Month) {
    const row = await this.prisma.settlement.findFirst({
      where: { merchantId, month: month.value, status: 'PENDING' },
      include: { lines: true },
    });
    return row ? SettlementMapper.toDomain(row) : null; // 도메인은 Prisma 타입을 모른다
  }

  async save(s: Settlement) {
    const row = SettlementMapper.toRow(s);
    await this.prisma.settlement.update({ where: { id: row.id }, data: row });
  }
}

Fowler의 정의에서 Repository는 도메인과 데이터 매핑 사이를 중재하며, 마치 인메모리 컬렉션인 것처럼 도메인 객체를 다루게 해 주는 경계입니다. 위 인터페이스에는 SQL도, include도, Prisma라는 단어도 없습니다. 그 모든 저장 기술의 사정은 SettlementMapper가 있는 구현체 한 곳에 모여 있습니다. 컬렉션이라는 환상을 유지하기 위해 행과 객체를 번역하는 자리, 그게 Data Mapper의 역할입니다. 이름을 알고 나니 Prisma가 어디까지 해 주고 어디부터 제 일인지가 정리됐습니다.

5. Unit of Work와 Identity Map — 프레임워크의 몫

저장소를 컬렉션처럼 다루기로 하고 나면, 여러 번의 저장을 하나의 원자적 작업으로 묶는 일은 누가 하는지가 남습니다. 책은 이 역할에 Unit of Work라는 이름을 줍니다. 하나의 업무 트랜잭션이 건드린 객체들을 모아 두었다가, 변경을 한꺼번에 써 내리고 동시성 문제를 정리하는 장치입니다. 정산 확정을 유스케이스로 옮겨 보니 이 경계가 눈에 들어왔습니다.

// confirm-settlement.usecase.ts — $transaction 콜백 = 하나의 작업 단위
@Injectable()
export class ConfirmSettlement {
  constructor(private prisma: PrismaService, private bank: BankClient, private feePolicy: FeePolicy) {}

  async execute(merchantId: string, month: Month) {
    const payout = await this.prisma.$transaction(async (tx) => {
      const repo = new PrismaSettlementRepository(tx); // 저장소를 트랜잭션에 묶는다
      const s = await repo.findPending(merchantId, month);
      if (!s) throw new NotFound('정산 대상 없음');
      s.confirm();                          // 도메인 규칙
      await repo.save(s);                     // 이 콜백 안의 쓰기는 전부 원자적
      return s.payout(this.feePolicy);
    });
    await this.bank.transfer(merchantId, payout); // 통보는 커밋 이후로
    return payout;
  }
}

Prisma의 $transaction 콜백이 그대로 하나의 작업 단위입니다. 콜백 안의 모든 쓰기가 함께 커밋되거나 함께 롤백되고, 저장소는 트랜잭션 클라이언트 tx에 묶여 있습니다. 2장에서 트랜잭션 한복판에 있던 은행 이체는 커밋 바깥으로 옮겼습니다. 통보 후 롤백되어 "정산됐다고 문자는 왔는데 실제로는 안 된" 상태를 만들고 싶지 않아서였는데, 그 판단을 내릴 자리가 Unit of Work의 경계였습니다.

Unit of Work에는 짝이 되는 패턴이 하나 있습니다. 같은 객체가 두 번 로드되지 않도록 이미 읽은 객체를 맵에 담아 두는 Identity Map입니다. Prisma에는 이 Identity Map이 없는데, 저는 이름을 알기 전까지 그 사실을 의식한 적이 없었습니다.

!

한 트랜잭션 안에서 같은 행을 두 번 findPending하면 Prisma는 서로 다른 객체 두 개를 돌려줍니다. JPA·Hibernate였다면 영속성 컨텍스트가 같은 인스턴스를 돌려줬을 텐데, 그 영속성 컨텍스트가 Unit of Work와 Identity Map을 한 몸에 넣은 구현이기 때문입니다. 예전 프레임워크가 알아서 해 주던 두 패턴의 이름을 알고 나서야 Prisma가 그걸 해 주지 않는다는 사실이 눈에 들어왔습니다. 그래서 한 작업 단위 안에서는 같은 애그리거트를 한 번만 읽어 변수로 들고 다닙니다.

이건 Prisma의 결함이라기보다 설계 철학의 차이라고 생각합니다. JPA는 Unit of Work와 Identity Map을 내장해 편의를 주는 대신 "언제 DB에 나가는지"를 흐리게 만들고, Prisma는 그 둘을 빼는 대신 쿼리를 투명하게 남깁니다. 어느 쪽이 옳은지를 가리기보다, 내 도구가 이 패턴들을 주는지 안 주는지를 물을 수 있게 된 것이 제게 달라진 점이었습니다.

6. 패턴은 처방이 아니라 지도다

이 책을 목차 그대로 체크리스트처럼 읽기 쉽습니다. Repository, Unit of Work, Service Layer를 전부 깔아야 "제대로 된" 아키텍처라는 식으로요. 저도 한동안 그렇게 읽었는데, 정작 Fowler는 그 반대를 이야기합니다. 그는 패턴을 반쯤 익은(half-baked) 것이라 부릅니다. 완성품이 아니라, 각자의 환경에서 직접 마저 구워야 하는 시작점이라는 뜻입니다.

실제로 저도 정산에서 원칙을 일부러 깬 자리가 있습니다. 월별 집계 대시보드는 도메인 객체를 수만 건 로딩해 합산할 이유가 없어서 읽기 전용 쿼리로 우회했습니다. 보호할 불변식이 없는 읽기 경로에서는 Repository가 도메인 객체를 돌려줄 이유가 없다고 봤습니다. 트랜잭션 스크립트가 여전히 맞는 자리도 많습니다. 단순한 관리자 CRUD에 도메인 모델을 씌우는 건, 앞의 그래프에서 교차점 왼쪽에 있으면서 비싼 쪽을 고르는 일입니다.

덧붙이면, 도메인 모델을 고르고도 로직을 전부 서비스에 몰아넣으면 — Fowler가 이후에 빈혈 도메인 모델이라 이름 붙인 — 껍데기만 객체인 구조가 됩니다. 데이터와 행동을 다시 갈라놓았으니 도메인 모델의 이점은 얻지 못하고 비용만 치르는 셈입니다. 패턴의 이름을 아는 일과 패턴을 제대로 쓰는 일은 다르고, 이 책은 전자를 줄 뿐 후자는 판단의 몫으로 남겨 둡니다. 그래서 처방이 아니라 지도라고 적어 두었습니다.

7. 정리 — 이름이 생기면 보인다

20년도 더 된 책이 정산 리팩토링과 NestJS 설계에 쓸모 있었던 이유를 목록으로 남겨 둡니다.

  • 패턴 카탈로그가 남기는 것은 코드가 아니라 어휘다. 이름이 있어야 설계 회의에서 가리키고, 반대하고, 의도적으로 선택하거나 버릴 수 있다.
  • 트랜잭션 스크립트는 잘못이 아니라 조건부 선택이다. 단순할 땐 가장 정직하고, 도메인 로직의 복잡도가 교차점을 넘을 때 도메인 모델로 간다.
  • Service Layer는 응용 로직이 사는 층이다. 트랜잭션 경계·외부 통보·순서는 여기에, 업무 규칙은 도메인 객체 안에 둔다. 둘은 다른 이유로 바뀐다.
  • Repository는 컬렉션의 환상, Data Mapper는 그 환상을 유지하는 번역기다. Prisma는 후자의 절반만 주므로 매핑은 내 몫으로 남는다.
  • Unit of Work와 Identity Map은 프레임워크가 줄 수도, 안 줄 수도 있다. 이름을 알아야 내 도구가 무엇을 안 해 주는지 보인다.
  • 패턴은 처방이 아니라 지도다. Fowler 스스로 반쯤 익은 시작점이라 부르며, 완성은 각자의 환경에 맡긴다.

돌아보면 제가 이 책에서 얻은 건 새로운 기술이 아니라 이미 하던 일을 부를 수 있는 말이었습니다. 부를 수 있게 되고 나서야 그 일을 조금씩 다르게 할 수 있었습니다. 이름이 생기면 보이고, 보이면 고칠 수 있었습니다. 이 어휘를 실제 레거시 위에서 하나씩 적용한 이야기는 아래 글로 이어집니다.
