서평 · 아키텍처

서평: 엔터프라이즈 애플리케이션 아키텍처 패턴 — 이름이 생기면 보인다

2023년 3월• PoEAA · Fowler · 패턴

제 책장에서 가장 손때가 많이 묻은 책을 하나 고르라면 Martin Fowler의 《Patterns of Enterprise Application Architecture》(2002)를 꼽습니다. 표지가 수수한 이 두꺼운 카탈로그가 왜 아직도 유효한가. 답은 이 책이 기술이 아니라 이름을 다루기 때문입니다. 제가 이미 짜고 있던 코드에 이름이 붙는 순간, 비로소 그것을 두고 논쟁할 수 있게 되었습니다.

먼저 짚고 가기
트랜잭션 스크립트Transaction Script
요청 하나를 위에서 아래로 쭉 읽히는 절차 하나로 처리하는 방식입니다. 요리 레시피처럼 순서대로 적는 셈이라, 로직이 단순할 땐 가장 정직합니다. 이 글은 정산 코드가 왜 이 방식에서 시작해 무너졌는지를 출발점으로 삼습니다.
도메인 모델Domain Model
데이터와 그 데이터를 다루는 행동(계산 규칙)을 한 객체 안에 함께 담아, 객체들이 서로 얽힌 그물로 업무를 표현하는 방식입니다. 규칙이 복잡해질수록 트랜잭션 스크립트보다 유리해지는 대안으로 등장합니다.
Service LayerService Layer
한 번의 요청을 처리하는 흐름 — 트랜잭션 경계, 외부 호출, 순서 조율 — 을 담는 층입니다. 업무 규칙(도메인)과 흐름을 다른 층에 나눠 두어, 한쪽을 고치다 다른 쪽을 깨뜨리지 않게 합니다.
RepositoryRepository
데이터베이스를 마치 메모리 속 객체 컬렉션인 것처럼 다루게 해 주는 경계입니다. SQL 같은 저장 기술의 사정을 숨겨, 도메인 코드가 "어디에 어떻게 저장되는지"를 몰라도 되게 합니다.
Unit of WorkUnit of Work
한 업무가 건드린 여러 번의 저장을 하나의 작업 단위로 묶어, 전부 함께 반영하거나 전부 취소하는 장치입니다. 이 글 후반의 "은행 이체를 커밋 뒤로 미루는" 판단이 바로 이 경계에서 나옵니다.

1. 이름이 없으면 논쟁도 없다

이 책은 두 부분으로 나뉩니다. 앞쪽 절반은 계층화·도메인 로직·객체와 관계형 DB의 매핑을 설명하는 서술(narratives)이고, 뒤쪽 절반은 50개가 넘는 패턴을 하나씩 정의한 참조용 카탈로그입니다. 처음 읽을 땐 뒤쪽이 사전처럼 지루해 보였는데, 몇 년 뒤 다시 펴 보니 이 책의 진짜 산출물이 바로 그 사전이었습니다. 코드가 아니라 어휘였던 것입니다.

제 첫 실무는 오래된 웹 시스템을 유지보수하고 화면을 찍어내는 일이었습니다. 그때도 저는 지금 돌아보면 이름이 있는 구조를 짜고 있었는데, 그 이름을 몰랐습니다. 그래서 코드 리뷰에서 "이건 이대로 두는 게 낫다"거나 "여긴 갈아엎어야 한다"를 느낌으로만 말할 수 있었습니다. 가리킬 단어가 없으니 논증이 아니라 취향 싸움이 됐죠. 패턴의 값은 정답을 주는 데 있지 않고, 설계 회의에서 손가락으로 가리킬 대상을 만들어 주는 데 있었습니다.

Fowler는 이 패턴들을 GoF와 크리스토퍼 알렉산더의 전통 위에 올려놓습니다. 패턴은 문제·맥락·해법을 한 이름 아래 묶은 공용어라는 관점이죠. 아래 다섯 개 패턴의 이름 — 트랜잭션 스크립트, 도메인 모델, Service Layer, Repository, Unit of Work — 이 제 설계 언어를 어떻게 바꿨는지가 이 글의 내용입니다.

2. 트랜잭션 스크립트와 도메인 모델의 분기

책에서 가장 자주 다시 읽는 장은 도메인 로직을 조직하는 방법을 다루는 2장입니다. Fowler는 선택지를 셋으로 정리합니다. 트랜잭션 스크립트(요청 하나를 절차 하나로 처리), 도메인 모델(데이터와 행동을 가진 객체망), 그리고 그 사이의 테이블 모듈(테이블 단위로 로직을 모으는 .NET DataSet 계열)입니다. 제가 정산 시스템에서 처음 마주한 코드는 전형적인 트랜잭션 스크립트였습니다.

// settlement.service.ts — 트랜잭션 스크립트: 한 메서드가 전부 한다
@Injectable()
export class SettlementService {
  constructor(private prisma: PrismaService, private bank: BankClient) {}

  async confirm(merchantId: string, month: string) {
    // 1) 원장 조회
    const rows = await this.prisma.ledger.findMany({
      where: { merchantId, month, status: 'PENDING' },
    });
    // 2) 금액 계산 — 수수료·세금·보정이 if로 쌓인다
    let payout = 0;
    for (const r of rows) {
      let fee = r.amount * 0.033;                // 실수(float)로 다루는 돈 — 이미 버그다
      if (r.channel === 'CARD') fee = r.amount * 0.025;
      if (r.promo) fee = fee - r.amount * 0.005;
      payout += r.amount - Math.round(fee);
    }
    // 3) 저장과 4) 외부 통보가 한 메서드에 뒤섞인다
    await this.prisma.settlement.create({
      data: { merchantId, month, payout, status: 'CONFIRMED' },
    });
    await this.bank.transfer(merchantId, payout); // 커밋과 무관하게 날아가는 이체
    return payout;
  }
}

Fowler가 못 박는 핵심은, 트랜잭션 스크립트 자체는 잘못이 아니라는 점입니다. 로직이 단순할 때는 이 방식이 가장 정직합니다. 절차가 위에서 아래로 읽히고, 새 개발자도 즉시 따라갑니다. 이 구조가 무너지는 건 도메인 로직의 복잡도가 자랄 때입니다. 위 코드의 요금 계산이 지점·채널·감면·정산주기로 곱해지기 시작하면, if 중첩이 사람이 추적할 수 있는 깊이를 넘어섭니다.

이 지점을 Fowler는 유명한 그래프 하나로 설명합니다. 가로축은 도메인 로직의 복잡도, 세로축은 변경에 드는 노력입니다. 트랜잭션 스크립트는 낮게 시작해 가파르게 치솟고, 도메인 모델은 높게 시작해 완만하게 오릅니다. 두 선은 어디선가 교차하고, 그 교차점을 넘어서면 도메인 모델이 싸집니다. 정산 코드는 이미 교차점을 한참 지나 있었습니다.

*

Fowler 본인도 이 그래프가 측정에서 나온 게 아니라 경험에 기댄 인상이라고 분명히 밝힙니다. 교차점의 위치는 팀의 객체지향 설계 숙련도에 따라 좌우로 움직이고요. 그래서 이 그림의 쓸모는 "언제 갈아타라"는 답이 아니라, "지금 우리가 어느 축 위에 있는가"를 묻게 만드는 데 있습니다.

3. Service Layer — 응용 로직이 사는 층

도메인 모델로 넘어가기로 했다면 곧바로 부딪히는 질문이 있습니다. 계산 규칙은 객체 안으로 넣는다 치고, 트랜잭션·통보·순서 같은 건 어디에 두나. 위 예제 3·4번 — 저장하고 은행에 이체를 날리는 부분 — 은 정산의 업무 규칙이 아니라 한 번의 요청을 처리하는 흐름입니다. 이걸 도메인 객체에 넣으면 객체가 인프라를 알게 되고, 아무 데도 안 넣으면 컨트롤러가 비대해집니다.

Fowler가 이 자리에 놓는 패턴이 Service Layer입니다. 응용의 경계를 서비스 한 층으로 정의해, 시스템이 제공하는 오퍼레이션의 목록과 각 오퍼레이션의 응답 조율(트랜잭션 경계, 외부 호출, 이벤트 발행)을 담당하게 하는 것이죠. 책은 두 가지 결을 구분합니다. 도메인 모델 위에 얇게 얹는 도메인 파사드형과, 응용 로직까지 두툼하게 품는 오퍼레이션 스크립트형입니다.

SafeIn 백엔드를 NestJS로 설계할 때 저는 이 구분을 그대로 썼습니다. Provider 하나가 유스케이스 하나가 되고, 업무 규칙은 도메인 객체에, 흐름은 서비스 계층에. 중요한 건 규칙과 흐름이 서로 다른 이유로 바뀐다는 사실입니다. 수수료율이 바뀌는 것과 "이체는 커밋 뒤에 한다"가 바뀌는 것은 완전히 다른 사건이고, 둘을 다른 층에 두면 한쪽을 고치다 다른 쪽을 깨뜨리지 않습니다. Service Layer라는 이름을 알기 전에는 이 둘을 한 메서드에 욱여넣고도 무엇이 잘못됐는지 설명하지 못했습니다.

4. Repository와 Data Mapper — Prisma를 다시 보다

도메인 모델을 세우면 다음 질문은 저장입니다. 객체는 메모리에 살고 데이터는 테이블에 사는데, 이 둘을 잇는 방식을 책은 Active Record와 Data Mapper로 나눕니다. Active Record는 행을 감싼 객체가 DB 접근까지 스스로 하는 방식이고, Data Mapper는 객체와 DB, 그리고 매퍼가 서로를 모르게 갈라놓는 방식입니다. 도메인 모델과 테이블 구조가 어긋나기 시작하면 후자가 답입니다.

여기서 Prisma를 다시 보게 됐습니다. Prisma가 돌려주는 것은 메서드 없는 평범한 데이터 객체입니다 — 즉 Active Record가 아닙니다. TypeORM이 문서에서 "Active Record냐 Data Mapper냐"를 대놓고 선택지로 제시하는 것과 대조적이죠. 그 책의 어휘가 20년 뒤 ORM 문서의 목차가 되어 있는 셈입니다. Prisma는 데이터 매핑의 절반(행 읽기)만 해 주고, 행을 도메인 객체로 바꾸는 나머지 절반은 제 몫으로 남깁니다. 그래서 저는 도메인과 Money를 먼저 세웠습니다.

// money.ts — Fowler의 Money 패턴: 금액은 '정수 최소단위 + 통화'다
export class Money {
  private constructor(readonly minor: bigint, readonly currency: Currency) {}

  static krw(won: number) { return new Money(BigInt(won), 'KRW'); }
  plus(o: Money) { return new Money(this.minor + o.minor, this.currency); }
  rate(bp: number) { return new Money(this.minor * BigInt(bp) / 10000n, this.currency); }
}

// settlement.ts — 도메인 모델: 계산 규칙이 객체 안에서 산다
export class Settlement {
  private constructor(
    readonly id: SettlementId,
    private status: SettlementStatus,
    private readonly lines: LedgerLine[],
  ) {}

  payout(policy: FeePolicy): Money {
    return this.lines.reduce(
      (sum, line) => sum.plus(line.net(policy)),
      Money.krw(0),
    );
  }

  confirm() {
    if (this.status !== 'PENDING') throw new DomainError('이미 확정된 정산');
    this.status = 'CONFIRMED';
  }
}

2장 예제의 r.amount * 0.033 같은 실수 연산이 사라진 것에 주목해 주세요. 책의 Base Patterns 장에 있는 Money 패턴은 돈을 부동소수점이 아니라 최소 단위 정수 + 통화로 다루라고 말합니다. 정산 도메인에서 이건 취향이 아니라 정확성의 문제입니다. 그리고 이제 payout()이라는 업무 규칙이 서비스가 아니라 Settlement 안에 살죠. 이 도메인 객체를 저장소가 컬렉션처럼 담아 줍니다.

// settlement.repository.ts — 인메모리 컬렉션인 '척'하는 경계
export interface SettlementRepository {
  findPending(merchantId: string, month: Month): Promise<Settlement | null>;
  save(settlement: Settlement): Promise<void>;
}

// prisma-settlement.repository.ts — Data Mapper: 행 ↔ 도메인 변환은 여기서만
@Injectable()
export class PrismaSettlementRepository implements SettlementRepository {
  constructor(private prisma: PrismaClient) {}

  async findPending(merchantId: string, month: Month) {
    const row = await this.prisma.settlement.findFirst({
      where: { merchantId, month: month.value, status: 'PENDING' },
      include: { lines: true },
    });
    return row ? SettlementMapper.toDomain(row) : null; // 도메인은 Prisma 타입을 모른다
  }

  async save(s: Settlement) {
    const row = SettlementMapper.toRow(s);
    await this.prisma.settlement.update({ where: { id: row.id }, data: row });
  }
}

Fowler의 정의에서 Repository는 도메인과 데이터 매핑 사이를 중재하며, 마치 인메모리 컬렉션인 것처럼 도메인 객체를 다루게 해 주는 경계입니다. 위 인터페이스에는 SQL도, include도, Prisma라는 단어도 없습니다. 그 모든 저장 기술의 사정은 SettlementMapper가 있는 구현체 한 곳에 갇혀 있죠. 이것이 Data Mapper의 역할입니다 — 컬렉션이라는 환상을 유지하기 위해 행과 객체를 번역하는 자리. 이름을 알고 나니, Prisma가 어디까지 해 주고 어디부터 내 일인지가 또렷해졌습니다.

5. Unit of Work와 Identity Map — 프레임워크의 몫

저장소를 컬렉션처럼 다루기로 했다면, "여러 번의 저장을 하나의 원자적 작업으로 묶는 일"은 누가 하나요. 책은 이 역할에 Unit of Work라는 이름을 줍니다. 하나의 업무 트랜잭션이 건드린 객체들을 모아 두었다가, 변경을 한꺼번에 써 내리고 동시성 문제를 정리하는 장치입니다. 정산 확정을 유스케이스로 옮기면 이 경계가 눈에 보입니다.

// confirm-settlement.usecase.ts — $transaction 콜백 = 하나의 작업 단위
@Injectable()
export class ConfirmSettlement {
  constructor(private prisma: PrismaService, private bank: BankClient, private feePolicy: FeePolicy) {}

  async execute(merchantId: string, month: Month) {
    const payout = await this.prisma.$transaction(async (tx) => {
      const repo = new PrismaSettlementRepository(tx); // 저장소를 트랜잭션에 묶는다
      const s = await repo.findPending(merchantId, month);
      if (!s) throw new NotFound('정산 대상 없음');
      s.confirm();                          // 도메인 규칙
      await repo.save(s);                     // 이 콜백 안의 쓰기는 전부 원자적
      return s.payout(this.feePolicy);
    });
    await this.bank.transfer(merchantId, payout); // 통보는 커밋 이후로
    return payout;
  }
}

Prisma의 $transaction 콜백이 그대로 하나의 작업 단위입니다. 콜백 안의 모든 쓰기가 함께 커밋되거나 함께 롤백되고, 저장소는 트랜잭션 클라이언트 tx에 묶여 있죠. 그리고 2장에서 트랜잭션 한복판에 있던 은행 이체가 커밋 바깥으로 밀려났습니다. 통보 후 롤백되어 "정산됐다고 문자는 왔는데 실제로는 안 된" 상태를 만들지 않으려는 것이고, 그 판단을 내릴 자리가 바로 Unit of Work의 경계입니다.

여기서 이 책이 준 가장 값진 깨달음이 옵니다. Unit of Work에는 짝이 되는 패턴 Identity Map이 있습니다 — 같은 객체가 두 번 로드되지 않도록, 이미 읽은 객체를 맵에 담아 두는 장치죠. 그런데 Prisma에는 Identity Map이 없습니다.

!

한 트랜잭션 안에서 같은 행을 두 번 findPending하면 Prisma는 서로 다른 객체 두 개를 돌려줍니다. JPA·Hibernate였다면 영속성 컨텍스트가 같은 인스턴스를 돌려줬을 겁니다 — 그 영속성 컨텍스트가 바로 Unit of Work와 Identity Map을 한 몸에 넣은 구현이니까요. 프레임워크가 공짜로 주던 두 패턴의 이름을 알고 나서야, Prisma가 그걸 안 준다는 사실이 비로소 보였습니다. 그래서 한 작업 단위 안에서는 같은 애그리거트를 한 번만 읽어 변수로 들고 다닙니다.

이건 Prisma의 결함이 아니라 설계 철학의 차이입니다. JPA는 Unit of Work와 Identity Map을 내장해 편의를 주는 대신 "언제 DB에 나가는지"를 흐리게 만들고, Prisma는 그 둘을 빼는 대신 쿼리를 투명하게 남깁니다. 어느 쪽이 옳으냐가 아니라, 내 도구가 이 패턴들을 주는가 안 주는가를 물을 수 있게 됐다는 것 — 그게 이름의 힘이었습니다.

6. 패턴은 처방이 아니라 지도다

이 책을 오해하는 가장 흔한 방식은 목차를 체크리스트로 읽는 것입니다. Repository, Unit of Work, Service Layer를 전부 깔아야 "제대로 된" 아키텍처라고요. Fowler는 정확히 그 반대를 말합니다. 그는 패턴을 반쯤 익은(half-baked) 것이라 부릅니다 — 완성품이 아니라, 각자의 환경에서 직접 마저 구워야 하는 시작점이라는 뜻입니다.

실제로 저는 정산에서 원칙을 의도적으로 깬 자리가 있습니다. 월별 집계 대시보드는 도메인 객체를 수만 건 로딩해 합산할 이유가 없어서, 읽기 전용 쿼리로 우회했습니다. 보호할 불변식이 없는 읽기 경로에서는 Repository가 도메인 객체를 돌려줄 이유 자체가 사라지니까요. 트랜잭션 스크립트가 여전히 맞는 자리도 많습니다. 단순한 관리자 CRUD에 도메인 모델을 씌우는 건, 아까 그 그래프에서 교차점 왼쪽에 있으면서 비싼 쪽을 고르는 일입니다.

덧붙이면, 도메인 모델을 고르고도 로직을 전부 서비스에 몰아넣으면 — Fowler가 이후에 빈혈 도메인 모델이라 이름 붙인 — 껍데기만 객체인 구조가 됩니다. 데이터와 행동을 다시 갈라놓았으니 도메인 모델의 이점은 하나도 못 얻고 비용만 치르는 것이죠. 패턴의 이름을 아는 일과 패턴을 제대로 쓰는 일은 다르고, 이 책은 전자를 줄 뿐 후자는 여전히 판단의 몫으로 남깁니다. 그래서 지도이지 처방이 아닙니다.

7. 정리 — 이름이 생기면 보인다

20년도 더 된 책이 정산 리팩토링과 NestJS 설계에 여전히 쓸모 있었던 이유를 목록으로 남깁니다.

  • 패턴 카탈로그의 산출물은 코드가 아니라 어휘다. 이름이 있어야 설계 회의에서 가리키고, 반대하고, 의도적으로 선택하거나 버릴 수 있다.
  • 트랜잭션 스크립트는 죄가 아니라 조건부 선택이다. 단순할 땐 가장 정직하고, 도메인 로직의 복잡도가 교차점을 넘을 때 도메인 모델로 간다.
  • Service Layer는 응용 로직의 집이다. 트랜잭션 경계·외부 통보·순서는 여기에, 업무 규칙은 도메인 객체 안에. 둘은 다른 이유로 바뀐다.
  • Repository는 컬렉션의 환상, Data Mapper는 그 환상을 유지하는 번역기다. Prisma는 후자의 절반만 주므로 매핑은 내 몫이다.
  • Unit of Work와 Identity Map은 프레임워크가 줄 수도, 안 줄 수도 있다. 이름을 알아야 내 도구가 무엇을 안 해 주는지 보인다.
  • 패턴은 처방이 아니라 지도다. Fowler 스스로 반쯤 익은 시작점이라 부르며, 완성은 각자의 환경에 맡긴다.

돌아보면 제가 이 책에서 얻은 건 새로운 기술이 아니라 이미 하던 일을 부를 수 있는 말이었습니다. 그리고 부를 수 있게 되자 비로소 그것을 더 잘할 수 있게 됐습니다. 이름이 생기면 보이고, 보이면 고칠 수 있으니까요. 이 어휘를 실제 레거시 위에서 하나씩 적용한 이야기는 아래 글로 이어집니다.
