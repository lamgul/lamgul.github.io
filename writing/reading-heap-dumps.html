힙덤프 읽는 법 — MAT의 Dominator Tree로 범인 찾기

2025년 9월• Heap Dump · MAT · GC

서버가 7일마다 죽던 때가 있었습니다. 원인은 힙덤프 안에 있었는데, 정작 저는 힙덤프를 읽을 줄 몰랐습니다. 며칠 동안 붙잡고 익힌 순서를 잊지 않으려고 적어 둡니다. 막상 해보니 봐야 하는 화면은 몇 개 되지 않았습니다.

먼저 짚고 가기
힙덤프Heap Dump
어느 시점의 JVM 힙 전체를 파일로 직렬화한 바이너리 스냅샷입니다. 보통 HPROF 형식(.hprof)이고, 객체마다 소속 클래스·필드 값·다른 객체를 가리키는 참조까지 담고 있어 그 순간의 객체 그래프를 통째로 복원할 수 있습니다. 덤프를 뜨는 동안 JVM은 세이프포인트에서 멈추고, live 옵션을 주면 Full GC를 먼저 돌린 뒤 살아남은 객체만 기록합니다. 시간 축이 없는 한 장의 사진이라 "언제 누가 이 객체를 할당했는가"는 들어 있지 않습니다.
메모리 누수Memory Leak
논리적으로는 더 이상 쓰이지 않는데도 GC Root에서 참조 사슬이 끊기지 않아, GC가 회수하지 못한 채 계속 쌓이는 객체들을 가리킵니다. JVM 같은 관리형 런타임에서는 객체가 여전히 도달 가능(reachable)하다는 점이 핵심이라, 참조를 잃어 해제하지 못하는 C/C++식 누수와는 원인이 정반대입니다. 그래서 GC를 아무리 돌려도 사라지지 않고, 붙잡고 있는 참조 자체를 끊어야 회수됩니다.
GCGarbage Collection
GC Root 집합에서 출발해 참조를 따라가며 도달 가능한 객체를 살아 있다고 표시하고, 끝내 도달하지 못한 객체를 회수하는 자동 메모리 관리 방식입니다(추적형 GC, mark-and-sweep 계열). 판정 기준은 "실제로 쓰이는가"가 아니라 "루트에서 참조로 닿는가"라서, 도달은 되지만 논리적으로 안 쓰이는 객체는 회수 대상에서 빠집니다. 누수는 바로 이 틈에서 생깁니다.
Dominator Tree
객체 그래프를 지배(dominance) 관계로 다시 묶은 트리입니다. 객체 A가 B를 "지배한다"는 것은 GC Root에서 B에 이르는 모든 경로가 반드시 A를 거친다는 뜻이고, 그래서 A가 회수되면 A가 지배하는 객체는 전부 함께 회수됩니다. MAT는 이 트리로 각 객체의 Retained Heap(그 객체가 지배하는 하위 전체의 크기)을 계산해 큰 순서로 세웁니다. 트리의 부모·자식은 실제 참조 관계가 아니라 지배 관계라, 힙의 참조 구조와는 모양이 다를 수 있습니다.
GC Roots
힙 바깥에서 힙 안의 객체를 직접 붙잡고 있는 참조 지점들의 집합으로, GC의 도달 가능성 탐색이 여기서 시작됩니다. 실행 중인 스레드 스택의 지역 변수, 클래스의 static 필드, JNI 전역 참조, 모니터로 잠긴 객체 등이 루트가 됩니다. 어떤 객체든 이 루트 가운데 하나까지 참조 사슬로 이어져 있으면 도달 가능한 것으로 보아 회수되지 않습니다.

1. 사고 순간을 자동으로 잡는다

# OOM이 나는 그 순간을 자동으로 덤프
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/app/heap.hprof

# 살아있는 프로세스를 수동으로 뜰 때
jmap -dump:live,format=b,file=heap.hprof <pid>

누수는 재현하기가 어려워서, HeapDumpOnOutOfMemoryError를 걸어 두고 기다리는 쪽이 빨랐습니다. 터지는 순간의 스냅샷이 남아 있어야 그다음을 볼 수 있었습니다.

2. Dominator Tree로 "누가 붙잡고 있나"

Eclipse MAT를 열면 저는 Dominator Tree부터 봅니다. 여기서 봐야 하는 값은 Shallow Heap(그 객체 자체 크기)이 아니라 Retained Heap(그 객체가 사라지면 함께 회수될 총 크기)입니다. 두 값을 헷갈리면 엉뚱한 객체를 붙잡고 시간을 씁니다. Retained가 유독 큰 객체가 메모리를 쥐고 있는 쪽이고, 제 경우에는 특정 캐시 클래스 하나가 힙의 80%를 retained로 들고 있었습니다.

3. Leak Suspects와 GC Roots

MAT의 Leak Suspects 리포트는 의심 객체를 자동으로 짚어 줍니다. 확정은 아니지만 어디부터 볼지 정하는 데는 도움이 됐습니다. 그다음 "이게 왜 GC에 안 먹히지?"는 Path to GC Roots로 확인했습니다. 이 객체를 살려두는 참조 사슬을 뿌리까지 그려 줍니다. 제 경우 그 사슬 끝에는 동기화 없이 공유되던 static 컬렉션이 있었습니다. static은 프로그램이 켜져 있는 내내 사라지지 않기 때문에, 여기에 담긴 객체도 함께 회수되지 못한 채 남아 있었습니다.

*

"Path to GC Roots → exclude weak/soft references" 옵션은 켜 두고 봤습니다. 약참조는 어차피 회수되기 때문에, 객체를 실제로 붙잡고 있는 강참조 경로만 남겨야 사슬이 읽힙니다.

4. 증상이 아니라 원인

힙덤프가 알려준 것은 "무엇이 쌓였나"까지였고, "왜 쌓였나"는 코드로 돌아가서 봐야 했습니다. 멀티스레드에서 동기화가 빠져 객체가 회수되지 못하고 있었고, synchronized를 붙여 정리했습니다. 도구는 볼 곳을 좁혀 줄 뿐, 마지막은 코드를 읽는 일이었습니다.
