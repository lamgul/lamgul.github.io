힙덤프 읽는 법 — MAT의 Dominator Tree로 범인 찾기

2025년 9월• 약 8분• Heap Dump · MAT · GC

7일마다 죽던 서버의 진짜 원인은 힙덤프 안에 있었습니다. 문제는 힙덤프를 읽을 줄 몰랐다는 것. 그때 익힌 순서를 정리합니다. 겁먹을 것 없이, 몇 개의 화면만 보면 됩니다.

먼저 짚고 가기
힙덤프Heap Dump
어느 한순간 프로그램이 메모리에 쥐고 있던 모든 객체를 통째로 찍은 사진입니다. 사고 순간의 이 사진을 뜯어보는 것이 이 글의 전부입니다.
메모리 누수Memory Leak
더는 쓰지 않는데도 프로그램이 계속 붙잡고 있어 치워지지 않는 객체가 쌓이는 현상입니다. 방을 안 치우면 물건이 쌓이듯, 메모리가 서서히 꽉 차 결국 서버가 죽습니다.
GCGarbage Collection
아무도 더 참조하지 않는 객체를 자동으로 찾아 메모리에서 치워 주는 청소부입니다. 누수는 바로 이 청소부가 "아직 누군가 쓴다"고 착각해 못 치우는 상태를 말합니다.
Dominator Tree
"이 객체가 사라지면 덩달아 함께 사라질 객체가 얼마나 되는가"를 크기순으로 줄 세운 표입니다. 메모리를 가장 많이 붙잡은 범인을 한눈에 찾게 해 줍니다.
GC Roots
실행 중인 스레드, static 변수처럼 GC가 "여기서부터는 무조건 살아 있다"고 보는 출발점입니다. 어떤 객체가 이 뿌리까지 참조로 이어져 있으면 청소 대상에서 빠집니다.

1. 사고 순간을 자동으로 잡는다

# OOM이 나는 그 순간을 자동으로 덤프
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/app/heap.hprof

# 살아있는 프로세스를 수동으로 뜰 때
jmap -dump:live,format=b,file=heap.hprof <pid>

재현이 어려운 누수일수록 HeapDumpOnOutOfMemoryError가 생명줄입니다. 터지는 순간의 스냅샷이 곧 증거니까요.

2. Dominator Tree로 "누가 붙잡고 있나"

Eclipse MAT를 열면 가장 먼저 볼 것은 Dominator Tree입니다. 여기서 중요한 건 Shallow Heap(그 객체 자체 크기)이 아니라 Retained Heap(그 객체가 사라지면 함께 회수될 총 크기)입니다. Retained가 압도적으로 큰 객체가 범인입니다. 저는 특정 캐시 클래스 하나가 힙의 80%를 retained로 쥐고 있는 걸 봤습니다.

3. Leak Suspects와 GC Roots

MAT의 Leak Suspects 리포트는 의심 객체를 자동으로 짚어줍니다. 시작점으로 훌륭하죠. 그다음 "이게 왜 GC에 안 먹히지?"는 Path to GC Roots로 봅니다 — 이 객체를 살려두는 참조 사슬을 뿌리까지 그려줍니다. 제 경우 그 사슬 끝에 동기화 없이 공유되던 static 컬렉션이 있었습니다. static은 프로그램이 켜져 있는 내내 사라지지 않기 때문에, 여기에 담긴 객체도 함께 영영 회수되지 못한 것이죠.

*

"Path to GC Roots → exclude weak/soft references" 옵션을 꼭 켜세요. 약참조는 어차피 회수되니, 진짜로 객체를 붙잡는 강참조 경로만 봐야 범인이 보입니다.

4. 증상이 아니라 원인

힙덤프가 알려준 건 "무엇이 쌓였나"였고, 코드로 돌아가 "왜 쌓였나"를 봤습니다. 멀티스레드에서 동기화가 빠져 객체가 회수되지 못하고 있었죠. synchronized 한 줄이 답이었습니다. 도구는 범위를 좁혀줄 뿐, 마지막 한 걸음은 결국 코드를 읽는 일입니다.
