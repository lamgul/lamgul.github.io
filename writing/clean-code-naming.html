클린코드 · 이름 — 변수명이 주석을 대신할 때

2020년 4월• 약 12분• 클린코드 · 네이밍 · 가독성

정산 로직이 5곳에 중복돼 있던 시스템을 리팩토링할 때, 가장 오래 걸린 일은 패턴 적용이 아니라 중복을 찾는 것이었습니다. 같은 계산이 calcCharge, computeFee, getSettleAmt 같은 제각각의 이름으로 흩어져 있어 검색으로는 잡히지 않았기 때문입니다. 그때 배웠습니다 — 이름은 취향의 문제가 아니라 비용의 문제라는 것을. 이 글은 그 비용을 줄이는 구체적인 규칙을, 나쁜 예를 좋은 예로 고쳐가며 정리합니다.

먼저 짚고 가기
리팩토링Refactoring
동작(결과)은 그대로 두고 코드의 구조나 이름만 더 읽기 좋게 손보는 작업입니다. 방을 옮기지 않고 정리만 하는 것과 같습니다. 이 글에서 "나쁜 이름을 좋은 이름으로 바꾸기"가 바로 가장 작고 안전한 리팩토링입니다.
불리언Boolean
참(true) 아니면 거짓(false), 딱 두 값만 가지는 자료형입니다. 스위치의 켜짐/꺼짐과 같습니다. if 문의 조건은 결국 불리언이라, 이 이름을 잘 지으면 조건문이 자연어 문장처럼 읽힙니다.
헝가리안 표기법Hungarian notation
변수 이름 앞에 그 값의 타입을 약자로 붙이던 옛 관습입니다(예: 문자열이면 str, 정수면 i). IDE가 타입을 바로 보여주는 지금은 이름과 타입이 어긋날 위험만 남아, 이 글에서 "유물"로 다룹니다.
스코프Scope
변수가 살아 있고 접근 가능한 범위입니다. 함수 한 줄 안에서만 쓰는 변수는 스코프가 짧고, 모듈 전체에서 쓰면 깁니다. 이 글의 결론 하나가 "이름 길이는 스코프 길이에 비례해야 한다"이므로 꼭 필요한 개념입니다.
유비쿼터스 언어Ubiquitous Language
도메인 전문가와 개발자가 같은 뜻으로 합의해 회의·문서·코드에서 똑같이 쓰는 공통 용어집입니다. DDD(도메인 주도 설계)의 핵심 개념으로, 이름이 팀 사전과 1:1로 맞아야 검색과 소통이 통한다는 5장의 토대입니다.

1. 주석이 필요한 이름은 이미 실패한 이름

흔히 "이름이 좀 짧아도 주석을 달면 되지 않나"라고 합니다. 실제로 많이 보는 코드가 이렇습니다.

// Before — 이름이 못 하는 말을 주석이 대신한다
const list = getData(mid);          // mid: 가맹점 ID
let amt = 0;                        // 정산 기준 금액(원)
for (const t of list) {
  if (t.st === 2) amt += t.v;       // 2 = 결제 완료, v = 금액
}

// After — 주석이 하던 말을 이름이 한다
const transactions = findTransactionsByMerchantId(merchantId);
let settlementBaseAmountWon = 0;
for (const tx of transactions) {
  if (tx.status === TransactionStatus.Completed) {
    settlementBaseAmountWon += tx.amountWon;
  }
}

왜 주석보다 이름이어야 할까요. 논거는 두 가지입니다. 첫째, 주석은 선언 지점에만 있고, 이름은 모든 사용 지점에 따라다닙니다. amt를 선언부에서 30줄 아래에서 만나면 주석을 찾아 다시 올라가야 하지만, settlementBaseAmountWon은 만나는 자리에서 바로 읽힙니다. 이름은 사용처마다 자동으로 복사되는 주석인 셈입니다. 둘째, 주석은 컴파일러가 검증하지 않습니다. 코드가 바뀌어도 주석은 조용히 낡아가고, 어느 순간부터 거짓말을 합니다. 반면 이름은 IDE의 rename 리팩토링으로 모든 사용처가 함께 바뀝니다. 즉 이름은 도구가 관리해 주는 문서이고, 주석은 사람이 기억해서 관리해야 하는 문서입니다. 같은 정보라면 관리 비용이 낮은 쪽에 두는 것이 맞습니다.

그래서 저는 코드 리뷰에서 "여기 주석 좀 달아주세요"라고 말하고 싶어질 때마다 먼저 자문합니다. 이 주석이 필요해진 건, 이름이 일을 안 하고 있어서 아닌가? 대부분은 그렇습니다.

2. 축약어와 헝가리안 — 규칙이 아니라 유물

strDt, arrList, iCnt 같은 이름은 아직도 레거시 Java 코드에서 자주 만납니다. 이런 헝가리안 표기가 한때는 합리적이었다는 사실을 먼저 인정해야 설득이 됩니다. 타입 정보가 약한 언어에서, 선언부를 보려면 종이 출력물을 뒤져야 하던 시절에는 이름에 타입을 새기는 것(헝가리안 표기법)이 실제로 읽기 비용을 줄였습니다. 문제는 그 전제가 전부 사라졌다는 것입니다. 지금은 타입 시스템이 타입을 보증하고, IDE가 마우스만 올려도 타입을 보여줍니다.

// Before — 타입을 이름에 새기던 시절의 습관
String strDt = ord.getDt();             // 정산 일자… 인데 String?
List arrList = repo.findAll();          // 뭐가 들었는지 이름은 침묵
int iCnt = 0;                           // 무엇의 개수인가

// After — 타입은 타입 시스템에게, 이름은 의미에게
LocalDate settlementDate = order.getSettlementDate();
List<Settlement> pendingSettlements = settlementRepository.findAllPending();
int payoutRetryCount = 0;

전제가 사라진 표기법은 비용만 남깁니다. strDt를 LocalDate로 바꾸는 순간 이름이 거짓말을 시작하고, 정보가 타입과 이름에 이중으로 적혀 있으니 하나가 바뀔 때마다 불일치 위험이 생깁니다. 축약도 같은 구조입니다. 축약이 아끼는 것은 쓰는 사람의 타이핑인데, 자동완성 시대에 타이핑 비용은 0에 수렴합니다. 반면 읽는 비용 — Dt가 date인지 detail인지 추측하는 비용 — 은 읽을 때마다 매번 발생합니다. 코드는 한 번 쓰이고 수십 번 읽히므로, 이 거래는 항상 손해입니다.

*

실무 팁 — rename은 IDE가 제공하는 가장 안전한 리팩토링입니다(IntelliJ Shift+F6, VS Code F2). 테스트가 없어도 안전하게 돌릴 수 있는 거의 유일한 변경이니, 나쁜 이름을 발견하면 "나중에 크게 고칠 때"로 미루지 말고 그 자리에서 바꾸세요. PR에서 이름을 다듬는 것은 가장 싸게 사는 품질입니다.

3. 불리언 — 조건문이 문장이 되게

불리언 이름의 목표는 하나입니다. if 문을 소리 내어 읽었을 때 자연어 문장이 될 것. flag, check, status는 이 시험을 통과하지 못합니다. "만약 플래그라면"은 문장이 아니니까요.

// Before — flag는 아무것도 말하지 않는다
let flag = false;
if (!chkHoliday(dt) && u.state != 0 && !u.noAcct) {
  flag = true;
}

// After — 조건 자체가 문장이 된다
const isBusinessDay    = !isHoliday(settlementDate);
const isActiveMerchant = merchant.status === MerchantStatus.Active;
const hasBankAccount   = merchant.bankAccount !== null;

const canReceivePayout = isBusinessDay && isActiveMerchant && hasBankAccount;

규칙은 세 가지로 정리됩니다. 첫째, 접두사는 is(상태) · has(보유) · can(가능) · should(정책)로 통일합니다. 접두사가 곧 그 불리언의 범주를 알려주기 때문입니다. 둘째, 긍정형으로 짓습니다. Before의 u.noAcct를 보세요 — 부정형 이름은 반드시 !u.noAcct 같은 이중 부정을 낳고, 이중 부정은 사람이 머릿속에서 한 번 뒤집어 읽어야 하는 인지 비용입니다. hasBankAccount로 뒤집어 두면 부정이 필요한 쪽에서만 !를 쓰면 됩니다. 셋째, 복합 조건에는 중간 이름을 부여합니다. After에서 canReceivePayout 한 줄이 추가된 대가로, 조건이 틀렸을 때 "어느 항이 틀렸는가"를 디버거 없이 이름 단위로 좁힐 수 있게 됐습니다. NestJS 권한 가드처럼 조건이 정책인 코드에서 특히 효과가 큽니다.

4. 컬렉션·시간·단위 — 형태가 정해주는 이름

값의 형태마다 검증된 관례가 있습니다. 관례를 따르는 이유는 미학이 아니라, 읽는 사람의 추측을 없애기 위해서입니다.

// Before — 숫자 3의 단위를 아는 사람은 작성자뿐
const data = getList(mid);
const total = data.map(x => x.amount).reduce((a, b) => a + b, 0);
schedule(total, 3);                      // 3… 일? 시간? 퍼센트?

// After — 복수형·xByY·~At·~Date·단위 접미사
const settlements = findSettlementsByMerchantId(merchantId);
const settlementByOrderId = new Map(settlements.map(s => [s.orderId, s]));
const netAmountWon = sumWon(settlements.map(s => s.netAmountWon));

schedulePayout({
  amountWon:   netAmountWon,
  requestedAt: clock.now(),              // 시각(순간)은 ~At
  payoutDate:  nextBusinessDay(today),   // 날짜는 ~Date
  holdDays:    3,                        // 기간은 단위를 이름에
});

하나씩 근거를 보겠습니다. 컬렉션은 복수형(settlements)으로 — list, data는 "여러 개"라는 것만 말하고 "무엇의 여러 개"인지는 숨깁니다. Map은 valueByKey(settlementByOrderId) 형태로 — 이 관례 하나로 "키가 뭐지?"라는 질문이 사라지고, settlementByOrderId.get(orderId)처럼 사용부가 자기 검증 문장이 됩니다. 시간은 시각과 기간을 구분합니다. settledAt은 순간(timestamp), settlementDate는 달력상 날짜, holdDays·timeoutMs는 길이입니다. 이 셋을 전부 time이라 부르는 코드에서 시차 버그가 나옵니다. 마지막으로 돈에는 반드시 단위와 성격을 붙입니다. 정산 도메인에서 amount 하나만 있는 필드는 사실상 지뢰입니다 — 수수료 공제 전(grossAmount)인지 후(netAmount)인지에 따라 가맹점에 입금되는 금액이 달라지니까요.

!

함정 — 단위 없는 돈은 이름의 문제로 끝나지 않습니다. 결제 수단마다 요금이 달랐던 정산 버그를 추적할 때, 어떤 코드는 수수료 포함 금액을, 어떤 코드는 공제 후 금액을 같은 이름 amount로 주고받고 있었습니다. 이름을 grossAmountWon / netAmountWon으로 갈라 놓기만 해도 잘못된 값을 넘기는 코드가 리뷰에서 눈에 띄기 시작합니다. 여기서 한 걸음 더 가면 Money 타입으로 컴파일러에게 감시를 맡길 수 있습니다.

5. 유비쿼터스 언어 — 팀의 사전이 곧 코드

지금까지는 이름 하나를 잘 짓는 규칙이었습니다. 그런데 실무에서 더 큰 사고는 이름들 사이의 불일치에서 납니다. 서두의 정산 리팩토링이 정확히 그 사례였습니다. 웹 결제·키오스크·카카오페이·월정액·관리자 정산, 다섯 곳이 같은 계산을 각자의 이름으로 갖고 있었고, 이름이 다르니 grep으로도 잡히지 않았습니다. 할인 정책이 바뀔 때 다섯 곳 중 몇 곳만 수정되는 사고는 이 지형에서 필연이었습니다.

// Before — 같은 개념이 세 가지 이름으로 흩어져 있다
webPayment.calcCharge(order);       // 웹 결제의 "정산"
kiosk.computeFee(order);            // 키오스크의 "정산"
adminBatch.getSettleAmt(order);     // 수동 정산의 "정산"

// After — 용어집의 단어가 그대로 타입과 메서드가 된다
Settlement settlement = settlementPolicy.settle(order);   // 정산: 금액 확정
Payout payout = payoutScheduler.schedule(settlement);     // 지급: 실제 송금

DDD가 말하는 유비쿼터스 언어는 거창한 것이 아닙니다. 도메인 전문가와 개발자가 합의한 용어집을 만들고, 그 단어를 회의·문서·코드에서 똑같이 쓰자는 약속입니다. 정산 도메인이라면 이렇게 됩니다 — 정산(Settlement)은 주기 단위로 금액을 확정하는 일, 지급(Payout)은 확정된 금액을 실제 송금하는 일, 수수료(Fee)는 공제 항목, 보류(Hold)는 지급을 미루는 상태. 이 네 단어가 코드의 클래스명·메서드명과 1:1로 대응하면, 기획자가 "지급 보류 조건을 바꿔주세요"라고 말한 순간 수정할 클래스가 이미 특정됩니다. 크롤링 파이프라인을 DDD로 재설계했을 때도 효과의 절반은 아키텍처가 아니라 이 언어 통일에서 왔습니다.

규칙은 대칭 쌍으로 기억하면 좋습니다. 하나의 개념에 하나의 단어(같은 것을 settle/calc/compute로 부르지 않기), 그리고 하나의 단어에 하나의 개념(fee가 어디서는 수수료, 어디서는 요금 전체를 뜻하지 않기). 전자가 깨지면 검색이 안 되고, 후자가 깨지면 검색 결과를 믿을 수 없게 됩니다.

6. 규칙이 깨지는 곳 — 짧은 이름이 옳을 때

여기까지 읽으면 "이름은 무조건 길고 자세하게"로 오해하기 쉽습니다. 그러나 위 규칙들이 성립한 근거를 다시 보면 — 이름은 사용처마다 따라다니는 주석이다 — 사용처가 코앞 한 줄뿐이라면 긴 이름의 근거도 사라집니다. 이름 길이는 스코프 길이에 비례해야 합니다.

// 이 s는 좋은 이름이다 — 수명이 화살표 함수 한 줄이니까
const totalNetWon = settlements.reduce((sum, s) => sum + s.netAmountWon, 0);

// 오히려 과잉 이름이 신호를 잡음으로 덮는다
const totalNetWon2 = settlements.reduce(
  (accumulatedNetAmountWon, settlementListItem) =>
    accumulatedNetAmountWon + settlementListItem.netAmountWon, 0);

// 같은 한 글자라도 수명이 모듈 전체라면 폭탄이 된다
export let c = loadConfig();   // 6개월 뒤의 c는 아무도 못 읽는다

람다 매개변수의 s, 두 줄짜리 루프의 i는 선언과 사용이 한 시야에 들어오므로 추측 비용이 0입니다. 이때 settlementListItem 같은 이름은 정보를 더하는 게 아니라 패턴 인식을 방해합니다. 팀 사전에 등록된 관례적 축약 — id, url, Express/NestJS의 req/res — 도 같은 이유로 허용됩니다. 이미 모두가 아는 단어는 축약이 아니라 어휘입니다.

한 가지 한계도 정직하게 적어둡니다. 이름은 "무엇"까지만 말할 수 있고 "왜"는 말하지 못합니다. holdDays: 3이라는 이름은 3일이 보류 기간이라는 사실은 전하지만, 왜 3일인지 — 카드사 취소 가능 기간 때문인지, 정산 주기 협약 때문인지 — 는 전하지 못합니다. 그 "왜"는 여전히 주석과 커밋 메시지, 설계 문서의 몫입니다. 변수명이 주석을 대신하는 범위는 딱 "무엇"까지입니다. 이 경계를 인정해야 "주석 무용론" 같은 극단으로 가지 않습니다.

7. 정리 — 이름 리뷰 체크리스트

PR을 리뷰할 때 실제로 쓰는 순서대로 정리합니다.

  • 주석 테스트 — 이 변수에 주석이 필요한가? 필요하다면 이름이 일을 안 하는 것. 주석의 내용을 이름으로 옮긴다.
  • 축약·헝가리안 금지 — 타입 정보는 타입 시스템에게. 단, 팀 사전에 등록된 축약(id, url, req/res)은 어휘로 인정.
  • 불리언은 문장으로 — is/has/can/should + 긍정형. 이중 부정이 보이면 이름을 뒤집는다. 복합 조건엔 중간 이름을.
  • 컬렉션은 복수형, Map은 valueByKey — 필터된 결과라면 조건까지 이름에(pendingSettlements).
  • 시간은 세 갈래 — 시각은 ~At, 날짜는 ~Date, 기간은 단위 접미사(~Days, ~Ms).
  • 돈에는 성격과 단위 — gross/net/fee 구분, 통화 단위 명시. 가능하면 타입으로 승격.
  • 하나의 개념 = 하나의 단어 — 용어집을 먼저 합의하고, 회의에서 쓰는 단어를 코드에 그대로 쓴다.
  • 이름 길이 ∝ 스코프 길이 — 한 줄 람다의 한 글자는 무죄, 모듈 전역의 한 글자는 유죄.
  • "왜"는 이름 밖에 — 이름은 "무엇"까지. 이유는 주석·커밋·문서에 남긴다.

다음 글에서는 이 이름들이 모여 사는 곳 — 함수를 다룹니다. 좋은 이름의 함수가 왜 짧아질 수밖에 없는지가 이 글의 자연스러운 다음 장입니다.
