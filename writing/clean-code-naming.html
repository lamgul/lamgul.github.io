클린코드 · 이름 — 변수명이 주석을 대신할 때

2020년 4월• 클린코드 · 네이밍 · 가독성

정산 로직이 5곳에 중복돼 있던 시스템을 리팩토링할 때, 시간이 가장 많이 든 일은 패턴을 적용하는 쪽이 아니라 중복을 찾는 쪽이었습니다. 같은 계산이 calcCharge, computeFee, getSettleAmt처럼 제각각의 이름으로 흩어져 있어서 검색으로는 걸리지 않았기 때문입니다. 그전까지 저는 이름을 취향의 문제로 여겼는데, 그 일을 겪고 나서는 비용의 문제로 보게 됐습니다. 그 비용을 줄이려고 정리해 둔 규칙들을, 나쁜 예를 좋은 예로 고쳐가며 적어 둡니다.

먼저 짚고 가기
리팩토링Refactoring
외부에서 관찰되는 동작을 바꾸지 않은 채 코드의 내부 구조만 고쳐 이해와 수정 비용을 낮추는 작업을 말합니다(Martin Fowler, 『Refactoring』). 기능 추가·버그 수정과 갈리는 기준이 '관찰 가능한 동작의 보존'이고, 그 보존을 보증하는 것은 작성자의 의도가 아니라 통과하는 테스트입니다. 이름 바꾸기(rename)는 변경 범위가 선언 한 곳과 그 이름의 모든 참조로 한정되는, 이 작업의 가장 작은 단위입니다.
불리언Boolean
참(true)과 거짓(false) 두 값만 갖는 자료형입니다. 비교 연산자(===, >)가 돌려주는 값의 타입이자 if·while이 분기를 판단할 때 보는 값이라, 불리언에 붙인 이름은 조건문에서 문장 자리에 그대로 놓입니다. 다만 조건절에 놓인 값이 언제나 불리언인 것은 아니어서, JavaScript는 truthy/falsy 규칙으로 다른 타입을 대신 평가하고 &&·||도 불리언이 아니라 피연산자 중 하나를 그대로 돌려줍니다. Java의 Boolean처럼 null을 허용하는 래퍼 타입에서는 상태가 셋이 되므로 '두 값'이라는 전제 자체가 깨집니다.
헝가리안 표기법Hungarian notation
변수 이름 앞에 그 값의 성질을 나타내는 약자 접두사를 붙이는 명명 규칙으로, Charles Simonyi가 고안해 마이크로소프트와 Windows API 문서를 통해 퍼졌습니다. 접두사가 저장 타입을 가리키는 쪽(strDt, iCnt)을 시스템 헝가리안, 값의 의미나 단위를 가리키는 쪽(안전한 문자열인지 아닌지, 좌표가 어느 계인지)을 앱스 헝가리안이라 나눠 부릅니다. 이 글이 유물로 다루는 것은 타입 시스템·IDE와 정보가 이중으로 겹쳐 불일치만 남기는 앞쪽이고, 뒤쪽의 발상은 amountWon처럼 이름에 단위를 새기는 4장의 규칙으로 남아 있습니다.
스코프Scope
어떤 이름이 그 선언을 가리키며 유효하게 참조되는, 소스 코드상의 범위입니다. 대부분의 현대 언어는 렉시컬(정적) 스코프를 써서 실행 중의 호출 순서가 아니라 코드의 중첩 구조만 보고 어느 선언에 묶일지를 결정하며, JavaScript라면 let·const는 블록, var는 함수, export한 이름은 모듈 전체가 그 범위가 됩니다. 값이 메모리에 남아 있는 기간과는 축이 달라서, 클로저가 붙잡은 변수는 이름을 부를 수 있는 범위를 벗어난 뒤에도 값 자체는 계속 살아 있습니다.
유비쿼터스 언어Ubiquitous Language
Eric Evans가 『Domain-Driven Design』(2003)에서 제시한 설계 패턴으로, 도메인 전문가와 개발자가 어휘 하나를 합의한 뒤 회의·문서·코드에서 그 단어를 같은 뜻으로만 쓰는 것을 가리킵니다. 용어집을 따로 두는 데서 끝나지 않고 그 단어가 클래스·메서드 이름이 되어야 하며, 대화에서 단어의 뜻이 바뀌면 코드의 이름도 함께 바뀌는 것이 이 패턴이 작동하는 방식입니다. 다만 통일이 요구되는 범위는 시스템 전체가 아니라 바운디드 컨텍스트(Bounded Context) 하나여서, 맥락이 다르면 같은 fee가 다른 뜻일 수 있고 그 경계에서는 번역을 둡니다. 5장의 "하나의 단어에 하나의 개념"도 한 컨텍스트 안에서의 규칙입니다.

1. 주석이 필요한 이름은 이미 실패한 이름

이름이 좀 짧아도 주석을 달면 되지 않느냐는 말을 종종 듣습니다. 실제로 실무에서 자주 마주치는 코드도 대개 아래와 같은 모양이었습니다.

// Before — 이름이 못 하는 말을 주석이 대신한다
const list = getData(mid);          // mid: 가맹점 ID
let amt = 0;                        // 정산 기준 금액(원)
for (const t of list) {
  if (t.st === 2) amt += t.v;       // 2 = 결제 완료, v = 금액
}

// After — 주석이 하던 말을 이름이 한다
const transactions = findTransactionsByMerchantId(merchantId);
let settlementBaseAmountWon = 0;
for (const tx of transactions) {
  if (tx.status === TransactionStatus.Completed) {
    settlementBaseAmountWon += tx.amountWon;
  }
}

주석보다 이름이 나은 이유는 두 가지로 정리해 두었습니다. 첫째, 주석은 선언 지점에만 남지만 이름은 모든 사용 지점에 따라다닙니다. amt를 선언부에서 30줄 아래에서 만나면 주석을 찾아 다시 올라가야 하는데, settlementBaseAmountWon은 만나는 자리에서 그대로 읽힙니다. 이름은 사용처마다 자동으로 복사되는 주석인 셈입니다. 둘째, 주석은 컴파일러가 검증하지 않습니다. 코드가 바뀌어도 주석은 그 자리에 남아 낡아가고, 어느 시점부터는 사실과 다른 말을 하게 됩니다. 반면 이름은 IDE의 rename 리팩토링으로 모든 사용처가 함께 바뀝니다. 이름은 도구가 관리해 주는 문서이고 주석은 사람이 기억해서 관리해야 하는 문서라, 같은 정보라면 관리 비용이 낮은 쪽에 두는 편이 낫다고 생각하게 됐습니다.

그래서 코드 리뷰에 "여기 주석 좀 달아주세요"라고 쓰려다가, 이 주석이 필요해진 게 이름이 일을 안 하고 있어서는 아닌지 한 번 더 확인하는 습관이 생겼습니다. 확인해 보면 이름을 고치는 쪽으로 끝나는 경우가 많았습니다.

2. 축약어와 헝가리안 — 규칙이 아니라 유물

strDt, arrList, iCnt 같은 이름은 레거시 Java 코드에서 아직도 자주 만납니다. 처음에는 왜 이렇게 썼을까 싶었는데, 알아보니 이런 헝가리안 표기는 한때 합리적인 선택이었습니다. 타입 정보가 약한 언어에서, 선언부를 보려면 종이 출력물을 뒤져야 하던 시절에는 이름에 타입을 새기는 것(헝가리안 표기법)이 실제로 읽는 비용을 줄여 줬습니다. 다만 지금은 그 전제가 남아 있지 않습니다. 타입 시스템이 타입을 보증하고, IDE가 마우스만 올려도 타입을 보여줍니다.

// Before — 타입을 이름에 새기던 시절의 습관
String strDt = ord.getDt();             // 정산 일자… 인데 String?
List arrList = repo.findAll();          // 뭐가 들었는지 이름은 침묵
int iCnt = 0;                           // 무엇의 개수인가

// After — 타입은 타입 시스템에게, 이름은 의미에게
LocalDate settlementDate = order.getSettlementDate();
List<Settlement> pendingSettlements = settlementRepository.findAllPending();
int payoutRetryCount = 0;

전제가 사라지고 나면 표기법에는 비용만 남습니다. strDt를 LocalDate로 바꾸는 순간 이름이 사실과 어긋나고, 같은 정보가 타입과 이름에 이중으로 적혀 있으니 한쪽이 바뀔 때마다 불일치가 생깁니다. 축약도 구조가 같습니다. 축약이 아껴 주는 것은 쓰는 사람의 타이핑인데, 자동완성이 있는 지금은 그 비용이 0에 가깝습니다. 반면 Dt가 date인지 detail인지 추측하는 읽는 비용은 읽을 때마다 매번 발생합니다. 코드는 한 번 쓰이고 수십 번 읽히니, 제가 겪은 범위에서 이 거래는 늘 손해였습니다.

*

실무 팁 — rename은 IDE가 제공하는 리팩토링 중 가장 안전한 축에 듭니다(IntelliJ Shift+F6, VS Code F2). 테스트가 없는 코드에서도 비교적 마음 편히 돌릴 수 있어서, 나쁜 이름을 발견하면 "나중에 크게 고칠 때"로 미루지 않고 그 자리에서 바꿔 두는 편입니다. PR에서 이름을 다듬는 정도는 부담이 적은 작업이었습니다.

3. 불리언 — 조건문이 문장이 되게

불리언 이름은 if 문을 소리 내어 읽었을 때 자연어 문장이 되는지를 기준으로 짓고 있습니다. flag, check, status는 이 기준을 통과하지 못합니다. "만약 플래그라면"은 문장이 되지 않기 때문입니다.

// Before — flag는 아무것도 말하지 않는다
let flag = false;
if (!chkHoliday(dt) && u.state != 0 && !u.noAcct) {
  flag = true;
}

// After — 조건 자체가 문장이 된다
const isBusinessDay    = !isHoliday(settlementDate);
const isActiveMerchant = merchant.status === MerchantStatus.Active;
const hasBankAccount   = merchant.bankAccount !== null;

const canReceivePayout = isBusinessDay && isActiveMerchant && hasBankAccount;

규칙은 세 가지로 정리해 두었습니다. 첫째, 접두사는 is(상태) · has(보유) · can(가능) · should(정책)로 통일합니다. 접두사가 그 불리언의 범주를 먼저 알려주기 때문입니다. 둘째, 긍정형으로 짓습니다. Before의 u.noAcct처럼 부정형으로 지어 두면 !u.noAcct 같은 이중 부정이 따라오고, 이중 부정은 읽는 사람이 머릿속에서 한 번 뒤집어야 하는 인지 비용이 됩니다. hasBankAccount로 뒤집어 두면 부정이 필요한 쪽에서만 !를 쓰면 됩니다. 셋째, 복합 조건에는 중간 이름을 붙입니다. After에서 canReceivePayout 한 줄을 추가한 대가로, 조건이 틀렸을 때 어느 항이 틀렸는지를 디버거 없이 이름 단위로 좁힐 수 있었습니다. NestJS 권한 가드처럼 조건이 곧 정책인 코드에서 특히 도움이 됐습니다.

4. 컬렉션·시간·단위 — 형태가 정해주는 이름

값의 형태마다 오래 쓰여 온 관례가 있습니다. 관례를 따르는 이유는 보기 좋아서라기보다 읽는 사람의 추측을 줄이기 위해서입니다.

// Before — 숫자 3의 단위를 아는 사람은 작성자뿐
const data = getList(mid);
const total = data.map(x => x.amount).reduce((a, b) => a + b, 0);
schedule(total, 3);                      // 3… 일? 시간? 퍼센트?

// After — 복수형·xByY·~At·~Date·단위 접미사
const settlements = findSettlementsByMerchantId(merchantId);
const settlementByOrderId = new Map(settlements.map(s => [s.orderId, s]));
const netAmountWon = sumWon(settlements.map(s => s.netAmountWon));

schedulePayout({
  amountWon:   netAmountWon,
  requestedAt: clock.now(),              // 시각(순간)은 ~At
  payoutDate:  nextBusinessDay(today),   // 날짜는 ~Date
  holdDays:    3,                        // 기간은 단위를 이름에
});

하나씩 근거를 적어 둡니다. 컬렉션은 복수형(settlements)으로 씁니다. list나 data는 "여러 개"라는 것만 말하고 무엇의 여러 개인지는 숨기기 때문입니다. Map은 valueByKey(settlementByOrderId) 형태로 씁니다. 이 관례 하나로 "키가 뭐지?"라는 질문이 사라지고, settlementByOrderId.get(orderId)처럼 사용부가 스스로를 설명하는 문장이 됩니다. 시간은 시각과 기간을 구분합니다. settledAt은 순간(timestamp), settlementDate는 달력상 날짜, holdDays·timeoutMs는 길이입니다. 이 셋을 모두 time이라 부르는 코드에서 시차 버그가 생깁니다. 돈에는 단위와 성격을 함께 붙입니다. 정산 도메인에서 amount 하나만 있는 필드는 수수료 공제 전(grossAmount)인지 후(netAmount)인지에 따라 가맹점에 입금되는 금액이 달라져서, 특히 조심하게 됐습니다.

!

함정 — 단위 없는 돈은 이름만의 문제로 끝나지 않았습니다. 결제 수단마다 요금이 달랐던 정산 버그를 추적할 때, 어떤 코드는 수수료 포함 금액을, 어떤 코드는 공제 후 금액을 같은 amount라는 이름으로 주고받고 있었습니다. 이름을 grossAmountWon / netAmountWon으로 갈라 두는 것만으로도 잘못된 값을 넘기는 코드가 리뷰에서 눈에 띄었습니다. 여기서 한 걸음 더 가면 Money 타입으로 컴파일러에게 감시를 맡길 수 있습니다.

5. 유비쿼터스 언어 — 팀의 사전이 곧 코드

여기까지는 이름 하나를 잘 짓는 규칙이었습니다. 다만 제가 실무에서 겪은 더 큰 문제는 이름들 사이의 불일치였습니다. 서두에 적은 정산 리팩토링이 그런 경우였습니다. 웹 결제·키오스크·카카오페이·월정액·관리자 정산, 다섯 곳이 같은 계산을 각자의 이름으로 갖고 있었고, 이름이 다르니 grep으로도 한 번에 잡히지 않았습니다. 할인 정책이 바뀔 때 다섯 곳 중 몇 곳만 수정되는 일은 이런 구조에서는 반복될 수밖에 없었습니다.

// Before — 같은 개념이 세 가지 이름으로 흩어져 있다
webPayment.calcCharge(order);       // 웹 결제의 "정산"
kiosk.computeFee(order);            // 키오스크의 "정산"
adminBatch.getSettleAmt(order);     // 수동 정산의 "정산"

// After — 용어집의 단어가 그대로 타입과 메서드가 된다
Settlement settlement = settlementPolicy.settle(order);   // 정산: 금액 확정
Payout payout = payoutScheduler.schedule(settlement);     // 지급: 실제 송금

DDD가 말하는 유비쿼터스 언어는 거창한 것이 아니었습니다. 도메인 전문가와 개발자가 합의한 용어집을 만들고, 그 단어를 회의·문서·코드에서 같게 쓰자는 약속입니다. 정산 도메인이라면 정산(Settlement)은 주기 단위로 금액을 확정하는 일, 지급(Payout)은 확정된 금액을 실제로 송금하는 일, 수수료(Fee)는 공제 항목, 보류(Hold)는 지급을 미루는 상태가 됩니다. 이 네 단어를 코드의 클래스명·메서드명과 1:1로 맞춰 두고 나니, 기획자가 "지급 보류 조건을 바꿔주세요"라고 말하는 순간 수정할 클래스가 특정됐습니다. 크롤링 파이프라인을 DDD로 재설계했을 때도 효과의 절반쯤은 아키텍처보다 이 언어 통일에서 온 것 같습니다.

규칙은 대칭 쌍으로 묶어서 기억해 두고 있습니다. 하나의 개념에 하나의 단어(같은 것을 settle/calc/compute로 부르지 않기), 그리고 하나의 단어에 하나의 개념(fee가 어디서는 수수료, 어디서는 요금 전체를 뜻하지 않기)입니다. 앞쪽이 깨지면 검색이 안 되고, 뒤쪽이 깨지면 검색 결과를 믿을 수 없게 됩니다.

6. 규칙이 깨지는 곳 — 짧은 이름이 옳을 때

여기까지 적고 보니 "이름은 무조건 길고 자세하게"로 읽힐 것 같아 한 가지를 덧붙입니다. 앞의 규칙들이 기댄 근거는 이름이 사용처마다 따라다니는 주석이라는 점이었는데, 사용처가 코앞 한 줄뿐이라면 긴 이름의 근거도 함께 약해집니다. 이름 길이는 스코프 길이에 비례하는 편이 맞습니다.

// 이 s는 좋은 이름이다 — 수명이 화살표 함수 한 줄이니까
const totalNetWon = settlements.reduce((sum, s) => sum + s.netAmountWon, 0);

// 오히려 과잉 이름이 신호를 잡음으로 덮는다
const totalNetWon2 = settlements.reduce(
  (accumulatedNetAmountWon, settlementListItem) =>
    accumulatedNetAmountWon + settlementListItem.netAmountWon, 0);

// 같은 한 글자라도 수명이 모듈 전체라면 폭탄이 된다
export let c = loadConfig();   // 6개월 뒤의 c는 아무도 못 읽는다

람다 매개변수의 s나 두 줄짜리 루프의 i는 선언과 사용이 한 시야에 들어오므로 추측할 비용이 없습니다. 이런 자리에 settlementListItem 같은 이름을 붙이면 정보가 늘기보다 패턴 인식이 방해받습니다. 팀 사전에 등록된 관례적 축약인 id, url, Express/NestJS의 req/res도 같은 이유로 쓰고 있습니다. 이미 모두가 아는 단어라면 축약이라기보다 어휘에 가깝습니다.

한 가지 한계도 함께 적어 둡니다. 이름은 "무엇"까지는 말해 주지만 "왜"는 말해 주지 못합니다. holdDays: 3이라는 이름은 3일이 보류 기간이라는 사실은 전하지만, 왜 3일인지 — 카드사 취소 가능 기간 때문인지, 정산 주기 협약 때문인지 — 는 전하지 못합니다. 그 "왜"는 여전히 주석과 커밋 메시지, 설계 문서의 몫이었습니다. 변수명이 주석을 대신하는 범위는 "무엇"까지라고 선을 그어 두는 편이, 저에게는 "주석 무용론"으로 기울지 않는 데 도움이 됐습니다.

7. 정리 — 이름 리뷰 체크리스트

PR을 리뷰할 때 실제로 보는 순서대로 정리해 둡니다.

  • 주석 테스트 — 이 변수에 주석이 필요한지 본다. 필요하다면 이름이 일을 못 하고 있는 쪽이므로, 주석의 내용을 이름으로 옮긴다.
  • 축약·헝가리안 피하기 — 타입 정보는 타입 시스템에 맡긴다. 단, 팀 사전에 등록된 축약(id, url, req/res)은 어휘로 인정한다.
  • 불리언은 문장으로 — is/has/can/should + 긍정형. 이중 부정이 보이면 이름을 뒤집고, 복합 조건에는 중간 이름을 붙인다.
  • 컬렉션은 복수형, Map은 valueByKey — 필터된 결과라면 조건까지 이름에 담는다(pendingSettlements).
  • 시간은 세 갈래 — 시각은 ~At, 날짜는 ~Date, 기간은 단위 접미사(~Days, ~Ms)를 쓴다.
  • 돈에는 성격과 단위 — gross/net/fee를 구분하고 통화 단위를 명시한다. 가능하면 타입으로 올린다.
  • 하나의 개념 = 하나의 단어 — 용어집을 먼저 합의하고, 회의에서 쓰는 단어를 코드에 그대로 쓴다.
  • 이름 길이 ∝ 스코프 길이 — 한 줄 람다의 한 글자는 괜찮고, 모듈 전역의 한 글자는 곤란하다.
  • "왜"는 이름 밖에 — 이름은 "무엇"까지만 맡고, 이유는 주석·커밋·문서에 남긴다.

다음 글에서는 이 이름들이 모여 있는 함수를 다룹니다. 좋은 이름을 붙인 함수가 왜 짧아지는지를 이어서 적어 두려고 합니다.
