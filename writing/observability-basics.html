관측 가능성 — 새벽에 안 깨려면 무엇을 남겨야 하나

2024년 3월• 로그 · 메트릭 · 트레이스

운영을 하다 보면 장애는 납니다. 그래서 제 목표도 장애를 없애는 쪽이 아니라, 터졌을 때 원인을 아는 데 걸리는 시간을 30분에서 3분으로 줄이는 쪽이 되었습니다. 그 시간을 줄여 준 것이 관측 가능성이었고, 남겨 두길 잘했다 싶었던 것들을 정리해 둡니다.

먼저 짚고 가기
관측 가능성Observability
시스템이 밖으로 내보낸 신호(로그·메트릭·트레이스)만으로 내부 상태를 어디까지 추론할 수 있는지를 가리키는 시스템의 성질입니다. 출력값만 보고 내부 상태를 결정할 수 있는가를 따지던 제어이론의 용어를 소프트웨어 운영에 가져다 쓴 것입니다. 미리 정해 둔 실패 조건을 지켜보는 모니터링과 달리, 코드를 새로 배포하지 않고도 처음 받아 보는 질문에 답할 수 있느냐가 기준이 됩니다. 그래서 도구를 도입하면 생기는 것이 아니라, 무엇을 내보내도록 계측해 뒀는지에 달려 있습니다.
메트릭Metric
이름과 라벨(키-값)로 식별되는 계열에 시각과 숫자 값을 시간순으로 쌓아 둔 시계열 데이터입니다. 값의 성격에 따라 누적만 하는 카운터, 오르내리는 게이지, 구간별 개수를 세는 히스토그램 등으로 나눠 수집하고, 일정 주기로 집계해 저장하기 때문에 보관 비용이 요청 수에 비례해 늘지 않습니다. 대신 집계하면서 개별 요청의 정체가 사라지므로 에러율이 올랐다까지는 알아도 어느 요청이 실패했는지는 답하지 못하고, 히스토그램에서 뽑은 p99도 버킷 경계에서 보간한 근사값입니다.
로그Log
프로그램이 실행 중 특정 지점을 지날 때 남기는, 시각이 찍힌 개별 사건 기록입니다. 문자열 한 줄로 남기면 사람이 읽을 수는 있어도 결국 grep으로 뒤지는 수밖에 없고, JSON처럼 키-값 필드로 남기면(구조화 로깅) 필드 단위 검색·필터·집계가 됩니다. 다만 로그에는 코드를 쓸 때 남기기로 정한 값만 담기기 때문에, 그때 찍어 두지 않은 정보는 사후에 복원할 수 없습니다.
트레이스Trace
요청 하나가 거쳐 간 구간들을 스팬(span)으로 쪼개고, 각 스팬에 시작·종료 시각과 부모 스팬 ID를 달아 트리 모양으로 이어 붙인 기록입니다. 같은 trace ID를 서비스 경계 너머로 넘기며(HTTP라면 W3C Trace Context의 traceparent 헤더) 호출 관계를 복원하고, 스팬별 소요 시간을 보고 어느 구간이 오래 걸렸는지 짚습니다. 계측하지 않은 구간은 트레이스에 아예 나타나지 않고, 보통 전량이 아니라 표본만 저장하므로 정작 문제가 된 그 요청이 남아 있지 않을 수도 있습니다.
상관 IDCorrelation / Trace ID
요청이 시스템에 처음 들어오는 지점에서 한 번 발급해, 그 요청이 만들어 내는 모든 로그와 하위 호출에 같이 실어 보내는 식별자입니다. 요청 범위 컨텍스트(자바라면 MDC 같은 자리)에 담아 뒀다가 로그 한 줄마다 자동으로 찍고, 다른 서비스를 부를 때는 헤더에 실어 넘겨 사슬을 잇습니다. 그러면 계층마다 흩어져 있던 로그를 이 값 하나로 검색해 한 요청의 흐름으로 모아 볼 수 있습니다. 컨텍스트가 저절로 따라가지 않는 경계 — 비동기 스레드, 메시지 큐, 배치 — 에서는 직접 복사해 넘기지 않으면 거기서 사슬이 끊깁니다.

세 기둥 — 로그·메트릭·트레이스

  • 메트릭은 무엇이 잘못됐는지를 숫자로 보여 줍니다. 응답시간 p99, 에러율, GC 횟수 같은 값들이고, 대시보드와 알림을 만들 때 재료로 씁니다.
  • 로그는 왜 그랬는지를 담습니다. 다만 검색이 되는 형태로 남겨 두지 않으면 쌓아 놓고도 찾지 못합니다.
  • 트레이스는 어디서 느렸는지를 한 요청의 여정으로 보여 줍니다. 7일마다 죽던 서버를 볼 때도 Scouter APM의 트레이스를 따라가다 시간을 잡아먹는 구간을 찾았습니다.

구조화 로깅과 상관 ID

// 문자열이 아니라 검색 가능한 필드로
log.info({ event: "report.resolve", reportId, userId,
           traceId: ctx.traceId, ms: elapsed });
// traceId 하나로 관리자 Web → API → DB 로그를 꿰어 본다

처음에는 로그를 그냥 문자열로 남겼는데, 나중에 뭔가 찾을 일이 생길 때마다 grep으로 뒤지느라 시간을 많이 썼습니다. JSON 같은 구조화 로그로 바꾸고 나서는 필드 단위로 검색과 집계가 됐습니다. 여기에 요청마다 상관 ID(trace/correlation id)를 부여해 전 계층 로그에 함께 실어 두면, 흩어져 있던 로그를 한 요청의 이야기로 이어 볼 수 있습니다.

알림은 증상이 아니라 영향에

한동안 "CPU 90%" 같은 알림을 걸어 뒀는데, 새벽에 울려서 확인해 보면 별일 아닌 경우가 대부분이었습니다. 그래서 기준을 사용자에게 영향을 주는 지표 쪽으로 옮겼습니다. 에러율 급증, p99 지연, 큐 적체 같은 것들입니다. 이렇게 바꾸고 나니 알림이 울릴 때 실제로 봐야 할 내용이 있는 경우가 늘었습니다.
