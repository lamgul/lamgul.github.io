관측 가능성 — 새벽에 안 깨려면 무엇을 남겨야 하나

2024년 3월• 약 6분• 로그 · 메트릭 · 트레이스

장애는 반드시 납니다. 그러니 목표는 "장애 제로"가 아니라 "터졌을 때 30분이 아니라 3분 만에 원인을 아는 것"입니다. 그걸 가능하게 하는 게 관측 가능성입니다.

먼저 짚고 가기
관측 가능성Observability
시스템이 밖으로 남긴 흔적(로그·숫자·요청 기록)만 보고도 "지금 안에서 무슨 일이 벌어지는지"를 알아낼 수 있는 성질입니다. 자동차 계기판이 엔진을 열어보지 않아도 상태를 알려주는 것과 같습니다. 이 흔적을 잘 남겨 둬야 장애가 났을 때 빠르게 원인을 찾습니다.
메트릭Metric
시간에 따라 변하는 값을 숫자로 잰 것입니다. 에러율, 응답시간, 초당 요청 수처럼요. 체온계 눈금처럼 "무언가 이상하다"를 한눈에 보여 주기 때문에 대시보드와 알림의 재료가 됩니다.
로그Log
프로그램이 언제 무엇을 했는지 시간 순서로 남기는 기록입니다. 일기장에 가깝습니다. 숫자(메트릭)가 "이상하다"고 알려 주면, 로그는 "왜 그랬는지"의 사연을 담습니다.
트레이스Trace
요청 하나가 웹 → API → DB 등 여러 부품을 거치는 전체 여정을 이어 붙인 기록입니다. 택배 배송 추적처럼 어느 구간에서 시간을 잡아먹었는지 짚어 주기 때문에 "어디서 느렸나"를 찾을 때 씁니다.
상관 IDCorrelation / Trace ID
요청 하나마다 붙이는 고유한 꼬리표입니다. 이 꼬리표를 모든 계층 로그에 함께 남기면, 흩어진 로그를 이 번호로 검색해 한 요청의 이야기로 꿰어 볼 수 있습니다.

세 기둥 — 로그·메트릭·트레이스

  • 메트릭은 "무엇이 잘못됐나"를 숫자로 (응답시간 p99, 에러율, GC 횟수). 대시보드와 알림의 재료.
  • 로그는 "왜 그랬나"의 서사. 단, 검색 가능해야 쓸모가 있습니다.
  • 트레이스는 "어디서 느렸나"를 한 요청의 여정으로. 7일마다 죽던 서버도 결국 Scouter APM의 트레이스가 범인을 짚어줬습니다.

구조화 로깅과 상관 ID

// 문자열이 아니라 검색 가능한 필드로
log.info({ event: "report.resolve", reportId, userId,
           traceId: ctx.traceId, ms: elapsed });
// traceId 하나로 관리자 Web → API → DB 로그를 꿰어 본다

로그를 문자열로 남기면 grep의 지옥이 열립니다. JSON 같은 구조화 로그로 남기면 필드로 검색·집계가 됩니다. 그리고 요청마다 상관 ID(trace/correlation id)를 부여해 전 계층 로그에 실으면, 흩어진 로그가 하나의 이야기로 꿰어집니다.

알림은 증상이 아니라 영향에

"CPU 90%" 알림은 새벽에 사람을 깨우기만 하고 대개 아무 일도 아닙니다. 알림은 사용자에게 영향을 주는 것에 걸어야 합니다 — 에러율 급증, p99 지연, 큐 적체. 그래야 알림이 "양치기 소년"이 되지 않고, 울리면 진짜 봐야 하는 신호가 됩니다.
