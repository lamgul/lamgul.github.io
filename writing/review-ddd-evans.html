서평: 도메인 주도 설계 — 유비쿼터스 언어의 힘

2023년 10월• 약 10분• 서평 · DDD · Evans

개발을 하다 보면 어느 시점엔가 모두가 이 파란 책을 폅니다. 저는 엔티티와 애그리거트 같은 당장 쓸 수 있는 코드 패턴이 궁금해서 폈습니다. 다 읽고 나서야 이 책의 진짜 무게가 그 반대쪽 — 언어와 경계 — 에 실려 있다는 걸 알았습니다. 크롤링 파이프라인에 경계를 그으며 다시 읽은 기록입니다.

먼저 짚고 가기
도메인Domain
소프트웨어가 다루려는 현실의 문제 영역입니다. 예컨대 이 글의 '크롤링'이나 '정산'처럼요. 이 책 제목의 '도메인 주도'란, 기술이 아니라 이 문제 영역을 설계의 중심에 둔다는 뜻입니다.
유비쿼터스 언어Ubiquitous Language
기획자·개발자·코드가 같은 단어를 같은 뜻으로 쓰자는 약속입니다. 회의에서 "수집 결과"라 부른 것을 코드에서도 그대로 부르는 것이죠. 이 책의 심장에 해당하는 개념이라 글 곳곳에서 반복됩니다.
엔티티와 값 객체Entity / Value Object
속성이 다 바뀌어도 '같은 것'으로 추적해야 하면 엔티티(사람의 주민번호처럼), 값만 같으면 같은 것으로 봐도 되면 값 객체(1000원짜리 지폐처럼)입니다. 둘을 나누는 기준은 '무엇으로 동일함을 판단하는가'입니다.
애그리거트Aggregate
함께 지켜져야 할 규칙을 공유하는 객체들을 묶은 일관성의 경계입니다. 단순한 '폴더'가 아니라, 한 번의 저장(트랜잭션)에서 반드시 함께 맞아야 하는 것들의 울타리라고 보면 됩니다. 이 글에서 가장 자주 오해받는 패턴으로 소개됩니다.
바운디드 컨텍스트Bounded Context
하나의 모델(용어와 규칙)이 유효하게 통하는 경계입니다. 같은 'Account'라는 단어도 크롤링 팀과 정산 팀에서 다른 뜻이듯, 경계마다 자기 언어를 가집니다. 큰 시스템을 어떻게 쪼갤지에 대한 이 책의 핵심 답입니다.

1. 책의 두 얼굴 — 전술과 전략

에릭 에반스(Eric Evans)의 Domain-Driven Design은 2003년에 나온 책이고, 커뮤니티에서는 표지 색을 따 블루 북이라 부릅니다. 두껍고, 절반쯤은 솔직히 지루합니다. 그런데 이 책을 여러 번 다시 펴게 만드는 힘은 두께가 아니라 구조에 있습니다.

책은 크게 두 얼굴을 가지고 있습니다. 하나는 전술적 설계 — 엔티티, 값 객체, 도메인 서비스, 애그리거트, 리포지토리처럼 코드로 바로 옮겨지는 빌딩 블록입니다. 다른 하나는 책의 마지막을 채우는 전략적 설계 — 바운디드 컨텍스트, 컨텍스트 맵, 핵심 도메인 증류처럼 시스템을 어떻게 쪼갤지에 관한 이야기입니다. 대부분의 독자는 앞쪽 전술 패턴에 끌려서 책을 집고, 뒤쪽 전략 설계에서 진짜를 만납니다.

저도 그랬습니다. 처음엔 "리포지토리는 이렇게 짜는 거구나" 하는 실용 지식을 챙겼고, 몇 년 뒤 크롤링 도메인을 혼자 설계하면서야 이 책이 내내 하려던 말이 언어와 경계였다는 걸 이해했습니다.

*

에반스는 훗날 여러 강연에서, 사람들이 이 책을 빌딩 블록 카탈로그로만 읽고 정작 중요한 유비쿼터스 언어와 전략적 설계를 지나쳤다는 취지로 아쉬움을 말했습니다. 처음 이 책을 폈을 때의 제가 정확히 그 독자였습니다.

2. 유비쿼터스 언어 — 코드에 없으면 없는 말

책의 심장은 유비쿼터스 언어(Ubiquitous Language)입니다. 도메인 전문가와 개발자가 회의에서 쓰는 말, 화이트보드에 그리는 그림, 그리고 코드가 전부 같은 단어를 써야 한다는 원칙입니다. 말은 이렇게 부르고 코드에는 다르게 적는 순간, 번역 비용과 오해가 시스템 곳곳에 스며듭니다.

제가 이 원칙에서 가장 오래 곱씹은 문장은 이겁니다. 어떤 용어가 코드에 없다면, 그건 아직 팀의 언어가 아니다. 회의에서는 "수집 결과"라고 부르면서 코드에는 data, item, result 같은 무색무취한 이름만 있다면, 유비쿼터스 언어는 말로만 존재하는 것입니다.

// 언어가 코드에 없을 때 — 도메인이 사라진다
class DataManager {
  process(item) { /* item? process? 무엇을 처리하나 */ }
  handleResult(r) { /* r 은 무엇의 결과인가 */ }
}

// 유비쿼터스 언어가 코드에 스민 뒤 — 도메인 전문가도 읽는다
class CrawlSchedule {
  enqueue(target: CrawlTarget) { /* 크롤 대상을 대기열에 */ }
  recordFetch(result: FetchResult) { /* 수집 결과를 기록 */ }
}

에반스는 이걸 모델 주도 설계(Model-Driven Design)로 밀어붙입니다. 분석가가 그린 모델과 개발자가 짠 모델이 둘이 아니라 하나여야 한다는 것입니다. SafeIn 백엔드를 혼자 설계할 때 이 원칙이 오히려 더 무거웠습니다. 나와 도메인 전문가가 분리돼 있지 않으니, 내가 대충 붙인 이름이 곧 팀 전체의 언어가 되어버렸기 때문입니다. 그래서 이름 하나를 정하는 데 함수 하나를 짜는 것보다 더 오래 고민하는 습관이 생겼습니다.

3. 엔티티와 값 객체 — 동일성의 문제

전술 패턴 중 실무에서 가장 자주 헷갈리는 건 엔티티(Entity)와 값 객체(Value Object)의 구분입니다. 책의 기준은 단순합니다. 속성이 다 바뀌어도 같은 것으로 추적해야 하면 엔티티, 값이 같으면 같은 것으로 봐도 되면 값 객체입니다. 동일성(identity)을 무엇으로 정하느냐의 문제죠.

이 구분이 중요한 이유는 값 객체에 딸려 오는 규칙 때문입니다. 값 객체는 불변(immutable)으로 만들고, 동등성은 값으로 비교합니다. 그래야 여러 곳에서 공유해도 안전하고, "같은 값인데 다른 객체"라는 미묘한 버그가 사라집니다.

// 값 객체 — 정체성이 없다. 값이 같으면 같은 것이고, 불변이다.
class CrawlTarget {
  constructor(readonly url: string, readonly normalized: string) {}
  equals(o: CrawlTarget) { return this.normalized === o.normalized; }
}

// 엔티티 — 속성이 다 바뀌어도 같은 것. 동일성은 id가 정한다.
class CrawlJob {
  constructor(readonly id: CrawlJobId, private status: JobStatus) {}
  equals(o: CrawlJob) { return this.id.equals(o.id); }
}

정산 리팩토링에서 이 구분은 추상적인 취향 문제가 아니었습니다. 금액은 전형적인 값 객체입니다 — Money(1000, "KRW")는 언제 어디서 만들든 같은 값이면 같아야 하고, 절대 몰래 바뀌면 안 됩니다. 금액을 원시 타입 number로 다루던 코드를 불변 값 객체로 감싸는 것만으로, 통화 단위가 뒤섞이거나 중간에 값이 오염되는 부류의 버그가 통째로 사라졌습니다.

엔티티와 값 객체 어디에도 자연스럽게 들어가지 않는 연산이 있습니다. 여러 객체에 걸쳐 있는 순수한 동작이죠. 책은 이걸 억지로 어느 객체에 욱여넣지 말고 도메인 서비스로 빼라고 합니다. 다만 서비스는 남발하기 쉬워서, 저는 "이 로직이 정말 어떤 엔티티의 책임도 아닌가"를 한 번 더 물은 뒤에야 서비스로 꺼냅니다.

4. 애그리거트 — 가장 오해받는 패턴

애그리거트(Aggregate)는 이 책에서 가장 자주 읽히면서 가장 자주 오해받는 패턴입니다. 흔한 오해는 "관련 있는 객체를 묶어두는 폴더" 정도로 이해하는 것입니다. 그렇게 보면 무엇을 묶고 무엇을 뗄지 기준이 없어집니다.

책의 정의에서 실무적으로 중요한 부분은 묶음이 아니라 일관성 경계입니다. 애그리거트에는 루트가 되는 엔티티가 하나 있고, 바깥에서는 그 루트를 통해서만 내부에 접근하며, 하나의 트랜잭션이 커밋되는 순간 지켜져야 할 불변식이 그 경계 안에 담깁니다. 다른 애그리거트는 객체가 아니라 ID로 참조하라는 규칙도 여기서 나옵니다.

이 패턴을 크롤링 파이프라인에 실제로 적용하면서, 저는 "어디까지가 하나의 애그리거트인가"라는 질문 하나에 재설계의 절반을 썼습니다. 경계를 잘못 그으면 코드가 아니라 트랜잭션과 잠금이 무너진다는 걸 그때 배웠습니다. 그 결정 과정 — 잘못 그은 선과 다시 그은 선 — 은 이 서평의 범위를 넘어서니, 아래 별도 글에 따로 정리해 두었습니다.

5. 바운디드 컨텍스트 — 진짜 어려운 절반

전술 패턴이 코드 안의 이야기라면, 바운디드 컨텍스트(Bounded Context)는 코드 바깥, 시스템을 어떻게 쪼갤지에 관한 이야기입니다. 그리고 이쪽이 훨씬 어렵습니다. 핵심 통찰은 이것입니다 — 하나의 모델은 특정 경계 안에서만 유효하다. 온 회사에 통하는 단 하나의 완벽한 모델 같은 건 없습니다.

같은 단어라도 경계가 다르면 다른 것을 가리킵니다. 크롤링 컨텍스트에서 Account는 "대상 사이트에 로그인하는 자격"이고, 정산 컨텍스트에서 Account는 "고객의 청구 계정"입니다. 이 둘을 하나의 Account 클래스로 통일하려는 유혹이야말로 대형 시스템을 진창으로 만드는 지름길입니다.

# 크롤링 컨텍스트 — Account 는 '대상 사이트 로그인 자격'
class Account:            # module: crawling
    host: str
    cookie_jar: CookieJar
    logged_in_at: datetime

# 정산 컨텍스트 — 같은 단어, 전혀 다른 모델
class Account:            # module: billing
    customer_id: CustomerId
    balance: Money
    currency: str

바운디드 컨텍스트는 각 컨텍스트가 자기 언어의 주인이 되게 합니다. 크롤링 쪽 Account가 청구 로직을 신경 쓸 필요가 없고, 정산 쪽 Account가 쿠키를 알 필요가 없습니다. 저는 크롤링·정산·모델 서빙을 서로 다른 컨텍스트로 못 박고 나서야, 온프레미스 vLLM 서빙이 도메인 로직과 뒤엉키지 않도록 선을 그을 수 있었습니다.

*

컨텍스트 경계를 찾는 가장 실용적인 신호는 언어가 깨지는 지점입니다. 같은 단어가 팀마다 다른 뜻으로 쓰이거나, 한 용어에 자꾸 수식어를 붙여야("정산 쪽 계정", "크롤링 쪽 계정") 뜻이 통한다면, 거기가 바운디드 컨텍스트의 이음매입니다. 저는 클래스 다이어그램보다 회의실의 말버릇에서 경계를 먼저 봤습니다.

6. 컨텍스트 맵과 부패 방지 계층

컨텍스트를 나눴다면, 그다음은 그 사이의 관계를 솔직하게 그리는 일입니다. 책은 이걸 컨텍스트 맵(Context Map)이라 부릅니다. 컨텍스트들이 서로 어떻게 엮이는지 — 공유 커널, 고객·공급자, 준수자, 오픈 호스트 서비스, 발행된 언어 등 — 을 이름 붙여 드러내는 것입니다. 중요한 건 그림의 정교함이 아니라, 관계가 존재한다는 사실을 인정하고 이름 붙이는 것입니다.

이 지도에서 제가 실무적으로 가장 많이 쓴 패턴은 부패 방지 계층(Anticorruption Layer)입니다. 외부 시스템이나 레거시의 지저분한 모델이 내 도메인 언어로 그대로 새어 들어오지 못하게, 경계에 번역 계층을 세우는 것입니다. 크롤링에서 사이트마다 제각각인 응답을 우리 도메인 용어로 옮기는 어댑터가 정확히 이 역할을 합니다.

# anticorruption_layer.py — 외부 사이트의 언어를 우리 도메인 언어로 번역
class SiteBGateway:
    def to_fetch_result(self, raw: dict) -> FetchResult:
        # 저쪽의 'st' · 'ts' · 'body' 를 우리 도메인 용어로 옮긴다
        return FetchResult(
            status=self._map_status(raw["st"]),
            fetched_at=parse_epoch(raw["ts"]),
            content=raw["body"],
        )

대흥정보에서 레거시 시스템을 유지보수하고 웹을 개발할 때, 저는 이 패턴의 이름을 몰랐을 뿐 이미 그 일을 하고 있었습니다. 손대기 어려운 외부 모델과 우리가 지키고 싶은 모델 사이에 얇은 번역 층을 두는 것 — 책은 제가 감으로 하던 방어에 부패 방지 계층이라는 이름과 근거를 줬습니다. 이름이 생기자 그 층을 언제 세우고 언제 걷어낼지도 팀과 논의할 수 있게 됐습니다.

7. 정리 — 언어가 먼저다

이 책을 다시 읽고 제 우선순위는 뒤집혔습니다. 처음엔 전술 패턴이 본론이고 전략 설계는 부록처럼 보였는데, 지금은 반대입니다. 바운디드 컨텍스트와 유비쿼터스 언어가 본론이고, 엔티티·값 객체·애그리거트는 그 언어를 코드로 옮기는 도구입니다. 경계와 언어를 먼저 정하지 않은 채 전술 패턴부터 꺼내면, 잘 만든 엔티티들이 잘못된 자리에 놓입니다.

2003년 책이라 낡은 구석도 있습니다. 리포지토리나 팩토리를 둘러싼 무거운 격식은 요즘 언어와 프레임워크에서는 덜어내도 됩니다. 하지만 도메인의 언어를 코드에 새기고, 모델이 유효한 경계를 긋는다는 두 축은 20여 년이 지나도 낡지 않았습니다. 오히려 마이크로서비스 시대에 더 절실해졌습니다.

과거의 저에게 한마디 남긴다면 이렇게 말하겠습니다. 이 책에서 패턴 목록을 베끼려 하지 마라. 도메인 전문가와 같은 단어를 쓰는 법과 모델이 어디까지 유효한지 아는 법 — 이 둘을 가져가라. 나머지 코드 패턴은 그 두 가지가 자리 잡은 뒤에야 제자리를 찾습니다. 애그리거트 경계를 크롤링 파이프라인에 실제로 어떻게 그었는지는 아래 글에 이어집니다.
