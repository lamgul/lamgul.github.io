보안: 레이트 리미팅 — 미들웨어로 그은 방어선

2024년 6월• Rate Limiting · Redis · 미들웨어

온프레미스로 vLLM 추론 서버를 세우고 나서 가장 먼저 한 일은 기능 추가가 아니라 한도를 긋는 것이었습니다. GPU 한 장으로 버티는 엔드포인트에서, 한 클라이언트가 쉬지 않고 두드리면 나머지 모두가 굶습니다. 레이트 리미팅은 그 방어선을 요청이 비싼 자원에 닿기 전에, 미들웨어에 긋는 일입니다. 이 글은 알고리즘을 고르고, Redis로 분산 환경에서 원자적으로 세는 데까지의 기록입니다 — 원칙의 나열이 아니라 왜 그 선택인지와 어디서 무너지는지까지.

먼저 짚고 가기
레이트 리미팅Rate Limiting
정해진 시간 동안 한 사용자가 보낼 수 있는 요청 수에 상한을 두는 것입니다. 놀이공원에서 "1인당 하루 3회 탑승"처럼 횟수 제한을 거는 셈이죠. 이 글이 다루는 방어선 전체가 바로 이 한도를 어떻게 정확히 세느냐의 이야기입니다.
미들웨어Middleware
요청이 실제 처리 코드(핸들러)에 도착하기 전에 먼저 거쳐 가는 중간 검문소입니다. 건물 입구의 경비실처럼, 안쪽 사무실까지 들이기 전에 여기서 걸러 냅니다. 리미터를 여기 두어야 비싼 작업(DB·GPU)에 닿기 전에 막을 수 있습니다.
원자적 연산Atomic Operation
여러 단계를 "전부 실행되거나 전혀 안 되거나"로 묶어, 도중에 다른 작업이 끼어들 수 없게 만든 연산입니다. 값을 읽고 고쳐 쓰는 사이 다른 요청이 끼어들면 한도가 새는데, 원자성이 그 틈을 막아 줍니다.
Redis
데이터를 디스크가 아닌 메모리에 두어 아주 빠르게 읽고 쓰는 저장소입니다. 서버가 여러 대여도 모두 이 한 곳을 함께 보게 하면, 카운터를 한군데에 모아 분산 환경에서도 일관되게 셀 수 있습니다.
TTLTime To Live
저장한 값이 자동으로 사라지기까지의 수명입니다. "1분 뒤 폐기"라고 붙여 둔 유통기한 스티커라고 보면 됩니다. 창이 지난 카운터를 알아서 지워 주므로, 리미터가 오래된 기록을 직접 청소할 필요가 없습니다.

무엇을 막고, 어디에 긋는가

레이트 리미팅을 "트래픽 조절" 정도로만 보면 절반을 놓칩니다. 이건 보안 통제입니다. 로그인 폼에 초당 수백 번 비밀번호를 밀어 넣는 무차별 대입, 유출된 계정 목록을 자동으로 대입해 보는 크리덴셜 스터핑, 남의 데이터를 통째로 긁어 가는 스크래핑 — 이들의 공통점은 하나입니다. 한 번은 무해하지만 무한 반복이면 공격이 된다는 것. 한도는 그 반복 자체에 값을 매기는 장치입니다.

그다음 질문은 "어디에 긋느냐"입니다. 저는 가능한 한 바깥, 요청이 비싼 일을 시작하기 전이라는 원칙을 씁니다. 인증을 통과시키고, DB를 조회하고, GPU에 프롬프트를 밀어 넣은 뒤에 "당신은 한도 초과입니다"라고 말하면, 막으려던 그 비용을 이미 다 치른 뒤입니다. 그래서 리미터는 애플리케이션의 미들웨어 계층, 핸들러보다 앞에 있어야 합니다.

키를 무엇으로 잡느냐가 남습니다. 정답은 엔드포인트의 성격이 정합니다. 인증 전 구간(로그인, 회원가입)은 IP 기준으로 세야 합니다. 아직 사용자가 없으니까요. 인증 후 구간은 사용자·API 키 기준이라야 합니다. NAT 뒤 수천 명이 한 IP를 공유하는 상황에서 IP로 한도를 걸면 애먼 사람이 막히니까요. 이 구분은 곧 미들웨어의 실행 순서 문제로 이어지는데, 뒤의 NestJS 절에서 다시 짚습니다.

고정 윈도우와 경계 폭주

가장 단순한 알고리즘부터 봅니다. 고정 윈도우 카운터는 시간을 1분 같은 고정 창으로 자르고, 창마다 카운터를 하나 둡니다. Redis라면 창을 키에 박아 넣고 INCR로 세면 끝입니다.

// 고정 윈도우 — 1분에 100회. 창은 분 단위로 갈린다
async function fixedWindow(id: string, limit = 100, windowSec = 60) {
  const slot = Math.floor(Date.now() / 1000 / windowSec);
  const key  = `rl:${id}:${slot}`;
  const count = await redis.incr(key);            // INCR 자체는 원자적
  if (count === 1) await redis.expire(key, windowSec); // 첫 요청에만 TTL
  return count <= limit;
}

INCR은 원자적이라 카운트 자체는 경쟁 상태에 안전합니다. 값싸고 메모리도 카운터 하나뿐이라 매력적이죠. 하지만 경계에서 폭주합니다. 창이 분 단위로 갈리므로, 00:59.9에 100회를 채우고 01:00.1에 다시 100회를 채우면 — 각 창의 한도는 지켰지만 0.2초 사이에 200회가 통과합니다. 한도를 "분당 100"이라 적어 놓고 실제로는 순간 200을 허용하는 겁니다. 무차별 대입을 막으려던 방어선이 창 경계에서 두 배로 벌어집니다.

!

더 조용한 함정 — TTL 없는 카운터. 위 코드는 INCR 다음에 EXPIRE를 겁니다. 만약 프로세스가 그 두 명령 사이에서 죽으면, 그 키는 값이 1인 채로 영영 만료되지 않습니다. 다음 창이 와도 카운터가 리셋되지 않아, 해당 사용자는 원인 모를 영구 차단에 걸립니다. 한도 로직이 여러 명령으로 쪼개지는 순간 이런 틈이 생깁니다. 결론은 하나 — 세는 일은 "한 번에" 끝나야 한다. 뒤에서 Lua로 이 틈을 닫습니다.

슬라이딩 윈도우 — 로그와 카운터

경계 폭주의 원인은 창이 고정돼 있다는 데 있습니다. 그러니 창을 요청 시점 기준으로 미끄러뜨리면 됩니다. 가장 정확한 방식이 슬라이딩 윈도우 로그입니다. 요청 타임스탬프를 정렬 집합(ZSET)에 쌓아 두고, "지금부터 1분 전"보다 오래된 것을 버린 뒤 남은 개수를 셉니다.

// 슬라이딩 윈도우 로그 — 타임스탬프를 정렬 집합에 쌓는다
const now = Date.now();
const key = `rl:log:${id}`;
const res = await redis.multi()                      // 네 명령을 한 트랜잭션으로
  .zremrangebyscore(key, 0, now - windowMs)          // 창 밖 기록 제거
  .zadd(key, now, `${now}:${nanoid()}`)             // 이번 요청 기록
  .zcard(key)                                         // 창 안 요청 수
  .pexpire(key, windowMs)
  .exec();
const count = res[2][1];                            // zcard 결과
return count <= limit;

정확합니다. 경계도 없습니다. 어느 순간에 재도 직전 1분의 실제 요청 수를 보니까요. 대가는 메모리입니다. 요청 하나마다 멤버 하나가 쌓입니다. 게다가 위 코드는 검사 전에 zadd부터 하므로, 폭주 공격이 들어오면 거절할 요청까지 집합에 기록돼 메모리가 부풀죠. 한도가 100인데 초당 만 번을 맞으면, 정작 막는 데 쓰는 자원이 공격의 먹잇감이 됩니다.

그 절충이 슬라이딩 윈도우 카운터입니다. 타임스탬프를 다 저장하는 대신, 현재 창과 직전 창의 카운터 두 개만 들고, 직전 창을 겹친 비율만큼 빌려 씁니다 — 대략 현재 창 수 + 직전 창 수 × 겹침 비율. 메모리는 카운터 두 개로 고정이고, 경계 폭주는 눌러집니다. 대신 직전 창의 요청이 고르게 분포했다고 가정하는 근사라, 완벽하진 않습니다. 이 근사가 실제로 얼마나 쓸 만한지는 뒤의 책 이야기에서 숫자로 다시 봅니다.

토큰 버킷과 Lua 원자성

지금까지는 "몇 번 왔나"를 셌습니다. 토큰 버킷은 관점이 다릅니다. 용량이 정해진 버킷에 토큰이 일정 속도로 채워지고, 요청은 토큰을 하나씩 씁니다. 토큰이 있으면 통과, 없으면 거절. 이 방식의 진짜 미덕은 버스트를 우아하게 허용한다는 데 있습니다. 평소 놀다가 잠깐 몰아 쓰는 사용자는 쌓인 토큰으로 통과시키고, 지속적인 남용만 막습니다. 사람이 화면을 클릭하는 패턴은 원래 몰렸다 뜸했다 하니, 사용자 대면 API에는 이 관용이 딱 맞습니다. vLLM 앞에 이걸 고른 이유이기도 합니다.

그런데 토큰 버킷은 INCR 하나로 안 끝납니다. 토큰을 읽고 → 흐른 시간만큼 채우고 → 검사하고 → 차감해 다시 쓰는 읽기-수정-쓰기입니다. 이걸 애플리케이션 코드에서 GET 후 SET으로 하면, 두 요청이 동시에 같은 잔량을 읽고 둘 다 통과시켜 버립니다. 한도가 새는 겁니다. 여러 대의 API 서버가 한 Redis를 볼 때 이 경쟁은 반드시 터집니다. 해법은 이 전체를 하나의 원자적 연산으로 묶는 것 — Redis에서는 Lua 스크립트가 그 역할을 합니다.

-- token_bucket.lua  |  KEYS[1]=버킷,  ARGV=capacity, refill, now, cost
local tokens = tonumber(redis.call('HGET', KEYS[1], 'tokens'))
local ts     = tonumber(redis.call('HGET', KEYS[1], 'ts'))
local cap    = tonumber(ARGV[1])
local refill = tonumber(ARGV[2])          -- 초당 채워지는 토큰
local now    = tonumber(ARGV[3])
local cost   = tonumber(ARGV[4])

if tokens == nil then tokens = cap; ts = now end
tokens = math.min(cap, tokens + (now - ts) * refill)  -- 흐른 시간만큼 지연 충전

local ok = tokens >= cost
if ok then tokens = tokens - cost end

redis.call('HSET', KEYS[1], 'tokens', tokens, 'ts', now)
redis.call('EXPIRE', KEYS[1], math.ceil(cap / refill) * 2)  -- 노는 버킷은 자동 소멸
return {ok and 1 or 0, tokens}

두 가지가 여기서 해결됩니다. 첫째, 지연 충전(lazy refill) — 타이머를 돌려 토큰을 채우지 않고, 마지막으로 만진 시각과의 차이로 "그동안 채워졌어야 할 양"을 그때그때 계산합니다. 버킷이 백만 개여도 백그라운드 작업이 필요 없습니다. 둘째, 앞 절 콜아웃의 TTL 틈도 닫힙니다. 읽기·검사·차감·EXPIRE가 한 스크립트 안에 있으니, 중간에 끊길 자리가 없습니다.

*

왜 Lua가 원자적인가. Redis는 명령을 단일 스레드로 처리하고, EVAL로 넘긴 스크립트는 끝까지 실행되는 동안 다른 명령을 끼워 넣지 않습니다. 그래서 스크립트 안의 읽기-수정-쓰기는 절대 인터리빙되지 않죠. 단, 바로 그 이유로 스크립트는 짧아야 합니다 — 실행되는 동안 서버 전체가 멈추니까요. 그리고 클러스터에서 옳게 라우팅되려면 만지는 키를 반드시 KEYS[]로 넘겨야 합니다. 하드코딩한 키 이름은 슬롯 계산을 건너뜁니다.

NestJS 가드로 얹고 429를 돌려준다

이제 리미터를 요청 경로에 얹을 차례입니다. NestJS에는 걸 곳이 셋 있는데, 셋의 생애주기 위치가 다릅니다. 미들웨어는 가장 먼저, 가드보다도 앞에서 돕니다. 가드는 "이 요청을 통과시킬까"를 결정하고, 인터셉터는 핸들러 실행을 감쌉니다. 레이트 리미팅의 의미는 정확히 "통과 여부 결정"이라 가드가 가장 잘 맞습니다. 인터셉터는 핸들러를 이미 감싼 뒤라 늦고, 미들웨어는 인증 컨텍스트(req.user)를 아직 못 볼 수 있습니다.

// 429의 출구를 한곳으로 — 가드가 접근 여부를 결정한다
@Injectable()
export class RateLimitGuard implements CanActivate {
  constructor(private readonly limiter: TokenBucket, private readonly metrics: Metrics) {}

  async canActivate(ctx: ExecutionContext): Promise<boolean> {
    const req = ctx.switchToHttp().getRequest<Request>();
    const res = ctx.switchToHttp().getResponse<Response>();
    const id  = req.user?.id ?? req.ip;          // 인증됐으면 사용자, 아니면 IP

    const { allowed, remaining, resetSec } = await this.limiter.take(id);
    res.setHeader('X-RateLimit-Limit', this.limiter.capacity);
    res.setHeader('X-RateLimit-Remaining', Math.max(0, Math.floor(remaining)));
    if (allowed) return true;

    res.setHeader('Retry-After', resetSec);          // '언제 다시'를 알려준다
    this.metrics.increment('ratelimit.blocked', { route: req.path });
    throw new HttpException('Too Many Requests', 429);
  }
}

거절은 반드시 429 Too Many Requests이고, Retry-After로 "언제 다시 오라"를 알려 줘야 합니다. 이건 예의가 아니라 계약입니다. 크롤러를 DDD로 설계하며 반대편에 서 본 입장에서 뼈저리게 배운 건데, 상대가 Retry-After를 안 주면 클라이언트는 언제 재시도할지 몰라 더 자주 두드립니다. 좋은 리미터는 막으면서 동시에 물러설 방법을 알려 줍니다. 이 429를 받은 클라이언트가 어떻게 지수 백오프로 물러서야 하는지는 백오프 글에서 다뤘습니다.

마지막으로 ratelimit.blocked 메트릭을 빠뜨리지 마세요. 429가 갑자기 치솟는 건 그 자체가 신호입니다 — 공격이 시작됐거나, 배포한 클라이언트가 버그로 우리를 두드리고 있거나. 둘 다 사람이 알아야 할 일이죠. 리미터는 방어이자 센서입니다. 이 신호를 어떻게 알림으로 연결하는지는 관측 가능성 글의 "증상이 아니라 영향에 알림을 건다"와 이어집니다.

책 한 권 — "레이트 리미터를 설계하라"

이 주제를 처음 메뉴판처럼 정리해 준 건 알렉스 쉬(Alex Xu)의 『System Design Interview』 1권, "레이트 리미터를 설계하라(Design a Rate Limiter)" 장이었습니다. 이 책의 모든 장은 같은 4단계 틀을 따릅니다 — 문제와 설계 범위를 확정하고 → 개략 설계를 제시하고 → 심화하고 → 마무리하는 순서. 레이트 리미터 장의 진짜 값어치는 코드가 아니라 이 질문의 순서에 있었습니다.

책은 다섯 알고리즘을 나란히 놓고 장단을 따집니다 — 토큰 버킷, 리키 버킷, 고정 윈도우 카운터, 슬라이딩 윈도우 로그, 슬라이딩 윈도우 카운터. 토큰 버킷은 아마존과 스트라이프가 API 스로틀링에 쓰는 예로, 리키 버킷은 쇼피파이의 예로 듭니다. 고정 윈도우의 경계 폭주, 슬라이딩 로그의 메모리 비용, 슬라이딩 카운터의 근사 — 이 글의 뼈대가 사실 이 장의 뼈대입니다. 카운터의 근사가 얼마나 정확하냐를 두고 책은 클라우드플레어의 분석을 인용해 극히 일부 요청에서만 오차가 난다고 짚는데, 그 근사 공식은 코드로 옮기면 이렇게 단순합니다.

// 책의 슬라이딩 윈도우 카운터 — 이전 창을 '겹친 비율'만큼 빌려 쓴다
const elapsed  = (now % windowMs) / windowMs;   // 현재 창에서 흐른 비율
const estimate = curr + prev * (1 - elapsed);
// 예) 한도 7, 이전 창 5, 현재 창 3, 창의 30% 지점
//     → 3 + 5 × 0.7 = 6.5 → 6, 한도 이하이므로 통과
return estimate <= limit;

하지만 실무에 옮길 때 가장 크게 남은 건 알고리즘이 아니라 분산 환경 절이었습니다. 책은 여러 대의 리미터가 생기는 순간 두 문제가 따라온다고 못 박습니다 — 경쟁 상태와 동기화. 카운터를 읽고 쓰는 사이 다른 요청이 끼어드는 경쟁은 Lua 스크립트나 정렬 집합으로 풀고, 여러 서버가 상태를 공유하려면 스티키 세션이 아니라 중앙 저장소(Redis)를 둬야 한다는 것. 앞 절에서 제가 토큰 버킷의 읽기-수정-쓰기를 통째로 Lua에 밀어 넣은 건, 책이 "여기서 조용히 깨진다"고 표시해 둔 바로 그 지점을 막은 것이었습니다.

세부까지 실무 감각에 붙는 책입니다. 429 응답과 함께 X-Ratelimit-Remaining·X-Ratelimit-Limit·X-Ratelimit-Retry-After 같은 헤더로 남은 한도와 재시도 시점을 돌려주라는 계약, Redis의 INCR·EXPIRE로 카운터를 구현하는 밑그림까지 짚어 줍니다. 인터뷰 대비서라 깊이는 넓고 얕지만, 결정을 어떤 순서로 내려야 하는지를 몸에 새겨 준다는 점에서, 실제로 리미터를 얹기 전에 한 번 통과하기 좋은 장이었습니다.

정리 — 방어선을 어디에 긋나

레이트 리미팅은 라이브러리를 하나 붙이는 일처럼 보이지만, 실은 일련의 결정입니다. 그 결정들을 순서대로 추리면 이렇습니다.

  • 바깥에, 비싼 일 앞에 긋는다. 인증·DB·GPU에 닿기 전 미들웨어 계층에서 막아야, 막으려던 비용을 안 치른다.
  • 키는 엔드포인트가 정한다. 인증 전은 IP, 인증 후는 사용자·API 키. NAT 뒤 다수가 한 IP를 공유하는 걸 잊지 않는다.
  • 고정 윈도우는 경계에서 두 배로 샌다. 값싸지만, 창 경계의 순간 폭주를 감수할 수 있을 때만.
  • 정확함이 필요하면 슬라이딩 로그, 절충이면 카운터. 로그는 메모리를, 카운터는 근사를 대가로 낸다.
  • 사용자 대면 API엔 토큰 버킷. 버스트를 관용하고 지속적 남용만 막는 게 실제 트래픽 모양에 맞는다.
  • 세는 일은 한 번에 — Lua로 원자화한다. 읽기-수정-쓰기가 쪼개지면 한도가 새고, TTL 틈으로 영구 차단까지 난다.
  • 거절은 429 + Retry-After + 남은 한도 헤더. 막되, 언제 물러서야 하는지를 알려 주는 게 계약이다.
  • 429 급증은 알림거리다. 리미터는 방어선이자 센서 — 막은 횟수를 반드시 메트릭으로 남긴다.

결국 한도를 정하는 건 기술이 아니라 요구사항입니다. "이 엔드포인트는 얼마나 두드려도 되는가"를 도메인에 먼저 묻지 않으면, 어떤 알고리즘을 골라도 숫자는 감으로 찍은 값이 됩니다. 알고리즘은 그 답을 새지 않게 집행하는 방법일 뿐입니다.
