보안: 레이트 리미팅 — 미들웨어로 그은 방어선

2024년 6월• Rate Limiting · Redis · 미들웨어

온프레미스로 vLLM 추론 서버를 세우고 나서, 기능을 더 붙이기 전에 한도부터 걸었습니다. GPU 한 장으로 버티는 엔드포인트라 한 클라이언트가 쉬지 않고 두드리면 나머지 요청이 전부 밀리는 구조였습니다. 그래서 요청이 비싼 자원에 닿기 전, 미들웨어 자리에서 먼저 세기로 했습니다. 알고리즘을 고르고 Redis로 분산 환경에서 원자적으로 세기까지 어떤 선택을 했고 그 선택이 어디서 잘 깨지는지를 잊지 않으려고 적어 둡니다.

먼저 짚고 가기
레이트 리미팅Rate Limiting
요청을 특정 키(IP·사용자·API 키)로 묶어 단위 시간당 허용 횟수의 상한을 정하고, 그 상한을 넘는 요청을 거절하는 접근 통제 기법입니다. 개별 요청의 내용이 아니라 반복 빈도를 기준으로 삼기 때문에, 정상 요청과 악성 요청을 한 건씩 떼어 놓고는 구분하지 못하고 오직 누적 횟수로만 남용을 걸러 냅니다. 상한을 어떤 키로 어느 창(window)에서 세느냐가 곧 이 기법의 설계 전부입니다.
미들웨어Middleware
요청이 라우트 핸들러에 닿기 전에 파이프라인에서 먼저 실행되는 함수 계층입니다. 요청·응답 객체와 다음 단계로 넘기는 next 함수를 받아, 요청을 검사하거나 고치거나 여기서 응답을 돌려주고 체인을 끊어 버릴 수 있습니다. 다만 NestJS에서는 미들웨어가 가드보다 앞서 돌아 인증 컨텍스트(req.user)가 아직 채워지지 않은 경우가 있어, 이 글에서는 사용자 식별이 필요한 리미터를 미들웨어 대신 가드에 두었습니다.
원자적 연산Atomic Operation
다른 연산의 관점에서 더 쪼갤 수 없는 단위로 실행되는 성질입니다. 값을 읽고 고쳐 쓰는 중간 상태를 다른 연산이 관찰하지도, 그 사이에 끼어들지도 못합니다(비인터리빙). Redis는 명령을 단일 스레드로 처리하므로 INCR 한 명령이나 EVAL로 넘긴 Lua 스크립트 하나가 이 단위가 되어, 두 요청이 같은 잔량을 읽어 둘 다 통과시키는 틈을 막습니다. 다만 여기서의 원자성은 '중간에 끼어들지 않음'을 뜻할 뿐, 스크립트가 도중에 실패해도 앞서 끝낸 쓰기를 되돌리는 롤백까지 보장하지는 않습니다.
Redis
데이터를 디스크가 아닌 메모리에 두고 명령을 단일 스레드로 처리하는 인메모리 키-값 저장소입니다. 단순 문자열뿐 아니라 해시(HASH)·정렬 집합(ZSET) 같은 자료구조와 EVAL로 실행하는 서버 측 Lua 스크립팅을 제공해서, 이 글의 토큰 버킷(HASH)과 슬라이딩 윈도우 로그(ZSET)가 모두 여기에 얹힙니다. 여러 API 서버가 이 한 곳을 함께 바라보게 하면 카운터를 중앙에 모아 분산 환경에서도 일관되게 셀 수 있고, 대신 전체 용량이 물리 메모리에 묶이며 지속성은 별도 설정(RDB·AOF)에 달려 있습니다.
TTLTime To Live
키에 걸어 두는 남은 수명으로, EXPIRE·PEXPIRE로 설정하면 그 시간이 지난 뒤 키가 자동 삭제됩니다. 삭제는 만료 순간에 즉시 일어나기보다 키에 접근할 때(수동)와 주기적으로 표본을 훑는 백그라운드 정리(능동)로 처리됩니다. 리미터는 이걸로 지난 창의 카운터를 스스로 청소하지 않아도 되지만, TTL을 INCR과 별개 명령으로 걸다 프로세스가 그 사이에서 죽으면 수명 없는 키가 남는데, 본문 콜아웃의 영구 차단이 바로 그 틈에서 생깁니다.

무엇을 막고, 어디에 긋는가

처음에는 레이트 리미팅을 트래픽 조절 장치 정도로 생각했는데, 실제로 걸어 보니 보안 통제에 가까웠습니다. 로그인 폼에 초당 수백 번 비밀번호를 밀어 넣는 무차별 대입, 유출된 계정 목록을 자동으로 대입해 보는 크리덴셜 스터핑, 남의 데이터를 통째로 긁어 가는 스크래핑은 요청 하나만 놓고 보면 정상 트래픽과 구분되지 않습니다. 차이는 반복 횟수에서만 드러나기 때문에, 한도를 걸어 그 반복에 비용을 매기는 편이 실질적인 방어가 됩니다.

그다음에 정할 것은 어디에 긋느냐였습니다. 저는 요청이 비싼 일을 시작하기 전, 되도록 바깥쪽에 두는 쪽을 택했습니다. 인증을 통과시키고 DB를 조회하고 GPU에 프롬프트까지 밀어 넣은 다음에 한도 초과라고 답하면, 막으려던 비용은 이미 다 치른 뒤가 되기 때문입니다. 그래서 리미터는 핸들러보다 앞, 미들웨어 계층에 두었습니다.

남은 것은 무엇을 키로 삼느냐였는데, 이건 엔드포인트마다 달랐습니다. 로그인이나 회원가입처럼 인증 전 구간은 아직 사용자가 없으니 IP 기준으로 셀 수밖에 없었고, 인증 후 구간은 사용자나 API 키를 키로 잡았습니다. NAT 뒤에서 수천 명이 한 IP를 공유하는 경우 IP로 한도를 걸면 관계없는 사람까지 같이 막히기 때문입니다. 이 구분은 미들웨어의 실행 순서 문제로도 이어지는데, 그 이야기는 뒤의 NestJS 절에 적었습니다.

고정 윈도우와 경계 폭주

가장 단순한 것부터 살펴봤습니다. 고정 윈도우 카운터는 시간을 1분 같은 고정 창으로 자르고 창마다 카운터를 하나 두는 방식입니다. Redis에서는 창 번호를 키에 넣고 INCR로 세면 됩니다.

// 고정 윈도우 — 1분에 100회. 창은 분 단위로 갈린다
async function fixedWindow(id: string, limit = 100, windowSec = 60) {
  const slot = Math.floor(Date.now() / 1000 / windowSec);
  const key  = `rl:${id}:${slot}`;
  const count = await redis.incr(key);            // INCR 자체는 원자적
  if (count === 1) await redis.expire(key, windowSec); // 첫 요청에만 TTL
  return count <= limit;
}

INCR은 원자적이라 카운트 자체는 경쟁 상태에 안전하고, 메모리도 카운터 하나면 되니 비용이 거의 들지 않습니다. 다만 창 경계에서 한도가 벌어집니다. 창이 분 단위로 갈리므로 00:59.9에 100회를 채우고 01:00.1에 다시 100회를 채우면, 각 창의 한도는 지켰는데도 0.2초 사이에 200회가 통과합니다. 분당 100이라고 적어 둔 한도가 순간적으로는 200까지 허용되는 셈이라, 무차별 대입을 막으려던 방어선이 경계에서 두 배로 느슨해집니다.

!

TTL이 붙지 않고 남는 카운터. 위 코드는 INCR 다음에 EXPIRE를 겁니다. 프로세스가 그 두 명령 사이에서 죽으면 키는 값이 1인 채로 만료되지 않고 남습니다. 다음 창이 와도 카운터가 리셋되지 않아 해당 사용자는 이유를 알 수 없는 영구 차단에 걸립니다. 한도 로직이 여러 명령으로 쪼개지면 이런 틈이 생긴다는 걸 여기서 알았고, 그래서 세는 일을 한 번에 끝내는 방법을 찾게 됐습니다. 뒤에서 Lua로 이 틈을 닫습니다.

슬라이딩 윈도우 — 로그와 카운터

경계에서 한도가 벌어지는 건 창이 고정돼 있기 때문이니, 창을 요청 시점 기준으로 미끄러뜨리면 해결됩니다. 그중 가장 정확한 방식이 슬라이딩 윈도우 로그였습니다. 요청 타임스탬프를 정렬 집합(ZSET)에 쌓아 두고, 지금으로부터 1분보다 오래된 기록을 버린 뒤 남은 개수를 셉니다.

// 슬라이딩 윈도우 로그 — 타임스탬프를 정렬 집합에 쌓는다
const now = Date.now();
const key = `rl:log:${id}`;
const res = await redis.multi()                      // 네 명령을 한 트랜잭션으로
  .zremrangebyscore(key, 0, now - windowMs)          // 창 밖 기록 제거
  .zadd(key, now, `${now}:${nanoid()}`)             // 이번 요청 기록
  .zcard(key)                                         // 창 안 요청 수
  .pexpire(key, windowMs)
  .exec();
const count = res[2][1];                            // zcard 결과
return count <= limit;

어느 순간에 재도 직전 1분의 실제 요청 수가 나오니 경계 문제는 사라집니다. 대신 메모리를 씁니다. 요청 하나마다 멤버가 하나씩 쌓이고, 위 코드는 검사보다 zadd를 먼저 하기 때문에 폭주가 들어오면 거절할 요청까지 집합에 기록됩니다. 한도가 100인데 초당 만 번을 맞는 상황이라면, 막는 데 쓰는 자원이 오히려 공격의 표적이 됩니다.

그 사이의 절충이 슬라이딩 윈도우 카운터입니다. 타임스탬프를 전부 저장하는 대신 현재 창과 직전 창의 카운터 두 개만 두고, 직전 창을 겹친 비율만큼 빌려 씁니다. 대략 현재 창 수 + 직전 창 수 × 겹침 비율입니다. 메모리는 카운터 두 개로 고정되고 경계 문제도 눌러지지만, 직전 창의 요청이 고르게 분포했다고 가정하는 근사라 정확하지는 않습니다. 이 근사가 실제로 얼마나 쓸 만한지는 뒤의 책 이야기에서 숫자로 다시 적었습니다.

토큰 버킷과 Lua 원자성

여기까지는 몇 번 왔는지를 세는 방식이었습니다. 토큰 버킷은 세는 대상이 조금 다릅니다. 용량이 정해진 버킷에 토큰이 일정 속도로 채워지고, 요청은 토큰을 하나씩 씁니다. 토큰이 남아 있으면 통과시키고 없으면 거절합니다. 이 방식은 버스트를 어느 정도 허용합니다. 평소에 쉬다가 잠깐 몰아 쓰는 사용자는 그동안 쌓인 토큰으로 통과하고, 지속적으로 두드리는 쪽만 막힙니다. 사람이 화면을 클릭하는 패턴 자체가 몰렸다 뜸했다 하는 모양이라, vLLM 앞에는 이 방식을 골랐습니다.

고정 윈도우 모드에서 [버스트 10발]을 창 경계 직전·직후에 나눠 쏘면, "직전 5초 실제 통과"가 한도의 두 배까지 벌어지는 걸 볼 수 있습니다

다만 토큰 버킷은 INCR 하나로 끝나지 않습니다. 토큰을 읽고, 흐른 시간만큼 채우고, 검사한 다음, 차감해서 다시 쓰는 읽기-수정-쓰기입니다. 이걸 애플리케이션 코드에서 GET 후 SET으로 처리하면 두 요청이 같은 잔량을 읽고 둘 다 통과하는 일이 생기고, 그만큼 한도가 샙니다. API 서버가 여러 대여서 한 Redis를 같이 볼수록 이 경쟁이 자주 드러납니다. 그래서 이 과정 전체를 하나의 원자적 연산으로 묶었습니다. Redis에서는 Lua 스크립트가 그 역할을 합니다.

-- token_bucket.lua  |  KEYS[1]=버킷,  ARGV=capacity, refill, now, cost
local tokens = tonumber(redis.call('HGET', KEYS[1], 'tokens'))
local ts     = tonumber(redis.call('HGET', KEYS[1], 'ts'))
local cap    = tonumber(ARGV[1])
local refill = tonumber(ARGV[2])          -- 초당 채워지는 토큰
local now    = tonumber(ARGV[3])
local cost   = tonumber(ARGV[4])

if tokens == nil then tokens = cap; ts = now end
tokens = math.min(cap, tokens + (now - ts) * refill)  -- 흐른 시간만큼 지연 충전

local ok = tokens >= cost
if ok then tokens = tokens - cost end

redis.call('HSET', KEYS[1], 'tokens', tokens, 'ts', now)
redis.call('EXPIRE', KEYS[1], math.ceil(cap / refill) * 2)  -- 노는 버킷은 자동 소멸
return {ok and 1 or 0, tokens}

여기서 두 가지가 같이 정리됩니다. 하나는 지연 충전(lazy refill)입니다. 타이머를 돌려 토큰을 채우는 대신, 마지막으로 만진 시각과의 차이로 그동안 채워졌어야 할 양을 그때그때 계산합니다. 덕분에 버킷이 백만 개여도 백그라운드 작업이 필요 없습니다. 다른 하나는 앞 절에 적어 둔 TTL 틈입니다. 읽기와 검사, 차감, EXPIRE가 한 스크립트 안에 있으니 중간에 끊길 자리가 없어집니다.

*

Lua가 원자적인 이유. Redis는 명령을 단일 스레드로 처리하고, EVAL로 넘긴 스크립트는 끝까지 실행되는 동안 다른 명령을 끼워 넣지 않습니다. 그래서 스크립트 안의 읽기-수정-쓰기는 인터리빙되지 않습니다. 같은 이유로 스크립트는 짧게 유지해야 합니다. 실행되는 동안 서버 전체가 멈추기 때문입니다. 그리고 클러스터에서 제대로 라우팅되려면 만지는 키를 KEYS[]로 넘겨야 합니다. 키 이름을 하드코딩하면 슬롯 계산을 건너뛰게 됩니다.

NestJS 가드로 얹고 429를 돌려준다

다음은 리미터를 요청 경로에 얹는 일이었습니다. NestJS에는 걸 수 있는 자리가 셋 있고, 각각 생애주기 위치가 다릅니다. 미들웨어는 가드보다도 앞에서 가장 먼저 돌고, 가드는 이 요청을 통과시킬지를 결정하고, 인터셉터는 핸들러 실행을 감쌉니다. 레이트 리미팅이 하는 일이 통과 여부 결정이라 가드에 두었습니다. 인터셉터는 핸들러를 이미 감싼 뒤라 시점이 늦고, 미들웨어는 인증 컨텍스트(req.user)를 아직 못 보는 경우가 있었습니다.

// 429의 출구를 한곳으로 — 가드가 접근 여부를 결정한다
@Injectable()
export class RateLimitGuard implements CanActivate {
  constructor(private readonly limiter: TokenBucket, private readonly metrics: Metrics) {}

  async canActivate(ctx: ExecutionContext): Promise<boolean> {
    const req = ctx.switchToHttp().getRequest<Request>();
    const res = ctx.switchToHttp().getResponse<Response>();
    const id  = req.user?.id ?? req.ip;          // 인증됐으면 사용자, 아니면 IP

    const { allowed, remaining, resetSec } = await this.limiter.take(id);
    res.setHeader('X-RateLimit-Limit', this.limiter.capacity);
    res.setHeader('X-RateLimit-Remaining', Math.max(0, Math.floor(remaining)));
    if (allowed) return true;

    res.setHeader('Retry-After', resetSec);          // '언제 다시'를 알려준다
    this.metrics.increment('ratelimit.blocked', { route: req.path });
    throw new HttpException('Too Many Requests', 429);
  }
}

거절은 429 Too Many Requests로 돌려주고, Retry-After로 언제 다시 와도 되는지를 같이 알려 줍니다. 크롤러를 DDD로 설계하며 반대편에 서 봤을 때 배운 부분인데, 상대 서버가 Retry-After를 주지 않으면 클라이언트는 재시도 시점을 알 수 없어 오히려 더 자주 두드리게 됩니다. 막는 일과 물러설 방법을 알려 주는 일은 같이 가야 한다고 생각합니다. 429를 받은 클라이언트가 지수 백오프로 물러서는 방법은 백오프 글에 적어 두었습니다.

마지막으로 ratelimit.blocked 메트릭을 같이 남겨 두었습니다. 429가 갑자기 늘어나는 건 공격이 시작됐거나 방금 배포한 클라이언트가 버그로 우리를 두드리고 있다는 뜻이고, 둘 다 사람이 알아야 하는 일이었습니다. 리미터는 막는 장치이면서 동시에 센서 역할도 합니다. 이 신호를 알림으로 연결하는 방법은 관측 가능성 글의 "증상이 아니라 영향에 알림을 건다"와 이어집니다.

책 한 권 — "레이트 리미터를 설계하라"

이 주제를 한자리에 정리해 준 건 알렉스 쉬(Alex Xu)의 『System Design Interview』 1권, "레이트 리미터를 설계하라(Design a Rate Limiter)" 장이었습니다. 이 책은 모든 장이 같은 4단계 틀을 따릅니다. 문제와 설계 범위를 확정하고, 개략 설계를 제시하고, 심화하고, 마무리하는 순서입니다. 레이트 리미터 장에서 제게 남은 건 코드보다 이 질문의 순서였습니다.

책은 다섯 알고리즘을 나란히 놓고 장단을 따집니다. 토큰 버킷, 리키 버킷, 고정 윈도우 카운터, 슬라이딩 윈도우 로그, 슬라이딩 윈도우 카운터입니다. 토큰 버킷은 아마존과 스트라이프가 API 스로틀링에 쓰는 예로, 리키 버킷은 쇼피파이의 예로 듭니다. 고정 윈도우의 경계 문제, 슬라이딩 로그의 메모리 비용, 슬라이딩 카운터의 근사까지, 지금 이 글의 뼈대도 사실 이 장을 따라간 것입니다. 카운터의 근사가 얼마나 정확한지를 두고 책은 클라우드플레어의 분석을 인용해 극히 일부 요청에서만 오차가 난다고 짚습니다. 그 근사 공식은 코드로 옮기면 꽤 단순합니다.

// 책의 슬라이딩 윈도우 카운터 — 이전 창을 '겹친 비율'만큼 빌려 쓴다
const elapsed  = (now % windowMs) / windowMs;   // 현재 창에서 흐른 비율
const estimate = curr + prev * (1 - elapsed);
// 예) 한도 7, 이전 창 5, 현재 창 3, 창의 30% 지점
//     → 3 + 5 × 0.7 = 6.5 → 6, 한도 이하이므로 통과
return estimate <= limit;

실무에 옮길 때 가장 도움이 된 건 알고리즘 비교보다 분산 환경을 다루는 절이었습니다. 책은 리미터가 여러 대로 늘어나는 순간 경쟁 상태와 동기화라는 두 문제가 따라온다고 정리합니다. 카운터를 읽고 쓰는 사이 다른 요청이 끼어드는 경쟁은 Lua 스크립트나 정렬 집합으로 풀고, 여러 서버가 상태를 공유하려면 스티키 세션 대신 중앙 저장소(Redis)를 두라는 이야기였습니다. 앞 절에서 토큰 버킷의 읽기-수정-쓰기를 통째로 Lua에 밀어 넣은 것도, 책이 여기서 깨진다고 표시해 둔 지점을 그대로 막은 것이었습니다.

세부도 실무에 가깝게 적혀 있습니다. 429 응답과 함께 X-Ratelimit-Remaining·X-Ratelimit-Limit·X-Ratelimit-Retry-After 같은 헤더로 남은 한도와 재시도 시점을 돌려주라는 계약, Redis의 INCR·EXPIRE로 카운터를 구현하는 밑그림까지 나옵니다. 인터뷰 대비서라 폭은 넓고 깊이는 얕은 편이지만, 결정을 어떤 순서로 내려야 하는지를 잡아 준다는 점에서 리미터를 실제로 얹기 전에 한 번 읽어 두기 좋은 장이었습니다.

정리 — 방어선을 어디에 긋나

레이트 리미팅은 라이브러리 하나 붙이면 되는 일처럼 보였는데, 해 보니 결정을 순서대로 내리는 일에 가까웠습니다. 그 순서를 정리하면 이렇습니다.

  • 비싼 일 앞, 바깥쪽에 긋는다. 인증·DB·GPU에 닿기 전 미들웨어 계층에서 막아야 막으려던 비용을 치르지 않는다.
  • 키는 엔드포인트가 정한다. 인증 전은 IP, 인증 후는 사용자·API 키. NAT 뒤 다수가 한 IP를 공유하는 걸 잊지 않는다.
  • 고정 윈도우는 경계에서 한도가 두 배까지 벌어진다. 비용은 가장 싸지만, 그 순간의 초과를 감수할 수 있을 때만 쓴다.
  • 정확함이 필요하면 슬라이딩 로그, 절충이면 카운터. 로그는 메모리를, 카운터는 근사를 대가로 낸다.
  • 사용자 대면 API엔 토큰 버킷. 버스트를 관용하고 지속적 남용만 막는 게 실제 트래픽 모양에 맞는다.
  • 세는 일은 한 번에 끝낸다. 읽기-수정-쓰기가 쪼개지면 한도가 새고 TTL 틈으로 영구 차단까지 생기니, Lua로 묶는다.
  • 거절은 429 + Retry-After + 남은 한도 헤더. 막되, 언제 물러서야 하는지를 알려 주는 게 계약이다.
  • 429 급증은 알림으로 잇는다. 리미터는 방어선이면서 센서이기도 하니, 막은 횟수를 메트릭으로 남긴다.

돌아보면 한도를 정하는 건 기술보다 요구사항에 가까운 문제였습니다. 이 엔드포인트를 얼마나 두드려도 되는지를 도메인에 먼저 묻지 않으면, 어떤 알고리즘을 골라도 숫자는 감으로 찍은 값이 됩니다. 알고리즘은 그렇게 정한 값을 새지 않게 집행하는 방법이었습니다.
