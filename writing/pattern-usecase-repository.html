아키텍처 패턴 · UseCase와 Repository — 레거시에 경계 긋기

2022년 11월• UseCase · Repository · 스트랭글러

레거시 개선 사례 글에는 6개월간 신뢰를 쌓은 뒤에야 구조 개선을 제안할 수 있었다고 적었습니다. 정작 팀이 동의해 준 다음 날부터 무엇을 어떤 순서로 옮겼는지는 남겨 두지 않아서, 그 부분을 여기에 적어 둡니다. 운영 중인 시스템이었으니 전면 재작성은 애초에 선택지가 아니었고, 할 수 있는 일은 경계를 하나씩 긋는 것 정도였습니다.

먼저 짚고 가기
트랜잭션 스크립트Transaction Script
마틴 파울러가 《엔터프라이즈 애플리케이션 아키텍처 패턴》(PoEAA)에서 분류한 도메인 로직 패턴 중 하나로, 요청 하나를 프로시저 하나가 처음부터 끝까지 처리하는 구성입니다. 검증·계산·DB 접근·외부 호출이 모두 그 메서드의 지역 흐름으로 이어지고, 데이터는 도메인 객체가 아니라 레코드나 Map 형태로 오갑니다. 안티패턴이 아닙니다 — PoEAA는 로직이 단순하면 Domain Model보다 이쪽이 낫다고 명시합니다. 이 방식이 무너지는 것은 절차들 사이에 공유되는 업무 규칙이 생기고, 그 규칙이 복사되기 시작한 뒤입니다.
UseCase유스케이스
야콥슨이 요구사항 기법으로 정리한 use case를 코드의 단위로 삼은 애플리케이션 계층 객체입니다. 클린 아키텍처는 같은 것을 인터랙터(Interactor)라고 부릅니다. 하는 일은 조율입니다 — 입력을 받아 Repository로 도메인 객체를 불러오고, 정해진 순서로 호출하고, 트랜잭션 경계를 소유합니다. 다만 업무 규칙 자체는 도메인 객체에 남겨 둡니다. 규칙까지 이 안으로 들어오면 이름만 바뀐 트랜잭션 스크립트가 되기 때문에, 이 글에서는 업무 문장 하나 = 클래스 하나 = 진입점 execute 하나를 규칙으로 씁니다.
Repository리포지토리
같은 책(PoEAA)의 데이터 접근 패턴으로, 도메인 객체의 컬렉션처럼 보이는 인터페이스 뒤에 매핑 계층을 감추는 것이 정의입니다. 호출부는 SQL·테이블 구조·ORM 사용 여부 대신 도메인 언어로 된 메서드(예: findActiveByCarNo)를 호출하고, 저장 기술의 선택은 구현체 안에만 남습니다. 인터페이스가 넓다고 추상화가 되는 것은 아닙니다 — 임의 조건을 Map으로 받는 범용 조회 메서드는 테이블 사정을 파라미터 키 이름으로 흘려보내므로 경계는 이름만 남습니다. 커밋은 이 계층의 책임이 아니고, 트랜잭션 경계는 위쪽 UseCase가 갖습니다.
불변식Invariant
객체가 외부에서 관찰되는 모든 시점에 참이어야 하는 조건입니다. 예를 들어 "만료된 정기권은 연장 상태로 넘어갈 수 없다" 같은 것으로, 생성자가 이 조건이 성립한 상태로만 객체를 만들고 상태를 바꾸는 모든 메서드가 종료 시점에 다시 성립시키는 방식으로 강제합니다. 메서드 실행 도중 일시적으로 깨지는 것은 허용됩니다 — 밖에서 볼 수 있는 시점에만 참이면 되기 때문입니다. 뒤집어 말하면 setter로 필드를 열어 두거나 UPDATE 문이 객체를 우회하는 순간, 불변식은 코드에 적혀 있어도 아무것도 강제하지 못합니다.
스트랭글러 패턴Strangler
마틴 파울러가 이름 붙인 레거시 이행 전략입니다(원문 Strangler Fig Application — 숙주 나무를 감고 올라가 끝내 대체하는 교살 무화과에서 따온 이름입니다). 기존 호출 지점에 가로채는 층(파사드·라우팅·위임 껍데기)을 두고 기능을 하나씩 새 구현으로 옮긴 뒤, 옛 경로의 호출자가 0이 되면 그 조각만 지웁니다. 전면 재작성과 갈리는 지점은 매 단계가 배포 가능한 상태로 유지된다는 것입니다. 대신 두 구현이 공존하는 기간이 반드시 생기므로, 같은 데이터에 대한 쓰기 소유권은 한 시점에 한쪽만 갖게 해야 규칙이 우회되지 않습니다.

1. 트랜잭션 스크립트가 무너지는 조건

제가 맡은 코드는 2008년 주차 시스템을 기반으로 자라온 것이었습니다. 요청이 들어오면 서비스 메서드 하나가 검증부터 저장, 알림까지 전부 처리했습니다. 이런 구조를 트랜잭션 스크립트라고 부른다는 것도 나중에 알았습니다. 실제 모습은 대략 이랬습니다.

// SeasonTicketService.java — 2008년부터 자라온 메서드
public Map<String, Object> regist(HttpServletRequest req) {
  String carNo   = req.getParameter("car_no");
  String lotType = req.getParameter("lot_type");
  String phone   = req.getParameter("phone");
  // 1) 검증 — 같은 규칙이 화면마다 복사되어 있다
  if (carNo == null || carNo.length() < 7) { /* ... */ }
  // 2) 요금 계산 — 지점 × 차종 × 감면이 if로 중첩
  int fee = 100000;
  if ("B".equals(lotType))  fee = 80000;
  if (isDiscount(req))      fee = fee / 2;
  // 3) 저장 — SQL이 서비스 안에 그대로
  jdbc.update("INSERT INTO ST_MST (CAR_NO, FEE, USE_YN) VALUES (?, ?, 'Y')", carNo, fee);
  // 4) 알림 — 트랜잭션 한가운데서 외부 호출
  smsClient.send(phone, "정기권이 등록되었습니다");
  return resultMap("OK");
}

다만 트랜잭션 스크립트 자체가 결함인 것은 아닙니다. 《엔터프라이즈 애플리케이션 아키텍처 패턴》도 로직이 단순할 때는 이 방식이 가장 정직한 선택이라고 말합니다. 절차가 위에서 아래로 그대로 읽히기 때문입니다. 이 구조가 무너지는 건 두 조건이 겹칠 때인데, 제가 본 코드에는 두 조건이 다 있었습니다.

첫째는 분기의 조합 폭발이었습니다. 지점별 요금 × 차종 × 감면 조건이 곱해지면서 if 중첩이 사람이 추적할 수 있는 깊이를 넘어서 있었습니다. 둘째는 화면 단위 복사였습니다. 새 화면을 만들 때 기존 메서드를 복사해 고치는 관행이 쌓여서, 같은 업무 규칙이 몇 군데에 있는지 아무도 알지 못했습니다. 이렇게 되면 변경 비용은 코드량이 아니라 "이 규칙이 어디에 또 있는지 모른다"는 불확실성에서 나옵니다. 고치는 시간보다 찾는 시간이 길었습니다. 제가 그은 경계는 이 두 가지를 겨냥한 것이었습니다.

2. 첫 번째 경계 — UseCase는 시스템이 하는 '한 문장'이다

UseCase 계층의 단위는 클래스나 메서드가 아니라 업무 문장 하나로 잡았습니다. "관리자가 정기권을 등록한다" 같은 문장 하나가 클래스 하나가 되고, 진입점은 execute 하나, 입력과 출력은 그 유스케이스 전용 타입을 씁니다. 위의 regist를 이 규칙대로 옮기면 다음과 같습니다.

// 유스케이스 하나 = 클래스 하나 = 진입점 하나
public class RegisterSeasonTicket {

  public record Command(String carNo, Long lotId, DiscountKind discount) {}
  public record Result(Long ticketId, Money fee) {}

  private final SeasonTicketRepository tickets;
  private final ParkingLotRepository lots;
  private final FeePolicy feePolicy;

  @Transactional   // 트랜잭션 경계는 유스케이스가 소유한다
  public Result execute(Command cmd) {
    ParkingLot lot = lots.getById(cmd.lotId());
    tickets.findActiveByCarNo(cmd.carNo())
           .ifPresent(t -> { throw new AlreadyRegistered(t); });
    Money fee = feePolicy.calculate(lot, cmd.discount());
    SeasonTicket ticket = SeasonTicket.issue(cmd.carNo(), lot, fee);
    tickets.save(ticket);
    return new Result(ticket.getId(), fee);  // SMS는 커밋 후 이벤트로
  }
}

서비스 클래스에 메서드를 하나 더 추가하지 않고 클래스 단위로 쪼갠 이유는 변경의 축이 달라서입니다. 하나의 서비스 클래스에 등록·해지·연장 메서드가 쌓이면 서로 다른 이유로 바뀌는 코드가 한 파일에 살게 됩니다. 등록 규칙을 고치다 해지를 깨뜨리거나, 두 사람이 같은 파일을 고치다 충돌하기 쉬웠습니다. 유스케이스를 클래스 단위로 두니 따라오는 것이 세 가지 있었습니다. 디렉터리 목록이 그대로 기능 목록이 되어 새로 온 사람이 코드를 열면 시스템이 무슨 일을 하는지 보이고, 테스트 단위가 클래스 하나 = 시나리오 하나로 정해지고, 기능을 내리는 일이 클래스를 지우는 일이 됩니다. 레거시에서는 마지막 항목이 생각보다 자주 필요했습니다.

원래 코드의 4번, 트랜잭션 한가운데의 SMS 발송이 사라진 것도 경계를 그은 결과입니다. 외부 호출이 트랜잭션 안에 있으면 SMS 지연이 DB 커넥션을 물고 있게 되고, 발송 후 롤백되면 "등록됐다는 문자를 받았는데 등록이 안 된" 상태가 됩니다. 알림은 커밋 이후로 밀어내는 편이 맞는데, 그 판단을 내릴 자리가 트랜잭션 경계를 소유한 UseCase였습니다.

*

이름을 보면 경계가 어긋난 것을 알 수 있습니다. 유스케이스 이름은 동사+목적어로 지었습니다 — RegisterSeasonTicket, CancelSeasonTicket. 이름에 Manager, Processor, Service 같은 단어가 필요해진다면 문장 두 개를 한 클래스에 담으려 한다는 신호로 봤고, 그럴 때는 쪼갰습니다.

3. 두 번째 경계 — Repository가 숨겨야 할 것, 드러내야 할 것

UseCase가 위쪽 경계라면 Repository는 아래쪽 경계입니다. 그런데 "저장소를 추상화한다"는 말만 알고 있을 때는 막상 무엇을 해야 할지 잘 몰랐습니다. 도움이 된 건 무엇을 숨기고 무엇을 드러낼지 목록으로 적어 두는 일이었습니다.

숨겨야 하는 것은 저장 기술의 사정입니다. SQL 문자열, ORM 사용 여부, 테이블·조인 구조, soft delete 플래그, 캐시를 타는지 여부 같은 것들입니다. 이것들이 시그니처로 새어 나오면 호출부가 기술에 결합되고, 경계는 이름만 남습니다. 반대로 드러내야 하는 것은 도메인의 사정입니다. '없을 수 있음'은 Optional로 타입에 드러나야 하고, '활성인', '만료된' 같은 업무 조건은 메서드 이름에 드러나야 합니다. 그리고 앞서 적었듯 트랜잭션 경계는 Repository가 갖지 않습니다. 저장소가 스스로 커밋해 버리면 유스케이스가 여러 저장을 하나의 원자적 작업으로 묶을 방법을 잃기 때문입니다.

// 나쁜 경계 — 저장 기술이 시그니처로 샌다
List<Map<String,Object>> selectList(Map<String,Object> param); // 무엇이든 물어보세요
void executeQuery(String sql);                                  // 경계가 아예 없다
Entity findByCarNoAndUseYnAndDelYn(String c, String u, String d); // 테이블 사정이 호출부로

// 좋은 경계 — 도메인의 질문만 남긴다
public interface SeasonTicketRepository {
  Optional<SeasonTicket> findActiveByCarNo(String carNo); // '없음'을 타입으로
  SeasonTicket getById(Long id);   // 없으면 예외 — get 접두사의 계약
  void save(SeasonTicket ticket);
}

나쁜 예의 첫 줄 selectList(Map)은 레거시에서 자주 만나는 형태입니다. 언뜻 유연해 보이지만 실제로는 모든 호출부가 파라미터 키 이름이라는 문서화되지 않은 계약에 묶입니다. 인터페이스가 넓다고 추상화가 되는 것은 아니었습니다. 쓸 만한 Repository 인터페이스는 오히려 좁고, 메서드 하나하나가 도메인의 질문 하나에 대응합니다. 인터페이스를 도메인 언어로 좁혀 두니 테스트도 같이 쉬워졌습니다. findActiveByCarNo는 가짜 구현으로 바꾸기 쉽지만, selectList(Map)의 가짜 구현은 사실상 DB를 다시 만드는 일이 됩니다.

4. 원칙이 깨지는 곳 — 집계 쿼리와 읽기 모델

여기까지의 원칙을 한 줄로 줄이면 "모든 데이터 접근은 Repository를 통해 도메인 객체로"입니다. 그런데 이 원칙을 그대로 지키기 어려운 지점이 있었습니다. 월 정산 집계가 그랬습니다. 지점별 월 매출을 뽑는 화면을 원칙대로 만들면 정기권 수만 건을 도메인 객체로 로딩해 메모리에서 합산하게 됩니다. 성능도 성능이지만, 도메인 객체가 이 작업에서 하는 일이 없다는 점이 더 걸렸습니다.

Repository가 도메인 객체를 반환하는 이유를 되짚어 보면 쓰기 경로의 불변식을 보호하기 위해서입니다. "만료된 정기권은 연장할 수 없다" 같은 규칙을 객체가 스스로 강제하게 하려는 것입니다. 그런데 읽기 전용 집계에는 보호할 불변식이 없으니, 원칙이 서 있던 전제 자체가 없는 셈이었습니다. 그래서 이런 곳에서는 읽기 전용 쿼리 모델로 의도적으로 우회했습니다.

// 읽기 전용 쿼리 모델 — 원칙을 '알고' 깬다
public interface SettlementQuery {
  List<LotMonthlyRow> monthlyByLot(YearMonth month);  // 반환은 DTO — 도메인 객체인 척하지 않는다
}

// 구현은 SQL 한 방 — 집계는 DB가 가장 잘한다
public List<LotMonthlyRow> monthlyByLot(YearMonth m) {
  return jdbc.query("""
    SELECT LOT_ID, COUNT(*) AS CNT, SUM(FEE) AS TOTAL
      FROM ST_MST
     WHERE ISSUED_AT BETWEEN ? AND ?
     GROUP BY LOT_ID""", rowMapper, start(m), end(m));
}

대신 우회에는 두 가지 조건을 걸어 두었습니다. 하나는 읽기 전용이라는 것으로, 이 경로로는 쓰지 않습니다. 다른 하나는 DTO를 반환한다는 것입니다. 집계 결과가 도메인 객체 행세를 하기 시작하면 두 모델이 섞이면서 경계가 다시 무너지기 때문입니다. 이건 CQRS를 가볍게 적용한 형태인데, 이름은 나중에 알았습니다. 그보다 도움이 된 건 원칙이 어떤 전제 위에 서 있는지 알아 두면 전제가 없는 자리에서 마음 편히 깰 수 있다는 점이었습니다. 요금 계산 쪽의 FeePolicy도 같은 맥락에서 Strategy로 분리해 두었는데, 이 구조는 이후 정산 시스템을 Template Method + Strategy로 리팩토링할 때 더 큰 규모로 다시 쓰게 됩니다.

5. 스트랭글러 — 옮기는 순서가 곧 전략이다

구조가 정해졌어도 레거시 전체를 한 번에 옮길 수는 없었습니다. 스트랭글러 패턴의 요점은 새 구조가 옛 구조를 감싸며 자라게 하는 것인데, 해 보니 실제로 중요한 건 옮기는 순서였습니다. 저는 다음 순서로 옮겼습니다.

먼저 새 기능입니다. 새로 들어오는 요구사항은 처음부터 UseCase로 작성했습니다. 기존 코드를 한 줄도 건드리지 않으니 위험이 없고, 팀은 실제 코드로 패턴을 익히고, 새 방식이 돌아간다는 증거도 쌓입니다. 다음은 버그가 난 지점이었습니다. 어차피 손을 대야 하는 코드이니, 고치기 전에 현재 동작을 그대로 붙잡아 두는 특성 테스트(characterization test)를 먼저 만들고, 테스트가 초록인 상태에서 UseCase로 이관한 뒤 버그를 고쳤습니다. 수정 비용에 이관 비용을 얹는다기보다 이관이 수정을 안전하게 만들어 주는 쪽에 가까웠습니다. 마지막이 자주 바뀌는 지점입니다. 변경 빈도가 곧 투자 대비 수익이라, 반대로 몇 년째 바뀌지 않은 코드는 끝까지 옮기지 않아도 괜찮았습니다. 레거시를 없애는 것이 목표가 아니라 변경이 일어나는 자리에 경계가 있게 하는 것이 목표였기 때문입니다.

이관한 뒤에도 기존 화면단 호출부는 그대로 두었습니다. 레거시 서비스 메서드를 껍데기로 만들어 위임하는 식입니다.

// 레거시 서비스는 껍데기로 — 호출부는 그대로 둔다
/** @deprecated RegisterSeasonTicket 으로 이관됨. 새 코드에서 호출 금지. */
@Deprecated
public Map<String, Object> regist(HttpServletRequest req) {
  var cmd = new RegisterSeasonTicket.Command(
      req.getParameter("car_no"),
      Long.valueOf(req.getParameter("lot_id")),
      DiscountKind.from(req.getParameter("dc")));
  var result = registerSeasonTicket.execute(cmd);
  return legacyResponse(result);  // 기존 화면이 기대하는 Map 형태 유지
}

이 껍데기가 하는 일은 번역뿐입니다. 낡은 인터페이스(HttpServletRequest와 Map)와 새 경계(Command와 Result) 사이를 옮겨 주고, 업무 로직은 전부 경계 안쪽으로 들어갔습니다. 껍데기의 호출자가 하나씩 줄어들다 0이 되면 그때 지웁니다.

!

이관 기간에 가장 조심한 것은 쓰기 경로가 둘로 나뉘는 상황이었습니다. 같은 테이블을 레거시 DAO와 새 Repository가 동시에 쓰면, 도메인 객체가 강제하던 불변식이 레거시 경로로 우회됩니다. "만료 정기권은 연장 불가"를 UseCase가 지켜도 옛 화면의 UPDATE 한 줄이 그대로 통과합니다. 읽기는 공유하더라도 쓰기 소유권은 한 시점에 한쪽에만 두는 편이 안전했습니다. 테이블 단위로 소유권을 넘기고, 넘긴 뒤에는 레거시 쪽 쓰기 코드를 바로 위임으로 바꿨습니다.

이 순서는 기술적인 판단이기도 했지만 신뢰의 문제이기도 했습니다. 새 기능부터 시작하면 누구의 코드도 부수지 않으면서 증거를 만들 수 있습니다. 6개월을 기다려 얻은 발언권을 첫 이관에서 장애를 내며 잃고 싶지 않았습니다. 그 신뢰에 대한 이야기는 사례 글에 적어 두었습니다. 덧붙이면 이 경계는 프레임워크와 상관이 없었습니다. 이후 NestJS로 백엔드를 설계할 때도 UseCase를 Provider로 두는 같은 규칙을 그대로 썼습니다.

6. 정리 — 경계를 긋는 규칙들

이 글에서 적은 규칙을 목록으로 남겨 둡니다. 레거시 위에서 하나씩 확인한 것들입니다.

  • 트랜잭션 스크립트는 조건부 부채다. 분기의 조합 폭발과 규칙 복사가 겹치는 순간부터 비용이 불확실성으로 바뀐다.
  • UseCase의 단위는 업무 문장 하나. 클래스 하나, 진입점 하나, 전용 입출력 타입. 이름은 동사+목적어 — Manager가 필요해지면 쪼갤 때다.
  • 트랜잭션 경계는 UseCase가 소유한다. Repository는 커밋하지 않고, 외부 호출(알림 등)은 커밋 뒤로 보낸다.
  • Repository는 기술을 숨기고 도메인을 드러낸다. SQL·테이블 사정은 감추고, '없음'은 Optional로, 업무 조건은 메서드 이름으로. 인터페이스가 넓다고 추상화가 되지는 않는다.
  • 읽기 전용 집계에서는 원칙을 알고 깬다. 보호할 불변식이 없으면 쿼리 모델로 우회하되, 읽기 전용 + DTO 반환이라는 조건을 건다.
  • 이관 순서는 새 기능 → 버그 지점 → 변경 빈도 순. 레거시를 없애는 것이 아니라, 변경이 일어나는 자리에 경계를 두는 것이 목표다.
  • 스트랭글러 기간에도 쓰기 소유권은 한쪽에만. 레거시 메서드는 위임 껍데기로 남기고, 호출자가 0이 되면 지운다.

경계를 긋는 일은 코드를 다시 쓰는 일보다 지루하고 오래 걸립니다. 재작성은 한 번의 결단으로 끝나지만 경계는 계속 지켜야 하는 규율이기 때문입니다. 그래도 운영 중인 시스템에서 고를 수 있는 쪽은 대개 후자였고, 그렇게 하나씩 옮겨 두었습니다.
