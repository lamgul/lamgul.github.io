논문 분석: MapReduce — 큰 데이터를 작게 쪼개는 법

2021년 12월• MapReduce · 분산처리 · 논문

크롤러를 만들면서 데이터를 모으는 쪽은 사이트별, 페이지별로 쪼개 여러 워커에 맡기면 됐는데, 모아 온 데이터를 집계하는 쪽은 늘 한 프로세스에 남아 있었습니다. 데이터가 늘수록 그 집계 스크립트 하나가 병목이 됐고, 어떻게 나눠야 할지 몰라 한동안 그대로 두었습니다. 같은 문제를 2004년의 구글이 먼저 정리해 둔 것이 MapReduce 논문이어서, 크롤러를 만드는 사람의 눈으로 다시 읽고 정리해 둡니다.

출처

MapReduce: Simplified Data Processing on Large Clusters

Jeffrey Dean, Sanjay Ghemawat · OSDI 2004

공식 PDF ↗ · Google Research ↗

먼저 짚고 가기
Map
입력 (키, 값) 쌍 하나를 받아 중간 (키, 값) 쌍들의 목록을 내놓는 사용자 정의 함수입니다 — 논문 2절의 표기로 map(k1, v1) → list(k2, v2). 워드카운트에서는 (문서 이름, 문서 내용)을 받아 단어마다 EmitIntermediate(w, "1")을 부릅니다. 호출 하나가 자기 입력 조각 바깥의 상태를 보지 않기 때문에 M개 조각을 아무 기계에나 흩어 돌려도 결과가 같습니다. 다만 방출한 중간 결과는 전역 파일시스템이 아니라 그 워커의 로컬 디스크에 남고, 이 사실이 뒤에서 볼 장애 복구 규칙을 정합니다.
Reduce
중간 키 하나와 그 키로 모인 값들의 이터레이터를 받아 값을 접어 출력하는 사용자 정의 함수입니다 — reduce(k2, list(v2)) → list(v2), 호출 한 번에 보통 값 0개나 1개를 냅니다. 값을 리스트가 아니라 이터레이터로 넘기는 것은 한 키에 딸린 값이 메모리보다 커도 처리할 수 있게 하기 위해서입니다(논문 2절). 키별로 값을 모아 주는 일은 사용자가 아니라 라이브러리가 하므로, 이 함수는 자기 키 바깥의 값을 아예 볼 수 없습니다.
셔플Shuffle
map의 중간 출력을 키 기준으로 재배치해, 한 키의 모든 값이 담당 reduce 워커 한 곳에 모이게 하는 단계입니다. 구체적으로는 map 워커가 출력을 partition(key) = hash(key) mod R로 R개 파일에 나눠 로컬 디스크에 쓰고, reduce 워커가 RPC로 그 파일들을 원격 읽기 한 뒤 중간 키로 정렬해 같은 키를 인접시킵니다(논문 3.1절). 사용자가 코드를 한 줄도 쓰지 않는 유일한 단계이지만 중간 데이터 전체가 네트워크를 건너므로, 논문 4.3절의 combiner는 바로 이 구간으로 나가는 양을 줄이려는 장치입니다.
데이터 지역성Data Locality
데이터를 계산 쪽으로 옮기는 대신, 계산을 그 데이터의 복제본이 이미 있는 기계에 배정하는 스케줄링 원칙입니다. 마스터는 GFS가 64MB 블록 단위로 보통 3벌씩 복제해 둔 입력 블록의 위치를 알고 있어서, map 태스크를 복제본을 가진 기계에 우선 배정하고 실패하면 같은 네트워크 스위치에 물린 가까운 기계를 찾습니다(논문 3.4절). 여기서 절약되는 것은 map이 입력을 읽는 대역폭이지, 셔플이 중간 데이터를 나르는 대역폭은 아닙니다.
스트래글러Straggler
작업 막바지에 남은 몇 개의 태스크를 유난히 오래 붙들고 있는 기계를 가리키는 논문의 용어입니다(논문 3.6절). 죽은 게 아니라 느리게나마 진행하기 때문에 핑 기반 장애 감지에 걸리지 않고, 전체 작업은 마지막 태스크가 끝나야 끝나므로 이 한 대가 수천 대짜리 클러스터의 완료 시간을 정합니다. 논문이 드는 원인은 보정 가능한 디스크 오류로 읽기가 30MB/s에서 1MB/s로 떨어진 기계, 다른 태스크와 자원을 다투는 경우, 프로세서 캐시가 꺼진 채 초기화된 기계 같은 것들입니다. 대응은 원인 진단이 아니라 백업 태스크 중복 실행이고, 이 장치를 끄자 같은 정렬 작업이 44% 더 걸렸습니다(논문 5.4절).

1. 하루치는 되는데 한 달치는 안 됩니다

제가 쓰던 집계 스크립트도 처음에는 단순했습니다. 파일을 열고, 한 줄씩 읽고, 딕셔너리에 세고, 마지막에 정렬해서 출력하는 정도였고, 크롤링한 데이터가 하루치일 때는 그걸로 충분했습니다. 그런데 데이터는 티 나지 않게 늘어났습니다. 수집 쪽은 워커를 늘려 따라갔지만 집계는 여전히 한 프로세스가 전부 떠안고 있어서, 어느 시점부터는 메모리가 부족해지거나 처리 시간이 감당하기 어려워졌습니다.

여러 대에서 나눠 돌리면 되지 않을까 싶었지만, 막상 쪼개려고 하니 원래 세는 로직과는 상관없는 질문들이 먼저 걸렸습니다.

  • 데이터를 어떤 기준으로 몇 조각으로 나눌 것인가
  • 나눠 돌리다 한 대가 죽으면 어디서부터 다시 할 것인가
  • 각자 센 부분 결과를 어떻게 안전하게 합칠 것인가

논문 1장이 출발하는 지점도 여기와 비슷합니다. 저자들은 구글 안에서 크롤링한 문서 처리, 역색인 만들기, 웹 그래프 집계 같은 수백 개의 특수 목적 계산이 저마다 병렬화·분산·장애 처리 코드를 반복해서 짊어지고 있는 상황을 두고, 계산 자체는 단순한데 그걸 감싸는 분산 코드가 원래 로직을 뒤덮는다고 정리합니다. MapReduce는 그 감싸는 부분을 라이브러리 쪽으로 옮겨 둔 결과물입니다.

2. 함수 두 개만 남기고 전부 숨긴다

논문이 사용자에게 남겨 두는 것은 map과 reduce 두 함수뿐입니다. 논문 2.1절의 워드카운트 예제가 사용자가 직접 작성하는 코드의 전부입니다.

// 논문 2.1절의 워드카운트 — 사용자가 쓰는 코드의 전부
map(String key, String value):
    // key: 문서 이름, value: 문서 내용
    for each word w in value:
        EmitIntermediate(w, "1");

reduce(String key, Iterator values):
    // key: 단어 하나, values: 그 단어에 대해 모인 "1"들
    int result = 0;
    for each v in values:
        result += ParseInt(v);
    Emit(AsString(result));

왜 이 두 함수인지는 각각이 어떤 모양으로 병렬화되는지를 보면 드러납니다. map은 입력 조각 하나만 보고 결과를 내므로 조각 수만큼 병렬이 되고, reduce는 키 하나에 모인 값만 보므로 키를 나눈 조각(R) 수만큼 병렬이 됩니다. 두 함수 모두 "내 몫 바깥"을 모르도록 설계되어 있어서, 분배와 재시도와 취합을 라이브러리가 알아서 처리해도 사용자의 계산 결과는 달라지지 않습니다. 앞에서 제가 걸렸던 질문들, 그러니까 어떻게 나누고 죽으면 어쩌고 어떻게 합치느냐가 전부 라이브러리 쪽 문제로 옮겨 가는 셈입니다. 좋은 추상화는 기능을 더하는 쪽보다 사용자가 신경 쓸 목록에서 항목을 지우는 쪽에 가깝다는 생각을, 이 절을 읽으면서 하게 됐습니다.

3. 숨은 세 번째 단계, shuffle

이름은 MapReduce지만 실제 실행은 map → shuffle → reduce 세 단계로 이루어집니다. map이 뱉은 (단어, 1)들은 여러 워커의 로컬 디스크에 흩어져 있는데, "big"이라는 단어 하나를 합산하려면 워커 1이 만든 (big, 1)도 워커 3이 만든 (big, 1)도 같은 reduce 워커에 도착해야 합니다. 이 재배치가 셔플이고, 배달 주소는 아래 수식으로 정해집니다.

# 어느 map 워커가 만든 (키, 값)이든 같은 키는 같은 reduce 조각으로 (논문 4.1절)
partition(key) = hash(key) mod R      # R = reduce 조각 수

# 입력은 M개 조각으로 쪼갠다 — 조각 하나가 보통 16~64MB (논문 3.1절)
M = ceil(전체 입력 크기 / 조각 크기)

해시가 결정적이라 워커들끼리 따로 조율하지 않아도 "big은 3번 reduce로"라는 같은 결론이 나옵니다. 대신 셔플은 중간 데이터 전체를 네트워크에 태우는 단계라 비용이 적지 않습니다. 논문이 M과 R을 워커 수보다 훨씬 크게 잡으라고 권하는 것도(논문 3.5절) 태스크가 잘게 쪼개져야 부하 분산이 매끄럽고 워커가 죽었을 때 복구도 빨라지기 때문이고, 아래에서 볼 데이터 지역성과 백업 태스크는 이 단계에서 생기는 네트워크 비용과 꼬리 지연을 겨냥한 장치들입니다.

*

워드카운트라면 "the" 같은 단어가 한 워커 안에서만 수천 번 나옵니다. 이걸 전부 (the, 1)로 네트워크에 태우는 대신, 논문 4.3절의 combiner는 map이 끝난 그 자리에서 부분합을 미리 만들어 셔플로 나가는 양을 줄입니다. 네트워크로 내보내기 전에 로컬에서 줄일 수 있는 만큼 줄인다는 이야기인데, 제가 다루는 파이프라인에도 그대로 옮겨 볼 만한 발상이었습니다.

세 단계가 실제로 어떻게 이어지는지는 글로만 읽어서는 잘 잡히지 않아서, 직접 돌려 볼 수 있는 위젯을 하나 만들어 두었습니다. 문장을 바꾸거나 워커 수를 조절해 보면, 워커를 늘렸을 때 map 입력이 몇 조각으로 나뉘는지 그리고 그런데도 셔플과 reduce의 결과는 왜 같은지를 눈으로 확인할 수 있습니다.

워드카운트로 보는 map → shuffle → reduce직접 조작해 보세요
1 · map(단어, 1)
2 · shuffle키별 그룹
3 · reduce합계

워커 수를 바꾸면 map 입력이 다시 쪼개지지만, shuffle과 reduce의 결과는 변하지 않습니다

4. 데이터를 옮기지 말고 계산을 옮긴다

제가 집계 스크립트를 짜던 습관에서는 데이터를 저장소에서 계산하는 프로세스 쪽으로 끌어왔습니다. DB에서 애플리케이션으로 전부 당겨 와 메모리에서 세는 식이었습니다. 논문 3.4절은 반대 방향을 택합니다. 입력 파일은 GFS가 64MB 블록 단위로 여러 기계에 복제해 두는데(복제본 보통 3개), 마스터는 map 태스크를 배정할 때 그 입력 블록의 복제본을 이미 들고 있는 기계에 우선 배정합니다. 실패하면 같은 네트워크 스위치에 물린 가까운 기계라도 찾습니다. 대규모 작업에서는 대부분의 입력을 로컬 디스크에서 읽게 되어 네트워크를 거의 쓰지 않는다고 논문은 보고합니다.

당시 구글 클러스터는 평범한 IDE 디스크를 단 저가 장비 수천 대와 상대적으로 귀한 네트워크 대역폭으로 이루어져 있었습니다(논문 3장). 디스크는 기계마다 있으니 흔하고, 기계 사이를 잇는 네트워크는 모두가 나눠 쓰니 귀합니다. 그렇다면 큰 데이터를 귀한 길로 옮기는 대신 작은 코드를 데이터 곁으로 보내는 편이 낫다는 계산입니다. 제약이 설계를 정한 사례인데, 저는 이 절을 읽고 나서야 제 집계 스크립트에서 시간을 잡아먹던 쪽이 "세는 시간"이 아니라 데이터를 프로세스까지 옮기는 시간이었다는 걸 알았습니다. 집계 쿼리를 DB 쪽에서 끝내고 요약만 받아 오는 것도 따지고 보면 같은 원칙의 작은 실천이었습니다.

5. 제일 느린 한 대가 전체 속도를 정한다

수천 대를 동원해도 작업은 마지막 태스크가 끝나야 끝납니다. 그런데 현실의 클러스터에는 유난히 느린 기계가 꼭 하나씩 있습니다. 논문 3.6절이 드는 예는 디스크 오류 보정 때문에 읽기 속도가 30MB/s에서 1MB/s까지 떨어진 기계인데, 고장이 아니라서 죽지도 않고 느리게나마 일은 하기 때문에 교체 대상도 아닙니다. 이런 스트래글러 한 대가 전체 완료 시간을 붙잡습니다.

논문의 대응은 원인을 진단해 고치는 쪽이 아닙니다. 작업이 거의 끝나갈 무렵, 아직 진행 중인 태스크들의 복제본(백업 태스크)을 다른 기계에서 하나 더 돌립니다. 원본이든 백업이든 먼저 끝나는 쪽을 채택하고 나머지는 버립니다. 계산을 이중으로 하는 낭비를 몇 퍼센트 감수하는 대신 꼬리에서 시간을 버는 거래인데, 논문 5장의 실험에 그 값어치가 나와 있습니다. 약 1,800대 클러스터에서 100바이트 레코드 100억 개(약 1TB)를 정렬하는 데 891초가 걸렸고(논문 5.3절), 백업 태스크를 끄고 같은 작업을 돌리니 1,283초, 그러니까 44% 더 걸렸습니다(논문 5.4절). 소수의 스트래글러를 위해 전체가 기다리는 비용이 그만큼 컸다는 뜻입니다.

수천 대 클러스터를 다루지 않아도 이 관점은 쓸 데가 있었습니다. 여러 워커에 나눠 도는 크롤링 잡의 완료 시간도 결국 가장 느린 워커, 그러니까 응답이 유난히 느린 사이트를 배정받은 워커가 정합니다. 평균이 아니라 꼬리가 전체를 정한다는 것, 그리고 원인을 진단하는 것보다 중복 실행이 쌀 때가 있다는 것을 이 절에서 메모해 두었습니다.

6. 장애 복구가 "다시 실행" 한 줄이 되는 이유

수천 대 규모에서 기계 고장은 예외 상황이 아니라 일상에 가깝습니다. 논문 3.3절의 복구 전략은 그래서 오히려 단순합니다. 마스터는 워커들에게 주기적으로 핑을 보내고, 응답이 끊긴 워커의 태스크를 다른 기계에서 처음부터 다시 실행합니다. 상태를 백업했다가 복원하는 방식이 아니라 그냥 다시 계산하는 방식입니다.

# 논문 3.3절의 워커 장애 복구 규칙 — 옮겨 보면 이게 전부입니다
def on_worker_failure(worker):
    for task in worker.map_tasks:       # 완료된 map 태스크까지 포함
        task.state = IDLE                # 결과가 죽은 기계의 로컬 디스크에 있으므로
        reschedule(task)                 # 다른 워커에서 처음부터 다시
    for task in worker.reduce_tasks:
        if task.state != DONE:           # 완료된 reduce 출력은 전역 파일시스템에 있음
            reschedule(task)

여기서 두 가지 디테일을 따로 적어 두었습니다. 첫째로 완료된 map 태스크도 다시 돌립니다. map의 출력은 그 기계의 로컬 디스크에만 있어서 기계가 죽으면 접근할 수 없기 때문입니다. 반면 reduce의 출력은 전역 파일시스템에 이미 올라가 있으니 다시 돌릴 필요가 없습니다. 결과를 어디에 두느냐가 복구 규칙을 정하는 셈입니다. 둘째로 reduce는 결과를 임시 파일에 쓴 뒤 원자적인 이름 변경(rename)으로 최종 파일을 만듭니다. 같은 태스크가 두 번 실행되어도 최종 파일은 한 벌만 남습니다.

*

다시 실행하는 것만으로 복구가 되려면 map과 reduce가 결정적이어야 합니다. 같은 입력에 늘 같은 출력을 내는 함수라면 재실행한 결과와 원래 결과가 구분되지 않으므로 따로 복구 로직을 둘 필요가 없습니다. 논문 3.3절이 "결정적 연산이라면 분산 실행 결과가 순차 실행 결과와 같다"고 보장하는 것도 이 전제 위에서입니다. 반대로 함수 안에 난수나 현재 시각, 외부 상태가 끼어들면 이 보장은 성립하지 않습니다.

이 절은 크롤러를 만들면서 겪은 일과 겹칩니다. 수집 작업이 중간에 죽었을 때 어디까지 했는지를 복원하려 들면 복구 코드가 본 코드보다 복잡해지곤 했습니다. 작업을 다시 실행해도 안전한 단위로 잘라 두고 실패하면 그 단위를 통째로 다시 돌리는 쪽이 코드도 짧고 검증도 쉬웠습니다. 재시도를 안전하게 만드는 건 재시도 코드보다 재시도해도 되는 작업의 모양이라는 정리를, 3.3절을 읽으면서 하게 됐습니다.

7. 정리 — 쪼개기, 묶기, 다시 하기

논문을 덮고 나서 손에 남은 건 동사 세 개였습니다. 서로를 모르는 조각으로 쪼개고(map), 같은 키끼리 묶고(shuffle), 실패하면 다시 하는 것(결정적 재실행)입니다. 데이터 지역성과 백업 태스크는 이 골격 위에 얹힌 보정으로, 네트워크는 귀하고 꼬리는 길다는 현실을 감안한 장치들입니다.

이 모델은 오픈소스 구현인 하둡(Hadoop)을 거쳐 업계 표준이 됐고, 이후의 Spark 같은 시스템들도 결정적 계산의 재실행으로 장애를 복구한다는 이 논문의 뼈대 위에서 출발했습니다. 규모와 무관하게, 집계 스크립트가 무거워지기 시작한 저 같은 사람에게 이 논문은 질문 목록으로 남습니다. 내 계산은 서로를 모르는 조각으로 쪼개지는가, 조각들을 묶는 기준(키)은 무엇인가, 실패한 조각을 처음부터 다시 돌려도 안전한가. 이 세 질문에 답할 수 있는 작업은 데이터가 커져도 잘 무너지지 않았습니다. 크롤링 파이프라인의 수집·정제 작업을 설계할 때 제가 계속 꺼내 쓰는 질문들이기도 합니다.
