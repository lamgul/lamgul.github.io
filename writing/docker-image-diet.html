Docker 이미지를 반으로 — 멀티스테이지와 레이어 캐시

2025년 10월• 약 6분• Docker · 멀티스테이지 · 캐시

배포가 느리고 이미지가 1GB를 넘는다면, 십중팔구 빌드에 쓴 도구를 런타임까지 들고 가는 탓입니다. 컴파일러, 개발 의존성, 캐시 — 실행에는 필요 없는 것들이죠.

먼저 짚고 가기
도커 이미지Docker Image
프로그램과 그 실행에 필요한 것들을 한 덩어리로 묶어 둔 '완성된 도시락'입니다. 이 도시락이 크면 옮기고 배포하는 데 오래 걸리므로, 필요 없는 반찬을 빼는 것이 이 글의 목표입니다.
런타임Runtime
프로그램을 실제로 '실행하는' 순간과 그때 필요한 환경을 뜻합니다. 요리를 만드는 데 쓰는 도구(빌드)와 다 만든 음식을 먹는 순간(런타임)이 다르듯, 실행에는 컴파일러 같은 빌드 도구가 필요 없다는 점이 핵심입니다.
멀티스테이지 빌드Multi-stage Build
이미지를 만드는 과정을 여러 단계(스테이지)로 나누고, 앞 단계에서 만든 결과물만 마지막 단계로 옮겨 담는 방식입니다. 무거운 도구는 앞 단계에 두고 버려서, 최종 이미지에는 실행에 필요한 알맹이만 남깁니다.
레이어 캐시Layer Cache
도커는 이미지를 여러 층(레이어)으로 쌓아 만들고, 바뀌지 않은 층은 다시 만들지 않고 재사용합니다. 이 재사용분이 캐시입니다. 어떤 명령을 위에 두느냐에 따라 재빌드 속도가 크게 달라집니다.
빌드 컨텍스트Build Context
도커가 이미지를 만들 때 참고하려고 통째로 읽어 들이는 폴더 전체입니다. 여기에 불필요한 파일이 많으면 그만큼 빌드가 느려지므로, .dockerignore로 걸러 냅니다.

멀티스테이지 — 빌드와 런타임을 분리

# 1) 빌드 스테이지: 무거운 도구 총동원
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 2) 런타임 스테이지: 결과물만 복사
FROM node:20-slim
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
USER node                          # 비루트로 실행
CMD ["node", "dist/main.js"]

런타임 이미지에는 소스도, 빌드 캐시도 없습니다. 필요한 산출물만 --from=build로 건져 옵니다. 이것만으로 이미지가 절반 이하로 줄곤 합니다.

레이어 캐시는 '순서'가 전부

Docker는 레이어를 위에서부터 캐시합니다. 자주 바뀌는 걸 아래에 두는 게 핵심입니다. 의존성 파일을 먼저 COPY하고 설치한 뒤, 소스를 나중에 복사하세요. 소스만 바뀌면 npm ci 레이어는 캐시가 그대로 살아, 재빌드가 몇 초로 끝납니다. 순서를 뒤집으면 매번 전부 다시 깝니다.

빼먹기 쉬운 두 가지 — .dockerignore와 비루트

*

.dockerignore를 빼먹지 마세요. node_modules·.git이 빌드 컨텍스트로 딸려 들어가면 캐시가 매번 깨지고 빌드가 느려집니다. 그리고 비루트 유저로 실행하는 건 이미지 다이어트가 아니라 기본 위생입니다.
