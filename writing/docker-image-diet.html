Docker 이미지를 반으로 — 멀티스테이지와 레이어 캐시

2025년 10월• Docker · 멀티스테이지 · 캐시

배포가 느려서 이미지 크기를 재 보니 1GB가 넘었습니다. 원인은 빌드에 쓴 도구를 런타임 이미지까지 그대로 들고 간 것이었습니다. 컴파일러와 개발 의존성, 빌드 캐시는 실행할 때 쓰지 않는데도 최종 이미지 안에 그대로 남아 있었습니다. 멀티스테이지 빌드로 실행에 필요한 것만 남기고 레이어 순서를 손봐 재빌드 시간을 줄인 과정을 적어 둡니다.

먼저 짚고 가기
도커 이미지Docker Image
컨테이너를 실행하는 데 쓰는 읽기 전용 레이어(각각 tar 아카이브) 목록과 실행 설정(ENTRYPOINT·CMD·ENV·WORKDIR 등)을 담은 config를 manifest 하나로 묶은 패키지 포맷입니다(OCI Image Spec). 레이어는 각각 sha256 다이제스트로 식별되고, 실행 시점에는 overlay2 같은 union 파일시스템이 이 레이어들을 순서대로 겹쳐 올린 뒤 맨 위에 쓰기 가능한 레이어를 얹습니다. 이미 쌓인 레이어는 불변이라, 뒤 레이어에서 RUN rm으로 파일을 지워도 삭제 표시만 덧붙을 뿐 아래 레이어의 원본 바이트는 그대로 남아 이미지 크기는 줄지 않습니다.
런타임Runtime
컨테이너 프로세스가 실제로 돌아가는 시점, 그리고 그 시점에 프로세스가 참조하는 파일 일체를 가리키는 말입니다(runc·containerd 같은 '컨테이너 런타임' 소프트웨어와는 다른 뜻입니다). 멀티스테이지 Dockerfile에서는 마지막 FROM 이후에 쌓인 파일시스템과 config가 그대로 런타임 환경이 되므로, 거기에 복사해 두지 않은 파일은 실행 중에 아예 존재하지 않습니다. 다만 '빌드 도구는 전부 버려도 된다'가 늘 성립하지는 않아서, 인터프리터와 동적 링크되는 라이브러리는 런타임에도 남아 있어야 합니다 — glibc 기반에서 빌드한 네이티브 모듈을 musl 기반 alpine 이미지로 옮기면 실행되지 않습니다.
멀티스테이지 빌드Multi-stage Build
Dockerfile 하나에 FROM을 여러 번 써서 빌드를 독립된 스테이지로 나누고, COPY --from=<스테이지>로 앞 스테이지의 파일시스템에서 필요한 경로만 꺼내 오는 빌드 방식입니다(Docker 17.05부터). 최종 이미지의 manifest에는 마지막 FROM 이후에 쌓인 레이어만 들어가므로, 앞 스테이지에 설치한 컴파일러와 개발 의존성은 최종 이미지에 흔적이 남지 않습니다. 스테이지는 AS 이름이나 인덱스 번호로 참조하고, --target으로 중간 스테이지에서 빌드를 멈출 수도 있습니다. 줄어드는 것은 배포되는 이미지 크기이지, 빌드 머신의 디스크는 아닙니다 — 버려진 스테이지도 빌드 캐시로는 로컬에 그대로 남습니다.
레이어 캐시Layer Cache
RUN·COPY·ADD는 각각 새 파일시스템 레이어를 만들고(나머지 명령은 메타데이터만 바꿉니다), 빌더는 '직전까지의 이미지 상태 + 이번 명령'으로 캐시 키를 계산해 같은 키의 결과가 있으면 명령을 실행하지 않고 기존 레이어를 재사용합니다. COPY·ADD는 명령 문자열뿐 아니라 복사 대상 파일들의 내용 체크섬까지 키에 넣지만, RUN은 기본적으로 명령 문자열만 보기 때문에 apt-get update처럼 실행할 때마다 결과가 달라지는 명령도 캐시에 걸려 넘어갑니다. 한 단계에서 키가 어긋나면 그 아래 모든 단계가 함께 무효화되므로, 재빌드 시간은 '무엇을 캐시하느냐'보다 '가장 자주 바뀌는 명령이 몇 번째 줄에 있느냐'로 결정됩니다.
빌드 컨텍스트Build Context
docker build에 인자로 넘긴 경로의 파일 트리 전체를 말하며, 빌드가 시작될 때 이 트리가 빌더로 전송됩니다. Dockerfile의 COPY·ADD는 이 트리 안의 경로만 참조할 수 있어 컨텍스트 바깥의 상위 디렉터리 파일은 복사할 수 없고, 반대로 컨텍스트에 들어온 파일은 COPY하지 않더라도 전송과 해싱 대상에는 포함됩니다. .dockerignore에 적은 패턴은 이 대상에서 제외되므로 전송량과 COPY 캐시 키 계산 범위가 함께 줄어듭니다. BuildKit은 매번 전부 보내는 대신 바뀐 파일만 증분 전송하지만, 걸러 두지 않은 .git은 커밋할 때마다 내용이 바뀌어 COPY . .의 캐시 키를 흔듭니다.

멀티스테이지 — 빌드와 런타임을 분리

# 1) 빌드 스테이지: 무거운 도구 총동원
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 2) 런타임 스테이지: 결과물만 복사
FROM node:20-slim
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
USER node                          # 비루트로 실행
CMD ["node", "dist/main.js"]

이렇게 나눠 두면 런타임 이미지에는 소스도 빌드 캐시도 남지 않고, 필요한 산출물만 --from=build로 넘어옵니다. 이 변경만으로 이미지가 절반 이하로 줄어드는 경우가 많았습니다.

레이어 캐시는 '순서'가 전부

Docker는 레이어를 위에서부터 캐시하기 때문에, 자주 바뀌는 명령일수록 아래에 두는 편이 낫습니다. 그래서 의존성 파일을 먼저 COPY하고 설치한 다음, 소스는 그 뒤에 복사하도록 순서를 바꿨습니다. 이렇게 두면 소스만 고쳤을 때 npm ci 레이어의 캐시가 그대로 남아서 재빌드가 몇 초 만에 끝납니다. 순서를 반대로 써 두면 소스 한 군데만 고쳐도 의존성 설치가 매번 처음부터 다시 돕니다.

빼먹기 쉬운 두 가지 — .dockerignore와 비루트

*

여기까지 하고 나서 .dockerignore도 같이 챙겼습니다. node_modules나 .git이 빌드 컨텍스트로 딸려 들어가면 캐시가 매번 깨지고 빌드가 느려집니다. 이미지 크기만 보고 있으면 잘 드러나지 않는 부분이었습니다. 비루트 유저로 실행하는 건 이미지 크기와는 상관없는 이야기지만, 기본 위생에 가까워서 같은 작업에 묶어 두었습니다.
