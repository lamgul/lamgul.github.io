Docker 이미지를 반으로 — 멀티스테이지와 레이어 캐시

2025년 10월• 약 6분• Docker · 멀티스테이지 · 캐시

배포가 느리고 이미지가 1GB를 넘는다면, 십중팔구 빌드에 쓴 도구를 런타임까지 들고 가는 탓입니다. 컴파일러, 개발 의존성, 캐시 — 실행에는 필요 없는 것들이죠.

멀티스테이지 — 빌드와 런타임을 분리

# 1) 빌드 스테이지: 무거운 도구 총동원
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 2) 런타임 스테이지: 결과물만 복사
FROM node:20-slim
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
USER node                          # 비루트로 실행
CMD ["node", "dist/main.js"]

런타임 이미지에는 소스도, 빌드 캐시도 없습니다. 필요한 산출물만 --from=build로 건져 옵니다. 이것만으로 이미지가 절반 이하로 줄곤 합니다.

레이어 캐시는 '순서'가 전부

Docker는 레이어를 위에서부터 캐시합니다. 자주 바뀌는 걸 아래에 두는 게 핵심입니다. 의존성 파일을 먼저 COPY하고 설치한 뒤, 소스를 나중에 복사하세요. 소스만 바뀌면 npm ci 레이어는 캐시가 그대로 살아, 재빌드가 몇 초로 끝납니다. 순서를 뒤집으면 매번 전부 다시 깝니다.

빼먹기 쉬운 두 가지 — .dockerignore와 비루트

*

.dockerignore를 빼먹지 마세요. node_modules·.git이 빌드 컨텍스트로 딸려 들어가면 캐시가 매번 깨지고 빌드가 느려집니다. 그리고 비루트 유저로 실행하는 건 이미지 다이어트가 아니라 기본 위생입니다.
