온프레미스에서 LLM을 서빙한다는 것 — vLLM과 양자화

2026년 5월• vLLM · 양자화 · GPU

산업안전 현장의 데이터를 외부로 전송할 수 없어서, 모델을 직접 GPU에 올려 써야 했습니다. API 키 하나로 불러 쓰던 때와는 사정이 달랐습니다. 모델을 부르는 일과 운영하는 일이 서로 다른 문제라는 걸 그때 알았고, 그 과정을 적어 둡니다.

먼저 짚고 가기
온프레미스on-premise
애플리케이션과 데이터를 외부 클라우드가 아니라 조직이 소유·통제하는 자체 인프라(사내 서버실·공장 전산실 등)에서 실행하고 저장하는 배포 방식입니다. 물리적·네트워크 경계가 조직 내부에 있어 데이터가 외부망으로 나가지 않는 대신, 하드웨어 조달과 확장·유지보수 책임을 운영 주체가 집니다. 클라우드(IaaS/SaaS)와 대비되는 개념으로, 규제나 보안 제약으로 데이터 반출이 금지될 때 선택합니다.
서빙serving
학습이 끝난 모델을 프로세스로 상주시켜, 네트워크 요청이 들어오면 추론(forward pass)을 실행해 응답을 돌려주는 운영 단계입니다. 모델을 학습시키는 일(training)이나 스크립트로 한 번 실행해 보는 추론과 달리, 다수의 동시 요청을 받아 지연(latency)과 처리량(throughput)을 관리하며 엔드포인트를 상시 유지하는 것이 핵심입니다. 이 글에서는 OpenAI 호환 HTTP 엔드포인트 형태로 서빙합니다.
KV 캐시KV cache
트랜스포머가 토큰을 하나씩 생성(자기회귀 추론)할 때, 각 레이어 어텐션의 Key·Value 텐서를 저장해 두는 GPU 메모리 구조입니다. 이미 처리한 토큰의 K·V를 다시 계산하지 않고 재사용하려는 것으로, Query는 캐시하지 않고 매 스텝 새로 계산합니다. 크기가 시퀀스 길이와 동시 처리 요청 수에 비례해 선형으로 늘어 VRAM을 점유하므로, 이 메모리를 어떻게 배치·관리하느냐가 서빙 처리량을 좌우합니다.
vLLM
대규모 언어모델의 추론·서빙에 특화된 오픈소스 엔진입니다. 핵심은 KV 캐시를 고정 크기 블록(페이지) 단위로 비연속 할당하는 PagedAttention과, 매 디코딩 스텝마다 완료된 시퀀스 자리에 대기 중인 새 요청을 채워 넣는 연속 배칭(continuous batching)으로, 같은 GPU에서 처리량을 크게 끌어올립니다. UC 버클리에서 시작됐고 PagedAttention 기법은 SOSP 2023 논문으로 공개됐으며, 학습이 아니라 추론 서빙 전용 엔진입니다.
양자화quantization
모델 가중치를 FP16/FP32 부동소수점에서 INT4·INT8 같은 저비트 정수로 매핑해 저장하는 압축 기법입니다. 각 가중치 그룹의 스케일(과 영점)로 실수 범위를 정수 격자에 근사하는 것이라 정보 손실이 있는 근사이며, 본문의 AWQ·GPTQ는 어떤 가중치가 출력에 더 중요한지를 반영해 그 손실을 줄인 4비트 weight-only 방식입니다. VRAM 사용량이 비트 수에 비례해 줄고, LLM 추론이 메모리 대역폭 병목이라 읽어 오는 데이터가 줄어 지연도 함께 감소합니다.

왜 그냥 transformers로는 안 됐나

처음에는 model.generate()로 붙여 두었는데, 데모까지가 한계였습니다. 요청이 하나씩 순차로 처리되고, KV 캐시가 GPU 메모리를 파편화해서 동시 사용자가 늘면 버티지 못했습니다. 번역 요청이 몰리는 아침 조회 시간에 이걸로는 감당이 안 돼서 vLLM으로 옮겼습니다.

PagedAttention과 연속 배칭

vLLM에서 실제로 효과를 본 건 두 가지였습니다. PagedAttention은 KV 캐시를 OS의 가상 메모리처럼 페이지 단위로 관리해 파편화를 없앱니다. 연속 배칭(continuous batching)은 한 요청이 끝나기를 기다리지 않고, 매 스텝 완료된 슬롯에 새 요청을 채워 넣는 방식입니다. 같은 GPU인데도 처리량이 몇 배로 올라갔습니다.

# OpenAI 호환 엔드포인트로 바로 서빙
vllm serve Qwen2.5-7B-Instruct-AWQ \
  --quantization awq \
  --gpu-memory-utilization 0.90 \   # KV 캐시가 쓸 VRAM 비중
  --max-num-seqs 64 \               # 동시 처리 슬롯
  --max-model-len 4096

양자화 — VRAM 반, 속도 덤

7B 모델도 FP16이면 14GB가 넘습니다. AWQ/GPTQ 4비트 양자화를 적용하니 4GB대까지 내려갔고, 메모리 대역폭이 병목인 추론이라 지연도 함께 줄었습니다. 품질 손실은 걱정했던 것과 달리 번역·요약 같은 태스크에선 체감되지 않았습니다. 같은 VRAM이라면 "더 큰 모델을 4비트로" 올리는 편이 "작은 모델을 FP16으로" 올리는 것보다 나은 경우가 많았습니다.

*

튜닝에서 손이 가장 많이 간 건 gpu-memory-utilization과 max-num-seqs의 균형이었습니다. 슬롯을 너무 늘리면 KV 캐시가 터지고(OOM), 너무 줄이면 GPU가 놉니다. 미리 계산해서 정할 수 있는 값은 아니어서, 실제 트래픽으로 부하를 걸어 가며 값을 찾았습니다.

배운 것

  • 처리량과 지연은 트레이드오프였습니다. 배치를 키우면 처리량이 오르는 대신 개별 응답은 느려져서, 공지 번역은 처리량 쪽으로, 대화형은 지연 쪽으로 태스크마다 다르게 잡았습니다.
  • 이 태스크에서는 양자화로 잃은 게 거의 없었습니다. 다만 코드 생성이나 수리 논증처럼 정밀도가 중요한 태스크라면 벤치마크를 따로 돌려 봐야 할 것 같습니다.
  • 돌아보면 LLM 서빙은 모델보다 인프라 쪽 일에 가까웠습니다. 모델을 올리는 데는 30분이면 됐고, 지연을 관리하며 운영에 태우는 일은 그 뒤로도 한참 남아 있었습니다.
