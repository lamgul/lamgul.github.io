온프레미스에서 LLM을 서빙한다는 것 — vLLM과 양자화

2026년 5월• 약 9분• vLLM · 양자화 · GPU

API 키 하나면 최고의 모델을 부를 수 있는 시대에, 저는 모델을 직접 GPU에 올려야 했습니다. 산업안전 현장의 데이터를 외부로 전송할 수 없었기 때문입니다. 모델을 부르는 것과 운영하는 것은 완전히 다른 문제였습니다.

먼저 짚고 가기
온프레미스on-premise
서비스를 남의 클라우드가 아니라 우리 사무실·공장 안에 직접 둔 서버에서 돌리는 방식입니다. 데이터가 건물 밖으로 나가지 않아, 이 글처럼 현장 데이터를 외부로 못 보내는 상황에서 어쩔 수 없이 택하게 됩니다.
서빙serving
학습이 끝난 모델을 실제 사용자 요청에 24시간 응답하도록 띄워 두고 운영하는 일입니다. 모델을 한 번 불러 보는 것과 달리, 여러 사람이 동시에 써도 안 무너지게 만드는 게 핵심입니다.
KV 캐시KV cache
LLM이 문장을 한 글자씩 만들 때, 앞서 읽은 내용을 매번 다시 계산하지 않으려고 중간 결과를 저장해 두는 GPU 메모리 공간입니다. 응답이 길어질수록 커져서, 이 메모리를 어떻게 관리하느냐가 서빙 성능을 좌우합니다.
vLLM
LLM을 빠르고 효율적으로 서빙하려고 만든 오픈소스 엔진입니다. 뒤에 나오는 PagedAttention과 연속 배칭으로, 같은 GPU에서 훨씬 많은 요청을 처리하게 해 줍니다.
양자화quantization
모델의 숫자(가중치)를 32비트·16비트에서 4비트처럼 더 거친 단위로 줄여 저장하는 기술입니다. 사진을 저화질로 압축하듯 용량이 확 줄어, 작은 GPU에도 큰 모델을 올릴 수 있게 됩니다.

왜 그냥 transformers로는 안 됐나

model.generate()는 데모용입니다. 요청이 하나씩 순차 처리되고, KV 캐시가 GPU 메모리를 파편화하며, 동시 사용자가 늘면 무너집니다. 번역 요청이 몰리는 아침 조회 시간에 이걸로는 감당이 안 됐습니다. 그래서 vLLM으로 갈아탔습니다.

PagedAttention과 연속 배칭

vLLM의 두 가지가 판을 바꿉니다. PagedAttention은 KV 캐시를 OS의 가상 메모리처럼 페이지 단위로 관리해 파편화를 없앱니다. 연속 배칭(continuous batching)은 한 요청이 끝나기를 기다리지 않고, 매 스텝 완료된 슬롯에 새 요청을 밀어 넣습니다. 처리량이 몇 배로 뜁니다.

# OpenAI 호환 엔드포인트로 바로 서빙
vllm serve Qwen2.5-7B-Instruct-AWQ \
  --quantization awq \
  --gpu-memory-utilization 0.90 \   # KV 캐시가 쓸 VRAM 비중
  --max-num-seqs 64 \               # 동시 처리 슬롯
  --max-model-len 4096

양자화 — VRAM 반, 속도 덤

7B 모델도 FP16이면 14GB가 넘습니다. AWQ/GPTQ 4비트 양자화로 4GB대까지 내려가고, 메모리 대역폭이 병목인 추론에서는 오히려 지연도 줄어듭니다. 품질 손실은 번역·요약 같은 태스크에선 체감이 거의 없었습니다. "더 큰 모델을 4비트로"가 "작은 모델을 FP16으로"보다 나은 경우가 많았습니다.

*

튜닝의 핵심은 gpu-memory-utilization과 max-num-seqs의 균형이었습니다. 슬롯을 너무 늘리면 KV 캐시가 터지고(OOM), 너무 줄이면 GPU가 논다. 실제 트래픽으로 부하를 걸어 스윗스팟을 찾는 수밖에 없었습니다.

배운 것

  • 처리량과 지연은 트레이드오프다. 배치를 키우면 처리량이 오르지만 개별 응답은 느려진다. 공지 번역은 처리량, 대화형은 지연 — 태스크마다 다르게 잡았다.
  • 양자화는 공짜 점심에 가깝다(이 태스크에선). 다만 코드 생성·수리 논증처럼 정밀도가 중요한 태스크라면 반드시 벤치마크를 돌려야 한다.
  • 결국 LLM 서빙은 인프라 문제였다. 모델은 30분이면 올리지만, 지연을 관리하며 운영에 태우는 건 그 뒤의 일이다.
