FastAPI 의존성 주입 — 크롤러 어댑터를 갈아끼우는 법

2025년 2월• 약 10분• FastAPI · Python · DI

크롤링 파이프라인을 설계하면서 가장 오래 붙잡았던 질문은 "사이트별 크롤러를 누가 고르고, 누가 넣어주는가"였습니다. fashion_scrapy에서는 dependency-injector 라이브러리로 풀었고, FastAPI에는 Depends가 내장되어 있습니다. 이 글은 Depends가 실제로 어떻게 동작하는지 뜯어보고, 같은 문제 — 어댑터 교체 — 를 두 도구로 각각 풀어서 비교합니다.

먼저 짚고 가기
의존성 주입Dependency Injection
어떤 객체가 필요로 하는 부품(의존성)을 자기가 직접 만들지 않고, 밖에서 만들어 건네받는 방식입니다. 요리사가 재료를 직접 재배하지 않고 받아 쓰는 것과 같습니다. 이 글의 핵심 주제로, "사이트별 크롤러를 누가 골라 넣어주는가"라는 문제를 푸는 열쇠입니다.
어댑터 패턴Adapter Pattern
사이트마다 다른 세부 구현을 공통된 껍데기(인터페이스) 뒤에 숨겨, 바깥에서는 다 똑같은 모양으로 다루게 하는 설계입니다. 이 글에서 사이트마다 다른 URL·셀렉터·차단 정책을 CrawlerAdapter 하나로 격리하는 근거입니다.
인터페이스Interface / Protocol
"이런 기능은 반드시 있어야 한다"는 약속(계약)만 정하고 실제 내용은 비워 둔 틀입니다. 파이썬에서는 Protocol로 표현합니다. 사용하는 쪽은 이 약속만 믿으면 되므로, 구현체를 자유롭게 갈아끼울 수 있습니다.
요청 스코프Request Scope
웹 서버가 요청 하나를 처리하는 동안만 유효한 생명 범위입니다. 이 범위 안에서 만든 값은 그 요청이 끝나면 정리됩니다. Depends가 요청마다 자원을 만들고 뒤처리하는 단위가 바로 이것입니다.
싱글턴Singleton
프로그램 전체에서 인스턴스를 딱 하나만 만들어 계속 재사용하는 방식입니다. 커넥션 풀처럼 매번 새로 만들면 낭비인 무거운 자원에 씁니다. Depends에는 없고 컨테이너 방식에는 내장돼 있어, 두 도구를 비교하는 기준이 됩니다.

1. 크롤러에 왜 의존성 주입인가

크롤러의 본질적인 어려움은 크롤링 자체가 아니라 사이트마다 전부 다르다는 데 있습니다. URL 구조가 다르고, HTML 셀렉터가 다르고, 로그인·차단 정책이 다릅니다. 이 다름을 방치하면 if site == "musinsa" 분기가 파이프라인 전체로 번집니다.

그래서 크롤링 파이프라인을 설계할 때 사이트별 차이를 CrawlerAdapter 인터페이스 뒤에 격리했습니다. 파이프라인(수집→정제→변환→저장)은 인터페이스만 알고, 구현은 사이트 수만큼 존재합니다. 여기까지가 Adapter 패턴이고, 남는 질문이 하나 있습니다. 실행 시점에 어떤 구현체를 골라서, 어떻게 넣어주는가. 그 "고르고 넣어주는" 일을 코드 바깥으로 빼는 것이 의존성 주입(DI)의 전부입니다. 도구는 여러 가지지만 원리는 하나입니다 — 사용하는 쪽은 만들지 않는다.

2. Depends는 마법이 아니다 — 시그니처가 그래프가 된다

FastAPI의 Depends는 프레임워크가 미리 준비한 마법 키워드가 아닙니다. 동작은 세 단계로 정리됩니다.

첫째, 라우트를 등록하는 순간 FastAPI는 inspect로 핸들러의 시그니처를 읽어 의존성 트리(내부적으로 Dependant 객체)를 만듭니다. 파라미터의 타입 힌트와 Depends(...) 표시를 보고, 의존성 함수의 시그니처도 재귀적으로 읽습니다. 경로·쿼리·헤더 파라미터가 해석되는 것과 완전히 같은 메커니즘입니다. 둘째, 요청이 들어오면 트리를 잎부터 차례로 실행해서 값을 채웁니다. 셋째, 같은 요청 안에서는 같은 의존성 함수의 결과를 캐시해 한 번만 실행합니다.

from typing import Annotated
import httpx
from fastapi import Depends, FastAPI

app = FastAPI()

async def get_client():
    async with httpx.AsyncClient(timeout=10.0) as client:
        yield client  # 응답이 나간 뒤 여기로 돌아와 정리됩니다

async def get_html(url: str,
                   client: Annotated[httpx.AsyncClient, Depends(get_client)]) -> str:
    r = await client.get(url)
    r.raise_for_status()
    return r.text

@app.get("/preview")
async def preview(html: Annotated[str, Depends(get_html)]):
    return {"length": len(html)}

주목할 부분이 두 곳입니다. get_html의 url: str은 어디서 올까요 — 쿼리 파라미터입니다. 의존성 함수의 시그니처도 라우트와 똑같이 해석되기 때문에, /preview?url=...의 값이 핸들러를 거치지 않고 의존성으로 직접 흘러듭니다. 그리고 get_client의 yield — 한 번 양보한 뒤 응답이 전송되고 나서 async with 블록이 닫힙니다. 커넥션 정리 같은 뒤처리를 의존성 안에 함께 두는 방식이고, DB 세션 패턴(get_db)이 전부 이 구조입니다.

*

같은 요청 안에서 같은 의존성은 기본으로 한 번만 실행되고 캐시됩니다. get_client를 세 곳에서 Depends해도 클라이언트는 하나입니다. 반대로 호출마다 새 값이 필요하면 Depends(fn, use_cache=False)를 명시해야 합니다. 이걸 모르면 "왜 값이 안 바뀌지"를 꽤 오래 헤매게 됩니다.

3. 어댑터를 주입으로 갈아끼운다

이제 크롤러 문제로 돌아갑니다. 인터페이스는 Protocol로 선언하고, 구현체 선택을 의존성 함수 하나에 몰아넣습니다.

from typing import Annotated, Protocol
from fastapi import Depends, FastAPI, HTTPException

class CrawlerAdapter(Protocol):
    async def fetch(self) -> str: ...
    def parse(self, html: str) -> list[dict]: ...

class MusinsaAdapter:
    def __init__(self, client: httpx.AsyncClient):
        self.client = client
    async def fetch(self) -> str:
        r = await self.client.get("https://magazine.musinsa.com")
        return r.text
    def parse(self, html: str) -> list[dict]:
        ...  # 사이트 고유의 셀렉터·정제 규칙은 전부 이 안에 갇힙니다

class W29Adapter: ...  # 같은 모양, 다른 규칙

_ADAPTERS: dict[str, type] = {"musinsa": MusinsaAdapter, "w29": W29Adapter}

def get_adapter(site: str,
                client: Annotated[httpx.AsyncClient, Depends(get_client)]) -> CrawlerAdapter:
    try:
        return _ADAPTERS[site](client)
    except KeyError:
        raise HTTPException(404, f"unknown site: {site}")

@app.get("/crawl/{site}")
async def crawl(adapter: Annotated[CrawlerAdapter, Depends(get_adapter)]):
    html = await adapter.fetch()
    return adapter.parse(html)

핵심은 라우트 시그니처에 site가 없다는 점입니다. 경로의 {site}는 get_adapter의 시그니처가 끌어가고, 라우트는 CrawlerAdapter라는 계약만 봅니다. 구현체 선택 + HTTP 클라이언트 조립이 get_adapter 한 곳에서 끝나므로, 새 사이트 추가는 클래스 하나 + 딕셔너리 한 줄입니다. 파이프라인 코드는 그대로 — 크롤링 파이프라인 사례에서 지키려던 "새 소스를 추가할 때 기존 코드를 건드리지 않는다"가 웹 계층에서도 유지됩니다.

4. dependency-injector와 비교 — 컨테이너 대 시그니처

fashion_scrapy는 API 서버가 아니라 스케줄러가 매일 돌리는 배치였습니다. 요청(request)이라는 실행 단위가 없으니 Depends를 쓸 자리도 없었고, 프레임워크 무관하게 동작하는 dependency-injector를 썼습니다. 객체 그래프를 컨테이너 클래스 한 곳에 명시적으로 선언하는 방식입니다.

from dependency_injector import containers, providers
from dependency_injector.wiring import Provide, inject

class Container(containers.DeclarativeContainer):
    config = providers.Configuration()
    client = providers.Singleton(httpx.AsyncClient, timeout=10.0)  # 앱 전역 1개
    adapter = providers.Selector(config.site,
        musinsa=providers.Factory(MusinsaAdapter, client=client),
        w29=providers.Factory(W29Adapter, client=client))
    pipeline = providers.Factory(CrawlPipeline, adapter=adapter)

@inject
async def run(pipeline: CrawlPipeline = Provide[Container.pipeline]):
    await pipeline.execute()

container = Container()
container.config.site.from_env("CRAWL_SITE", default="musinsa")
container.wire(modules=[__name__])

두 도구의 차이는 "어디에 그래프를 적는가"입니다. dependency-injector는 컨테이너 한 파일에 전체 조립도가 모입니다. Singleton·Factory·Resource로 수명(스코프)을 명시적으로 고를 수 있고, 설정(Configuration)과 선택(Selector)이 일급으로 지원됩니다. 대신 @inject와 wire()라는 부트스트랩 비용이 있고, 팀원이 처음 보면 낯설어합니다.

Depends는 반대로 그래프가 시그니처에 분산됩니다. 조립도를 한눈에 보긴 어렵지만, 경로·쿼리·헤더와 자연스럽게 섞이고 요청 스코프가 공짜로 따라옵니다. 다만 싱글턴 스코프는 내장되어 있지 않습니다 — 앱 전역 자원(커넥션 풀, 모델 핸들 등)은 lifespan에서 만들어 app.state에 두거나, 설정처럼 가벼운 것은 @lru_cache를 붙인 함수로 흉내 냅니다. 온프레미스 vLLM 서빙에서 게이트웨이를 만들 때도 백엔드 클라이언트는 이렇게 lifespan에서 한 번 만들고, 요청마다 바뀌는 것만 Depends로 풀었습니다.

5. 테스트에서 갈아끼우기 — dependency_overrides

주입 구조의 진짜 배당금은 테스트에서 나옵니다. FastAPI는 의존성 함수를 통째로 갈아끼우는 공식 통로 app.dependency_overrides를 제공합니다. 실제 사이트를 때리지 않고 어댑터 계약만 검증합니다.

from fastapi.testclient import TestClient

class FakeAdapter:
    async def fetch(self) -> str:
        return "<html>fixture</html>"
    def parse(self, html: str) -> list[dict]:
        return [{"title": "고정된 기사"}]

def test_crawl_returns_parsed_articles():
    app.dependency_overrides[get_adapter] = FakeAdapter
    try:
        res = TestClient(app).get("/crawl/musinsa")
        assert res.json() == [{"title": "고정된 기사"}]
    finally:
        app.dependency_overrides.clear()

오버라이드 대상은 함수 객체 자체(get_adapter)가 키입니다. 네트워크도, 실제 어댑터도, get_client도 전부 우회됩니다 — get_adapter가 통째로 대체되니 그 하위 의존성도 실행되지 않습니다. dependency-injector에도 같은 용도의 container.override()가 있어서, 배치 쪽 테스트도 같은 감각으로 작성할 수 있었습니다.

*

dependency_overrides는 앱 전역 딕셔너리입니다. 테스트가 끝나면 반드시 clear()로 되돌리세요. pytest라면 fixture의 yield 뒤에서 정리하는 편이 안전합니다. 어느 테스트의 가짜 어댑터가 다른 테스트로 새어 들어가는 순간, 실패가 실행 순서에 따라 랜덤하게 보이기 시작합니다.

6. 무엇을 언제 쓰나

제 기준은 단순합니다. 실행 단위가 HTTP 요청이면 Depends로 충분합니다. 요청 스코프·정리(yield)·오버라이드까지 내장이고, 의존성이 곧 문서(OpenAPI 파라미터)로 이어집니다. 배치·CLI거나 웹 프레임워크 밖에서도 같은 그래프를 써야 하면 컨테이너(dependency-injector)가 맞습니다. 조립도가 한 파일에 모이고 수명 제어가 명시적입니다.

도구가 무엇이든 원리는 같습니다. 정산 모듈을 리팩토링할 때도 결론은 결국 — 시계와 수수료 정책을 밖에서 주입하고 나서야 테스트가 가능해졌다는 것이었습니다. 사용하는 쪽은 만들지 않는다. 이 한 문장을 지키면, 크롤러 어댑터든 LLM 백엔드든 갈아끼우는 일은 설정 한 줄의 문제가 됩니다.
