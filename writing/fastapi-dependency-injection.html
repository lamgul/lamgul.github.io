FastAPI 의존성 주입 — 크롤러 어댑터를 갈아끼우는 법

2025년 2월• FastAPI · Python · DI

크롤링 파이프라인을 설계하면서 사이트별 크롤러를 누가 고르고 누가 넣어주는가를 오래 고민했습니다. fashion_scrapy에서는 dependency-injector 라이브러리로 풀었고, 이후 FastAPI를 쓰면서는 같은 일을 내장된 Depends로 하게 됐습니다. 그때 Depends가 실제로 어떻게 동작하는지 뜯어본 내용과, 어댑터 교체라는 같은 문제를 두 도구로 각각 풀어 본 기록을 남겨 둡니다.

먼저 짚고 가기
의존성 주입Dependency Injection
객체가 협력할 다른 객체를 자기 안에서 직접 생성하지 않고, 생성자 인자나 함수 파라미터로 외부에서 건네받게 하는 설계 원칙입니다. 제어의 역전(IoC)을 객체 생성에 적용한 형태로, "무엇을 쓸지 고르고 조립하는 책임"을 사용하는 코드 바깥으로 옮깁니다. 컨테이너나 프레임워크가 있어야 성립하는 것은 아니어서 인자로 넘기는 순간 이미 주입이고, Depends나 dependency-injector는 그 조립을 대신해 주는 도구일 뿐입니다.
어댑터 패턴Adapter Pattern
호출하는 쪽이 기대하는 인터페이스와 실제 구현이 가진 인터페이스가 다를 때, 그 사이에 변환 계층을 두어 호출을 위임하는 구조 패턴(GoF)입니다. 이 글에서는 사이트마다 다른 URL·셀렉터·차단 정책을 MusinsaAdapter·W29Adapter가 각각 흡수하고, 바깥에는 fetch()/parse()라는 같은 시그니처만 남깁니다. 통일되는 것은 시그니처까지이며 응답 지연이나 실패하는 방식까지 같아지지는 않으므로, 타임아웃·재시도 같은 정책은 어댑터 바깥에서 따로 정해야 합니다.
인터페이스Interface / Protocol
구현은 비워 두고 메서드 이름과 시그니처만 정해 둔 타입 수준의 계약입니다. 파이썬의 typing.Protocol은 구조적 부분 타이핑(PEP 544)이라, 아래 MusinsaAdapter처럼 CrawlerAdapter를 상속하지 않아도 fetch()와 parse()만 갖추면 그 타입으로 인정됩니다. 다만 이 판정은 mypy 같은 정적 타입 검사기가 하는 일이라 런타임에는 강제되지 않고, @runtime_checkable을 붙여 isinstance로 확인해도 메서드가 존재하는지만 볼 뿐 시그니처가 맞는지까지는 보지 않습니다.
요청 스코프Request Scope
HTTP 요청 하나가 들어와 응답이 나갈 때까지로 한정된 객체 수명 범위입니다. FastAPI는 이 범위 안에서 의존성 함수의 결과를 캐시하므로, 같은 함수를 여러 곳에서 Depends해도 요청당 한 번만 실행됩니다(기본값 use_cache=True). yield로 값을 내보낸 의존성은 응답이 전송된 뒤에야 yield 아래의 정리 코드가 실행되고, 다음 요청은 이 캐시를 물려받지 않아 처음부터 다시 만듭니다.
싱글턴Singleton
인스턴스를 한 번만 만들어 두고 이후의 모든 요청이 같은 참조를 돌려받게 하는 수명 범위이자 생성 패턴입니다. 커넥션 풀처럼 만드는 비용이 크고 상태를 공유해야 이득인 자원에 씁니다. Depends에는 이 스코프가 내장돼 있지 않아 lifespan과 app.state, 또는 @lru_cache를 붙인 함수로 대신하고, dependency-injector에는 providers.Singleton으로 들어 있습니다. 여기서 "하나"는 프로세스 단위라, 워커를 여러 개 띄우면 워커 수만큼 생긴다는 점은 따로 염두에 둬야 합니다.

1. 크롤러에 왜 의존성 주입인가

크롤러를 만들면서 힘들었던 건 크롤링 자체보다 사이트마다 전부 다르다는 점이었습니다. URL 구조가 다르고, HTML 셀렉터가 다르고, 로그인·차단 정책이 다릅니다. 이 차이를 그대로 두면 if site == "musinsa" 분기가 파이프라인 전체로 번져 나갑니다.

그래서 크롤링 파이프라인을 설계할 때 사이트별 차이를 CrawlerAdapter 인터페이스 뒤에 격리했습니다. 파이프라인(수집→정제→변환→저장)은 인터페이스만 알고, 구현은 사이트 수만큼 두었습니다. 여기까지가 Adapter 패턴인데, 실행 시점에 어떤 구현체를 골라서 어떻게 넣어줄지는 여전히 남는 문제였습니다. 그 고르고 넣어주는 일을 코드 바깥으로 빼는 것이 의존성 주입(DI)이 하는 일입니다. 도구는 여러 가지지만, 사용하는 쪽이 자기가 쓸 것을 직접 만들지 않는다는 원리는 같습니다.

2. Depends는 마법이 아니다 — 시그니처가 그래프가 된다

처음엔 Depends가 프레임워크가 알아서 해 주는 마법 같은 키워드인 줄 알았습니다. 소스를 따라가 보니 동작은 세 단계로 정리됐습니다.

먼저 라우트를 등록하는 순간, FastAPI는 inspect로 핸들러의 시그니처를 읽어 의존성 트리(내부적으로 Dependant 객체)를 만듭니다. 파라미터의 타입 힌트와 Depends(...) 표시를 보고, 의존성 함수의 시그니처도 재귀적으로 읽습니다. 경로·쿼리·헤더 파라미터가 해석되는 것과 같은 메커니즘입니다. 그다음 요청이 들어오면 트리를 잎부터 차례로 실행해서 값을 채우고, 같은 요청 안에서는 같은 의존성 함수의 결과를 캐시해 한 번만 실행합니다.

from typing import Annotated
import httpx
from fastapi import Depends, FastAPI

app = FastAPI()

async def get_client():
    async with httpx.AsyncClient(timeout=10.0) as client:
        yield client  # 응답이 나간 뒤 여기로 돌아와 정리됩니다

async def get_html(url: str,
                   client: Annotated[httpx.AsyncClient, Depends(get_client)]) -> str:
    r = await client.get(url)
    r.raise_for_status()
    return r.text

@app.get("/preview")
async def preview(html: Annotated[str, Depends(get_html)]):
    return {"length": len(html)}

이 코드에서 한참 들여다본 곳이 두 군데입니다. 하나는 get_html의 url: str인데, 이 값은 쿼리 파라미터에서 옵니다. 의존성 함수의 시그니처도 라우트와 똑같이 해석되기 때문에, /preview?url=...의 값이 핸들러를 거치지 않고 의존성으로 바로 들어옵니다. 다른 하나는 get_client의 yield로, 여기서 한 번 양보한 뒤 응답이 전송되고 나서야 async with 블록이 닫힙니다. 커넥션 정리 같은 뒤처리를 의존성 안에 함께 두는 방식이고, 흔히 쓰는 DB 세션 패턴(get_db)도 같은 구조였습니다.

*

같은 요청 안에서 같은 의존성은 기본으로 한 번만 실행되고 캐시됩니다. get_client를 세 곳에서 Depends해도 클라이언트는 하나입니다. 호출마다 새 값이 필요하면 Depends(fn, use_cache=False)를 명시해야 하는데, 저는 이걸 모르고 "왜 값이 안 바뀌지"를 한동안 헤맸습니다.

3. 어댑터를 주입으로 갈아끼운다

처음의 크롤러 문제로 돌아오면, 인터페이스는 Protocol로 선언하고 구현체를 고르는 일은 의존성 함수 하나에 모아 두었습니다.

from typing import Annotated, Protocol
from fastapi import Depends, FastAPI, HTTPException

class CrawlerAdapter(Protocol):
    async def fetch(self) -> str: ...
    def parse(self, html: str) -> list[dict]: ...

class MusinsaAdapter:
    def __init__(self, client: httpx.AsyncClient):
        self.client = client
    async def fetch(self) -> str:
        r = await self.client.get("https://magazine.musinsa.com")
        return r.text
    def parse(self, html: str) -> list[dict]:
        ...  # 사이트 고유의 셀렉터·정제 규칙은 전부 이 안에 갇힙니다

class W29Adapter: ...  # 같은 모양, 다른 규칙

_ADAPTERS: dict[str, type] = {"musinsa": MusinsaAdapter, "w29": W29Adapter}

def get_adapter(site: str,
                client: Annotated[httpx.AsyncClient, Depends(get_client)]) -> CrawlerAdapter:
    try:
        return _ADAPTERS[site](client)
    except KeyError:
        raise HTTPException(404, f"unknown site: {site}")

@app.get("/crawl/{site}")
async def crawl(adapter: Annotated[CrawlerAdapter, Depends(get_adapter)]):
    html = await adapter.fetch()
    return adapter.parse(html)

써 놓고 보니 라우트 시그니처에 site가 없습니다. 경로의 {site}는 get_adapter의 시그니처가 가져가고, 라우트는 CrawlerAdapter라는 계약만 봅니다. 구현체 선택과 HTTP 클라이언트 조립이 get_adapter 한 곳에서 끝나기 때문에, 새 사이트를 붙일 때 하는 일은 클래스 하나를 쓰고 딕셔너리에 한 줄을 더하는 것이었습니다. 파이프라인 코드는 건드리지 않았고, 크롤링 파이프라인 사례에서 지키려던 "새 소스를 추가할 때 기존 코드를 건드리지 않는다"가 웹 계층에서도 그대로 유지됐습니다.

4. dependency-injector와 비교 — 컨테이너 대 시그니처

fashion_scrapy는 API 서버가 아니라 스케줄러가 매일 돌리는 배치였습니다. 요청(request)이라는 실행 단위가 없어서 Depends를 쓸 자리도 없었고, 프레임워크와 무관하게 동작하는 dependency-injector를 골랐습니다. 객체 그래프를 컨테이너 클래스 한 곳에 명시적으로 선언하는 방식입니다.

from dependency_injector import containers, providers
from dependency_injector.wiring import Provide, inject

class Container(containers.DeclarativeContainer):
    config = providers.Configuration()
    client = providers.Singleton(httpx.AsyncClient, timeout=10.0)  # 앱 전역 1개
    adapter = providers.Selector(config.site,
        musinsa=providers.Factory(MusinsaAdapter, client=client),
        w29=providers.Factory(W29Adapter, client=client))
    pipeline = providers.Factory(CrawlPipeline, adapter=adapter)

@inject
async def run(pipeline: CrawlPipeline = Provide[Container.pipeline]):
    await pipeline.execute()

container = Container()
container.config.site.from_env("CRAWL_SITE", default="musinsa")
container.wire(modules=[__name__])

두 도구를 다 써 보고 느낀 차이는 그래프를 어디에 적느냐였습니다. dependency-injector는 컨테이너 한 파일에 전체 조립도가 모입니다. Singleton·Factory·Resource로 수명(스코프)을 명시적으로 고를 수 있고, 설정(Configuration)과 선택(Selector)도 일급으로 지원됩니다. 대신 @inject와 wire()를 갖춰 두는 부트스트랩 비용이 있고, 이 코드를 처음 보는 사람에게는 낯선 방식입니다.

Depends는 반대로 그래프가 시그니처에 흩어져 있습니다. 조립도를 한눈에 보기는 어렵지만, 경로·쿼리·헤더와 자연스럽게 섞이고 요청 스코프는 따로 준비하지 않아도 따라옵니다. 다만 싱글턴 스코프는 내장되어 있지 않아서, 앱 전역 자원(커넥션 풀, 모델 핸들 등)은 lifespan에서 만들어 app.state에 두거나 설정처럼 가벼운 것은 @lru_cache를 붙인 함수로 흉내 냈습니다. 온프레미스 vLLM 서빙에서 게이트웨이를 만들 때도 백엔드 클라이언트는 lifespan에서 한 번 만들고, 요청마다 바뀌는 것만 Depends로 풀었습니다.

5. 테스트에서 갈아끼우기 — dependency_overrides

주입 구조가 가장 고마웠던 건 테스트를 쓸 때였습니다. FastAPI는 의존성 함수를 통째로 갈아끼우는 공식 통로로 app.dependency_overrides를 제공합니다. 덕분에 실제 사이트를 호출하지 않고 어댑터 계약만 검증할 수 있었습니다.

from fastapi.testclient import TestClient

class FakeAdapter:
    async def fetch(self) -> str:
        return "<html>fixture</html>"
    def parse(self, html: str) -> list[dict]:
        return [{"title": "고정된 기사"}]

def test_crawl_returns_parsed_articles():
    app.dependency_overrides[get_adapter] = FakeAdapter
    try:
        res = TestClient(app).get("/crawl/musinsa")
        assert res.json() == [{"title": "고정된 기사"}]
    finally:
        app.dependency_overrides.clear()

오버라이드의 키는 함수 객체 자체(get_adapter)입니다. get_adapter가 통째로 대체되기 때문에 그 아래에 달린 get_client도, 실제 어댑터도, 네트워크 호출도 실행되지 않습니다. dependency-injector에도 같은 용도의 container.override()가 있어서, 배치 쪽 테스트도 비슷한 감각으로 작성했습니다.

*

dependency_overrides는 앱 전역 딕셔너리라서, 테스트가 끝나면 clear()로 되돌려 놓아야 합니다. pytest라면 fixture의 yield 뒤에서 정리하는 편이 안전합니다. 한 테스트의 가짜 어댑터가 다른 테스트로 새어 들어가면, 실패가 실행 순서에 따라 들쭉날쭉하게 보이기 시작합니다.

6. 무엇을 언제 쓰나

지금 제가 쓰는 기준은 단순합니다. 실행 단위가 HTTP 요청이면 Depends로 충분했습니다. 요청 스코프와 정리(yield), 오버라이드가 모두 내장이고, 의존성이 그대로 문서(OpenAPI 파라미터)로 이어집니다. 배치나 CLI처럼 웹 프레임워크 밖에서도 같은 그래프를 써야 하면 컨테이너(dependency-injector) 쪽이 편했습니다. 조립도가 한 파일에 모이고 수명 제어가 명시적입니다.

도구가 무엇이든 원리는 같았습니다. 정산 모듈을 리팩토링할 때도 시계와 수수료 정책을 밖에서 주입하고 나서야 테스트를 쓸 수 있었습니다. 사용하는 쪽이 자기가 쓸 것을 직접 만들지 않게 해 두면, 크롤러 어댑터든 LLM 백엔드든 갈아끼우는 일이 설정을 바꾸는 정도로 줄어듭니다.
