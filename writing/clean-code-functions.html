클린코드 · 함수 — 한 가지 일의 실제 크기

2020년 6월• 클린코드 · 함수 분리 · 추상화 수준

정산 로직을 리팩토링하다가 80줄짜리 함수 앞에서 한참 멈춰 있던 때가 있었습니다. "함수는 한 가지 일만 해야 한다"는 말은 알고 있었는데, 그 '한 가지'가 얼마만 한 크기인지는 몰랐습니다. 그때 기준으로 삼았던 추상화 수준, 부수효과, 인자를 실제로 밟은 순서대로 코드와 함께 적어 둡니다. 쪼갰다가 오히려 더 읽기 어려워졌던 경우도 같이 남깁니다.

먼저 짚고 가기
추상화 수준Abstraction Level
"무엇을 하는가(정산한다)"와 "어떻게 하는가(수수료율 0.05를 곱한다)" 사이의 높낮이입니다. 지도의 축척과 같아서, 나라 전체를 보는 지도와 골목을 보는 지도를 한 장에 겹쳐 그리면 헷갈리듯 한 함수 안에 서로 다른 층위가 섞이면 읽기 어려워집니다. 이 글에서 "한 가지 일"의 기준으로 삼는 것도 이 층위입니다.
부수효과Side Effect
함수가 값을 돌려주는 것 말고 바깥 세상을 바꾸는 일 — DB에 저장하거나, 슬랙 알림을 보내는 것 같은 동작입니다. 같은 입력에도 결과가 달라지거나 흔적을 남기기 때문에, 계산 로직과 섞이면 테스트가 어려워집니다.
시그니처Signature
함수의 이름·인자·반환 타입만 모아 놓은 "겉모습"입니다. commission_fee(amount, grade) -> int처럼, 본문을 읽지 않아도 무엇을 받아 무엇을 내놓는지 알려 주는 명함이라 보면 됩니다. 이 글에서 함수를 나누는 목적도 이 명함을 정직하게 만드는 데 있습니다.
특성화 테스트Characterization Test
"올바른 동작"이 아니라 "지금 이대로의 동작"을 그대로 사진 찍어 두는 테스트입니다. 낡은 코드를 손대기 전에 현재 결과를 고정해 두면, 고치는 중에 동작이 바뀌었는지 바로 알아챌 수 있습니다.
목(mock)Mock
테스트할 때 진짜 DB나 외부 서비스 대신 끼워 넣는 가짜 대역입니다. 대역을 많이 준비해야 한다는 건 그 함수가 바깥 세계에 그만큼 얽혀 있다는 신호라서, 이 글에서는 mock이 사라지는 것을 좋은 분리의 증거로 봅니다.

80줄 함수는 어떻게 태어나는가

그 함수도 처음부터 80줄은 아니었습니다. 처음엔 15줄이었습니다. 판매자 등급별 수수료가 들어오며 18줄이 붙었고, 프로모션 할인이 생기며 16줄, 재정산 요구가 생기며 플래그 인자가 하나, 슬랙 알림이 붙으며 다시 8줄이 늘었습니다. 한 번 한 번의 추가는 그때마다 합리적인 판단이었습니다. "이 다섯 줄 때문에 함수를 새로 파나" 싶은 순간이 열 번쯤 쌓이면 이런 모양이 됩니다.

def settle_order(order_id, retry=False, notify=True):
    # ── 조회 (11줄) ──
    order = order_repo.find(order_id)
    if order is None:
        raise OrderNotFound(order_id)
    seller = seller_repo.find(order.seller_id)
    promotions = promo_repo.find_active(order.created_at)
    # ── 검증 (9줄) ──
    if order.status != "PAID":
        raise NotSettleable(order_id, order.status)
    if not retry and settlement_repo.exists(order_id):
        raise AlreadySettled(order_id)
    # ── 수수료 계산 (18줄) ──
    if seller.grade == "PARTNER":
        fee_rate = 0.05
    elif seller.grade == "GENERAL" and order.amount >= 1_000_000:
        fee_rate = 0.07
    else:
        fee_rate = 0.09
    fee = order.amount * fee_rate
    # ── 할인 반영 (16줄) ──
    discount = 0
    for promo in promotions:
        if promo.applies_to(order):
            discount += promo.amount_for(order)
    # ── 저장 (13줄) ──
    settlement = Settlement(order_id, fee, discount, order.amount - fee - discount)
    settlement_repo.save(settlement)
    order.mark_settled()
    order_repo.save(order)
    # ── 알림 (8줄) ──
    if notify:
        slack.send(f"정산 완료: {order_id}")
    return settlement

실제 80줄을 다 옮기는 대신 구조만 남겼습니다. 다시 보니 눈에 들어온 건 코드보다 구획 주석이었습니다. # ── 조회 ──, # ── 검증 ── 같은 주석을 달았다는 건, 이 함수 안에 서로 다른 일이 여섯 개 있다고 작성자가 스스로 적어 둔 셈입니다. 그래서 구획 주석은 리팩토링이 필요하다는 표시이면서 동시에 설계도이기도 했습니다. 각 주석이 그대로 추출할 함수의 이름 후보가 되니까요.

"한 가지 일"의 기준 — 줄 수가 아니라 추상화 수준

저도 한동안 "함수는 5줄 이내로", "10줄 넘으면 쪼개라" 같은 줄 수 규칙을 지키려 했는데 잘 되지 않았습니다. 같은 20줄이라도 어떤 20줄은 한 가지 일이고(복잡한 수식 하나를 옮겨 적은 것), 어떤 20줄은 세 가지 일이었기 때문입니다(조회하고, 계산하고, 저장하는 것). 줄 수는 결과이지 기준이 아니었습니다.

그 뒤로 제가 기준으로 삼은 것은 함수 본문의 모든 문장이 함수 이름보다 정확히 한 단계 낮은 추상화 수준에 있는가입니다. 위 함수가 걸린 지점도 길이가 아니었습니다. "정산한다"라는 이름 아래에 "판매자 등급이 PARTNER면 수수료율 0.05" 같은 세 단계 아래의 문장이 섞여 있었고, 그러면 읽는 사람은 문단 사이가 아니라 층 사이를 오르내리게 됩니다. 이 함수를 읽을 때마다 앞부분을 자꾸 놓쳤던 것도 그 때문이라고 생각합니다.

이 기준으로 목표 형태를 먼저 그려 봤습니다.

def settle_order(order_id: OrderId) -> Settlement:
    target = load_settlement_target(order_id)   # 조회 (I/O)
    ensure_settleable(target)                   # 검증
    result = calculate_settlement(target)       # 계산 (순수)
    return commit_settlement(target, result)    # 반영 (I/O)

네 줄이 모두 같은 층위에 있습니다 — 조회, 검증, 계산, 반영. 정산 담당자에게 업무를 설명할 때 쓰던 문장과 거의 그대로 겹쳤습니다. 클린코드가 말하는 내려가기 규칙(stepdown rule)이 이런 뜻이었구나 싶었습니다. 코드를 위에서 아래로 읽을 때 함수 하나를 지날 때마다 추상화 수준이 한 계단씩 내려가면 되고, 그 계단이 유지되는 한 함수가 4줄이든 20줄이든 한 가지 일을 하는 것으로 봅니다.

실제 리팩토링 순서 — 주석이 함수 이름이 된다

목표 형태를 그려 놓고도 어디부터 손대야 할지 몰라 며칠을 미뤘습니다. 정산 코드에서는 세 단계로 옮겼습니다.

1단계 — 동작을 먼저 고정합니다. 리팩토링은 정의상 동작을 바꾸지 않는 작업이므로, 바뀌지 않았음을 확인할 수단이 먼저 필요했습니다. 80줄 함수에 단위 테스트가 있을 리 없어서, 대표 입력 몇 개에 대한 현재 출력을 그대로 스냅샷으로 잡는 특성화 테스트(characterization test)를 만들었습니다. 이 테스트가 지키는 건 "옳은 동작"이 아니라 "현재 동작"입니다. 처음엔 그게 이상하게 느껴졌는데, 고치는 중에 동작이 달라졌는지 알아내는 데는 이것으로 충분했습니다.

*

추출 1건 = 커밋 1건. 함수 하나를 빼낼 때마다 테스트를 돌리고 커밋했습니다. 번거롭긴 하지만, 여섯 개를 한꺼번에 빼다가 테스트가 깨지면 어느 추출 때문인지 알 수 없습니다. 커밋이 잘게 쪼개져 있으면 잘못됐을 때 git revert 한 번으로 돌아올 수 있습니다.

2단계 — 구획 주석을 그대로 Extract Method 합니다. 이름은 따로 고민하지 않아도 됐습니다. 주석이 이미 이름이었습니다. # 수수료 계산은 commission_fee가 되고, # 검증은 ensure_settleable이 됐습니다.

3단계 — 추출이 드러낸 것을 봅니다. 함수를 빼내고 나서야 원래 코드에서는 보이지 않던 것이 보였습니다. 수수료·할인 계산을 빼내려니 인자로 order, seller, promotions가 줄줄이 필요했습니다. 같은 인자 묶음이 함수마다 함께 다닌다면 이름 붙지 않은 개념이 숨어 있다는 뜻입니다. 그렇게 SettlementTarget이라는 데이터 덩어리를 찾았고, 계산 함수들은 이렇게 정리됐습니다.

@dataclass(frozen=True)
class SettlementTarget:
    order: Order
    seller: Seller
    promotions: list[Promotion]

def calculate_settlement(t: SettlementTarget) -> SettlementResult:
    fee = commission_fee(t.order.amount, t.seller.grade)
    discount = promotion_discount(t.order, t.promotions)
    return SettlementResult(fee, discount, payout=t.order.amount - fee - discount)

def commission_fee(amount: int, grade: SellerGrade) -> int:
    # 18줄짜리 if-elif가 정책 테이블 하나로 줄어든다
    rate = FEE_RATES[grade].rate_for(amount)
    return int(amount * rate)

commission_fee의 시그니처를 다시 봤습니다. 금액과 등급을 받아 수수료를 돌려준다 — 함수가 무엇에 의존하고 무엇을 내놓는지 시그니처만으로 드러납니다. 80줄 버전에서는 같은 정보가 본문 어딘가에 파묻혀 있어서 알아내려면 읽는 수밖에 없었습니다. 함수를 나눠서 얻은 것은 짧아졌다는 사실보다 의존성이 시그니처로 올라왔다는 점이었습니다.

부수효과를 가장자리로 — 계산과 I/O의 분리

목표 형태의 네 줄에 (I/O)와 (순수)를 표시해 둔 이유가 있습니다. 80줄 함수에서 저를 가장 힘들게 한 건 길이가 아니라, 수수료 정책 하나를 검증하려고 DB 리포지토리 네 개와 슬랙 클라이언트까지 mock 해야 한다는 점이었습니다. 계산(순수 로직)과 부수효과(DB 쓰기, 알림)가 한 몸이라서, 가장 자주 바뀌고 가장 자주 테스트해야 하는 수수료·할인 로직이 하필 가장 테스트하기 어려운 자리에 있었습니다.

그래서 쪼갤 때 그냥 여섯 조각으로 나누지 않고, 순수한 계산을 가운데로 모으고 부수효과를 양쪽 가장자리로 밀었습니다. 조회(읽기 I/O) → 계산(순수) → 반영(쓰기 I/O). 흔히 functional core, imperative shell이라 부르는 배치입니다.

def commit_settlement(t: SettlementTarget, r: SettlementResult) -> Settlement:
    # 쓰기 부수효과는 전부 여기, 한 곳에만 있다
    settlement = settlement_repo.save(Settlement.of(t.order.id, r))
    order_repo.save(t.order.mark_settled())
    events.publish(SettlementCompleted(t.order.id))  # 알림은 구독자의 일로
    return settlement

# 핵심 로직의 테스트에는 mock이 하나도 없다
def test_partner_fee_rate():
    t = target(grade="PARTNER", amount=200_000)
    assert calculate_settlement(t).fee == 10_000

이렇게 배치한 이유는 작업을 하면서 알게 됐습니다. 부수효과가 여기저기 흩어져 있으면 이 함수를 지금 불러도 되는지를 호출부에서 판단할 수 없습니다. "두 번 불러도 되나, 트랜잭션 안에서 불러도 되나" 같은 것을 알려면 본문을 다 읽어야 했습니다. 부수효과를 commit_settlement 한 곳에 모으고 나면 나머지 함수들은 값을 받아 값을 돌려주는 일만 하므로, 몇 번을 부르든 어디서 부르든 안전합니다. 저는 "한 가지 일"을 계산이거나 부수효과이거나, 둘 다는 아닌 것 정도로 이해하고 있습니다. 이후 정산 채널이 늘어나 계산부를 Template Method와 Strategy로 재구성할 때도, 이 분리가 되어 있어서 교체 대상이 순수 함수뿐이라 작업이 수월했습니다.

인자 — 개수보다 정직함

클린코드는 인자 0~2개를 권하고 3개부터 의심하라고 합니다. 다만 제가 걸렸던 건 개수보다 인자가 함수에 대해 거짓말을 하는가 쪽이었습니다. 원래 시그니처를 다시 옮겨 봅니다.

# Before — 호출부는 암호가 된다
settle_order(order_id, True, False)   # True가 뭐고 False가 뭐더라?

# After — 다른 일에는 다른 이름을
def settle_order(order_id: OrderId) -> Settlement: ...
def resettle_order(order_id: OrderId) -> Settlement:
    # 기존 정산 무효화 + 재정산: 검증 규칙 자체가 다른 별개의 유스케이스
    ...

돌아보면 retry=False 같은 불리언 플래그 인자는 이 함수가 사실 두 개라는 표시였습니다. 플래그에 따라 검증 로직이 갈라진다면 그건 분기가 아니라 서로 다른 유스케이스입니다. 첫 정산과 재정산은 사전 조건도, 실패했을 때 처리도 다릅니다. 함수를 둘로 나누고 나니 각각의 이름이 정확해졌고, 호출부의 True, False가 무슨 뜻인지 되묻는 일도 없어졌습니다.

notify=True는 다른 종류의 문제였습니다. 정산 함수가 슬랙을 아는 순간, 알림 채널이 바뀔 때마다 정산 코드를 고쳐야 합니다. 의존 방향이 거꾸로 붙어 있었던 셈입니다. 그래서 위 코드처럼 SettlementCompleted 이벤트를 발행하는 것으로 바꾸고, 알림은 구독자의 일로 넘겼습니다. 인자 하나를 없애려던 작업이 모듈 경계를 긋는 일이 되어 있었습니다.

이른 추상화의 함정 — 잘못 쪼개면 더 나빠진다

여기까지는 잘 풀린 쪽이고, 잘못 쪼개서 원본보다 읽기 어려워진 경우도 있었습니다. 레거시 UseCase를 개선하다 만난, 규칙은 다 지켰는데 더 읽기 어려운 코드가 이런 모양이었습니다.

class SettlementService:
    def settle(self, order_id):
        self._load(order_id)      # self.order, self.seller를 채운다
        self._validate()          # self.order를 몰래 읽는다
        self._calc_fee()          # self.fee를 몰래 채운다
        self._apply_discount()    # self.fee를 몰래 고친다!
        self._save()              # 전부 다 몰래 읽는다

    def _calc_fee(self):
        # 시그니처가 아무것도 말해주지 않는다: 입력도 출력도 안 보인다
        self.fee = self.order.amount * self._rate_for(self.seller)

함수는 전부 5줄 이내이니 줄 수 규칙으로는 문제가 없습니다. 다만 모든 함수가 self의 필드로 상태를 주고받으면 그 필드들은 클래스 범위의 전역변수가 됩니다. _calc_fee는 _load 다음, _apply_discount 이전에만 호출해야 안전한데, 그 순서 제약은 어디에도 적혀 있지 않았습니다. 원본 80줄 함수는 적어도 위에서 아래로 읽으면 순서가 보였는데, 이 코드는 다섯 함수를 오가며 어떤 필드가 언제 채워지는지 따라가야 했습니다. 함수는 짧아졌지만 한 번에 이해해야 하는 범위는 오히려 넓어진 셈입니다. 쪼개는 목적이 짧음이 아니라 독립성이라면, 이 코드는 이름만 여섯 개로 늘리고 독립성은 얻지 못했습니다.

그래서 예외로 두는 경우가 하나 있습니다. 중간 상태를 자연스럽게 공유하는 순차 로직 — 파서, 초기화 시퀀스, 한 곳에서만 쓰이는 마이그레이션 스크립트 — 은 다소 긴 함수 하나가 어설프게 쪼갠 다섯 함수보다 나았습니다. 지역변수는 함수 안에 있을 때 가장 안전한 상태 공유 수단입니다. 쪼갰을 때 각 조각이 시그니처만으로 이해되고 독립적으로 테스트된다면 쪼개는 편이 맞고, 조각들이 필드나 호출 순서로 몰래 엮인다면 합쳐 두는 편이 맞습니다. 지금도 매번 헷갈리지만, 판정 기준은 줄 수가 아니라 추출 후 시그니처가 정직한가로 두고 있습니다.

*

구조는 세 번째 반복에서 세웁니다. 정산 리팩토링에서 Template Method와 Strategy를 넣은 시점은 처음이 아니라, 채널별 분기가 세 곳에서 같은 모양으로 반복되는 걸 확인한 뒤였습니다. 두 코드가 비슷해 보여도 바뀌는 이유가 다르면(우연한 중복) 묶고 나서 한쪽을 고칠 때 다른 쪽까지 끌려옵니다. 두 번까지는 중복을 견디고 세 번째에 패턴이 분명해졌을 때 추상화하는 편이, 되돌리는 비용까지 넣고 보면 대체로 저렴했습니다.

정리 — 함수를 쪼개기 전에 확인하는 것

80줄 함수에서 출발해 정리된 기준을 목록으로 남겨 둡니다.

  • 구획 주석은 추출할 함수의 이름 후보입니다. # ── 검증 ──이 보이면 그 주석을 그대로 함수 이름으로 옮겼습니다.
  • 기준은 줄 수가 아니라 추상화 수준입니다. 본문의 모든 문장이 함수 이름보다 한 단계만 낮으면, 길이와 무관하게 한 가지 일을 하는 함수로 봅니다.
  • 계산은 가운데로, 부수효과는 가장자리로 둡니다. 계산이거나 부수효과이거나, 둘 다는 아니게. 핵심 로직의 테스트에서 mock이 사라지면 잘 나뉜 것으로 봤습니다.
  • 불리언 플래그 인자는 함수 두 개를 합쳐 놓은 것입니다. 플래그에 따라 검증이 갈라지면 유스케이스가 다른 것이라 이름을 나눴습니다.
  • 추출한 조각들이 필드·호출 순서로 엮이면 잘못 쪼갠 것입니다. 시그니처가 입력과 출력을 정직하게 말하지 못하는 분리는 되돌렸습니다.
  • 추상화는 세 번째 반복에서 합니다. 우연한 중복을 묶으면 한쪽을 고칠 때 다른 쪽이 끌려옵니다.
  • 동작 고정 테스트 먼저, 추출 1건 = 커밋 1건. 리팩토링의 안전망은 신중함이 아니라 절차였습니다.

적어 놓고 보니 기준들이 가리키는 곳은 비슷했습니다. 함수를 나누는 일은 코드를 짧게 만드는 작업이라기보다 의존성과 의도를 시그니처라는 공개된 자리로 끌어올리는 작업에 가까웠습니다. 다음 편에서는 그 시그니처가 실패를 어떻게 말해야 하는지, 에러 처리로 이어가겠습니다.
