보안: OWASP Top 10 — 백엔드에서 실제로 막는 법

2024년 3월• 약 15분• OWASP · 인젝션 · 접근 통제

OWASP Top 10은 모두가 북마크하지만, 정작 코드로 옮기는 사람은 드뭅니다. 목록은 위험의 순위일 뿐, 그 자체로는 한 줄의 방어도 되지 못하기 때문입니다. SafeIn 산업안전 시스템 백엔드를 혼자 설계하며 보안 점검을 준비할 때, 제게 필요했던 건 열 개의 항목이 아니라 "이 중 무엇을 내 코드가 직접 결정하고, 어느 한 줄이 그것을 막는가"였습니다. 이 글은 Top 10에서 백엔드가 온전히 책임지는 항목만 골라, 막는 코드까지 내려갑니다.

보안 전문가들의 비영리 단체(OWASP)가 몇 년마다 발표하는, 웹 애플리케이션에서 가장 흔하고 위험한 취약점 10가지 순위표입니다. 이 글은 그 목록 자체가 아니라 "그중 백엔드 코드가 직접 막을 수 있는 항목"을 코드로 옮기는 데 초점을 둡니다.
인젝션Injection
사용자가 입력한 값이 단순한 데이터가 아니라 프로그램의 명령어로 잘못 해석되어 실행되는 공격입니다. 예를 들어 검색창에 넣은 글자가 데이터베이스 명령의 일부로 둔갑하는 식이죠. 이 글에서 가장 먼저, 가장 길게 다루는 위험입니다.
파라미터 바인딩Parameter Binding
데이터베이스에 명령을 보낼 때, "쿼리의 뼈대"와 "사용자가 넣은 값"을 아예 다른 통로로 따로 전달하는 방식입니다. 값은 언제나 값으로만 취급되어, 인젝션이 구조적으로 불가능해집니다 — 이 글이 제시하는 인젝션의 핵심 방어입니다.
접근 통제Access Control
로그인한 사용자가 "무엇을 볼 수 있고 무엇은 안 되는지"를 서버가 강제하는 규칙입니다. 도서관에서 회원증이 있어도 아무 서고나 들어갈 수 없는 것과 비슷합니다. 화면에서 버튼만 숨기는 것으로는 부족하고, 요청마다 서버가 다시 확인해야 한다는 것이 이 글의 요지입니다.
해시Hash
원본을 되돌릴 수 없는 고정 길이의 값으로 변환하는 일방향 계산입니다. 비밀번호는 나중에 다시 풀어 볼 필요가 없으므로, 암호화(복호화 가능) 대신 해시로 저장해 유출되어도 원문을 알 수 없게 만듭니다.

1. Top 10에서 백엔드의 몫

2021년 개정판의 열 개 항목은 프런트·백엔드·인프라에 걸쳐 있습니다. 클릭재킹이나 보안 설정 오류처럼 서버·브라우저·배포가 함께 책임지는 항목도 있죠. 그러나 그중 몇 개는 거의 전적으로 애플리케이션 코드가 결과를 결정합니다. 저는 그 항목들에만 집중하기로 했습니다 — 나머지에 대한 방어도, 결국은 이 코어를 막을 수 있느냐에서 갈리기 때문입니다.

  • A01 접근 통제 실패 — 2021년 판의 1위. 인증은 됐지만 넘보면 안 되는 것에 접근한다.
  • A03 인젝션 — 사용자 입력이 코드로 해석된다. SQL·OS 명령·NoSQL이 같은 뿌리다.
  • A07 인증 실패 — 신원을 허술하게 증명하거나, 무차별 대입을 방치한다.
  • A02 암호화 실패 — 2017년의 '민감정보 노출'이 이름을 바꿨다. 흘리거나, 평문으로 둔다.

이 넷을 고른 기준은 하나입니다. 방어의 마지막 선이 서버라는 것. 프런트에서 걸어 둔 어떤 검증도, 감춘 어떤 버튼도 공격자에게는 우회의 대상일 뿐입니다. 브라우저는 통제할 수 없고, 요청은 조작할 수 있으므로, 신뢰가 실제로 성립하는 유일한 장소는 서버 코드입니다. 무차별 대입 방어(레이트 리미팅), 전송 구간 암호화(TLS), 토큰 발급(JWT)은 각각 별도의 글로 다루고, 여기서는 인젝션·접근 통제·인증·민감정보를 코드로 막는 데 집중합니다.

2. 인젝션 — 코드와 데이터를 섞지 않기

인젝션의 뿌리는 한 문장으로 요약됩니다. 인터프리터가 코드와 데이터가 뒤섞인 문자열을 받는 것. SQL 엔진에게 "... WHERE email = '" + email + "'"를 넘기면, 엔진은 어디까지가 우리가 짠 쿼리이고 어디부터가 사용자 값인지 알 방법이 없습니다. 공격자가 email 자리에 ' OR '1'='1을 넣는 순간, 데이터였어야 할 값이 쿼리의 논리가 됩니다. 대흥정보에서 레거시 웹을 유지보수할 때, 이렇게 문자열을 이어 붙인 SQL은 곳곳에 있었습니다.

// 취약 — 문자열을 이어 붙여 SQL을 만든다
const sql = "SELECT * FROM users WHERE email = '" + email + "'";
db.query(sql);   // email = ' OR '1'='1  이면 전체 행이 나온다

// 안전 — 구조와 데이터를 분리한다 (파라미터 바인딩)
db.query("SELECT * FROM users WHERE email = $1", [email]);

// Prisma — 쿼리 빌더는 애초에 바인딩된다
await prisma.user.findUnique({ where: { email } });

해법은 이스케이프가 아니라 파라미터 바인딩입니다. 차이가 중요합니다 — 이스케이프는 위험한 문자를 우리가 일일이 골라 무력화하는 방식이라, 한 곳만 빠뜨려도 뚫립니다. 반면 파라미터 바인딩은 드라이버가 쿼리의 골격과 값을 아예 다른 채널로 데이터베이스에 보냅니다. 값은 항상 값으로만 취급되고, 무슨 문자가 들었든 다시 SQL로 해석될 경로 자체가 없습니다. 그래서 이건 "조심하는" 방어가 아니라 구조적으로 불가능하게 만드는 방어입니다.

ORM을 쓰면 대부분 이 바인딩이 기본값입니다. Prisma의 쿼리 빌더(findUnique, where)는 값을 자동으로 바인딩하고, 원시 SQL이 필요할 때도 태그드 템플릿은 안전하게 처리됩니다. 문제는 그 옆에 있는 탈출구입니다.

// 태그드 템플릿 — Prisma가 ${...}를 자동으로 바인딩한다 (안전)
await prisma.$queryRaw`SELECT * FROM users WHERE email = ${email}`;

// Unsafe — 완성된 문자열을 그대로 실행한다 (주입 가능)
await prisma.$queryRawUnsafe(`SELECT * FROM users WHERE email = '${email}'`);

이름 그대로입니다. $queryRaw는 ${email}을 바인딩 파라미터로 바꿔 주지만, $queryRawUnsafe는 우리가 이미 완성한 문자열을 받아 그대로 실행합니다. 인젝션은 SQL만의 문제도 아닙니다. NoSQL 연산자 주입, OS 명령 주입, LDAP 주입이 전부 같은 구조를 공유합니다 — 인터프리터에게 넘길 입력을 문자열 조립으로 만들지 않는다는 한 문장이 이 항목 전체의 방어입니다.

!

함정 — ORM은 방패이지 면죄부가 아닙니다. 바인딩할 수 없는 자리가 있기 때문입니다. 대표적으로 동적 정렬 컬럼(ORDER BY ${col})은 값이 아니라 식별자라 파라미터로 넘길 수 없어, 클라이언트가 준 컬럼명을 그대로 쓰면 주입 경로가 됩니다. 이런 자리는 반드시 허용된 컬럼 목록과 대조해서 통과시켜야 합니다. $queryRawUnsafe, 문자열로 만든 IN 리스트도 같은 예외 지대입니다.

3. 입력 검증 — 경계에서 계약을 강제한다

먼저 오해를 하나 걷어내겠습니다. 입력 검증은 인젝션의 주된 방어가 아닙니다. 인젝션은 앞서 본 파라미터 바인딩이 막고, 검증은 다른 일을 합니다 — 서버가 받아들일 입력의 모양을 경계에서 계약으로 못 박는 것이죠. 이건 심층 방어이자 정확성의 문제이고, 뒤에 나올 접근 통제와도 직접 맞닿습니다. NestJS라면 DTO와 class-validator가 이 계약을 선언적으로 표현합니다.

// signup.dto.ts — 경계에서 받을 수 있는 모양을 선언한다
export class SignupDto {
  @IsEmail()
  email: string;

  @IsString() @MinLength(10)
  password: string;

  @IsOptional() @IsString() @MaxLength(40)
  displayName?: string;
}

// main.ts — 전역으로 강제한다
app.useGlobalPipes(new ValidationPipe({
  whitelist: true,            // DTO에 없는 속성은 제거
  forbidNonWhitelisted: true, // 있으면 400으로 거절
  transform: true,            // 페이로드를 DTO 인스턴스로 변환
}));

여기서 보안상 가장 중요한 옵션은 whitelist입니다. 이유는 대량 할당(mass assignment) 때문입니다. 검증이 없으면 클라이언트가 요청 본문에 { email, password, role: "admin" }처럼 선언하지 않은 필드를 끼워 넣을 수 있고, 코드가 무심코 userRepo.create(dto)로 전체 객체를 넘기면 role까지 그대로 저장됩니다. whitelist: true는 DTO에 없는 속성을 잘라내고, forbidNonWhitelisted: true는 그런 속성이 오면 아예 400으로 거절합니다. 검증이 접근 통제의 첫 관문이 되는 지점입니다.

핵심 원칙은 방향입니다. 서버가 받을 것을 긍정형으로 열거하고, 나머지는 전부 거절합니다. "위험한 걸 골라 막는" 블랙리스트가 아니라, "허용된 것만 통과시키는" 화이트리스트여야 합니다. 블랙리스트는 공격자가 목록에 없는 한 가지만 찾으면 뚫리지만, 화이트리스트는 우리가 명시한 것 외에는 처음부터 존재하지 않기 때문입니다. 이 사고방식은 뒤의 모든 섹션에서 반복됩니다.

4. 접근 통제 — 수직과 수평

2021년 판에서 1위로 올라온 항목입니다. 실무에서 가장 자주, 가장 조용히 뚫리는 곳이기도 하죠. 접근 통제 실패는 두 축으로 나눠 보면 선명해집니다. 수직은 일반 사용자가 위쪽 권한(관리자 기능)에 닿는 것이고, 수평은 같은 권한의 사용자가 옆 사람의 자원에 닿는 것입니다.

수직 통제부터 봅니다. 관리자 API는 요청마다 서버에서 역할을 검사해야 합니다. 화면에서 관리자 버튼을 숨기는 건 접근 통제가 아니라 장식입니다 — 엔드포인트는 URL만 알면 직접 호출되니까요. NestJS라면 가드가 이 검사를 담당합니다.

// roles.guard.ts — 수직 통제: 요청마다 서버에서 역할을 확인한다
@Injectable()
export class RolesGuard implements CanActivate {
  canActivate(ctx: ExecutionContext): boolean {
    const required = this.reflector.getAllAndOverride<Role[]>('roles', [
      ctx.getHandler(), ctx.getClass(),
    ]);
    if (!required) return true;
    const { user } = ctx.switchToHttp().getRequest();
    return required.some((r) => user.roles?.includes(r));
  }
}

// 컨트롤러 — 선언으로 정책을 붙인다
@Roles('admin')
@Get('admin/users')
listUsers() { /* ... */ }

가드 배선의 전체 그림은 역할 기반 API 설계 글에서 더 자세히 다뤘습니다. 여기서 강조하고 싶은 건 수평 통제 쪽입니다. 역할 검사는 통과했는데도 뚫리는 경우가 바로 이곳이거든요. 정산 시스템을 예로 들면, 모든 가맹점은 merchant 역할을 갖지만 각자 자기 정산만 봐야 합니다. 그런데 GET /settlements/:id가 단순히 id로 조회만 하면, 가맹점 A가 :id만 바꿔 가맹점 B의 정산을 열람할 수 있습니다. 이것이 IDOR(안전하지 않은 직접 객체 참조)입니다.

// 취약 — 역할은 통과했지만 "누구의 것인가"를 묻지 않는다
@Get('settlements/:id')
getSettlement(@Param('id') id: string) {
  return this.settlements.findById(id);   // 남의 정산도 그대로 나간다 (IDOR)
}

// 안전 — 인증 주체에 소유권을 묶어서 조회한다
@Get('settlements/:id')
async getSettlement(@Param('id') id: string, @CurrentUser() user: User) {
  const row = await this.settlements.findOne({
    where: { id, merchantId: user.merchantId },   // 소유자가 아니면 조회 자체가 안 된다
  });
  if (!row) throw new NotFoundException();      // 403이 아니라 404 — 존재조차 흘리지 않는다
  return row;
}

고친 코드의 핵심은 소유권을 조회 조건에 넣은 것입니다. where에 인증된 사용자의 merchantId를 함께 걸면, 내 것이 아닌 행은 애초에 결과에 들어오지 않습니다. 권한 검사를 서비스 로직 뒤가 아니라 데이터 접근 계층에서, 그것도 클라이언트가 준 id가 아니라 인증 주체에 묶어 수행하는 것 — 이게 수평 통제의 본질입니다. 없을 때 403 대신 404를 반환하는 것도 의도된 선택입니다. 403은 "그건 있는데 너는 못 본다"고 알려주는 셈이라, 리소스의 존재 여부라는 정보까지 흘리기 때문입니다.

*

기본은 차단(deny by default) — 접근 통제는 "열어야 할 것을 명시적으로 여는" 방향으로 설계해야 합니다. 전역 가드를 깔아 두면 새 엔드포인트는 기본이 차단이고, 공개하려면 @Public() 같은 표식을 일부러 붙여야 합니다. 반대로 기본이 '허용'이면, 언젠가 누군가 @Roles를 빠뜨리는 순간 그 구멍은 소리 없이 열립니다. 실수가 났을 때 닫히는 쪽으로 기울여 두는 것이 설계의 몫입니다.

5. 인증과 민감정보 — 흘리지 않기

인증 실패(A07)와 암호화 실패(A02)는 결이 다르지만, 백엔드에서 막는 방식은 한 문장으로 이어집니다 — 흘리지 않기. 먼저 비밀번호입니다. 비밀번호는 되돌릴 수 없게 저장해야 하고, 그래서 암호화가 아니라 해시입니다. MD5·SHA-1이나 평문은 논외이고, bcrypt·argon2처럼 비용 인자가 있는 알고리즘을 써서 무차별 대입을 느리게 만듭니다.

// 저장 — 비용 인자를 준 단방향 해시
const passwordHash = await bcrypt.hash(password, 12);

// 로그인 — 성공/실패 메시지를 하나로 통일한다
const user = await this.users.findByEmail(email);
const ok = user && await bcrypt.compare(password, user.passwordHash);
if (!ok) {
  // 이메일 존재 여부를 흘리지 않는다 — 계정 열거 방지
  throw new UnauthorizedException('이메일 또는 비밀번호가 올바르지 않습니다');
}

실패 메시지를 하나로 합친 것이 사소해 보이지만 중요합니다. "없는 이메일입니다"와 "비밀번호가 틀렸습니다"를 구분해 주면, 공격자는 어떤 이메일이 가입돼 있는지를 하나씩 확인할 수 있습니다(계정 열거). 한 걸음 더 들어가면 타이밍도 신호가 됩니다 — 이메일이 없을 때 bcrypt.compare를 건너뛰면 응답이 빨라져, 그 시간차만으로 존재 여부가 새기도 합니다. 그래서 사용자가 없을 때도 더미 해시와 비교해 응답 시간을 균일하게 맞추는 편이 안전합니다. 무차별 대입 자체를 늦추는 건 레이트 리미팅의 몫이고, 토큰 발급과 검증은 뒤의 JWT 인증 미들웨어 글로 미룹니다.

민감정보 쪽에서 개발자가 가장 자주 잊는 유출 경로는 두 곳입니다. 응답과 로그. 엔티티를 통째로 직렬화해 반환하면 passwordHash나 개인정보 컬럼까지 그대로 응답에 실립니다. SafeIn처럼 근로자 개인정보와 사고 기록을 다루는 시스템에서는, 응답에 무엇이 실리는지가 곧 사고 여부를 가릅니다. 막는 방법은 명시적으로 고를 것만 고르는 것입니다.

// 취약 — 엔티티 전체를 직렬화하면 passwordHash까지 응답에 실린다
return await prisma.user.findUnique({ where: { id } });

// 안전 — 필요한 컬럼만 select 한다. 민감 컬럼은 애초에 안 꺼낸다
return await prisma.user.findUnique({
  where: { id },
  select: { id: true, email: true, displayName: true },
});

select로 필요한 컬럼만 꺼내면, 실수로 민감 컬럼을 응답에 흘릴 경로가 사라집니다(응답 DTO나 class-transformer의 @Exclude()도 같은 목적입니다). 나머지 한 곳, 로그는 더 은근합니다. 요청 본문을 통째로 로깅하는 인터셉터 하나가 비밀번호·토큰·주민번호를 평문으로 디스크에 쌓습니다 — 그래서 로그에 남기기 전에 민감 필드는 마스킹해야 합니다. 전송 구간의 암호화(TLS)와 리버스 프록시 설정은 다음 글 TLS와 리버스 프록시에서, 서버·헤더 레벨의 하드닝은 웹 서버 하드닝 체크리스트에서 이어집니다.

6. 서평 — 모든 입력은 적이다

지금까지의 코드에는 하나의 태도가 관통합니다 — 클라이언트를 믿지 않는다는 것. 이 태도를 가장 철저하게 훈련시켜 준 책이 데이비드 스터타드(Dafydd Stuttard)와 마커스 핀토의 『The Web Application Hacker's Handbook』(2판, Wiley, 2011)입니다. 저자 스터타드는 침투 테스트 도구 Burp Suite를 만든 PortSwigger의 창업자로, 이 책은 방어자가 아니라 공격자의 방법론 순서로 쓰여 있습니다.

구성부터가 공격의 흐름입니다. 먼저 애플리케이션을 매핑해 공격 표면을 파악하고, 이어서 각 메커니즘을 차례로 공략합니다 — 클라이언트 측 통제를 우회하고, 인증을 공격하고, 세션 관리를 공격하고, 접근 통제를 공격하고, 인터프리터에 코드를 주입하고, 애플리케이션 로직의 허점을 찌릅니다. 눈치채셨겠지만, 이 목차는 이 글의 섹션 순서와 거의 겹칩니다. 우연이 아닙니다. 방어의 지도는 공격의 지도를 뒤집은 것이기 때문입니다.

책이 처음부터 끝까지 반복하는 명제는 하나입니다. 사용자는 임의의 입력을 보낼 수 있고, 클라이언트에 심어 둔 어떤 통제도 우회될 수 있다. 그러니 모든 입력은 잠재적으로 적대적이라고 가정하라는 것이죠. 백엔드 개발자로서 이 책을 읽으면, 기본값이 뒤집힙니다. 클라이언트가 보낸 값을 "일단 맞겠지"로 두던 습관이, "이 값을 서버가 다시 세우지 않으면 아무것도 보장되지 않는다"로 바뀝니다.

// 책의 태도를 한 줄로: 클라이언트가 뭐라 주장하든, 서버가 다시 세운다
const merchantId = req.user.merchantId;   // 세션에서 (O)  —  req.body.merchantId (X)
const row = await this.settlements.findOne({ where: { id, merchantId } });

이 관점으로 되돌아보면 앞 섹션들이 왜 그렇게 생겼는지가 설명됩니다. 소유권을 조회 조건에 넣은 것은 클라이언트가 준 id를 믿지 않아서이고, 화이트리스트 검증은 본문의 모양을 믿지 않아서이며, 요청마다 역할을 다시 검사하는 것은 이전 요청의 권한이 지금도 유효하다고 믿지 않아서입니다. 전부 같은 태도의 다른 얼굴입니다.

정직하게 한계도 적어 둡니다. 2011년 책이라 등장하는 도구와 기술 스택은 지금 기준으로 낡았고, SPA·JWT가 표준이 된 오늘의 지형을 그대로 담지는 못합니다. 그러나 공격자의 사고 모델은 특정 취약점보다 오래 갑니다. 개별 익스플로잇은 유행을 타도, "클라이언트를 신뢰하지 말라"는 명제는 프레임워크가 몇 번 바뀌어도 그대로 유효했습니다. 서버 코드를 쓰는 방식을 바꾸는 것은 결국 이 모델입니다.

7. 정리 — 백엔드 보안 체크리스트

PR을 리뷰하거나 새 엔드포인트를 열 때 실제로 짚는 순서로 정리합니다.

  • 인젝션 — 문자열 조립 금지, 파라미터 바인딩·ORM 빌더로. 원시 SQL은 태그드 템플릿만, $queryRawUnsafe·동적 정렬 컬럼은 허용 목록과 대조.
  • 입력 검증 — DTO + whitelist·forbidNonWhitelisted. 받을 모양을 긍정형으로 열거하고 나머지는 거절. 대량 할당을 막는다.
  • 수직 통제 — 요청마다 서버에서 역할 검사. 화면에서 버튼 숨기기는 통제가 아니다. 기본은 차단(deny by default).
  • 수평 통제 — 조회 조건에 인증 주체의 소유권을 묶는다. 클라이언트가 준 id를 믿지 않는다. 없을 때는 404로 존재를 감춘다.
  • 인증 — bcrypt·argon2로 단방향 해시, 로그인 실패 메시지 통일(계정 열거 방지), 무차별 대입은 레이트 리미팅.
  • 민감정보 — 응답은 select·DTO로 골라 내보내고, 로그는 마스킹, 전송은 TLS.
  • 관통하는 원칙 — 클라이언트가 주장하는 모든 것을 서버가 다시 검증한다.

OWASP Top 10은 한 번 쌓고 끝나는 성벽이 아니라, 모든 경계에서 반복되는 습관입니다. 목록의 항목을 외우는 것보다, 입력을 받는 자리마다 "이 값을 서버가 다시 세웠는가"를 묻는 편이 훨씬 멀리 갑니다. 그 경계 중 하나인 인증은 따로 한 편에서 코드로 파고듭니다 — 토큰을 검증하는 미들웨어를 어떻게 짜는지가 자연스러운 다음 장입니다.
