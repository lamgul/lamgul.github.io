보안: OWASP Top 10 — 백엔드에서 실제로 막는 법

2024년 3월• OWASP · 인젝션 · 접근 통제

SafeIn 산업안전 시스템 백엔드를 혼자 설계하면서 보안 점검을 준비하던 때, OWASP Top 10 문서를 띄워 놓고 한참을 들여다봤습니다. 목록은 위험의 순위일 뿐이라 그것만 읽어서는 제 코드가 한 줄도 달라지지 않았습니다. 알고 싶었던 건 열 개 항목 중 무엇을 제 코드가 직접 결정하는지, 그리고 어느 줄이 그것을 막아 주는지였습니다. 그때 정리한 내용을 적어 둡니다. Top 10에서 백엔드가 책임지는 항목만 골라, 막는 코드까지 옮겼습니다.

웹 애플리케이션 보안을 다루는 비영리 단체 OWASP가 몇 년마다 개정해 발표하는 표준 인식 문서로, 여러 조직에서 수집한 데이터를 바탕으로 취약점을 발생 빈도·악용 가능성·영향으로 추려 10개 위험 범주로 순위 매긴 목록입니다. 2021년 판은 각 범주에 A01~A10 식별자를 붙이며, 이 글이 다루는 접근 통제 실패(A01)·인젝션(A03)·인증 실패(A07)·암호화 실패(A02)가 여기 속합니다. 개별 취약점의 해결책 모음이나 보안 인증 기준이 아니라 무엇이 흔하고 위험한지를 가리키는 우선순위표이며, 각 항목도 단일 취약점이 아닌 넓은 범주라는 점이 경계입니다.
인젝션Injection
인터프리터(SQL 엔진, OS 셸, LDAP 등)가 코드와 데이터가 한 문자열로 섞인 입력을 받아, 신뢰할 수 없는 사용자 값을 데이터가 아니라 명령의 구조로 해석·실행하게 되는 취약점 범주입니다(OWASP A03). 쿼리를 문자열로 이어 붙이면 엔진은 어디까지가 개발자의 구문이고 어디부터가 입력값인지 구분할 수 없어, 값으로 들어온 문자열이 쿼리의 논리 자체를 바꿔 버립니다. SQL 인젝션 외에 NoSQL 연산자 주입·OS 명령 주입·LDAP 주입이 같은 구조를 공유하며, 특정 위험 문자가 원인이 아니라 코드와 데이터를 한 채널에 섞는 것이 원인이라는 점이 핵심입니다.
파라미터 바인딩Parameter Binding
쿼리문에는 값이 들어갈 자리를 자리표시자($1, ?, :name)로만 남겨 두고, 실제 값은 드라이버가 별도 채널로 데이터베이스에 전달하는 기법입니다(준비된 구문, prepared statement). 데이터베이스는 구문을 먼저 파싱한 뒤 값을 끼워 넣으므로, 값에 어떤 문자가 들어 있든 다시 SQL로 해석될 경로가 없어 인젝션이 구조적으로 성립하지 않습니다. 다만 바인딩되는 대상은 값뿐이라 테이블·컬럼명이나 ORDER BY 정렬 대상 같은 식별자 자리는 자리표시자로 넘길 수 없고, 그런 자리는 허용 목록과 대조하는 별도 방어가 필요합니다.
접근 통제Access Control
인증으로 신원이 확인된 주체가 특정 자원에 어떤 동작을 할 자격이 있는지 판정해 강제하는 인가(authorization) 메커니즘으로, 신원 자체를 증명하는 인증(authentication)과는 구분되는 단계입니다(OWASP A01). 서버가 요청마다 주체의 역할이나 소유권을 정책과 대조해 허용 여부를 결정하며, 상위 권한을 넘보는 수직 통제와 같은 권한으로 남의 자원에 접근하는 수평 통제(IDOR)로 나뉩니다. 화면에서 버튼을 숨기는 클라이언트 측 처리는 엔드포인트를 URL로 직접 호출하면 우회되므로 접근 통제가 아니고, 판정은 반드시 서버에서 이뤄져야 합니다.
해시Hash
임의 길이의 입력을 되돌릴 수 없는 고정 길이 값(다이제스트)으로 변환하는 일방향 함수로, 복호화가 가능한 암호화와 달리 원문을 계산으로 복원할 수 없습니다. 비밀번호는 저장해 둔 해시와 입력값의 해시를 비교만 하면 되므로 암호화가 아닌 해시로 저장하는데, 이때는 범용 해시(SHA-256 등)가 아니라 사용자마다 다른 솔트와 조절 가능한 비용 인자를 갖춘 bcrypt·argon2를 씁니다. MD5·SHA-1이나 비용 인자가 없는 빠른 해시는 유출 시 무차별 대입을 충분히 늦추지 못해 비밀번호 저장에는 부적합하다는 점이 경계입니다.

1. Top 10에서 백엔드의 몫

2021년 개정판의 열 개 항목은 프런트·백엔드·인프라에 걸쳐 있습니다. 클릭재킹이나 보안 설정 오류처럼 서버·브라우저·배포가 함께 책임지는 항목도 있습니다. 그중 몇 개는 애플리케이션 코드가 결과를 거의 그대로 결정하는 항목이어서, 저는 그 넷을 먼저 정리했습니다.

  • A01 접근 통제 실패 — 2021년 판의 1위. 인증은 됐지만 넘보면 안 되는 것에 접근한다.
  • A03 인젝션 — 사용자 입력이 코드로 해석된다. SQL·OS 명령·NoSQL이 같은 뿌리다.
  • A07 인증 실패 — 신원을 허술하게 증명하거나, 무차별 대입을 방치한다.
  • A02 암호화 실패 — 2017년의 '민감정보 노출'이 이름을 바꿨다. 흘리거나, 평문으로 둔다.

이 넷을 고른 기준은 방어의 마지막 선이 서버라는 점이었습니다. 프런트에 걸어 둔 검증이나 감춰 둔 버튼은 공격자 입장에서는 우회하면 그만이고, 브라우저는 제가 통제할 수 없으며 요청은 얼마든지 조작됩니다. 그래서 신뢰가 실제로 성립하는 자리는 서버 코드뿐이었습니다. 무차별 대입 방어(레이트 리미팅), 전송 구간 암호화(TLS), 토큰 발급(JWT)은 각각 별도의 글로 다루고, 여기서는 인젝션·접근 통제·인증·민감정보를 코드로 막는 부분만 적습니다.

2. 인젝션 — 코드와 데이터를 섞지 않기

인젝션은 인터프리터가 코드와 데이터가 뒤섞인 문자열을 받는 데서 생깁니다. SQL 엔진에게 "... WHERE email = '" + email + "'"를 넘기면, 엔진은 어디까지가 제가 짠 쿼리이고 어디부터가 사용자 값인지 구분할 방법이 없습니다. 공격자가 email 자리에 ' OR '1'='1을 넣으면 데이터였어야 할 값이 쿼리의 논리로 해석됩니다. 대흥정보에서 레거시 웹을 유지보수할 때, 이렇게 문자열을 이어 붙인 SQL을 곳곳에서 봤습니다.

// 취약 — 문자열을 이어 붙여 SQL을 만든다
const sql = "SELECT * FROM users WHERE email = '" + email + "'";
db.query(sql);   // email = ' OR '1'='1  이면 전체 행이 나온다

// 안전 — 구조와 데이터를 분리한다 (파라미터 바인딩)
db.query("SELECT * FROM users WHERE email = $1", [email]);

// Prisma — 쿼리 빌더는 애초에 바인딩된다
await prisma.user.findUnique({ where: { email } });

해법은 이스케이프가 아니라 파라미터 바인딩입니다. 이스케이프는 위험한 문자를 하나씩 골라 무력화하는 방식이라 한 곳만 빠뜨려도 뚫리는 반면, 파라미터 바인딩은 드라이버가 쿼리의 골격과 값을 아예 다른 채널로 데이터베이스에 보냅니다. 값은 항상 값으로만 취급되고, 어떤 문자가 들어 있든 다시 SQL로 해석될 경로가 없습니다. 조심해서 피하는 방어가 아니라 구조적으로 성립할 수 없게 만드는 방어라, 저는 이쪽을 기본으로 두었습니다.

ORM을 쓰면 대부분 이 바인딩이 기본값입니다. Prisma의 쿼리 빌더(findUnique, where)는 값을 자동으로 바인딩하고, 원시 SQL이 필요할 때도 태그드 템플릿은 안전하게 처리됩니다. 다만 그 옆에 탈출구가 하나 있습니다.

// 태그드 템플릿 — Prisma가 ${...}를 자동으로 바인딩한다 (안전)
await prisma.$queryRaw`SELECT * FROM users WHERE email = ${email}`;

// Unsafe — 완성된 문자열을 그대로 실행한다 (주입 가능)
await prisma.$queryRawUnsafe(`SELECT * FROM users WHERE email = '${email}'`);

이름이 그대로 알려 줍니다. $queryRaw는 ${email}을 바인딩 파라미터로 바꿔 주지만, $queryRawUnsafe는 이미 완성된 문자열을 받아 그대로 실행합니다. 인젝션이 SQL만의 문제도 아닙니다. NoSQL 연산자 주입, OS 명령 주입, LDAP 주입이 같은 구조를 공유해서, 인터프리터에게 넘길 입력을 문자열 조립으로 만들지 않는다는 원칙 하나가 이 항목 전체에 걸립니다.

!

함정 — ORM을 쓴다고 전부 덮이지는 않습니다. 바인딩할 수 없는 자리가 남기 때문입니다. 대표적으로 동적 정렬 컬럼(ORDER BY ${col})은 값이 아니라 식별자라 파라미터로 넘길 수 없어, 클라이언트가 준 컬럼명을 그대로 쓰면 그 자리가 주입 경로가 됩니다. 이런 자리는 허용된 컬럼 목록과 대조한 뒤 통과시켜야 합니다. $queryRawUnsafe나 문자열로 만든 IN 리스트도 같은 예외 지대입니다.

3. 입력 검증 — 경계에서 계약을 강제한다

입력 검증을 인젝션 방어로 묶어 생각하기 쉽지만, 둘은 하는 일이 다릅니다. 인젝션은 앞서 본 파라미터 바인딩이 막고, 검증은 서버가 받아들일 입력의 모양을 경계에서 계약으로 정해 두는 쪽입니다. 심층 방어이자 정확성의 문제이고, 뒤에 나올 접근 통제와도 맞닿아 있습니다. NestJS라면 DTO와 class-validator가 이 계약을 선언적으로 표현합니다.

// signup.dto.ts — 경계에서 받을 수 있는 모양을 선언한다
export class SignupDto {
  @IsEmail()
  email: string;

  @IsString() @MinLength(10)
  password: string;

  @IsOptional() @IsString() @MaxLength(40)
  displayName?: string;
}

// main.ts — 전역으로 강제한다
app.useGlobalPipes(new ValidationPipe({
  whitelist: true,            // DTO에 없는 속성은 제거
  forbidNonWhitelisted: true, // 있으면 400으로 거절
  transform: true,            // 페이로드를 DTO 인스턴스로 변환
}));

여기서 보안상 중요한 옵션은 whitelist입니다. 대량 할당(mass assignment) 때문인데, 검증이 없으면 클라이언트가 요청 본문에 { email, password, role: "admin" }처럼 선언하지 않은 필드를 끼워 넣을 수 있고, 코드가 userRepo.create(dto)로 전체 객체를 넘기면 role까지 그대로 저장됩니다. whitelist: true는 DTO에 없는 속성을 잘라내고, forbidNonWhitelisted: true는 그런 속성이 오면 400으로 거절합니다. 검증이 접근 통제의 첫 관문이 되는 지점입니다.

원칙은 방향에 있습니다. 서버가 받을 것을 긍정형으로 열거하고 나머지는 거절합니다. 위험한 걸 골라 막는 블랙리스트가 아니라, 허용된 것만 통과시키는 화이트리스트입니다. 블랙리스트는 공격자가 목록에 없는 한 가지만 찾으면 뚫리지만, 화이트리스트는 명시한 것 외에는 처음부터 존재하지 않기 때문입니다. 이 방향은 아래 섹션들에서도 계속 되풀이됩니다.

4. 접근 통제 — 수직과 수평

2021년 판에서 1위로 올라온 항목이고, 뚫려도 겉으로는 티가 잘 나지 않는 쪽입니다. 접근 통제 실패는 두 축으로 나눠 보니 정리하기 쉬웠습니다. 수직은 일반 사용자가 위쪽 권한(관리자 기능)에 닿는 것이고, 수평은 같은 권한의 사용자가 옆 사람의 자원에 닿는 것입니다.

수직 통제부터 봅니다. 관리자 API는 요청마다 서버에서 역할을 검사해야 합니다. 화면에서 관리자 버튼을 숨기는 것만으로는 통제가 되지 않는데, 엔드포인트는 URL만 알면 직접 호출할 수 있기 때문입니다. NestJS라면 가드가 이 검사를 담당합니다.

// roles.guard.ts — 수직 통제: 요청마다 서버에서 역할을 확인한다
@Injectable()
export class RolesGuard implements CanActivate {
  canActivate(ctx: ExecutionContext): boolean {
    const required = this.reflector.getAllAndOverride<Role[]>('roles', [
      ctx.getHandler(), ctx.getClass(),
    ]);
    if (!required) return true;
    const { user } = ctx.switchToHttp().getRequest();
    return required.some((r) => user.roles?.includes(r));
  }
}

// 컨트롤러 — 선언으로 정책을 붙인다
@Roles('admin')
@Get('admin/users')
listUsers() { /* ... */ }

가드 배선의 전체 그림은 역할 기반 API 설계 글에서 더 자세히 다뤘습니다. 여기서 더 적어 두고 싶은 쪽은 수평 통제입니다. 역할 검사를 통과하고도 뚫리는 자리가 여기입니다. 정산 시스템을 예로 들면, 모든 가맹점은 merchant 역할을 갖지만 각자 자기 정산만 봐야 합니다. 그런데 GET /settlements/:id가 단순히 id로 조회만 하면, 가맹점 A가 :id만 바꿔 가맹점 B의 정산을 열람할 수 있습니다. 이것이 IDOR(안전하지 않은 직접 객체 참조)입니다.

// 취약 — 역할은 통과했지만 "누구의 것인가"를 묻지 않는다
@Get('settlements/:id')
getSettlement(@Param('id') id: string) {
  return this.settlements.findById(id);   // 남의 정산도 그대로 나간다 (IDOR)
}

// 안전 — 인증 주체에 소유권을 묶어서 조회한다
@Get('settlements/:id')
async getSettlement(@Param('id') id: string, @CurrentUser() user: User) {
  const row = await this.settlements.findOne({
    where: { id, merchantId: user.merchantId },   // 소유자가 아니면 조회 자체가 안 된다
  });
  if (!row) throw new NotFoundException();      // 403이 아니라 404 — 존재조차 흘리지 않는다
  return row;
}

고친 코드는 소유권을 조회 조건에 넣었습니다. where에 인증된 사용자의 merchantId를 함께 걸면, 내 것이 아닌 행은 애초에 결과에 들어오지 않습니다. 권한 검사를 서비스 로직 뒤가 아니라 데이터 접근 계층에서, 클라이언트가 준 id가 아니라 인증 주체에 묶어 수행하는 셈입니다. 없을 때 403 대신 404를 돌려주는 것도 의도한 부분인데, 403은 "그건 있는데 너는 못 본다"고 알려 주는 셈이라 리소스의 존재 여부까지 흘리기 때문입니다.

*

기본은 차단(deny by default) — 접근 통제는 열어야 할 것을 명시적으로 여는 방향으로 설계하는 편이 낫습니다. 전역 가드를 깔아 두면 새 엔드포인트는 기본이 차단이고, 공개하려면 @Public() 같은 표식을 일부러 붙이게 됩니다. 반대로 기본이 허용이면 누군가 @Roles를 빠뜨린 순간 구멍이 조용히 열립니다. 실수했을 때 닫히는 쪽으로 기울여 두는 것이 설계의 몫입니다.

5. 인증과 민감정보 — 흘리지 않기

인증 실패(A07)와 암호화 실패(A02)는 결이 다르지만, 백엔드에서 막는 방식은 흘리지 않기로 이어집니다. 먼저 비밀번호는 되돌릴 수 없게 저장해야 해서, 암호화가 아니라 해시를 씁니다. MD5·SHA-1이나 평문은 논외이고, bcrypt·argon2처럼 비용 인자가 있는 알고리즘을 써서 무차별 대입을 느리게 만듭니다.

// 저장 — 비용 인자를 준 단방향 해시
const passwordHash = await bcrypt.hash(password, 12);

// 로그인 — 성공/실패 메시지를 하나로 통일한다
const user = await this.users.findByEmail(email);
const ok = user && await bcrypt.compare(password, user.passwordHash);
if (!ok) {
  // 이메일 존재 여부를 흘리지 않는다 — 계정 열거 방지
  throw new UnauthorizedException('이메일 또는 비밀번호가 올바르지 않습니다');
}

실패 메시지를 하나로 합친 부분은 사소해 보이지만 필요합니다. "없는 이메일입니다"와 "비밀번호가 틀렸습니다"를 구분해 주면, 공격자가 어떤 이메일이 가입돼 있는지를 하나씩 확인할 수 있습니다(계정 열거). 메시지를 합쳐도 타이밍이 남습니다. 이메일이 없을 때 bcrypt.compare를 건너뛰면 응답이 빨라져서, 그 시간차만으로 존재 여부가 새기도 합니다. 그래서 사용자가 없을 때도 더미 해시와 비교해 응답 시간을 비슷하게 맞추는 편이 안전합니다. 무차별 대입 자체를 늦추는 건 레이트 리미팅의 몫이고, 토큰 발급과 검증은 뒤의 JWT 인증 미들웨어 글로 미룹니다.

민감정보가 새는 경로는 응답과 로그 두 곳인데, 둘 다 눈에 잘 띄지 않습니다. 엔티티를 통째로 직렬화해 반환하면 passwordHash나 개인정보 컬럼까지 그대로 응답에 실립니다. SafeIn처럼 근로자 개인정보와 사고 기록을 다루는 시스템에서는 응답에 무엇이 실리는지가 그대로 사고로 이어질 수 있어서, 필요한 것만 골라 내보내는 쪽으로 정리했습니다.

// 취약 — 엔티티 전체를 직렬화하면 passwordHash까지 응답에 실린다
return await prisma.user.findUnique({ where: { id } });

// 안전 — 필요한 컬럼만 select 한다. 민감 컬럼은 애초에 안 꺼낸다
return await prisma.user.findUnique({
  where: { id },
  select: { id: true, email: true, displayName: true },
});

select로 필요한 컬럼만 꺼내면, 실수로 민감 컬럼을 응답에 흘릴 경로가 사라집니다(응답 DTO나 class-transformer의 @Exclude()도 같은 목적입니다). 나머지 한 곳인 로그는 더 눈에 띄지 않습니다. 요청 본문을 통째로 로깅하는 인터셉터가 하나 있으면 비밀번호·토큰·주민번호가 평문으로 디스크에 쌓이므로, 로그로 남기기 전에 민감 필드를 마스킹해 두었습니다. 전송 구간의 암호화(TLS)와 리버스 프록시 설정은 다음 글 TLS와 리버스 프록시에서, 서버·헤더 레벨의 하드닝은 웹 서버 하드닝 체크리스트에서 이어집니다.

6. 서평 — 모든 입력은 적이다

지금까지 적은 코드에는 클라이언트를 믿지 않는다는 태도가 공통으로 깔려 있습니다. 그 태도를 정리해 준 책이 데이비드 스터타드(Dafydd Stuttard)와 마커스 핀토의 『The Web Application Hacker's Handbook』(2판, Wiley, 2011)입니다. 저자 스터타드는 침투 테스트 도구 Burp Suite를 만든 PortSwigger의 창업자이고, 이 책은 방어자가 아니라 공격자의 방법론 순서로 쓰여 있습니다.

목차 자체가 공격의 흐름입니다. 먼저 애플리케이션을 매핑해 공격 표면을 파악하고, 이어서 각 메커니즘을 차례로 공략합니다. 클라이언트 측 통제를 우회하고, 인증을 공격하고, 세션 관리를 공격하고, 접근 통제를 공격하고, 인터프리터에 코드를 주입하고, 애플리케이션 로직의 허점을 찌릅니다. 읽다 보니 이 순서가 이 글의 섹션 구성과 거의 겹쳤는데, 방어의 지도가 공격의 지도를 뒤집은 것이라 그런 듯합니다.

책이 반복해서 말하는 명제는 사용자는 임의의 입력을 보낼 수 있고, 클라이언트에 심어 둔 어떤 통제도 우회될 수 있다는 것입니다. 그러니 모든 입력을 잠재적으로 적대적이라고 가정하라는 이야기입니다. 읽고 나서 제 기본값이 바뀌었습니다. 클라이언트가 보낸 값을 일단 맞겠거니 두던 습관이, 이 값을 서버가 다시 세우지 않으면 보장되는 게 없다는 쪽으로 옮겨 갔습니다.

// 책의 태도를 한 줄로: 클라이언트가 뭐라 주장하든, 서버가 다시 세운다
const merchantId = req.user.merchantId;   // 세션에서 (O)  —  req.body.merchantId (X)
const row = await this.settlements.findOne({ where: { id, merchantId } });

이 관점에서 앞 섹션들을 다시 보면 왜 그런 모양이 됐는지 설명이 됩니다. 소유권을 조회 조건에 넣은 것은 클라이언트가 준 id를 믿지 않아서이고, 화이트리스트 검증은 본문의 모양을 믿지 않아서이며, 요청마다 역할을 다시 검사하는 것은 이전 요청의 권한이 지금도 유효하다고 보지 않아서입니다. 같은 태도가 자리만 바꿔 나타난 셈입니다.

한계도 적어 둡니다. 2011년 책이라 등장하는 도구와 기술 스택은 지금 기준으로 낡았고, SPA·JWT가 표준이 된 지금의 지형을 그대로 담지는 못합니다. 다만 공격자의 사고 모델은 특정 취약점보다 오래 갑니다. 개별 익스플로잇은 유행을 타도, 클라이언트를 신뢰하지 말라는 명제는 프레임워크가 몇 번 바뀌는 동안에도 그대로 유효했습니다. 제 서버 코드 쓰는 방식을 바꾼 것도 개별 기법이 아니라 이 모델이었습니다.

7. 정리 — 백엔드 보안 체크리스트

PR을 리뷰하거나 새 엔드포인트를 열 때 실제로 짚는 순서로 정리합니다.

  • 인젝션 — 문자열 조립 금지, 파라미터 바인딩·ORM 빌더로. 원시 SQL은 태그드 템플릿만, $queryRawUnsafe·동적 정렬 컬럼은 허용 목록과 대조.
  • 입력 검증 — DTO + whitelist·forbidNonWhitelisted. 받을 모양을 긍정형으로 열거하고 나머지는 거절. 대량 할당을 막는다.
  • 수직 통제 — 요청마다 서버에서 역할 검사. 화면에서 버튼 숨기기는 통제가 아니다. 기본은 차단(deny by default).
  • 수평 통제 — 조회 조건에 인증 주체의 소유권을 묶는다. 클라이언트가 준 id를 믿지 않는다. 없을 때는 404로 존재를 감춘다.
  • 인증 — bcrypt·argon2로 단방향 해시, 로그인 실패 메시지 통일(계정 열거 방지), 무차별 대입은 레이트 리미팅.
  • 민감정보 — 응답은 select·DTO로 골라 내보내고, 로그는 마스킹, 전송은 TLS.
  • 관통하는 원칙 — 클라이언트가 주장하는 모든 것을 서버가 다시 검증한다.

정리하고 보니 OWASP Top 10은 한 번 쌓고 끝나는 성벽이라기보다 경계마다 되풀이하는 습관에 가까웠습니다. 목록의 항목을 외우는 것보다, 입력을 받는 자리마다 이 값을 서버가 다시 세웠는지 묻는 편이 오래 남습니다. 그 경계 중 하나인 인증은 따로 한 편에서 코드로 적어 두었습니다. 토큰을 검증하는 미들웨어를 어떻게 짜는지가 다음 순서입니다.
