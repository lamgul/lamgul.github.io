논문 분석: PagedAttention — vLLM이 빠른 진짜 이유

2026년 4월• vLLM · PagedAttention · 논문

온프레미스에서 vLLM으로 모델을 서빙하던 때, gpu_memory_utilization 값을 조금 조정했을 뿐인데 동시 처리량이 크게 달라지는 것을 보고도 이유를 설명하지 못했습니다. 그래서 vLLM의 근간이 된 논문 — Efficient Memory Management for Large Language Model Serving with PagedAttention(Kwon et al., SOSP 2023) — 을 읽고 정리해 두었습니다. 읽어 보니 어텐션을 바꾼 논문이라기보다 KV 캐시 메모리를 어떻게 관리할 것인가를 다룬 논문이었고, 제가 운영하며 겪던 현상도 대부분 여기서 설명이 됐습니다.

출처

Efficient Memory Management for Large Language Model Serving with PagedAttention

Woosuk Kwon 외 · SOSP 2023 · arXiv:2309.06180

PDF ↗ · arXiv ↗

먼저 짚고 가기
KV 캐시KV Cache
트랜스포머 디코더가 이미 처리한 토큰마다 레이어별로 계산해 둔 키(K)·값(V) 벡터를 GPU 메모리에 그대로 보관하는 텐서입니다. 확정된 토큰의 K·V는 이후 스텝에서도 값이 변하지 않으므로, 새 토큰의 쿼리는 이 캐시를 읽어 어텐션을 계산하고 자기 K·V를 뒤에 덧붙이기만 합니다. 연산을 메모리와 맞바꾼 구조라 크기가 토큰 수에 비례해 선형으로 늘고, 토큰 하나가 2 × 레이어 수 × 히든 차원 × dtype 바이트를 차지합니다(논문의 OPT-13B 예시로 800 KiB). 값이 고정된 가중치와 달리 요청마다 따로 생겼다 사라지는 메모리라는 점이 관리를 어렵게 만듭니다.
가상 메모리 · 페이징Virtual Memory · Paging
운영체제가 프로세스의 주소 공간을 고정 크기 페이지로 나누고, 페이지 테이블로 가상 페이지 번호를 물리 프레임 번호에 매핑하는 메모리 관리 기법입니다. 프로세스는 연속된 주소 공간을 보지만 실제 페이지는 물리 메모리 어디에 흩어져 있어도 되므로, 할당할 때 연속된 빈 자리를 찾아야 할 이유가 사라집니다. 대신 접근할 때마다 주소 변환이라는 간접 참조가 한 겹 붙는 비용을 치릅니다. 논문은 이 대응을 페이지→KV 블록, 바이트→토큰, 프로세스→시퀀스로 그대로 옮겨 옵니다.
단편화Fragmentation
할당은 되어 있지만 실제 데이터가 들어 있지 않아 어느 요청도 쓸 수 없는 메모리입니다. 할당 단위 안쪽에 남는 잉여가 내부 단편화(2048칸을 예약하고 37토큰만 쓴 경우), 할당된 덩어리들 사이에 크기가 맞지 않아 배정되지 못하는 틈새가 외부 단편화입니다. 논문은 여기에 '예약 낭비'를 따로 세는데, 그 자리는 언젠가 같은 시퀀스가 쓸 공간이라 엄밀히는 단편화와 구분되지만 지금 다른 요청이 못 쓴다는 점은 같습니다. 셋을 합쳐 기존 시스템의 KV 캐시 중 실제 토큰 상태를 담은 비율이 20.4~38.2%에 그쳤다는 것이 논문의 측정입니다.
copy-on-writeCopy-on-Write
여러 주체가 같은 데이터를 볼 동안에는 물리 사본 하나를 공유하다가, 쓰기가 들어오는 순간 그 단위만 복제해 쓰는 쪽에 넘기는 자원 관리 기법입니다. 공유 단위마다 참조 카운트를 두고, 카운트가 2 이상인 단위에 쓰기가 오면 복사한 뒤 원본의 카운트를 하나 내리는 식으로 구현합니다. vLLM은 이 단위를 운영체제의 페이지 대신 KV 블록(기본 16토큰)으로 잡아, 병렬 샘플링이나 빔 서치에서 공통 프롬프트 블록을 한 벌만 유지합니다. 절감은 내용이 실제로 같은 구간에서만 생기므로, 시퀀스가 갈라진 뒤 생성되는 토큰의 블록은 각자 따로 가집니다.
선점Preemption
실행 중인 작업에서 자원을 도로 회수해 다른 작업에 재배정하는 스케줄링 동작입니다. vLLM에서는 KV 블록 풀이 바닥나면 스케줄러가 나중에 도착한 시퀀스를 골라 그 시퀀스의 블록을 전부 회수하는데, 다음 디코딩 스텝에 모든 블록이 함께 필요하므로 일부만 내려놓는 선택지가 없습니다. 회수한 블록은 CPU 메모리로 옮겨 두거나(스왑) 아예 버렸다가 재개 시점에 다시 계산합니다. 선점된 요청은 취소되는 것이 아니라 지연될 뿐이지만, 로그에 preempted가 반복된다면 풀 크기에 비해 동시 슬롯이 많다는 신호입니다.

1. KV 캐시 — 가중치보다 다루기 어려운 메모리

디코딩은 토큰을 하나씩 생성합니다. 매 스텝마다 새 토큰의 쿼리 q는 이전 모든 토큰의 키·값과 어텐션을 계산해야 하는데, 매번 다시 계산하면 낭비이므로 각 토큰의 K·V 벡터를 레이어마다 저장해 두고 재사용합니다. 이것이 KV 캐시이고, 토큰 하나가 차지하는 양은 모델 구조만 알면 그대로 계산됩니다.

# KV 캐시가 토큰 하나에 차지하는 메모리 (논문의 예시: OPT-13B, FP16)
n_layers, hidden, dtype_bytes = 40, 5120, 2
kv_per_token = 2 * n_layers * hidden * dtype_bytes  # 2 = K와 V
print(f"{kv_per_token / 1024:.0f} KiB")             # 800 KiB

# 최대 길이 2048토큰을 요청마다 미리 잡아두면
print(f"{kv_per_token * 2048 / 2**30:.2f} GiB")     # 요청 하나에 1.56 GiB

논문의 수치로, OPT-13B를 A100 40GB에 올리면 가중치가 약 26GB(65%)를 차지하고 KV 캐시에 쓸 수 있는 공간은 30% 남짓입니다. 가중치는 고정된 양이라, 동시에 몇 개의 요청을 배치에 태울 수 있는가는 이 30%를 얼마나 알뜰하게 쓰느냐에 달려 있습니다. 서빙 처리량의 병목이 연산이 아니라 메모리에 있다는 것이 논문의 출발점입니다.

2. 문제: 연속 할당이 만드는 세 가지 낭비

기존 시스템들은 KV 캐시를 연속된 텐서로 다뤘습니다. 시퀀스 길이를 미리 알 수 없으니 요청이 들어오면 최대 길이(예: 2048토큰)만큼을 한 덩어리로 예약하는 방식입니다. 논문은 여기서 생기는 낭비를 세 가지로 분해합니다.

  • 내부 단편화 — 2048칸을 예약했는데 답변이 37토큰에서 끝나면 나머지는 그냥 버려진 공간입니다.
  • 예약 낭비 — 아직 생성되지 않은 미래 토큰의 자리는 지금 비어 있지만, 다른 요청이 쓸 수 없습니다.
  • 외부 단편화 — 요청마다 예약 크기가 달라서, 할당기(버디 할당기류)가 쪼개 놓은 틈새 메모리는 어느 요청에도 못 들어갑니다.

논문의 측정에서는 기존 시스템의 KV 캐시 메모리 중 실제 토큰 상태를 담고 있는 비율이 20.4%~38.2%였습니다. 확보한 캐시 공간의 6~8할이 쓰이지 못하고 있었다는 뜻입니다. vLLM은 이 낭비를 4% 미만으로 줄였다고 보고합니다.

3. 발상: 블록 테이블 — KV 캐시의 가상 메모리

논문이 꺼내 든 해법은 운영체제 수업에서 배우는 가상 메모리였습니다. 가상 메모리가 프로세스의 주소 공간을 페이지로 쪼개 물리 메모리 아무 곳에나 흩뿌리듯, PagedAttention은 시퀀스의 KV 캐시를 고정 크기 블록(vLLM 기본값 16토큰)으로 쪼갭니다. 블록이 페이지, 토큰이 바이트, 시퀀스가 프로세스에 대응하는 구조입니다. 시퀀스마다 블록 테이블이 논리 블록 번호를 물리 블록 번호로 매핑하고, 블록은 토큰이 실제로 채워질 때 하나씩 할당됩니다.

# 블록 테이블의 동작을 최소한으로 재현한 시뮬레이션
BLOCK = 16                                # vLLM 기본 블록 크기(토큰)

class BlockTable:
    def __init__(self, pool):
        self.pool = pool                  # 비어 있는 물리 블록 번호 풀
        self.blocks = []                  # 논리 순서 → 물리 블록 번호

    def grow_to(self, n_tokens):
        need = (n_tokens + BLOCK - 1) // BLOCK
        while len(self.blocks) < need:     # 마지막 블록이 찼을 때만 할당
            self.blocks.append(self.pool.pop())

    def free(self):
        self.pool.extend(self.blocks); self.blocks.clear()

pool = [301, 12, 990, 47, 588, 3]  # 먼저 끝난 시퀀스들이 반납해 뒤섞인 빈 블록 풀
seq = BlockTable(pool)
seq.grow_to(37)          # 37토큰 → 블록 3개. 낭비는 마지막 블록의 11칸뿐
print(seq.blocks)        # [3, 588, 47] — 논리 순서는 0,1,2지만 물리 위치는 흩어져 있다

K·V가 더 이상 연속 텐서가 아니게 되니 어텐션 커널도 함께 바뀝니다. 어텐션의 정의 자체는 그대로여서 출력은 여전히 o_i = Σ_j softmax(q_i·Kᵀ/√d)_j · v_j이고, 다만 PagedAttention 커널은 이 합을 블록 단위로 순회하며 계산합니다. 블록 테이블에서 물리 블록 주소를 찾아 K 블록·V 블록을 하나씩 가져와 부분 어텐션을 누적하는 식입니다. 논문은 이 간접 참조 때문에 커널 지연이 소폭 늘어난다는 점도 함께 적어 두었는데, 그 비용을 내더라도 메모리 낭비를 없애 배치를 키우는 편이 처리량에서 더 이득이라고 계산합니다. 초록에 적힌 수치로는 동일 지연 수준에서 기존 최고 시스템(FasterTransformer, Orca) 대비 처리량 2~4배이고, 시퀀스가 길수록 모델이 클수록 격차가 더 벌어진다고 합니다.

4. 공유와 copy-on-write — 페이징이 공짜로 주는 것

페이징을 도입하고 나면 운영체제에서와 마찬가지로 공유도 따라옵니다. 한 프롬프트에서 후보 답변을 n개 뽑는 병렬 샘플링에서는 프롬프트 구간의 KV 캐시가 n개 시퀀스 모두 동일합니다. vLLM은 물리 블록에 참조 카운트를 두고 프롬프트 블록을 그대로 공유하다가, 시퀀스가 갈라져 공유 블록에 새 토큰을 써야 하는 순간에만 그 블록 하나를 복사합니다. 운영체제의 copy-on-write와 같은 방식이고, 단위가 페이지 대신 블록일 뿐입니다. 빔 서치처럼 후보들이 긴 앞부분을 공유하는 디코딩일수록 절감 폭이 커지고, 여러 요청이 같은 시스템 프롬프트를 쓰는 경우에도 같은 원리가 적용됩니다.

# 병렬 샘플링 — 프롬프트 블록은 4개 시퀀스가 공유한다
from vllm import LLM, SamplingParams

llm = LLM("Qwen2.5-7B-Instruct-AWQ", gpu_memory_utilization=0.90)
params = SamplingParams(n=4, temperature=0.8, max_tokens=256)

# 프롬프트 KV는 물리 블록 한 벌 + 참조 카운트 4.
# 갈라지는 블록만 copy-on-write로 복사된다.
outs = llm.generate(["다음 공지를 세 문장으로 요약하라: ..."], params)

5. 블록이 바닥나면 — 선점과 스왑

블록을 미리 예약하지 않는 대신, 트래픽이 몰리면 풀이 실제로 바닥날 수 있습니다. 이때 vLLM 스케줄러는 나중에 온 시퀀스를 선점(preemption)합니다. 눈에 띈 설계가 하나 있는데, 시퀀스의 블록을 일부만 내리지 않고 전부 내리거나 전부 유지한다는 점이었습니다. 어차피 다음 스텝에 그 시퀀스의 모든 블록이 함께 필요하기 때문입니다. 내리는 방법은 두 가지로, 블록을 CPU 메모리로 옮겨 두는 스왑(--swap-space가 이 공간)과, 블록을 그냥 버렸다가 나중에 프롬프트와 생성분을 한 번의 배치 연산으로 다시 계산하는 재계산입니다. 저는 재계산이 더 느릴 줄 알았는데, 디코딩 때와 달리 복원할 때는 이미 아는 토큰들이라 한 번에 처리되기 때문에 오히려 더 빠른 경우도 있다고 합니다.

6. 실서빙과 연결 — gpu_memory_utilization의 진짜 의미

여기까지 읽고 나니 산업안전 서비스에서 번역·요약용 모델을 서빙하며 만졌던 설정들이 다시 보였습니다. vLLM은 기동할 때 gpu_memory_utilization만큼의 VRAM을 확보한 뒤, 가중치와 활성값 작업 공간을 빼고 남는 부분을 모두 KV 블록 풀로 만듭니다. 그러니까 이 값은 GPU를 얼마나 쓸지 정하는 값이라기보다 동시에 담아 둘 수 있는 토큰 총량을 정하는 다이얼에 가깝습니다. 당시엔 그냥 메모리 여유를 두는 옵션 정도로 생각하고 있었습니다.

# 실제 서빙에서 쓰던 형태의 기동 옵션
vllm serve Qwen2.5-7B-Instruct-AWQ \
  --quantization awq \
  --gpu-memory-utilization 0.90 \  # 가중치를 뺀 나머지가 KV 블록 풀
  --max-model-len 4096 \           # 이제 "예약"이 아니라 상한일 뿐
  --max-num-seqs 64 \
  --swap-space 8                   # 선점 시 블록을 옮길 CPU 공간(GiB)

# 기동 로그에서 이 줄을 봐야 한다 — 풀의 실제 크기
# INFO ... # GPU blocks: 9490, # CPU blocks: 2048
# 동시 수용 가능한 토큰 ≈ 9490 × 16 (블록 크기)

연속 할당 방식이었다면 max-model-len 4096은 요청마다 4096칸을 예약한다는 뜻이었을 겁니다. PagedAttention 아래에서는 상한일 뿐이라, 평균 수백 토큰짜리 요청이 대부분인 워크로드에서는 동시성이 크게 올라갑니다. 제가 씨름하던 gpu-memory-utilization과 max-num-seqs의 균형도 블록 풀 크기와 슬롯 수의 균형 문제였고, 슬롯을 과하게 늘렸을 때 처리량이 떨어지던 것도 OOM이 아니라 풀이 바닥나 선점과 스왑이 이어지던 상황이었습니다. 논문을 읽고 나서야 기동 로그의 어느 줄을 봐야 하는지 알게 됐습니다.

*

vLLM 로그에 preempted 경고가 반복되면 블록 풀이 모자란다는 신호입니다. 저는 이럴 때 max_num_seqs를 낮추거나, max_model_len을 워크로드의 실제 상한까지 줄이거나, 양자화 수준을 올려 가중치를 줄이고 풀을 키우는 쪽으로 대응했습니다. 기동 로그의 # GPU blocks 줄과 운영 중 KV 캐시 사용률 지표를 함께 보면 어느 쪽 다이얼을 돌려야 할지 판단하기 수월했습니다.

7. 남는 것

읽고 나서 정리해 보니 이 논문에서 새로 발명된 알고리즘은 거의 없었습니다. 페이징, 블록 테이블, copy-on-write, 스왑 모두 수십 년 된 운영체제 기법입니다. 대신 LLM 서빙의 병목을 정확히 진단한 부분이 이 논문의 몫이었습니다. 병목이 연산이 아니라 KV 캐시 메모리라는 것을 숫자로 보인 뒤, 그 문제 모양에 맞는 오래된 해법을 가져다 붙였습니다. vLLM을 그저 빠른 추론 엔진으로만 알고 있었는데, 빠른 이유가 버리는 메모리가 적어 배치를 크게 잡을 수 있기 때문이라는 것이 서빙을 운영하는 입장에서 가장 쓸모 있는 결론이었습니다.
