논문 분석: PagedAttention — vLLM이 빠른 진짜 이유

2026년 4월• vLLM · PagedAttention · 논문

온프레미스에서 vLLM으로 서빙하면서 gpu_memory_utilization 숫자 하나에 동시 처리량이 널뛰는 것을 봤습니다. 왜 그런지 제대로 알고 싶어서 vLLM의 근간이 된 논문 — Efficient Memory Management for Large Language Model Serving with PagedAttention(Kwon et al., SOSP 2023) — 을 읽었습니다. 결론부터 말하면 이건 어텐션 논문이 아니라 메모리 관리 논문입니다. 그리고 그게 핵심입니다.

먼저 짚고 가기
KV 캐시KV Cache
모델이 다음 단어를 만들 때는 앞서 나온 모든 단어를 매번 다시 참고해야 합니다. 그 참고용 계산 결과(키·값)를 버리지 않고 저장해 두는 메모장이 KV 캐시입니다. 이 메모장이 GPU 메모리를 얼마나 알뜰하게 쓰느냐가 이 글 전체의 주제입니다.
가상 메모리 · 페이징Virtual Memory · Paging
운영체제가 메모리를 작은 칸(페이지)으로 쪼개 물리적으로 흩어진 자리에 담아 두고, 주소표로 이어 붙여 마치 하나처럼 쓰는 기법입니다. PagedAttention은 바로 이 오래된 아이디어를 KV 캐시에 그대로 가져온 것이라, 논문의 발상 전체가 여기서 나옵니다.
단편화Fragmentation
메모리를 미리 크게 잡아 두면 실제로 안 쓴 빈칸이나 여기저기 남은 자투리가 생겨 아무도 못 쓰는 낭비가 됩니다. 이 낭비가 기존 방식에서 캐시 공간의 6~8할을 잡아먹었다는 것이 논문이 지적한 문제입니다.
copy-on-writeCopy-on-Write
여럿이 같은 내용을 볼 때는 하나만 두고 함께 쓰다가, 누군가 그 내용을 고치려는 순간에만 비로소 복사본을 만드는 방식입니다. 같은 프롬프트로 답을 여러 개 뽑을 때 앞부분 메모리를 통째로 아끼는 원리입니다.
선점Preemption
자리가 부족하면 나중에 온 작업을 잠시 밀어내 자원을 회수하는 것입니다. vLLM은 메모리 풀이 바닥나면 늦게 들어온 요청을 이렇게 잠깐 내려놓았다가 다시 태웁니다.

1. KV 캐시 — 가중치보다 다루기 어려운 메모리

디코딩은 토큰을 하나씩 생성합니다. 매 스텝 새 토큰의 쿼리 q는 이전 모든 토큰의 키·값과 어텐션을 계산해야 하는데, 이걸 매번 다시 계산하면 낭비이므로 각 토큰의 K·V 벡터를 레이어마다 저장해 둡니다. 이것이 KV 캐시입니다. 토큰 하나가 먹는 양은 정확히 계산됩니다.

# KV 캐시가 토큰 하나에 차지하는 메모리 (논문의 예시: OPT-13B, FP16)
n_layers, hidden, dtype_bytes = 40, 5120, 2
kv_per_token = 2 * n_layers * hidden * dtype_bytes  # 2 = K와 V
print(f"{kv_per_token / 1024:.0f} KiB")             # 800 KiB

# 최대 길이 2048토큰을 요청마다 미리 잡아두면
print(f"{kv_per_token * 2048 / 2**30:.2f} GiB")     # 요청 하나에 1.56 GiB

논문의 수치로, OPT-13B를 A100 40GB에 올리면 가중치가 약 26GB(65%)를 차지하고 KV 캐시에 쓸 수 있는 공간은 30% 남짓입니다. 가중치는 고정이니, 동시에 몇 개의 요청을 배치에 태울 수 있는가는 결국 이 30%를 얼마나 알뜰하게 쓰느냐가 결정합니다. 서빙 처리량의 병목이 연산이 아니라 메모리라는 것 — 논문의 출발점입니다.

2. 문제: 연속 할당이 만드는 세 가지 낭비

기존 시스템들은 KV 캐시를 연속된 텐서로 다뤘습니다. 시퀀스 길이를 미리 알 수 없으니 요청이 들어오면 최대 길이(예: 2048토큰)만큼을 한 덩어리로 예약하는 방식입니다. 논문은 여기서 생기는 낭비를 세 가지로 분해합니다.

  • 내부 단편화 — 2048칸을 예약했는데 답변이 37토큰에서 끝나면 나머지는 그냥 버려진 공간입니다.
  • 예약 낭비 — 아직 생성되지 않은 미래 토큰의 자리는 지금 비어 있지만, 다른 요청이 쓸 수 없습니다.
  • 외부 단편화 — 요청마다 예약 크기가 달라서, 할당기(버디 할당기류)가 쪼개 놓은 틈새 메모리는 어느 요청에도 못 들어갑니다.

논문의 측정으로는, 기존 시스템에서 KV 캐시 메모리 중 실제 토큰 상태를 담고 있는 비율이 20.4%~38.2%에 그칩니다. 확보한 캐시 공간의 6~8할을 버리고 있었다는 뜻입니다. vLLM은 이 낭비를 4% 미만으로 줄였다고 보고합니다.

3. 발상: 블록 테이블 — KV 캐시의 가상 메모리

해법은 OS 수업에서 본 그것입니다. 가상 메모리가 프로세스의 주소 공간을 페이지로 쪼개 물리 메모리 아무 곳에나 흩뿌리듯, PagedAttention은 시퀀스의 KV 캐시를 고정 크기 블록(vLLM 기본값 16토큰)으로 쪼갭니다. 대응 관계가 정확합니다 — 블록은 페이지, 토큰은 바이트, 시퀀스는 프로세스. 시퀀스마다 블록 테이블이 논리 블록 번호를 물리 블록 번호로 매핑하고, 블록은 토큰이 실제로 채워질 때 하나씩 할당됩니다.

# 블록 테이블의 동작을 최소한으로 재현한 시뮬레이션
BLOCK = 16                                # vLLM 기본 블록 크기(토큰)

class BlockTable:
    def __init__(self, pool):
        self.pool = pool                  # 비어 있는 물리 블록 번호 풀
        self.blocks = []                  # 논리 순서 → 물리 블록 번호

    def grow_to(self, n_tokens):
        need = (n_tokens + BLOCK - 1) // BLOCK
        while len(self.blocks) < need:     # 마지막 블록이 찼을 때만 할당
            self.blocks.append(self.pool.pop())

    def free(self):
        self.pool.extend(self.blocks); self.blocks.clear()

pool = list(range(1024))
seq = BlockTable(pool)
seq.grow_to(37)          # 37토큰 → 블록 3개. 낭비는 마지막 블록의 11칸뿐
print(seq.blocks)        # [1023, 1022, 1021] — 물리적으로는 불연속

이러면 어텐션 커널이 달라져야 합니다. K·V가 연속 텐서가 아니니까요. 어텐션의 정의 자체는 그대로입니다 — 출력은 o_i = Σ_j softmax(q_i·Kᵀ/√d)_j · v_j — 다만 PagedAttention 커널은 이 합을 블록 단위로 순회하며 계산합니다. 블록 테이블에서 물리 블록 주소를 찾아 K 블록·V 블록을 하나씩 가져와 부분 어텐션을 누적하는 식입니다. 논문은 이 간접 참조 때문에 커널 지연이 소폭 늘어난다는 것도 숨기지 않는데, 그 비용을 내고 메모리 낭비를 없애 배치를 키우는 쪽이 처리량에서 압도적으로 이득이라는 게 논문의 계산입니다. 초록의 주장 그대로, 동일 지연 수준에서 기존 최고 시스템(FasterTransformer, Orca) 대비 처리량 2~4배. 시퀀스가 길수록, 모델이 클수록 격차는 더 벌어집니다.

4. 공유와 copy-on-write — 페이징이 공짜로 주는 것

페이징을 도입하면 OS가 그랬듯 공유도 따라옵니다. 한 프롬프트에서 후보 답변을 n개 뽑는 병렬 샘플링을 생각해 보면, 프롬프트 구간의 KV 캐시는 n개 시퀀스가 완전히 동일합니다. vLLM은 물리 블록에 참조 카운트를 두고 프롬프트 블록을 그대로 공유하다가, 시퀀스가 갈라져 공유 블록에 새 토큰을 써야 하는 순간에만 그 블록 하나를 복사합니다. 정확히 copy-on-write이고, 단위는 블록입니다. 빔 서치처럼 후보들이 긴 앞부분을 공유하는 디코딩일수록 절감 폭이 커지고, 여러 요청이 같은 시스템 프롬프트를 쓰는 경우에도 같은 원리가 적용됩니다.

# 병렬 샘플링 — 프롬프트 블록은 4개 시퀀스가 공유한다
from vllm import LLM, SamplingParams

llm = LLM("Qwen2.5-7B-Instruct-AWQ", gpu_memory_utilization=0.90)
params = SamplingParams(n=4, temperature=0.8, max_tokens=256)

# 프롬프트 KV는 물리 블록 한 벌 + 참조 카운트 4.
# 갈라지는 블록만 copy-on-write로 복사된다.
outs = llm.generate(["다음 공지를 세 문장으로 요약하라: ..."], params)

5. 블록이 바닥나면 — 선점과 스왑

블록을 미리 예약하지 않는 대신, 트래픽이 몰리면 풀이 진짜로 바닥날 수 있습니다. 이때 vLLM 스케줄러는 나중에 온 시퀀스를 선점(preemption)합니다. 흥미로운 설계가 하나 있는데, 시퀀스의 블록을 일부만 내리는 게 아니라 전부 내리거나 전부 유지합니다. 어차피 다음 스텝에 그 시퀀스의 모든 블록이 함께 필요하기 때문입니다. 내리는 방법은 두 가지입니다 — 블록을 CPU 메모리로 옮겨 두는 스왑(--swap-space가 이 공간), 아니면 블록을 그냥 버리고 나중에 프롬프트+생성분을 한 번의 배치 연산으로 다시 계산하는 재계산. 재계산이 직관과 달리 더 빠른 경우도 있습니다. 디코딩 때는 토큰을 하나씩 만들었지만 복원할 때는 이미 아는 토큰들이라 한 방에 처리되기 때문입니다.

6. 실서빙과 연결 — gpu_memory_utilization의 진짜 의미

여기까지 읽고 나니, 제가 산업안전 서비스에서 번역·요약용 모델을 서빙하며 만졌던 설정들이 전부 다시 보였습니다. vLLM은 기동 시 gpu_memory_utilization만큼의 VRAM을 확보한 뒤, 가중치와 활성값 작업 공간을 빼고 남는 전부를 KV 블록 풀로 만듭니다. 즉 이 숫자는 "GPU를 얼마나 쓸까"가 아니라 사실상 동시 처리 가능한 토큰 총량을 정하는 다이얼입니다.

# 실제 서빙에서 쓰던 형태의 기동 옵션
vllm serve Qwen2.5-7B-Instruct-AWQ \
  --quantization awq \
  --gpu-memory-utilization 0.90 \  # 가중치를 뺀 나머지가 KV 블록 풀
  --max-model-len 4096 \           # 이제 "예약"이 아니라 상한일 뿐
  --max-num-seqs 64 \
  --swap-space 8                   # 선점 시 블록을 옮길 CPU 공간(GiB)

# 기동 로그에서 이 줄을 봐야 한다 — 풀의 실제 크기
# INFO ... # GPU blocks: 9490, # CPU blocks: 2048
# 동시 수용 가능한 토큰 ≈ 9490 × 16 (블록 크기)

연속 할당 방식이었다면 max-model-len 4096은 요청마다 4096칸 예약을 뜻했을 겁니다. PagedAttention 아래에서는 상한일 뿐이라, 평균 수백 토큰짜리 요청이 대부분인 워크로드에서 동시성이 확 올라갑니다. 제가 겪었던 "gpu-memory-utilization과 max-num-seqs의 균형 잡기"도 결국 블록 풀 크기와 슬롯 수의 균형 문제였던 것이고, 슬롯을 과하게 늘렸을 때 터지던 OOM은 풀이 바닥나 선점·스왑이 연쇄되던 상황이었습니다. 논문을 읽고 나서야 로그의 어떤 줄을 봐야 하는지 알게 됐습니다.

*

vLLM 로그에 preempted 경고가 반복되면 블록 풀이 모자란다는 신호입니다. 처리량이 조용히 무너지기 전에 — max_num_seqs를 낮추거나, max_model_len을 워크로드의 실제 상한으로 줄이거나, 더 센 양자화로 가중치를 줄여 풀을 키우세요. 기동 로그의 # GPU blocks 줄과 운영 중 KV 캐시 사용률 지표를 함께 보면 어느 쪽 다이얼을 돌려야 할지 보입니다.

7. 남는 것

이 논문에서 새로 발명된 알고리즘은 사실상 없습니다. 페이징, 블록 테이블, copy-on-write, 스왑 — 전부 수십 년 된 OS 기법입니다. 통찰은 LLM 서빙의 병목을 정확히 진단한 것입니다. 병목이 연산이 아니라 KV 캐시 메모리라는 걸 숫자로 보인 뒤, 그 문제 모양에 맞는 오래된 해법을 정확히 가져다 붙였습니다. 좋은 시스템 논문의 전형이고, "빠른 추론 엔진"이라는 인상과 달리 vLLM이 빠른 진짜 이유는 버리는 메모리가 없어서 배치가 크기 때문이라는 것 — 서빙을 운영하는 입장에서 가장 쓸모 있는 결론이었습니다.
