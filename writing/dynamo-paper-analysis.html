논문 분석: Dynamo — 항상 쓸 수 있는 저장소의 대가

2022년 8월• Dynamo · 일관성 · 논문

아마존에는 절대 실패하면 안 되는 버튼이 하나 있습니다. 장바구니에 담기입니다. 디스크가 죽고, 네트워크가 널뛰고, 데이터센터가 토네이도에 날아가는 중에도 고객의 "담기"는 성공해야 한다 — 논문 서론에 실제로 나오는 표현입니다. 2007년 SOSP에 발표된 Dynamo 논문은 이 한 줄의 요구에서 출발해, 관계형 데이터베이스의 상식이던 강한 일관성을 의도적으로 양보하는 저장소를 설계합니다. 중요한 건 이 양보가 공짜가 아니라는 점입니다. 가용성을 사면 청구서는 다른 이름으로 날아옵니다. 그 청구서를 항목별로 읽어봤습니다.

출처

Dynamo: Amazon's Highly Available Key-value Store

Giuseppe DeCandia 외 · SOSP 2007

공식 PDF ↗

먼저 짚고 가기
가용성Availability
요청이 왔을 때 "지금은 안 됩니다"라고 거절하지 않고 응답을 돌려줄 수 있는 성질입니다. 이 논문에서는 특히 쓰기를 절대 거부하지 않는 것(always writeable)을 뜻합니다.
최종 일관성Eventual Consistency
여러 대에 복사해 둔 데이터가 잠시 서로 다른 값을 가질 수 있지만, 새 쓰기가 멈추면 결국 모두 같은 값으로 수렴한다는 약속입니다. "지금 당장 같음"을 포기하는 대신 "언젠가 같아짐"을 보장합니다.
컨시스턴트 해싱Consistent Hashing
키와 서버를 같은 원(링) 위에 해시로 올려두고, 키는 시계 방향으로 처음 만나는 서버가 맡게 하는 분배법입니다. 서버가 늘거나 줄어도 이웃 구간의 키만 이사하게 됩니다.
쿼럼Quorum
읽기와 쓰기를 각각 몇 대가 응답해야 성공으로 칠지 정하는 정족수입니다. 읽기 정족수 R과 쓰기 정족수 W의 합이 복제본 수 N보다 크면(R+W>N) 두 집합이 반드시 겹칩니다.
벡터 클록Vector Clock
"몇 시에 썼나"가 아니라 "어느 노드가 몇 번째로 썼나"를 기록하는 장부입니다. 두 버전을 비교해 선후 관계인지, 아니면 동시에 갈라진 충돌인지 판별하게 해 줍니다.

1. 문제 설정 — 99.9번째 백분위수의 세계

이 논문이 특별한 이유는 알고리즘보다 먼저 요구사항의 모양에 있습니다. 아마존의 내부 서비스들은 SLA를 평균이 아니라 99.9번째 백분위수로 계약합니다. 논문 2.2절의 예시는 "초당 500 요청의 피크 부하에서 요청의 99.9%를 300ms 안에 응답"입니다. 평균이나 중앙값으로 재면 소수의 느린 요청이 통계 뒤에 숨습니다. 논문이 굳이 꼬리를 계약 조건으로 끌어올린 이유는 대다수가 아니라 모든 고객이 좋은 경험을 하는 시스템을 목표로 삼았기 때문입니다.

그리고 규모가 문제를 바꿉니다. 수백 대의 노드로 이루어진 시스템에서 장애는 예외 상황이 아니라 상수입니다. 항상 어딘가의 디스크가 죽어 있고 어딘가의 스위치가 재부팅 중입니다. 이 조건에서 강한 일관성을 고집하면 — 네트워크가 갈라졌을 때 일관성과 가용성을 동시에 가질 수 없다는 것은 알려진 트레이드오프입니다 — 시스템은 정직하게 쓰기를 거부하는 쪽을 택하게 됩니다. 장바구니 서비스 입장에서 그건 최악입니다. 고객이 물건을 담으려는 순간 "지금은 복제본 점검 중이라 안 됩니다"라고 답하는 셈이니까요.

그래서 Dynamo는 방향을 뒤집습니다. 쓰기는 절대 거부하지 않는다. 대신 충돌의 해소를 쓰기 시점에서 읽기 시점으로 미루고, "어떻게 합칠 것인가"의 결정권을 저장소가 아니라 애플리케이션에게 넘깁니다. 장바구니라면 두 버전의 합집합을 취하는 식으로요. 이 두 가지 결정에서 이후의 모든 설계 — 컨시스턴트 해싱, 벡터 클록, 쿼럼, 힌티드 핸드오프 — 가 필연적으로 따라 나옵니다.

2. 컨시스턴트 해싱 — 노드가 죽어도 링은 돈다

데이터를 수백 대에 나눠 놓으려면 먼저 "이 키는 누구 것인가"를 정해야 합니다. 순진한 답은 hash(key) % N입니다. 균등하고 빠르지만 치명적인 약점이 있습니다. 노드 수 N이 바뀌는 순간 나머지 연산의 결과가 거의 전부 달라져서, 대부분의 키가 한꺼번에 주인을 바꿉니다. 캐시라면 미스 폭풍이고, 저장소라면 전면 대이사입니다. 장애가 상수인 환경에서 노드 증감은 일상이므로 이 방식은 출발부터 탈락입니다.

Dynamo가 채택한 컨시스턴트 해싱(논문 4.2절)은 해시 공간의 양 끝을 이어 붙여 원으로 만듭니다. 키도 노드도 같은 원 위의 점이고, 키는 자기 위치에서 시계 방향으로 처음 만나는 노드가 맡습니다. 노드 하나가 들어오거나 나가면 바로 이웃 구간의 키만 주인이 바뀌고, 링의 나머지는 아무 일도 없었다는 듯 돌아갑니다.

# 링 위에서 키의 주인 찾기 — 시계 방향 첫 노드 (논문 4.2절)
import bisect, hashlib

def token(s):  # 문자열 → 링 위의 위치(큰 정수)
    return int(hashlib.md5(s.encode()).hexdigest(), 16)

class Ring:
    def __init__(self, nodes, vnodes=32):
        # 가상 노드: 물리 노드 하나를 링 위 여러 점으로 흩뿌린다
        self.points = sorted(
            (token(f"{n}#vn{i}"), n) for n in nodes for i in range(vnodes))

    def owner(self, key):
        i = bisect.bisect(self.points, (token(key),)) % len(self.points)
        return self.points[i][1]  # 시계 방향으로 처음 만나는 노드

글로 읽으면 당연해 보이지만, 노드를 실제로 넣고 빼면서 몇 개의 키가 움직이는지 세어 보면 모듈러 방식과의 차이가 극적으로 드러납니다. 아래 링에서 직접 확인해 보세요.

컨시스턴트 해싱 링 — 노드 증감 시 이동하는 키직접 조작해 보세요
노드 3대 · 키 24개 노드를 추가하거나 제거해 보세요

키(작은 점)는 시계 방향으로 처음 만나는 노드(큰 점)가 맡습니다 — 주인이 바뀐 키만 주황색으로 강조됩니다

위젯에서 한 가지가 더 보일 겁니다. 노드를 링에 한 점씩만 찍으면 운에 따라 어떤 노드는 넓은 구간을, 어떤 노드는 좁은 구간을 맡아 부하가 쏠립니다. Dynamo의 답이 가상 노드(virtual node)입니다. 물리 노드 하나를 링 위의 여러 점(토큰)으로 흩뿌려서 담당 구간을 잘게 쪼개면, 첫째로 부하가 통계적으로 고르게 퍼지고, 둘째로 노드 하나가 죽었을 때 그 부담이 한 이웃이 아니라 여러 노드로 분산되며, 셋째로 성능이 좋은 장비에는 토큰을 더 많이 줘서 용량 차이까지 흡수할 수 있습니다(논문 4.2절). 복제는 이 링 위에서 자연스럽게 정의됩니다 — 키의 주인 노드가 자기 뒤를 잇는 N−1개의 노드에 사본을 두고, 이 담당 노드 목록을 선호 목록(preference list)이라 부릅니다(4.3절). 실제 목록은 장애에 대비해 N개보다 길게 두고, 가상 노드 때문에 같은 물리 노드가 중복되지 않도록 링 위의 위치를 건너뛰며 고릅니다.

*

이 아이디어는 Dynamo 밖에서 표준이 됐습니다. Cassandra의 토큰 링이 직계 후손이고, Redis Cluster는 키를 16384개의 해시 슬롯에 고정 매핑한 뒤 슬롯을 노드에 나눠 주는 변형으로 같은 문제를 풉니다. 실무에서 Redis를 캐시로 쓰면서 노드를 늘려도 미스 폭풍이 일지 않는 건 이 논문 세대가 만들어 둔 길 위에 있기 때문입니다.

3. 벡터 클록 — 시간이 아니라 인과를 기록한다

쓰기를 절대 거부하지 않기로 했으니, 네트워크가 갈라진 동안 같은 키에 서로 다른 쓰기가 쌓이는 일은 정상 동작입니다. 그럼 나중에 두 버전을 만났을 때 무엇이 최신인지 어떻게 알까요? 타임스탬프는 답이 못 됩니다. 분산된 노드들의 벽시계는 서로 어긋나 있고, "시계가 빠른 노드가 이긴다"는 규칙은 멀쩡한 쓰기를 조용히 삼켜 버립니다.

Dynamo의 답은 벡터 클록(논문 4.4절)입니다. 각 버전에 {노드: 카운터} 쌍의 목록을 붙여서, 시각이 아니라 인과 관계를 기록합니다. 버전 A의 장부가 버전 B의 장부를 모든 항목에서 앞서거나 같으면 A는 B를 알고 쓰인 후손이므로 B를 버려도 됩니다. 어느 쪽도 상대를 덮지 못하면 두 쓰기는 동시에 갈라진 것이고, 이때는 둘 다 보관했다가 읽을 때 화해시킵니다.

# 벡터 클록 — {노드: 카운터}로 인과관계를 판별 (논문 4.4절, Figure 3의 예)
def descends(a, b):
    # a가 b의 모든 사건을 알고 있는가 = a는 b 이후의 버전인가
    return all(a.get(node, 0) >= n for node, n in b.items())

def reconcile(a, b):
    if descends(a, b): return [a]   # a가 후손 — b는 안심하고 버린다
    if descends(b, a): return [b]
    return [a, b]  # 인과관계 없음 = 동시 쓰기 — 둘 다 보관, 읽을 때 화해

# 노드 Sx가 두 번 쓴 뒤, 분단된 사이 Sy와 Sz가 각자 이어 쓴 상황
D2 = {"Sx": 2}
D3 = {"Sx": 2, "Sy": 1}
D4 = {"Sx": 2, "Sz": 1}
reconcile(D3, D2)  # → [D3]      : D3이 D2의 후손
reconcile(D3, D4)  # → [D3, D4]  : 충돌 — 장바구니라면 합집합으로 병합

화해에는 두 단계가 있습니다. 장부만 보고 선후를 가리는 구문적 화해는 저장소가 알아서 하고, 진짜 충돌은 의미적 화해로 애플리케이션이 처리합니다. 장바구니의 병합 규칙은 합집합이라, 담은 물건은 절대 사라지지 않는 대신 지운 물건이 되살아날 수는 있습니다(4.4절). 일관성을 양보한 대가가 어디로 청구되는지 보여주는 정직한 예시죠. 참고로 장부가 무한히 자라는 걸 막기 위해 쌍의 개수가 임계치(논문의 예시는 10개)를 넘으면 가장 오래된 쌍부터 잘라내는 절단 규칙도 함께 둡니다.

그래서 실제로 충돌은 얼마나 잦을까요? 논문 6.3절의 24시간 실측에서 요청의 99.94%는 버전을 정확히 하나만 봤습니다. 갈라짐은 드물고, 그마저 사람이 아니라 자동화된 클라이언트 로봇의 동시 쓰기 폭주가 주범이었다고 합니다. 최악을 위해 설계하되, 최악이 일상은 아니라는 것까지 숫자로 보여주는 부분입니다.

4. R+W>N — 일관성을 다이얼로 만들다

복제본이 N개 있을 때, 쓰기를 몇 대가 받아야 성공으로 치고(W), 읽기를 몇 대에서 모아야 할까요(R)? Dynamo는 이걸 고정하지 않고 서비스가 돌리는 다이얼로 만들었습니다(논문 4.5절). 핵심 부등식은 하나입니다. R + W > N — 읽기에 참여한 R대와 쓰기에 참여한 W대의 합이 N을 넘으면, 비둘기집 원리에 의해 두 집합은 최소 한 대를 공유합니다. 즉 어떤 읽기든 최신 쓰기를 아는 노드를 반드시 하나는 만나게 됩니다.

// R + W > N 이면 읽기 집합과 쓰기 집합은 반드시 겹친다 (논문 4.5절)
const N = 3;             // 복제본 수
const configs = [
  { R: 2, W: 2 },  // 균형형 — 흔한 운영값 (3,2,2), 논문 6절
  { R: 1, W: 3 },  // 읽기 최적화 — 읽기는 1대만 물어봐도 된다
  { R: 3, W: 1 },  // 쓰기 최적화 — 1대만 받아도 성공, 가용성 최대
];
for (const { R, W } of configs) {
  const overlap = R + W - N;  // 읽기·쓰기 집합의 최소 겹침
  console.log(`R=${R} W=${W} → 겹침 ≥ ${overlap}개`);
}
// R + W ≤ N 으로 낮추면? 더 빠르고 더 잘 살아남지만,
// 읽기가 낡은 값만 만나는 경우가 생긴다 — 그것도 '선택지'다.

이 다이얼이 좋은 이유는 트레이드오프를 코드가 아니라 설정으로 드러내기 때문입니다. 항상 쓸 수 있어야 하는 장바구니는 W를 낮추고, 낡은 값을 보여주면 안 되는 서비스는 R을 올립니다. 논문 6절에 따르면 사내의 여러 Dynamo 인스턴스가 흔히 쓰는 값은 (N, R, W) = (3, 2, 2)입니다. 읽기와 쓰기 어느 쪽도 한 대의 장애로 멈추지 않으면서 겹침은 보장되는, 이름 그대로의 균형값입니다.

*

주의할 점 하나. R+W>N의 겹침 보장은 읽기와 쓰기가 같은 N대의 집합을 상대한다는 전제 위에 서 있습니다. 다음 절의 슬로피 쿼럼이 켜지면 장애 시 그 집합 자체가 옆 노드로 미끄러지기 때문에, 이 보장은 절대적이 아니라 근사적이 됩니다(논문 4.6절). Dynamo는 그 틈을 알고도 가용성을 위해 감수했고, 벡터 클록이 그 틈의 뒷수습을 맡는 구조입니다.

5. 슬로피 쿼럼과 힌티드 핸드오프 — 그래도 쓰기를 받는다

엄격한 쿼럼이라면 키를 맡은 N대 중 W대가 응답하지 못할 때 쓰기는 실패해야 합니다. 하지만 그건 "쓰기를 절대 거부하지 않는다"와 충돌하죠. Dynamo의 답은 슬로피 쿼럼(sloppy quorum, 논문 4.6절)입니다. 정확히 그 N대가 아니라, 선호 목록을 따라 내려가며 만나는 건강한 첫 N대가 읽기·쓰기를 받습니다. 원래 주인 A가 쓰러져 있으면 목록의 다음 노드 D가 대신 받는 식입니다.

이때 D는 그 사본을 자기 데이터인 척 섞어 두지 않고, "이건 원래 A의 것"이라는 힌트를 메타데이터로 붙여 별도 보관합니다. 그리고 A가 살아난 것이 감지되면 힌트가 달린 사본을 A에게 배달하고 자기 것을 지웁니다. 이것이 힌티드 핸드오프(hinted handoff)입니다. 노드가 잠깐 앓는 일시 장애는 이렇게 링의 이웃들이 품앗이로 흡수하고, 시스템 전체로는 쓰기가 한 번도 거부되지 않은 것처럼 보입니다.

물론 힌트를 들고 있던 노드마저 죽는 영구 장애도 있습니다. 그 뒷정리는 반엔트로피(anti-entropy) 프로토콜의 몫입니다. 복제본끼리 머클 트리(Merkle tree)로 키 범위의 해시를 위에서부터 비교해 내려가면, 전체 데이터를 읽지 않고도 어긋난 부분만 콕 집어 동기화할 수 있습니다(4.7절). 멤버십과 장애 감지는 중앙 관리자 없이 가십 프로토콜로 퍼뜨립니다. 논문 4장의 표 1이 이 기법들을 문제 상황별로 정리해 두고 있는데 — 분할은 컨시스턴트 해싱, 버전 충돌은 벡터 클록, 일시 장애는 슬로피 쿼럼과 힌티드 핸드오프, 영구 장애는 머클 트리 반엔트로피 — 분산 시스템 공부의 지도로 삼기 좋은 한 페이지입니다.

6. 정리 — 대가를 알고 지불하기

처음의 청구서 이야기로 돌아가겠습니다. Dynamo가 가용성을 사고 지불한 값은 명확합니다. 충돌 해소의 책임이 애플리케이션으로 넘어왔고, 지운 물건이 되살아날 수 있으며, 읽기가 낡은 값을 만날 수 있고, 링·힌트·반엔트로피라는 운영 복잡성이 생겼습니다. 이 논문이 훌륭한 건 그 대가를 숨기지 않고 N, R, W라는 다이얼로 노출해서, 값을 치를지 말지를 서비스마다 스스로 정하게 했다는 점입니다.

저는 이 규모의 저장소를 만들 일이 없는 백엔드 개발자지만, 이 계산법 자체는 제 크기의 시스템에도 그대로 적용됐습니다. SafeIn 백엔드를 혼자 설계하고 운영하면서 DB 앞에 Redis 캐시를 두는 순간, 저는 이미 원본과 사본이 잠시 어긋나는 최종 일관성 시스템을 하나 굴리고 있었던 셈입니다. "TTL 동안의 낡음을 어디까지 허용할 것인가"는 R과 W를 고르는 일의 축소판이었죠. 반대로 정산 코드를 리팩토링할 때는 다이얼을 정반대로 돌렸습니다. 돈 계산은 낡은 읽기가 한 건도 허용되지 않는 영역이라, 가용성을 조금 내주더라도 트랜잭션의 확정된 상태 위에서만 움직이게 했습니다.

결국 이 논문에서 가져갈 문장은 하나입니다. 일관성이냐 가용성이냐는 시스템 전체가 아니라 연산 단위로 정하는 문제다. 장바구니의 "담기"와 결제의 "청구"는 같은 서비스 안에서도 다른 다이얼 값을 가져야 합니다. Dynamo는 그 선택지를 처음으로 상품 진열대에 올려 보여준 논문이고, 그 아이디어는 Cassandra와 Riak으로, 이름은 DynamoDB로 이어졌습니다. 15년이 지난 지금 읽어도, 청구서를 먼저 보여주는 설계의 정직함은 낡지 않았습니다.
