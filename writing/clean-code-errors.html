클린코드 · 에러 — 예외를 삼키지 않는 법

2020년 11월• 약 12분• 클린코드 · 예외 처리 · NestJS

민원이 매달 200건씩 쌓이던 시절, 원인을 파다 보면 자주 만나는 코드가 있었습니다. catch (e) { logger.error(e); } — 로그 한 줄 찍고 아무 일 없었다는 듯 지나가는 코드. 시스템은 "정상"이었고, 실패는 로그 파일 속에서만 조용히 쌓였으며, 그 실패를 처음 알려주는 건 언제나 고객의 민원이었습니다. 이 글은 그 경험에서 나온 예외 처리의 규칙들입니다. 원칙의 나열이 아니라, 왜 그 규칙이 성립하는지와 어디서 깨지는지까지 다룹니다.

먼저 짚고 가기
예외Exception
프로그램이 실행되다 정상 흐름을 이어갈 수 없는 문제(없는 데이터, 끊긴 DB 등)를 만났을 때 "여기서 뭔가 잘못됐다"고 알리는 신호입니다. 화재경보기처럼, 울렸다면 누군가는 반응해야 합니다. 이 글의 핵심은 그 경보를 끄지만 말고 제대로 받는 법입니다.
도메인 예외 vs 인프라 예외domain / infrastructure
실패 원인이 우리가 정한 업무 규칙 안에 있으면 도메인 예외(예: 이미 처리된 신고), 우리 규칙 밖의 바깥 환경에 있으면 인프라 예외(예: DB 끊김)입니다. 어느 쪽인지에 따라 대응(사용자에게 알림 vs 재시도·운영자 호출)이 완전히 달라지기 때문에 나눕니다.
멱등성Idempotency
같은 요청을 몇 번을 반복해도 결과가 한 번 한 것과 똑같이 유지되는 성질입니다. 엘리베이터 버튼을 여러 번 눌러도 한 번 누른 것과 같은 것처럼요. 조회는 원래 멱등하지만 결제·저장 같은 쓰기는 그렇지 않아서, 안전하게 재시도하려면 별도 장치가 필요합니다.
부패 방지 계층Anti-Corruption Layer
바깥 시스템(ORM·외부 API)의 사정과 언어가 우리 코드 속으로 새어 들어오지 못하게 막고, 그 경계에서 우리 도메인의 언어로 번역해 주는 통역관 같은 층입니다. 덕분에 바깥을 갈아끼워도 안쪽 코드는 흔들리지 않습니다.
전역 예외 필터Exception Filter
NestJS에서 어디서 발생했든 처리되지 않은 예외를 맨 마지막에 한곳으로 모아 HTTP 응답으로 바꿔 주는 장치입니다. 예외의 유일한 '출구'를 하나로 만들어, 로그와 응답 형식을 일관되게 지킬 수 있습니다.

로그만 찍고 지나간 자리에서 장애가 자란다

먼저 "예외를 삼킨다"의 정확한 의미부터 잡겠습니다. 예외를 catch한 뒤 복구하지도, 다시 던지지도 않고 정상 흐름으로 복귀하는 것. 로그를 찍었어도 마찬가지입니다. 로그는 처리의 흔적이지 처리가 아니니까요.

// Before — 실패가 성공처럼 흘러간다
async approveReport(id: string): Promise<void> {
  try {
    const report = await this.repo.findOneOrFail(id);
    report.approve();
    await this.repo.save(report);
  } catch (e) {
    this.logger.error(e);   // 여기서 끝. 호출자는 아무것도 모른다
  }
}

이 코드가 왜 위험한지 구체적으로 따져 보겠습니다. save가 실패해도 이 메서드는 정상 반환합니다. 컨트롤러는 HTTP 200을 내려주고, 사용자 화면에는 "처리 완료"가 뜹니다. 하지만 DB에는 아무것도 반영되지 않았습니다. 이 지점에서 세 가지가 동시에 무너집니다.

  • 계약 위반 — 호출자는 "반환됐다 = 성공했다"를 믿고 다음 로직을 진행합니다. 그 믿음 위에 쌓인 모든 후속 동작이 거짓 전제 위에 서 있게 됩니다.
  • 정합성 붕괴 — 사용자가 아는 상태와 시스템의 상태가 갈라집니다. 이 간극은 시간이 지날수록 벌어지고, 나중에는 어느 쪽이 진실인지조차 알기 어려워집니다.
  • 발견 지연 — 장애가 터진 곳이 아니라 전혀 다른 곳에서, 한참 뒤에 발견됩니다. 실패 지점과 발견 지점이 멀수록 디버깅 비용은 기하급수로 늘어납니다.

민원 200건 시절 반복 민원의 상당수가 이 패턴이었습니다. "처리했다고 떴는데 안 됐어요"라는 민원은 코드 어딘가에서 예외가 삼켜졌다는 신호였고, 우리는 그 신호를 로그가 아니라 고객의 입으로 받고 있었던 겁니다. 그래서 첫 번째 규칙은 이렇습니다 — 복구할 수 없다면 던져라. 예외를 멈춰 세울 자격은 "처리할 수 있는 코드"에게만 있다.

예외를 계층으로 설계한다 — 도메인 vs 인프라

"던져라"만으로는 부족합니다. 받는 쪽이 무엇을 받았는지 구분할 수 없으면 결국 catch (e)로 뭉뚱그려 잡게 되고, 삼키는 코드로 되돌아갑니다. 그래서 예외에도 설계가 필요합니다. 제가 SafeIn 백엔드를 설계하며 세운 기준은 단 하나의 질문이었습니다 — "이 실패의 원인이 우리 도메인 규칙 안에 있는가, 바깥에 있는가?"

  • 도메인 예외 — 이미 처리된 신고를 다시 처리하려 함, 권한 없는 접근, 검증 실패. 요청 자체가 규칙에 어긋난 것이므로 같은 요청을 다시 보내도 결과는 같습니다. 사용자에게 이유를 돌려줘야 할 대상입니다.
  • 인프라 예외 — DB 커넥션 끊김, 외부 API 타임아웃, 네트워크 오류. 우리 규칙과 무관한 바깥의 실패이므로 시간이 지나면 성공할 수도 있습니다. 사용자가 아니라 운영자가 알아야 할 대상입니다.
// 도메인 예외 — 규칙 위반. 재시도해도 결과는 같다
export abstract class DomainException extends Error {
  abstract readonly code: string;        // 'REPORT_ALREADY_RESOLVED' 등
  readonly retryable = false;
  constructor(message: string, readonly context?: Record<string, unknown>) {
    super(message);
  }
}

export class ReportAlreadyResolvedError extends DomainException {
  readonly code = 'REPORT_ALREADY_RESOLVED';
  constructor(reportId: string) {
    super(`이미 처리된 신고입니다: ${reportId}`, { reportId });
  }
}

// 인프라 예외 — 바깥의 실패. 재시도 여부를 함께 실어 보낸다
export class InfrastructureException extends Error {
  constructor(
    message: string,
    readonly retryable: boolean,
    options?: { cause?: unknown },
  ) {
    super(message, options);   // cause 체인으로 원본을 보존
  }
}

이 구분이 왜 성립하느냐 — 분류가 곧 처리 전략을 결정하기 때문입니다. 도메인 예외는 4xx로 변환해 사용자에게 이유를 돌려주면 끝이고, 알림을 울릴 필요가 없습니다. 인프라 예외는 5xx이고, 재시도 후보이며, 반복되면 운영자를 깨워야 합니다. 예외 클래스가 이 정보를 스스로 들고 있으면, 받는 쪽은 instanceof 한 번으로 전략을 고를 수 있습니다. 반대로 모든 예외가 Error 하나라면 받는 쪽은 메시지 문자열을 파싱하거나 — 최악의 경우 — 그냥 삼키게 됩니다. 예외 계층은 던지는 쪽의 사치가 아니라 받는 쪽의 생존 조건입니다.

재시도 가능한가 — 그리고 그 분류가 깨지는 지점

인프라 예외에 retryable을 실은 이유는 크롤링 파이프라인을 운영하며 배웠습니다. 수십 개 외부 사이트를 긁다 보면 실패는 일상입니다. 그런데 실패에는 두 부류가 있었습니다. 일시적인 타임아웃·429는 기다렸다 다시 하면 되는 실패이고, 파싱 구조 변경·404는 백 번 재시도해도 안 되는 실패입니다. 이 둘을 구분하지 않고 일괄 재시도하면, 안 될 요청을 반복해 상대 서버를 두드리며 시간과 신뢰를 함께 태웁니다.

// 재시도는 '재시도해도 되는 실패'에만 — 지수 백오프와 함께
async fetchPage(url: string): Promise<RawPage> {
  return withRetry(() => this.http.get(url), {
    retries: 3,
    backoff: (n) => 2 ** n * 1_000,      // 2s → 4s → 8s
    retryOn: (e) =>
      e instanceof InfrastructureException && e.retryable,
  });
}

여기까지는 "도메인 = 재시도 불가, 인프라 = 재시도 가능"이라는 깔끔한 규칙처럼 보입니다. 하지만 이 규칙에는 분명한 경계 사례가 있습니다. 쓰기 요청의 타임아웃입니다. 결제 API에 요청을 보냈는데 타임아웃이 났다고 합시다. 분류상 명백한 인프라 예외지만 — 타임아웃은 실패의 확인이 아니라 결과를 모른다는 뜻입니다. 상대 서버는 요청을 정상 처리했는데 응답만 늦었을 수 있습니다. 여기서 기계적으로 재시도하면 이중 결제가 됩니다. 즉, "인프라 예외 = 재시도 가능"은 연산이 멱등할 때만 성립하는 규칙입니다. 읽기는 마음껏 재시도해도 되지만, 쓰기는 멱등 키(idempotency key)를 먼저 갖추지 않았다면 재시도해서는 안 됩니다. 정산 시스템을 리팩토링할 때 이 원칙 때문에 재시도 로직보다 멱등 키 설계를 먼저 했습니다. 순서가 반대면 재시도는 기능이 아니라 흉기가 됩니다.

경계에서 번역한다 — catch가 있어야 할 자리

예외 계층을 만들어도, 변환을 아무 데서나 하면 코드가 다시 지저분해집니다. 레거시 UseCase 코드를 개선하며 가장 많이 지운 패턴이 이것이었습니다 — 서비스 계층 곳곳에 흩어진 ORM 예외 처리.

// Before — ORM의 사정이 서비스 계층까지 새어 나온다
async createReport(dto: CreateReportDto) {
  try {
    return await this.reportRepo.save(dto.toEntity());
  } catch (e) {
    // 서비스가 PostgreSQL 에러 코드를 알아야 한다?
    if (e instanceof QueryFailedError && e.driverError.code === '23505') {
      throw new ConflictException();   // HTTP 예외가 비즈니스 로직 한복판에
    }
    throw e;
  }
}

이 코드의 문제는 동작이 아니라 지식의 위치입니다. "PostgreSQL의 23505는 유니크 제약 위반"이라는 지식은 영속성 계층의 사정인데, 그것이 비즈니스 로직 한가운데에 박혀 있습니다. ORM을 바꾸면 서비스 전체를 수정해야 하고, 같은 판별 코드가 save를 호출하는 모든 곳에 복제됩니다. 해결은 경계에서 한 번만 번역하는 것입니다.

// After — 리포지토리(경계)가 바깥의 언어를 도메인의 언어로 번역한다
async save(report: Report): Promise<Report> {
  try {
    return await this.orm.save(report);
  } catch (e) {
    if (isUniqueViolation(e)) {
      throw new DuplicateReportError(report.id);    // 도메인의 언어
    }
    throw new InfrastructureException(
      'report 저장 실패',
      isTransientDbError(e),        // 커넥션류면 true, 제약 위반이면 false
      { cause: e },                 // 원본 스택은 cause로 보존
    );
  }
}

이제 서비스는 DuplicateReportError라는 도메인의 언어만 다루면 되고, ORM 교체는 리포지토리 안에서 끝납니다. 이것이 DDD에서 말하는 부패 방지 계층(Anti-Corruption Layer)의 예외 버전입니다. 크롤링 파이프라인에서 사이트별 Adapter가 각기 다른 HTTP 라이브러리의 예외를 공통 예외로 번역했던 것과 같은 원리입니다. 규칙으로 정리하면 — catch는 두 곳에만 둔다. 바깥 세계와 만나는 경계(번역), 그리고 최종 처리 지점(응답 변환). 그 사이의 모든 계층은 예외를 그냥 통과시킨다. 중간 계층의 try/catch는 대부분 삼킴의 시작입니다.

NestJS 전역 필터 — 처리를 한곳으로 모은다

"최종 처리 지점"을 NestJS에서는 전역 exception filter로 만듭니다. 모든 컨트롤러에 흩어져 있던 try/catch를 걷어내고, 예외 → HTTP 응답 변환을 이 한곳에 모읍니다.

// 전역 필터 — 예외의 '출구'는 여기 하나뿐이다
@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  private readonly logger = new Logger('Exceptions');

  catch(exception: unknown, host: ArgumentsHost) {
    const res = host.switchToHttp().getResponse<Response>();

    if (exception instanceof DomainException) {
      // 규칙 위반은 장애가 아니다 — warn이면 충분
      this.logger.warn(`${exception.code}: ${exception.message}`);
      return res.status(statusOf(exception.code))   // code → 4xx 매핑표
        .json({ code: exception.code, message: exception.message });
    }

    if (exception instanceof HttpException) {
      return res.status(exception.getStatus()).json(exception.getResponse());
    }

    // 여기 도달한 예외만이 진짜 '장애'다 — 스택까지 남기고 500
    this.logger.error(
      exception instanceof Error ? exception.stack : String(exception),
    );
    return res.status(500)
      .json({ code: 'INTERNAL_ERROR', message: '일시적인 오류가 발생했습니다.' });
  }
}

이 구조가 주는 이득은 세 가지입니다. 첫째, 도메인 예외의 code가 그대로 API 에러 코드가 됩니다. 프런트엔드는 문자열 메시지가 아니라 안정된 코드로 분기할 수 있습니다. 둘째, 500 응답에서 내부 정보가 새지 않습니다. 스택과 원본 메시지는 로그에만 남고, 사용자에게는 정제된 문장만 나갑니다. 셋째, 가장 중요한 것 — 로그 레벨이 예외의 의미와 일치합니다. 사용자의 잘못된 요청(도메인 예외)은 warn, 시스템의 실패만 error. 이 구분이 없으면 "이미 처리된 신고입니다"가 하루 수백 건씩 error로 쌓이고, 진짜 장애 로그가 그 소음에 묻힙니다. 민원 시절 로그를 열 때마다 겪던 일입니다. 모든 것이 error인 로그는 아무것도 error가 아닌 로그와 같습니다.

*

실무 팁 — 로그는 출구에서 한 번, 원본은 cause로. 각 계층에서 잡을 때마다 로그를 찍으면 예외 하나가 로그 4줄이 되어 집계와 알림 기준을 다 흐립니다. 로깅은 전역 필터(출구) 한 곳에서만 하고, 중간 계층은 new InfrastructureException(msg, retryable, { cause: e })처럼 cause 체인으로 원본 스택을 보존해서 던지기만 하세요. Node 16.9+의 Error cause는 이걸 위해 있습니다. 번역하면서 원본을 버리면, 필터에 도착한 예외는 "저장 실패"라고만 말할 뿐 어느 쿼리가 왜 죽었는지 말하지 못합니다.

삼켜도 되는 예외는 있다 — 원칙의 경계

"절대 삼키지 마라"를 문자 그대로 지키면 오히려 가용성이 무너지는 지점이 있습니다. 주 흐름의 성공과 무관한 부수 작업입니다. 신고 처리가 끝난 뒤의 푸시 알림 발송, 캐시 갱신, 통계 이벤트 적재 — 이런 작업의 실패로 이미 성공한 본 처리를 500으로 되돌리는 건 사용자 입장에서 더 큰 거짓말입니다. 처리는 됐는데 "실패했습니다"라고 말하는 셈이니까요.

// 유일하게 허용되는 삼킴 — 명시적으로, 흔적과 함께
const report = await this.reports.resolve(id, admin.id);   // 주 흐름 — 실패하면 그대로 던진다

try {
  await this.notifier.send(report.workerId, '처리 완료');
} catch (e) {
  // 알림 실패 ≠ 처리 실패. 주 흐름을 막지 않는다 — 단, 반드시 관측 가능하게
  this.logger.warn(`알림 발송 실패 reportId=${report.id}`, e);
  this.metrics.increment('notification.failure');
}

다만 이 예외적 허용에는 조건이 붙습니다. 이 조건들이 빠지면 그냥 첫 번째 절의 나쁜 코드로 되돌아갑니다.

  • 범위가 좁을 것 — try 블록은 부수 작업 한 줄만 감쌉니다. 주 흐름까지 감싸는 순간 삼킴이 됩니다.
  • 의도가 코드에 적혀 있을 것 — "알림 실패는 처리 실패가 아니다"라는 판단을 주석으로 남깁니다. 실수로 삼킨 것과 결정하고 삼킨 것은 다른 코드입니다.
  • 관측 가능할 것 — warn 로그와 메트릭 카운터를 남깁니다. 알림 실패율이 오르면 사람이 알아야 합니다. 흔적 없는 삼킴은 결정이 아니라 은폐입니다.
!

함정 — await 없는 비동기 호출은 필터도 못 잡습니다. 부수 작업이라고 this.notifier.send(...)를 await(또는 명시적 .catch) 없이 던져두면, 그 Promise의 rejection은 요청 컨텍스트를 벗어나 unhandledRejection으로 흘러갑니다. 앞서 만든 전역 필터에도 안 잡히고, 응답과의 연결 고리도 끊깁니다. try/catch가 없는 것보다 나쁜, 구조적으로 삼켜진 예외입니다. 부수 작업을 의도적으로 분리할 때도 반드시 .catch를 붙여 흔적을 남기세요.

결국 "예외를 삼키지 마라"의 본질은 try/catch 문법의 문제가 아니라 실패라는 정보를 소멸시키지 마라는 것입니다. 호출자에게 던지든, 응답 코드로 변환하든, 메트릭으로 남기든 — 실패는 반드시 어딘가에 도착해야 합니다. 도착지 없이 사라지는 실패가 하나씩 늘어날 때마다, 시스템은 조금씩 거짓말쟁이가 됩니다.

정리 — 예외 처리 체크리스트

민원 200건을 30건으로 줄이는 과정의 절반은 프로세스였지만, 나머지 절반은 이렇게 실패가 조용히 사라지던 길목을 하나씩 막는 일이었습니다. 핵심 규칙만 추리면 다음과 같습니다.

  • 복구할 수 없으면 던진다. 로그만 찍고 정상 복귀하는 catch는 처리 흔적이지 처리가 아니다.
  • 도메인 예외와 인프라 예외를 계층으로 가른다. 원인이 규칙 안이면 4xx·재시도 무의미, 바깥이면 5xx·재시도 후보. 분류가 처리 전략을 결정한다.
  • 재시도는 retryable일 때만, 그리고 연산이 멱등할 때만. 쓰기의 타임아웃은 실패 확인이 아니다 — 멱등 키 없이 재시도하지 않는다.
  • catch는 경계와 출구, 두 곳에만. 경계(리포지토리·어댑터)에서는 바깥의 예외를 도메인의 언어로 번역하고, 중간 계층은 통과시킨다.
  • 출구는 전역 필터 하나로. 도메인 예외는 warn + 4xx, 미지의 예외만 error + 스택 + 500. 로그 레벨이 예외의 의미와 일치해야 알림이 살아남는다.
  • 로그는 출구에서 한 번, 원본은 cause로 보존한다. 계층마다 찍는 중복 로그는 집계를 죽인다.
  • 부수 작업만 예외적으로 삼킬 수 있다. 좁은 범위 + 주석으로 남긴 의도 + 메트릭. 셋 중 하나라도 빠지면 은폐다.
  • 비동기 호출은 반드시 await 또는 .catch. 떠다니는 Promise의 rejection은 필터조차 잡지 못한다.
