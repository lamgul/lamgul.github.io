클린코드 · 에러 — 예외를 삼키지 않는 법

2020년 11월• 클린코드 · 예외 처리 · NestJS

민원이 매달 200건씩 쌓이던 때가 있었습니다. 원인을 하나씩 따라가다 보면 catch (e) { logger.error(e); } 같은 코드를 자주 만났습니다. 로그 한 줄만 남기고 아무 일 없었다는 듯 지나가는 코드였습니다. 지표상 시스템은 정상이었고 실패는 로그 파일 안에만 쌓였는데, 정작 그 실패를 저에게 알려주는 건 대개 고객의 민원이었습니다. 그때 정리해 둔 예외 처리 규칙들을 적어 둡니다. 규칙만 나열하면 나중에 제가 다시 봐도 이유를 잊을 것 같아서, 왜 그렇게 정했는지와 어디서 그 규칙이 잘 맞지 않았는지도 함께 남깁니다.

먼저 짚고 가기
예외Exception
정상 제어 흐름을 이어갈 수 없을 때 던져지는 객체이자, 호출 스택을 거슬러 올라가며 처리할 곳을 찾는 제어 흐름 장치입니다. throw가 실행되면 그 함수는 값을 반환하지 못한 채 즉시 중단되고, 런타임은 조건에 맞는 catch 프레임을 만날 때까지 스택을 되감습니다. 자바스크립트에는 checked exception이 없어 어떤 예외가 올라올지는 시그니처에 드러나지 않고, 값이면 무엇이든 던질 수 있어 catch 변수의 타입은 unknown입니다. 그래서 중간 계층에서 한 번 붙잡히고 끝나면 그 실패는 호출자 쪽에 아무 흔적도 남기지 않습니다.
도메인 예외 vs 인프라 예외domain / infrastructure
예외가 던져진 위치가 아니라 실패의 원인이 어디에 있는지로 예외 계층을 가르는 분류 축입니다. 요청이 우리가 정의한 업무 불변식을 어긴 것이면 도메인 예외(이미 처리된 신고, 권한 없음), 프로세스 바깥의 자원이 기대한 응답을 주지 못한 것이면 인프라 예외(커넥션 끊김, 타임아웃)입니다. 이 글에서는 그 분류를 DomainException·InfrastructureException 두 기반 클래스로 만들고 code와 retryable을 예외 객체에 실어, 받는 쪽이 instanceof 한 번으로 상태 코드·로그 레벨·재시도 여부를 정하게 했습니다. 기준이 원인이라서, 리포지토리 안에서 잡힌 유니크 제약 위반도 도메인 예외 쪽에 놓입니다.
멱등성Idempotency
같은 인자로 연산을 두 번 이상 적용해도 시스템 상태가 한 번 적용했을 때와 같아지는 성질입니다(f(f(x)) = f(x)). 상태가 같다는 뜻이지 응답까지 같다는 뜻은 아니어서, DELETE는 멱등이지만 두 번째 호출은 404를 돌려줄 수 있습니다. HTTP 명세(RFC 9110)는 GET·PUT·DELETE를 멱등한 메서드로, POST를 그렇지 않은 메서드로 규정합니다. 쓰기를 멱등하게 만들려면 클라이언트가 붙인 멱등 키를 서버가 저장해 두고, 같은 키의 재요청은 다시 실행하는 대신 처음 결과를 돌려주도록 해야 합니다.
부패 방지 계층Anti-Corruption Layer
에릭 에반스의 Domain-Driven Design(2003)에 나오는 컨텍스트 통합 패턴으로, 외부 모델을 우리 모델로 옮기는 전용 번역 계층을 경계에 세우는 설계입니다. 레거시 시스템·서드파티 API·ORM 같은 바깥의 개념이 안쪽 도메인 모델의 어휘를 오염시키지 못하게 막는 것이 목적입니다. 이 글에서는 리포지토리가 QueryFailedError와 SQLSTATE 23505(유니크 제약 위반)를 DuplicateReportError로 바꾸는 예외 번역이 그 역할을 맡습니다. 바깥 타입을 그대로 통과시키는 얇은 래퍼와는 다릅니다. 안쪽 코드가 여전히 바깥 타입을 알아야 한다면 계층만 하나 늘어난 것입니다.
전역 예외 필터Exception Filter
NestJS 요청 파이프라인의 마지막에 놓이는 구성 요소로, ExceptionFilter 인터페이스를 구현하고 @Catch()로 받을 예외 타입을 지정하는 클래스입니다. 라우트 핸들러 밖으로 빠져나온 예외가 도착하면 catch(exception, host)가 호출되고, ArgumentsHost로 실행 컨텍스트를 열어 응답 객체를 직접 다룹니다. @Catch()에 인자를 주지 않으면 모든 예외를 받고, useGlobalFilters()나 APP_FILTER 프로바이더로 등록하면 전 라우트에 걸립니다. 다만 요청 수명주기를 타고 전파된 예외만 도착해서, await 없이 떠다니는 Promise의 rejection은 여기에 닿지 않습니다.

로그만 찍고 지나간 자리에서 장애가 자란다

먼저 "예외를 삼킨다"가 무엇인지부터 정리하겠습니다. 예외를 catch한 뒤 복구하지도 않고 다시 던지지도 않은 채 정상 흐름으로 돌아가는 경우를 말합니다. 로그를 남겼더라도 마찬가지입니다. 로그는 처리한 흔적일 뿐 처리 자체는 아니기 때문입니다.

// Before — 실패가 성공처럼 흘러간다
async approveReport(id: string): Promise<void> {
  try {
    const report = await this.repo.findOneOrFail(id);
    report.approve();
    await this.repo.save(report);
  } catch (e) {
    this.logger.error(e);   // 여기서 끝. 호출자는 아무것도 모른다
  }
}

이 코드가 왜 위험한지 하나씩 따져 보면 이렇습니다. save가 실패해도 이 메서드는 정상적으로 반환하고, 그래서 컨트롤러는 HTTP 200을 내려주고 사용자 화면에는 "처리 완료"가 뜹니다. 정작 DB에는 아무것도 반영되지 않았는데도 그렇습니다. 여기서 세 가지 문제가 함께 생깁니다.

  • 계약 위반 — 호출자는 "반환됐다 = 성공했다"로 보고 다음 로직을 진행합니다. 그 뒤에 이어지는 동작들이 전부 틀린 전제 위에 놓입니다.
  • 정합성 붕괴 — 사용자가 아는 상태와 시스템의 상태가 갈라집니다. 이 간극은 시간이 지날수록 벌어져서, 나중에는 어느 쪽이 맞는지 확인하는 데만도 시간이 걸립니다.
  • 발견 지연 — 장애가 발생한 곳이 아니라 다른 곳에서, 한참 뒤에 드러납니다. 실패 지점과 발견 지점이 멀수록 디버깅에 드는 시간도 그만큼 늘어납니다.

민원 200건 시절 반복해서 들어오던 민원의 상당수가 이 패턴이었습니다. "처리했다고 떴는데 안 됐어요"라는 민원은 코드 어딘가에서 예외가 삼켜졌다는 신호였는데, 저희는 그 신호를 로그가 아니라 고객에게서 받고 있었습니다. 그래서 첫 번째 규칙을 복구할 수 없으면 던진다로 정했습니다. 예외를 멈춰 세우는 건 그 예외를 처리할 수 있는 코드만 하기로 했습니다.

예외를 계층으로 설계한다 — 도메인 vs 인프라

그런데 던지기만 해서는 부족했습니다. 받는 쪽에서 무엇을 받았는지 구분할 수 없으면 catch (e)로 뭉뚱그려 잡게 되고, 그러다 보면 다시 삼키는 코드로 돌아갔습니다. 예외 자체에도 설계가 필요했던 셈입니다. SafeIn 백엔드를 설계할 때는 "이 실패의 원인이 우리 도메인 규칙 안에 있는가, 바깥에 있는가"를 기준으로 나눴습니다.

  • 도메인 예외 — 이미 처리된 신고를 다시 처리하려는 요청, 권한 없는 접근, 검증 실패입니다. 요청 자체가 규칙에 어긋난 것이라 같은 요청을 다시 보내도 결과는 같습니다. 사용자에게 이유를 돌려줘야 하는 쪽입니다.
  • 인프라 예외 — DB 커넥션 끊김, 외부 API 타임아웃, 네트워크 오류입니다. 우리 규칙과 무관한 바깥의 실패라서 시간이 지나면 성공할 수도 있습니다. 사용자가 아니라 운영자가 알아야 하는 쪽입니다.
// 도메인 예외 — 규칙 위반. 재시도해도 결과는 같다
export abstract class DomainException extends Error {
  abstract readonly code: string;        // 'REPORT_ALREADY_RESOLVED' 등
  readonly retryable = false;
  constructor(message: string, readonly context?: Record<string, unknown>) {
    super(message);
  }
}

export class ReportAlreadyResolvedError extends DomainException {
  readonly code = 'REPORT_ALREADY_RESOLVED';
  constructor(reportId: string) {
    super(`이미 처리된 신고입니다: ${reportId}`, { reportId });
  }
}

// 인프라 예외 — 바깥의 실패. 재시도 여부를 함께 실어 보낸다
export class InfrastructureException extends Error {
  constructor(
    message: string,
    readonly retryable: boolean,
    options?: { cause?: unknown },
  ) {
    super(message, options);   // cause 체인으로 원본을 보존
  }
}

이렇게 나눈 건 분류가 그대로 처리 전략이 되기 때문이었습니다. 도메인 예외는 4xx로 변환해 사용자에게 이유를 돌려주면 되고, 알림을 울릴 필요가 없습니다. 인프라 예외는 5xx이고, 재시도 후보이며, 반복되면 운영자에게 알려야 합니다. 예외 클래스가 이 정보를 스스로 들고 있으면 받는 쪽은 instanceof 한 번으로 전략을 고를 수 있습니다. 반대로 모든 예외가 Error 하나뿐이면 받는 쪽은 메시지 문자열을 파싱하거나 그냥 삼키게 됩니다. 예외 계층은 던지는 쪽보다 받는 쪽을 위한 장치라고 생각하게 된 이유입니다.

재시도 가능한가 — 그리고 그 분류가 깨지는 지점

인프라 예외에 retryable을 실어 둔 건 크롤링 파이프라인을 운영하면서 필요해진 것입니다. 수십 개 외부 사이트를 긁다 보면 실패는 늘 있는 일이었는데, 실패의 성격이 두 가지로 나뉘었습니다. 일시적인 타임아웃이나 429는 조금 기다렸다 다시 요청하면 되는 실패였고, 파싱 구조 변경이나 404는 몇 번을 다시 보내도 같은 결과였습니다. 이 둘을 구분하지 않고 전부 재시도하다 보니 될 리 없는 요청으로 상대 서버를 계속 두드리게 됐습니다.

// 재시도는 '재시도해도 되는 실패'에만 — 지수 백오프와 함께
async fetchPage(url: string): Promise<RawPage> {
  return withRetry(() => this.http.get(url), {
    retries: 3,
    backoff: (n) => 2 ** n * 1_000,      // 2s → 4s → 8s
    retryOn: (e) =>
      e instanceof InfrastructureException && e.retryable,
  });
}

여기까지만 보면 "도메인 = 재시도 불가, 인프라 = 재시도 가능"으로 깔끔하게 나뉘는 것 같은데, 잘 맞지 않는 경우가 있었습니다. 쓰기 요청의 타임아웃입니다. 결제 API에 요청을 보냈는데 타임아웃이 났다면 분류상으로는 인프라 예외입니다. 다만 타임아웃은 실패했다는 확인이 아니라 결과를 모른다는 뜻입니다. 상대 서버는 요청을 정상 처리했는데 응답만 늦었을 수도 있습니다. 이 상태에서 그대로 재시도하면 이중 결제가 됩니다. 그래서 "인프라 예외 = 재시도 가능"은 연산이 멱등할 때만 성립하는 규칙으로 다시 적어 두었습니다. 읽기는 재시도해도 괜찮지만, 쓰기는 멱등 키(idempotency key)를 갖추기 전에는 재시도하지 않기로 했습니다. 정산 시스템을 리팩토링할 때도 이 순서 때문에 재시도 로직보다 멱등 키 설계를 먼저 했습니다. 순서를 반대로 잡았다면 재시도가 오히려 사고를 키웠을 것 같습니다.

경계에서 번역한다 — catch가 있어야 할 자리

예외 계층을 만들어 두어도 변환을 아무 데서나 하면 코드가 다시 지저분해졌습니다. 레거시 UseCase 코드를 개선할 때 가장 많이 지운 것도 서비스 계층 곳곳에 흩어져 있던 ORM 예외 처리였습니다.

// Before — ORM의 사정이 서비스 계층까지 새어 나온다
async createReport(dto: CreateReportDto) {
  try {
    return await this.reportRepo.save(dto.toEntity());
  } catch (e) {
    // 서비스가 PostgreSQL 에러 코드를 알아야 한다?
    if (e instanceof QueryFailedError && e.driverError.code === '23505') {
      throw new ConflictException();   // HTTP 예외가 비즈니스 로직 한복판에
    }
    throw e;
  }
}

이 코드는 동작이 틀린 게 아니라 지식이 놓인 위치가 어긋나 있습니다. "PostgreSQL의 23505는 유니크 제약 위반"이라는 지식은 영속성 계층의 사정인데, 그것이 비즈니스 로직 한가운데에 들어와 있습니다. 이대로 두면 ORM을 바꿀 때 서비스 전체를 고쳐야 하고, 같은 판별 코드가 save를 호출하는 곳마다 복제됩니다. 그래서 경계에서 한 번만 번역하도록 옮겼습니다.

// After — 리포지토리(경계)가 바깥의 언어를 도메인의 언어로 번역한다
async save(report: Report): Promise<Report> {
  try {
    return await this.orm.save(report);
  } catch (e) {
    if (isUniqueViolation(e)) {
      throw new DuplicateReportError(report.id);    // 도메인의 언어
    }
    throw new InfrastructureException(
      'report 저장 실패',
      isTransientDbError(e),        // 커넥션류면 true, 제약 위반이면 false
      { cause: e },                 // 원본 스택은 cause로 보존
    );
  }
}

이제 서비스는 DuplicateReportError라는 도메인의 언어만 다루면 되고, ORM 교체는 리포지토리 안에서 끝납니다. 이것이 DDD에서 말하는 부패 방지 계층(Anti-Corruption Layer)의 예외 버전입니다. 크롤링 파이프라인에서 사이트별 Adapter가 각기 다른 HTTP 라이브러리의 예외를 공통 예외로 번역했던 것과 같은 원리입니다. 규칙으로 정리하면 catch는 두 곳에만 둡니다. 바깥 세계와 만나는 경계에서 번역할 때, 그리고 최종 처리 지점에서 응답으로 바꿀 때입니다. 그 사이의 계층은 예외를 그냥 통과시킵니다. 중간 계층에 try/catch가 보이면 대체로 삼킴이 시작되는 자리였습니다.

NestJS 전역 필터 — 처리를 한곳으로 모은다

앞에서 말한 "최종 처리 지점"은 NestJS에서 전역 exception filter로 만들었습니다. 컨트롤러마다 흩어져 있던 try/catch를 걷어내고, 예외 → HTTP 응답 변환을 이 한곳에 모았습니다.

// 전역 필터 — 예외의 '출구'는 여기 하나뿐이다
@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  private readonly logger = new Logger('Exceptions');

  catch(exception: unknown, host: ArgumentsHost) {
    const res = host.switchToHttp().getResponse<Response>();

    if (exception instanceof DomainException) {
      // 규칙 위반은 장애가 아니다 — warn이면 충분
      this.logger.warn(`${exception.code}: ${exception.message}`);
      return res.status(statusOf(exception.code))   // code → 4xx 매핑표
        .json({ code: exception.code, message: exception.message });
    }

    if (exception instanceof HttpException) {
      return res.status(exception.getStatus()).json(exception.getResponse());
    }

    // 여기 도달한 예외만이 진짜 '장애'다 — 스택까지 남기고 500
    this.logger.error(
      exception instanceof Error ? exception.stack : String(exception),
    );
    return res.status(500)
      .json({ code: 'INTERNAL_ERROR', message: '일시적인 오류가 발생했습니다.' });
  }
}

이 구조로 바꾸고 나서 좋았던 점은 세 가지였습니다. 첫째, 도메인 예외의 code가 그대로 API 에러 코드가 되어서 프런트엔드가 문자열 메시지 대신 안정된 코드로 분기할 수 있었습니다. 둘째, 500 응답에서 내부 정보가 새지 않습니다. 스택과 원본 메시지는 로그에만 남고 사용자에게는 정제된 문장만 나갑니다. 셋째, 로그 레벨이 예외의 의미와 일치합니다. 사용자의 잘못된 요청인 도메인 예외는 warn으로, 시스템의 실패만 error로 남깁니다. 이 구분이 없던 때는 "이미 처리된 신고입니다"가 하루 수백 건씩 error로 쌓여서, 정작 봐야 할 장애 로그가 그 사이에 묻혔습니다. 민원이 많던 시절 로그를 열 때마다 겪던 일입니다. 모든 줄이 error인 로그는 아무 줄도 error가 아닌 로그와 다르지 않았습니다.

*

실무 팁 — 로그는 출구에서 한 번, 원본은 cause로. 각 계층에서 잡을 때마다 로그를 찍으면 예외 하나가 로그 4줄이 되어 집계와 알림 기준이 흐려집니다. 그래서 로깅은 전역 필터(출구) 한 곳에서만 하고, 중간 계층은 new InfrastructureException(msg, retryable, { cause: e })처럼 cause 체인으로 원본 스택을 보존해 던지기만 하도록 두었습니다. Node 16.9+의 Error cause가 이 용도로 쓰입니다. 번역하면서 원본을 버리면 필터에 도착한 예외는 "저장 실패"라고만 말할 뿐, 어느 쿼리가 왜 실패했는지는 알려주지 못합니다.

삼켜도 되는 예외는 있다 — 원칙의 경계

"절대 삼키지 마라"를 문자 그대로 지키다 보면 오히려 가용성이 나빠지는 자리도 있었습니다. 주 흐름의 성공과 무관한 부수 작업입니다. 신고 처리가 끝난 뒤에 나가는 푸시 알림 발송, 캐시 갱신, 통계 이벤트 적재 같은 것들입니다. 이런 작업이 실패했다고 이미 성공한 본 처리를 500으로 되돌리면, 처리는 됐는데 실패했다고 알리는 셈이 됩니다.

// 유일하게 허용되는 삼킴 — 명시적으로, 흔적과 함께
const report = await this.reports.resolve(id, admin.id);   // 주 흐름 — 실패하면 그대로 던진다

try {
  await this.notifier.send(report.workerId, '처리 완료');
} catch (e) {
  // 알림 실패 ≠ 처리 실패. 주 흐름을 막지 않는다 — 단, 반드시 관측 가능하게
  this.logger.warn(`알림 발송 실패 reportId=${report.id}`, e);
  this.metrics.increment('notification.failure');
}

다만 여기에는 조건을 붙여 두었습니다. 이 조건이 빠지면 앞에서 본 코드와 다를 게 없어집니다.

  • 범위가 좁을 것 — try 블록은 부수 작업 한 줄만 감쌉니다. 주 흐름까지 감싸면 그때부터는 삼킨 것이 됩니다.
  • 의도가 코드에 적혀 있을 것 — "알림 실패는 처리 실패가 아니다"라는 판단을 주석으로 남깁니다. 실수로 삼킨 것과 정하고 삼킨 것은 나중에 읽을 때 차이가 큽니다.
  • 관측 가능할 것 — warn 로그와 메트릭 카운터를 남깁니다. 알림 실패율이 오르면 사람이 알아야 합니다. 흔적이 없으면 결정했다기보다 감춘 쪽에 가깝습니다.
!

함정 — await 없는 비동기 호출은 필터도 못 잡습니다. 부수 작업이라고 this.notifier.send(...)를 await(또는 명시적 .catch) 없이 던져두면, 그 Promise의 rejection은 요청 컨텍스트를 벗어나 unhandledRejection으로 흘러갑니다. 앞서 만든 전역 필터에도 잡히지 않고, 응답과의 연결 고리도 끊깁니다. try/catch를 아예 쓰지 않은 것보다 찾기 어려운 형태로 삼켜진 예외입니다. 그래서 부수 작업을 의도적으로 분리할 때도 .catch를 붙여 흔적을 남기도록 했습니다.

정리하고 보니 "예외를 삼키지 마라"는 try/catch 문법의 문제가 아니라 실패라는 정보를 없애지 말자는 이야기였습니다. 호출자에게 던지든, 응답 코드로 바꾸든, 메트릭으로 남기든 실패는 어딘가에는 도착해야 합니다. 도착지 없이 사라지는 실패가 늘어날수록 시스템이 말하는 상태와 실제 상태가 조금씩 어긋났습니다.

정리 — 예외 처리 체크리스트

민원을 200건에서 30건으로 줄이는 과정의 절반은 프로세스 개선이었고, 나머지 절반은 이렇게 실패가 조용히 사라지던 자리를 하나씩 막는 일이었습니다. 그때 정리한 규칙만 추리면 다음과 같습니다.

  • 복구할 수 없으면 던진다. 로그만 찍고 정상 복귀하는 catch는 처리한 흔적일 뿐 처리가 아니다.
  • 도메인 예외와 인프라 예외를 계층으로 가른다. 원인이 규칙 안이면 4xx이고 재시도는 의미가 없다. 바깥이면 5xx이고 재시도 후보다. 분류가 처리 전략을 정한다.
  • 재시도는 retryable이면서 연산이 멱등할 때만 한다. 쓰기의 타임아웃은 실패했다는 확인이 아니므로, 멱등 키 없이 재시도하지 않는다.
  • catch는 경계와 출구, 두 곳에만 둔다. 경계(리포지토리·어댑터)에서는 바깥의 예외를 도메인의 언어로 번역하고, 중간 계층은 통과시킨다.
  • 출구는 전역 필터 하나로 모은다. 도메인 예외는 warn + 4xx, 알 수 없는 예외만 error + 스택 + 500. 로그 레벨이 예외의 의미와 맞아야 알림이 제 역할을 한다.
  • 로그는 출구에서 한 번, 원본은 cause로 보존한다. 계층마다 찍는 중복 로그는 집계를 어렵게 만든다.
  • 부수 작업만 예외적으로 삼킬 수 있다. 좁은 범위, 주석으로 남긴 의도, 메트릭 셋 중 하나라도 빠지면 그냥 삼킨 것이다.
  • 비동기 호출에는 await 또는 .catch를 붙인다. 떠다니는 Promise의 rejection은 필터도 잡지 못한다.

다음 글에는 이 예외들이 실제로 던져지는지, 실패 경로를 어떻게 테스트했는지 적어 두려고 합니다.
