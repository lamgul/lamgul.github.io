논문 분석: Spanner — 시계로 트랜잭션을 줄 세우다

2024년 8월• Spanner · TrueTime · 논문

Google의 Spanner는 전 세계 데이터센터에 흩어진 트랜잭션의 순서를 물리 시계로 보장하는 데이터베이스입니다. 제가 배운 분산 시스템은 그 반대였습니다. 서버마다 시계가 조금씩 다르게 가니 이벤트의 순서는 시계가 아니라 메시지로 정하라고 했고, 저도 그렇게 알고 있었습니다. Spanner는 시계를 완벽하게 만들어서가 아니라, 시계가 얼마나 틀릴 수 있는지를 보장하고 그만큼 기다리는 방법으로 순서를 세웁니다. 그게 어떻게 성립하는지 궁금해서, 정산 시스템에서 트랜잭션 순서를 고민해 본 백엔드 개발자의 눈으로 논문을 읽고 남은 것을 적어 둡니다.

출처

Spanner: Google's Globally-Distributed Database

James C. Corbett, Jeffrey Dean 외 · OSDI 2012

공식 PDF ↗ · Google Research 페이지 ↗

먼저 짚고 가기
외부 일관성external consistency
트랜잭션 T1의 커밋이 T2의 시작보다 실제(절대) 시간에서 앞섰다면, 시스템이 부여하는 직렬화 순서에서도 반드시 s1 < s2가 되어야 한다는 정합성 조건입니다. 직렬성(serializability)에 이 실시간 순서 제약을 더한 것으로, 흔히 엄격한 직렬성(strict serializability)이라 부르는 가장 강한 보장입니다. 직렬성만으로는 실제로 먼저 끝난 트랜잭션이 직렬화 순서에서 뒤로 밀리는 재배열도 허용되지만, 외부 일관성은 그것을 금지합니다.
TrueTime
현재 시각을 단일 값이 아니라 구간 [earliest, latest]로 돌려주는 시계 API입니다(논문 3장, Table 1). 핵심 메서드 TT.now()는 호출 순간의 진짜 절대 시각 t가 언제나 earliest ≤ t ≤ latest를 만족한다는 불변식을 보장하고, TT.after()·TT.before()는 어떤 시각이 확실히 지났는지·아직 오지 않았는지를 이 구간 위에서 판정합니다. 오차를 없애는 것이 아니라 구간 폭으로 오차의 상한을 드러내는 장치라, 그 보장은 "시계 드리프트가 정해진 상한을 넘지 않는다"는 운영상의 전제 위에 섭니다.
불확실성 구간[earliest, latest]
TT.now()가 돌려주는 구간(논문의 TTinterval)으로, 진짜 절대 시각이 반드시 이 안에 있음이 보장됩니다. 구간의 반폭을 ε(엡실론)이라 하며, 이는 오차의 실측값이 아니라 보수적으로 잡은 상한입니다. ε은 마스터와 동기화한 직후 약 1ms까지 줄었다가 다음 동기화 전까지 초당 200μs씩 커지는 톱니 모양을 그리고(평균 약 4ms), commit-wait가 이 ε에 정비례하므로 시계 품질이 곧 커밋 지연으로 이어집니다.
커밋 대기commit-wait
읽기-쓰기 트랜잭션의 커밋 절차에 들어가는 규칙입니다(논문 4.1.2절). 잠금을 잡은 채 커밋 타임스탬프 s = TT.now().latest를 배정한 뒤, TT.after(s)가 참이 될 때까지—즉 s가 어느 시계로 보아도 확실한 과거가 될 때까지—잠금 해제와 커밋 노출을 미룹니다. 이 대기는 대략 2ε(논문 4.2.1절의 표현으로 최소 2ε̄)이며, 바로 이 절차가 "타임스탬프 순서 = 실제 시간 순서"를 성립시킵니다. 잠금 없이 과거를 읽는 읽기 전용 트랜잭션에는 적용되지 않습니다.
스냅샷 읽기snapshot read
클라이언트가 고른 과거 타임스탬프 t 시점의 데이터를 읽는 다중 버전(MVCC) 읽기입니다. 타임스탬프가 전역에서 순서를 가지므로 잠금 없이, t까지의 쓰기를 모두 반영한 복제본이면 어디서든 처리할 수 있습니다—각 복제본은 자신의 tsafe(논문 4.1.3절)가 t 이상인지만 확인합니다. 정합적인 과거 스냅샷을 보는 방식이라 방금 커밋된 최신 쓰기는 보이지 않을 수 있고, 읽기-쓰기 트랜잭션과 서로 막지 않는 것이 이 방식의 이점입니다.

1. 시계를 믿지 말라던 세계에서

분산 시스템이 물리 시계를 불신하는 데는 이유가 있습니다. NTP로 동기화한 서버들의 시계도 수 밀리초에서 수십 밀리초씩 어긋납니다. 서버 A의 시계로 10:00:00.005에 커밋된 트랜잭션과 서버 B의 시계로 10:00:00.003에 커밋된 트랜잭션이 있을 때, 어느 쪽이 먼저인지 시계만 보고는 알 수 없습니다. B의 시계가 4ms 느리게 가고 있었다면, B가 찍은 .003은 실제로는 .007이어서 A가 먼저였을 테니까요. 그래서 Lamport의 1978년 논문 이래로 분산 시스템은 논리 시계를 씁니다. "메시지를 주고받은 순서"라는 인과 관계로만 순서를 정하고, 벽시계는 참고용으로 밀어두는 접근입니다.

다만 논리 시계는 서로 통신한 적 없는 두 트랜잭션 사이의 순서까지는 정해 주지 못합니다. 사용자가 A 시스템에서 결제를 마치고 그 사실을 전화로 전달받은 동료가 B 시스템에서 결과를 조회하는 경우, 두 시스템은 메시지를 주고받은 적이 없지만 사람의 눈에는 선후 관계가 분명합니다. 이 "시스템 바깥의 순서"까지 지키는 것이 외부 일관성이고, Spanner는 세계 규모로 분산된 데이터베이스가 외부 일관성을 제공한 첫 사례라고 논문 1장에서 밝힙니다.

Spanner는 이 문제를 뒤집어서 접근합니다. 시계의 오차를 없애는 대신 측정 가능한 범위 안에 묶어 두고, 오차가 문제되는 그 구간만큼만 기다립니다. 오차를 0으로 만드는 것은 물리적으로 불가능하지만, "오차가 최대 ε을 넘지 않는다"는 보장은 공학으로 만들 수 있습니다. 그 공학이 TrueTime입니다.

2. TrueTime — 시각이 아니라 구간을 답하는 API

TrueTime의 API는 크지 않습니다(논문 3장, Table 1). 핵심은 TT.now()가 시각 하나가 아니라 구간 [earliest, latest]를 돌려준다는 점입니다. 호출한 순간의 진짜 절대 시각은 반드시 이 구간 안에 있습니다. 어디인지는 모르지만 밖은 아니라는 보장이고, 이후의 설계는 전부 이 한 가지 위에 서 있습니다.

// TrueTime API의 동작 모형 — "몇 시냐"에 구간으로 답한다 (논문 3장)
function now() {
  const t = localClock();                    // 마지막 동기화 이후 로컬로 흐른 시각
  const eps = base + DRIFT * sinceSync();    // 동기화 후 200μs/s씩 자라는 오차 상한
  return { earliest: t - eps, latest: t + eps }; // 진짜 시각은 반드시 이 안
}
const after  = (s) => now().earliest > s;   // s는 "확실히" 지났다
const before = (s) => now().latest < s;     // s는 "확실히" 오지 않았다

구간의 폭은 인프라가 정합니다. 각 데이터센터에는 GPS 수신기를 단 타임 마스터와, GPS 장애에 대비해 원자시계를 단 마스터가 함께 있고, 모든 머신의 데몬이 30초마다 여러 마스터에 시각을 물어 자기 시계를 맞춥니다(논문 3장). 동기화 직후의 ε은 마스터까지의 통신 지연이 지배하는 약 1ms이고, 다음 동기화까지는 "로컬 시계는 초당 최대 200μs 어긋난다"는 보수적 가정으로 ε을 키워 갑니다. 그래서 ε은 약 1ms에서 7ms 사이를 오가는 톱니 모양이 되고, 평균은 4ms 정도입니다(논문 3장). 오차를 없앤 것이 아니라 오차의 상한을 알고 있는 상태를 만들어 둔 셈입니다.

*

TrueTime은 알고리즘이라기보다 인프라 투자에 가깝습니다. 보장의 뿌리는 "200μs/s 이상 어긋나는 시계는 없다"는 가정인데, 논문은 이 가정이 깨지는 머신을 걸러내며 운영한 경험까지 적어 둡니다. 자기들의 장애 통계에서 나쁜 시계보다 나쁜 CPU가 6배 흔했다는 관찰(논문 5.3절)은, 잘 관리된 서버 시계가 생각보다 믿을 만하다는 근거로 인용됩니다. 증명은 수학이지만 그 수학이 딛고 선 전제는 운영에서 나온다는 점이 기억에 남았습니다.

3. commit-wait — 확신이 설 때까지 기다린다

이 구간으로 순서를 만드는 절차는 이렇습니다. 읽기-쓰기 트랜잭션이 커밋을 요청하면, 코디네이터는 잠금을 잡은 상태에서 타임스탬프 s = TT.now().latest를 배정합니다. "지금일 수도 있는 가장 늦은 시각"이니, s는 배정되는 순간 기준으로 미래이거나 기껏해야 현재입니다. 그다음 TT.after(s)가 참이 될 때까지, 그러니까 s가 어느 서버의 시계로 보더라도 확실한 과거가 될 때까지 커밋 결과를 클라이언트에게 보여주지 않습니다. 이것이 commit-wait입니다(논문 4.1.2절).

// 읽기-쓰기 트랜잭션의 커밋 뼈대 (논문 4.1.2절 · 4.2.1절)
function commit(txn) {
  acquireLocks(txn);                  // 2단계 잠금 — 겹치는 트랜잭션은 잠금이 거른다
  const s = TT.now().latest;          // ① 타임스탬프 배정: 확실히 현재 이후인 시각
  while (!TT.after(s)) wait();        // ② commit-wait: s가 확실한 과거가 될 때까지
  applyWrites(txn, s);                // ③ 이 시점부터 s는 전 세계 어느 시계로도 과거
  releaseLocks(txn);
  return s;                           // 응답은 반드시 대기 이후 — 순서 보장의 근거
}

기다리면 순서가 생기는 이유는 계산해 보면 드러납니다. 배정 시점에 latest = t + ε이었으니 s는 진짜 시각보다 최대 ε 앞서 있고, TT.after(s)는 earliest = t − ε이 s를 넘어야 참이 되므로, 대기는 대략 2ε이 됩니다. 논문 4.2.1절의 표현으로는 최소 2ε̄입니다. 이 대기가 끝난 뒤 응답을 받은 클라이언트가 무엇을 하든, 그 행동은 s보다 확실히 늦은 절대 시각에 일어납니다. 그래서 다음 트랜잭션이 어느 데이터센터의 어느 서버에서 시작하더라도 그쪽이 배정할 타임스탬프는 s보다 클 수밖에 없습니다. 서버들이 서로 통신하지 않고도 각자 자기 시계만 보고 전역 순서에 합의하게 되는 구조입니다.

글로만 읽었을 때는 잘 와닿지 않아서, ε을 바꿔 가며 커밋을 실행해 볼 수 있게 아래에 위젯을 만들어 두었습니다. 불확실성 구간이 타임라인 위를 흘러가다가, earliest가 s를 넘어서는 순간 커밋이 확정됩니다.

ε을 줄이면 대기(≈2ε)가 그대로 줄어듭니다 — 더 좋은 시계가 곧 더 빠른 커밋인 이유입니다.

슬라이더를 움직여 보면 트레이드오프가 숫자로 보입니다. ε이 7ms면 커밋마다 약 14ms를 기다려야 하고, 같은 데이터를 잇달아 고치는 트랜잭션은 잠금 위에서 이 대기가 직렬로 쌓이므로 초당 처리량에 상한이 생깁니다. ε을 1ms로 줄이면 대기도 2ms로 줄지만, 그 1ms는 GPS와 원자시계, 촘촘한 동기화 주기라는 인프라를 갖춘 대가입니다. 시계에 쓴 돈이 그대로 트랜잭션 지연으로 돌아오는 구조라, 처음 봤을 때는 낯설었습니다.

*

커밋마다 수 ms씩 기다리면 느리지 않을까 싶었는데, 논문 4.2.1절은 이 대기가 대개 Paxos 복제 통신과 겹쳐서 숨는다고 설명합니다. 커밋 확정을 위해 어차피 과반 복제본의 응답을 기다려야 하는데, 그 왕복 시간 동안 commit-wait도 함께 흘러갑니다. 논문 5.1절의 복제본 1개 실험에서 commit-wait는 약 5ms, Paxos 지연은 약 9ms로 측정됐으니, 대기가 통신 뒤에 가려질 만한 크기입니다. 없앨 수 없는 지연을 다른 필수 지연 뒤에 겹쳐 두는 방식은 실무에서 쓰는 파이프라이닝과 크게 다르지 않아 보였습니다.

4. 외부 일관성 — 4줄짜리 증명

이 설계에서 마음에 남은 건 정확성 논증이 짧다는 점이었습니다. T1이 커밋을 마친 뒤에 T2가 시작했다고 합시다. 두 트랜잭션이 서로 다른 대륙의 서버에서 처리되어도, 타임스탬프 순서 s1 < s2는 다음 네 줄로 보장됩니다(논문 4.1.2절의 증명을 풀어 쓴 것입니다).

# 외부 일관성 증명의 뼈대 (논문 4.1.2절)
s1 < t_abs(T1 응답)          # commit-wait: 응답 시점에 s1은 이미 확실한 과거
t_abs(T1 응답) < t_abs(T2 시작)   # 가정: T2는 T1이 끝난 것을 보고 시작했다
t_abs(T2 시작) ≤ s2          # 배정 규칙: s2 = TT.now().latest ≥ 진짜 현재 시각
∴ s1 < s2                    # 실제 시간의 순서가 곧 타임스탬프의 순서

첫 줄이 commit-wait의 몫이고, 셋째 줄이 latest로 배정하는 규칙의 몫입니다. 두 규칙 모두 "확실해질 때까지 보수적으로"라는 같은 원칙의 양면인 셈입니다. 이 보장이 서 있으면 따라오는 것들이 있습니다. 타임스탬프가 전역 순서를 가지므로, 과거 시점 t를 정한 스냅샷 읽기는 잠금 없이 아무 복제본에서나 처리할 수 있습니다. 복제본은 자신이 t까지의 쓰기를 다 반영했는지만 확인하면 됩니다(논문 4.1.3절의 tsafe). 읽기 트래픽이 쓰기를 방해하지 않는 MVCC를, 단일 DB가 아니라 여러 대륙에 걸친 규모에서 구현한 셈입니다. 논문 5.4절에는 샤딩된 MySQL로 버티던 Google의 광고 백엔드 F1이 Spanner로 옮겨 온 이야기가 나오는데, 이 보장이 실제 워크로드를 감당했다는 기록으로 읽었습니다.

5. 정산 시스템에서 다시 읽기

이 논문이 남의 이야기로만 읽히지 않았던 건 정산 코드를 리팩토링하던 때의 기억 때문입니다. 정산은 "어느 시점까지의 거래를 확정으로 볼 것인가"를 정하는 일입니다. 단일 데이터베이스 위에서는 이 고민이 얕아도 됩니다. 커밋 순서를 DB가 하나의 타임라인으로 줄 세워 주니, 트랜잭션 격리 수준만 바르게 고르면 "정산 배치가 시작된 시점의 스냅샷"이라는 개념이 공짜로 성립합니다. 제가 SafeIn 백엔드를 혼자 설계하면서 단일 DB의 트랜잭션 경계 안에 정합성이 중요한 로직을 몰아넣으려 애썼던 것도, 돌아보면 이 공짜 보장 위에 서 있으려는 본능이었습니다.

그 공짜가 사라지는 자리에서 문제가 생깁니다. DB 앞에 Redis 캐시를 두면 그때부터는 작은 분산 시스템입니다. 원본을 고치고 캐시를 무효화하는 사이의 틈, 무효화와 재적재가 엇갈리는 순서처럼 캐시 실무에서 겪은 골칫거리들은 "두 저장소의 상태 변화에 전역 순서가 없다"는 문제의 변주였습니다. Spanner를 읽고 나서는 이런 문제 앞에서 물어볼 것이 생겼습니다. 지금 내 시스템의 순서 보장은 어디에서 오는가(단일 DB의 커밋 로그인가, 잠금인가, 운인가), 그 보장이 닿지 않는 틈은 어디인가, 그 틈을 메우는 비용은 무엇인가. Spanner는 그 비용을 "2ε의 대기"라는 값으로 적어 두었는데, 저는 제가 만든 시스템에 그런 가격표를 붙여 본 적이 없었습니다.

6. 정리 — 불확실성을 없애지 말고 계량하라

논문을 덮으면서 정리해 둔 것은 세 가지입니다.

첫째, 불확실성은 없애기보다 재는 편이 나을 때가 있다는 것입니다. 시계 오차를 0으로 만들 수 없다는 사실 앞에서, 논문은 "그렇다면 상한을 보장하자"로 방향을 틉니다. 오차가 ε 안에 있다는 보장만 있으면 2ε의 대기로 완전한 순서를 살 수 있습니다. 재시도 횟수든 복제 지연이든, 없앨 수 없는 것의 상한을 재고 그 상한만큼 보수적으로 행동하는 패턴은 분산 시스템 전반에서 반복됩니다.

둘째, 보장의 가격을 수치로 밝혀 둔 점이 오래 남았습니다. Spanner는 외부 일관성이라는 강한 보장을 주면서 그 값이 커밋당 최소 2ε̄의 대기임을 논문에 그대로 적고, 그 값을 낮추는 방법(더 좋은 시계 인프라)까지 트레이드오프로 제시합니다. 위 위젯의 슬라이더가 보여주는 직선적인 관계가 그 정직함의 다른 표현입니다.

셋째, 어려운 문제가 층을 바꿔서 풀리는 경우를 봤습니다. 트랜잭션 순서라는 소프트웨어 문제를, Spanner는 GPS 안테나와 원자시계라는 하드웨어 층의 투자로 풀었습니다. 애플리케이션 코드에서 수백 줄로 씨름하던 문제가 인프라 한 층 아래에서는 한 줄의 보장으로 끝나는 경우를, 캐시 무효화 로직이 뒤엉키기 전에 TTL과 데이터 모델을 다시 보는 식으로 실무에서도 만납니다. 어느 층에서 풀 것인가를 정하는 일 자체가 설계라는 걸, 이 논문을 읽으며 다시 생각했습니다.
