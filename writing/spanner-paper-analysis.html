논문 분석: Spanner — 시계로 트랜잭션을 줄 세우다

2024년 8월• Spanner · TrueTime · 논문

분산 시스템 교과서의 첫 페이지에는 대개 이런 경고가 있습니다. "물리 시계를 믿지 마라." 서버마다 시계가 조금씩 다르게 가니, 이벤트의 순서는 시계가 아니라 메시지로 정하라는 것이죠. 그런데 Google의 Spanner는 정확히 그 반대로 갑니다. 전 세계 데이터센터에 흩어진 트랜잭션의 순서를 물리 시계로 보장합니다. 시계를 완벽하게 만들어서가 아니라, 시계가 얼마나 틀릴 수 있는지를 보장하고, 딱 그만큼 기다리는 방법으로요. 이 역발상이 어떻게 성립하는지, 정산 시스템에서 트랜잭션 순서를 고민해 본 백엔드 개발자의 눈으로 읽어봤습니다.

출처

Spanner: Google's Globally-Distributed Database

James C. Corbett, Jeffrey Dean 외 · OSDI 2012

1. 시계를 믿지 말라던 세계에서

왜 분산 시스템은 물리 시계를 불신할까요? NTP로 동기화한 서버들의 시계도 수 밀리초에서 수십 밀리초씩 어긋납니다. 서버 A의 시계로 10:00:00.005에 커밋된 트랜잭션과 서버 B의 시계로 10:00:00.003에 커밋된 트랜잭션이 있을 때, 어느 쪽이 먼저인지 시계만 보고는 알 수 없습니다. B의 시계가 4ms 느리게 가고 있었다면, B가 찍은 .003은 실제로는 .007이어서 A가 먼저였을 테니까요. 그래서 Lamport의 1978년 논문 이래로 분산 시스템은 논리 시계를 씁니다. "메시지를 주고받은 순서"라는 인과 관계로만 순서를 정하고, 벽시계는 참고용으로 밀어두는 접근입니다.

하지만 논리 시계에는 한계가 있습니다. 서로 통신한 적 없는 두 트랜잭션 사이의 순서는 정해 주지 못합니다. 사용자가 A 시스템에서 결제를 마치고, 전화로 전달받은 동료가 B 시스템에서 그 결과를 조회하는 상황 — 두 시스템은 메시지를 주고받은 적이 없지만, 사람의 눈에는 명백한 선후 관계가 있습니다. 이 "시스템 바깥의 순서"까지 지키는 것이 외부 일관성이고, Spanner는 세계 규모로 분산된 데이터베이스가 외부 일관성을 제공한 첫 사례라고 논문 1장에서 밝힙니다.

Spanner의 통찰은 문제를 뒤집은 데 있습니다. 시계의 오차를 없애는 대신 측정 가능하게 묶어두고, 오차가 문제되는 딱 그 구간만큼만 기다리자는 것입니다. 오차를 0으로 만드는 것은 물리적으로 불가능하지만, "오차가 최대 ε을 넘지 않는다"는 보장은 공학으로 만들 수 있습니다. 그 공학이 TrueTime입니다.

2. TrueTime — 시각이 아니라 구간을 답하는 API

TrueTime의 API는 놀랄 만큼 작습니다(논문 3장, Table 1). 핵심은 TT.now()가 시각 하나가 아니라 구간 [earliest, latest]를 돌려준다는 점입니다. 호출한 순간의 진짜 절대 시각은 반드시 이 구간 안에 있습니다. 어디인지는 모르지만, 밖은 아니라는 보장 — 이 정직한 무지(無知)가 전체 설계의 주춧돌입니다.

// TrueTime API의 동작 모형 — "몇 시냐"에 구간으로 답한다 (논문 3장)
function now() {
  const t = localClock();                    // 마지막 동기화 이후 로컬로 흐른 시각
  const eps = base + DRIFT * sinceSync();    // 동기화 후 200μs/s씩 자라는 오차 상한
  return { earliest: t - eps, latest: t + eps }; // 진짜 시각은 반드시 이 안
}
const after  = (s) => now().earliest > s;   // s는 "확실히" 지났다
const before = (s) => now().latest < s;     // s는 "확실히" 오지 않았다

구간의 폭은 인프라가 정합니다. 각 데이터센터에는 GPS 수신기를 단 타임 마스터와, GPS 장애에 대비해 원자시계를 단 마스터가 함께 있고, 모든 머신의 데몬이 30초마다 여러 마스터에 시각을 물어 자기 시계를 맞춥니다(논문 3장). 동기화 직후의 ε은 마스터까지의 통신 지연이 지배하는 약 1ms이고, 다음 동기화까지는 "로컬 시계는 초당 최대 200μs 어긋난다"는 보수적 가정으로 ε을 키워 갑니다. 그래서 ε은 약 1ms에서 7ms 사이를 오가는 톱니 모양이 되고, 평균은 4ms 정도입니다(논문 3장). 오차를 없앤 것이 아니라, 오차의 상한이 그리는 그래프를 알고 있는 상태를 만든 것입니다.

*

TrueTime은 알고리즘이 아니라 인프라 투자입니다. 보장의 뿌리는 "200μs/s 이상 어긋나는 시계는 없다"는 가정인데, 논문은 이 가정이 깨지는 머신을 걸러내며 운영한 경험까지 적어 둡니다. 자기들의 장애 통계에서 나쁜 시계보다 나쁜 CPU가 6배 흔했다는 관찰(논문 5.3절)은, 잘 관리된 서버 시계가 생각보다 믿을 만하다는 근거로 인용됩니다. 보장은 수학에서 오지만, 그 수학의 전제는 운영에서 온다는 점이 이 논문의 숨은 교훈입니다.

3. commit-wait — 확신이 설 때까지 기다린다

이제 이 구간으로 순서를 만드는 방법입니다. 읽기-쓰기 트랜잭션이 커밋을 요청하면, 코디네이터는 잠금을 잡은 상태에서 타임스탬프 s = TT.now().latest를 배정합니다. "지금일 수도 있는 가장 늦은 시각"이니, s는 배정되는 순간 기준으로 미래이거나 기껏해야 현재입니다. 그리고 핵심 규칙이 나옵니다. TT.after(s)가 참이 될 때까지 — s가 어느 서버의 시계로 보더라도 확실한 과거가 될 때까지 — 커밋 결과를 클라이언트에게 보여주지 않습니다. 이것이 commit-wait입니다(논문 4.1.2절).

// 읽기-쓰기 트랜잭션의 커밋 뼈대 (논문 4.1.2절 · 4.2.1절)
function commit(txn) {
  acquireLocks(txn);                  // 2단계 잠금 — 겹치는 트랜잭션은 잠금이 거른다
  const s = TT.now().latest;          // ① 타임스탬프 배정: 확실히 현재 이후인 시각
  while (!TT.after(s)) wait();        // ② commit-wait: s가 확실한 과거가 될 때까지
  applyWrites(txn, s);                // ③ 이 시점부터 s는 전 세계 어느 시계로도 과거
  releaseLocks(txn);
  return s;                           // 응답은 반드시 대기 이후 — 순서 보장의 근거
}

왜 기다리면 순서가 생길까요? 배정 시점에 latest = t + ε이었으니 s는 진짜 시각보다 최대 ε 앞서 있고, TT.after(s)는 earliest = t − ε이 s를 넘어야 참이 되므로, 대기는 대략 2ε — 논문 4.2.1절의 표현으로는 최소 2ε̄입니다. 이 대기가 끝난 뒤 응답을 받은 클라이언트가 무엇을 하든, 그 행동은 s보다 확실히 늦은 절대 시각에 일어납니다. 그래서 다음 트랜잭션이 어느 데이터센터의 어느 서버에서 시작하더라도, 그쪽이 배정할 타임스탬프는 s보다 클 수밖에 없습니다. 서버들이 서로 통신하지 않고도, 각자 자기 시계만 보고 전역 순서에 합의하게 되는 것입니다.

말로 들으면 미묘한데, 손으로 움직여 보면 명확해집니다. 아래에서 ε을 바꿔 가며 커밋을 실행해 보세요. 불확실성 구간이 타임라인 위를 흘러가다가, earliest가 s를 넘어서는 순간 커밋이 확정됩니다.

ε을 줄이면 대기(≈2ε)가 그대로 줄어듭니다 — 더 좋은 시계가 곧 더 빠른 커밋인 이유입니다.

슬라이더를 움직여 보면 트레이드오프가 숫자로 보입니다. ε이 7ms면 커밋마다 약 14ms를 기다려야 하고, 같은 데이터를 잇달아 고치는 트랜잭션은 잠금 위에서 이 대기가 직렬로 쌓이므로 초당 처리량에 상한이 생깁니다. ε을 1ms로 줄이면 대기는 2ms로 줄지만, 그 1ms는 공짜가 아닙니다 — GPS와 원자시계, 촘촘한 동기화 주기라는 인프라의 값입니다. Spanner가 "시계에 돈을 쓰면 트랜잭션이 빨라진다"는 특이한 등식을 성립시킨 것이죠.

*

"커밋마다 수 ms씩 기다리면 너무 느리지 않나?"라는 의문이 들 텐데, 논문 4.2.1절은 이 대기가 대개 Paxos 복제 통신과 겹쳐서 숨는다고 답합니다. 커밋 확정을 위해 어차피 과반 복제본의 응답을 기다려야 하는데, 그 왕복 시간 동안 commit-wait도 함께 흘러가는 것입니다. 실제로 논문 5.1절의 복제본 1개 실험에서 commit-wait는 약 5ms, Paxos 지연은 약 9ms로 측정됐습니다 — 대기가 통신 뒤에 가려질 만한 크기입니다. 지연을 없앨 수 없다면 다른 필수 지연 뒤에 숨기는 것 — 실무의 파이프라이닝과 같은 감각입니다.

4. 외부 일관성 — 4줄짜리 증명

이 설계의 아름다움은 정확성 논증이 짧다는 데 있습니다. T1이 커밋을 마친 뒤에 T2가 시작했다고 합시다. 두 트랜잭션이 서로 다른 대륙의 서버에서 처리되어도, 타임스탬프 순서 s1 < s2는 다음 네 줄로 보장됩니다(논문 4.1.2절의 증명을 풀어 쓴 것입니다).

# 외부 일관성 증명의 뼈대 (논문 4.1.2절)
s1 < t_abs(T1 응답)          # commit-wait: 응답 시점에 s1은 이미 확실한 과거
t_abs(T1 응답) < t_abs(T2 시작)   # 가정: T2는 T1이 끝난 것을 보고 시작했다
t_abs(T2 시작) ≤ s2          # 배정 규칙: s2 = TT.now().latest ≥ 진짜 현재 시각
∴ s1 < s2                    # 실제 시간의 순서가 곧 타임스탬프의 순서

첫 줄이 commit-wait의 몫이고, 셋째 줄이 latest로 배정하는 규칙의 몫입니다. 두 규칙 모두 "확실해질 때까지 보수적으로"라는 같은 원칙의 양면인 셈입니다. 그리고 이 보장이 서 있으면 좋은 것들이 줄줄이 따라옵니다. 타임스탬프가 전역 순서를 가지므로, 과거 시점 t를 정한 스냅샷 읽기는 잠금 없이 아무 복제본에서나 처리할 수 있습니다. 복제본은 자신이 t까지의 쓰기를 다 반영했는지만 확인하면 됩니다(논문 4.1.3절의 tsafe). 읽기 트래픽이 쓰기를 방해하지 않는 MVCC를, 단일 DB가 아니라 행성 규모에서 해낸 것입니다. 논문 5.4절의 F1 사례 — 샤딩된 MySQL로 버티던 Google의 광고 백엔드가 Spanner로 옮겨 온 이야기 — 는 이 보장이 실제 워크로드를 감당했다는 증거로 읽힙니다.

5. 정산 시스템에서 다시 읽기

제가 이 논문을 남의 이야기로만 읽을 수 없었던 건 정산 코드를 리팩토링했던 경험 때문입니다. 정산은 본질적으로 "어느 시점까지의 거래를 확정으로 볼 것인가"를 정하는 일입니다. 단일 데이터베이스 위에서는 이 고민이 얕아도 됩니다. 커밋 순서를 DB가 하나의 타임라인으로 줄 세워 주니, 트랜잭션 격리 수준만 바르게 고르면 "정산 배치가 시작된 시점의 스냅샷"이라는 개념이 공짜로 성립합니다. 제가 SafeIn 백엔드를 혼자 설계하면서 단일 DB의 트랜잭션 경계 안에 정합성이 중요한 로직을 몰아넣으려 애썼던 것도, 돌아보면 이 공짜 보장 위에 서 있으려는 본능이었습니다.

그 공짜가 사라지는 순간이 문제입니다. 예컨대 DB 앞에 Redis 캐시를 두는 순간, 이미 작은 분산 시스템이 시작됩니다. 원본을 고치고 캐시를 무효화하는 사이의 틈, 무효화와 재적재가 엇갈리는 순서 — 캐시 실무에서 겪는 골칫거리들은 결국 "두 저장소의 상태 변화에 전역 순서가 없다"는 문제의 변주입니다. Spanner를 읽고 나면 이런 문제를 보는 언어가 생깁니다. 지금 내 시스템의 순서 보장은 어디에서 오는가(단일 DB의 커밋 로그인가, 잠금인가, 운인가), 그 보장이 닿지 않는 틈은 어디인가, 그 틈을 메우는 비용은 무엇인가. Spanner의 답이 "2ε의 대기"라는 명시적인 가격표였다는 점이 중요합니다. 보장에는 반드시 비용이 있고, 좋은 설계는 그 비용을 숨기는 게 아니라 측정 가능하게 드러내는 것이니까요.

6. 정리 — 불확실성을 없애지 말고 계량하라

Spanner 논문에서 가져갈 것을 세 가지로 정리합니다.

첫째, 불확실성은 제거의 대상이 아니라 계량의 대상입니다. 시계 오차를 0으로 만들 수 없다는 사실 앞에서, 논문은 "그렇다면 상한을 보장하자"로 방향을 틉니다. 오차가 ε 안에 있다는 보장만 있으면, 2ε의 대기로 완전한 순서를 살 수 있습니다. 재시도 횟수든 복제 지연이든, 없앨 수 없는 것의 상한을 재고 그 상한만큼 보수적으로 행동하는 이 패턴은 분산 시스템 전반에서 반복됩니다.

둘째, 보장의 가격을 명시하는 설계가 좋은 설계입니다. Spanner는 외부 일관성이라는 최상급 보장을 주면서 그 값이 커밋당 최소 2ε̄의 대기임을 논문에 수치로 못 박고, 그 값을 낮추는 방법(더 좋은 시계 인프라)까지 트레이드오프로 제시합니다. 위 위젯의 슬라이더가 보여주는 그 직선적인 관계가, 이 시스템이 정직하게 설계됐다는 증거입니다.

셋째, 어려운 문제는 종종 층을 바꿔서 풀립니다. 트랜잭션 순서라는 소프트웨어 문제를, Spanner는 GPS 안테나와 원자시계라는 하드웨어 층의 투자로 풀었습니다. 애플리케이션 코드에서 수백 줄로 씨름하던 문제가 인프라 한 층 아래에서는 한 줄의 보장으로 끝나는 경우 — 캐시 무효화 로직으로 뒤엉키기 전에 TTL과 데이터 모델을 다시 보는 일 같은 — 를 실무에서도 자주 만납니다. 어느 층에서 풀 것인가를 정하는 것 자체가 설계라는 것을, 이 논문만큼 크게 보여주는 사례는 드뭅니다.
