Adapter로 크롤러를 무한 확장하고, Backoff로 안 잘리기

2025년 8월• 약 7분• Adapter · Backoff · 크롤링

크롤러가 지저분해지는 이유는 하나입니다. 사이트마다 다른데, 그 다름이 코드 전체로 번지기 때문입니다. 새 사이트를 추가할 때 기존 코드를 건드리지 않으려면 변화의 축을 한 군데 가둬야 합니다.

먼저 짚고 가기
Adapter 패턴Adapter Pattern
서로 다른 대상을 똑같은 모양의 창구 하나로 감싸는 설계 방식입니다. 나라마다 콘센트가 달라도 여행용 어댑터 하나면 꽂히듯, 사이트마다 다른 HTML을 각 어댑터가 흡수해 파이프라인은 창구 규격만 알면 됩니다. 이 글에서 "새 사이트 = 새 어댑터 하나"가 가능한 이유입니다.
Rate limit / 429Too Many Requests
서버가 "짧은 시간에 너무 많이 요청했다"며 잠시 거절하는 신호입니다. HTTP 응답 코드 429가 그 대표적인 표시로, 계속 무시하고 두드리면 아예 차단으로 이어집니다. 크롤러가 상대 서버와 오래 공존하려면 반드시 다뤄야 하는 반응입니다.
지수 백오프Exponential Backoff
실패할 때마다 다음 재시도까지 기다리는 시간을 2배씩 늘리는 방법입니다(1초, 2초, 4초…). 거절당했는데 곧바로 또 조르면 상황만 나빠지니, 물러서는 간격을 점점 벌려 서버가 숨 쉴 틈을 줍니다.
지터Jitter
대기 시간에 일부러 섞는 약간의 무작위 값입니다. 여러 워커가 똑같이 "4초 뒤 재시도"하면 그 순간 다 같이 몰려 서버를 또 때리는데, 각자 4.3초·4.7초처럼 흩어 놓으면 충돌을 피할 수 있습니다.
천둥 떼 문제Thundering Herd
대기하던 요청들이 같은 순간에 한꺼번에 깨어나 서버로 쏟아지는 현상입니다. 지터가 없을 때 벌어지며, 이 글에서 지터 한 줄을 넣는 이유가 바로 이 떼거리를 시간축에 흩기 위해서입니다.

Adapter — 다름을 인터페이스 뒤로

class CrawlerAdapter(Protocol):
    def fetch(self, url: str) -> str: ...
    def parse(self, html: str) -> list[Article]: ...

# 새 사이트 = 새 Adapter 하나. 파이프라인은 그대로.
class MusinsaAdapter(CrawlerAdapter): ...
class W29Adapter(CrawlerAdapter): ...

파이프라인(수집→정제→변환→저장)은 CrawlerAdapter만 압니다. 사이트가 10개든 100개든, 공통 로직은 한 줄도 바뀌지 않습니다. 개방-폐쇄 원칙이 추상이 아니라 배포 속도로 돌아옵니다.

Backoff — 서버를 화나게 하지 않기

공격적으로 긁으면 429(Too Many Requests)나 차단이 돌아옵니다. 실패하면 무작정 재시도하는 대신, 지수적으로 간격을 늘리고 지터를 섞습니다.

async def fetch_with_backoff(url, tries=5):
    for i in range(tries):
        r = await client.get(url)
        if r.status_code != 429: return r
        wait = min(60, 2 ** i) + random.uniform(0, 1)  # 지터
        await asyncio.sleep(wait)
    raise RateLimited(url)
*

지터가 없으면 여러 워커가 동시에 재시도해 서버를 다시 두드립니다(thundering herd). 2**i에 랜덤을 섞는 한 줄이, 재시도를 시간축에 흩어 서로를 밟지 않게 합니다.

그리고 관측 가능성

장애는 반드시 납니다. 중요한 건 빨리 아는 것. 크롤러가 연속 실패하면 Slack으로 알림이 오게 했습니다. 매일 정해진 시간 스케줄러가 돌고, 실패율이 임계치를 넘으면 사람을 부릅니다. 무인 자동화의 조건은 "조용할 때만 무인"입니다.
