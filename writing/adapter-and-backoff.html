Adapter로 크롤러를 무한 확장하고, Backoff로 안 잘리기

2025년 8월• Adapter · Backoff · 크롤링

크롤링할 사이트가 하나둘 늘면서 코드가 점점 지저분해졌습니다. 사이트마다 HTML이 다른 거야 어쩔 수 없는데, 그 다름이 파이프라인 전체로 번지는 게 문제였습니다. 새 사이트를 하나 붙일 때마다 멀쩡한 기존 코드를 건드리고 있길래, 변하는 부분을 한 군데에 가둬 두는 쪽으로 구조를 바꿨습니다. 그때 남은 두 축이 Adapter와 Backoff입니다.

먼저 짚고 가기
Adapter 패턴Adapter Pattern
호환되지 않는 인터페이스를 호출하는 쪽이 기대하는 형태로 변환하는 GoF 구조 패턴입니다. 이 글에서는 CrawlerAdapter라는 Protocol이 fetch(url) -> str과 parse(html) -> list[Article] 두 시그니처를 규격으로 못 박고, 사이트별 구현체가 그 규격만 만족시키면 파이프라인은 구현체를 구분하지 않습니다. Adapter는 인터페이스의 모양만 바꿀 뿐 기능을 더하지 않는다는 점에서, 동작을 덧씌우는 Decorator나 여러 하위 시스템을 단일 창구로 가리는 Facade와 구분됩니다.
Rate limit / 429Too Many Requests
일정 시간 창 안에 허용된 요청 수를 클라이언트가 넘겼을 때 서버가 돌려주는 HTTP 상태 코드입니다(429 Too Many Requests, RFC 6585 §4). 응답에 Retry-After 헤더를 실어 언제 다시 오라고 알려줄 수 있고, 이를 무시하고 계속 두드리면 IP나 계정 단위 차단으로 넘어가기도 합니다. 다만 제한을 세는 방식(고정 윈도우·슬라이딩 윈도우·토큰 버킷)은 스펙이 정하지 않으며, 모든 서버가 제한을 429로 알려주지도 않아 403이나 빈 응답으로 돌아오는 경우도 있습니다.
지수 백오프Exponential Backoff
n번째 재시도의 대기 시간을 base × 2ⁿ처럼 지수적으로 늘리는 재시도 스케줄링 알고리즘입니다. 이더넷 CSMA/CD의 이진 지수 백오프에서 온 방식으로, 실패가 이어질수록 클라이언트가 스스로 요청 빈도를 낮춰 서버 쪽 부하가 걷힐 시간을 줍니다. 대기 시간이 무한정 커지지 않게 상한을 씌운 형태를 capped backoff라 부르고, 이 글 코드의 min(60, 2 ** i)가 그것입니다. 재시도 횟수 상한(tries=5)을 함께 두지 않으면 복구 불가능한 실패를 늦게 알아차리게 되는 것이 이 방식의 대가입니다.
지터Jitter
백오프로 계산한 대기 시간에 난수를 섞어 재시도 시각을 확률적으로 흩뜨리는 기법입니다. 이 글의 코드는 2 ** i에 random.uniform(0, 1)을 더하는 가산 지터라, 워커들의 재시도 시점이 1초 폭 안에서 벌어집니다. 흩어지는 폭이 백오프 값에 비해 좁으면 분산 효과도 그만큼 작아서, 동시 클라이언트가 많을 때는 대기 시간을 random.uniform(0, cap)처럼 구간 전체에서 뽑는 full jitter가 더 고르게 퍼집니다.
천둥 떼 문제Thundering Herd
대기하던 다수의 주체가 같은 순간에 한꺼번에 깨어나 하나의 자원으로 몰리는 현상입니다. 재시도 맥락에서는 같은 시각에 429를 받은 워커들이 똑같은 백오프 값을 계산하는 바람에 재시도 시점까지 서로 맞춰지면서 벌어지고, 그 동기화를 깨뜨리는 장치가 지터입니다. 원래는 커넥션 하나가 도착했을 때 대기 중인 프로세스를 전부 깨워 버리는 운영체제 쪽 문제를 가리키던 말이라, 그쪽 해법(배타적 웨이크업)과 여기서 쓰는 해법(지터)은 서로 다릅니다.

Adapter — 다름을 인터페이스 뒤로

class CrawlerAdapter(Protocol):
    def fetch(self, url: str) -> str: ...
    def parse(self, html: str) -> list[Article]: ...

# 새 사이트 = 새 Adapter 하나. 파이프라인은 그대로.
class MusinsaAdapter(CrawlerAdapter): ...
class W29Adapter(CrawlerAdapter): ...

파이프라인(수집→정제→변환→저장)은 CrawlerAdapter라는 창구 하나만 알고 있습니다. 그래서 사이트가 10개로 늘든 100개로 늘든 공통 로직은 그대로 두고 어댑터만 하나씩 더 쓰면 됐습니다. 개방-폐쇄 원칙이라는 말은 그전까지 저에게 좀 추상적이었는데, 새 사이트 하나를 붙이는 데 드는 시간이 줄어드는 걸 보고 나서야 무슨 뜻인지 알 것 같았습니다.

Backoff — 서버를 화나게 하지 않기

요청을 촘촘히 보내면 429(Too Many Requests)가 돌아오고, 그걸 무시하고 계속 두드리면 차단으로 이어집니다. 그래서 실패했을 때 곧바로 다시 요청하는 대신, 재시도 간격을 지수적으로 늘리고 거기에 지터를 섞도록 고쳤습니다.

async def fetch_with_backoff(url, tries=5):
    for i in range(tries):
        r = await client.get(url)
        if r.status_code != 429: return r
        wait = min(60, 2 ** i) + random.uniform(0, 1)  # 지터
        await asyncio.sleep(wait)
    raise RateLimited(url)
*

지터가 없으면 여러 워커가 같은 시점에 재시도해 서버를 다시 두드리게 됩니다(thundering herd). 2**i에 랜덤 값을 더해 두면 재시도 시점이 조금씩 어긋나서, 워커끼리 겹치는 일이 줄어듭니다.

그리고 관측 가능성

구조를 정리해 두어도 장애는 납니다. 그래서 막는 쪽보다 언제 났는지 빨리 아는 쪽에 손을 더 썼습니다. 크롤러가 연속으로 실패하면 Slack 알림이 오게 해 두었고, 스케줄러는 매일 정해진 시간에 돌면서 실패율이 임계치를 넘을 때 사람을 부릅니다. 무인으로 돌려 놨다고 생각했지만, 실제로는 조용할 때만 무인이었습니다.
