아키텍처 패턴 · Pipe & Filter — 무수정 확장의 구조

2023년 6월• 약 12분• 아키텍처 · Pipe&Filter · Adapter

크롤링 파이프라인 사례에서 저는 "새 사이트를 추가할 때 기존 코드를 건드리지 않을 것"이 핵심 요구였다고 썼습니다. 이번 글은 그 문장의 기술적 증명입니다. 어떤 인터페이스 계약이 그 약속을 성립시키는지, 필터의 정의부터 합성, 어댑터 등록, 그리고 배압과 에러 전파 같은 현실 문제까지 Python 코드로 처음부터 다시 짓습니다.

먼저 짚고 가기
파이프 & 필터Pipe & Filter
데이터를 여러 개의 작은 처리 단계(필터)에 차례로 통과시키는 구조입니다. 정수기가 물을 여러 필터로 거르듯, 데이터가 정제·중복 제거·분류 단계를 한 줄로 흘러가며 다듬어집니다. 이 글은 바로 이 구조로 크롤러를 다시 짓는 이야기입니다.
제너레이터 · 지연 평가Generator · Lazy Evaluation
결과를 한꺼번에 다 만들어 쌓아두지 않고, 뒤 단계가 "한 건 줘"라고 요청할 때 그때그때 한 건씩만 만들어 넘기는 방식입니다. Python의 yield가 이 역할을 합니다. 기사 수만 건을 메모리에 쌓지 않아도 되는 이유가 여기에 있습니다.
어댑터 · 레지스트리Adapter · Registry
어댑터는 사이트마다 제각각인 수집 방식을 똑같은 모양의 인터페이스로 맞춰주는 변환 부품이고, 레지스트리는 그 부품들을 이름표를 붙여 모아두는 목록입니다. 새 소스를 "목록에 부품 하나 더 꽂기"로 끝내는 것이 무수정 확장의 열쇠입니다.
배압Backpressure
뒷 단계가 느려지면 그 느림이 앞 단계까지 거꾸로 전해져 앞 단계도 알아서 속도를 늦추는 성질입니다. 수도관 끝을 막으면 물이 위로 밀리는 것과 같습니다. 이게 없으면 처리 못 한 데이터가 무한정 쌓여 프로그램이 메모리로 죽습니다.
개방-폐쇄 원칙Open-Closed Principle
기존 코드는 고치지 않은 채(폐쇄), 새 코드를 더하는 것만으로 기능을 늘릴 수 있어야(개방) 한다는 설계 원칙입니다. 이 글이 증명하려는 "코드를 안 건드리고 새 소스 추가"가 바로 이 원칙의 실현입니다.

1. 출발점 — 잘 돌던 크롤러가 자라는 방식

크롤러는 처음엔 다들 멀쩡하게 시작합니다. 소스 하나, 함수 하나. 문제는 두 번째 소스가 들어올 때부터입니다. "일단 elif 하나만" 붙이는 순간, 코드는 이렇게 자랍니다.

# before — 소스가 늘 때마다 이 함수도 함께 자란다
def crawl_all():
    articles, seen = [], set()
    for source in ["musinsa", "w29"]:
        if source == "musinsa":
            html = fetch_musinsa()
            for item in parse_musinsa(html):
                item["title"] = item["title"].strip()
                if item["url"] in seen: continue
                seen.add(item["url"])
                articles.append(to_article(item))
        elif source == "w29":
            html = fetch_w29()
            ...  # 거의 같은 코드가 한 벌 더, 미묘하게 다르게
    save_all(articles)

이 함수의 진짜 문제는 길이가 아닙니다. 서로 다른 세 개의 변화 축이 한 몸에 얽혀 있다는 것입니다. ① 소스가 늘어나는 축(사이트마다 다른 fetch·parse), ② 처리 규칙이 바뀌는 축(정제·중복 제거·변환), ③ 저장 방식이 바뀌는 축. 축이 얽혀 있으면 어느 축의 변화든 기존 코드 수정이 되고, 수정은 곧 회귀 위험입니다. 소스 10개짜리 크롤러에서 11번째 소스를 붙이다가 3번째 소스를 깨뜨리는 일이 실제로 일어납니다.

이전 글에서 정산 시스템의 if-else 증식을 Template Method와 Strategy로 눌렀던 것과 같은 병입니다. 다른 점은 처방입니다. 정산은 "흐름은 같고 단계 구현만 다른" 문제였지만, 크롤러는 데이터가 단계를 통과하며 변형되는 문제입니다. 이런 형태에는 Pipe & Filter가 정확히 맞습니다.

2. 필터 프로토콜 — 계약은 작을수록 강하다

Pipe & Filter의 전부는 사실 계약 하나입니다. 모든 필터는 스트림을 받아 스트림을 돌려준다. Python으로는 Protocol 하나와 제너레이터 함수들로 충분합니다.

from typing import Iterable, Protocol

class Filter(Protocol):
    def __call__(self, stream: Iterable[Article]) -> Iterable[Article]: ...

# 필터는 제너레이터 함수 하나면 충분합니다
def clean(stream: Iterable[Article]) -> Iterable[Article]:
    for a in stream:
        yield a.with_title(normalize(a.title))

def dedup(stream: Iterable[Article]) -> Iterable[Article]:
    seen: set[str] = set()
    for a in stream:
        if a.url not in seen:
            seen.add(a.url)
            yield a

왜 시그니처를 Iterable → Iterable 하나로 통일해야 할까요. 근거는 두 가지입니다.

첫째, 합성이 닫혀 있으려면 출력 타입이 곧 다음 입력 타입이어야 합니다. 필터 A의 출력이 필터 B의 입력과 같은 타입이면, A와 B를 이은 것 역시 같은 타입의 필터입니다. 이 성질 덕분에 필터를 아무 순서로 갈아 끼우고, 중간에 새 필터를 삽입하고, 빼는 일이 전부 국소 변경이 됩니다. 만약 clean은 dict를 받고 dedup은 Article을 받는 식으로 타입이 어긋나면, 필터 사이마다 변환 코드가 끼어들고 그 변환 코드가 다시 결합점이 됩니다.

둘째, 제너레이터라서 지연 평가가 공짜입니다. yield 기반 필터는 다음 단계가 당겨갈 때(pull)만 한 건을 처리합니다. 기사 1만 건을 리스트로 쌓아 넘기는 게 아니라, 한 건이 파이프 전체를 관통한 뒤 다음 건이 출발합니다. 메모리 사용량이 스트림 길이와 무관해지고 — 뒤에서 다루겠지만 — 배압(backpressure)의 절반이 여기서 저절로 해결됩니다.

*

필터는 순수하게, IO는 양 끝으로. 필터 안에서 DB를 조회하거나 API를 부르기 시작하면 테스트가 즉시 무거워집니다. 변형 로직만 필터에 두면 테스트는 list(clean([article])) — 리스트를 넣고 리스트를 받는 것으로 끝납니다. 저는 이 규칙 덕에 크롤러의 정제·변환 로직을 네트워크 없이 전부 단위 테스트로 커버할 수 있었습니다.

3. 파이프 — 합성은 반복문 하나로 끝난다

계약이 통일돼 있으면 파이프, 즉 합성기는 놀랄 만큼 짧습니다.

class Pipeline:
    def __init__(self, *filters: Filter):
        self.filters = filters

    def __call__(self, stream: Iterable[Article]) -> Iterable[Article]:
        for f in self.filters:
            stream = f(stream)        # f3(f2(f1(stream)))과 같다
        return stream

pipeline = Pipeline(clean, dedup, categorize)
for article in pipeline(fetched):
    repo.save(article)                # IO는 파이프 바깥, 맨 끝에서

반복문의 stream = f(stream)은 수학의 함수 합성 그대로입니다. 그리고 여기서 계약 통일의 보상이 하나 더 나옵니다. Pipeline 자체가 Iterable → Iterable이므로, 파이프라인도 하나의 필터입니다. 정제 파이프라인을 만들어 더 큰 파이프라인의 한 단계로 끼워 넣을 수 있습니다. Composite 패턴이 별도 코드 없이 계약에서 저절로 따라 나오는 셈입니다.

주의할 점은 실행 시점입니다. pipeline(fetched)를 호출하는 순간에는 아무 일도 일어나지 않습니다. 제너레이터가 사슬로 엮일 뿐이고, 실제 처리는 맨 끝의 소비자 — 위 코드에선 for 루프 — 가 당길 때 시작됩니다. "파이프를 조립하는 코드"와 "물을 흘리는 코드"가 분리된다는 뜻이고, 조립 결과는 실행 없이도 검증할 수 있습니다.

4. 어댑터 등록 — 신규 소스의 diff를 0줄로

이제 남은 축은 하나, 소스별 차이입니다. 사이트마다 fetch 방식과 HTML 구조가 다르므로, 그 다름을 SourceAdapter 인터페이스 뒤에 가두고 레지스트리에 선언적으로 등록합니다.

class SourceAdapter(Protocol):
    name: str
    def fetch(self) -> Iterable[RawPage]: ...
    def parse(self, page: RawPage) -> Iterable[Article]: ...

_REGISTRY: dict[str, SourceAdapter] = {}

def register(cls):
    _REGISTRY[cls.name] = cls()
    return cls

def run_all(pipeline: Filter):
    for adapter in _REGISTRY.values():
        stream = (a for page in adapter.fetch()
                    for a in adapter.parse(page))
        for article in pipeline(stream):
            repo.save(article)

run_all을 다시 읽어보면, 이 코드 어디에도 소스의 이름이 없습니다. "musinsa"도 "w29"도 모릅니다. 파이프라인은 오직 SourceAdapter라는 추상만 알고, 구체 어댑터는 레지스트리를 통해서만 만납니다. 이 무지(無知)가 무수정 확장의 핵심 조건입니다. 실제로 신규 소스를 추가하는 작업은 파일 하나를 새로 만드는 것으로 끝납니다.

# 신규 소스 추가 = 새 파일 하나. 기존 코드 diff는 0줄.
# sources/trendpost.py
@register
class TrendPostAdapter:
    name = "trendpost"

    def fetch(self):
        yield from paged_get(BASE_URL, pages=5)

    def parse(self, page):
        for node in page.select("article.card"):
            yield Article(
                title=node.text(".title"),
                url=node.attr("a", "href"),
                source=self.name,
            )

정리하면, 무수정 확장은 세 조건이 동시에 성립할 때만 얻어집니다. ① 변화의 축이 어댑터 경계와 일치할 것 — "소스가 늘어난다"는 변화가 정확히 SourceAdapter 하나의 구현으로 흡수돼야 합니다. ② 등록이 선언적일 것 — 새 어댑터를 어딘가의 if문에 끼워 넣어야 한다면 그 if문이 곧 수정 지점입니다. ③ 파이프라인이 구체 소스를 모를 것. 셋 중 하나라도 무너지면 "새 소스 추가"는 다시 "기존 코드 수정"으로 되돌아갑니다. 개방-폐쇄 원칙은 공짜 성질이 아니라, 이 세 조건을 설계로 지불한 대가입니다.

5. 현실 문제 — 배압과 에러 전파

여기까지는 교과서가 그려주는 그림입니다. 운영에 들어가면 두 가지가 바로 찾아옵니다.

배압. 동기 제너레이터 사슬에서는 배압이 공짜였습니다. 소비자(저장)가 당길 때만 생산자(수집)가 움직이는 pull 구조라, 저장이 느려지면 수집도 저절로 느려집니다. 그런데 수집을 빠르게 하려고 소스별 fetch를 병렬화하는 순간, 생산자와 소비자 사이에 큐가 생기고 이 성질이 깨집니다. 큐가 무한이면 DB가 잠깐 느려지는 동안 큐에 기사가 무한정 쌓이고, 프로세스는 메모리로 죽습니다. 답은 한 단어, 유한 큐입니다.

# 병렬화로 파이프가 큐로 바뀌는 순간, 배압은 공짜가 아니게 된다
queue: asyncio.Queue[Article] = asyncio.Queue(maxsize=200)

async def producer(adapter):
    for page in adapter.fetch():
        for a in adapter.parse(page):
            await queue.put(a)  # 가득 차면 여기서 멈춘다 — 이것이 배압

maxsize에 도달한 put()이 블록되는 것은 버그가 아니라 설계입니다. 소비자의 속도가 큐를 타고 생산자까지 역방향으로 전파되는 것, 그게 배압의 정의니까요. 크롤러에서는 덤으로 요청 속도가 저장 속도에 묶이므로, 대상 서버를 필요 이상으로 두드리지 않는 효과도 있습니다.

에러 전파. 순진한 제너레이터 사슬에는 치명적인 성질이 하나 있습니다. 필터 안에서 예외가 터지면 사슬 전체가 죽습니다. 기사 1만 건 중 1건의 HTML이 깨졌다고 그날 수집 전체가 사라지는 건 받아들일 수 없습니다. 그렇다고 예외를 그냥 삼키면 더 나쁜 일이 생깁니다. 저는 실패를 한 건 단위로 격리하되, 반드시 기록하는 데코레이터로 해결했습니다.

def guarded(f: Filter) -> Filter:
    # 상태 없는 필터 전용 — 매 건마다 f를 새로 실행해 격리한다
    def wrapper(stream):
        for a in stream:
            try:
                yield from f(iter([a]))          # 한 건씩 격리 실행
            except Exception as e:
                dead_letters.put(a, error=e)        # 삼키지 말고 적재
                metrics.incr("filter_error", tag=f.__name__)
    return wrapper

pipeline = Pipeline(guarded(clean), dedup, guarded(categorize))

조립 줄에서 dedup만 맨몸인 것은 실수가 아닙니다. 제너레이터는 예외가 새어 나가는 순간 닫혀버리므로, 스트림 전체를 하나의 try로 감싸는 방식으로는 "한 건 건너뛰고 계속"이 불가능합니다. 그래서 guarded는 매 건마다 f를 새로 실행해 격리하는데, 이 방식은 상태 없는 필터에만 쓸 수 있습니다. dedup처럼 건 사이 상태(seen)를 유지하는 필터를 이렇게 감싸면 매 건 상태가 초기화돼 중복 제거가 조용히 무력화됩니다. 격리 장치 하나에도 필터의 상태 유무라는 성질이 따라붙는다는 것 — 이것도 계약의 일부입니다.

*

함정 — 조용한 예외 처리는 조용한 장애가 됩니다. except: pass로 삼키기만 하는 크롤러는 소스 하나가 개편돼 파싱이 전멸해도 아무 소리를 내지 않습니다. "수집 0건"을 며칠 뒤 데이터 쪽에서 먼저 발견하게 되죠. 실패는 dead letter로 쌓고, 실패율이 임계치를 넘으면 사람을 부르는 알림까지가 한 세트입니다. 저는 이 임계치 알림을 Slack으로 받도록 붙여두고서야 스케줄러에 크롤러를 온전히 맡길 수 있었습니다.

6. 경계 — 이 구조가 깨지는 곳

패턴을 아는 것의 절반은 패턴이 안 통하는 지점을 아는 것입니다. Pipe & Filter가 깨지는 경계를 세 개 짚습니다.

전 구간 집계가 필요한 필터. "오늘 수집분 전체에서 카테고리별 비중을 계산해 태그를 단다" 같은 필터는 스트림 전체를 봐야 결과를 낼 수 있습니다. 구현하려면 필터 내부에서 list(stream)으로 실체화할 수밖에 없는데, 그 지점에서 지연 평가가 끊기고 배압도 함께 끊깁니다. 하나쯤은 감수할 수 있지만, 이런 필터가 늘어난 파이프라인은 겉모양만 파이프인 배치 덩어리입니다. 필터 절반이 실체화를 하고 있다면 처음부터 배치 잡으로 설계하는 편이 정직합니다.

단방향이 아닌 대화. 필터 계약은 한 방향으로만 흐릅니다. 그런데 수집은 그렇지 않습니다. 페이징은 "파싱 결과를 보고 다음 요청을 결정"하는 양방향 대화이고, 세션 만료나 봇 감지 대응도 마찬가지입니다. 제가 fetch·parse를 필터가 아니라 어댑터 내부에 둔 이유가 이것입니다. 양방향 대화는 어댑터 안에 가두고, 파이프에는 그 대화의 결과물인 단방향 스트림만 흘립니다. "수집도 단계니까 필터로 만들자"라고 계약을 억지로 씌우면, 필터끼리 콜백과 공유 상태로 뒷문 대화를 시작하고 구조는 안에서부터 무너집니다.

변화의 축을 잘못 짚었을 때. 가장 아픈 경계입니다. 개방-폐쇄 원칙은 "모든 변화에 닫혀 있다"가 아니라 "예측한 축의 변화에 닫혀 있다"입니다. 이 구조는 "소스가 늘어난다"에는 diff 0줄로 답하지만, Article 도메인 모델 자체가 바뀌는 변화 앞에서는 모든 필터와 모든 어댑터가 함께 흔들립니다. 우리 파이프라인이 무수정 확장을 누린 건 운이 아니라, 이 도메인에서 가장 자주 오는 변화가 "신규 소스"라는 판단이 맞았기 때문입니다. 축을 못 맞히면 같은 구조가 간접층만 늘린 비용으로 남습니다. 패턴 선택보다 변화 축의 예측이 먼저인 이유입니다.

7. 정리 — 무수정 확장의 규칙

크롤링 파이프라인에서 검증한 규칙을 요약합니다.

  • 필터 계약은 하나로 통일한다 — Iterable in, Iterable out. 합성이 닫혀 있어야 삽입·제거·재배열이 국소 변경이 됩니다.
  • 필터는 순수하게, IO는 양 끝으로 — 테스트가 리스트 넣고 리스트 받기로 끝나야 합니다.
  • 파이프라인은 구체 소스를 몰라야 한다 — 소스 이름이 파이프라인 코드에 등장하는 순간 무수정 확장은 끝납니다.
  • 등록은 선언적으로 — 레지스트리(또는 엔트리포인트)가 diff 0줄 확장의 마지막 조각입니다.
  • 병렬화로 큐가 생기면 반드시 유한 큐 — put()의 블록이 곧 배압입니다. 무한 큐는 장애를 메모리로 이연할 뿐입니다.
  • 실패는 격리하되 침묵시키지 않는다 — 한 건의 실패가 배치를 죽여도, 소리 없이 사라져도 안 됩니다. dead letter + 임계치 알림이 한 세트입니다.
  • 축이 다른 변화가 보이면 패턴을 접을 용기 — 전 구간 집계·양방향 대화·분기와 조인이 늘어나면, 파이프를 고집하지 말고 배치 잡이나 DAG 오케스트레이션으로 갈아탑니다.

다음 글에서는 이 파이프라인의 바깥 껍데기 — 비즈니스 흐름을 담는 UseCase 계층과 저장을 추상화하는 Repository — 를 다룹니다. 필터가 "데이터의 변형"을 격리했다면, UseCase와 Repository는 "의도와 영속성"을 격리합니다.
