논문 분석: Attention Is All You Need — 수식을 손으로 풀다

2021년 3월• Transformer · Attention · 논문

온프레미스에서 vLLM으로 LLM을 서빙하는 동안 "KV 캐시"라는 말을 매일 썼는데, 정작 그 K와 V가 무엇인지는 설명하지 못했습니다. 그게 계속 걸려서 2017년 Vaswani 등의 논문을 펴고, 거기 있는 수식 하나를 numpy로 직접 풀어 봤습니다. 그때 확인한 것들을 잊지 않으려고 적어 둡니다.

출처

Attention Is All You Need

Ashish Vaswani 외 · NeurIPS 2017 · arXiv:1706.03762

소프트맥스softmax
여러 점수를 모두 더하면 1이 되는 비율로 바꿔 주는 함수입니다. "어디를 얼마나 볼지"를 확률처럼 나눠 주는 역할을 하며, 어텐션의 가중치가 여기서 나옵니다.

1. RNN은 왜 병목이었나

Transformer 이전의 기계번역은 대부분 RNN 계열이었습니다. 논문을 읽으며 정리한 RNN의 구조적 한계는 두 가지였습니다. 하나는 h_t를 계산하려면 h_(t-1)이 먼저 나와야 해서 토큰을 순서대로 하나씩 처리할 수밖에 없다는 점입니다. GPU는 병렬 연산에 맞춰진 장치인데, 문장 길이만큼의 순차 단계가 학습 속도를 붙잡아 둡니다. 다른 하나는 문장 앞의 단어와 뒤의 단어가 관계를 맺으려면 그 사이의 모든 스텝을 거쳐야 해서, 멀리 떨어진 의존 관계일수록 신호가 희미해진다는 점입니다.

논문은 재귀를 아예 쓰지 않는 쪽을 택했습니다. 모든 토큰 쌍의 관계를 행렬곱 한 번으로 계산하면 순차 단계가 사라지고, 어떤 두 토큰 사이의 경로 길이도 상수가 됩니다. 논문이 보고한 수치는 base 모델 기준 P100 GPU 8장으로 약 12시간 학습, WMT 2014 영→독 번역에서 big 모델 28.4 BLEU였고, 당시 앙상블 모델들을 웃도는 결과였습니다.

2. 수식 하나 — softmax(QKᵀ/√d_k)V

논문에서 반복해 나오는 식은 Attention(Q, K, V) = softmax(QKᵀ/√d_k)V 한 줄입니다. 기호만 봐서는 잘 안 읽혀서, 검색에 비유해 놓고서야 이해가 됐습니다. Q(query)는 각 토큰이 던지는 질의, K(key)는 각 토큰이 내건 색인, V(value)는 실제 내용입니다. QKᵀ로 모든 질의-색인 쌍의 유사도를 구하고, softmax로 행마다 합이 1인 확률분포, 그러니까 "어디를 볼 것인가"를 만든 다음, 그 가중치로 V를 섞습니다.

import numpy as np

def softmax(x, axis=-1):
    x = x - x.max(axis=axis, keepdims=True)  # 오버플로 방지
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)

def attention(Q, K, V):
    d_k = Q.shape[-1]
    scores  = Q @ K.swapaxes(-2, -1) / np.sqrt(d_k)  # (n, n) 유사도
    weights = softmax(scores)            # 행마다 합이 1인 분포
    return weights @ V, weights

n, d_k = 4, 8                            # 토큰 4개, 차원 8
rng = np.random.default_rng(0)
x = rng.normal(size=(n, d_k))            # 토큰 임베딩이라 치자
out, w = attention(x, x, x)              # self-attention: Q=K=V=x
print(w.round(2))                        # 토큰별 "어디를 보는가"

실제 모델에서는 Q, K, V가 같은 입력 x에 학습된 가중치 W_q, W_k, W_v를 곱해 만들어집니다. 위 예제는 그 투영을 생략한 최소형인데, 그래도 수식의 뼈대는 10줄 남짓에 다 담깁니다.

*

논문 수식이 안 읽힐 때는 기호보다 shape를 먼저 따라가는 편이 저에게는 나았습니다. (n, d_k) @ (d_k, n) → (n, n), 거기에 (n, n) @ (n, d_v) → (n, d_v). shape가 맞아떨어지고 나서야 수식이 코드로 읽혔습니다.

위 코드의 print(w.round(2))가 뱉는 (n, n) 행렬은 숫자로만 보면 감이 잘 안 왔습니다. 그래서 토이 문장 두 개의 QKᵀ/√d_k 점수 행렬을 미리 만들어 두고, 거기에 softmax를 실제로 돌려 색으로 그려 봤습니다. 단어를 클릭하면 그 단어가 문장의 어디를 얼마나 보는지가 퍼센트로 나옵니다. 아래쪽 온도 슬라이더는 softmax에 넣기 전 점수를 T로 나누는 장치인데, 이 나눗셈이 왜 필요한지는 다음 절의 √d_k 이야기와 이어집니다.

어텐션 히트맵 — softmax(QKᵀ/√d_k) 돌려보기직접 조작해 보세요

0.3 · 뾰족 — 한 곳에 몰림3.0 · 평평 — 고르게 분산

행 = 질의(Q)를 던지는 단어, 열 = 그 단어가 바라보는 상대(K). 색이 진할수록 가중치가 큽니다. 단어 셀을 클릭해 보세요.

3. √d_k로 나누는 이유

처음엔 √d_k를 장식처럼 넘겼는데, 논문은 각주에 이유를 적어 두었습니다. 성분이 평균 0, 분산 1인 두 벡터의 내적은 분산이 d_k만큼 커진다는 것이었습니다. 글로만 봐서는 크기가 와닿지 않아 직접 찍어 봤습니다.

d = 512
q = rng.normal(size=(10000, d))
k = rng.normal(size=(10000, d))
dots = (q * k).sum(axis=1)

print(dots.std())                # ≈ 22.6 ≈ sqrt(512)
print((dots / np.sqrt(d)).std()) # ≈ 1.0

표준편차가 22를 넘는 값들이 softmax에 들어가면 지수 함수가 큰 값 하나로 쏠려 사실상 one-hot에 가까워지고, 그러면 기울기가 0 근처로 내려가 학습이 진척되지 않습니다. √d_k로 나누는 한 줄은 softmax를 기울기가 살아 있는 구간에 붙잡아 두는 장치였습니다. 수식에 붙은 상수를 눈으로만 훑고 넘기던 습관은 이 각주를 읽고 나서 조금 고쳤습니다.

4. 멀티헤드 — 관점을 8개로 쪼갠다

attention을 d_model=512 전체 차원으로 한 번 수행하는 대신, 논문은 64차원짜리 헤드 8개로 쪼갭니다(base 모델 기준 h=8, d_k=d_v=64). 각 헤드는 자기만의 투영 행렬을 가지므로, 어떤 헤드는 인접 단어를, 어떤 헤드는 문법적 호응을 보는 식으로 서로 다른 관계를 학습할 수 있습니다. 총 연산량은 전체 차원으로 한 번 계산할 때와 거의 같습니다.

def multi_head(x, W_q, W_k, W_v, W_o, h=8):
    n, d_model = x.shape
    d_k = d_model // h                   # 512 // 8 = 64
    # (n, d_model) → (h, n, d_k): 헤드 축을 앞으로
    Q = (x @ W_q).reshape(n, h, d_k).swapaxes(0, 1)
    K = (x @ W_k).reshape(n, h, d_k).swapaxes(0, 1)
    V = (x @ W_v).reshape(n, h, d_k).swapaxes(0, 1)
    heads, _ = attention(Q, K, V)        # 헤드 8개를 한 번에
    concat = heads.swapaxes(0, 1).reshape(n, d_model)
    return concat @ W_o                  # 다시 하나로 섞는다

d_model = 512
W = [rng.normal(size=(d_model, d_model)) / np.sqrt(d_model) for _ in range(4)]
x = rng.normal(size=(6, d_model))       # 토큰 6개짜리 문장
print(multi_head(x, *W).shape)           # (6, 512) — 들어온 모양 그대로

여기서는 2절의 attention 함수를 고치지 않고 그대로 다시 썼습니다. 앞에서 swapaxes(-2, -1)로 적어 둔 덕에 헤드 축이 배치처럼 앞에 붙어도 같은 코드가 돌아갔습니다. 코드로 옮겨 놓고 보니 멀티헤드는 reshape 두 번과 행렬곱 하나였습니다.

5. 포지셔널 인코딩 — 순서를 더하기로 복원한다

attention에는 걸리는 성질이 하나 있습니다. 토큰을 뒤섞어 넣어도 각 토큰이 받는 결과는 행 자리만 따라 옮겨질 뿐 그대로인데, 이를 순열 등변이라고 합니다. 어순만으로 갈리는 "누가 물었고 누가 물렸는지"를 구분할 근거가 식 안에 없다는 뜻입니다. 논문은 순서 정보를 담은 벡터를 임베딩에 그냥 더하는 방식을 택했고, 그 벡터를 주기가 기하급수적으로 다른 사인·코사인으로 만들었습니다.

def positional_encoding(n, d_model):
    pos = np.arange(n)[:, None]              # (n, 1) 위치
    i   = np.arange(d_model // 2)[None, :]   # (1, d/2) 차원 인덱스
    angle = pos / 10000 ** (2 * i / d_model)
    pe = np.empty((n, d_model))
    pe[:, 0::2] = np.sin(angle)              # 짝수 차원
    pe[:, 1::2] = np.cos(angle)              # 홀수 차원
    return pe

x = x + positional_encoding(6, d_model)      # 임베딩에 더하면 끝

낮은 차원은 토큰마다 빠르게 진동해 이웃을 구분하고, 높은 차원은 문장 전체에 걸쳐 천천히 변해 대략적 위치를 담습니다. 사인·코사인 쌍으로 만들면 위치 차이(상대 위치)가 선형 변환으로 표현된다는 성질도 논문이 이 형태를 고른 이유로 제시됩니다. 처음엔 concat이 아니라 덧셈이라는 게 이상하게 느껴졌는데, 512차원 공간은 순서 신호가 얹혀도 내용 신호와 얽히지 않을 만큼 넓다는 게 요지였습니다.

6. 이 수식이 서빙 병목이 되기까지

여기까지가 논문 쪽 이야기고, 제가 이 논문을 다시 편 이유는 운영 쪽에 있었습니다. 추론 단계에서 토큰을 하나씩 생성할 때, 새 토큰의 Q는 이전 모든 토큰의 K와 V를 상대로 attention을 수행합니다. 이전 토큰들의 K·V는 매번 다시 계산할 필요가 없어 저장해 두는데, 그것이 KV 캐시였습니다. 2절 코드에 나온 K, V 행렬이 GPU 메모리에 그대로 쌓이는 셈입니다.

문맥이 길어질수록 이 캐시는 토큰 수에 비례해 자라고, QKᵀ의 (n, n) 연산은 제곱으로 자랍니다. 온프레미스 서빙에서 gpu-memory-utilization과 동시 처리 슬롯 수를 두고 씨름했던 일이 이 두 줄의 수식과 닿아 있었습니다. vLLM의 PagedAttention은 이 KV 캐시를 페이지 단위로 관리하는 기법인데, 그 논문은 다음 글에서 따로 다룹니다.

*

서빙 파라미터를 만지기 전에 수식으로 돌아가면 감이 잡혔습니다. KV 캐시 크기는 대략 2 × 레이어 수 × 토큰 수 × d_model × 바이트 수입니다. 최대 문맥 길이와 동시 슬롯 수를 정할 때 이 곱셈을 먼저 해 두면, OOM이 나기 전에 대략의 예산이 보입니다.

7. 손으로 풀어 남은 것

  • 수식 한 줄에서 갈라져 나왔다. QKV도 멀티헤드도 softmax(QKᵀ/√d_k)V의 변주였고, numpy 20줄로 뼈대까지는 따라갈 수 있었다.
  • 상수에도 이유가 있었다. √d_k는 장식이 아니라 기울기를 살려 두는 장치였고, 분산을 찍어 보는 다섯 줄로 확인됐다.
  • 이론이 운영 쪽 언어와 붙었다. K·V가 무엇인지 알고 나니 KV 캐시, 문맥 길이, 메모리 예산이 한 그림으로 이어졌다. 그전까지는 같은 값들을 감으로 조정하고 있었다.
