논문 분석: Attention Is All You Need — 수식을 손으로 풀다

2021년 3월• Transformer · Attention · 논문

온프레미스에서 vLLM으로 LLM을 서빙하면서 매일 "KV 캐시"라는 말을 썼습니다. 그런데 그 K와 V가 정확히 무엇이냐고 스스로에게 물었을 때 말문이 막혔습니다. 그래서 원점으로 돌아갔습니다 — 2017년 Vaswani 등의 논문에 있는 수식 하나를 numpy로 직접 풀어본 기록입니다.

먼저 짚고 가기
임베딩embedding
단어를 컴퓨터가 계산할 수 있도록 숫자 여러 개(벡터)로 바꾼 것입니다. 비슷한 뜻의 단어는 비슷한 숫자 묶음이 되도록 학습됩니다. 이 글의 모든 계산은 결국 이 숫자 묶음들을 서로 섞는 일입니다.
RNNRecurrent Neural Network
문장을 앞에서부터 한 단어씩 순서대로 읽어 나가는 옛 방식의 신경망입니다. 앞 계산이 끝나야 다음을 계산할 수 있어 느린데, Transformer가 이 순차 처리를 없애며 대체했습니다.
어텐션attention
한 단어가 문장 속 다른 단어들 중 어디에 집중할지를 정하는 장치입니다. "물었다"가 "개"와 "사람" 중 누구에 얼마나 관련되는지 가중치로 매겨 섞습니다. 이 논문의 핵심 아이디어입니다.
Q · K · VQuery · Key · Value
어텐션을 도서관 검색에 비유한 세 역할입니다. Q는 던지는 질문, K는 각 자료에 붙은 색인표, V는 자료의 실제 내용입니다. 질문(Q)과 색인(K)이 잘 맞는 곳의 내용(V)을 많이 가져옵니다.
소프트맥스softmax
여러 점수를 모두 더하면 1이 되는 비율로 바꿔 주는 함수입니다. "어디를 얼마나 볼지"를 확률처럼 나눠 주는 역할을 하며, 어텐션의 가중치가 여기서 나옵니다.

1. RNN은 왜 병목이었나

Transformer 이전의 기계번역은 RNN 계열이 지배했습니다. RNN의 구조적 한계는 두 가지입니다. 첫째, h_t를 계산하려면 h_(t-1)이 먼저 나와야 하므로 토큰을 순서대로 하나씩 처리할 수밖에 없습니다. GPU는 병렬 연산 기계인데, 문장 길이만큼의 순차 단계가 학습을 가로막습니다. 둘째, 문장 앞의 단어와 뒤의 단어가 관계를 맺으려면 그 사이 모든 스텝을 거쳐야 해서, 멀리 떨어진 의존 관계일수록 신호가 희미해집니다.

논문의 답은 재귀를 통째로 버리는 것이었습니다. 모든 토큰 쌍의 관계를 행렬곱 한 번으로 계산하면, 순차 단계는 사라지고 어떤 두 토큰 사이의 경로 길이도 상수가 됩니다. 그 결과가 base 모델 기준 P100 GPU 8장으로 약 12시간 학습, WMT 2014 영→독 번역에서 big 모델 28.4 BLEU — 당시 앙상블 모델까지 제친 수치였습니다.

2. 수식 하나 — softmax(QKᵀ/√d_k)V

논문의 핵심은 이 한 줄입니다. Attention(Q, K, V) = softmax(QKᵀ/√d_k)V. 저는 이걸 검색에 비유해서야 이해했습니다. Q(query)는 각 토큰이 던지는 질의, K(key)는 각 토큰이 내건 색인, V(value)는 실제 내용입니다. QKᵀ로 모든 질의-색인 쌍의 유사도를 구하고, softmax로 행마다 합이 1인 확률분포("어디를 볼 것인가")를 만들어, 그 가중치로 V를 섞습니다.

import numpy as np

def softmax(x, axis=-1):
    x = x - x.max(axis=axis, keepdims=True)  # 오버플로 방지
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)

def attention(Q, K, V):
    d_k = Q.shape[-1]
    scores  = Q @ K.swapaxes(-2, -1) / np.sqrt(d_k)  # (n, n) 유사도
    weights = softmax(scores)            # 행마다 합이 1인 분포
    return weights @ V, weights

n, d_k = 4, 8                            # 토큰 4개, 차원 8
rng = np.random.default_rng(0)
x = rng.normal(size=(n, d_k))            # 토큰 임베딩이라 치자
out, w = attention(x, x, x)              # self-attention: Q=K=V=x
print(w.round(2))                        # 토큰별 "어디를 보는가"

실제 모델에서는 Q, K, V가 같은 입력 x에 학습된 가중치 W_q, W_k, W_v를 곱해 만들어집니다. 위 예제는 투영을 생략한 최소형이지만, 수식의 뼈대는 이게 전부입니다. 10줄 남짓입니다.

*

논문 수식이 안 읽힐 때는 기호보다 shape를 먼저 따라가세요. (n, d_k) @ (d_k, n) → (n, n), 거기에 (n, n) @ (n, d_v) → (n, d_v). shape가 맞아떨어지는 순간 수식이 코드로 읽히기 시작합니다.

3. √d_k로 나누는 이유

처음엔 √d_k를 장식처럼 지나쳤는데, 논문은 각주에서 이유를 분명히 밝힙니다. 성분이 평균 0, 분산 1인 두 벡터의 내적은 분산이 d_k만큼 커집니다. 직접 확인해 봤습니다.

d = 512
q = rng.normal(size=(10000, d))
k = rng.normal(size=(10000, d))
dots = (q * k).sum(axis=1)

print(dots.std())                # ≈ 22.6 ≈ sqrt(512)
print((dots / np.sqrt(d)).std()) # ≈ 1.0

표준편차 22가 넘는 값들이 softmax에 들어가면 지수 함수가 큰 값 하나로 쏠려 사실상 one-hot이 됩니다. 기울기는 0에 가까워지고 학습이 멈춥니다. √d_k로 나누는 한 줄은 softmax를 기울기가 살아 있는 구간에 붙잡아 두는 장치입니다. 수식의 "잡다한 상수"에도 다 이유가 있다는 걸 이 논문에서 배웠습니다.

4. 멀티헤드 — 관점을 8개로 쪼갠다

attention을 d_model=512 전체 차원으로 한 번 수행하는 대신, 논문은 64차원짜리 헤드 8개로 쪼갭니다(base 모델 기준 h=8, d_k=d_v=64). 각 헤드는 자기만의 투영 행렬을 가지므로 서로 다른 관계 — 어떤 헤드는 인접 단어, 어떤 헤드는 문법적 호응 — 를 보도록 학습될 수 있습니다. 총 연산량은 전체 차원 한 방과 거의 같습니다.

def multi_head(x, W_q, W_k, W_v, W_o, h=8):
    n, d_model = x.shape
    d_k = d_model // h                   # 512 // 8 = 64
    # (n, d_model) → (h, n, d_k): 헤드 축을 앞으로
    Q = (x @ W_q).reshape(n, h, d_k).swapaxes(0, 1)
    K = (x @ W_k).reshape(n, h, d_k).swapaxes(0, 1)
    V = (x @ W_v).reshape(n, h, d_k).swapaxes(0, 1)
    heads, _ = attention(Q, K, V)        # 헤드 8개를 한 번에
    concat = heads.swapaxes(0, 1).reshape(n, d_model)
    return concat @ W_o                  # 다시 하나로 섞는다

d_model = 512
W = [rng.normal(size=(d_model, d_model)) / np.sqrt(d_model) for _ in range(4)]
x = rng.normal(size=(6, d_model))       # 토큰 6개짜리 문장
print(multi_head(x, *W).shape)           # (6, 512) — 들어온 모양 그대로

2절의 attention 함수를 그대로 재사용했다는 점이 중요합니다. swapaxes(-2, -1)로 써 둔 덕에 헤드 축이 배치처럼 붙어도 같은 코드가 돌아갑니다. reshape 두 번과 행렬곱 하나 — 멀티헤드의 실체는 이게 전부입니다.

5. 포지셔널 인코딩 — 순서를 더하기로 복원한다

attention에는 치명적인 성질이 하나 있습니다. 토큰의 순서를 바꿔도 결과가 (행 순서만 바뀐 채) 똑같다는 것 — 순열 불변입니다. "개가 사람을 물었다"와 "사람이 개를 물었다"를 구분하지 못합니다. 논문은 순서 정보를 담은 벡터를 임베딩에 그냥 더하는 방식을 택했고, 그 벡터를 주기가 기하급수적으로 다른 사인·코사인으로 만들었습니다.

def positional_encoding(n, d_model):
    pos = np.arange(n)[:, None]              # (n, 1) 위치
    i   = np.arange(d_model // 2)[None, :]   # (1, d/2) 차원 인덱스
    angle = pos / 10000 ** (2 * i / d_model)
    pe = np.empty((n, d_model))
    pe[:, 0::2] = np.sin(angle)              # 짝수 차원
    pe[:, 1::2] = np.cos(angle)              # 홀수 차원
    return pe

x = x + positional_encoding(6, d_model)      # 임베딩에 더하면 끝

낮은 차원은 토큰마다 빠르게 진동해 이웃을 구분하고, 높은 차원은 문장 전체에 걸쳐 천천히 변해 대략적 위치를 담습니다. 사인·코사인 쌍으로 만들면 위치 차이(상대 위치)가 선형 변환으로 표현된다는 성질도 논문이 이 형태를 고른 이유로 제시됩니다. "concat이 아니라 덧셈이라니" 싶었지만, 512차원 공간은 순서 신호가 얹혀도 내용 신호와 얽히지 않을 만큼 넓다는 게 요지입니다.

6. 이 수식이 서빙 병목이 되기까지

여기까지가 이론이고, 제가 이 논문을 다시 편 이유는 운영 쪽에 있습니다. 추론 단계에서 토큰을 하나씩 생성할 때, 새 토큰의 Q는 이전 모든 토큰의 K와 V를 상대로 attention을 수행합니다. 이전 토큰들의 K·V는 매번 다시 계산할 필요가 없으니 저장해 두는데 — 그게 바로 KV 캐시입니다. 2절 코드의 K, V 행렬이 GPU 메모리에 그대로 쌓이는 셈입니다.

문맥이 길어질수록 이 캐시는 토큰 수에 비례해 자라고, QKᵀ의 (n, n) 연산은 제곱으로 자랍니다. 제가 온프레미스 서빙에서 gpu-memory-utilization과 동시 처리 슬롯 수를 붙잡고 씨름했던 이유가 결국 이 두 줄의 수식이었습니다. vLLM의 PagedAttention은 이 KV 캐시를 페이지 단위로 관리하는 기법인데, 그 논문은 다음 글에서 따로 다룹니다.

*

서빙 파라미터를 튜닝하기 전에 수식으로 돌아가면 감이 잡힙니다. KV 캐시 크기는 대략 2 × 레이어 수 × 토큰 수 × d_model × 바이트 수 — 최대 문맥 길이와 동시 슬롯 수를 정할 때 이 곱셈부터 해 보면, OOM이 나기 전에 예산이 보입니다.

7. 손으로 풀어 남은 것

  • 수식 하나가 전부였다. QKV도, 멀티헤드도, 결국 softmax(QKᵀ/√d_k)V의 변주다. numpy 20줄이면 뼈대가 다 나온다.
  • 상수에도 이유가 있다. √d_k는 장식이 아니라 기울기를 살리는 장치였다. 분산 실험 다섯 줄로 확인할 수 있었다.
  • 이론은 운영의 언어가 된다. K·V가 뭔지 알고 나니 KV 캐시, 문맥 길이, 메모리 예산이 하나의 그림으로 연결됐다. 도구를 쓰는 것과 이해하고 쓰는 것의 차이였다.
