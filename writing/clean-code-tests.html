클린코드 · 테스트 — 테스트가 설계를 바꾼다

2021년 1월• 약 12분• 클린코드 · 테스트 · 의존성 주입

테스트가 안 짜지는 코드를 만나면 저는 테스트를 탓하기 전에 설계를 봅니다. 테스트하기 어려움은 결합도의 다른 이름이기 때문입니다. 이 글은 "커버리지를 올리는 법"이 아니라, 테스트라는 렌즈로 시간·랜덤·IO 의존을 찾아내 설계를 고치고, 그렇게 만든 테스트를 읽히는 문서로 유지하는 법을 다룹니다.

먼저 짚고 가기
결합도Coupling
어떤 코드가 다른 코드에 얼마나 얽혀 있는지를 나타내는 정도입니다. 실을 많이 엮어 놓으면 하나만 당겨도 전체가 끌려오듯, 결합도가 높으면 한 부분을 테스트하려 해도 관련된 것들이 줄줄이 딸려 옵니다. 이 글은 "테스트가 어렵다"를 곧 "결합도가 높다"는 신호로 읽습니다.
의존성 주입Dependency Injection
필요한 부품(시계, 정책 등)을 객체가 스스로 만들지 않고, 밖에서 만들어 넣어 주는 방식입니다. 요리사가 재료를 직접 밭에서 캐 오는 대신 손질된 재료를 받아 쓰는 것과 같습니다. 재료를 바꿔 넣을 수 있으니, 테스트에서는 "가짜 시계"처럼 통제된 부품을 끼워 검증할 수 있습니다.
비결정성Non-determinism
같은 코드를 같은 조건으로 두 번 돌려도 결과가 매번 달라질 수 있는 성질입니다. 시간·랜덤·IO가 대표적인 원인입니다. 결과를 예측할 수 없으면 "무엇이 맞는지" 못 박을 수 없으니, 이 글은 비결정성을 코드 밖 경계로 밀어내는 것을 핵심 전략으로 삼습니다.
순수 함수Pure Function
같은 입력에 항상 같은 결과를 내고, 바깥 세계를 건드리지 않는(DB 저장 등이 없는) 함수입니다. 계산기처럼 입력만으로 결과가 정해지므로, DB나 네트워크 없이도 값만 넣어 보면 검증이 끝납니다. 계산과 저장을 분리하면 계산 부분이 순수 함수가 됩니다.
목 · 스텁Mock · Stub
진짜 부품 대신 세워 두는 가짜 대역입니다. 스텁은 미리 정한 답을 돌려주는 대역, 목은 "제대로 호출됐는지"까지 확인하는 대역입니다. 결제 게이트웨이처럼 진짜로 부를 수 없는 경계에는 유용하지만, 남용하면 테스트가 동작이 아니라 구현 방식을 못 박아 리팩토링을 방해합니다.

테스트하기 어려운 코드는 설계가 보내는 신호입니다

테스트를 짠다는 것은 결국 코드의 모든 입력을 내 손으로 통제하고, 모든 출력을 내 눈으로 관찰하는 일입니다. 그런데 어떤 코드는 아무리 애써도 통제가 안 됩니다. 함수 안에서 스스로 시계를 보고, 전역 상태를 읽고, DB에 곧장 씁니다. 통제할 수 없는 입력이 있다는 건 취향의 문제가 아니라 이 함수의 진짜 입력이 시그니처에 다 드러나 있지 않다는 형식적 증거입니다. 정산 로직을 리팩토링하기 전, 제가 마주했던 코드가 정확히 그랬습니다.

// Before — 이 코드는 "오늘"을 스스로 안다
public class SettlementService {

  public void settle(Order order) {
    LocalDate today = LocalDate.now();          // 숨은 입력 1: 시간
    FeePolicy policy = FeePolicy.current();     // 숨은 입력 2: 전역 상태
    BigDecimal fee = policy.feeOf(order, today);
    repository.save(new Settlement(order, fee, today)); // 숨은 출력: IO
  }
}

이 코드의 주말 수수료 분기를 검증하려면 선택지가 둘뿐입니다. 토요일까지 기다리거나, 시스템 시계를 조작하거나. 반환값도 없어서 결과를 보려면 DB를 다시 읽어야 합니다. 테스트가 어려운 게 아니라, 이 함수가 자신의 입력과 출력을 숨기고 있는 것입니다. 그래서 "테스트 먼저"라는 구호가 실무에서 힘을 갖습니다. 테스트를 먼저 생각하는 순간, 숨은 의존이 전부 수면 위로 올라오기 때문입니다.

시간·랜덤·IO — 비결정성을 경계로 밀어냅니다

테스트를 막는 3대 원흉은 시간, 랜덤, IO입니다. 셋의 공통점은 비결정성 — 같은 코드를 두 번 실행해도 같은 결과가 보장되지 않는다는 점입니다. 결정적이지 않은 함수는 검증할 수 없고, 검증할 수 없는 코드는 명세가 없는 것과 같습니다. 해법은 하나의 원칙으로 수렴합니다. 비결정성을 함수 밖으로, 시스템의 경계로 밀어내라. 가장 쉬운 도구가 의존성 주입입니다.

// After — "오늘"은 밖에서 들어오고, 저장은 호출자의 일이 된다
public class SettlementService {
  private final Clock clock;
  private final FeePolicy feePolicy;

  public SettlementService(Clock clock, FeePolicy feePolicy) {
    this.clock = clock; this.feePolicy = feePolicy;
  }

  public Settlement settle(Order order) {
    LocalDate today = LocalDate.now(clock);
    return Settlement.of(order, feePolicy.feeOf(order, today), today);
  }
}

// JUnit — 토요일을 기다릴 필요가 없다
@Test
void 주말_주문에는_주말_수수료율이_적용된다() {
  Clock saturday = Clock.fixed(Instant.parse("2026-07-18T00:00:00Z"), SEOUL);
  SettlementService service = new SettlementService(saturday, defaultPolicy());

  Settlement result = service.settle(orderOf(10_000));

  assertThat(result.feeRate()).isEqualTo(WEEKEND_RATE);
}

바뀐 것은 세 가지입니다. 시간은 Clock으로 주입받고, 정책은 인터페이스 뒤로 갔고, save는 사라지고 결과를 반환합니다. 계산과 저장을 분리하면 계산은 순수 함수가 되어 DB 없이 검증되고, 저장은 얇아져서 통합 테스트 몇 개면 충분해집니다. 정산 시스템을 Template Method와 Strategy로 리팩토링했을 때 체감한 것이 이것입니다. 요율 정책이 Strategy로 분리되자 정책 × 케이스의 조합을 순수 단위 테스트로 다 깔 수 있었고, 테스트가 쉬워진 만큼 새 정책을 추가하는 일도 쉬워졌습니다. 테스트 가능성과 확장 가능성은 같은 설계의 양면입니다.

다만 이 원칙에도 경계가 있습니다. "모든 곳에 Clock을 주입하라"가 되면 곤란합니다. 도메인 깊숙한 객체까지 Clock을 끌고 가면 생성자들이 시계 배달 통로가 되고, 주입 자체가 새로운 결합이 됩니다. 더 좋은 형태는 유스케이스 진입점(컨트롤러·스케줄러)에서 now를 한 번만 찍어 값으로 내려보내는 것입니다. 그러면 도메인 함수는 주입조차 필요 없는 settle(order, today)가 됩니다. 즉 상위 규칙은 "주입하라"가 아니라 "비결정성을 경계에서 한 번만 해석하라"이고, 주입은 그 수단 중 하나일 뿐입니다.

*

NestJS에서 Jest의 jest.useFakeTimers()는 전역 시계를 조작하는 방식이라, 디바운스·재시도 지연처럼 "흐르는 시간"을 검증할 때만 쓰는 게 안전합니다. 도메인이 "오늘"을 아는 문제는 가짜 타이머가 아니라 주입 또는 파라미터로 푸세요. 전역 조작은 같은 파일의 test.concurrent, 그리고 실제 타이머에 기대는 라이브러리(HTTP 타임아웃·커넥션 유지)와 얽히는 순간 새기 시작합니다.

준비-실행-검증 — 테스트는 실행되는 횟수보다 읽히는 횟수가 많습니다

테스트를 실행하는 건 CI지만, 읽는 건 사람입니다. 6개월 뒤의 저는 요구사항 문서가 아니라 테스트를 열어 "이 시스템이 약속한 동작"을 확인합니다. 그래서 테스트는 준비(Arrange) → 실행(Act) → 검증(Assert)의 리듬이 눈에 보여야 하고, 이름은 완결된 문장이어야 합니다. SafeIn 백엔드를 단독으로 설계했을 때는 리뷰어가 없었기 때문에, 테스트 이름을 문장으로 쓰는 습관이 사실상 코드 리뷰를 대신했습니다. 이름만 위에서 아래로 읽으면 명세서가 되도록.

// Jest — 이름만 모아 읽으면 명세가 되게
describe('CouponService.apply', () => {
  it('만료된 쿠폰이면 ExpiredCouponError를 던진다', async () => {
    // 준비 — 이 테스트의 조건만 눈에 띄게
    const coupon  = couponFixture({ expiresAt: daysAgo(1) });
    const service = new CouponService(fixedClock('2026-07-23'));

    // 실행 — 한 줄
    const apply = () => service.apply(anOrder(), coupon);

    // 검증 — 한 관심사
    await expect(apply).rejects.toThrow(ExpiredCouponError);
  });
});

이 리듬은 미관이 아니라 진단 도구입니다. 실행이 두 줄 이상이면 대상 함수가 두 가지 일을 하고 있다는 신호이고(함수 편에서 다룬 "한 가지 일"이 여기서 다시 등장합니다), 검증이 서로 다른 관심사를 여러 개 확인하면 테스트가 두 개여야 한다는 신호입니다. 준비가 화면을 넘길 만큼 길다면 대상 객체의 생성이 너무 무겁다는 뜻이니, 그것도 설계로 되돌아갈 신호입니다. AAA가 안 지켜지는 테스트는 대개 테스트의 잘못이 아닙니다.

목 남용 — 테스트가 구현을 못 박을 때

테스트의 존재 이유는 동작을 보존한 채 구현을 바꿀 자유, 즉 리팩토링을 가능하게 하는 것입니다. 그런데 목(mock)으로 호출 여부와 인자를 일일이 검증하는 테스트는 동작이 아니라 구현의 현재 모습을 고정합니다. 레거시 UseCase를 개선할 때 이걸 뼈저리게 겪었습니다. 목이 겹겹이 쌓인 테스트들은 리팩토링 첫 커밋 — 동작은 하나도 바뀌지 않은 커밋 — 에서 전부 빨갛게 물들었습니다. 안전망이어야 할 테스트가 족쇄가 된 겁니다.

// Bad — 결과가 아니라 "어떻게"를 검증한다
it('정산을 계산한다', async () => {
  await service.settle(order);

  expect(feePolicy.feeOf).toHaveBeenCalledTimes(1);
  expect(feePolicy.feeOf).toHaveBeenCalledWith(order, expect.anything());
  expect(repo.save).toHaveBeenCalledWith(
    expect.objectContaining({ fee: 300 })
  );
});
// 내부에 캐시를 넣거나 호출 순서를 바꾸면 — 동작은 같아도 — 깨진다

// Good — 검증은 결과 상태로
it('1만 원 카드 주문의 수수료는 300원이다', async () => {
  const result = await service.settle(cardOrderOf(10_000));

  expect(result.fee).toBe(300);
});

왜 상태 검증이 우월할까요. 리팩토링의 정의가 "관찰 가능한 동작을 보존하는 변경"이기 때문입니다. 반환값과 상태는 관찰 가능한 동작이지만, 내부 협력자를 몇 번 불렀는지는 관찰 가능한 동작이 아니라 구현 세부입니다. 구현 세부를 검증하는 테스트는 정의상 모든 리팩토링에서 깨질 수밖에 없습니다.

그렇다고 "목 금지"를 새 규칙으로 삼으면 반대쪽 벽에 부딪힙니다. 여기가 이 원칙의 경계입니다. 프로세스 경계 밖 — 결제 게이트웨이, SMTP, 남의 서버 — 는 목·스텁이 정답입니다. 실물을 부를 수 없고, 불러서도 안 되니까요. DDD로 크롤링 파이프라인을 만들 때는 대상 사이트의 HTML을 픽스처 파일로 박제해 스텁으로 썼습니다. 사이트가 바뀌면 픽스처를 갱신하면 되고, 파서 테스트는 네트워크 없이 밀리초에 돕니다. 정리하면 이렇습니다. 내가 소유한 코드는 실물로, 소유하지 않은 경계는 목으로. 목은 경계 표시선이지, 편의 도구가 아닙니다.

*

한 테스트에 toHaveBeenCalledWith가 세 개 이상 보이면 그 테스트는 명세가 아니라 구현의 스냅샷입니다. 리팩토링 때마다 깨진다면 지키는 게 없다는 뜻 — 상태 검증으로 바꾸거나, 바꿀 수 없다면 지우는 쪽이 코드베이스에 이롭습니다.

픽스처에도 의도를 — 데이터가 문장이 되게

테스트 본문을 아무리 다듬어도 준비 단계의 데이터가 불투명하면 문서가 되지 못합니다. 인자 일곱 개짜리 생성자 호출에서 "이 테스트의 주인공이 어느 값인지"를 알아내는 건 독자의 몫이 되니까요.

// Bad — 어떤 값이 이 테스트의 주인공인지 알 수 없다
const order = new Order('u-1', 10_000, 'CARD', true, null, 3);

// Good — 결론에 기여하는 값만 앞으로, 나머지는 기본값 뒤로
const order = anOrder().amount(10_000).paidBy('CARD').build();

// 더 좋은 것 — 의도가 이름이 된 픽스처
const weekendOrder  = orderOn('2026-07-18');        // 토요일
const expiredCoupon = couponExpiredSince(daysAgo(1));

규칙은 하나입니다. 테스트의 결론에 기여하는 값만 보이게 하고, 나머지는 빌더의 기본값 뒤로 숨긴다. 이 규칙이 성립하는 이유는 관련성의 문제입니다. 화면에 보이는 모든 값은 독자에게 "이 값이 결과에 영향을 주는가?"라는 질문을 강제합니다. 무관한 값 다섯 개가 보이면 독자는 다섯 번 헛질문을 합니다. 반대로 weekendOrder, expiredCoupon처럼 의도가 이름이 된 픽스처는 질문 자체를 없앱니다. 부수 효과도 큽니다. 필드가 추가돼도 빌더의 기본값 한 곳만 고치면 수백 개의 테스트가 그대로 살아남습니다.

게임에서 배운 것 — 자동 플레이테스트 20종

게임 프로젝트에서 자동 플레이테스트 20종을 만들어 돌린 적이 있습니다. 스폰, 이동, 전투, 보상 같은 시나리오를 봇이 자동으로 수행하고 결과를 검증하는 방식이었는데, 여기서도 이 글의 원칙이 그대로 성립했습니다. 랜덤이 낀 전투는 시드를 고정해야 재현 가능했고 — 랜덤 주입의 다른 얼굴입니다 — 시나리오 이름은 실패 로그만 보고도 무엇이 깨졌는지 알 수 있게 문장으로 지어야 했습니다. 그리고 가장 큰 수확은 심리적인 것이었습니다. 20종이 매번 돌아준다는 믿음이 생기자 밸런스 수치를 바꾸는 일이 더 이상 겁나지 않았습니다. 서버 코드도 똑같습니다. 테스트가 주는 진짜 가치는 버그 검출이 아니라, 변경을 두려워하지 않게 되는 것입니다.

정리 — 오늘 가져갈 규칙

테스트는 품질 검사 도구이기 전에 설계 도구입니다. 테스트가 괴로울 때 고쳐야 할 것은 대부분 테스트가 아니라 코드 쪽입니다.

  • 테스트가 안 짜지면 설계를 의심합니다. 통제할 수 없는 입력 = 시그니처에 숨은 의존이 있다는 증거입니다.
  • 시간·랜덤·IO는 경계로 밀어냅니다. 주입하거나, 더 좋게는 진입점에서 한 번 해석해 값으로 내려보냅니다.
  • 계산과 저장을 분리합니다. 계산은 순수 함수로 단위 테스트, 저장은 얇게 만들어 통합 테스트 몇 개로.
  • 준비-실행-검증. 실행은 한 줄, 검증은 한 관심사. 길어지면 테스트가 아니라 대상 함수를 쪼갭니다.
  • 이름은 문장으로. 테스트 이름만 모아 읽으면 명세서가 되어야 합니다.
  • 내가 소유한 코드는 실물로, 소유하지 않은 프로세스 경계만 목으로. 호출 검증이 쌓이면 구현을 못 박은 것입니다.
  • 픽스처는 결론에 기여하는 값만 보이게. 나머지는 빌더 기본값 뒤로 숨깁니다.
