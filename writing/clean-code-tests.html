클린코드 · 테스트 — 테스트가 설계를 바꾼다

2021년 1월• 클린코드 · 테스트 · 의존성 주입

테스트가 잘 안 짜이는 코드를 여러 번 만났습니다. 처음엔 제가 테스트를 못 쓰는 줄 알았는데, 들여다볼 때마다 문제는 코드 쪽에 있었습니다. 테스트하기 어렵다는 건 대개 결합도가 높다는 말이었습니다. 커버리지를 올리는 방법은 아니고, 테스트를 통해 시간·랜덤·IO 의존을 찾아내 설계를 고쳤던 과정과 그렇게 만든 테스트를 나중에도 읽히게 유지한 방식을 적어 둡니다.

먼저 짚고 가기
결합도Coupling
한 모듈이 다른 모듈의 내부를 얼마나 알고 있는지, 그래서 저쪽이 바뀔 때 이쪽도 같이 고쳐야 할 확률이 얼마나 되는지를 가리키는 설계 지표입니다. 응집도(cohesion)와 짝을 이루는 구조적 설계 용어로, 협력자를 직접 new 하거나 LocalDate.now() 같은 정적 메서드를 부르는 순간 그 선택이 코드에 박혀 호출자가 대체할 수 없게 됩니다. 결합도는 0으로 만드는 것이 목표가 아니라 — 모듈은 어차피 협력해야 합니다 — 어느 방향으로 어떤 형태로 걸지 고르는 문제입니다. 이 글은 "테스트가 어렵다"를 곧 "결합도가 높다"는 신호로 읽습니다.
의존성 주입Dependency Injection
객체가 필요한 협력자를 스스로 new 하거나 정적 팩토리로 찾아오지 않고, 생성자 인자·세터·메서드 파라미터로 밖에서 건네받게 하는 패턴입니다. 제어의 역전(IoC)의 한 형태로, 협력자를 고르는 결정이 호출자 쪽으로 올라가기 때문에 같은 자리에 운영에서는 시스템 시계를, 테스트에서는 Clock.fixed(...)를 끼울 수 있습니다. 다만 대상이 교체 가능한 타입일 때만 성립하고, 주입할 것이 생성자를 타고 도메인 깊숙이 번지면 주입 자체가 새로운 결합이 됩니다.
비결정성Non-determinism
같은 인자와 같은 초기 상태로 실행해도 결과가 달라질 수 있는 성질입니다. 원인은 대개 시그니처에 드러나지 않는 암묵적 입력(시스템 시계, 난수 생성기, 전역 상태)이거나 DB·네트워크처럼 프로세스 밖에서 오는 응답입니다. 테스트는 "이 입력이면 이 출력"이라는 대응을 못 박는 일이라, 실제 입력의 일부가 시그니처 밖에 있으면 기대값을 정할 수 없습니다. 비결정성을 아예 없앨 수는 없고(언젠가는 실제 시각을 읽고 실제로 저장해야 합니다), 해석하는 지점을 경계 한 곳으로 모으는 것이 이 글이 쓰는 전략입니다.
순수 함수Pure Function
같은 인자에 항상 같은 값을 반환하고(결정적), 바깥에서 관찰되는 부수효과 — 외부 상태 변경, IO, 인자 자체의 변형 — 가 없는 함수입니다. 두 조건이 함께 성립하면 호출을 그 반환값으로 바꿔 써도 프로그램의 의미가 달라지지 않습니다(참조 투명성). 검증에 필요한 것이 인자와 반환값뿐이라 DB나 네트워크 없이 값 비교만으로 테스트가 끝나고, 호출 순서나 횟수에도 영향받지 않습니다. 함수 안에서 지역 변수를 바꾸는 것은 밖에서 관찰되지 않으니 순수성을 깨지 않지만, 프로그램 어딘가는 반드시 저장하고 응답해야 하므로 전부를 순수하게 만들 수는 없습니다.
목 · 스텁Mock · Stub
둘 다 실제 협력자 자리에 끼워 넣는 테스트 대역(test double)이지만 역할이 다릅니다. 스텁은 호출에 미리 정한 값을 돌려주어 대상에게 들어가는 간접 입력을 통제하는 대역이고, 목은 기대한 호출이 기대한 인자로 일어났는지까지 단언하는 대역이라 검증 대상이 반환값이 아니라 상호작용입니다. 그래서 목으로 검증하면 협력자를 부르는 방식이 테스트에 고정되어, 동작이 그대로여도 구현을 바꾸면 실패합니다. 같은 jest.fn()이라도 반환값만 정해 두면 스텁이고 toHaveBeenCalledWith로 단언하면 목이라, 둘을 가르는 것은 도구가 아니라 검증 방식입니다.

테스트하기 어려운 코드는 설계가 보내는 신호입니다

테스트를 짠다는 건 코드의 입력을 제가 통제하고 출력을 제가 관찰하는 일이었습니다. 그런데 어떤 코드는 손을 대도 통제가 되지 않았습니다. 함수 안에서 스스로 시계를 보고, 전역 상태를 읽고, DB에 곧장 쓰기 때문입니다. 통제할 수 없는 입력이 있다는 건 취향의 문제가 아니라 함수의 실제 입력이 시그니처에 다 드러나 있지 않다는 증거였습니다. 정산 로직을 리팩토링하기 전에 제가 보던 코드가 그런 모양이었습니다.

// Before — 이 코드는 "오늘"을 스스로 안다
public class SettlementService {

  public void settle(Order order) {
    LocalDate today = LocalDate.now();          // 숨은 입력 1: 시간
    FeePolicy policy = FeePolicy.current();     // 숨은 입력 2: 전역 상태
    BigDecimal fee = policy.feeOf(order, today);
    repository.save(new Settlement(order, fee, today)); // 숨은 출력: IO
  }
}

이 코드에서 주말 수수료 분기를 검증하려면 토요일까지 기다리거나 시스템 시계를 조작하는 방법밖에 없었습니다. 반환값도 없어서 결과를 보려면 DB를 다시 읽어야 했습니다. 당시엔 테스트가 어렵다고만 생각했는데, 실제로는 함수가 자신의 입력과 출력을 숨기고 있던 것이었습니다. "테스트를 먼저 짜라"는 말이 실무에서 왜 나오는지도 이때 이해했습니다. 테스트를 먼저 생각하면 숨어 있던 의존이 먼저 드러나기 때문입니다.

시간·랜덤·IO — 비결정성을 경계로 밀어냅니다

제 경우 테스트를 막은 것은 대부분 시간, 랜덤, IO 이 셋이었습니다. 공통점은 비결정성으로, 같은 코드를 두 번 실행해도 같은 결과가 보장되지 않는다는 점입니다. 결정적이지 않은 함수는 검증할 수 없고, 검증할 수 없는 코드는 명세가 없는 것과 같습니다. 그래서 방법은 비결정성을 함수 밖, 시스템의 경계로 밀어내는 것 하나로 모였습니다. 그중 손이 가장 덜 가는 도구가 의존성 주입이었습니다.

// After — "오늘"은 밖에서 들어오고, 저장은 호출자의 일이 된다
public class SettlementService {
  private final Clock clock;
  private final FeePolicy feePolicy;

  public SettlementService(Clock clock, FeePolicy feePolicy) {
    this.clock = clock; this.feePolicy = feePolicy;
  }

  public Settlement settle(Order order) {
    LocalDate today = LocalDate.now(clock);
    return Settlement.of(order, feePolicy.feeOf(order, today), today);
  }
}

// JUnit — 토요일을 기다릴 필요가 없다
@Test
void 주말_주문에는_주말_수수료율이_적용된다() {
  Clock saturday = Clock.fixed(Instant.parse("2026-07-18T00:00:00Z"), SEOUL);
  SettlementService service = new SettlementService(saturday, defaultPolicy());

  Settlement result = service.settle(orderOf(10_000));

  assertThat(result.feeRate()).isEqualTo(WEEKEND_RATE);
}

바뀐 것은 세 가지였습니다. 시간은 Clock으로 주입받고, 정책은 인터페이스 뒤로 갔고, save가 빠진 자리에 결과 반환이 들어왔습니다. 계산과 저장을 분리하면 계산은 순수 함수가 되어 DB 없이 검증되고, 저장은 얇아져서 통합 테스트 몇 개로 덮을 수 있습니다. 정산 시스템을 Template Method와 Strategy로 리팩토링하면서 이 차이를 체감했습니다. 요율 정책이 Strategy로 분리되자 정책과 케이스의 조합을 순수 단위 테스트로 다 깔 수 있었고, 그러고 나니 새 정책을 추가하는 일도 같이 수월해졌습니다. 테스트 가능성과 확장 가능성이 한 설계의 양면이라는 말을 그때 납득했습니다.

다만 이 방법에도 선을 그어야 할 지점이 있었습니다. 주입이 편하다고 모든 곳에 Clock을 넣다 보면, 도메인 깊숙한 객체까지 시계를 끌고 가느라 생성자들이 시계 배달 통로가 되고 주입 자체가 새로운 결합이 됩니다. 지금은 유스케이스 진입점(컨트롤러·스케줄러)에서 now를 한 번만 찍어 값으로 내려보내는 쪽을 씁니다. 그러면 도메인 함수는 주입이 필요 없는 settle(order, today)가 됩니다. 정리하면 상위 규칙은 "주입하라"가 아니라 비결정성을 경계에서 한 번만 해석하라는 쪽이고, 주입은 그 수단 중 하나였습니다.

*

NestJS에서 Jest의 jest.useFakeTimers()는 전역 시계를 조작하는 방식이라, 디바운스·재시도 지연처럼 흐르는 시간을 검증할 때만 쓰는 편이 안전했습니다. 도메인이 "오늘"을 아는 문제는 가짜 타이머 대신 주입이나 파라미터로 푸는 쪽이 낫습니다. 전역 조작은 같은 파일의 test.concurrent나 실제 타이머에 기대는 라이브러리(HTTP 타임아웃·커넥션 유지)와 얽히면 그때부터 새기 시작합니다.

준비-실행-검증 — 테스트는 실행되는 횟수보다 읽히는 횟수가 많습니다

테스트를 실행하는 건 CI지만 읽는 건 사람이었습니다. 6개월 뒤의 저는 요구사항 문서보다 테스트를 먼저 열어서 이 시스템이 약속한 동작을 확인하곤 했습니다. 그래서 테스트는 준비(Arrange) → 실행(Act) → 검증(Assert)의 리듬이 눈에 보이게 두고, 이름은 완결된 문장으로 적어 두었습니다. SafeIn 백엔드를 혼자 설계할 때는 리뷰어가 없어서, 테스트 이름을 문장으로 쓰는 습관이 코드 리뷰를 대신하는 역할을 했습니다. 이름만 위에서 아래로 훑어도 명세서처럼 읽히게 하려는 의도였습니다.

// Jest — 이름만 모아 읽으면 명세가 되게
describe('CouponService.apply', () => {
  it('만료된 쿠폰이면 ExpiredCouponError를 던진다', async () => {
    // 준비 — 이 테스트의 조건만 눈에 띄게
    const coupon  = couponFixture({ expiresAt: daysAgo(1) });
    const service = new CouponService(fixedClock('2026-07-23'));

    // 실행 — 한 줄
    const apply = () => service.apply(anOrder(), coupon);

    // 검증 — 한 관심사
    await expect(apply).rejects.toThrow(ExpiredCouponError);
  });
});

쓰다 보니 이 리듬은 보기 좋으라고 지키는 형식이 아니라 진단 도구에 가까웠습니다. 실행이 두 줄 이상이면 대상 함수가 두 가지 일을 하고 있다는 신호이고(함수 편에서 다룬 "한 가지 일"이 여기서 다시 나옵니다), 검증이 서로 다른 관심사를 여러 개 확인하고 있으면 테스트를 두 개로 나눠야 한다는 신호였습니다. 준비가 화면을 넘길 만큼 길다면 대상 객체의 생성이 무겁다는 뜻이라, 그것도 설계로 돌아가 볼 지점이었습니다. AAA가 안 지켜지는 테스트를 만나면 저는 대개 테스트보다 대상 코드를 먼저 의심합니다.

목 남용 — 테스트가 구현을 못 박을 때

제가 테스트를 두는 이유는 동작을 보존한 채 구현을 바꿀 자유, 그러니까 리팩토링을 할 수 있게 하려는 것이었습니다. 그런데 목(mock)으로 호출 여부와 인자를 일일이 검증하는 테스트는 동작이 아니라 구현의 현재 모습을 고정합니다. 레거시 UseCase를 개선할 때 이걸 겪었습니다. 목이 겹겹이 쌓인 테스트들이 리팩토링 첫 커밋에서 전부 실패했는데, 동작은 하나도 바뀌지 않은 커밋이었습니다. 안전망으로 두었던 테스트가 오히려 손발을 묶고 있던 셈입니다.

// Bad — 결과가 아니라 "어떻게"를 검증한다
it('정산을 계산한다', async () => {
  await service.settle(order);

  expect(feePolicy.feeOf).toHaveBeenCalledTimes(1);
  expect(feePolicy.feeOf).toHaveBeenCalledWith(order, expect.anything());
  expect(repo.save).toHaveBeenCalledWith(
    expect.objectContaining({ fee: 300 })
  );
});
// 내부에 캐시를 넣거나 호출 순서를 바꾸면 — 동작은 같아도 — 깨진다

// Good — 검증은 결과 상태로
it('1만 원 카드 주문의 수수료는 300원이다', async () => {
  const result = await service.settle(cardOrderOf(10_000));

  expect(result.fee).toBe(300);
});

상태로 검증하는 편이 나은 이유는 리팩토링의 정의가 "관찰 가능한 동작을 보존하는 변경"이기 때문입니다. 반환값과 상태는 관찰 가능한 동작이지만, 내부 협력자를 몇 번 불렀는지는 관찰 가능한 동작이 아니라 구현 세부입니다. 구현 세부를 검증하는 테스트는 정의상 리팩토링마다 깨질 수밖에 없었습니다.

그렇다고 "목 금지"를 새 규칙으로 삼으면 반대쪽에서 막힙니다. 결제 게이트웨이, SMTP, 남의 서버처럼 프로세스 경계 밖에 있는 것들은 목이나 스텁으로 두는 편이 맞습니다. 실물을 부를 수 없고 불러서도 안 되는 대상이기 때문입니다. DDD로 크롤링 파이프라인을 만들 때는 대상 사이트의 HTML을 픽스처 파일로 떠서 스텁으로 썼습니다. 사이트가 바뀌면 픽스처만 갱신하면 되고, 파서 테스트는 네트워크 없이 밀리초 단위로 돌았습니다. 지금은 소유한 코드는 실물로, 소유하지 않은 경계는 목으로 정도의 기준을 두고 씁니다. 목을 편의 도구보다는 경계 표시선으로 생각하는 편입니다.

*

한 테스트에 toHaveBeenCalledWith가 세 개 이상 보이면, 그건 명세라기보다 구현의 스냅샷에 가깝습니다. 리팩토링 때마다 깨지기만 한다면 실제로 지켜 주는 게 없다는 뜻이라, 상태 검증으로 바꾸거나 그럴 수 없으면 지우는 편이 낫습니다.

픽스처에도 의도를 — 데이터가 문장이 되게

테스트 본문을 다듬어 놓고도 준비 단계의 데이터가 불투명해서 읽히지 않는 경우가 있었습니다. 인자 일곱 개짜리 생성자 호출을 보면 이 테스트의 주인공이 어느 값인지 알아내는 일이 읽는 사람 몫으로 넘어갑니다.

// Bad — 어떤 값이 이 테스트의 주인공인지 알 수 없다
const order = new Order('u-1', 10_000, 'CARD', true, null, 3);

// Good — 결론에 기여하는 값만 앞으로, 나머지는 기본값 뒤로
const order = anOrder().amount(10_000).paidBy('CARD').build();

// 더 좋은 것 — 의도가 이름이 된 픽스처
const weekendOrder  = orderOn('2026-07-18');        // 토요일
const expiredCoupon = couponExpiredSince(daysAgo(1));

그래서 결론에 기여하는 값만 보이게 두고 나머지는 빌더의 기본값 뒤로 숨기는 방식으로 정리했습니다. 화면에 보이는 값은 읽는 사람에게 "이 값이 결과에 영향을 주는가"라는 질문을 하나씩 만들기 때문입니다. 무관한 값이 다섯 개 보이면 헛질문도 다섯 번 생깁니다. 반대로 weekendOrder나 expiredCoupon처럼 의도가 이름에 들어간 픽스처는 그 질문 자체를 없앱니다. 덤으로 필드가 추가돼도 빌더의 기본값 한 곳만 고치면 나머지 테스트는 그대로 살아남았습니다.

게임에서 배운 것 — 자동 플레이테스트 20종

게임 프로젝트에서 자동 플레이테스트 20종을 만들어 돌린 적이 있습니다. 스폰, 이동, 전투, 보상 같은 시나리오를 봇이 수행하고 결과를 검증하는 방식이었는데, 여기서도 앞의 이야기가 그대로 반복됐습니다. 랜덤이 낀 전투는 시드를 고정해야 재현이 됐고(랜덤 주입의 다른 얼굴이었습니다), 시나리오 이름은 실패 로그만 보고도 무엇이 깨졌는지 알 수 있게 문장으로 지어야 했습니다. 남은 것 중에는 심리적인 부분이 컸습니다. 20종이 매번 돌아 준다는 믿음이 생기고 나서는 밸런스 수치를 바꾸는 일이 덜 부담스러웠습니다. 서버 코드에서도 비슷했습니다. 테스트에서 제가 가장 많이 얻은 것은 버그 검출보다 변경을 덜 두려워하게 된 쪽이었습니다.

정리 — 오늘 가져갈 규칙

돌아보면 테스트는 품질 검사 도구이기 전에 설계 도구였습니다. 테스트가 괴로울 때 고칠 대상은 대부분 테스트가 아니라 코드 쪽이었습니다. 지금 지키고 있는 것들을 적어 둡니다.

  • 테스트가 안 짜이면 설계를 의심합니다. 통제할 수 없는 입력은 시그니처에 숨은 의존이 있다는 증거였습니다.
  • 시간·랜덤·IO는 경계로 밀어냅니다. 주입하거나, 가능하면 진입점에서 한 번 해석해 값으로 내려보냅니다.
  • 계산과 저장을 분리합니다. 계산은 순수 함수로 단위 테스트하고, 저장은 얇게 만들어 통합 테스트 몇 개로 덮습니다.
  • 준비-실행-검증을 지킵니다. 실행은 한 줄, 검증은 한 관심사로 두고, 길어지면 테스트가 아니라 대상 함수를 쪼갭니다.
  • 이름은 문장으로 씁니다. 테스트 이름만 모아 읽어도 명세서처럼 읽히게 합니다.
  • 소유한 코드는 실물로, 소유하지 않은 프로세스 경계만 목으로 둡니다. 호출 검증이 쌓이면 구현을 못 박은 상태입니다.
  • 픽스처는 결론에 기여하는 값만 보이게 합니다. 나머지는 빌더 기본값 뒤로 숨깁니다.
