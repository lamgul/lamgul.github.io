보안: CORS와 CSRF — 헷갈리는 둘을 정확히

2022년 7월• 약 11분• CORS · CSRF · SameSite

CORS 에러가 뜨면 손이 먼저 움직입니다. 스택오버플로를 뒤져 origin: '*'를 붙이고, 빨간 글씨가 사라지면 다음 일로 넘어가죠. 저도 오래 그랬습니다. 문제는 그 한 줄이 에러를 지운 게 아니라 방어를 지웠다는 것이었습니다. CORS와 CSRF는 이름도 비슷하고 늘 같이 등장하지만, 사실 정반대 방향을 봅니다. 이 글은 둘을 헷갈리지 않게 갈라 세우고, 미들웨어에서 무엇을 켜고 꺼야 하는지까지 정리합니다.

1. 같은 출처 정책은 하나가 아니다

CORS와 CSRF가 헷갈리는 건 이름 탓만이 아닙니다. 둘 다 같은 출처 정책(Same-Origin Policy) 위에 서 있기 때문입니다. 출처(origin)는 스킴·호스트·포트 세 가지로 정의되고, 셋 중 하나만 달라도 브라우저는 다른 출처로 봅니다. 경로(path)는 출처에 포함되지 않는다는 점이 자주 헷갈리는 지점입니다.

# 출처(origin) = 스킴 + 호스트 + 포트, 셋 중 하나만 달라도 다른 출처
https://app.example.com          # 기준
https://app.example.com/admin    # 같은 출처 (경로는 무관)
http://app.example.com           # 다른 출처 (스킴 http↔https)
https://api.example.com          # 다른 출처 (호스트 app↔api)
https://app.example.com:8443     # 다른 출처 (포트)

핵심은 같은 출처 정책이 무엇을 막느냐입니다. 이 정책은 다른 출처의 응답을 스크립트가 읽는 것을 막지, 요청이 나가는 것을 막지 않습니다. 브라우저는 다른 출처로 쿠키를 실은 POST도 얼마든지 보냅니다 — 다만 그 응답을 호출한 자바스크립트에게 넘겨주지 않을 뿐입니다. 이 '보내기는 되고 읽기는 안 되는' 비대칭이 CORS와 CSRF가 갈라서는 지점입니다.

그래서 둘은 정반대 방향을 봅니다. CORS는 이 읽기 제한을 의도적으로 푸는 쪽이고, CSRF는 보내기가 애초에 안 막힌다는 점을 악용하는 쪽입니다. 한쪽은 빗장을 풀고, 다른 쪽은 잠기지 않은 문을 노립니다. 이 문장 하나만 붙잡고 가면 나머지는 따라옵니다.

2. CORS는 방패가 아니라 빗장이다

CORS(Cross-Origin Resource Sharing)를 한 문장으로 줄이면 이렇습니다. 서버가 브라우저에게 "이 출처한테는 내 응답을 보여줘도 된다"고 허락하는 프로토콜. 잠금장치가 아니라 허가증입니다. 그래서 CORS 설정은 문을 잠그는 일이 아니라, 특정 손님에게만 문을 열어 주는 일에 가깝습니다.

요청은 두 부류로 나뉩니다. GET이나 조건을 만족하는 단순한 POST 같은 단순 요청은 곧장 서버로 갑니다. 반면 application/json 본문이나 커스텀 헤더, PUT·DELETE·PATCH가 붙으면 브라우저가 본 요청 전에 프리플라이트(preflight)라는 OPTIONS 요청을 먼저 보내 허락을 구합니다.

# 브라우저가 먼저 물어본다 (preflight) — 실제 요청 전에 자동 발생
OPTIONS /api/reports HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method: PATCH
Access-Control-Request-Headers: authorization, content-type

# 서버의 허가 응답 — 이 조합을 허용한다고 답해야 본 요청이 나간다
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, PATCH, DELETE
Access-Control-Allow-Headers: authorization, content-type
Access-Control-Max-Age: 600

여기서 가장 흔한 오해를 짚어야 합니다. CORS를 강제하는 주체는 서버가 아니라 브라우저입니다. curl이나 Postman, 다른 서버는 CORS 헤더를 그냥 무시하고 응답을 다 읽습니다. 즉 CORS는 API의 접근 제어가 아니라, 브라우저 사용자를 보호하는 장치입니다. 인증·인가는 여전히 별도로 해야 합니다 — CORS를 열어 뒀다고 API가 뚫린 게 아니고, 닫아 뒀다고 안전한 것도 아닙니다.

3. 자격증명을 실으면 별표가 막힌다

쿠키나 Authorization 헤더 같은 자격증명을 함께 실으면(credentials: 'include') 규칙이 갑자기 엄격해집니다. 브라우저는 자격증명이 붙은 요청에 대해 Access-Control-Allow-Origin: *를 거부합니다. 와일드카드 대신 구체적인 출처 하나를 돌려주고, Access-Control-Allow-Credentials: true를 붙이며, 캐시가 섞이지 않게 Vary: Origin까지 넣어야 합니다.

문제는 여러 출처를 지원하려고 들어온 Origin을 그대로 되돌려주는 코드입니다. 화이트리스트 없이 반사하면, 그건 사실상 '아무 출처나 자격증명과 함께 허용'이 되어 스펙이 금지한 *+자격증명을 손으로 재현한 꼴입니다. 이 순간 어떤 사이트든 로그인한 사용자를 흉내 내 인증된 응답을 읽어 갈 수 있습니다. 전형적인 CORS 설정 취약점입니다.

// 예전의 나 — 에러만 지우면 된다고 생각했다
app.enableCors({ origin: '*', credentials: true });
// 브라우저는 이 조합을 거부한다. '되게' 하려고 origin을 요청 그대로 반사하면
// 그건 '아무 출처나 자격증명과 함께 허용'이 되어 오히려 더 위험하다.

// 지금 — 출처를 화이트리스트로 고정하고, 자격증명은 명시적으로
app.enableCors({
  origin: ['https://app.example.com', 'https://admin.example.com'],
  credentials: true,
  methods: ['GET', 'POST', 'PATCH', 'DELETE'],
  allowedHeaders: ['Authorization', 'Content-Type'],
  maxAge: 600,
});

부끄럽지만 예전 공개 레포에서 제가 이 실수를 했습니다. 쿠키 인증을 쓰면서 CORS 에러를 지우려고 출처를 열어 뒀고, 브라우저에서 '되니까' 문제없다고 여겼습니다. 자격증명 동반 요청의 규칙을 손으로 따라가 보고 나서야, 그게 방어를 지운 설정이었다는 걸 알았습니다.

!

함정 — origin: '*'는 자격증명과 함께 쓸 수 없다는 규칙을, '들어온 Origin을 그대로 되돌려주기'로 우회하는 코드를 조심하세요. 화이트리스트 없이 반사하면 모든 사이트가 로그인한 사용자를 흉내 내 인증된 응답을 읽을 수 있습니다. 허용 목록은 배열이나 함수로 명시하고, 목록에 없는 출처엔 CORS 헤더를 아예 붙이지 마세요.

4. CSRF는 쿠키 자동 첨부를 노린다

CSRF(Cross-Site Request Forgery)는 판을 뒤집습니다. 응답을 읽으려 하지 않습니다 — 읽을 필요가 없으니까요. 대신 브라우저가 내 도메인으로 가는 요청에 쿠키를 자동으로 실어 준다는 성질을 악용합니다. 누가 그 요청을 유발했든 상관없이 말이죠. 같은 출처 정책은 응답을 가려 줄 뿐, 서버에서 일어난 상태 변경은 이미 끝난 뒤입니다.

고전적인 시나리오는 이렇습니다. 피해자가 bank.example.com에 로그인한 상태(세션 쿠키 보유)로 evil.com을 방문합니다. 그 페이지엔 bank.example.com/transfer로 향하는 자동 제출 폼이 숨어 있습니다. 폼이 제출되면 브라우저는 세션 쿠키를 실어 보내고, 서버는 유효한 세션으로 보고 송금을 처리합니다.

<!-- evil.com에 심어 둔 자동 제출 폼 — 피해자의 쿠키가 자동으로 실린다 -->
<form action="https://bank.example.com/transfer" method="POST" id="f">
  <input type="hidden" name="to"     value="attacker">
  <input type="hidden" name="amount" value="1000000">
</form>
<script>document.getElementById('f').submit();</script>

왜 CORS가 여기서 구해 주지 못할까요. 폼 POST는 application/x-www-form-urlencoded 본문이라 단순 요청이고, 단순 요청엔 프리플라이트가 없습니다. 요청은 그대로 나가서 처리되고, CORS는 기껏해야 응답을 가려 줄 뿐인데 공격자는 애초에 응답을 원하지 않았습니다. "우리 API는 CORS를 설정했으니 CSRF는 괜찮다"가 성립하지 않는 이유입니다.

5. 토큰과 SameSite — 두 겹의 방어

막는 방법은 두 겹입니다. 첫째는 CSRF 토큰입니다. 서버가 예측 불가능한 토큰을 발급해 같은 출처 코드만 읽을 수 있는 곳에 심어 두고, 상태 변경 요청마다 그 토큰을 되돌려받아 검증합니다. 공격자는 이 토큰을 읽을 수 없습니다 — 같은 출처 정책이 유일하게 확실히 보장하는 게 바로 '교차 출처 읽기 차단'이니까요. 쿠키에 토큰을 두고 헤더로 같은 값을 함께 보내 일치를 확인하는 이중 제출(double-submit) 방식은 서버 상태 없이도 됩니다.

둘째는 SameSite 쿠키입니다. SameSite=Lax(요즘 브라우저 기본값)는 교차 사이트에서 날아온 POST 같은 하위요청에 쿠키를 빼고 보냅니다 — 위조된 교차 사이트 POST가 세션 쿠키를 잃으니 고전적 CSRF가 대부분 무력화됩니다. Strict는 아예 교차 사이트에 안 보내고(로그인 복귀 흐름이 깨질 수 있음), None은 항상 보내되 Secure가 필수입니다.

// 세션 쿠키: 브라우저 단에서 자동 첨부를 1차로 통제한다
res.cookie('sid', sessionId, {
  httpOnly: true,   // JS에서 못 읽음 → XSS 탈취를 어렵게
  secure:   true,   // HTTPS에서만 전송
  sameSite: 'lax',  // 교차 사이트 하위요청엔 쿠키를 빼고 보낸다
});

// 상태 변경엔 토큰을 한 겹 더 — 공격자는 이 값을 읽을 수 없다(SOP)
@Post('transfer')
transfer(@Body() dto: TransferDto, @Req() req) {
  if (req.headers['x-csrf-token'] !== req.cookies.csrf) {
    throw new ForbiddenException('CSRF token mismatch');  // double-submit 검증
  }
  return this.account.transfer(dto);
}

왜 두 겹을 다 두냐면, 어느 하나도 완전하지 않기 때문입니다. SameSite=Lax는 싸고 강력한 기본값이지만, 최상위 GET 내비게이션에는 쿠키가 실립니다 — 상태를 바꾸는 GET을 두면 Lax도 못 막으니, 상태 변경은 반드시 POST·PATCH 등으로 해야 합니다. 서드파티 임베드처럼 SameSite=None이 필요한 맥락도 있고, 그때는 토큰이 빈자리를 메웁니다. 다층 방어가 원칙입니다.

축을 하나 더 짚자면, 순수 API에서 자격증명을 쿠키 대신 Authorization: Bearer 헤더로 실으면 CSRF 표면 자체가 줄어듭니다. 헤더 토큰은 브라우저가 자동으로 붙여 주지 않으므로, 공격자의 위조 요청엔 자격증명이 실리지 않습니다. 앞 글에서 다룬 JWT 인증이 세션 쿠키보다 CSRF에 강한 이유가 이것입니다 — 물론 쿠키는 HttpOnly로 XSS 탈취를 막는 장점이 있어, 무엇이 낫다기보다 트레이드오프입니다.

6. The Tangled Web가 바꾼 시선

제가 오랫동안 CORS와 CSRF를 헷갈린 진짜 이유는, 하나의 같은 출처 정책을 찾고 있었기 때문입니다. Michal Zalewski의 The Tangled Web: A Guide to Securing Modern Web Applications(No Starch Press, 2011)가 그 습관을 깨 준 책입니다. 저자는 AFL 퍼저로 유명한 보안 연구자이고, 이 책은 브라우저 보안 모델을 정면으로 해부합니다.

책의 핵심 논지는 이렇습니다. 웹 플랫폼의 보안은 설계된 게 아니라 쌓여 온 것이라, 단 하나의 같은 출처 정책이란 존재하지 않는다 — DOM, 쿠키, XMLHttpRequest, 이미지와 프레임에 제각기 다르게 적용되는 느슨한 규칙들의 묶음일 뿐이라는 것입니다. 같은 단어가 구석마다 다르게 행동합니다. 이 지도를 머릿속에 넣고 나니 CORS와 CSRF가 깨끗하게 갈라졌습니다 — CORS는 fetch의 읽기 규칙을 협상하고, 쿠키는 자기만의 첨부 규칙을 따르며(그래서 쿠키 쪽 레버는 CORS가 아니라 SameSite입니다), 둘은 처음부터 다른 축이었습니다.

책에서 가져온 구체적 습관도 있습니다. 각 장은 "Security Engineering Cheat Sheet"로 끝나는데, 그 장의 결론을 '이렇게 하라'는 짧은 목록으로 압축해 줍니다. 저는 CORS·CSRF를 매번 원리로 다시 유도하려던 걸 그만두고 제 체크리스트를 적었습니다(그게 다음 절입니다). 이 플랫폼은 머릿속으로만 붙들기엔 너무 엉켜 있으니 규칙을 문서로 굳히라는 태도가, 제 경험과 정확히 맞았습니다.

정직하게 덧붙이면, 2011년 책이라 세부는 옮겨 갔습니다. SameSite=Lax 기본값, 와일드카드+자격증명 금지의 명문화, CSP 레벨 같은 건 이후의 변화입니다. 하지만 틀 — 같은 출처 정책을 패치워크로 보고, 방어를 '메커니즘 하나하나를 지도로 그리는 일'로 대하는 관점 — 은 완벽하게 나이 들었습니다. 저는 헤더 문법이 아니라 사고 모델을 얻으려 이 책을 읽었습니다.

*

Zalewski의 관점을 한 줄로 줄이면 이렇습니다. '같은 출처 정책'을 찾지 말고, 각 메커니즘이 무엇을 읽고 무엇을 보내는지를 따로 적어라. fetch의 읽기 규칙(CORS), 쿠키의 첨부 규칙(SameSite), 자격증명의 동반 규칙 — 축이 다른 세 가지를 하나의 스위치로 다루려 할 때 구멍이 생깁니다.

7. 정리 — 무엇을 켜고 끌 것인가

결국 두 축을 떼어 놓으면 전부 한 페이지에 들어옵니다. 미들웨어를 만질 때 실제로 확인하는 순서대로 적습니다.

  • CORS는 보안이 아니다 — 브라우저의 읽기 제한을 의도적으로 푸는 스위치. 인증·인가는 별개로 둔다.
  • origin: '*'와 자격증명은 공존 불가 — 출처는 화이트리스트로 고정하고, 목록에 없으면 CORS 헤더를 붙이지 않는다. 반사(reflect) 금지.
  • 자격증명 응답 3종 세트 — 구체적 Origin + Access-Control-Allow-Credentials: true + Vary: Origin.
  • CSRF는 CORS로 못 막는다 — 폼 POST는 단순 요청이라 프리플라이트가 없다.
  • 쿠키 세션이면 두 겹 — SameSite=Lax(기본) + CSRF 토큰(동기화/이중 제출)을 함께.
  • 순수 API라면 헤더 토큰 — 자격증명을 쿠키 대신 Authorization: Bearer로. 자동 첨부가 없어 CSRF 표면이 준다.
  • 한 곳에서만 일관되게 — 같은 CORS 헤더를 엣지(NGINX)와 앱(NestJS) 양쪽에서 중복으로 붙이면 값이 겹쳐 되레 깨진다.
# NGINX 엣지 — 허용 출처만 화이트리스트로 매핑(목록에 없으면 빈 값 → 헤더 안 붙음)
map $http_origin $cors_origin {
  default                     "";
  "https://app.example.com"   $http_origin;
  "https://admin.example.com" $http_origin;
}

server {
  location /api/ {
    # 프리플라이트(OPTIONS)는 엣지에서 바로 끊는다
    if ($request_method = OPTIONS) {
      add_header Access-Control-Allow-Origin      $cors_origin always;
      add_header Access-Control-Allow-Methods     "GET, POST, PATCH, DELETE" always;
      add_header Access-Control-Allow-Headers     "Authorization, Content-Type" always;
      add_header Access-Control-Allow-Credentials "true" always;
      add_header Access-Control-Max-Age           600 always;
      add_header Vary Origin always;
      return 204;
    }
    add_header Access-Control-Allow-Origin      $cors_origin always;
    add_header Access-Control-Allow-Credentials "true" always;
    add_header Vary Origin always;              # 출처별로 캐시가 갈리도록
    proxy_pass http://app_upstream;
  }
}

예전의 저는 origin: '*' 한 줄로 에러를 지웠지만, 실은 방어를 지우고 있었습니다. CORS와 CSRF를 한 축으로 뭉뚱그린 탓이었죠. 둘을 '읽기를 푸는 쪽'과 '보내기를 막는 쪽'으로 갈라 세우고 나면, 미들웨어 설정은 더 이상 주술이 아니라 체크리스트가 됩니다. 여기에 보안 헤더 하드닝까지 이으면, 서버 앞단의 방어가 하나의 그림으로 맞춰집니다.
