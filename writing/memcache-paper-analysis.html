논문 분석: Scaling Memcache — 캐시 한 층의 전쟁

2023년 12월• Memcache · 캐시 · 논문

Redis로 목록 캐시를 운영하면서 캐시는 넣는 것보다 지우는 게 어렵다는 걸 배웠습니다. 그 무렵 읽은 논문이 Scaling Memcache at Facebook입니다. Facebook이 memcached라는 단순한 키-값 저장소를 세계 최대의 캐시 계층으로 키우며 겪은 일을 담았는데, cache-aside 패턴의 교과서 설명이 끝나는 지점에서 시작합니다. 만료된 인기 키에 요청이 몰릴 때, 처리 순서가 꼬여 낡은 값이 캐시에 남을 때, DB가 바뀐 것을 수천 대의 캐시 서버에 알려야 할 때 무슨 일이 벌어지는지를 하나씩 짚습니다. 제가 Redis 한 대에서 겪던 문제의 확대판이라, 읽는 동안 남 이야기 같지 않았습니다.

출처

Scaling Memcache at Facebook

Rajesh Nishtala 외 · NSDI 2013

먼저 짚고 가기
캐시 어사이드Look-Aside Cache
애플리케이션이 캐시 적재를 직접 책임지는 캐시 배치 방식이고, 논문의 표현으로는 demand-filled look-aside cache입니다. 읽기는 get으로 시작해 미스가 나면 애플리케이션이 DB를 조회하고 그 값을 set으로 넣으며, 캐시 서버는 DB의 존재도 스키마도 모른 채 키-값 저장만 합니다. 캐시 계층이 적재 로직을 갖는 read-through와 달리 미스 처리와 무효화 순서가 전부 애플리케이션 코드에 남고, 이 글에서 다루는 문제 대부분이 그 남겨진 책임에서 시작합니다.
썬더링 허드Thundering Herd
키 하나가 비는 순간 그 키를 읽던 동시 요청이 전부 미스를 맞고 각자 DB로 내려가, 같은 값을 중복 조회하고 중복으로 set 하는 부하 폭주 현상입니다. 논문 3.2.1절은 읽기 빈도가 높으면서 갱신(=삭제)이 잦은 키에서 이 조합이 나타난다고 지적합니다. 원인은 캐시 용량이 아니라 값이 비어 있는 짧은 구간의 동시성이므로, 캐시 서버를 늘리거나 메모리를 키워도 사라지지 않습니다.
리스Lease
미스가 난 클라이언트에게 memcached가 발급하는 64비트 토큰으로, 캐시를 채울 권한을 한 명으로 좁히는 장치입니다. 한 키당 기본 10초에 한 번만 발급되어 토큰 없이 미스한 요청은 잠시 뒤 다시 읽으라는 응답을 받고, 값을 쓰는 lease-set은 토큰이 아직 유효할 때만 받아들여집니다. 그 사이 같은 키에 delete가 도착하면 발급된 토큰이 무효화되므로 낡은 값을 든 뒤늦은 lease-set은 거부되는데, 토큰 하나가 썬더링 허드와 스테일 셋을 함께 막는 이유가 여기에 있습니다. 상호 배제를 보장하는 분산 락이 아니라 늦게 도착한 쓰기를 걸러 내는 검증 표식에 가깝습니다.
스테일 셋Stale Set
캐시에 마지막으로 남은 값이 DB의 최신 값이 아니게 되는 쓰기 사고입니다. 클라이언트가 DB에서 값을 읽은 시점과 그 값을 set 하는 시점 사이에 다른 쓰기가 DB를 갱신하고 무효화를 보내면, 뒤늦게 도착한 set이 방금 지워진 자리에 낡은 값을 도로 심습니다. 캐시는 자기가 든 값이 낡았다는 걸 스스로 알 수 없어서 TTL이 끝나거나 다음 무효화가 올 때까지 그대로 남고, 그래서 논문은 set의 수락 여부를 값이 아니라 lease 토큰의 유효성으로 판정합니다.
무효화Invalidation
DB가 바뀐 뒤 캐시에 남은 옛 항목을 제거해 두 저장소가 어긋난 구간을 닫는 연산입니다. Facebook은 이를 set이 아니라 delete로만 수행하는데, 삭제는 멱등이라 중복 배달·재전송·순서 역전이 있어도 "캐시에 없음"이라는 같은 상태로 수렴하기 때문입니다. 리전 규모에서는 웹서버가 아니라 mcsqueal이 각 DB의 커밋 로그에서 커밋된 SQL문에 함께 심어 둔 키를 뽑아 delete를 만들고, mcrouter를 거쳐 담당 memcached 서버로 배달합니다. 무효화는 캐시를 비울 뿐 새 값을 채워 주지는 않으므로 반드시 다음 읽기의 미스를 동반합니다.

1. 왜 set이 아니라 delete인가 — look-aside의 규칙

논문 2장이 그리는 구조는 우리가 아는 cache-aside 그대로였습니다. 읽기는 get → 미스면 DB 조회 → set으로 채우고(demand-filled), 쓰기는 DB를 갱신한 뒤 캐시를 손봅니다. 다른 것은 규모뿐인데, 논문 3.1절에 따르면 인기 페이지 한 장을 띄우는 데 평균 521개의 서로 다른 아이템을 memcache에서 읽는다고 합니다. 페이지 한 장을 그리는 데 캐시 요청이 수백 개씩 나가는 셈입니다.

# look-aside 캐시의 읽기·쓰기 경로 (논문 2장)
def read(key):
    value = memcache.get(key)
    if value is None:               # miss
        value = db.query(key)
        memcache.set(key, value)      # 채우는 책임은 읽는 쪽에 있다
    return value

def write(key, value):
    db.update(key, value)
    memcache.delete(key)              # set이 아니라 delete — 삭제는 멱등이다

논문이 처음으로 분명하게 밝히는 선택은 쓰기 후에 캐시를 갱신(set)하지 않고 삭제(delete)한다는 점입니다. 삭제가 멱등이기 때문입니다(논문 2장). 두 개의 쓰기가 경합할 때 set은 어느 쪽이 마지막에 도착하느냐에 따라 낡은 값이 이길 수 있지만, delete는 몇 번을 어떤 순서로 실행해도 "캐시에 없음"이라는 같은 결과로 수렴합니다. 무효화 메시지가 중복 배달되거나 재전송되어도 안전하다는 뜻이고, 이 성질이 뒤에 나올 무효화 파이프라인 전체를 받치고 있습니다. 저도 Redis에서 목록 캐시를 갱신으로 유지하려다 포기하고 삭제로 돌아섰던 적이 있어서, 이 문단은 밑줄을 그으며 읽었습니다.

2. 만료의 순간 — thundering herd

다만 delete 기반 무효화에는 대가가 따릅니다. 삭제된 인기 키를 다시 읽을 때 캐시에는 아무것도 없고, 초당 수만 번 읽히는 키라면 그 짧은 공백 동안 미스를 맞은 요청이 전부 DB로 갑니다. 읽기가 많은 키일수록, 또 쓰기(=삭제)가 잦은 키일수록 이 공백이 자주 열립니다. 논문 3.2.1절도 같은 조합을 지적하는데, thundering herd는 많이 읽히면서 자주 갱신되는 키에서 생깁니다.

이 현상이 문제가 되는 건 평소의 DB 용량이 캐시가 걸러 준 만큼만 감당하도록 잡혀 있기 때문입니다. 히트율 99%의 캐시 뒤에 있는 DB는 트래픽의 1%만 보다가, 인기 키 하나가 지워지면 그 키의 트래픽 100%를 받게 됩니다. 캐시가 평소에 잘 막아 줄수록 미스가 났을 때의 낙차도 커집니다. 캐시 장애가 그대로 DB 장애로 이어지고, DB가 느려지면 미스가 더 오래 지속되어 herd가 더 커지는 되먹임도 걸립니다.

3. lease — 토큰 하나로 두 문제를 잡다

논문 3.2.1절이 내놓는 해법은 단순한 편입니다. 미스가 난 키에 대해 memcached가 64비트 lease 토큰을 발급하되, 키당 기본 10초에 한 번만 발급합니다. 토큰을 받은 클라이언트만 DB에서 값을 읽어 와 채울 자격이 있고, 그 10초 안에 같은 키를 미스한 나머지 요청들은 곧 채워질 테니 잠깐 기다렸다 다시 읽으라는 응답을 받습니다. 토큰 보유자가 대개 수 밀리초 안에 값을 채우기 때문에, 기다린 쪽은 재시도에서 캐시 히트를 얻습니다. 그래서 DB로 가는 요청은 1개로 줄어듭니다.

# lease-get / lease-set 의 뼈대 (논문 §3.2.1)
def read_with_lease(key):
    value, token = memcache.lease_get(key)
    if value is not None:
        return value                     # hit — 평소엔 대부분 여기서 끝난다
    if token is None:                    # hot miss: 토큰은 이미 남이 가져갔다
        sleep_briefly()
        return read_with_lease(key)      # 잠깐 기다리면 대부분 히트가 된다
    value = db.query(key)                # 토큰 보유자만 DB로 간다
    memcache.lease_set(key, value, token)  # 토큰이 아직 유효할 때만 저장된다
    return value

읽으면서 인상 깊었던 건 같은 토큰이 stale set까지 함께 막아 준다는 점이었습니다. 그 사이에 누군가 이 키에 delete를 보내면, 즉 DB가 또 바뀌었으면 memcached는 발급했던 토큰을 무효로 만듭니다. 낡은 값을 들고 뒤늦게 도착한 lease_set은 토큰 검증에서 거부됩니다. 내가 DB에서 읽은 시점과 캐시에 쓰는 시점 사이에 값이 바뀌었는지를 캐시 서버가 토큰의 유효 여부로 판정해 주는 셈입니다. 검증 없이 낙관적으로 set 하면서 생기던 경쟁 조건이 토큰이라는 버전 표식으로 정리됩니다.

효과는 논문이 숫자로 적어 두었습니다. 3.2.1절은 썬더링 허드에 특히 취약한 키들을 골라 일주일간 캐시 미스를 관찰했는데, lease가 없을 때 DB 쿼리 피크가 17K/s였던 것이 lease를 켜자 1.3K/s로 내려갔습니다. 피크 부하를 기준으로 DB를 증설하는 팀이라면 이 차이가 장비 대수로 그대로 환산됩니다.

*

Redis에는 lease-get이 없지만 같은 모양을 만들 수 있습니다. 미스가 나면 SET lock:key ... NX PX 3000으로 짧은 락을 선점한 쪽만 DB를 읽어 채우고, 실패한 쪽은 수십 ms 기다렸다 재시도하는 식입니다. 제가 Redis 무효화를 다룬 글에서 삭제 직후의 미스 폭주를 고민했던 자리에, 이 논문은 10년 전에 이미 같은 답을 적어 두고 있었습니다.

4. 무효화를 배달하는 법 — 커밋 로그에서 뽑는 delete

다음은 규모의 문제입니다. 클러스터에는 수천 대의 memcached 서버가 있고, 같은 키가 여러 클러스터에 복제되어 있습니다. DB의 한 행이 바뀌면 그 키를 들고 있는 모든 캐시 서버에 delete를 배달해야 하는데, 그 배달을 누가 맡느냐가 남습니다.

가장 단순한 답은 쓰기를 수행한 웹서버가 직접 delete를 뿌리는 것이고, 실제로 Facebook도 클러스터 안에서는 이렇게 합니다. 다만 리전 전체로 넓히면 논문 4.1절이 지적하는 두 가지 약점이 드러납니다. 웹서버는 무효화를 묶어 보내는 데 서툴러 패킷 오버헤드가 커지고, 설정 실수로 delete가 엉뚱한 곳으로 가면 되돌릴 방법이 없습니다. 논문은 과거에 이런 일이 memcache 인프라 전체의 롤링 재시작으로 이어지곤 했다고 적어 두었습니다. 그래서 4.1절은 배달의 출발점을 애플리케이션에서 DB 커밋 로그로 옮깁니다. mcsqueal이라는 무효화 데몬이 모든 DB에 붙어 자기 DB가 커밋한 SQL문을 들여다보고, 거기에 함께 심어 둔 memcache 키를 추출해 delete를 만듭니다. 이걸 낱개로 쏘면 패킷 수를 감당할 수 없으므로 mcsqueal이 묶음으로 포장해 각 프론트엔드 클러스터의 mcrouter로 보내고, mcrouter가 묶음을 풀어 키마다 담당 memcached 서버로 나눠 줍니다.

커밋 로그를 출발점으로 삼으면 두 가지가 따라옵니다. 첫째, 커밋된 변경만 무효화를 만들기 때문에 롤백된 트랜잭션이 캐시를 지우는 일이 없습니다. 둘째, 로그가 남아 있으므로 배달이 실패하면 다시 재생할 수 있습니다. 재생이 안전한 이유는 앞에서 본 delete의 멱등성에 있습니다. 같은 delete가 두 번 도착해도 아무 일도 일어나지 않습니다. 개별 기법보다도 이렇게 설계가 서로를 받쳐 주는 구조가 더 배울 점이라고 생각하며 읽었습니다.

*

이 패턴은 오늘날 CDC(Change Data Capture)라 부르는 것의 원형입니다. Debezium으로 binlog를 읽어 캐시를 지우거나 검색 인덱스를 맞추는 요즘의 구성은, mcsqueal이 하던 일에 이름과 범용 도구가 붙은 것에 가깝습니다. 정합성이 중요한 전파는 애플리케이션이 아니라 커밋 로그에서 시작한다는 원칙으로 적어 두었습니다.

5. 리전을 넘어서 — 복제 지연과 remote marker

마지막으로 지리적 거리라는 조건이 붙습니다. 논문 5장에서 Facebook은 마스터 리전 하나에 쓰기를 모으고, 다른 리전들은 MySQL 복제로 데이터를 받아 읽기를 처리합니다. 여기서 걸리는 것이 복제 지연입니다. 복제 리전의 사용자가 글을 고치면 쓰기는 마스터로 가는데, 자기 리전의 복제 DB에는 그 변경이 아직 도착하지 않았습니다. 이 상태에서 로컬 캐시만 지우면 다음 읽기가 낡은 복제 DB에서 옛 값을 읽어 캐시에 도로 채워 넣습니다. 사용자 입장에서는 방금 고친 글이 옛날 내용으로 돌아간 것처럼 보입니다.

논문이 제시하는 remote marker는, 이 키에 대해 우리 리전의 데이터가 낡았을 수 있다는 표식을 캐시 계층에 남겨 두는 방식입니다.

# 비마스터 리전에서의 쓰기와 읽기 (논문 5장, remote marker)
def write_from_replica_region(key, value):
    memcache.set(marker(key), 1)       # 1) 우리 리전에 표식을 세운다
    master_db.update(key, value)       # 2) 마스터에 쓴다 — SQL에 marker 삭제도 심는다
    memcache.delete(key)               # 3) 로컬 캐시의 낡은 값을 지운다

def read_in_replica_region(key):
    value = memcache.get(key)
    if value is None:
        if memcache.get(marker(key)):  # 표식이 있다 = 복제가 아직 낡았을 수 있다
            value = master_db.query(key)   # 지연을 감수하고 마스터에서 읽는다
        else:
            value = replica_db.query(key)  # 평소에는 가까운 복제본에서
        memcache.set(key, value)
    return value

표식의 수명은 복제 스트림이 정합니다. 마스터에 보낸 SQL문에는 키 k의 무효화와 함께 marker의 삭제도 심어져 있어서, 그 쓰기가 복제를 타고 우리 리전 DB에 도착하면 mcsqueal이 marker를 지웁니다. 복제가 따라잡히면 표식도 사라지는 셈이고, 별도의 타이머 없이 복제 스트림 자체가 그 시점을 보장합니다. 리전 간 왕복이라는 비싼 비용은 표식이 살아 있는 짧은 구간에서만 치릅니다. 논문은 이것이 완벽한 정합성이 아니라 낡은 값을 읽을 확률을 지연과 맞바꾸는 절충임을 그대로 적어 두는데, 이렇게 트레이드오프를 분명히 밝히는 대목이 시스템 논문을 읽으면서 가장 도움이 됐습니다.

6. 정리 — Redis 한 대에서 시작해도 같은 지도를 쓴다

이 논문을 덮으면서 세 가지를 적어 두었습니다. 첫째, 무효화는 갱신이 아니라 삭제로 합니다. 멱등성 덕분에 재전송과 경합을 별도 장치 없이 견딜 수 있습니다. 둘째, 미스의 폭주는 채울 권한을 하나로 줄여서 막습니다. lease든 Redis의 SET NX 락이든 DB로 가는 문을 한 명만 통과시킨다는 점은 같습니다. 셋째, 전파는 커밋 로그에서 시작합니다. 애플리케이션은 죽어도 로그는 다시 재생할 수 있습니다.

SafeIn 백엔드에서 목록 캐시를 붙였을 때 제가 부딪힌 건 이 지도의 첫 번째와 두 번째 갈림길이었습니다. 그때는 시행착오로 배웠는데, 이 논문을 먼저 읽었다면 갈림길마다 표지판을 하나씩 세워 둘 수 있었을 것 같습니다. 규모가 천 배 달라도 cache-aside가 앓는 문제의 목록은 같고, Facebook이 겪은 순서를 우리도 그대로 겪게 됩니다. 캐시를 붙이는 날보다 지우는 규칙을 정하는 날이 설계의 시작이라는 걸, 이 글을 읽고 나서야 정리할 수 있었습니다.
