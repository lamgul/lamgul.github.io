논문 분석: Scaling Memcache — 캐시 한 층의 전쟁

2023년 12월• Memcache · 캐시 · 논문

Redis로 목록 캐시를 운영하면서 배운 게 하나 있습니다. 캐시는 넣는 것보다 지우는 게 어렵다는 것. 그런데 이 문제를 지구 규모로 앓아 본 팀이 있습니다. Facebook이 memcached라는 단순한 키-값 저장소를 세계 최대의 캐시 계층으로 키우며 겪은 일을 담은 이 논문은, cache-aside 패턴의 교과서 설명이 끝나는 지점에서 시작합니다. 만료된 인기 키에 요청이 쇄도하면? 순서가 꼬여 낡은 값이 캐시에 남으면? DB가 바뀐 걸 수천 대의 캐시 서버에 알리려면? 제가 Redis 한 대에서 겪은 문제들의 확대판이라, 남 이야기 같지 않게 읽었습니다.

출처

Rajesh Nishtala 외 · NSDI 2013

먼저 짚고 가기
캐시 어사이드Look-Aside Cache
애플리케이션이 캐시를 먼저 들여다보고, 없으면 DB에서 읽어 와 캐시에 직접 채워 넣는 구조입니다. 캐시는 DB를 전혀 모르고, 채우는 책임이 전부 읽는 쪽에 있다는 점이 이 글의 모든 문제의 출발점입니다.
썬더링 허드Thundering Herd
인기 키가 캐시에서 사라진 순간, 그 키를 읽던 수많은 요청이 동시에 미스를 맞고 전부 DB로 몰려가는 현상입니다. 소 떼(herd)가 우르르 달려가는 그림 그대로, 캐시가 지켜주던 DB가 한순간에 두들겨 맞습니다.
리스Lease
캐시 미스가 났을 때 "네가 DB에서 읽어 와 채워 넣어라"라는 권한을 딱 한 클라이언트에게만 주는 토큰입니다. 나머지는 잠깐 기다렸다가 그 클라이언트가 채운 값을 받아 갑니다.
스테일 셋Stale Set
여러 요청의 처리 순서가 꼬여서, 이미 낡아 버린 값이 캐시에 마지막으로 쓰이는 사고입니다. 한번 들어간 낡은 값은 다음 무효화가 올 때까지 캐시에 그대로 남습니다.
무효화Invalidation
DB의 데이터가 바뀌었을 때 캐시에 남은 옛 값을 지워서 두 저장소의 이야기를 맞추는 일입니다. 이 글의 절반은 결국 "무효화를 어떻게 안전하게 배달하는가"에 대한 이야기입니다.

1. 왜 set이 아니라 delete인가 — look-aside의 규칙

논문 2장이 그리는 구조는 우리가 아는 cache-aside 그대로입니다. 읽기는 get → 미스면 DB 조회 → set으로 채우고(demand-filled), 쓰기는 DB를 갱신한 뒤 캐시를 손봅니다. 규모가 다를 뿐입니다 — 논문 3.1절에 따르면 인기 페이지 한 장을 띄우는 데 평균 521개의 서로 다른 아이템을 memcache에서 읽습니다. 페이지 한 장이 곧 캐시 요청 수백 개인 세계입니다.

# look-aside 캐시의 읽기·쓰기 경로 (논문 2장)
def read(key):
    value = memcache.get(key)
    if value is None:               # miss
        value = db.query(key)
        memcache.set(key, value)      # 채우는 책임은 읽는 쪽에 있다
    return value

def write(key, value):
    db.update(key, value)
    memcache.delete(key)              # set이 아니라 delete — 삭제는 멱등이다

여기서 논문이 명시적으로 밝히는 첫 번째 선택이 눈에 띕니다. 쓰기 후에 캐시를 갱신(set)하지 않고 삭제(delete)한다는 것. 이유는 삭제가 멱등이기 때문입니다(논문 2장). 두 개의 쓰기가 경합할 때 set은 어느 쪽이 마지막에 도착하느냐에 따라 낡은 값이 이길 수 있지만, delete는 몇 번을 어떤 순서로 실행해도 "캐시에 없음"이라는 같은 결과로 수렴합니다. 무효화 메시지가 중복 배달되거나 재전송되어도 안전하다는 뜻이고, 이 성질이 뒤에 나올 무효화 파이프라인 전체를 떠받칩니다. 저도 Redis에서 목록 캐시를 갱신으로 유지하려다 포기하고 삭제로 돌아섰던 경험이 있어서, 이 문단은 밑줄을 그으며 읽었습니다.

2. 만료의 순간 — thundering herd

그런데 delete 기반 무효화에는 대가가 있습니다. 삭제된 인기 키를 처음 읽는 순간, 캐시에는 아무것도 없습니다. 초당 수만 번 읽히는 키라면 그 짧은 공백 동안 미스를 맞은 요청이 전부 DB로 직행합니다. 읽기가 많은 키일수록, 그리고 쓰기(=삭제)가 잦은 키일수록 이 공백이 자주 열립니다. 논문 3.2.1절이 이 조합을 정확히 지적합니다 — thundering herd는 "많이 읽히면서 자주 갱신되는 키"에서 터집니다.

이게 무서운 이유는 평소의 DB가 캐시가 걸러 준 만큼만 감당하도록 용량이 잡혀 있기 때문입니다. 히트율 99%의 캐시 뒤에 있는 DB는 트래픽의 1%만 보다가, 인기 키 하나가 지워진 순간 그 키의 트래픽 100%를 맞습니다. 캐시 계층이 평소에 잘할수록 사고의 낙차는 커집니다. 캐시 장애가 곧 DB 장애가 되고, DB가 느려지면 미스가 더 오래 지속되어 herd가 더 커지는 되먹임까지 걸립니다.

3. lease — 토큰 하나로 두 문제를 잡다

논문 3.2.1절의 해법은 단순합니다. 미스가 난 키에 대해 memcached가 64비트 lease 토큰을 발급하되, 키당 기본 10초에 한 번만 발급합니다. 토큰을 받은 클라이언트만 DB에서 값을 읽어 와 채울 자격이 있고, 그 10초 안에 같은 키를 미스한 나머지 요청들은 "곧 채워질 테니 잠깐 기다렸다 다시 읽어라"라는 응답을 받습니다. 실제로는 토큰 보유자가 수 밀리초 안에 값을 채우기 때문에, 기다린 쪽은 재시도에서 캐시 히트를 얻습니다. DB로 가는 요청은 결국 1개입니다.

# lease-get / lease-set 의 뼈대 (논문 §3.2.1)
def read_with_lease(key):
    value, token = memcache.lease_get(key)
    if value is not None:
        return value                     # hit — 평소엔 대부분 여기서 끝난다
    if token is None:                    # hot miss: 토큰은 이미 남이 가져갔다
        sleep_briefly()
        return read_with_lease(key)      # 잠깐 기다리면 대부분 히트가 된다
    value = db.query(key)                # 토큰 보유자만 DB로 간다
    memcache.lease_set(key, value, token)  # 토큰이 아직 유효할 때만 저장된다
    return value

이 장치의 아름다운 점은 stale set까지 같은 토큰으로 잡는다는 것입니다. 그 사이에 누군가 이 키에 delete를 보내면 — 즉 DB가 또 바뀌었으면 — memcached는 발급했던 토큰을 무효로 만듭니다. 낡은 값을 들고 뒤늦게 도착한 lease_set은 토큰 검증에서 거부됩니다. "내가 DB에서 읽은 시점"과 "캐시에 쓰는 시점" 사이에 세상이 바뀌었는지를, 캐시 서버가 토큰의 생사로 판정해 주는 셈입니다. 검증되지 않은 낙관적 set이 만들던 경쟁 조건이, 토큰이라는 버전 표식 하나로 정리됩니다.

효과는 논문이 숫자로 보여줍니다. 3.2.1절은 썬더링 허드에 특히 취약한 키들을 골라 일주일간 캐시 미스를 관찰했는데, lease가 없을 때 DB 쿼리 피크가 17K/s였던 것이 lease를 켜자 1.3K/s로 내려갔습니다. 피크 부하를 기준으로 DB를 증설하는 팀에게, 알고리즘 한 줌이 장비 몇 대 몫을 하는 순간입니다.

*

Redis에는 lease-get이 없지만 같은 모양을 만들 수 있습니다. 미스가 나면 SET lock:key ... NX PX 3000으로 짧은 락을 선점한 쪽만 DB를 읽어 채우고, 실패한 쪽은 수십 ms 기다렸다 재시도하는 식입니다. 제가 Redis 무효화를 다룬 글에서 삭제 직후의 미스 폭주를 고민했던 것과 정확히 같은 자리에, 이 논문은 10년 전에 이미 답을 놓아두고 있었습니다.

4. 무효화를 배달하는 법 — 커밋 로그에서 뽑는 delete

이제 규모의 문제가 남습니다. 클러스터에는 수천 대의 memcached 서버가 있고, 같은 키가 여러 클러스터에 복제되어 있습니다. DB의 한 행이 바뀌었을 때, 그 키를 들고 있는 모든 캐시 서버에 delete를 배달해야 합니다. 누가 배달하느냐가 문제입니다.

순진한 답은 "쓰기를 수행한 웹서버가 직접 delete를 뿌린다"입니다. 실제로 Facebook도 클러스터 안에서는 이렇게 합니다. 하지만 리전 전체로 넓히면 논문 4.1절이 지적하는 두 가지 약점이 드러납니다. 웹서버는 무효화를 묶어 보내는 데 서툴러 패킷 오버헤드가 커지고, 설정 실수로 delete가 엉뚱한 곳으로 가는 사고가 터지면 되돌릴 방법이 없습니다 — 논문은 과거에 이런 일이 memcache 인프라 전체의 롤링 재시작으로 이어지곤 했다고 적습니다. 그래서 4.1절의 선택은 배달의 출발점을 애플리케이션에서 DB 커밋 로그로 옮기는 것입니다. mcsqueal이라는 무효화 데몬이 모든 DB에 붙어 자기 DB가 커밋한 SQL문을 들여다보고, 거기에 함께 심어 둔 memcache 키를 추출해 delete를 만듭니다. 이걸 낱개로 쏘면 패킷 수가 감당이 안 되므로 mcsqueal이 묶음으로 포장해 각 프론트엔드 클러스터의 mcrouter로 보내고, mcrouter가 묶음을 풀어 키마다 담당 memcached 서버로 나눠 줍니다.

커밋 로그를 출발점으로 삼는 순간 두 가지가 공짜로 따라옵니다. 첫째, 커밋된 변경만 무효화를 만듭니다 — 롤백된 트랜잭션이 캐시를 지우는 일이 없습니다. 둘째, 로그는 남아 있으므로 배달이 실패하면 다시 재생할 수 있습니다. 그리고 재생이 안전한 이유가 바로 앞 장의 그 선택, delete의 멱등성입니다. 같은 delete가 두 번 도착해도 아무 일도 일어나지 않으니까요. 논문의 설계들이 서로를 받쳐 주는 이 맞물림이, 개별 기법보다 더 배울 점이라고 생각합니다.

*

이 패턴은 오늘날 CDC(Change Data Capture)라 부르는 것의 원형입니다. Debezium으로 binlog를 읽어 캐시를 지우거나 검색 인덱스를 맞추는 요즘의 구성은, mcsqueal이 하던 일에 이름과 범용 도구가 붙은 것에 가깝습니다. "정합성이 중요한 전파는 애플리케이션의 선의가 아니라 커밋 로그에서 시작하라"는 원칙으로 기억해 두면 두고두고 씁니다.

5. 리전을 넘어서 — 복제 지연과 remote marker

마지막 관문은 지리입니다. 논문 5장에서 Facebook은 마스터 리전 하나에 쓰기를 모으고, 다른 리전들은 MySQL 복제로 데이터를 받아 읽기를 처리합니다. 문제는 복제 지연입니다. 복제 리전의 사용자가 글을 고치면 쓰기는 마스터로 가는데, 자기 리전의 복제 DB에는 그 변경이 아직 도착하지 않았습니다. 이 상태에서 로컬 캐시만 지우면, 다음 읽기가 낡은 복제 DB에서 옛 값을 읽어 캐시에 도로 채워 넣습니다. 사용자 입장에서는 방금 고친 글이 다시 옛날로 돌아간 것처럼 보입니다.

해법인 remote marker는 "이 키는 지금 우리 리전이 낡았을 수 있다"는 표식을 캐시 계층에 남기는 것입니다.

# 비마스터 리전에서의 쓰기와 읽기 (논문 5장, remote marker)
def write_from_replica_region(key, value):
    memcache.set(marker(key), 1)       # 1) 우리 리전에 표식을 세운다
    master_db.update(key, value)       # 2) 마스터에 쓴다 — SQL에 marker 삭제도 심는다
    memcache.delete(key)               # 3) 로컬 캐시의 낡은 값을 지운다

def read_in_replica_region(key):
    value = memcache.get(key)
    if value is None:
        if memcache.get(marker(key)):  # 표식이 있다 = 복제가 아직 낡았을 수 있다
            value = master_db.query(key)   # 지연을 감수하고 마스터에서 읽는다
        else:
            value = replica_db.query(key)  # 평소에는 가까운 복제본에서
        memcache.set(key, value)
    return value

표식의 수명이 절묘합니다. 마스터에 보낸 SQL문에는 키 k의 무효화와 함께 marker의 삭제도 심어져 있어서, 그 쓰기가 복제를 타고 우리 리전 DB에 도착하는 순간 mcsqueal이 marker를 지웁니다. 즉 "복제가 따라잡히면 표식도 사라진다"가 별도의 타이머 없이 복제 스트림 그 자체로 보장됩니다. 리전 간 왕복이라는 비싼 값은 표식이 살아 있는 짧은 창에서만 치릅니다. 논문은 이것이 완벽한 정합성이 아니라 낡은 값을 읽을 확률을 지연과 맞바꾸는 절충임을 숨기지 않는데, 트레이드오프를 명시하는 이런 정직함이 시스템 논문을 읽는 맛입니다.

6. 정리 — Redis 한 대에서 시작해도 같은 지도를 쓴다

이 논문에서 제가 가져간 건 세 문장입니다. 첫째, 무효화는 갱신이 아니라 삭제로 하라. 멱등성이 재전송과 경합을 공짜로 견디게 해 줍니다. 둘째, 미스의 폭주는 채울 권한을 하나로 줄여서 막아라. lease든 Redis의 SET NX 락이든, 핵심은 "DB로 가는 문을 한 명만 통과시키는 것"입니다. 셋째, 전파는 커밋 로그에서 시작하라. 애플리케이션은 죽지만 로그는 재생됩니다.

SafeIn 백엔드에서 목록 캐시를 붙였을 때 제가 부딪힌 건 이 지도의 첫 번째와 두 번째 갈림길이었습니다. 그때는 시행착오로 배웠지만, 이 논문을 먼저 읽었다면 갈림길마다 표지판이 있었을 겁니다. 규모가 천 배 달라도 cache-aside가 앓는 병의 목록은 같고, Facebook이 겪은 순서대로 우리도 겪게 됩니다. 캐시를 "붙이는" 날이 아니라 "지우는 규칙을 정하는" 날이 진짜 설계가 시작되는 날이라는 것 — 이 한 층의 전쟁에서 배울 가장 큰 교훈입니다.
