서평: Clean Architecture — 의존성은 안쪽을 향한다

2024년 1월• Clean Architecture · 의존성 규칙 · SOLID

Robert C. Martin의 『Clean Architecture』는 소프트웨어의 가치가 '지금 동작하는 것'보다 '바꾸기 쉬운 것'에 있다는 주장을 처음부터 끝까지 밀고 갑니다. 저는 SafeIn 백엔드를 헥사고날로 옮기면서 경계를 긋고 포트·어댑터를 붙이는 일을 손으로 먼저 익혔는데, 그 감각이 어떤 원리 위에 서 있었는지는 이 책을 읽고 나서야 정리가 됐습니다. SOLID부터 아키텍처 경계까지를 의존성 규칙 하나로 꿰는 책이고, 요지는 화살표가 언제나 안쪽을 향해야 한다는 것입니다. 읽으며 정리한 내용을 남겨 둡니다.

먼저 짚고 가기
의존성dependency
어떤 코드가 다른 코드를 알고, 그것이 없으면 못 도는 관계입니다. A가 B를 import해서 쓰면 'A는 B에 의존한다'고 말합니다. 이 글의 화살표(→)는 전부 '누가 누구에게 의존하는가'의 방향을 가리킵니다.
의존성 규칙Dependency Rule
이 책 전체를 관통하는 원칙입니다. 소스 코드의 의존성은 오직 '안쪽'(핵심 업무 규칙)을 향해야 하고, 안쪽은 바깥(DB·프레임워크)의 존재조차 몰라야 한다는 원칙입니다. 자주 바뀌는 것이 잘 안 바뀌는 것에 기대게 만들어, 바깥이 흔들려도 핵심은 멀쩡하도록 지켜 줍니다.
DIP · 의존성 역전 원칙Dependency Inversion Principle
SOLID의 마지막 글자입니다. 원래대로면 핵심 로직이 DB 같은 구체적 도구를 직접 부르며 의존하게 되는데, 그 사이에 인터페이스(약속)를 끼워 화살표를 거꾸로 뒤집는 기법입니다. 그러면 도구가 핵심의 약속을 따르게 되어, 의존성 규칙을 실제로 지킬 수 있습니다.
포트·어댑터 (헥사고날)Port & Adapter / Hexagonal
핵심이 '이런 기능이 필요하다'고 정해 둔 인터페이스가 포트(콘센트 구멍), 그 포트에 실제로 꽂히는 DB·API 구현이 어댑터(플러그)입니다. 콘센트 규격만 맞으면 플러그를 갈아 끼우듯, DB나 외부 API를 바꿔도 핵심은 손대지 않습니다. 이 글에서 저자가 실무에 적용한 설계 방식입니다.
합성 루트Composition Root
추상(포트)과 구체(어댑터)를 실제로 이어 주는, 프로그램에서 딱 한 곳뿐인 조립 지점입니다. NestJS에서는 모듈의 프로바이더 등록이 그 역할을 합니다. 연결을 여기 한곳에 모아 두어야 나머지 코드가 구체적 도구를 몰라도 됩니다.

두 개의 가치, 그리고 구조의 편

책은 「A Tale of Two Values」에서 소프트웨어에 두 가지 가치가 있다고 말합니다. 하나는 지금 요구대로 동작하는가를 뜻하는 행위(behavior)이고, 다른 하나는 그 행위를 얼마나 싸게 바꿀 수 있는가를 뜻하는 구조(architecture)입니다. Martin은 길게 보면 두 번째가 더 중요하다고 봅니다. 동작하지만 못 바꾸는 프로그램은 요구가 바뀌는 순간 쓸모를 잃는 반면, 틀렸지만 무른 프로그램은 고쳐서 계속 살릴 수 있기 때문입니다.

다만 조직은 늘 급한 쪽을 고르게 됩니다. 아이젠하워의 사분면으로 치면 기능은 '긴급'이고 구조는 '중요'인데, 실제로는 긴급이 중요를 매번 이깁니다. 그래서 구조의 편에 서서 목소리를 내는 일이 아키텍트의 몫이라고 책은 말합니다.

정산 로직을 리팩토링하던 때가 생각났습니다. 코드는 분명히 동작했지만, 수수료 정책 한 줄을 바꾸려면 컨트롤러와 서비스, 쿼리, DTO를 모두 헤집어야 했습니다. 행위는 멀쩡한데 구조 쪽은 손대기 어려운 상태였습니다. 당시에는 그 상태를 뭐라고 불러야 할지 몰랐는데, 아키텍처의 목표가 우아함이 아니라 생애 비용의 최소화, 그러니까 만들고 유지하는 데 드는 총 노력을 줄이는 것이라는 대목에서 리팩토링의 기준선이 잡혔습니다.

SOLID: 결국 DIP로 수렴한다

3부는 SOLID를 다루는데, 흔히 소개되는 '클래스 잘 짜는 요령'과는 초점이 다릅니다. Martin은 SOLID를 모듈을 어떻게 배치하느냐의 규칙으로 읽습니다. 이를테면 SRP는 '한 가지 일만 하라'가 아니라 '한 액터에게만 책임지라', 곧 변경의 이유가 하나여야 한다는 뜻입니다. 급여 클래스 하나가 CFO·COO·CTO 세 부서의 요구를 동시에 떠안으면, 한 부서 때문에 고친 코드가 다른 부서를 부수게 됩니다.

OCP·LSP·ISP도 컴포넌트 수준에서 다시 읽힙니다. 확장에는 열리고 수정에는 닫히려면 무엇이 무엇에 의존하는지 그 방향을 설계해야 하고, 그 방향을 뒤집는 도구가 다섯 글자 중 마지막인 DIP입니다. 소스 코드의 의존성은 구체가 아니라 추상을 향해야 하며, 자주 변하는 구체(DB·프레임워크·외부 API)는 안정된 핵심이 소유한 인터페이스 뒤에 숨깁니다.

크롤러를 DDD로 짤 때 이 원칙을 설계의 뼈대로 삼았습니다. 사이트마다 HTML은 제멋대로 바뀌는 휘발성 세부입니다. 도메인이 특정 사이트의 마크업에 의존하면 사이트가 바뀔 때마다 도메인이 흔들리기 때문에, 도메인은 Fetcher·Parser 같은 추상에만 기대게 하고 사이트별 구현은 바깥에 뒀습니다. 앞 네 글자를 아키텍처의 이야기로 만들어 주는 것이 DIP라는 말을 책은 여러 번 반복합니다.

컴포넌트 — 응집과 결합의 저울

클래스를 다 짜고 나면 무엇을 함께 배포할 단위로 묶을지가 다음 질문이 되는데, 4부의 컴포넌트 원칙이 여기에 답합니다. 응집에는 세 원칙(REP·CCP·CRP)이 있고, 셋은 서로 잡아당깁니다. 함께 바뀌는 것은 함께 묶으라(CCP)와 함께 쓰지 않는 것은 떼어 내라(CRP)가 충돌하기 때문에, 셋을 동시에 만족시킬 수는 없고 프로젝트의 성숙도에 따라 저울추를 옮기게 됩니다.

결합에는 ADP·SDP·SAP가 있습니다. 의존성 그래프에 순환이 없어야 한다는 것(ADP)과 덜 안정된 것이 더 안정된 쪽을 향해 의존해야 한다는 것(SDP)이 뼈대이고, 안정된 것은 그만큼 추상적이어야(SAP) 유연함을 잃지 않습니다. 이 셋의 균형점을 책은 'Main Sequence'라는 선으로 그립니다.

NestJS 모놀리스의 모듈을 쪼갤 때 이 저울을 기준으로 삼았습니다. 정산 모듈과 크롤링 모듈이 서로를 참조하기 시작하면 순환이 생기고, 한쪽을 바꿀 때 다른 쪽이 딸려 옵니다. 그럴 때는 공유 커널에 인터페이스를 두고 화살표를 한 방향으로 정리했습니다. 컴포넌트의 경계가 변경을 가둬 두는 벽이라는 것을 이 장에서 배웠습니다.

의존성 규칙 — 안쪽을 향하는 화살표

5부에는 자주 인용되는 동심원 그림이 나옵니다. 안에서 밖으로 엔티티, 유스케이스, 인터페이스 어댑터, 프레임워크와 드라이버가 놓이고, 이 전체를 지배하는 한 문장이 의존성 규칙입니다. 소스 코드의 의존성은 오직 안쪽을 향해야 하며, 안쪽 원은 바깥 원의 존재조차 몰라야 한다는 것입니다.

읽으면서 한동안 헷갈린 지점이 있습니다. 제어 흐름과 소스 의존성의 방향이 다르다는 점입니다. 유스케이스가 DB를 호출하니 제어는 바깥으로 흐르지만, 소스 코드의 의존성은 반대로 안쪽을 향해야 합니다. 이 둘이 어긋나는 자리가 DIP를 꽂는 자리입니다.

*

제어 흐름은 바깥으로 흐르지만(유스케이스 → DB), 소스 코드 의존성은 안쪽을 향해야 합니다. 이 둘이 어긋나는 지점에 인터페이스를 세워 화살표를 뒤집는 일이 경계를 긋는다는 말의 실제 내용입니다.

말로만 두면 잘 잡히지 않아 NestJS 코드로 옮겨 봤습니다. 유스케이스가 포트(인터페이스)를 소유하고, 바깥의 어댑터가 그 포트를 구현합니다. 두 조각을 실제로 잇는 곳은 합성 루트(Main) 한 군데뿐이고, NestJS에서는 모듈의 프로바이더 등록이 그 역할을 합니다.

// application/ports/settlement-repository.port.ts — 안쪽(유스케이스)이 소유하는 계약
export const SETTLEMENT_REPOSITORY = Symbol('SettlementRepositoryPort');
export interface SettlementRepositoryPort {
  findPending(vendorId: VendorId): Promise<Settlement[]>;
  save(s: Settlement): Promise<void>;
}

// application/settle-payouts.usecase.ts — 프레임워크도 DB도 모른다
export class SettlePayoutsUseCase {
  constructor(
    @Inject(SETTLEMENT_REPOSITORY) private readonly repo: SettlementRepositoryPort,
  ) {}
  async execute(vendorId: VendorId) {
    const pending = await this.repo.findPending(vendorId);
    for (const s of pending) await this.repo.save(s.confirm());
  }
}

// infrastructure/typeorm-settlement.repository.ts — 바깥이 안쪽 계약을 구현
@Injectable()
export class TypeormSettlementRepository implements SettlementRepositoryPort {
  constructor(@InjectRepository(Row) private rows: Repository<Row>) {}
  findPending(vendorId: VendorId) { /* ORM 질의 → 도메인 매핑 */ }
  save(s: Settlement) { /* 도메인 → 행 매핑 후 저장 */ }
}

// app.module.ts — 합성 루트(Main): 여기서만 구체가 추상에 꽂힌다
providers: [
  SettlePayoutsUseCase,
  { provide: SETTLEMENT_REPOSITORY, useClass: TypeormSettlementRepository },
]

화살표를 하나씩 따라가 보면 방향이 보입니다. TypeormSettlementRepository는 안쪽의 SettlementRepositoryPort를 향해 의존하지만, 유스케이스는 TypeORM의 존재를 모릅니다. 제어는 바깥으로 흘러도 소스 의존성은 안쪽을 향한 채로 남고, 동심원이 지키려는 것이 이 상태입니다.

네 원을 NestJS 레이어에 앉히기

동심원은 폴더 구조로 옮겨 보고 나서야 손에 잡혔습니다. 가장 안쪽의 엔티티는 프레임워크를 모르는 순수한 도메인 클래스입니다. 정산 규칙처럼 'SafeIn이라는 서비스가 없어도 참인' 핵심 업무 규칙이 여기 놓이고, 데코레이터도 ORM도 붙이지 않습니다.

// domain/settlement.ts — Entity: SafeIn이라는 서비스가 없어도 참인 규칙
export class Settlement {
  private constructor(
    readonly vendorId: VendorId,
    readonly gross: Money,
    readonly feeRate: Rate,
  ) {}

  // 핵심 업무 규칙: 수수료를 뗀 실지급액
  net(): Money {
    return this.gross.minus(this.gross.times(this.feeRate));
  }

  confirm(): Settlement { /* 상태 전이도 도메인 안에서 */ }
}

그 바깥의 유스케이스는 애플리케이션 고유의 흐름을 담고 포트를 소유합니다. 인터페이스 어댑터에는 @Controller와 DTO·프레젠터 매퍼, 리포지토리 구현이 살고, 가장 바깥의 프레임워크와 드라이버에는 NestJS·TypeORM·HTTP와 vLLM 클라이언트처럼 '세부'에 해당하는 것들이 모입니다. 폴더로 옮기면 네 원이 그대로 네 디렉터리가 됩니다.

src/
  domain/          # Entities — 가장 안쪽, 아무것도 import 하지 않는다
  application/     # Use Cases — 포트(인터페이스)를 소유한다
  interface/       # Interface Adapters — 컨트롤러·프레젠터·매퍼
  infrastructure/  # Frameworks & Drivers — TypeORM·HTTP·vLLM 클라이언트

이 배치는 「The Database Is a Detail」, 「Frameworks Are Details」, 「Screaming Architecture」가 말한 바를 그대로 옮겨 놓은 것입니다. 폴더 트리를 열었을 때 'NestJS 프로젝트'가 아니라 '정산과 크롤링 시스템'으로 읽혀야 하고, DB도 웹도 프레임워크도 중심이 아니라 갈아 끼우는 세부라는 이야기입니다.

이 배치가 실제로 비용을 아껴 준 적이 있습니다. 호스팅형 LLM API를 온프레미스 vLLM 서빙으로 바꿀 때 유스케이스는 LlmPort에만 기대고 있었으므로 어댑터 한 장만 교체하면 됐습니다. 세부를 세부의 자리에 둔 덕이었습니다. 포트·어댑터(헥사고날)는 이 네 원을 구현하는 한 가지 형태이고, 저는 SafeIn을 레이어드에서 헥사고날로 옮기며 같은 규칙을 적용해 봤습니다.

규칙을 CI가 지키게 만든다

의존성 규칙에는 약점이 있습니다. 말뿐인 약속이라 조용히 무너진다는 점입니다. 누군가 도메인에서 인프라를 import 한 줄 해도 런타임은 멀쩡히 돌아가고 아무도 경고하지 않습니다. 그렇게 화살표가 하나둘 뒤집히면 반년 뒤에는 의존성이 다시 엉켜 있습니다.

그래서 규칙은 사람의 규율보다 기계의 검사에 맡기는 편이 낫다고 봅니다. dependency-cruiser나 eslint-plugin-boundaries로 '안쪽에서 바깥쪽으로의 import'를 CI에서 에러로 막아 두면, 의존성 규칙이 문서가 아니라 빌드를 깨는 불변식이 됩니다.

// .dependency-cruiser.js — "안쪽을 향한다"를 CI가 강제하게
module.exports = {
  forbidden: [{
    name: 'no-inward-violation',
    severity: 'error',
    comment: '도메인·유스케이스는 바깥 레이어를 import 할 수 없다',
    from: { path: '^src/(domain|application)' },
    to:   { path: '^src/(interface|infrastructure)' },
  }],
};
*

화살표는 소리 없이 뒤집힙니다. 잘못된 import 한 줄을 막는 정적 검사기의 값은 반년 뒤 뒤엉킨 의존성을 다시 푸는 비용보다 싸서, 규칙은 문서로 적어 두기보다 CI에 강제해 두는 편으로 정리했습니다.

같은 맥락에서 책은 테스트를 가장 바깥 원으로 봅니다(「The Test Boundary」). 테스트가 휘발성 구조에 밀착하면 사소한 변경에도 우수수 깨지는 'Fragile Test' 문제가 생기므로, 테스트도 안정된 API에 기대야 한다고 말합니다. 「Services: Great and Small」의 경고도 함께 적어 둡니다. 서비스를 쪼갠다고 아키텍처가 저절로 생기지는 않으며, 경계는 아키텍처지만 네트워크 홉은 그렇지 않다는 것입니다.

정리 — 무엇을 안쪽에 둘 것인가

다 읽고 정리해 보면 이 책은 하나의 생각을 세 축척으로 반복합니다. 클래스 수준의 SOLID, 모듈 수준의 컴포넌트 원칙, 시스템 수준의 의존성 규칙이 모두 같은 말을 합니다. 휘발성 세부는 바깥에, 정책은 안쪽에 두고 화살표를 안으로 향하게 하라는 것입니다.

책은 폴더 이름의 정답을 주지는 않습니다. 대신 화살표의 방향을 정해 줌으로써 나머지 결정을 미룰 수 있게 해 줍니다. DB와 프레임워크, 서빙 방식을 나중에 골라도 된다는 뜻이고, 아키텍처란 '내리지 않은 결정의 수를 최대로 남겨 두는 기술'이라는 정의가 저에게는 가장 오래 남았습니다.

메모에는 이렇게 적어 두었습니다. 아키텍처는 다이어그램이 아니라 생애 비용으로 측정되고, 그 비용을 좌우하는 것은 변경을 가두는 경계의 위치다. 남은 질문은 그 경계 안쪽, 곧 도메인 자체를 어떻게 모델링할 것인가였습니다. Evans의 도메인 주도 설계를 다음으로 이어 읽기로 한 이유이기도 합니다.
