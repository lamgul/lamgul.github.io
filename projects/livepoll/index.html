살아있는 투표Livepoll

데이터베이스를 켜지 않고, 실시간 상태를 어디까지 버틸 수 있을까. 메모리에 사는 투표와 WebSocket으로 흐르는 결과. 아래 데모는 지금 이 순간에도 봇들이 투표하는 중입니다.

Pages 배포본은 브라우저 안의 시뮬레이션 서버로 돕니다. 진짜 다중 접속은 server/를 띄우고 VITE_SERVER_URL만 걸면 됩니다.

저는 원래 "일단 DB부터 설계하는" 사람이었습니다. 커머스 백오피스를 오래 만들면 그렇게 됩니다. 그래서 반대로 가보고 싶었습니다 — 영속성을 아예 빼면 실시간 기능은 얼마나 단순해질까?

투표는 오래 살 필요가 없다

라이브 투표는 몇 분이면 끝나는 이벤트입니다. 끝나면 결과도 대개 의미가 없어집니다. 그렇다면 데이터를 디스크에 앉힐 이유가 없습니다. 전체 데이터셋을 Map 하나에 담고, 모든 변경은 O(1)이며, 오래된 투표는 주기적으로 쓸어냅니다. 재시작하면 다 날아가지만 — 그게 바로 이 설계의 거래 조건입니다.

// 한 소켓 = 한 표. 재투표는 표를 '옮긴다'. 접속이 끊기면 표를 회수한다.
vote(pollId, optionId, voterId) {
  const prev = poll.ballots.get(voterId);
  if (prev === optionId) return false;      // no-op
  if (prev) poll.options.get(prev).votes--;   // 옮기기 전 자리 비우기
  target.votes++;
  poll.ballots.set(voterId, optionId);
  return true;                              // 바뀌었으니 방송해라
}

"누가 보고 있나"는 서버에 묻지 않는다

접속자 수를 직접 세는 코드를 짜지 않았습니다. Socket.IO의 room이 곧 그 투표의 관중석이기 때문입니다. 방 크기를 물어보면 그게 시청자 수입니다. 상태가 바뀔 때마다 전체 스냅샷을 방 전체에 던지는 단순한 모델 — payload가 작으니 diff를 계산하느라 머리 쓸 이유가 없었습니다.

*

전체 스냅샷 방송은 선택지가 8개로 제한된 이 도메인에서만 정답입니다. 항목이 수천 개였다면 델타 전송이 필요했을 겁니다. 설계는 늘 도메인의 크기에 붙어 있습니다.

같은 앱이 라이브에서도, 데모에서도

문제가 하나 있었습니다. GitHub Pages에는 서버가 없습니다. 그런데 저는 이 프로젝트가 클릭하면 바로 살아 움직이길 바랐습니다. 그래서 클라이언트와 서버 사이에 얇은 트랜스포트 한 겹을 넣었습니다. 앱은 자기가 진짜 소켓과 말하는지, 브라우저 안의 시뮬레이터와 말하는지 모릅니다.

export function createTransport(): Transport {
  const url = import.meta.env.VITE_SERVER_URL;
  return url ? new LiveTransport(url)   // dev: 진짜 Socket.IO
                 : new SimTransport();     // Pages: 봇이 있는 시뮬레이터
}

시뮬레이터는 실제 서버의 의미(생성·참여·투표·스냅샷)를 그대로 흉내 내되, 가짜 지연과 봇 투표자를 얹었습니다. 살아있지 않은 실시간 투표는 그냥 막대그래프니까요. 위 데모에서 막대가 저절로 차오르는 건 그 봇들입니다.

남은 것 · 다음

  • 수평 확장. 지금은 단일 프로세스라 상태를 프로세스가 소유합니다. 여러 인스턴스로 늘리려면 Socket.IO의 Redis 어댑터로 방송을 공유하고, 표 상태도 Redis로 옮겨야 합니다. "인메모리"의 유효기간은 딱 거기까지입니다.
  • 정합성. 소켓 단위 1표라 새 탭을 열면 새 표가 됩니다. 진지한 투표라면 세션·핑거프린트가 필요합니다. 재미용으론 이게 정직한 선택이었습니다.
  • 마감. closed 필드는 뚫어뒀지만 UI는 아직입니다. 알고 남긴 자리입니다.
