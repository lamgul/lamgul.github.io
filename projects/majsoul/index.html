유지 = 14장 손패의 샹텐을 깎아 먹지 않는 버림 · 유효패 = 샹텐을 한 칸 내리는 츠모(종류 · 남은 장수) · 루트 카드나 표의 행을 누르면 해당 타패가 손패 위에 강조됩니다

프리셋 ①은 원본 레포 단위 테스트(MahjongAdvisorTest)의 손패 그대로입니다. 정답: 동을 버리면 텐파이.

작혼(Mahjong Soul)을 켜면 화면 아래에 내 손패 14장이 보입니다. 원본 앱은 그 화면을 캡처해 패를 읽고, 무엇을 버릴지 계산해서 게임 위에 얇은 패널로 답을 띄우는 안드로이드 도우미입니다. 서버도, 남의 마작 엔진도 없이 — 전부 폰 안에서 계산합니다. 그 계산의 심장을 이 페이지에 옮겨 왔습니다.

화면 캡처에서 오버레이까지

파이프라인은 네 칸입니다. MediaProjection이 가로 화면을 캡처하고, 고정 슬롯 좌표에서 손패 14장·도라 표시패·버림패·후로를 잘라 냅니다. 잘라 낸 조각은 tile_templates.bin의 타일 템플릿과 대조하는 nearest-template 분류기가 34종 중 하나로 읽습니다. 그 결과를 ShantenCalculator와 MahjongAdvisor가 받아 루트별 추천 타패를 계산하고, SYSTEM_ALERT_WINDOW 오버레이가 게임 위에 카드로 띄웁니다. 3인/4인 테이블은 화면 배치로 자동 판별합니다. 위 데모의 카드 패널이 그 오버레이의 재현입니다.

샹텐은 8에서 빼 내려간다

샹텐 수는 '텐파이까지 몇 걸음 남았나'입니다. 일반형 계산의 뼈대는 배점 하나입니다 — 완성 멘쯔는 2점, 부분 멘쯔(타아츠)와 머리는 1점, 8에서 그 합을 빼면 샹텐이 나옵니다. ShantenCalculator는 손패를 34칸 배열로 놓고, 가장 낮은 패부터 '코쯔 / 슌쯔 / 머리 / 타아츠 / 버림'의 다섯 갈래를 DFS로 전부 시도합니다. 아래는 원본 Java에서 그대로 가져온 발췌입니다.

private void dfs(int[] counts, int melds, int pairs, int taatsu) {
    int index = firstNonZero(counts);
    if (index == -1) {
        int usableTaatsu = Math.min(taatsu, 4 - melds);
        int shanten = 8 - (melds * 2) - usableTaatsu - pairs;
        if (shanten < best) best = shanten;
        return;
    }
    // ... 코쯔 갈래 생략 ...
    if (canSequence(index) && counts[index + 1] > 0 && counts[index + 2] > 0) {
        counts[index]--; counts[index + 1]--; counts[index + 2]--;
        dfs(counts, melds + 1, pairs, taatsu);   // 슌쯔로 묶어 본다
        counts[index]++; counts[index + 1]++; counts[index + 2]++;
    }
    // ... 머리·타아츠·버림 갈래 생략 ...
}

치또이츠는 6 - 또이츠 수, 국사무쌍은 13 - 요구패 종류 - 머리 보정 — 특수형 둘은 공식 한 줄이라 DFS 없이 바로 나옵니다. 데모 상단의 '일반형 · 치또이 · 국사' 세 숫자가 각각의 결과이고, 손패의 샹텐은 셋 중 최솟값입니다.

유효패는 서른네 번 다 넣어 본다

'무엇을 버릴까'는 사실 이중 루프입니다. 버림패 후보 최대 14가지 × 다음 츠모 34종을 전부 시뮬레이션해서, 샹텐이 실제로 줄어드는 츠모만 남은 장수로 합산합니다. 이게 유효패(ukeire)입니다.

for (int draw = 0; draw < TILE_TYPES; draw++) {
    int unavailable = afterDiscard[draw] + visibleCounts[draw]
            + (draw == discardedTile ? 1 : 0);
    if (unavailable >= 4) continue;      // 넉 장 다 보이는 패는 못 온다
    afterDiscard[draw]++;                  // 한 장 뽑아 넣어 보고
    if (routeShanten(afterDiscard, mode, fixedMelds) < baseShanten) {
        total += 4 - unavailable;          // 샹텐이 줄면 남은 장수만큼
    }
    afterDiscard[draw]--;
}

흥미로운 결정 하나: 여기엔 영리한 수학이 없습니다. 마작 손패는 34종 14장 — 도메인이 작아서, 후보마다 DFS를 통째로 다시 도는 완전 탐색으로 충분합니다. 데모 우측 상단의 ms가 그 비용의 실측값입니다. 보통의 손패는 한 자릿수 ms, DFS가 가장 싫어하는 긴 연속형(프리셋 ①)도 0.1초 안팎 — 클릭 한 번에 치르기엔 싼 값입니다. 알고리즘의 우아함보다 도메인의 크기를 먼저 재는 편입니다.

재현한 것, 생략한 것

이 데모의 계산부는 ShantenCalculator.java와 MahjongAdvisor.java를 함수 단위로 그대로 옮긴 JS입니다. DFS의 다섯 갈래, 루트별 점수식과 타이브레이크, 카드의 정렬 규칙과 문구까지 원본과 같고, 프리셋 ①은 레포 단위 테스트의 손패입니다. 포팅하며 보탠 것은 하나 — 같은 배열에 대한 DFS 결과를 분석 한 번 동안 캐시하는 메모이제이션뿐입니다.

생략한 것도 분명히 적어 둡니다. 화면 캡처와 타일 분류기는 뺐습니다(브라우저엔 읽을 게임 화면이 없으니 손패를 직접 편집하게 했습니다). 도라·버림패·후로 입력도 뺐는데, 그래서 남은 장수를 손패만 보고 세니 유효패가 실제 앱보다 후하게 계산됩니다. 3인 테이블 모드도 생략 — 데모는 4인 기준 136장 세계를 가정합니다.
