통제 콘솔SUMS CONSOLE

현장 근로자의 스마트폰을 구역·시간·행동조건으로 통제하는 관제 시스템. 구역을 그리고 정책을 조합하면 위반 로그가 흐릅니다.

역할설계 · 프론트엔드 · 백엔드
스택React · NestJS · Prisma · PostgreSQL · Leaflet
코드현재는 SafeIn으로 계승 — 세이프인 데모 ↗
sums console — 목데이터 · 네트워크 요청 0
스마트건설 주식회사

시간대별 위반 — 최근 24시간

경고차단

GET /api/statistics/dashboard · violation-history — 실제 백엔드 응답을 목데이터로 재현

구역을 클릭하면 상세 정보를 보여줍니다

GET /api/zones — 원본은 Leaflet 지도 위 위경도 지오펜스, 데모는 인라인 SVG 약도

정책 빌더 — 조건을 조합합니다

POST /api/policies — ControlPolicy가 Zone · TimePolicy · BehaviorCondition · HarmfulAppPreset을 M:N으로 참조

시간직원구역정책유형사유

GET /api/control-logs — 실서비스는 디바이스 앱이 밀어 넣고, 데모는 브라우저 타이머가 흘려보냅니다

산업현장과 물류 현장에서 근로자의 스마트폰은 편의이자 사고 원인입니다. SUMS는 위험 구역·작업 시간·행동 조건을 조합해 그 사용을 통제하는 관제 시스템입니다. 실구현은 비공개 레포에 있어, 이 페이지는 관리자 콘솔의 핵심 화면 네 개를 실제 도메인 데이터로 재현합니다.

정책은 소유하지 않고 참조합니다

실제 시스템은 React 콘솔(약 30개 화면)과 NestJS·Prisma·PostgreSQL API로 구성됩니다. 도메인의 중심은 통제 정책(ControlPolicy)인데, 정책이 조건들을 직접 소유하지 않습니다. 구역(Zone)·시간(TimePolicy)·행동 조건(BehaviorCondition)·유해앱 프리셋(HarmfulAppPreset)은 각자 독립된 테이블이고, 정책은 이들을 M:N으로 참조만 합니다. '주간 작업 시간' 하나를 여러 정책이 공유하므로, 점심시간 규정이 바뀌면 수정은 한 번이면 됩니다.

// sums-admin-backend/prisma/seed-simple.ts — 정책과 구역의 조인 테이블
const controlPolicyZones = zones.slice(0, 10).map((zone, idx) => ({
  controlPolicyId: `cp-${idx + 1}`,
  zoneId: zone.id,
}));
await prisma.controlPolicyZone.createMany({ data: controlPolicyZones });

데모가 재현한 것, 생략한 것

구역·정책·직원·위반 사유는 실제 콘솔의 mockData.ts에서 그대로 가져왔습니다. 크레인 작업 구역의 반경 50m, 시속 30km 차량속도 조건, '게임 차단' 프리셋의 패키지 목록까지 원본 그대로입니다.

// sums-admin-frontend/src/data/mockData.ts — 구역 모델
export interface Zone {
  id: string;
  name: string;
  type: 'danger' | 'normal' | 'work' | 'safe';
  shape: 'circle' | 'polygon';
  coordinates: { lat: number; lng: number }[];
  radius?: number;
}

생략한 것도 분명합니다. 원본의 Leaflet 지도는 타일 서버가 필요해서, 외부 요청 없는 이 데모에서는 인라인 SVG 공장 약도로 바꾸고 위경도 좌표를 약도 좌표로 다시 배치했습니다. 인증·권한, 실제 디바이스 연동, 30여 화면 중 26개도 뺐습니다. 위반 로그는 서버 푸시 대신 브라우저 타이머가 흘려보냅니다.

*

도메인 규칙 하나는 살려 두었습니다. 정책에 연결되지 않은 구역에서는 위반이 발생하지 않습니다. 휴게 구역에 로그가 없는 이유이고, 약도에 새 구역을 그려도 정책 탭에서 그 구역을 정책에 넣기 전까지는 조용한 이유입니다.

결정: 로그는 이름을 복사해 둡니다

통제 로그는 policyId 같은 외래키와 함께 당시의 정책명·구역명·직원명을 문자열로 함께 저장합니다. 정규화 관점에서는 중복이지만, 정책은 수정되고 삭제되는 반면 로그는 사건 당시를 증언해야 합니다. 이름을 조인으로 되찾으면 '지금의 이름'이 나오지, '그때의 이름'이 나오지 않습니다. 덤으로 로그 목록 조회에서 조인이 사라집니다. 이 데모의 로그 스트림도 같은 구조로 이름을 복사해 저장합니다.
