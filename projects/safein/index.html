safein. 성수 물류센터 신축현장

관제에서 TBM·공지를 발행하면 근로자 앱에 선택한 언어로 즉시 도착하고, 앱에서 '확인했습니다'를 누르면 콘솔의 실시율·안전점수가 올라갑니다. 데이터는 localStorage에 남습니다.

세이프인은 코드엣지에서 백엔드를 단독으로 설계·운영 중인 산업안전 시스템입니다. 상세 기획 없이 "산업안전을 관리하는 시스템"이라는 컨셉에서 출발해, 업무 흐름 분석부터 스키마·API·배포까지 전부 제 결정이었습니다. 위 데모는 그 제품의 핵심 한 장면 — 관리자의 발행이 근로자의 언어로 도착하고, 근로자의 확인이 관리자의 숫자가 되는 순간 — 을 가공 데이터로 재현한 것입니다.

실제 시스템이 어떻게 생겼나

Prisma 스키마 기준 모델 70개, enum 36개. 조직·근로자·기기, 통제정책(구역·시간·행동조건·허용 앱), TBM(작업 전 안전조회), 안전점검(체크리스트→배포→제출→지적→조치), 사고보고, 공지, AI 번역, 통계·감사 로그까지 8개가 넘는 도메인이 한 제품을 이룹니다. 클라이언트도 둘입니다 — 관리자 Web(관제 콘솔)과 근로자 App. 그래서 백엔드도 관리자 API와 앱 API로 분리되어 있고, 근로자 앱에는 관리자용 엔드포인트가 아예 존재하지 않습니다. 데모의 POST /app/tbms/:id/confirm은 실제 앱 백엔드의 확인 엔드포인트와 같은 모양입니다.

번역은 이 제품에서 장식이 아니라 도메인입니다. 현장에는 한국어가 익숙하지 않은 근로자가 많고, 안 읽히는 안전공지는 공지가 아니기 때문입니다. 공지·TBM·통제정책 같은 콘텐츠가 만들어지면 온프레미스 vLLM이 15개 언어의 번역을 생성해 저장하고, 각 근로자는 자기 언어로 된 푸시와 피드를 받습니다.

데모가 재현한 것, 생략한 것

재현한 것: TBM 작성(작업 선택 → 위험요인 체크 → 대책 자동 제안 → 발행)과 앱 도착·확인·실시율 반영, 안전점검의 판정 → 지적사항 → 조치완료 흐름, 공지의 AI 번역 후 발행, 그리고 앱의 언어 전환 — 6개 언어(한국어·English·中文·Tiếng Việt·ភាសាខ្មែរ·Русский)로 콘텐츠와 UI가 즉시 바뀝니다. 두 화면은 서버 대신 인메모리 스토어 하나를 함께 구독하고, 상태는 localStorage에 남습니다.

생략한 것: MDM 기기 제어, FCM 푸시(도착 연출로 대체), 그리고 실제 번역 서빙입니다. 데모의 번역문은 사전 준비된 목데이터이고, 실제 제품은 vLLM이 생성합니다 — 번역 API 자체의 시연은 사례 연구의 위젯이 맡고 있으니, 이 데모는 그 번역이 제품 흐름의 어디에 끼어 있는지를 보여주는 확장입니다. 화면의 현장·인명·수치·문안은 전부 이 데모를 위해 지어낸 가공 데이터로, 실제 고객사 데이터와 무관합니다.

설계 결정 하나: 같은 데이터를, 역할에 맞는 모양으로

관리자는 전체를 조망하고 처리하며, 근로자는 자기 것만 보고 확인합니다. 이 비대칭을 if (user.isAdmin)으로 덮는 대신 모듈·Guard·DTO 수준에서 갈랐는데, 그 이야기는 별도의 글로 정리해 뒀습니다. 스키마에도 같은 철학이 있습니다. 안전 도메인에서 "누가 언제 무엇을 받았고 확인했나"는 기능이 아니라 요구사항이라, 배정 시점의 이름·소속·언어를 스냅샷 필드로 동결하고, 번역은 원문 갱신 시각과 함께 별도 테이블에 쌓습니다.

// safein-prisma · schema.prisma — 구조 수준 발췌
model TbmParticipant {
  tbmId                 String
  employeeId            String?              // SetNull — 계정이 지워져도 기록은 남는다
  employeeIdAtAssign    String               // 배정 시점 스냅샷
  employeeNameAtAssign  String
  languageAtAssigned    AppLanguage          // 이 근로자가 받을 언어
  state                 TbmParticipantState  // PENDING → CONFIRMED
  confirmedAt           DateTime?
  confirmedLanguage     AppLanguage?         // 어떤 언어로 읽고 확인했나
}

model ContentTranslation {
  entityType       TranslatableEntityType   // NOTICE · TBM · SAFETY_CHECKLIST · ZONE …
  entityId         String
  fieldKey         String
  language         AppLanguage              // 15개 언어
  content          String
  sourceUpdatedAt  DateTime                 // 원문이 바뀌면 번역이 낡았음을 안다
  @@unique([entityType, entityId, fieldKey, language])
}

확인 기록에 confirmedLanguage가 남는 이유 — 사고가 났을 때 "공지는 나갔다"로는 부족합니다. 어떤 언어로 읽고 확인했는지까지 남아야 전달이 증명됩니다. 데모의 실시율 카운터는 이 레코드의 축약판입니다.

온프레미스 GPU에서 vLLM으로 15개 언어를 감당한 이야기는 '온프레미스에서 LLM을 서빙한다는 것'에, 프로젝트 전체 서사는 사례 연구에 있습니다.
