앱에서 예약을 만들면 관리자 테이블과 KPI가 즉시 갱신되고, 관리자에서 상태 뱃지를 누르면 앱의 예약 카드가 함께 바뀝니다. 데이터는 localStorage에 남습니다.

루키는 반려동물 슈퍼앱 프로토타입입니다. 고객 앱과 관리자 콘솔, 그리고 각각을 받치는 NestJS 백엔드 둘 — 레포 네 개가 한 제품을 이룹니다. 위 데모는 두 백엔드의 시드 데이터를 그대로 옮겨, 같은 데이터를 마주 보는 앱과 콘솔을 한 화면에 재현한 것입니다.

하나의 제품, 네 개의 레포

고객 쪽은 트위터형 피드·매거진·지도 예약·쇼핑·DM까지 40여 화면의 React 앱(비공개)이고, 업체 쪽은 대시보드부터 정산·CCTV까지 13개 화면의 관리자 콘솔(비공개)입니다. 그 뒤에 NestJS 11 + Prisma 6 + SQLite 백엔드가 둘 있습니다. 앱용은 /content/*, 관리자용은 /admin/*을 서빙하고, 프론트는 API 호출이 실패하면 내장 목데이터로 조용히 물러서는 구조라 백엔드 없이도 전체가 동작합니다. 데모라는 목적에 맞춘 이 fallback이 이번 정적 재현도 가능하게 했습니다.

시드가 곧 제품이다

목데이터를 Math.random()으로 만들면 새로고침마다 숫자가 흔들려 데모가 거짓말처럼 보입니다. 그래서 앱 백엔드의 시드는 참여도를 결정적 공식으로 만듭니다. 카테고리 가중치와 작성자별 부스트가 좋아요·댓글·조회수를 일관되게 끌어올리는 방식입니다.

// lookie-app-backend · src/mock-data/content.mock.ts
export const mockPosts: PostSchema[] = postBlueprints.map((post, index) => {
  const weight = categoryWeight[post.category];
  const base = 36 + index * 5 + weight * 18 + (post.engagementBoost ?? 0) * 20;
  return { id: index + 1, likes: base + 30, retweets: Math.floor(base * 0.16),
           comments: Math.floor(base * 0.24), views: base * 32, /* … */ };
});

위 데모의 피드는 이 공식을 자바스크립트로 그대로 이식했습니다. 화면의 좋아요 수는 원본 서버가 계산해 내려줬을 값과 동일한 숫자입니다.

배열 없는 DB에 배열 넣기

관리자 백엔드에서 흥미로웠던 결정 하나. SQLite에는 배열 컬럼이 없고 Prisma도 SQLite에서는 스칼라 리스트를 지원하지 않습니다. 상품의 이미지·태그, 고객의 반려동물 목록 같은 배열 필드를 어디에 둘 것인가 — 테이블을 쪼개는 대신 JSON 문자열 컬럼을 택하고, 직렬화와 역직렬화를 서비스 경계 한 곳에 모았습니다.

// lookie-admin-backend · src/admin/admin.service.ts
const JSON_FIELDS: Record<string, string[]> = {
  product: ['images', 'videos', 'tags'],
  customer: ['pets'],
};

private strip(record: Record<string, any>, modelKey?: string) {
  const { createdAt, updatedAt, ...rest } = record;
  if (modelKey && JSON_FIELDS[modelKey]) {
    for (const field of JSON_FIELDS[modelKey]) {
      if (typeof rest[field] === 'string') {
        try { rest[field] = JSON.parse(rest[field]); } catch { /* keep as-is */ }
      }
    }
  }
  return rest;
}
*

어드민 데모용 백엔드라는 규모에서는 정규화보다 경계 한 곳의 변환이 싸게 먹힙니다. 데이터가 커지고 배열 안을 쿼리해야 하는 순간이 오면 그때 테이블로 승격하면 됩니다. 미루는 것도 설계입니다.

데모가 재현한 것, 생략한 것

재현한 것: 홈 피드의 카테고리 필터와 좋아요·북마크 토글, 예약 생성, 상품 주문, 그리고 관리자 쪽 KPI 카드·예약 테이블·상태 뱃지 전환입니다. 두 화면은 서버 대신 인메모리 스토어 하나를 함께 구독하고, localStorage가 SQLite의 역할을 대신해 새로고침에도 데모 데이터가 남습니다. 유저 8명, 게시글 24건, 예약 펫 12마리, KPI 수치까지 모두 두 백엔드의 실제 시드입니다.

생략한 것: 40여 화면 중 대부분 — 매거진 블록 에디터, DM, CCTV 뷰어, 정산 — 은 걷어냈습니다. 원본 시드의 사진은 전부 Unsplash 핫링크인데, 이 데모는 외부 요청을 만들지 않는 것을 원칙으로 하므로 자리표시 타일로 바꿨습니다. 인증도 없습니다. 남긴 것은 이 제품의 핵심 한 줄 — 앱의 행동이 관리자의 숫자가 되는 순간입니다.
