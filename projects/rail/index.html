정규식 철길Rail

정규식은 한번 짜고 나면 다시 읽기 어렵습니다. 그래서 직접 파싱해서 철도 다이어그램으로 눕혀봤습니다. 아래 칸에 정규식을 넣으면, 손으로 짠 파서가 문법 트리를 세우고 레이아웃 엔진이 철길을 놓습니다.

예시
리터럴 이스케이프 (\d \w …) 문자 클래스 […] 그룹 (…) ↺ 아래 고리 = 반복 ⌒ 위 아치 = 생략 가능

정규식은 쓰기 전용 언어(write-only)라고들 합니다. 두 달 전의 내가 짠 (?:[a-z0-9._%+-]+)@…를 오늘의 내가 못 읽습니다. 읽으려면 결국 머릿속에서 파싱을 해야 하는데 — 그럼 그 파싱을 코드로 옮기면 되지 않나, 에서 시작했습니다.

문법이 그대로 코드가 된다

정규식은 우선순위가 뚜렷합니다. 선택(|)이 가장 느슨하고, 그 안에 연속이, 그 안에 수량자(*, +, ?, {})가, 그 안에 원자(문자·그룹·클래스)가 있습니다. 이 우선순위를 그대로 함수 하나씩으로 옮기면 재귀 하강 파서가 됩니다. 라이브러리가 필요 없습니다.

parseAlternation() {          // 가장 느슨한 단계
  const opts = [this.parseSequence()];
  while (this.peek() === "|") { this.next(); opts.push(this.parseSequence()); }
  return opts.length === 1 ? opts[0] : { t: "alt", opts };
}

토크나이저를 따로 두지 않았습니다. 정규식은 대부분 한 글자가 한 토큰이라, 문자열 위를 커서 하나로 훑는 게 더 정직했습니다. 잘못된 입력은 위치를 담은 에러로 던져서, 입력창 아래 ▲로 정확히 어디가 틀렸는지 가리킵니다. 위 데모에서 a{ 를 쳐보세요.

철길은 세 숫자로 놓인다

레이아웃 엔진의 모든 조각은 자기를 딱 세 숫자로 설명합니다 — 가로 width, 레일 위로 뻗는 up, 아래로 뻗는 down. 이 셋만 있으면 어떤 조각이든 (x, y)에 그릴 수 있고, 놀랍게도 수량자는 합성으로 공짜로 나옵니다.

// 0회 이상(*) = "생략 가능"과 "1회 이상"의 합성
case "star": return new Optional(new OneOrMore(build(node.node)));
*

분기 레일은 SVG의 arc 대신 큐빅 베지어로 그렸습니다. arc의 sweep 플래그를 방향마다 맞추다 두 번 틀린 뒤, "차라리 끝점 두 개로 곡선을 뽑자"로 바꿨더니 코드도 짧아지고 레일도 더 부드러워졌습니다. 가끔 정답은 더 단순한 도구입니다.

글자 폭은 재서 안다

박스 크기를 글자 수로 어림하면 한글·기호에서 어긋납니다. 그래서 화면 밖 <canvas>의 measureText로 실제 픽셀 폭을 잽니다. 브라우저가 이미 아는 걸 굳이 다시 추측할 이유가 없었습니다.

일부러 안 그린 것

  • 역참조(\1)와 유니코드 속성(\p{…}). 그리려면 다이어그램이 트리를 넘어 그래프가 됩니다. 이 도구의 목적(구조를 눈으로 보기)에는 지금 범위가 맞았습니다.
  • 매칭 엔진. 이건 문법을 그리는 도구지 실행하는 도구가 아닙니다. NFA로 확장하는 건 다음 주말의 즐거움으로 남겨뒀습니다.
