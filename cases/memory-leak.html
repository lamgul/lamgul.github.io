7일마다 죽던 서버근본 원인까지

입사 1주차에 들은 이야기 — "이 서비스는 7~9일마다 다운되는데, 벌써 1년째입니다." 급한 불을 끄는 것과 다시는 안 나게 만드는 것은 다른 일이었습니다. 저는 후자까지 가고 싶었습니다.

역할SM · 장애 분석 및 해결
도구Scouter APM · Heap Dump · JVM
유형트러블슈팅 · 성능
7일→30일+장애 주기사실상 재발 없음
+30%트랜잭션 속도GC 부담 해소
80%+한 클래스가 점유힙덤프에서 발견
1년→해결방치된 문제근본 원인 제거

'왜 이런 일이 1년이나 방치됐을까'라는 의문이 들었지만, 어쨌든 해결이 먼저였습니다. 흥미로운 건 이 문제가 여러 겹의 오해로 덮여 있었다는 점입니다. 하나씩 벗겨야 했습니다.

가설 ①

하드웨어 병목? — 아니었다

장애 시점에 CPU와 메모리 사용률이 100%에 도달하는 걸 확인했습니다. 자연스럽게 하드웨어 병목을 의심했습니다. 그런데 서버 사양은 충분히 넉넉했고, DB 성능도 정상이었습니다. 물리적 자원이 부족한 게 아니라면, 자원을 낭비하는 무언가가 있다는 뜻이었습니다.

단서

넉넉한 메모리, 그런데 쌓이는 스왑

물리 서버의 캐시 정보를 뒤지다 이상한 걸 봤습니다. PC 메모리는 넉넉한데 스왑 메모리가 계속 쌓이고 있었습니다. 메모리가 남는데 왜 디스크로 밀어내지? 여기서 문제가 애플리케이션 레이어, 특히 JVM 안에 있다는 감이 왔습니다.

진단

Scouter APM이 그림을 완성했다

Scouter APM으로 트랜잭션 동작을 추적하자 퍼즐이 맞춰졌습니다. DB에서 가져온 데이터를 자바 객체로 변환하는 과정에서 지연이 발생했고, 힙메모리가 부족해지자 빈번한 Full GC가 일어나면서 CPU를 100%로 밀어올리고 있었습니다. 100%는 원인이 아니라 증상이었던 겁니다.

임시 조치

먼저 급한 불부터, 정직하게 '임시'라고 부르며

당장 운영을 안정시키기 위해 힙메모리를 8GB에서 16GB로 증설하고, 클라이언트와 협의해 정기 재기동을 도입했습니다. 트래픽이 몰리는 시간에도 시스템이 쾌적해졌습니다. 하지만 저는 이걸 임시방편이라고 분명히 못 박아 두었습니다. 메모리를 늘리고 주기적으로 껐다 켜는 건, 누수를 늦추는 것이지 없애는 게 아니니까요.

근본 해결

퇴근 후 힙덤프를 배웠다

솔직히 그 시점의 저는 힙덤프를 제대로 분석할 능력이 부족했습니다. 마침 재직자 대상 트러블슈팅 교육이 있어 퇴근 후 녹화본으로 공부했습니다. 배운 대로 힙덤프를 떠서 분석하니, 특정 클래스 하나가 힙의 80% 이상을 붙들고 있었습니다. 코드를 따라가 보니 원인은 명확했습니다 — 멀티스레드 환경에서 객체를 참조하는 과정에 동기화 처리가 빠져 있었습니다.

// 여러 스레드가 동시에 접근하며 객체가 회수되지 못하고 쌓였다
public void register(Key k, Session s) {
  cache.put(k, s);          // 동기화 없음 → 상태가 어긋나며 참조가 남음
}
// 스레드 안전성을 확보하자 누수가 멈췄다
public synchronized void register(Key k, Session s) {
  cache.put(k, s);
}

synchronized로 스레드 안전성을 확보하자, 1년을 끌어온 메모리 누수가 사라졌습니다. 장애 주기는 7일에서 30일 이상으로 늘었고 — 사실상 재발하지 않았고 — Full GC 부담이 사라지며 트랜잭션 속도도 약 30% 개선됐습니다.

힙 사용률 — 누수 vs 근본 해결누수 있음
힙 30%
점선은 GC 임계치. BEFORE는 GC가 다 회수하지 못해 서서히 차오르다 재기동하고, AFTER는 매번 기준선으로 복귀합니다.
*

제가 이 사례를 아끼는 이유는 화려한 기술 때문이 아닙니다. 증상(CPU 100%)을 원인으로 착각하지 않고, 임시 대응과 근본 해결을 구분해 끝까지 간 것. 그리고 능력이 부족하면 퇴근 후에라도 배워서 채운 것. 이게 제가 문제를 대하는 방식입니다.

이 경험은 지금도 습관으로 남아 있습니다. 막히면 코드를 노려보기 전에 "내가 지금 뭘 사실이라고 가정하고 있지?"부터 적어봅니다. 그때 CPU 100%를 원인이라 가정했다면, 저는 아직도 서버를 껐다 켜고 있었을 겁니다.
