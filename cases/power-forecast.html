사내 AI 역량 팀으로 전력사용량 예측 경진대회에 참여했습니다. 모두가 같은 데이터를 받았습니다. 차이를 만든 건 모델이 아니라 데이터를 어떻게 이해했는가였습니다.

처음엔 단순하게 생각했습니다. "모든 데이터를 그대로 모델에 넣으면 되지 않을까?" 하지만 EDA(탐색적 데이터 분석)를 진행하면서 생각이 완전히 바뀌었습니다.

관찰

전력에는 시간의 리듬이 있다

데이터를 시각화하자 전력사용량에 뚜렷한 시간대 패턴이 보였습니다 — 22–05시 저점, 06–09시 상승, 10–16시 고점, 17–21시 하락. 이 리듬을 무시하고 데이터를 평평하게 다루면 모델이 중요한 구조를 놓칠 게 분명했습니다. 그래서 시간대와 건물을 축으로 데이터를 분류했습니다. 각 시간대·건물이 고유한 패턴을 갖기 때문입니다.

분석

상관관계가 알려준 것 — 그리고 함정

기상 요인을 분석하니 기온이 전력사용량과 가장 높은 상관을 보였고(r=0.85), 강수량은 거의 무관했습니다(r=0.12). 그런데 함정이 있었습니다. 기온·습도·풍속이 서로 높은 상관을 가지고 있었던 겁니다. 이대로 다 넣으면 다중공선성 문제로 모델이 흔들릴 수 있었습니다.

통찰

사람은 온도가 아니라 '불쾌함'에 반응한다

여기서 도메인 지식이 개입했습니다. "사람들이 냉방을 켜는 건 단순히 온도가 높아서가 아니라 불쾌하기 때문이 아닐까?" 그래서 기온과 습도를 결합해 불쾌지수를, 기온과 풍속을 결합해 체감온도를 파생 변수로 만들었습니다. 서로 얽혀 있던 세 변수를, 의미가 분명한 두 변수로 압축한 셈입니다. 반대로 강수량, 비전기 냉방시설, 태양광 보유 여부는 제거했습니다 — 상관이 낮거나 건물별로 동일해 정보가 없었으니까요.

# 얽힌 기상 변수를 '사람이 느끼는 것'으로 압축
df["discomfort"] = discomfort_index(df.temp, df.humidity)  # 불쾌지수
df["feels_like"] = apparent_temp(df.temp, df.wind)        # 체감온도
df = df.drop(columns=["rain", "nonelec_cooling", "solar"])   # 잡음·중복 제거

결과

모델이 아니라 데이터가 순위를 만들었다

결과는 AI 부문 리더보드 10위. 입상까지는 아니었지만, 이 프로젝트가 제게 남긴 건 순위보다 큰 확신이었습니다. 원시 데이터를 그대로 쓰는 게 아니라, 도메인 지식으로 가공하는 것이 성능을 얼마나 바꾸는지를 몸으로 체감했으니까요. 같은 데이터, 같은 모델이라도 Feature Engineering이 승부를 가릅니다.
