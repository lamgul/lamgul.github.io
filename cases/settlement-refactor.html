결제 수단마다요금이 달랐다

"주차 요금이 왜 다르게 계산되나요?" 확인해보니 웹에서 결제한 금액과 키오스크에서 결제한 금액이 달랐습니다. 같은 주차 시간인데도. 이건 단순 버그가 아니라 신뢰의 문제였습니다.

코드를 추적하니 원인이 드러났습니다. 정산 로직이 5곳에 중복되어 있었고, 각각 미묘하게 달랐습니다. 웹 결제, 키오스크, 카카오페이, 월정액, 관리자 수동 정산. 한 곳에서 할인 정책을 바꾸면 나머지 네 곳도 똑같이 바꿔야 했는데, 실수로 한둘을 빠뜨리면 결제 수단에 따라 요금이 달라지는 버그가 생겼습니다.

재정의

중복 코드가 아니라 일관성의 문제

처음엔 "중복 코드를 합치자"로 접근할 뻔했습니다. 하지만 한 발 물러서서 보니 이건 코드 스타일 문제가 아니었습니다. 사용자 입장에서는 어디서 결제하든 같은 금액이 나와야 한다 — 그게 지켜지지 않는 비즈니스 일관성의 문제였습니다. 문제를 어떻게 부르느냐가 해법의 방향을 바꿨습니다.

문제 ②

거대한 if-else, 그리고 늘어나는 결제 수단

또 다른 문제는 모든 정산 로직이 하나의 거대한 메서드에 if-else로 몰려 있다는 점이었습니다. 웹인지 키오스크인지 카카오페이인지 구분하는 분기가 복잡하게 얽혀 있었고, 새 결제 수단을 추가하려면 이 괴물 메서드에 또 다른 if 블록을 끼워 넣어야 했습니다. 확장할수록 위험해지는 구조였습니다.

설계

공통은 골격으로, 차이는 전략으로

Template Method와 Strategy 패턴을 조합해 재설계했습니다. 공통 정산 로직은 추상 클래스의 골격(template)에 정의하고, 결제 수단별로 달라지는 부분만 구체 클래스에서 구현했습니다. 할인 정책도 독립적인 전략으로 분리해 자유롭게 조합할 수 있게 했습니다.

abstract class Settlement {           // 골격은 한 곳에만
  final Money calculate(Parking p) {
    Money base = baseFare(p.duration());  // 공통
    return discount.apply(base);         // 전략으로 위임
  }
  abstract Money baseFare(Duration d);   // 수단별 차이만
}
// 웹·키오스크·카카오페이는 baseFare만 다르게 구현
결과

한 곳을 고치면 모두가 같아진다

리팩토링 후, 할인 정책을 바꿀 때 한 곳만 수정하면 모든 결제 수단에 일관되게 적용됐습니다. 금액 불일치 민원은 사라졌고, 새 결제 수단을 추가하는 시간도 절반으로 줄었습니다. 코드 라인 수는 약 40% 감소했지만, 오히려 더 명확하고 유지보수하기 쉬워졌습니다. 좋은 추상화는 코드를 줄이면서 동시에 더 읽히게 만든다는 걸 배운 사례입니다.
