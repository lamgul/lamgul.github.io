SafeIn산업안전관리 시스템

상세 기획서가 없었습니다. "산업안전을 관리하는 시스템이 필요하다"는 컨셉 수준의 요구에서 시작해, 업무 흐름을 직접 분석하고 시스템 전체를 설계하는 것부터가 제 일이었습니다. 백엔드는 단독으로 맡았습니다.

역할백엔드 단독 개발 · 시스템 설계
스택NestJS · PostgreSQL · NGINX · vLLM · AWS

데모관제 콘솔 + 근로자 앱 직접 클릭해보기 ↗

온프레미스GPU 추론 최적화vLLM · 양자화
Web + App역할별 API 분리관리자 · 근로자

산업안전관리 시스템에서는 데이터 누락이나 상태 불일치가 곧 현장 업무의 혼선으로 이어집니다. 근로자가 위험을 신고했는데 관리자 화면에 안 뜨거나, 안전공지가 일부에게만 전달되면 그건 단순 버그가 아니라 안전 사고의 틈이 됩니다. 그래서 이 프로젝트의 중심에 정합성을 뒀습니다.

Situation

기능이 아니라 '흐름'으로 정리하기

요구사항을 기능 단위로 나열하는 대신, 실제 현장의 업무 흐름을 기준으로 정리했습니다. 사업장 관리, 근로자 관리, 위험구역, 안전공지, 위험 신고, TBM(작업 전 안전점검), 안전 이벤트 이력, AI 번역 — 각 기능은 독립적으로 보이지만 현장에서는 하나의 흐름으로 이어져야 하기 때문입니다. 근로자의 신고가 관리자의 처리로, 관리자의 공지가 근로자의 알림으로 끊김 없이 흘러야 합니다.

Action

역할이 다르면 API도 나눈다

관리자와 근로자는 같은 데이터를 서로 다른 흐름으로 씁니다. 관리자는 전체를 조망하고 처리하며, 근로자는 자기 것만 보고 신고합니다. 그래서 관리자 Web과 근로자 App의 API를 분리하고 권한 구조를 따로 설계했습니다. 근로자의 신고·이벤트가 관리자의 처리 흐름으로 이어지도록, 백엔드에서 상태와 이력을 일관되게 관리하는 구조를 만들었습니다.

Action

외국인 근로자를 위한 AI 번역, 온프레미스로

현장에는 한국어가 익숙하지 않은 외국인 근로자가 많습니다. 안전공지가 안 읽히면 공지가 아닙니다. 그래서 공지를 15개 언어로 자동 번역하는 AI API를 붙였습니다. 번역은 Gemma, 음성 공지는 Whisper로 STT 처리합니다. 민감한 현장 데이터를 외부 API로 흘려보내지 않기 위해 온프레미스 GPU 서버를 직접 구축했고, vLLM과 양자화로 추론을 최적화해 현실적인 응답 속도를 확보했습니다.

*

AI를 "붙였다"가 아니라 운영 가능한 형태로 서빙했다가 핵심이었습니다. 모델을 부르는 건 쉽지만, 온프레미스 GPU에서 여러 요청을 감당하며 지연을 관리하는 건 다른 문제입니다. vLLM의 배치 처리와 양자화가 그 간극을 메웠습니다.

Result

확장이 구조를 흔들지 않도록

단독 개발이었기 때문에 기능 우선순위, API 설계, 데이터 구조, 예외 처리, 배포까지 모든 결정을 스스로 내리고 책임졌습니다. 그만큼 미래를 대비한 경계에 신경 썼습니다. 도메인별로 책임을 분리해 구현한 덕분에, 정책이 바뀌거나 새 도메인을 추가할 때 기존 코드의 수정 범위를 최소화할 수 있었습니다. 지금도 운영(AWS)과 유지보수를 이어가며, 실제 장애에 대응하고 기능을 넓히고 있습니다.

설계 결정들은 ADR과 기술 문서로 남겨, 저만 아는 시스템이 조직의 리스크가 되지 않도록 했습니다.
