사이트마다 HTML 구조가 다르고 크롤링 방식도 달라야 했습니다. 이걸 하나의 거대한 함수로 작성하면 유지보수는 불가능해집니다. 그래서 처음부터 변화의 축을 격리하는 구조를 고민했습니다.

구조 ①

흐름을 필터로 쪼갠다 — Pipe & Filter

먼저 데이터 처리 흐름을 단계로 나눴습니다. 수집 → 정제 → 변환 → 저장. 각 단계를 독립적인 필터로 구성하는 Pipe & Filter 아키텍처를 적용했습니다. 각 필터는 입력을 받아 처리 후 다음 필터로 넘깁니다. 단계 간 결합이 사라지니 각 단계를 독립적으로 테스트할 수 있게 됐습니다.

구조 ②

사이트별 차이는 Adapter 뒤에 숨긴다

다음은 사이트별 크롤러를 어떻게 통합하느냐였습니다. Adapter 패턴으로 공통 인터페이스 CrawlerInterface를 정의하고, 사이트마다 Adapter를 구현했습니다. 새 사이트를 추가할 때는 기존 코드를 단 한 줄도 건드리지 않고 새 Adapter만 추가하면 됩니다. 개방-폐쇄 원칙(OCP)이 코드로 실현된 지점입니다.

구조 ③

도메인을 명시적으로 — DDD

DDD 관점에서 Article, Category, Source 같은 도메인 모델을 명시적으로 정의했습니다. 비즈니스 로직은 도메인 객체에 캡슐화하고, UseCase 계층에서 이를 조합해 비즈니스 흐름을 표현했습니다. 크롤링이라는 지저분해지기 쉬운 영역에, 읽히는 뼈대를 세운 셈입니다.

운영

현실은 언제나 지저분하다

실무적인 문제도 있었습니다. 웹사이트의 Rate Limiting과 봇 감지를 우회해야 했습니다. Exponential Backoff 전략으로 요청 간격을 조절했고, 장애가 발생하면 Slack으로 알림이 오도록 했습니다. 스케줄러로 매일 정해진 시간에 자동 수집하게 만들었습니다. 그리고 Clean Code 원칙 — 명시적 네이밍, 단일 책임, 의존성 역전 — 을 지켜 팀원들이 읽기 쉽게 구조화했습니다.

결과

확장이 두렵지 않은 파이프라인

결과적으로 확장 가능하고 유지보수가 쉬운 크롤링 파이프라인을 얻었습니다. 새 데이터 소스를 추가할 때도 최소한의 코드 추가만으로 가능했습니다. "새 요구가 올 때마다 기존 코드를 고쳐야 하는" 구조와, "새 요구를 덧붙이기만 하면 되는" 구조의 차이 — 그 차이가 곧 시스템의 수명이라고 생각합니다.
